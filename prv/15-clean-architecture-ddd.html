Tại Sao Clean Architecture + DDD?

Khi codebase nhỏ (1 controller, 1 service, 1 entity), bạn không cần bất kỳ pattern nào. Code phẳng và "chạy được" là đủ. Nhưng khi:

  • Codebase phình lên 50K+ dòng
  • Có 5-10 dev cùng touch chung modules
  • Business rules thay đổi liên tục theo nghiệp vụ
  • Bạn cần swap infrastructure (đổi DB, đổi messaging) không break code

Thì code phẳng trở thành "big ball of mud": thay 1 dòng phải test 10 module, mỗi feature mới mất 1 tuần. Clean Architecture + DDD là boundary technique để giữ codebase navigable theo thời gian.

🎯 Vì sao file này là "Senior Signal"

Junior code "làm sao chạy". Senior code "làm sao maintain 3 năm sau". Interviewer hỏi architecture không phải để check syntax — họ check xem bạn có hiểu cost của coupling, cost của swap technology, và cost của onboarding. Đây là kiến thức không học được nếu chỉ làm CRUD app.

1. Bài Toán Thực Tế — Order System Của Một Sàn TMĐT

Tình huống: Sàn ShopVN có monolith hỗn loạn

Hệ thống order của sàn ShopVN có OrderController gọi trực tiếp OrderRepository, validate giỏ hàng, trừ stock, tạo invoice, ghi log, gửi email — tất cả trong 1 method controller dài 300 dòng. Khi cần thêm 1 phương thức thanh toán mới (ví dụ VNPay sau khi đã hỗ trợ Momo và COD), engineer phải đụng vào:

  • Controller (validation rules thay đổi)
  • Service layer (logic rải rác giữa OrderService và PaymentService)
  • Repository (orders table thêm column payment_method)
  • Entity classes (Order, Customer mọc thêm properties)
  • Database schema
  • External service (fee calculator, fraud check)

Business logic scatter khắp nơi: controllers validate, services orchestrate, entities mutate state của chính mình, repositories enforce rules. Thêm 1 feature = sửa 15 file + regression test 8 modules. Code review mất 3 ngày. PR conflict liên tục.

🔥 Triệu chứng "Big Ball of Mud"
  • Không biết business rule nằm ở đâu (Controller? Service? Entity? Stored procedure?)
  • Test unit phải boot full Spring context → mỗi test 5-10 giây
  • Đổi DB từ MySQL → PostgreSQL = rewrite 30% code
  • Junior mới onboard mất 3 tháng mới "biết nên sửa file nào"

5 câu hỏi suy ngẫm

  1. Vì sao thêm 1 payment method lại cần đụng controller?
  2. Làm sao isolate "payment method logic" để có thể thay đổi mà không ảnh hưởng order/stock logic?
  3. "Bounded context" là gì và vì sao ShopVN cần nhiều bounded context?
  4. Nếu Order tồn tại trong cả OrderContext và ShippingContext, 2 chỗ này khác nhau thế nào?
  5. Làm sao tránh Customer aggregate phình lên 50 fields vì trộn lẫn nhiều concern?

2. Mental Model

4 Layers & Dependency Rule

4 layers Clean Architecture — vòng trong (Domain) không biết gì về vòng ngoài

Dependency Rule: Code chỉ depend INWARD. Domain layer không biết HTTP, database, hoặc framework. Infrastructure implement domain interface.

🎯 Vì sao Domain layer phải "pure"?

Business rules là tài sản giá trị nhất của hệ thống — chúng thể hiện "công ty làm gì để kiếm tiền". Database, framework, HTTP là chi tiết kỹ thuật có thể thay đổi (MySQL → Postgres, Spring → Quarkus, REST → gRPC). Nếu domain biết về MySQL, đổi DB = rewrite business rules. Tách rời = business rules sống lâu hơn mọi technology choice.

🎯 Vì sao test domain mà cần boot Spring là dấu hiệu xấu?

Test domain logic nên chạy milliseconds, không phải seconds. Nếu phải @SpringBootTest mới test được business rule → có nghĩa domain đang phụ thuộc framework. Mỗi PR developer chạy 200 test trong 30s vs 30 phút = khác biệt productivity rõ rệt.

DDD Building Blocks

Block 1

Entity

Có identity duy nhất. Order với orderId — mutable (status đổi từ PENDING → PAID → SHIPPED), nhưng vẫn cùng 1 order. Identity tồn tại kể cả tất cả attribute đổi.

Block 2

Value Object

Không có identity, định nghĩa bằng attributes. Money(100, "VND") — immutable, cần Money khác → new instance. So sánh bằng value: Money(100,VND).equals(Money(100,VND)) → true.

Block 3

Aggregate

Cluster entities + value objects với 1 aggregate root. Transaction boundary = aggregate boundary. Save toàn bộ aggregate atomically. Không aggregate nào chứa aggregate root khác.

Block 4

Repository

Interface trong domain layer (business chỉ care abstraction). Implementation ở infrastructure. OrderRepository.findById(orderId) — không biết SQL/JPA/Mongo.

Block 5

Domain Service

Logic không thuộc về entity/VO nào cụ thể. Ví dụ PricingDomainService.calculateOrderTotal(order, promotions, taxRules) — operate trên multiple entities, không "thuộc" riêng entity nào.

Block 6

Domain Event

Something happened (OrderPlaced, ItemShipped). Immutable, publishable. Decouple bounded contexts → ShippingContext không cần biết OrderContext.

Bounded Context — Cùng từ "Order", khác nghĩa

Bounded context là boundary nơi 1 model áp dụng. Trong sàn ShopVN:

Context"Order" nghĩa làProperties quan tâm
OrderContext (checkout)Giỏ hàng đã confirm + payment intentitems, totalPrice, paymentMethod, customerId, status
InventoryContextLệnh trừ stockorderId, items[productId, quantity, warehouseId]
ShippingContextĐơn cần giaoorderId, address, weight, courier, trackingNumber
FinanceContextDoanh thu phải ghi nhậnorderId, invoiceNumber, taxAmount, settlementDate

Cùng từ "Order", 4 nghĩa khác nhau. Bounded context buộc bạn dịch tường minh (Anti-Corruption Layer) khi cross boundary.

🎯 Vì sao không gộp thành 1 "Order" mega-class?

Nếu có 1 class Order chứa tất cả fields (items, address, invoiceNumber, taxAmount, trackingNumber, ...) → class này có 50+ fields và 4 team cùng modify. Mỗi PR sẽ conflict. Mỗi test phải mock 30 dependencies. Mỗi DB query SELECT * FROM orders load 50 column dù chỉ cần 3. Bounded context tách thành 4 model nhỏ, mỗi team own 1 context, ship features độc lập.

3. Production-Grade Implementation

Money — Value Object (Immutable, Type-Safe)

package com.shopvn.order.domain;

import java.math.BigDecimal;
import java.math.RoundingMode;
import java.util.Currency;
import java.util.Objects;

// Value Object: No identity, immutable, compared by value
public final class Money {
    private final BigDecimal amount;
    private final Currency currency;

    // Private constructor — force factory method để validate
    private Money(BigDecimal amount, Currency currency) {
        if (amount.signum() < 0) {
            throw new IllegalArgumentException("Money amount cannot be negative");
        }
        if (currency == null) {
            throw new IllegalArgumentException("Currency cannot be null");
        }
        // setScale(2) → tránh floating-point error, đảm bảo 2 chữ số thập phân
        this.amount = amount.setScale(2, RoundingMode.HALF_UP);
        this.currency = currency;
    }

    public static Money of(BigDecimal amount, Currency currency) {
        return new Money(amount, currency);
    }

    public Money add(Money other) {
        // Cross-currency operation phải EXPLICIT — không silent bug
        // (VND + USD = meaningless sum, force caller convert trước)
        requireSameCurrency(other);
        return new Money(this.amount.add(other.amount), this.currency);
    }

    public Money multiply(int factor) {
        return new Money(this.amount.multiply(BigDecimal.valueOf(factor)), this.currency);
    }

    public boolean isGreaterThan(Money other) {
        requireSameCurrency(other);
        return this.amount.compareTo(other.amount) > 0;
    }

    private void requireSameCurrency(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new IllegalArgumentException(
                "Currency mismatch: " + this.currency + " vs " + other.currency);
        }
    }

    @Override public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Money)) return false;
        Money m = (Money) o;
        return amount.equals(m.amount) && currency.equals(m.currency);
    }
    @Override public int hashCode() { return Objects.hash(amount, currency); }
    @Override public String toString() { return amount.toPlainString() + " " + currency.getCurrencyCode(); }
}
🎯 Vì sao Money là Value Object, không phải BigDecimal trần?
  • Type safety: void charge(Money amount) không thể truyền nhầm quantity (cũng là BigDecimal). Compiler bảo vệ.
  • Currency built-in: tránh bug "trừ VND khỏi USD"
  • Validation centralized: chỉ 1 chỗ check non-negative, scale 2, không scatter
  • Self-documenting: Money totalPrice rõ ràng hơn BigDecimal totalPrice

Order — Aggregate Root với Domain Events

package com.shopvn.order.domain;

import java.time.Instant;
import java.util.*;

// Aggregate Root: Transaction boundary. Mọi thay đổi đi qua Order.
public class Order {
    private final OrderId orderId;
    private final CustomerId customerId;
    private final List<OrderLine> lines;  // Entity con — Order là root
    private Money totalPrice;
    private OrderStatus status;
    private final List<Object> domainEvents = new ArrayList<>();

    // Private constructor — force factory
    private Order(OrderId orderId, CustomerId customerId) {
        this.orderId = orderId;
        this.customerId = customerId;
        this.lines = new ArrayList<>();
        this.totalPrice = Money.of(BigDecimal.ZERO, Currency.getInstance("VND"));
        this.status = OrderStatus.DRAFT;
    }

    // Factory method — encapsulate logic tạo + emit initial event
    public static Order placeNewOrder(CustomerId customerId, List<OrderLine> items) {
        Order order = new Order(OrderId.generate(), customerId);
        items.forEach(order::addLine);
        order.confirm();
        return order;
    }

    public void addLine(OrderLine line) {
        // Business invariant: chỉ add line khi order DRAFT
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("Cannot modify confirmed order");
        }
        lines.add(line);
        recalculateTotal();
    }

    public void confirm() {
        if (lines.isEmpty()) {
            throw new IllegalStateException("Cannot confirm empty order");
        }
        this.status = OrderStatus.CONFIRMED;
        recordEvent(new OrderConfirmedEvent(orderId, customerId, totalPrice, Instant.now()));
    }

    public void markPaid(PaymentId paymentId) {
        if (status != OrderStatus.CONFIRMED) {
            throw new IllegalStateException("Order must be CONFIRMED before payment");
        }
        this.status = OrderStatus.PAID;
        recordEvent(new OrderPaidEvent(orderId, paymentId, Instant.now()));
    }

    public void cancel(String reason) {
        if (status == OrderStatus.SHIPPED || status == OrderStatus.DELIVERED) {
            throw new IllegalStateException("Cannot cancel shipped order");
        }
        this.status = OrderStatus.CANCELLED;
        recordEvent(new OrderCancelledEvent(orderId, reason, Instant.now()));
    }

    private void recalculateTotal() {
        Money sum = Money.of(BigDecimal.ZERO, Currency.getInstance("VND"));
        for (OrderLine line : lines) sum = sum.add(line.subtotal());
        this.totalPrice = sum;
    }

    // Record event nhưng KHÔNG publish ngay. Publish sau khi save thành công.
    private void recordEvent(Object event) { this.domainEvents.add(event); }

    public List<Object> pullDomainEvents() {
        List<Object> events = new ArrayList<>(domainEvents);
        domainEvents.clear();
        return events;
    }

    // Getters
    public OrderId getOrderId() { return orderId; }
    public OrderStatus getStatus() { return status; }
    public Money getTotalPrice() { return totalPrice; }
    public List<OrderLine> getLines() { return Collections.unmodifiableList(lines); }
}
🎯 Vì sao methods trên Order, không phải OrderService.confirm(order)?

Khi method thuộc về entity, code có 3 lợi ích:

  • Discoverability: IDE auto-complete order. hiện tất cả operations hợp lệ. Junior mới onboard nhìn class thấy ngay nghiệp vụ.
  • Encapsulation: validate state trong method, ai cũng không thể bypass. OrderService bên ngoài có thể quên check, entity thì không.
  • Single source of truth: logic "cancel order" chỉ ở 1 nơi (Order entity), không duplicate ở 3 service.

Đây là khác biệt giữa Anemic Domain Model (entity = data bag) và Rich Domain Model (entity = data + behavior).

🎯 Vì sao pullDomainEvents() thay vì publish ngay trong method?

Nếu confirm() publish event ngay → DB save fail mà event đã ra ngoài → downstream tưởng order confirmed, nhưng DB chưa lưu. Bug data inconsistency.

Pattern đúng: entity record events vào list, repository publish sau khi save thành công. Tách rõ "ý định emit" và "thực sự emit". Đi kèm pattern này thường có Outbox Pattern (xem #18) để guarantee delivery.

Repository Pattern — Interface trong Domain, Impl ở Infrastructure

Domain layer — Interface (pure Java, no framework)
package com.shopvn.order.domain;

public interface OrderRepository {
    Optional<Order> findById(OrderId orderId);
    void save(Order order);
    List<Order> findByCustomerId(CustomerId customerId);
    // KHÔNG có @Query, KHÔNG có Spring Page<T>, KHÔNG có Specification
    // Phức tạp hơn? Đó là việc của Read Model (xem CQRS ở #17)
}
Infrastructure layer — JPA Implementation
package com.shopvn.order.infrastructure.persistence;

import com.shopvn.order.domain.*;
import org.springframework.stereotype.Repository;

@Repository
public class JpaOrderRepository implements OrderRepository {
    private final SpringDataOrderRepo jpaRepo;
    private final DomainEventPublisher eventPublisher;

    public JpaOrderRepository(SpringDataOrderRepo jpaRepo, DomainEventPublisher eventPublisher) {
        this.jpaRepo = jpaRepo;
        this.eventPublisher = eventPublisher;
    }

    @Override
    public Optional<Order> findById(OrderId orderId) {
        return jpaRepo.findById(orderId.value())
            .map(this::toDomain);
    }

    @Override
    public void save(Order order) {
        // 1. Save aggregate
        OrderJpaEntity entity = toJpaEntity(order);
        jpaRepo.save(entity);

        // 2. Publish events SAU KHI save thành công
        // Nếu save fail → exception throw → events không được publish
        order.pullDomainEvents().forEach(eventPublisher::publish);
    }

    private Order toDomain(OrderJpaEntity entity) { /* ... mapping ... */ }
    private OrderJpaEntity toJpaEntity(Order order) { /* ... mapping ... */ }
}

// Spring Data JPA interface — internal, không leak ra ngoài
interface SpringDataOrderRepo extends JpaRepository<OrderJpaEntity, String> {}
🎯 Vì sao repository interface ở Domain, impl ở Infrastructure?

Đây là Dependency Inversion Principle applied. Domain declare what it needs ("tôi cần save Order"), Infrastructure provide how ("tôi dùng JPA"). Kết quả:

  • Test domain với InMemoryOrderRepository (Map<OrderId, Order>) — không cần boot DB
  • Swap impl: đổi JPA → MongoDB chỉ cần viết MongoOrderRepository implement cùng interface
  • Domain không "biết" JPA exist → không bị tempt sử dụng JPA-specific features (Specification, Page, etc.) lẫn vào business logic

Use Case / Application Service

package com.shopvn.order.application;

import com.shopvn.order.domain.*;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

@Service
public class PlaceOrderUseCase {
    private final OrderRepository orderRepo;
    private final ProductCatalog productCatalog;
    private final InventoryService inventoryService;

    public PlaceOrderUseCase(OrderRepository orderRepo, ProductCatalog productCatalog,
                              InventoryService inventoryService) {
        this.orderRepo = orderRepo;
        this.productCatalog = productCatalog;
        this.inventoryService = inventoryService;
    }

    @Transactional
    public OrderId execute(PlaceOrderCommand command) {
        // 1. Reserve inventory (orchestration với context khác)
        for (var line : command.lines()) {
            inventoryService.reserve(line.productId(), line.quantity());
        }

        // 2. Build domain objects từ command
        List<OrderLine> orderLines = command.lines().stream()
            .map(l -> {
                var product = productCatalog.findById(l.productId());
                return new OrderLine(l.productId(), product.name(), product.price(), l.quantity());
            })
            .toList();

        // 3. Invoke domain logic
        Order order = Order.placeNewOrder(command.customerId(), orderLines);

        // 4. Persist (repository sẽ publish events sau save)
        orderRepo.save(order);

        return order.getOrderId();
    }
}

// Command = pure data structure, không có behavior
public record PlaceOrderCommand(
    CustomerId customerId,
    List<OrderLineRequest> lines
) {}

public record OrderLineRequest(ProductId productId, int quantity) {}
🎯 Vì sao tách Use Case khỏi Controller?
  • Test không cần HTTP: unit test PlaceOrderUseCase với JUnit + Mockito, không cần MockMvc
  • Multiple entry points: cùng use case có thể trigger từ REST, GraphQL, message queue, scheduled job — không duplicate logic
  • Transaction boundary rõ ràng: @Transactional ở use case level — controller chỉ HTTP concerns, không quản lý DB transaction

💡 Quy tắc package — Verify Dependency Rule

Dùng ArchUnit để enforce ở CI:

@Test
void domainShouldNotDependOnInfrastructure() {
    noClasses().that().resideInAPackage("..domain..")
        .should().dependOnClassesThat().resideInAnyPackage(
            "..infrastructure..", "org.springframework..", "javax.persistence.."
        )
        .check(importedClasses);
}

Test này fail nếu ai đó accidentally import @Entity vào domain — bắt lỗi ở PR review chứ không phải production.

4. Trade-offs & Anti-Patterns

Khi nào dùng Clean Architecture + DDD?

✅ Dùng khi
  • Hệ thống phức tạp, business rules evolve liên tục
  • Nhiều team own domains khác nhau (cần bounded context tường minh)
  • Long-term maintenance critical (e-commerce, fintech, healthcare)
  • Cần swap infrastructure mà không break business
  • Codebase > 30K LOC hoặc dự kiến grow
❌ KHÔNG dùng nếu
  • CRUD app với 3 bảng, không có business logic phức tạp
  • Prototype / MVP — lean on framework conveniences trước
  • 1 developer, simple admin tool
  • Team chưa nắm OOP solid (DDD là pattern OOP nâng cao)

Anti-Pattern #1 — Anemic Domain Model

❌ Entity chỉ là data bag
@Entity
public class Order {
    private Long id;
    private BigDecimal total;
    private String status;
    // Chỉ getters/setters
}

@Service
public class OrderService {
    public void cancel(Long id) {
        Order o = repo.findById(id);
        if (o.getStatus().equals("SHIPPED"))
            throw new Exception();
        o.setStatus("CANCELLED");
        repo.save(o);
    }
}
✅ Rich Domain Model
public class Order {
    private OrderStatus status;

    public void cancel(String reason) {
        if (status == OrderStatus.SHIPPED)
            throw new IllegalStateException(
                "Cannot cancel shipped order");
        this.status = OrderStatus.CANCELLED;
        recordEvent(new OrderCancelledEvent(...));
    }
}
🎯 Vì sao Anemic Model có hại?
  • Logic scatter: 3 nơi gọi setStatus("CANCELLED") → 3 chỗ phải nhớ check shipped. Khi rule đổi → sửa 3 chỗ.
  • No encapsulation: bất cứ ai có order reference đều set status = "BANANA" → invariant broken.
  • Khó test: phải test thông qua service + repo, không test riêng business rule được.
  • "Service" phình to: cuối cùng OrderService 2000 dòng — God class.

Anti-Pattern #2 — Repository Trả Về Infrastructure Types

// ❌ Domain phụ thuộc Spring Data
public interface OrderRepository {
    Page<Order> findAll(Pageable pageable);  // Page là Spring type!
    Specification<Order> getActiveSpec();    // Spring Data Specification
}
🎯 Vì sao sai?
  • Domain layer phụ thuộc Spring → test domain phải có Spring → mất point của Clean Architecture
  • Đổi JPA → MongoDB phải rewrite interface (vì Spring Data interface không tương thích)
  • Pagination là infrastructure concern (offset vs cursor) — không thuộc domain logic

Fix: Domain repo trả về List<Order> hoặc domain-specific pagination object. Pagination handle ở Application layer.

Anti-Pattern #3 — Use Case Gọi Use Case

// ❌ Tạo coupling giữa use cases
@Service
public class PlaceOrderUseCase {
    public void execute(PlaceOrderCommand cmd) {
        // ...
        // Gửi welcome email
        sendWelcomeEmailUseCase.execute(new WelcomeCommand(cmd.customerId()));
        // Log audit
        auditUseCase.execute(new AuditCommand(...));
    }
}
🎯 Vì sao sai?

Use case ban đầu chỉ "place order" — giờ kéo theo email + audit. Khi cần thêm SMS notification, inventory event, fraud check... use case này phình to. Tighter coupling = harder testing + harder change.

Fix: Dùng Domain Events. Use case chỉ publish event OrderPlacedEvent. Email service, audit service, inventory service đều subscribe event đó độc lập.

@EventListener(OrderPlacedEvent.class)
public void onOrderPlaced(OrderPlacedEvent event) {
    emailService.sendOrderConfirmation(event.customerId(), event.orderId());
}

@EventListener(OrderPlacedEvent.class)
public void onOrderPlacedForAudit(OrderPlacedEvent event) {
    auditService.logOrderEvent(event);
}

Anti-Pattern #4 — Một Bounded Context = Một Microservice (Dogma)

// ❌ Over-segmentation — nanoservices
order-service/          (200 LOC)
order-pricing-service/  (150 LOC)
order-validation-service/ (100 LOC)
order-audit-service/    (80 LOC)

Mỗi service cần:
- CI/CD pipeline riêng
- Load balancer
- Database riêng
- Monitoring
- On-call rotation
// Network overhead > business value
🎯 Vì sao sai?

Microservice mang operational cost rất cao. Nếu service quá nhỏ, cost vận hành lớn hơn benefit của decoupling. Mỗi cuộc gọi giữa services = network latency + retry logic + circuit breaker + tracing setup.

Fix: 1 microservice có thể chứa nhiều bounded context nếu cùng team own. Khi team tách → tách service. Đừng tách trước.

Anti-Pattern #5 — Over-Engineering DDD Cho CRUD Đơn Giản

Form đăng ký user 4 field (name, email, phone, address) → có người design:

  • UserAggregate với 5 value objects
  • 3 domain services để "validate"
  • Event sourcing cho audit trail
  • 400 dòng code cho 4 fields
🎯 Vì sao sai?

DDD designed cho complex domain. Đăng ký user là CRUD đơn giản — không có business rule phức tạp, không có invariant cross-entity. Áp DDD = onboarding nightmare (junior mở 10 file mới hiểu 1 form), tests chậm (boot container), debug khó.

Fix: Tactical DDD cho phần core domain (complex business). Simple CRUD pattern cho admin/configuration/lookup tables. Mix tự nhiên trong cùng codebase.

Anti-Pattern #6 — Aggregate Quá Lớn

// ❌ Customer aggregate chứa TẤT CẢ orders + reviews + addresses
public class Customer {
    private CustomerId id;
    private List<Order> orders;        // 500 orders trong lifetime
    private List<Review> reviews;      // 200 reviews
    private List<Address> addresses;   // 10 shipping addresses
    // Load customer = load 700+ entities vào memory!
}
🎯 Vì sao sai?
  • Memory: 1 request load 700 objects → OOM khi 1000 concurrent users
  • Transaction lock: save 1 review = lock toàn customer aggregate → contention cao
  • Slow loading: query 700 rows mỗi lần dù chỉ cần customer name

Fix: Order, Review là aggregate riêng. Customer aggregate chỉ chứa data cần cho business rules của chính nó (profile, status, loyalty tier). Reference cross-aggregate bằng ID, không embed:

public class Customer {
    private CustomerId id;
    private String name;
    private LoyaltyTier tier;
    // KHÔNG có List<Order> — Order là aggregate riêng
}

public class Order {
    private OrderId id;
    private CustomerId customerId;  // Reference bằng ID
    // ...
}
📏 Rule of thumb

Aggregate nên chứa < 1000 entities ở load. Hơn thì tách. Quy tắc: "có gì cần consistent ngay lập tức?" → trong 1 aggregate. "có gì có thể eventual consistent?" → aggregate riêng + domain event.

5. Logging Trong Clean Architecture — Đặt Ở Đâu?

❌ Log trong Domain
package com.shopvn.order.domain;
import org.slf4j.Logger;  // Domain depend SLF4J!

public class Order {
    private static final Logger log =
        LoggerFactory.getLogger(Order.class);

    public void cancel(String reason) {
        log.info("Cancelling order {}", id);
        // Domain không nên biết về logging
    }
}
✅ Log ở Application layer
package com.shopvn.order.application;

@Service
@Slf4j
public class CancelOrderUseCase {
    public void execute(CancelOrderCommand cmd) {
        log.info("Cancel order: id={}, reason={}",
            cmd.orderId(), cmd.reason());
        try {
            // ... business logic
        } catch (IllegalStateException e) {
            log.warn("Cancel failed: {}", e.getMessage());
            throw e;
        }
    }
}
🎯 Vì sao Domain không nên log?
  • Logging framework (SLF4J/Logback) là infrastructure concern. Domain depend infrastructure = phá dependency rule.
  • Test domain phải mock logger → noise trong test.
  • Domain logic phải "silent" — chỉ raise events. Application layer quyết định log/notify/audit.

Domain communicate qua Domain Events. Application layer subscribe + log:

@Component
@Slf4j
public class OrderEventLogger {
    @EventListener
    public void onOrderConfirmed(OrderConfirmedEvent event) {
        log.info("Order confirmed: orderId={}, customerId={}, total={}",
            event.orderId(), event.customerId(), event.total());
    }

    @EventListener
    public void onOrderCancelled(OrderCancelledEvent event) {
        log.warn("Order cancelled: orderId={}, reason={}",
            event.orderId(), event.reason());
    }
}

6. Migration Strategy — Từ Anemic Sang Rich Domain Model

Không ai bắt đầu codebase mới sạch. Realistic là bạn join team với 500K LOC monolith Anemic. Migration đúng cách:

🎯 Vì sao Strangler Fig hiệu quả?

Code mới sống "bên cạnh" code cũ. Mỗi feature mới hoặc bug fix → cơ hội migrate 1 aggregate. Sau 6-12 tháng, code cũ tự nhiên "chết" vì không còn ai dùng. Pattern này tránh được "chúng tôi không thể rewrite vì business cần feature" — vì bạn vẫn ship feature, song song refactor.

7. Interview Framework

Tầng 1 — SurfaceMid-level gate
Q: Phân biệt Entity và Value Object?
"Entity có identity persist (Order với orderId — kể cả status đổi vẫn là cùng 1 order). Value Object không có identity, định nghĩa bằng value (Money = amount + currency). Entity mutable, Value Object immutable. Compare: entity bằng ID (order1.id.equals(order2.id)), value object bằng content (money1.equals(money2))."
Q: Repository trong DDD là gì?
"Repository là interface trong domain layer, đại diện cho persistence abstraction. Caller dùng như đang lưu aggregate vào collection: orderRepo.save(order), orderRepo.findById(orderId). Implementation (JPA, MongoDB, in-memory) ở infrastructure — domain không biết DB cụ thể, không biết SQL/NoSQL. Lý do: domain testable không cần DB, swap DB không break business logic."
Tầng 2 — Deep DiveMid → Senior
Q: Design use case "place order". Yêu cầu: (a) reserve inventory + tạo order atomic, (b) emit audit trail, (c) dễ thêm payment method mới sau này.
"Aggregate: Order là root, chứa OrderLine entities. Order method place() validate invariants (lines không empty, total > 0), emit OrderConfirmedEvent.

Use Case: PlaceOrderUseCase orchestrate inventory.reserve() + order.place() + orderRepo.save(). @Transactional ở use case → atomic giữa DB save và inventory reserve (cùng DB) hoặc dùng Saga (nếu inventory là service riêng).

Audit: Event-driven — OrderConfirmedEvent published sau khi save. AuditService @EventListener subscribe và log/persist. Không pollute use case với audit logic.

Payment method extensibility: PaymentMethod là Strategy pattern. PaymentGateway interface ở domain (port), implementation StripeGateway/MomoGateway/VNPayGateway ở infrastructure (adapter). Thêm method mới = add 1 implementation, không sửa use case."
Q: Có OrderContext và ShippingContext, đều có "Order" với attributes khác nhau. Giải thích bounded context + anti-corruption layer.
"Mỗi context own definition riêng của Order. OrderContext care: customerId, items, totalPrice, paymentStatus. ShippingContext care: orderId, address, weight, carrier, trackingNumber.

Khi ShippingContext cần info từ OrderContext (ví dụ địa chỉ giao), dùng Anti-Corruption Layer (ACL) — 1 adapter trong ShippingContext gọi OrderContext API và translate DTO của Order Context thành ShippingOrder model.

Lý do: nếu ShippingContext import OrderContext models trực tiếp → coupling chặt → OrderContext refactor tên field, ShippingContext break. ACL absorbs change. Khi OrderContext đổi customerId thành buyerId, chỉ ACL phải update."
Tầng 3 — ArchitectureSenior signal
Q: Sàn ShopVN sáp nhập với 1 sàn khác (Marketplace). Design bounded contexts cho hệ thống mới.
"Identify contexts cốt lõi:
1. CatalogContext — product master data, mỗi sàn brings existing catalog → cần unification strategy (deduplicate, merge SKUs)
2. OrderContext — checkout, payment intent (mỗi sàn process độc lập trước, eventual merge)
3. InventoryContext — stock của seller, cần aggregate cross-sàn để tránh oversell
4. CustomerContext — buyer profile, KYC (unify, deduplicate by email/phone)
5. SellerContext — vendor onboarding, payout (mỗi sàn có sellers riêng → namespace để tránh ID collision)
6. ShippingContext — fulfillment (có thể share carriers chung)
7. FinanceContext — consolidated revenue, settlement (new context, đọc từ OrderContext + SellerContext)

Migration strategy:
Phase 1: coexist — mỗi sàn vẫn chạy độc lập, sync data qua events (CDC từ DB cũ → Kafka → consumer).
Phase 2: shared contexts (CustomerContext, CatalogContext) migrate trước vì có dedupe value.
Phase 3: gradual data migration sang unified DB.
Phase 4: retire legacy systems.

Trong mỗi context, áp Clean Architecture (domain isolated). Giữa contexts: domain events + ACL. Tránh share DB cross-context — chỉ share via API/event."
