Oracle Deep Dive — SGA/PGA, UNDO/SCN, Tuning

Nhiều dev "dùng Oracle" nhưng không biết SGA khác PGA, vì sao có ORA-01555, hay ROWNUM khác ROW_NUMBER(). File này giải thích kiến trúc Oracle (bộ nhớ, MVCC, locking) và quy trình tune query/đọc AWR — đủ để tune chứ không chỉ viết SQL chạy được. Domain: ShopFlow — một sàn e-commerce chạy Oracle cho hệ thống đơn hàng.

Hiểu memory/MVCC/locking Oracle, tune query, đọc AWR/ASH

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Oracle với PostgreSQL gần như nhau, biết SQL là đủ."
Thực tế: MVCC khác cơ bản: Oracle update in-place + đẩy giá trị cũ vào UNDO segment (→ ORA-01555 khi UNDO bị ghi đè), còn PostgreSQL giữ version cũ ngay trong table (→ cần VACUUM dọn). Oracle không có REPEATABLE READ; "SERIALIZABLE" của Oracle chỉ là Snapshot Isolation. Mang giả định PostgreSQL sang Oracle = bug âm thầm.
Tưởng hiểu: "ROWNUM dùng để phân trang, WHERE ROWNUM BETWEEN 11 AND 20 lấy trang 2."
Thực tế: ROWNUM được gán khi row được chọn, trước ORDER BY, và điều kiện ROWNUM > 1 luôn rỗng — vì row đầu tiên phải có ROWNUM = 1, nếu bị loại thì không có row nào đạt ROWNUM = 2. Phân trang đúng cần subquery + ROW_NUMBER() hoặc OFFSET FETCH (12c+).
Tưởng hiểu: "Query chậm thì thêm index là xong."
Thực tế: Phải đọc execution plan + AWR/ASH để biết nghẽn thật ở đâu (full scan, sai index, bind variable peeking, contention, log file sync). Thêm index bừa làm chậm INSERT/UPDATE và đôi khi khiến optimizer chọn plan tệ hơn.

1. Bài toán thực tế — query 8s giờ cao điểm CORE ★

ShopFlow chạy Oracle. Trang "đơn hàng của tôi" bình thường 200ms, giờ cao điểm vọt 8s. Dev thêm index nhưng không đỡ. Có lúc một job report ném ORA-01555: snapshot too old. AWR report cho thấy "buffer busy waits" và "log file sync" cao. Mỗi triệu chứng trỏ tới một phần kiến trúc Oracle mà nếu không hiểu thì chỉ đoán mò.

⏸ Pause & Predict — trước khi đọc lời giải
Viết ra giấy:
  1. ORA-01555 nghĩa là gì? Nó liên quan UNDO thế nào, và vì sao long-running report dễ gặp?
  2. "Thêm index không đỡ" — những nguyên nhân nào khác ngoài thiếu index khiến query chậm giờ peak?
  3. Để biết nghẽn thật, bạn đọc report nào của Oracle và nhìn chỉ số gì trước?
Đáp án: ORA-01555 = phiên bản cũ trong UNDO bị ghi đè trước khi report đọc xong snapshot (2.2); chậm có thể do full scan/sai plan/contention/bind peeking (3.4); đọc AWR/ASH nhìn top wait events + SQL by elapsed time (3.4). Đừng đọc tiếp khi chưa giải thích được ORA-01555.

2.1 SGA + PGA — hai vùng bộ nhớ CORE ★

Oracle chia bộ nhớ thành hai vùng, hiểu chúng là điều kiện để đọc đúng AWR:

  • SGA (System Global Area) — vùng chia sẻ giữa mọi session. Gồm: buffer cache (cache các data block đọc từ disk — đọc trúng cache thì nhanh, miss thì phải đọc disk), shared pool (cache câu SQL đã parse + execution plan + library cache), và redo log buffer.
  • PGA (Program Global Area) — vùng riêng của mỗi session. Gồm workarea cho sort và hash join. Quan trọng: nếu một thao tác sort/hash lớn vượt PGA cho phép, nó tràn xuống temp tablespace (đĩa) → chậm hẳn.
Vì sao quan trọng cho tuning

Một query chậm có thể do PGA nhỏ (sort phải ghi đĩa), hoặc do buffer cache miss nhiều (đọc disk thay vì RAM). Biết SGA/PGA giúp bạn đọc đúng chỉ số trong AWR ("buffer cache hit ratio", "PGA used vs target") thay vì chỉ nhìn "query chậm".

2.2 MVCC: UNDO + SCN CORE ★

Như mọi DB MVCC, Oracle cho "readers don't block writers". Nhưng cơ chế khác PostgreSQL ở chỗ then chốt. Oracle update in-place (ghi đè row ngay trong data block), và trước khi ghi đè, nó copy giá trị cũ vào UNDO segment. SCN (System Change Number) là một đồng hồ logic tăng dần đánh dấu mỗi thay đổi. Khi một query bắt đầu, nó ghi nhớ SCN hiện tại làm "điểm nhìn"; nếu trong lúc chạy nó gặp một row đã bị sửa sau SCN đó, Oracle dùng UNDO để dựng lại phiên bản cũ đúng với SCN của query — đảm bảo query thấy một ảnh nhất quán tại thời điểm nó bắt đầu (read consistency).

Vấn đề: UNDO có dung lượng hữu hạn và bị tái sử dụng (ghi đè) khi đầy. Nếu một query chạy quá lâu (báo cáo lớn) và phiên bản cũ mà nó cần đã bị ghi đè trong UNDO bởi các giao dịch khác → Oracle không dựng lại được phiên bản đó → ném ORA-01555: snapshot too old. Đây là lý do report dài dễ gặp lỗi này. Khắc phục: tăng UNDO_RETENTION và kích thước UNDO tablespace, hoặc rút ngắn query / tách reporting khỏi OLTP.

L1 — Biết tên

Biết "Oracle có MVCC, không lock đọc". Symptom hiểu giả: không giải thích được ORA-01555 hay vai trò UNDO/SCN.

L2 — Explain được

Read consistency: query thấy snapshot tại SCN bắt đầu, phiên bản cũ dựng từ UNDO. ORA-01555 = UNDO chứa phiên bản cũ bị ghi đè (retention ngắn / UNDO nhỏ / nhiều DML) trước khi query dài đọc xong.

L3 — Apply được

Tune UNDO_RETENTION + UNDO tablespace; tránh report giữ snapshot quá lâu; tách OLTP/reporting. Đọc đúng error để biết chỉnh UNDO chứ không phải index.

2.3 Row lock qua ITL SUPPLEMENTARY

Oracle khoá ở mức row, nhưng không có "lock manager" tập trung như nhiều DB khác — thông tin lock được ghi ngay vào ITL (Interested Transaction List) trong phần header của data block chứa row đó. Hệ quả: Oracle không "leo thang lock" (lock escalation) từ row lên table khi có nhiều row bị khoá, và "writers don't block readers, readers don't block writers". Lock contention (nhiều giao dịch tranh khoá cùng row) biểu hiện qua wait event enq: TX - row lock contention trong AWR.

2.4 Redo log & recovery SUPPLEMENTARY

Để đảm bảo durability (cam kết D trong ACID) mà không phải ghi data file ngay lập tức (chậm, ngẫu nhiên), Oracle dùng write-ahead logging: mọi thay đổi được ghi vào redo log (ghi tuần tự, nhanh) trước khi data file được cập nhật. Khi COMMIT, Oracle flush redo xuống đĩa — đó là điểm đảm bảo "đã commit thì không mất". Vì vậy nếu trong AWR thấy "log file sync" cao, nghĩa là nghẽn ở khâu ghi redo lúc commit — thường do đĩa redo chậm hoặc do commit quá nhiều giao dịch nhỏ (mỗi commit một lần flush) → cân nhắc gom batch commit.

3.1 MERGE — upsert chuẩn Oracle CORE ★

"Upsert" (có thì update, chưa có thì insert) nếu làm bằng "đọc rồi quyết định" sẽ dính race condition giữa lúc đọc và lúc ghi. MERGE làm cả hai trong một câu lệnh atomic:

MERGE INTO orders t
USING (SELECT :id AS id, :total AS total FROM dual) s
ON (t.id = s.id)
WHEN MATCHED THEN
  UPDATE SET t.total = s.total, t.updated_at = SYSTIMESTAMP
WHEN NOT MATCHED THEN
  INSERT (id, total, created_at) VALUES (s.id, s.total, SYSTIMESTAMP);

3.2 Pagination đúng — vì sao ROWNUM lừa CORE ★

ROWNUM là một pseudo-column được Oracle gán khi một row được chọn vào kết quả — và quan trọng: gán trước khi ORDER BY sắp xếp, và theo cách "đếm tăng dần các row được giữ lại". Điều này dẫn tới một bẫy: điều kiện WHERE ROWNUM > 10 (hay BETWEEN 11 AND 20) luôn trả rỗng. Lý do: ROWNUM được kiểm tra khi gán — row đầu tiên được xét phải nhận ROWNUM = 1, nhưng 1 > 10 là false nên nó bị loại; row tiếp theo lại được xét để gán ROWNUM = 1 (vì row trước bị loại, bộ đếm chưa tăng)... cứ thế không row nào vượt được. Phải materialize (bọc subquery, gán số thứ tự xong) rồi mới lọc:

-- ❌ SAI: luôn rỗng, và ROWNUM gán trước ORDER BY
SELECT * FROM orders WHERE ROWNUM BETWEEN 11 AND 20 ORDER BY created_at DESC;

-- ✅ ĐÚNG (mọi version): bọc subquery, gán số thứ tự SAU khi đã sort
SELECT * FROM (
  SELECT o.*, ROW_NUMBER() OVER (ORDER BY created_at DESC) rn FROM orders o
) WHERE rn BETWEEN 11 AND 20;

-- ✅ ĐÚNG (12c+): cú pháp chuẩn SQL
SELECT * FROM orders ORDER BY created_at DESC
  OFFSET 10 ROWS FETCH NEXT 10 ROWS ONLY;

3.3 Analytic functions — vũ khí senior SUPPLEMENTARY

Analytic (window) functions tính toán trên một "cửa sổ" các row liên quan mà không gộp chúng lại (khác GROUP BY) — giải nhiều bài toán mà nếu không có chúng phải self-join phức tạp:

-- top 3 đơn giá trị nhất MỖI khách (không cần self-join)
SELECT customer_id, order_id, total FROM (
  SELECT customer_id, order_id, total,
         ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY total DESC) rn
  FROM orders
) WHERE rn <= 3;

-- running total + so với đơn liền trước
SELECT order_id, total,
       SUM(total) OVER (ORDER BY created_at) AS running_total,
       LAG(total) OVER (ORDER BY created_at) AS prev_total
FROM orders;

3.4 Index, partition, AWR/ASH CORE ★

  • Index: B-tree (mặc định, hợp cột nhiều giá trị phân biệt — high cardinality); bitmap (hợp cột ít giá trị — low cardinality, đọc nhiều ghi ít — KHÔNG dùng cho OLTP vì bitmap lock cả nhóm row khi update); composite (nhiều cột, thứ tự cột phải khớp cách dùng trong WHERE).
  • Partition: chia một table lớn thành nhiều phần theo range (vd theo tháng), list, hoặc hash → query có điều kiện trên cột phân vùng chỉ quét đúng phần cần (partition pruning) thay vì cả bảng.
Workflow chẩn đoán query chậm (dùng AWR/ASH thay vì đoán):
1. AWR report khoảng thời gian peak → "Top 10 Foreground Wait Events"
     buffer busy waits / log file sync / db file sequential read ... → biết loại nghẽn
2. AWR "SQL ordered by Elapsed Time" → tìm câu SQL tốn nhất
3. ASH (Active Session History) → "lúc 14:00 các session đang chờ gì"
4. EXPLAIN PLAN / DBMS_XPLAN cho SQL nghi vấn:
     FULL TABLE SCAN? sai index? bind variable peeking chọn plan tệ?
5. Sửa theo BẰNG CHỨNG: index đúng, bind variable, partition, tách reporting — không cảm tính

4. Anti-patterns CORE ★

🔧 Counter-example — khi "tối ưu" làm chậm hơn / sai
  • Bitmap index cho OLTP: bitmap khoá cả một nhóm row khi update một row → contention nặng dưới tải ghi. Bitmap chỉ cho data warehouse (đọc nhiều, ghi ít/batch).
  • Thêm index bừa: mỗi index làm chậm INSERT/UPDATE (phải cập nhật index) và có thể khiến optimizer chọn plan sai. Index theo query thật, đo trước.
  • PL/SQL SELECT INTO không bắt NO_DATA_FOUND / TOO_MANY_ROWS: lỗi runtime bất ngờ khi không có hoặc có nhiều row. Luôn handle exception.
  • Hardcode literal thay bind variable: mỗi giá trị khác nhau tạo một câu SQL mới phải hard-parse → library cache (trong shared pool) bùng nổ, tốn CPU parse. Dùng bind variable để reuse plan.
  • SELECT * trên table lớn: kéo cột thừa, phá khả năng index-only scan, tốn I/O.

Quy tắc: mọi tối ưu dựa trên execution plan + AWR thực đo, không cảm tính. Và đừng mang giả định PostgreSQL sang Oracle nguyên xi — hai DB khác ở nhiều chỗ căn bản.

5. 3-tier interview framework

Tầng 1 — SurfaceMid gate

Q: SGA và PGA khác gì?

SGA chia sẻ mọi session (buffer cache, shared pool, redo buffer). PGA riêng mỗi session (sort, hash workarea). Sort vượt PGA → tràn temp tablespace (đĩa, chậm). Biết để đọc đúng chỉ số AWR.

Tầng 2 — Deep DiveSenior gate

Q: Giải thích ORA-01555 và cách khắc phục.

Query dài đọc theo SCN cũ, cần phiên bản cũ từ UNDO; nhưng UNDO bị ghi đè (retention ngắn / UNDO nhỏ / nhiều DML đồng thời) trước khi query đọc xong → "snapshot too old". Fix: tăng UNDO_RETENTION + UNDO tablespace, rút ngắn query, tách reporting khỏi OLTP.

Tầng 3 — ArchitectureSenior+

Q: Query đột ngột chậm giờ cao điểm, workflow chẩn đoán?

AWR top wait events + SQL by elapsed → ASH xem session lúc đó chờ gì → execution plan của SQL nghi vấn (full scan? sai index? bind peeking?) → kiểm tra contention (enq: TX, buffer busy) / log file sync. Sửa theo bằng chứng: index đúng, bind variable, tách reporting, partition. Không đoán.

🔧 Lab bắt buộc — đọc plan & reproduce (2 bài)

Lab 1 — Pagination sai. Viết SELECT ... WHERE ROWNUM > 10 → quan sát trả rỗng. Sửa bằng subquery ROW_NUMBER() hoặc OFFSET FETCH. Tiêu chí: giải thích ROWNUM gán khi chọn row (trước ORDER BY) nên row đầu phải là 1, không thể có "ROWNUM > 10".

Lab 2 — Plan thay đổi vì thiếu index. Bảng orders lớn; chạy EXPLAIN PLAN cho query lọc theo customer_id chưa index → FULL TABLE SCAN. Thêm index → INDEX RANGE SCAN, đo thời gian. Sau đó thêm điều kiện làm index vô hiệu (function trên cột, vd UPPER(email)) → quan sát plan quay lại full scan. Tiêu chí: đọc được plan và biết vì sao index bị bỏ qua.
