1. Bài toán thực: Bottleneck production CORE

Một e-commerce platform xử lý 10K order/giây giờ cao điểm. Tính năng notification: mỗi order successful gửi push notification. Team phát hiện 3 bottleneck — mỗi cái là 1 wrong-DS choice:

Bottleneck 1: SELECT user_id FROM orders WHERE amount > 1000000
              → Full table scan, 5 giây cho 50M rows
              → Thiếu B-tree index trên cột amount

Bottleneck 2: Duplicate notification check
              "User X đã nhận notification về order Y trong 5 phút qua chưa?"
              → Đang scan ArrayList of all recent notifications → O(n)
              → Cần HashSet → O(1), hoặc Bloom Filter cho large scale

Bottleneck 3: Sliding window expiry cho dedup
              "Lưu notification_ids của 5 phút gần nhất"
              → Đang lưu HashSet nhưng KHÔNG có cleanup → memory leak
              → Cần sliding window structure (Deque + timestamp)

3. HashMap internals — Câu hỏi #1 phỏng vấn Java HIGH-SIGNAL

L2 check sau section: Explain được "Worst case HashMap khi nào? Java 8 fix thế nào?" trong < 30 giây không vướng.

Cấu trúc bên trong

HashMap<K,V> trong Java 8+:

internal: Node<K,V>[] table        ← array of buckets
default capacity: 16
load factor: 0.75 (resize khi size > capacity × 0.75)

Mỗi bucket có thể là:
- null (empty)
- single Node (no collision)
- LinkedList of Nodes (collision, chain length ≤ 8)
- Red-Black TreeNode (chain length > 8 và capacity ≥ 64)

Put operation — step by step

// Pseudo code (gần với JDK HashMap.put):
V put(K key, V value) {
    // Step 1: Compute hash
    int hash = key.hashCode();
    hash = hash ^ (hash >>> 16);  // Spreading high bits to low

    // Step 2: Find bucket index
    int index = hash & (table.length - 1);  // Bitwise AND, faster than %

    // Step 3: Check bucket
    Node node = table[index];
    if (node == null) {
        table[index] = new Node(hash, key, value);
        return null;
    }

    // Step 4: Walk chain hoặc tree, check key.equals()
    while (node != null) {
        if (node.hash == hash && node.key.equals(key)) {
            V old = node.value;
            node.value = value;  // Update existing
            return old;
        }
        node = node.next;
    }

    // Step 5: Append new node
    addToBucket(table, index, new Node(hash, key, value));

    // Step 6: Resize if needed
    if (++size > threshold) resize();
}

Collision & Worst case

Java 7 và trước (worst case O(n))

Khi nhiều key hash về cùng bucket:
- Chain LinkedList dài
- get(key) phải traverse chain: O(n)

Hash collision attack:
- Attacker craft keys cùng hash
- Server lookup → O(n) → DoS
- 2011: Java/Tomcat dính lỗi này → CVE-2011-4858

Java 8+ (fix với tree)

Khi chain > 8 nodes && capacity ≥ 64:
- Convert LinkedList → Red-Black Tree
- get(key) trên tree: O(log n)

Worst case O(n) → O(log n)
Plus: randomized hash seed (Java 7+)
→ Attacker không predict được collision

📌 Đáp án Pause & Predict — Vì sao DB index dùng B-tree?

3 lý do (không phải 1):
  1. Sorted order. B-tree iterate sorted → range query WHERE date > '2026-01-01' hiệu quả. HashMap không sorted.
  2. Disk-friendly layout. B-tree fan-out cao (mỗi node có 100-500 children) → cây "lùn", ít disk I/O. HashMap random distribution → mỗi lookup 1 disk seek không predictable.
  3. Persistence stability. B-tree on-disk format stable, không cần rehash khi resize. HashMap resize = move toàn bộ entries → expensive cho disk-based.
Nếu bạn chỉ viết được 1-2 lý do (chỉ "sorted") → vẫn L1. 3 lý do = L2.

Quy tắc thiết kế key cho HashMap

  1. Override equals + hashCode theo contract. Object.equals() compare reference — không đủ cho custom key.
  2. Key phải immutable (hoặc không modify sau put). Mutate key → hash thay đổi → lookup fail.
  3. hashCode phải distribute tốt. Đừng return constant (vd return 1;) → toàn bộ vào 1 bucket → O(n).
  4. String, Integer, UUID đã immutable + hashCode tốt — dùng làm key an toàn.

❌ Pitfall: Custom key không override equals/hashCode

class UserId {
    String value;
    UserId(String v) { value = v; }
    // ❌ Không override equals/hashCode
}

Map<UserId, User> map = new HashMap<>();
map.put(new UserId("u123"), user);

UserId lookup = new UserId("u123");
map.get(lookup);  // null! Khác instance, default equals = reference equality

// ✅ Fix:
class UserId {
    final String value;
    UserId(String v) { value = v; }
    @Override public boolean equals(Object o) {
        return o instanceof UserId && ((UserId)o).value.equals(value);
    }
    @Override public int hashCode() { return value.hashCode(); }
}

4. Tree traversal & recursive thinking CORE

4 cách traverse tree

Binary tree:
       1
      / \
     2   3
    / \ / \
   4  5 6  7

DFS (Depth-First — recursive natural):
  Preorder  (Root → Left → Right):  1, 2, 4, 5, 3, 6, 7
    → Copy tree, serialize structure (phải biết root trước)
  Inorder   (Left → Root → Right):  4, 2, 5, 1, 6, 3, 7
    → Sorted order cho BST!
  Postorder (Left → Right → Root):  4, 5, 2, 6, 7, 3, 1
    → Delete tree (xoá con trước), evaluate expression tree

BFS (Breadth-First — level by level):  1, 2, 3, 4, 5, 6, 7
  → Shortest path unweighted, level-wise processing

Recursive thinking — "leap of faith"

Đa số dev sợ recursion vì cố "trace stack". Sai — sẽ rối ngay với tree > 5 node. Đúng: tin recursive call đã đúng cho subtree, chỉ cần đảm bảo base case + combine.

// Height của tree
int height(TreeNode node) {
    if (node == null) return 0;            // base case
    // Tin height(left) đã đúng. Tin height(right) đã đúng.
    return 1 + Math.max(height(node.left), height(node.right));
}

// Validate BST
boolean isValidBST(TreeNode root) {
    return validate(root, Long.MIN_VALUE, Long.MAX_VALUE);
}
boolean validate(TreeNode node, long min, long max) {
    if (node == null) return true;
    if (node.val <= min || node.val >= max) return false;
    return validate(node.left, min, node.val)
        && validate(node.right, node.val, max);
}

// Lowest Common Ancestor
TreeNode lca(TreeNode root, TreeNode p, TreeNode q) {
    if (root == null || root == p || root == q) return root;
    TreeNode left = lca(root.left, p, q);
    TreeNode right = lca(root.right, p, q);
    if (left != null && right != null) return root;  // p và q ở 2 phía
    return left != null ? left : right;
}

BFS — iterative với Queue

List<List<Integer>> levelOrder(TreeNode root) {
    List<List<Integer>> result = new ArrayList<>();
    if (root == null) return result;

    Queue<TreeNode> queue = new ArrayDeque<>();   // ArrayDeque tốt hơn LinkedList
    queue.offer(root);

    while (!queue.isEmpty()) {
        int levelSize = queue.size();   // Chốt size để tách level
        List<Integer> level = new ArrayList<>();

        for (int i = 0; i < levelSize; i++) {
            TreeNode node = queue.poll();
            level.add(node.val);
            if (node.left != null) queue.offer(node.left);
            if (node.right != null) queue.offer(node.right);
        }
        result.add(level);
    }
    return result;
}
Khi nào DFS vs BFS:
DFS khi: explore all paths (backtracking, permutation), detect cycle, tree problem natural recursive.
BFS khi: shortest path unweighted, level-by-level processing, find anything trong "ít step" nhất.
Memory: DFS stack O(height). BFS queue O(width — max level size). Wide tree → DFS efficient hơn. Deep tree → BFS efficient hơn.

5. Sliding Window — Pattern cho substring/subarray CORE

2 loại window

LoạiĐặc điểmUse case
Fixed sizeWindow luôn k elements, slide 1 elementMax sum subarray size k, moving average
Variable sizeExpand khi OK, shrink khi violateLongest substring with property, rate limiter

Template variable size

int slidingWindow(int[] arr) {
    int left = 0, result = 0;

    for (int right = 0; right < arr.length; right++) {
        // 1. Expand: add arr[right] vào window
        addToWindow(arr[right]);

        // 2. Shrink: nếu violate constraint, di chuyển left
        while (violatesConstraint()) {
            removeFromWindow(arr[left]);
            left++;
        }

        // 3. Update result với window hợp lệ hiện tại
        result = Math.max(result, right - left + 1);
    }
    return result;
}

Why O(n) không O(n²)? Mặc dù có nested loop (for right + while left), mỗi element vào window đúng 1 lần và ra đúng 1 lần. Total operations = 2n → O(n).

Ví dụ: longest substring with K distinct chars

int longestKDistinct(String s, int k) {
    if (k == 0) return 0;
    Map<Character, Integer> count = new HashMap<>();
    int left = 0, maxLen = 0;

    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        count.merge(c, 1, Integer::sum);

        while (count.size() > k) {
            char leftChar = s.charAt(left);
            count.merge(leftChar, -1, Integer::sum);
            if (count.get(leftChar) == 0) count.remove(leftChar);
            left++;
        }

        maxLen = Math.max(maxLen, right - left + 1);
    }
    return maxLen;
}

🔨 Thử Phá Vỡ: Sliding window có hạn chế gì?

  • Chỉ work với contiguous (kề nhau). Subarray OK, subsequence (không cần kề) không dùng được.
  • Cần state có thể incremental update. Sum, count, max trong window OK. Nhưng median trong window phức tạp (cần 2 heap).
  • Negative values phá monotonicity. Kadane (max subarray) hoạt động với negative — sliding window không.

Big-O cheat sheet

Space complexity — hay bị quên

  • Recursive stack: O(depth). BST balanced O(log n). BST skewed O(n) — risk stack overflow.
  • Auxiliary structures: HashMap dedup O(n), queue BFS O(width).
  • Output space đôi khi không tính (nếu requirement bắt buộc return tất cả).

Space-time trade-off

Senior interviewer thường hỏi: "Dùng thêm O(n) space có giảm time từ O(n²) xuống O(n) không?"

Ví dụ kinh điển: Two Sum
- Brute force: O(n²) time, O(1) space — nested loop
- HashMap: O(n) time, O(n) space — trade space for time

Quy tắc thumb: Hash-based dedup hầu như luôn trade O(n) space cho O(n) time.
Big-O không phải tất cả: O(n) với constant lớn có thể chậm hơn O(n log n) với constant nhỏ. Cache locality, branch prediction, SIMD — đều ảnh hưởng. Big-O cho asymptotic; benchmark cho production.

Map<String, Integer> countByCategory(List<Product> products) {
    Map<String, Integer> freq = new HashMap<>();

        // getOrDefault: 1 lookup thay vì containsKey + get (2 lookups)

② Grouping

Map<String, List<Order>> groupByCustomer(List<Order> orders) {
    Map<String, List<Order>> groups = new HashMap<>();
    for (Order o : orders) {
        // computeIfAbsent: atomic, không NPE, tạo list khi cần
        groups.computeIfAbsent(o.getCustomerId(), k -> new ArrayList<>())
              .add(o);
    }
    return groups;
}

// Stream version (nếu prefer):
Map<String, List<Order>> groups = orders.stream()
    .collect(Collectors.groupingBy(Order::getCustomerId));

③ Deduplication

List<Order> dedup(List<Order> orders) {
    Set<String> seen = new HashSet<>();
    return orders.stream()
        .filter(o -> seen.add(o.getOrderId()))  // add returns false nếu đã có
        .collect(Collectors.toList());
}

④ LRU Cache (LinkedHashMap built-in)

public class LRUCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    public LRUCache(int capacity) {
        // accessOrder = true → LinkedHashMap maintain order theo access (LRU)
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;  // Auto-evict eldest khi vượt capacity
    }
}

// Usage:
LRUCache<String, User> cache = new LRUCache<>(1000);
cache.put("u1", user);  // Auto LRU behavior

// hoặc dùng Caffeine (production library).

① BFS level order (organization hierarchy print)

// Print org chart theo level
List<List<String>> printOrgChart(Employee ceo) {
    List<List<String>> levels = new ArrayList<>();
    if (ceo == null) return levels;

    Queue<Employee> q = new ArrayDeque<>();
    q.offer(ceo);

    while (!q.isEmpty()) {
        int n = q.size();
        List<String> level = new ArrayList<>();
        for (int i = 0; i < n; i++) {
            Employee e = q.poll();
            level.add(e.getName());
            for (Employee report : e.getDirectReports()) {
                q.offer(report);
            }
        }
        levels.add(level);
    }
    return levels;
}

③ LCA (find common manager of 2 employees)

// Lowest Common Ancestor — common manager
Employee commonManager(Employee root, Employee a, Employee b) {
    if (root == null || root == a || root == b) return root;

    Employee found = null;
    int hits = 0;
    for (Employee report : root.getDirectReports()) {
        Employee result = commonManager(report, a, b);
        if (result != null) {
            hits++;
            found = result;
        }
    }
    if (hits >= 2) return root;        // Cả a và b trong 2 subtree khác → root là LCA
    return found;                       // Chỉ 1 phía → tiếp tục bubble up
}

9. Sliding window rate limiter — Production code HIGH-SIGNAL

Câu hỏi interview senior phổ biến. Code phải production-ready (thread-safe, memory-bounded):

public class SlidingWindowRateLimiter {
    private final int maxRequests;
    private final long windowMs;
    // Per-user Deque of timestamps
    private final Map<String, Deque<Long>> perUser = new ConcurrentHashMap<>();

    public SlidingWindowRateLimiter(int maxRequests, long windowMs) {
        this.maxRequests = maxRequests;
        this.windowMs = windowMs;
    }

    public boolean tryAcquire(String userId) {
        long now = System.currentTimeMillis();
        long windowStart = now - windowMs;

        Deque<Long> timestamps = perUser.computeIfAbsent(
            userId, k -> new ArrayDeque<>());

        // Synchronize per-user deque (không lock global)
        synchronized (timestamps) {
            // Evict timestamps ngoài window (slide window)
            while (!timestamps.isEmpty() && timestamps.peekFirst() < windowStart) {
                timestamps.pollFirst();
            }

            if (timestamps.size() < maxRequests) {
                timestamps.addLast(now);
                return true;
            }
            return false;
        }
    }

    // ⚠️ Quan trọng: phải có cleanup, nếu không memory leak với inactive users
    @Scheduled(fixedRate = 60_000)
    public void cleanup() {
        long cutoff = System.currentTimeMillis() - windowMs * 2;
        perUser.entrySet().removeIf(entry -> {
            synchronized (entry.getValue()) {
                return entry.getValue().isEmpty()
                    || entry.getValue().peekLast() < cutoff;
            }
        });
    }
}

10. Two Pointer + Binary Search CORE

Two Pointer pattern

// Two Sum trên sorted array — O(n) time, O(1) space
int[] twoSumSorted(int[] arr, int target) {
    int left = 0, right = arr.length - 1;
    while (left < right) {
        int sum = arr[left] + arr[right];
        if (sum == target) return new int[]{left, right};
        else if (sum < target) left++;
        else right--;
    }
    return new int[]{};
}

// Two Sum trên unsorted — O(n) time, O(n) space (HashMap)
int[] twoSum(int[] arr, int target) {
    Map<Integer, Integer> seen = new HashMap<>();
    for (int i = 0; i < arr.length; i++) {
        int complement = target - arr[i];
        if (seen.containsKey(complement)) {
            return new int[]{seen.get(complement), i};
        }
        seen.put(arr[i], i);
    }
    return new int[]{};
}

Binary Search — Template KHÔNG sai off-by-one

// Standard binary search
int binarySearch(int[] arr, int target) {
    int left = 0, right = arr.length - 1;
    while (left <= right) {                          // <= với inclusive right
        int mid = left + (right - left) / 2;          // Tránh overflow
        if (arr[mid] == target) return mid;
        else if (arr[mid] < target) left = mid + 1;
        else right = mid - 1;
    }
    return -1;
}

// Find FIRST occurrence (lower bound)
int findFirst(int[] arr, int target) {
    int left = 0, right = arr.length - 1, result = -1;
    while (left <= right) {
        int mid = left + (right - left) / 2;
        if (arr[mid] == target) {
            result = mid;
            right = mid - 1;       // Tiếp tục search LEFT
        } else if (arr[mid] < target) left = mid + 1;
        else right = mid - 1;
    }
    return result;
}

// Find LAST occurrence (upper bound - 1)
int findLast(int[] arr, int target) {
    int left = 0, right = arr.length - 1, result = -1;
    while (left <= right) {
        int mid = left + (right - left) / 2;
        if (arr[mid] == target) {
            result = mid;
            left = mid + 1;        // Tiếp tục search RIGHT
        } else if (arr[mid] < target) left = mid + 1;
        else right = mid - 1;
    }
    return result;
}
Lý do dùng left + (right - left) / 2: Với (left + right) / 2, nếu left và right đều gần Integer.MAX_VALUE → tổng overflow → mid âm → bug. left + (right - left) / 2 luôn safe.

10b. Symptom → DS Choice Recipes CORE ★

Khi production có vấn đề, dấu hiệu nào báo "wrong DS choice"? Đây là tập recipe trace ngược từ symptom về quyết định DS.

🚨 Recipe #1 — Lookup nhanh khi cold, chậm sau vài giờ

Symptom: Service vừa start, hot map lookup 1ms. Sau 4-6h chạy, cùng API endpoint p99 tăng dần lên 50ms. JVM heap OK, CPU OK. Restart fix.

3 nguyên nhân thường gặp:

  1. HashMap không có max size, grow vô tận — code dùng Map<String, X> làm cache không bound. Rehash khi load factor cao tốn O(n). Fix: Caffeine.newBuilder().maximumSize(10_000).build() với LRU
  2. HashMap có collision cao do bad hashCode — key custom object, hashCode trùng nhau → bucket dài → O(n) lookup. Java 8+ chuyển sang Tree khi bucket > 8, nhưng vẫn O(log n). Tool: jmap -histo xem distribution
  3. String key có suffix random — hash distribute không đều. Fix: pre-hash key, dùng Long id

Debug: JFR/async-profiler flame graph sẽ show HashMap.getNode chiếm CPU bất thường.

🚨 Recipe #2 — "Tìm gần nhất" / "Top K" chạy chậm khi data tăng

Symptom: Endpoint "5 đơn hàng mới nhất của user" lúc data nhỏ chạy 10ms. Khi user có 10K đơn, latency lên 800ms. Code: list.stream().sorted().limit(5).

Root cause: Wrong DS choice — sort toàn bộ list (O(n log n)) để lấy top 5. Should use PriorityQueue (heap) → O(n log k) với k=5 nhỏ.

Fix pattern:

// ❌ Sort all để lấy top 5
List<Order> top5Bad = orders.stream()
    .sorted(Comparator.comparing(Order::getCreatedAt).reversed())
    .limit(5)
    .collect(Collectors.toList());

// ✅ Heap chỉ giữ 5 max
PriorityQueue<Order> heap = new PriorityQueue<>(
    Comparator.comparing(Order::getCreatedAt));
for (Order o : orders) {
    heap.offer(o);
    if (heap.size() > 5) heap.poll();
}

Tổng quát: Khi thấy "sort + limit K" với K nhỏ, suspect heap. Với data > 1M element, khác biệt 100ms vs 5s.

🚨 Recipe #3 — Memory tăng dần không OOM nhưng GC pause tăng

Symptom: Heap baseline 2GB, sau 8h chạy lên 5GB. GC pause từ 50ms → 800ms. jstat -gc show old gen growing.

3 nguyên nhân DS-related:

  1. LinkedList thay vì ArrayList khi không cần insert giữa — LinkedList overhead 24 byte per element (next/prev/value pointer). 1M Long element = 32MB list vs 12MB ArrayList
  2. HashMap với value lớn không cleanup — session store, retry tracker... key expire nhưng không remove. Fix: WeakHashMap hoặc explicit cleanup
  3. String concat trong loop — String s = ""; for(...) s += part; tạo O(n²) String garbage. Fix: StringBuilder

Tool:

jcmd <pid> GC.class_histogram | head -30

# Tìm map lớn nhất
jcmd <pid> GC.heap_info

⚠️ 5 anti-patterns CORE

❌ #1: Custom key không override equals/hashCode

Symptom: Put rồi get → null. Map "lost" entry.

Fix: Luôn override cả 2 cho custom key. Hoặc dùng Java record (Java 14+) — auto-generated.

❌ #2: Mutate key sau put

List<String> key = new ArrayList<>(List.of("a"));
map.put(key, "value");
key.add("b");           // ❌ Hash đã thay đổi
map.get(key);           // null — Map can't find

Fix: Dùng immutable keys (String, Integer, UUID, record).

❌ #3: ArrayList.contains() trong loop = O(n²)

List<String> seen = new ArrayList<>();
for (String item : items) {
    if (!seen.contains(item)) {      // ❌ O(n) mỗi check
        seen.add(item);
        process(item);
    }
}
// Total: O(n²) — disaster với n = 100K

Fix: Dùng HashSet — contains() O(1).

❌ #4: Code ngay không clarify

Symptom: Interview, đọc đề → code ngay. Bỏ sót edge case: empty input, null, duplicate, negative, overflow.

Fix: 2 phút clarify trước khi code. Hỏi: input size? sorted? duplicates? return what?

❌ #5: Không nói complexity

Symptom: Implement xong, interviewer hỏi "complexity?" — bí. Hoặc đoán mò.

Fix: Mỗi solution implement xong, nói thành tiếng "Time O(...), space O(...)". Reflex.

🎯 3-tier interview framework

Tier 1 — Surface (Junior)

Q1: HashMap vs TreeMap khác gì?

Trả lời: "HashMap dùng hash table, O(1) average put/get, không ordered. TreeMap dùng Red-Black Tree, O(log n) nhưng key sorted. HashMap khi cần fast lookup, TreeMap khi cần range queries hoặc sorted iteration."

Q2: DFS vs BFS — chọn khi nào?

Trả lời: "BFS cho shortest path unweighted, level-by-level. DFS cho explore all paths, backtracking, cycle detection. BFS dùng Queue, DFS dùng Stack (hoặc recursion). Memory: DFS O(depth), BFS O(width)."

Tier 2 — Deep Dive (Mid vs Senior)

Q3: HashMap worst case O(n) — khi nào? Java 8 fix?

"Worst case khi nhiều key hash về cùng bucket — hash collision. Trước Java 8, bucket = LinkedList → get O(n).

Java 8 fix: khi chain > 8 nodes && capacity ≥ 64 → convert LinkedList sang Red-Black Tree. Worst case O(n) → O(log n).

Plus Java 7+ có randomized hash seed → attacker không predict được collision → prevent DoS attack (CVE-2011-4858 cho Tomcat).

Trade-off: TreeNode ~2× memory của LinkedList Node. Conversion threshold 8 chọn vì xác suất chain dài đến 8 trong uniform hash distribution < 10^-7 — chỉ trigger với bad hashCode hoặc attack."

Q4: Implement rate limiter 100 req/phút. Trade-off?

"3 approach:
① Fixed window: Counter reset mỗi phút. Đơn giản, memory O(1)/user. Edge case: 100 req ở giây 59 + 100 ở giây 61 = 200 req trong 2s.
② Sliding window log: Lưu timestamps, đếm trong [now-60s, now]. Chính xác tuyệt đối. Cons: memory O(n requests)/user — vấn đề với 1M users.
③ Token bucket: Bucket có capacity, refill rate. Burst tolerant, memory O(1)/user. Smooth rate limiting. Phổ biến nhất production (Kong, Nginx).

Tôi chọn token bucket cho API public. Sliding window log nếu yêu cầu strict (vd financial limit). Implementation Redis với Lua script cho atomic distributed."

Tier 3 — Architecture (Senior)

Q5: Design search system tìm order history của user, scale 100M orders.

Phút 1 — Clarify:
  - Exact match (orderId) hay full-text search (description)?
  - Date range query?
  - Pagination requirement?
  - Real-time hay near-real-time?

Phút 2 — Data structures:
  - B-tree index (user_id, created_at) cho list user orders sorted by date
  - Hash index trên orderId cho exact lookup
  - Inverted index (Elasticsearch) cho full-text trên description

Phút 3 — Pagination:
  - Offset pagination: WHERE ... LIMIT 20 OFFSET N
    O(N) scan + skip → slow với large N
  - Cursor (keyset): WHERE id > last_seen ORDER BY id LIMIT 20
    O(log n) always → preferred for large datasets

Phút 4 — Caching:
  - Top users (Pareto): cache trong Redis Sorted Set
    ZADD orders:userId score=timestamp member=orderId
    ZREVRANGEBYSCORE → O(log n + k) range query
  - Cache TTL 5-15 phút, invalidate on new order

Phút 5 — Trade-offs:
  + B-tree composite index optimize most common pattern
  + Cursor pagination scales linearly
  + Redis ZSET cho hot users only (memory limit)
  - Schema change: thêm field cần migrate index
  - Multi-region: read replica + eventual consistency OK

🧠 Feynman test

🎓 Giải thích cho dev chưa biết — < 30s mỗi câu

  1. "HashMap worst case khi nào? Java 8 fix thế nào?" — Khi nhiều key hash về cùng bucket (collision). Trước Java 8 bucket = LinkedList O(n). Java 8 convert sang Red-Black Tree khi chain > 8 + capacity ≥ 64 → O(log n). Plus randomized hash seed prevent attack.
  2. "Vì sao DB index dùng B-tree không HashMap?" — 3 lý do: sorted (range query), disk-friendly (cây lùn, ít I/O), persistence stable (không cần rehash khi resize).
  3. "Vì sao sliding window O(n) không O(n²) dù có nested loop?" — Mỗi element vào window đúng 1 lần và ra đúng 1 lần. Total = 2n operations → O(n).
  4. "Khi nào TreeMap thay HashMap?" — Khi cần sorted iteration, range queries (headMap, tailMap, subMap), hoặc tìm closest key. Sacrifice O(log n) vs O(1) for ordering.
  5. "Space-time trade-off — ví dụ?" — Two Sum brute force O(n²) time O(1) space. HashMap version O(n) time O(n) space. Trade O(n) space để giảm time order of magnitude.
