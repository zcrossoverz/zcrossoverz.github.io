Clean Code & SOLID — Smell, Refactor, và Khi Nào Đừng

SOLID là thứ ai cũng đọc thuộc 5 chữ cái nhưng ít người dùng đúng — và over-apply SOLID còn hại hơn không biết nó. File này giải thích mỗi nguyên tắc qua vấn đề nó sinh ra để giải (không chỉ định nghĩa), kèm hậu quả cụ thể khi vi phạm và khi nào áp dụng là over-engineering — đọc một lần là biết nhìn code chỉ ra lỗi. Domain: ShopFlow — OrderPricingService tính giá/khuyến mãi của sàn e-commerce.

🎯 Mục tiêu
Nhận diện smell, refactor có kiểm soát, biết khi nào KHÔNG abstract

📚 Prereq

🟠 Cao — code review & design hỏi mỗi vòng senior

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Thuộc 5 chữ S-O-L-I-D và định nghĩa mỗi cái là hiểu SOLID."
Thực tế: Hiểu thật = nhìn một đoạn code chỉ ra nó vi phạm nguyên tắc nào và hậu quả cụ thể khi thêm feature. Đọc định nghĩa mà không nói được "vi phạm OCP thì sao" = hiểu giả. SOLID là công cụ chẩn đoán, không phải khẩu hiệu.
Tưởng hiểu: "Càng nhiều interface, abstraction, design pattern thì code càng SOLID, càng tốt."
Thực tế: Abstraction sớm/thừa là một code smell ngược: một class chỉ có một implementation mà bọc interface + factory + strategy = phải đọc 5 file để hiểu một việc. SOLID phục vụ khả năng thay đổi; chỗ không có trục thay đổi thì abstraction chỉ thêm nhiễu.
Tưởng hiểu: "Refactor là dọn code cho đẹp, làm lúc nào cũng được."
Thực tế: Theo định nghĩa, refactor là đổi cấu trúc mà KHÔNG đổi hành vi — nên bắt buộc có test phủ trước để chứng minh hành vi không đổi. Refactor không test = viết lại + cầu may, đó là "rewrite" chứ không phải refactor.

🗺️ Roadmap — nếu chỉ có X giờ thì học gì

Thứ tự ưu tiên
1 giờ (CORE ★): SRP + OCP (2.2, 2.3) — hai nguyên tắc dùng nhiều nhất; nhận diện 7 code smell (2.1). Đủ để pass phần "tìm lỗi SOLID" của interview.
+1 giờ (CORE ★): Refactor messy → clean theo bước có test (3), Tell-Don't-Ask (2.9), khi nào KHÔNG áp dụng SOLID (4).

1. Bài toán thực tế — PR tính giá 200 dòng CORE ★

Bạn review một PR: method calculateOrderTotal() dài 200 dòng, nhận 7 tham số, có magic number 0.05, 0.1 rải khắp, và trộn bốn việc khác nhau vào một chỗ — validate input, tính phí/khuyến mãi, ghi DB, gửi email. Tác giả nói "code chạy đúng mà". Vấn đề lộ ra khi bạn hỏi: "tháng sau thêm khuyến mãi Black Friday thì sửa ở đâu?" — câu trả lời là "sửa đúng cái method này", và mỗi lần sửa lại có rủi ro làm vỡ các loại khuyến mãi cũ vì tất cả nằm chung một chuỗi if-else khổng lồ.

⏸ Pause & Predict — trước khi đọc lời giải
Nhìn mô tả PR trên, viết ra giấy:
  1. Liệt kê đúng tên ít nhất 4 code smell trong đoạn này (không nói chung "code xấu").
  2. "Thêm 1 loại khuyến mãi phải sửa method này" — đây vi phạm nguyên tắc SOLID nào? Hậu quả khi có 10 loại khuyến mãi?
  3. Nếu refactor, bước ĐẦU TIÊN phải làm là gì trước khi đụng vào code (gợi ý: không phải tách method)?
Đáp án: Long Method, Long Parameter List, Magic Number, Mixed Concerns (2.1); vi phạm OCP (sửa thay vì mở rộng) + SRP (2.2, 2.3); bước đầu là viết test phủ hành vi hiện tại (4). Đừng đọc tiếp khi chưa đặt tên được 4 smell.

4 câu thinking

  1. Code smell nào đang có và mỗi cái gây hại gì khi maintain?
  2. Vi phạm nguyên tắc SOLID nào? Thêm feature thì vỡ ở đâu?
  3. Refactor theo bước nào để KHÔNG đổi hành vi?
  4. Khi nào dừng — abstraction tới đâu là đủ, đâu là over-engineering?

2.1 7 code smell — vì sao mỗi cái hại CORE ★

"Code smell" là dấu hiệu bề mặt gợi ý có vấn đề thiết kế sâu hơn — không phải bug (code vẫn chạy) nhưng làm code khó hiểu, khó sửa, dễ sinh bug khi thay đổi. Biết gọi tên smell giúp bạn nói chính xác trong code review thay vì "tôi thấy nó xấu".

SmellDấu hiệuVì sao hại khi maintain
Long MethodMethod 100+ dòng, nhiều cấp if lồngKhông giữ nổi toàn bộ trong đầu, không test được từng nhánh riêng
Long Parameter List7+ tham sốDễ truyền nhầm thứ tự (nhất là cùng kiểu), khó mở rộng — thêm 1 tham số đụng mọi caller
Magic Number* 0.05 không tênĐọc không biết 0.05 là gì; đổi thuế VAT phải tìm khắp codebase, dễ sót

2.2 S — Single Responsibility CORE ★

Định nghĩa: một class chỉ nên có một lý do để thay đổi. Cách diễn đạt chính xác hơn (Robert Martin): một class chỉ nên chịu trách nhiệm trước một nhóm người dùng/lý do nghiệp vụ. Vấn đề nó giải: khi một class gánh nhiều trách nhiệm, các trách nhiệm đó thay đổi vì những lý do khác nhau và vướng vào nhau. Ví dụ OrderPricingService vừa tính giá vừa gửi email có hai "lý do đổi": quy tắc giá (phòng Finance yêu cầu) và mẫu email (phòng Marketing yêu cầu). Khi Marketing sửa template, họ phải đụng vào file chứa logic tính tiền — rủi ro vô tình làm vỡ tính toán, và việc test/deploy hai mối quan tâm độc lập bị buộc chung. Tách ra: pricing đổi vì rule giá, notifier đổi vì kênh gửi — mỗi class một trục thay đổi.

❌ Fail — 2 lý do đổi trong 1 class

class OrderPricingService {
  BigDecimal price(Order o) { /* tính giá — Finance */ }
  void emailReceipt(Order o) { /* template — Marketing */ }
  void saveHistory(Order o) { /* ghi DB — Data */ }
}

✅ Success — mỗi class 1 trách nhiệm

class OrderPricingService { BigDecimal price(Order o){...} }
class ReceiptNotifier     { void email(Order o){...} }
class OrderHistoryStore   { void save(Order o){...} }

2.3 O — Open/Closed CORE ★

Định nghĩa: một module nên mở để mở rộng nhưng đóng để sửa đổi — nghĩa là thêm hành vi mới bằng cách thêm code, không phải sửa code đang chạy. Vấn đề nó giải: mỗi lần sửa code đang chạy là một lần có rủi ro làm vỡ thứ đang hoạt động. Một chuỗi if (type==PERCENT)...else if (type==FIXED)... để tính khuyến mãi vi phạm OCP: thêm loại "Black Friday" buộc bạn mở method đó, chèn một nhánh, và có nguy cơ ảnh hưởng các nhánh cũ (và phải test lại tất cả). Cách đạt OCP là đưa phần "thay đổi" ra sau một abstraction (interface): mỗi loại khuyến mãi thành một implementation của DiscountRule; thêm loại mới = thêm một class mới, code cũ không bị đụng tới nên không vỡ. Đây chính là Strategy pattern — và là lý do "replace conditional with polymorphism" được coi là refactor đạt OCP.

❌ Fail — mỗi loại mới sửa method này

BigDecimal discount(Order o) {
  if (o.type()==PERCENT)  return o.total().multiply(RATE);
  if (o.type()==FIXED)    return o.fixedOff();
  if (o.type()==BLACKFRI) return o.total().multiply(BF_RATE);
  // type mới → SỬA method này → rủi ro vỡ cũ + test lại hết
  return ZERO;
}

✅ Success — thêm rule = thêm class

interface DiscountRule { BigDecimal apply(Order o); }
class PercentRule    implements DiscountRule {...}
class FixedRule      implements DiscountRule {...}
class BlackFridayRule implements DiscountRule {...}  // THÊM, không sửa cũ

class Pricing {
  private final List<DiscountRule> rules;            // inject
  BigDecimal discount(Order o){
    return rules.stream().map(r->r.apply(o))
                .reduce(ZERO, BigDecimal::add);
  }
}

2.4 L — Liskov Substitution SUPPLEMENTARY

Định nghĩa: nếu S là subtype của T, thì đối tượng T ở mọi nơi đều có thể thay bằng đối tượng S mà không phá vỡ tính đúng của chương trình. Vấn đề nó giải: kế thừa hứa hẹn "subclass dùng được ở mọi chỗ superclass dùng được"; nếu subclass phá lời hứa đó (siết điều kiện đầu vào, ném exception cho thao tác mà superclass cho phép, trả giá trị ngoài hợp đồng), thì code client viết dựa trên superclass sẽ vỡ bất ngờ khi gặp subclass. Dấu hiệu kinh điển vi phạm LSP: bạn phải dùng instanceof để xử lý riêng một subclass, hoặc một subclass override method rồi throw nơi superclass chạy bình thường.

❌ Fail — subclass siết precondition

class Voucher { void redeem(){...} }
class SeasonalVoucher extends Voucher {
  @Override void redeem(){
    if(!isSeason()) throw new IllegalStateException(); // client vỡ
    super.redeem();
  }
}
// Voucher v = new SeasonalVoucher();  v.redeem();
// → ném exception, ngoài kỳ vọng "redeem được mọi lúc" của Voucher

✅ Success — giữ hợp đồng, đổi hành vi hợp lệ

class SeasonalVoucher extends Voucher {
  @Override void redeem(){
    if(!isSeason()) applyOffSeasonAdjustment();  // vẫn redeem được
    super.redeem();
  }
}

2.5 I — Interface Segregation SUPPLEMENTARY

Định nghĩa: client không nên bị buộc phụ thuộc vào những method nó không dùng. Vấn đề nó giải: một interface "béo" (fat interface) với 30 method khiến mọi implementation phải viết stub cho cả những method vô nghĩa với nó, và mọi test phải mock cả 30; tệ hơn, một thay đổi ở method không liên quan vẫn buộc recompile/đụng tới các client chỉ dùng phần khác. Tách interface lớn thành các interface nhỏ theo nhóm nhu cầu (vd tách read khỏi write) làm mỗi client chỉ phụ thuộc đúng phần nó cần.

❌ Fat interface

interface OrderGateway {
  Order place(...); List<Order> history(...);
  Report monthlyReport(...); void notify(...);
  void audit(...);                       // mọi impl phải làm hết
}

✅ Tách theo nhu cầu

interface OrderPlacer  { Order place(...); }
interface OrderReader  { List<Order> history(...); }
interface Reporter     { Report monthly(...); }
// client chỉ depend interface nó cần

2.6 D — Dependency Inversion CORE ★

Định nghĩa: module cấp cao (logic nghiệp vụ) không nên phụ thuộc module cấp thấp (chi tiết kỹ thuật như DB, HTTP client); cả hai nên phụ thuộc vào abstraction. Và abstraction không nên phụ thuộc chi tiết — chi tiết phụ thuộc abstraction. Vấn đề nó giải: nếu PricingService phụ thuộc trực tiếp class cụ thể JpaOrderRepository, thì đổi sang Mongo, hoặc inject một mock khi test, đều buộc sửa PricingService — logic nghiệp vụ bị xích vào chi tiết hạ tầng. Khi PricingService chỉ phụ thuộc interface OrderRepository, bạn "đảo ngược" hướng phụ thuộc: hạ tầng (JpaOrderRepository) implement interface do tầng nghiệp vụ định nghĩa → đổi implementation hoặc test mà không đụng logic. Đây là nền tảng cho việc test bằng mock và cho kiến trúc sạch (#15).

❌ Phụ thuộc class cụ thể

class PricingService {
  private JpaOrderRepository repo;     // concrete!
  // đổi sang Mongo / inject mock = phải sửa PricingService
}

✅ Phụ thuộc abstraction

interface OrderRepository {
  Order findById(long id); void save(Order o);
}
class PricingService {
  private final OrderRepository repo;  // abstraction
  PricingService(OrderRepository repo){ this.repo = repo; }
  // test: inject mock; prod: Jpa; tương lai: Mongo — không sửa logic
}

2.7 — 3 cấp độ mastery SOLID CORE ★

L1 — Biết tên

Đọc thuộc 5 chữ S-O-L-I-D và định nghĩa. Symptom hiểu giả: nhìn code không chỉ ra được nó vi phạm nguyên tắc nào, hay áp dụng máy móc gây over-engineering.

L2 — Explain được

Cho một đoạn code, chỉ ra nguyên tắc bị vi phạm + hậu quả cụ thể khi thêm feature (vd "vi phạm OCP → mỗi khuyến mãi mới sửa if-else → vỡ cũ + test lại hết", "vi phạm SRP → Marketing sửa email làm vỡ logic tính tiền").

L3 — Apply được

Refactor có kiểm soát: viết test trước, tách theo SRP/OCP, và biết dừng đúng lúc — không abstract chỗ chỉ có 1 implementation (rule of three). Cân nhắc ROI mỗi abstraction.

2.8 Refactoring techniques SUPPLEMENTARY

Refactoring là tập hợp các phép biến đổi nhỏ, có tên, đổi cấu trúc mà giữ hành vi. Vài phép hay dùng nhất:

  • Extract Method: tách một khối thành method có tên rõ → method gốc đọc như mục lục, và khối được tách trở nên test/đặt tên được.
  • Extract Class: nhóm các field/method liên quan ra một class riêng — cách chính để tách God Class.
  • Replace Magic Number with Constant: 0.08 → VAT_RATE để code tự giải thích và đổi một chỗ.
  • Replace Conditional with Polymorphism: switch(type) dài → mỗi nhánh thành một subclass/strategy. Đây chính là cách đạt OCP.
  • Introduce Parameter Object: gom 7 tham số rời thành một object (PricingRequest) — chữa Long Parameter List.

2.9 Tell-Don't-Ask SUPPLEMENTARY

Nguyên tắc: thay vì hỏi một object dữ liệu của nó rồi tự xử lý bên ngoài, hãy bảo object đó tự làm việc với dữ liệu của mình. Lý do: khi logic và dữ liệu nằm cùng chỗ (trong object), các invariant (điều kiện luôn đúng) được giữ ở một nơi và không thể bị quên ở chỗ khác. Đoạn "ask" lôi status và items ra ngoài để kiểm tra rồi mới set — nếu có nơi khác cũng làm vậy mà quên một điều kiện thì sinh bug; đoạn "tell" gói invariant vào trong Order.markShipped() nên mọi nơi gọi đều an toàn.

❌ Ask — lôi data ra ngoài xử lý

if (order.getStatus() == PAID
    && order.getItems().size() > 0) {
    order.setStatus(SHIPPED);
}  // invariant rải ra ngoài → dễ quên ở nơi khác

✅ Tell — giao việc cho object giữ data

order.markShipped();
// invariant (đã PAID, có item) nằm TRONG Order
// → không thể quên check ở nơi khác

3. Refactor messy → clean (có code, có test) CORE ★

Refactor an toàn không phải "viết lại cho đẹp" mà là một chuỗi bước nhỏ, mỗi bước chạy lại test để đảm bảo hành vi không đổi. Áp cho method calculateOrderTotal() 200 dòng ở phần 1:

❌ Trước — Long Method + Mixed Concerns + Magic Number

BigDecimal calculateOrderTotal(String customerId, List<Item> items,
    String couponCode, boolean isVip, String channel,
    LocalDate date, String currency) {
  // validate + tính subtotal + khuyến mãi (magic, if-else)
  // + thuế + GHI DB + GỬI MAIL — tất cả trong một hàm
}

✅ Sau — tách trách nhiệm, OCP cho discount

// B1: viết characterization test phủ hành vi HIỆN TẠI (kể cả bug) trước khi đụng code
record PricingRequest(String customerId, List<Item> items,
    String coupon, boolean vip, String channel,
    LocalDate date, String currency) {}                 // B2: Parameter Object

interface DiscountRule { BigDecimal apply(PricingRequest r, BigDecimal sub); } // B3: Strategy (OCP)
class OrderValidator { void validate(PricingRequest r){...} }                   // B4: tách validation (SRP)

class PricingCalculator {                               // B5: hàm tính THUẦN, không side-effect
  private final List<DiscountRule> rules;
  private static final BigDecimal VAT = new BigDecimal("0.08");
  Money total(PricingRequest r){
    var sub = r.items().stream().map(Item::lineTotal).reduce(ZERO, BigDecimal::add);
    var disc = rules.stream().map(x->x.apply(r, sub)).reduce(ZERO, BigDecimal::add);
    var taxed = sub.subtract(disc);
    return new Money(taxed.add(taxed.multiply(VAT)));
  }
}

class PlaceOrderUseCase {                               // B6: Facade nối các phần, side-effect tách
  Order place(PricingRequest r){
    validator.validate(r);
    Money total = calculator.total(r);
    Order o = orderRepo.save(Order.of(r, total));
    events.publish(new OrderPlaced(o.id()));            // notify async, KHÔNG trong hàm tính
    return o;
  }
}
Nguyên tắc vàng

Mỗi bước nhỏ + chạy test ngay sau mỗi bước. Test xanh sau mỗi commit = bạn luôn có điểm quay lui. Đây là khác biệt giữa "refactor" (an toàn, không đổi hành vi) và "rewrite" (rủi ro, dễ sinh bug mới).

4. Khi SOLID phản tác dụng CORE ★

🔧 Counter-example — khi nào KHÔNG áp dụng SOLID/abstraction

SOLID phục vụ khả năng thay đổi. Nơi không có trục thay đổi, áp dụng nó chỉ thêm chi phí đọc mà không có lợi ích — over-engineering hại không kém under-engineering:

  • Premature abstraction: tạo interface + factory cho cái chỉ có một implementation và chưa có dấu hiệu sẽ có cái thứ hai. Theo "rule of three": chờ đến lần thứ ba lặp lại mới abstract — vì lúc đó bạn mới biết đúng trục nào cần linh hoạt.
  • SOLID over-engineering: một CRUD đơn giản mà có 6 lớp (controller → service → usecase → gateway → mapper → repository) cho một findById = phải đọc 6 file để hiểu một việc tầm thường.
  • Tách SRP quá mịn: mỗi class 5 dòng, 40 class cho một flow → biến thành "shotgun surgery" ngược: thêm một field phải đụng 6 class.
  • Refactor không test: đổi hành vi âm thầm → bug. Characterization test trước, luôn.

Quy tắc: abstraction phục vụ thay đổi đã thấy hoặc rất khả năng xảy ra. Không có trục thay đổi → YAGNI (You Aren't Gonna Need It). Senior thể hiện ở chỗ biết dừng, kèm lý do — không phải ở chỗ dùng nhiều pattern nhất.

5. 3-tier interview framework

Tầng 1 — SurfaceMid gate

Q: Cho đoạn code, chỉ ra vi phạm SOLID.

Nói tên nguyên tắc + bằng chứng cụ thể: "method này thêm loại giảm giá phải sửa chuỗi if-else → vi phạm OCP", "class này vừa tính tiền vừa gửi mail → hai lý do đổi → vi phạm SRP". Không nói chung "code xấu".

Tầng 2 — Deep DiveSenior gate

Q: Refactor method 200 dòng này từng bước.

Characterization test phủ trước → Parameter Object → Strategy cho discount (OCP) → tách validation (SRP) → tách side-effect (DB/email ra khỏi hàm tính) → Facade orchestrator. Chạy test sau mỗi bước. Nhấn mạnh "không đổi hành vi" = refactor, không phải rewrite.

Tầng 3 — ArchitectureSenior+

Q: Khi nào KHÔNG nên áp dụng SOLID?

Khi không có trục thay đổi (YAGNI), code đơn giản/throwaway, 1 implementation duy nhất, hoặc abstraction làm tăng chi phí đọc > lợi ích. Rule of three. Senior cân bằng giữa flexibility và simplicity, không cargo-cult pattern.

🔧 Lab bắt buộc — refactor có kiểm soát (2 bài)

Lab 1 — OCP qua Strategy. Lấy một method tính khuyến mãi với if(type=="PERCENT")...else if(type=="FIXED").... Viết characterization test phủ các nhánh. Refactor sang DiscountRule interface + các implementation, inject list. Thêm một rule mới mà KHÔNG sửa code cũ. Tiêu chí: chứng minh test cũ vẫn xanh và rule mới chỉ thêm file.

Lab 2 — Nhận diện over-engineering. Lấy một đoạn code "SOLID quá đà" (interface + factory cho 1 implementation). Viết lại đơn giản hơn (xoá abstraction thừa) mà vẫn pass test. Tiêu chí: giải thích "rule of three" và vì sao bản đơn giản dễ đọc hơn ở quy mô hiện tại.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. SRP nói "một lý do để thay đổi" nghĩa là gì? Cho ví dụ một class vi phạm + hậu quả.
  2. OCP: "mở để mở rộng, đóng để sửa đổi" — vì sao sửa code đang chạy là rủi ro? Refactor if-else thành cái gì để đạt OCP?
  3. DIP "đảo ngược phụ thuộc" nghĩa là gì? Vì sao nó cho phép test bằng mock?
  4. Tại sao phải viết test TRƯỚC khi refactor? Khác gì với rewrite?
