#02 - JVM Internals: GC Tuning, Memory Leak, Heap Dump, ClassLoader

JVM là runtime vận hành code Java của bạn trong production. Senior backend cần biết JVM dùng memory thế nào, GC quyết định object sống/chết ra sao, vì sao memory leak vẫn xảy ra trong Java, và cần lấy evidence gì khi service OOM hoặc pause dài.

JVM Internals Là Kiến Thức Sống Còn Khi Production Có Sự Cố

Khi service chậm, lỗi không phải lúc nào cũng nằm ở SQL hay API bên ngoài. Có thể JVM đang dừng thế giới để GC, heap bị leak, thread stack quá nhiều, direct memory bị Netty/Kafka dùng hết, hoặc classloader giữ metadata không được unload. Nếu chỉ biết "Java có GC tự dọn rác", bạn sẽ không biết phải lấy heap dump, thread dump hay GC log trước khi restart.

Bài này viết theo kiểu giáo trình vận hành: trước hết học bản đồ memory, sau đó học GC roots và reachability, rồi học cách đọc dấu hiệu leak, cuối cùng là playbook debug và cấu hình production.

Cách học JVM để không hiểu giả

Đừng học JVM bằng cách thuộc tên vùng nhớ. Hãy học theo câu hỏi incident: service OOM vì heap hay non-heap? GC pause dài vì allocation spike hay leak? Heap dump object nào giữ memory? GC root nào giữ nó sống? Nếu restart ngay mà không lấy evidence, bạn chỉ xóa dấu vết.

1. Bài Toán Thực Tế - Service Resize Ảnh Chết Mỗi 3 Giờ

Một service xử lý ảnh sản phẩm nhận file từ queue, resize nhiều kích thước rồi upload lên object storage. Sau deploy bản mới, service bắt đầu có pattern: heap tăng đều, CPU GC tăng, latency upload tăng từ 200ms lên 8s, rồi pod restart vì OutOfMemoryError.

@Service
public class ImageMetadataService {
    private static final Map<String, Metadata> cache = new HashMap<>();

    public Metadata metadata(String imageId) {
        return cache.computeIfAbsent(imageId, this::loadMetadata);

}
Đúng: bounded cache
@Service
public class ImageMetadataService {
    private final Cache<String, Metadata> cache = Caffeine.newBuilder()
        .maximumSize(20_000)
        .expireAfterWrite(Duration.ofMinutes(30))
        .recordStats()
        .build();

    public Metadata metadata(String imageId) {
        return cache.get(imageId, this::loadMetadata);
    }
}

GC không collect được object trong static map vì object vẫn reachable từ GC root. Với JVM, "không còn business value" không đồng nghĩa "garbage". Chỉ object unreachable mới là garbage.

2.1 Container Memory Budget

Một lỗi production phổ biến là set -Xmx gần bằng toàn bộ memory limit của container. JVM không chỉ dùng heap. Nếu container 4GB và bạn set -Xmx4g, process vẫn cần metaspace, thread stacks, direct buffers, code cache và native memory. Kết quả có thể là container bị OOMKilled bởi kernel mà Java không kịp ném OutOfMemoryError.

Container 4GBBudget gợi ýLý do

Metaspace256MB - 512MBClass metadata, proxy, framework classes.
Thread stacks200MB - 500MBVí dụ 400 threads * 1MB stack là 400MB.

Container OOM khác Java OOM

Java OOM thường có stack trace và có thể tạo heap dump. Container OOMKilled có thể chỉ hiện trong Kubernetes event, không có heap dump. Vì vậy heap sizing phải để dư native memory.

2.2 Object Allocation Path

Hầu hết object mới sinh ở Eden. JVM tối ưu allocation rất mạnh bằng TLAB: mỗi thread có một vùng nhỏ trong Eden để tự cấp phát object mà không cần lock global. Allocation trong Java vì vậy thường rất rẻ; vấn đề không phải "tạo object nào cũng xấu", mà là allocation rate quá cao làm Minor GC chạy liên tục hoặc object sống quá lâu bị promote lên Old Gen.

Object ngắn hạn chết ở Young Gen là bình thường. Object sống lâu bất thường mới đáng nghi.

Ví dụ: tạo DTO response ngắn hạn cho mỗi request thường ổn. Nhưng collect 2 triệu DTO vào list để export report có thể gây pressure lớn. Khi xử lý data lớn, hãy stream/chunk thay vì materialize toàn bộ.

2.3 Object Layout: Vì Sao Nhiều Object Nhỏ Cũng Tốn Memory

Một object Java không chỉ chứa field của bạn. Nó có object header, alignment/padding, reference fields và có thể bị nén reference hoặc không tùy heap size/JVM config. Vì vậy một object "chỉ có vài field" vẫn có overhead đáng kể.

Thành phầnÝ nghĩa
Object headerMark word, class pointer, lock/hash metadata.
Instance fieldsPrimitive values và references tới object khác.
Padding/alignmentJVM align object size theo boundary nhất định.
Referenced objectsString, BigDecimal, List kéo theo object graph khác.
Học bằng đo đạc

Khi cần biết object size thật, dùng JOL (Java Object Layout) trong môi trường lab. Đừng đoán memory chỉ bằng cách cộng field size.

3. Garbage Collection

GC không "xóa object bạn không dùng". GC bắt đầu từ GC roots, đi theo reference graph. Object nào reachable thì sống; object nào unreachable thì collect. Vì vậy leak trong Java là object vẫn reachable dù business không cần nữa.

3.1 GC Roots

GC roots là điểm bắt đầu của graph traversal. Từ roots, JVM đi theo reference tới object khác. Những object đi tới được là live. Những object không đi tới được là garbage.

GC RootVí dụLeak thường gặp
Thread stackLocal variable trong method đang chạyTask giữ list lớn quá lâu trong stack frame.
Static fieldstatic Map, singleton registryCache không giới hạn, global list.
JNI/native referenceNative library giữ objectResource wrapper không close đúng.
Monitor lockObject đang bị synchronizedÍt phổ biến, nhưng object locked vẫn reachable.
Class metadataClass object, classloaderClassLoader leak khi plugin/hot reload.
Leak path thường chính là đường reference từ GC root tới object chiếm heap.

3.2 Generational Hypothesis

Generational GC dựa trên quan sát: phần lớn object chết trẻ, một số ít sống lâu. Vì vậy JVM tối ưu Young Gen để collect nhanh object ngắn hạn, và Old Gen cho object sống lâu.

  • Eden: object mới sinh.
  • Survivor spaces: object sống qua một vài Minor GC.
  • Old Gen: object sống đủ lâu hoặc object lớn được promote.
Tại sao leak thường hiện ở Old Gen?

Object leak thường sống qua nhiều GC vì vẫn reachable từ static map, queue, ThreadLocal hoặc listener. Sau vài cycle, chúng được promote lên Old Gen. Vì vậy old-gen after-GC baseline tăng đều là dấu hiệu quan trọng.

3.3 Minor GC, Major GC, Full GC

Object mới thường sinh ở Eden. Eden đầy thì Minor GC chạy, object sống được copy sang Survivor hoặc promote lên Old Gen. Old Gen đầy hoặc pressure cao thì Major/Full GC đắt hơn nhiều. Dấu hiệu leak là sau mỗi Full GC, heap used vẫn không quay về baseline.

GCKhi nàoPauseÝ nghĩa khi xuất hiện nhiều
Minor GCYoung Gen đầyThường ngắnAllocation rate cao, object ngắn hạn nhiều
Major/Old GCOld Gen pressureDài hơnObject sống lâu nhiều, có thể leak
Full GCJVM cần thu gom mạnh hoặc failureStop-the-world dàiDấu hiệu nghiêm trọng nếu lặp lại

3.4 G1GC vs ZGC: Mental Model

G1GC chia heap thành nhiều region và ưu tiên collect region có nhiều garbage. Nó cố giữ pause gần mục tiêu bạn cấu hình, ví dụ MaxGCPauseMillis=200, nhưng đây là mục tiêu mềm, không phải SLA guarantee.

ZGC thiết kế cho low-latency, làm nhiều việc concurrent với application thread để pause rất ngắn. Nó hợp với heap lớn và workload nhạy latency, nhưng yêu cầu Java version phù hợp và cần quan sát overhead thực tế.

CollectorMục tiêuDùng khiTrade-off
G1GCBalance throughput/latencyService backend general purposeCần heap headroom, pause target không phải guarantee tuyệt đối
ZGCPause rất thấpLatency-sensitive, heap lớnJava version mới, overhead và tuning khác
Parallel GCThroughput tối đaBatch/offline jobPause dài hơn, không hợp API latency nghiêm

4. Memory Leak: Cách Nhìn Đúng

Memory leak trong Java không phải "GC hỏng". Leak là reference graph còn giữ object. Các pattern phổ biến trong backend:

4.1 Leak Patterns Hay Gặp

  • Static map/list không giới hạn.
  • Cache thiếu eviction hoặc TTL.
  • ThreadLocal không remove() trong thread pool.
  • Listener/callback đăng ký nhưng không unregister.
  • Queue tích tụ vì consumer chậm hơn producer.
  • Load toàn bộ file/report vào memory thay vì stream/chunk.

ThreadLocal Leak

public class RequestContext {
    private static final ThreadLocal<UserContext> CTX = new ThreadLocal<>();

    public static void set(UserContext context) {
        CTX.set(context);
    }

    public static void clear() {
        CTX.remove();
    }
}

public void handle(Request request) {
    try {
        RequestContext.set(buildContext(request));
        doHandle(request);
    } finally {
        RequestContext.clear(); // bắt buộc vì thread pool reuse thread
    }
}

Queue Leak Do Backpressure Thiếu

// Sai: producer nhanh hơn consumer, queue unbounded
BlockingQueue<ImageJob> queue = new LinkedBlockingQueue<>();

// Đúng hơn: bounded queue + rejected/backpressure policy
BlockingQueue<ImageJob> queue = new ArrayBlockingQueue<>(10_000);

Listener Leak

Listener leak xảy ra khi object đăng ký callback vào publisher sống lâu hơn nó, nhưng không unregister. Publisher giữ reference tới listener, listener giữ object graph phía sau, GC không collect được.

public class PreviewSession implements AutoCloseable {
    private final EventBus eventBus;
    private final EventListener listener;

    public PreviewSession(EventBus eventBus) {
        this.eventBus = eventBus;
        this.listener = event -> refreshPreview(event);
        this.eventBus.register(listener);
    }

    @Override
    public void close() {
        eventBus.unregister(listener);
    }
}

Static Cache Leak

// Sai: static cache không có eviction
private static final Map<String, byte[]> thumbnails = new ConcurrentHashMap<>();

// Tốt hơn: cache có maximum size/weight
private final Cache<String, byte[]> thumbnails = Caffeine.newBuilder()
    .maximumWeight(256 * 1024 * 1024)
    .weigher((String key, byte[] value) -> value.length)
    .expireAfterAccess(Duration.ofMinutes(20))
    .build();

5. ClassLoader & JIT

5.1 ClassLoader

ClassLoader nạp class metadata vào Metaspace. Parent delegation giúp tránh class core bị override sai. Với plugin/hot reload, nếu custom classloader bị giữ reference, class metadata không unload được và Metaspace tăng mãi.

App classloader hỏi parent trước. Đây là parent delegation model.

5.2 JIT Warm-up

Java ban đầu interpret bytecode. Method nóng được JIT compile sang native code. Vì vậy benchmark Java phải có warm-up; đo ngay sau startup thường sai. Trong production autoscaling, pod mới cũng cần warm-up trước khi nhận toàn bộ traffic.

5.3 ClassLoader Leak Trong Plugin/Hot Reload

Class chỉ unload khi classloader của nó unreachable. Nếu một thread, static field, ThreadLocal hoặc singleton registry còn giữ object thuộc plugin classloader, toàn bộ classloader và class metadata phía sau vẫn sống trong Metaspace.

public class PluginManager {
    private URLClassLoader currentLoader;

    public void loadPlugin(Path jar) throws IOException {
        URLClassLoader loader = new URLClassLoader(
            new URL[] { jar.toUri().toURL() },
            getClass().getClassLoader()
        );
        this.currentLoader = loader;
    }

    public void unload() throws IOException {
        if (currentLoader != null) {
            currentLoader.close();      // release jar/native resources
            currentLoader = null;       // release strong reference
        }
    }
}

Trong thực tế, unload còn cần dừng thread do plugin tạo, unregister listener, clear ThreadLocal và xóa static registry. Chỉ close() classloader chưa đủ nếu object graph vẫn còn reference.

5.4 JIT Và Benchmark Sai

Java benchmark bằng vòng for và System.currentTimeMillis() rất dễ sai vì JIT warm-up, dead-code elimination, constant folding, escape analysis. Với microbenchmark, dùng JMH. Với production performance, dùng profiler và metrics thật.

Sai: đo thủ công dễ bị JIT tối ưu mất
long start = System.nanoTime();
for (int i = 0; i < 1_000_000; i++) {
    new BigDecimal("123").toString();
}
System.out.println(System.nanoTime() - start);
Đúng hướng: dùng JMH
@Benchmark
public String formatAmount() {
    return amount.toPlainString();
}

6.5 Heap Dump Và Security

Heap dump có thể chứa token, email, request body, password đã parse, dữ liệu người dùng và secret trong memory. Vì vậy heap dump là artifact nhạy cảm: lưu vùng private, giới hạn quyền đọc, xóa sau khi phân tích, không gửi qua kênh chat công khai.

7. JFR, Profiling Và GC Tuning Cookbook

Heap dump trả lời câu hỏi "ai đang giữ memory". GC log trả lời câu hỏi "GC đang làm gì". Thread dump trả lời câu hỏi "thread đang chờ ở đâu". Nhưng khi service chậm mà không OOM, bạn cần profiler: CPU đang cháy ở method nào, allocation đến từ call path nào, lock contention nằm ở monitor nào, request bị block ở socket hay database client?

7.1 Java Flight Recorder: Evidence Đầu Tiên Khi Latency Tăng

Java Flight Recorder là công cụ profiling có sẵn trong JDK, overhead thấp hơn nhiều so với cách log thủ công mọi method. JFR ghi lại event cấp JVM: CPU sample, allocation, GC, safepoint, thread park/block, socket I/O, file I/O, exception, class loading. Trong incident thật, JFR thường là artifact giàu thông tin nhất nếu service vẫn còn sống.

# Ghi JFR 5 phút trên process đang chạy
jcmd <pid> JFR.start name=incident settings=profile duration=5m filename=/tmp/app-incident.jfr

# Kiểm tra recording
jcmd <pid> JFR.check

# Dừng thủ công nếu không dùng duration
jcmd <pid> JFR.stop name=incident filename=/tmp/app-incident.jfr

Đầu tiên xem GC pause và heap trend để loại trừ GC. Sau đó xem Hot Methods để biết CPU đi đâu. Tiếp theo xem Allocation để tìm object churn. Cuối cùng xem Thread Park/Monitor Blocked/Socket I/O để phân biệt CPU-bound với blocking-bound.

7.2 Flame Graph: Đọc Profiling Không Bị Ngộ Nhận

Flame graph biểu diễn stack traces theo bề rộng. Khung càng rộng nghĩa là stack đó xuất hiện nhiều sample hơn, không nhất thiết method đó tự chạy lâu. Senior cần phân biệt inclusive time và self time: một method rộng có thể chỉ là wrapper gọi method con đắt đỏ.

Dạng flame graphÝ nghĩaHành động
CPU flame graphCPU sample nằm ở stack nàoTối ưu algorithm, cache kết quả, giảm serialization/regex/compression
Allocation flame graphObject được tạo từ call path nàoGiảm object churn, dùng primitive stream, reuse buffer có kiểm soát
Lock/block profileThread mất thời gian chờ ở đâuGiảm critical section, đổi lock granularity, bounded queue
Wall-clock profileThời gian thực bao gồm sleep/I/O/blockingPhân biệt CPU chậm với dependency chậm
 
Cách diễn giải:
Wide frame at ObjectMapper.writeValueAsString
  -> serialization chiếm nhiều CPU/allocation.

Wide frame at SocketInputStream.socketRead0
  -> thread đang chờ I/O, tối ưu CPU không giải quyết latency.

Wide frame at ReentrantLock.lock
  -> contention; cần xem ai giữ lock và critical section làm gì.
Bẫy khi đọc profiler

Không tối ưu method xuất hiện đầu tiên trong flame graph nếu nó chỉ là framework wrapper. Đi xuống stack tới method application hoặc library call thực sự tạo CPU/allocation/blocking. Luôn so với traffic và release gần nhất.

7.3 GC Tuning Cookbook: Chỉnh Ít Nhưng Có Evidence

GC tuning không bắt đầu bằng việc copy flags từ internet. Nó bắt đầu bằng SLA: pause tối đa chấp nhận bao nhiêu, throughput quan trọng đến đâu, heap live set bao nhiêu, allocation rate bao nhiêu, container memory giới hạn thế nào. Mỗi flag nên trả lời một vấn đề đo được.

ZGC: Khi Mục Tiêu Là Pause Rất Thấp

ZGC phù hợp khi pause thấp quan trọng hơn throughput tuyệt đối và bạn chạy trên JDK hỗ trợ ổn định. Tư duy với ZGC khác G1: bạn ít chỉnh region/young/old hơn, tập trung vào đủ heap headroom, allocation rate và CPU cho concurrent GC threads.

G1 phù hợp khi
  • Backend general-purpose.
  • Pause mục tiêu vài chục đến vài trăm ms.
  • Cần collector mặc định dễ vận hành.
ZGC phù hợp khi
  • Latency rất nhạy với stop-the-world pause.
  • Heap lớn, live set lớn.
  • Chấp nhận theo dõi CPU/headroom kỹ hơn.

Quy Trình Tuning Có Kỷ Luật

  1. Định nghĩa vấn đề: p99 latency tăng, Full GC, OOM, CPU GC cao hay RSS vượt limit.
  2. Lấy baseline: GC log, JFR, heap after-GC, allocation rate, traffic cùng thời điểm.
  3. Đổi một biến mỗi lần: heap size, pause target, cache size, chunk size, collector.
  4. So sánh bằng số: p95/p99, GC pause total, allocation rate, CPU, RSS, error rate.
  5. Ghi lại decision: vì sao chọn flag, rollback khi metric nào xấu.

7.5 Case Study: Media API Tăng p99 Sau Release

Bối cảnh: một API tạo thumbnail cho ảnh user upload. Sau release, p99 tăng từ 350ms lên 3s, CPU tăng, Young GC dày hơn, không có OOM. Restart tạm giảm latency nhưng sau vài giờ lại tăng.

Bước 2: Lấy JFR Và Đọc Hot Path

JFR allocation hot path:
ThumbnailController.upload
  ImagePipeline.process
    byte[] original = multipartFile.getBytes()
    BufferedImage image = ImageIO.read(new ByteArrayInputStream(original))
    byte[] preview = resizeAndEncode(image)
    metadata.put("preview", Base64.getEncoder().encodeToString(preview))

Kết luận: code mới load toàn bộ upload vào byte array, tạo thêm encoded string lớn, rồi giữ preview trong metadata map để trả về response. Không phải leak, mà là allocation amplification: mỗi request tạo nhiều bản copy của cùng dữ liệu.

Bước 3: Fix Thiết Kế Memory

  • Stream upload vào temp file hoặc object storage thay vì getBytes().
  • Resize theo stream/chunk nếu library hỗ trợ, tránh giữ nhiều bản copy.
  • Không nhúng preview base64 lớn vào metadata response; trả URL hoặc ID.
  • Giới hạn kích thước upload và reject sớm.
  • Thêm alert allocation rate và Young GC pause total.

Bước 4: Cách Trả Lời Interview

Không nói "tăng heap". Hãy nói: "Tôi sẽ lấy GC log và JFR. Nếu after-GC baseline không tăng nhưng allocation rate tăng, tôi nghi allocation churn. Tôi mở allocation flame graph để tìm call path tạo object lớn. Fix đúng là giảm bản copy/stream dữ liệu, không chỉ tuning GC."

9. Interview Framework

SurfaceJava có GC, sao vẫn memory leak?

GC chỉ collect object unreachable. Nếu object không còn cần cho business nhưng vẫn được static map, ThreadLocal, listener hoặc queue giữ reference, nó vẫn reachable từ GC root nên không bị collect.

DeepBạn debug OOM production thế nào?

Stabilize và lấy evidence: heap dump, thread dump, GC log. Phân tích GC log xem after-GC baseline tăng không. Mở heap dump bằng MAT, xem dominator tree và path to GC roots. Fix retention path thật: bounded cache, clear ThreadLocal, unregister listener, stream/chunk data.

ArchitectureCấu hình JVM cho service latency-sensitive

Chọn G1GC hoặc ZGC tùy SLA và Java version. Set heap cố định nhưng không chiếm hết RAM container. Bật GC log, heap dump on OOM, max metaspace, exit on OOM. Monitor GC pause, heap after GC, old-gen growth, direct memory và thread count.

Checklist Thành Thạo

  • Vẽ được JVM memory layout và phân biệt heap/non-heap.
  • Giải thích được GC roots, reachability và vì sao static map gây leak.
  • Phân biệt allocation spike với memory leak tăng đều qua after-GC baseline.
  • Biết lấy heap dump, thread dump, GC log bằng command line.
  • Đọc được JVM flags production cơ bản và lý do của từng flag.
  • Giải thích classloader leak và vì sao Metaspace có thể OOM.
  • Biết dùng JFR để phân biệt CPU-bound, allocation-heavy, blocking và GC pause.
  • Đọc được flame graph ở mức inclusive/self time và tránh tối ưu nhầm framework wrapper.
  • Biết tuning G1GC theo evidence: pause target, allocation rate, live set, humongous allocation.
  • Phân biệt heap OOM với container OOMKilled do direct memory, thread stack hoặc metaspace.
  • Trình bày được một case study latency tăng từ dashboard tới JFR, kết luận và fix thiết kế.
