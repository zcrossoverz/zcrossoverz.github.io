Spring Boot — Auto-configuration, Actuator, Custom Starter & Property Binding

Spring Boot không phải Spring khác — nó là convention layer trên Spring Core. Điểm phân biệt junior vs senior là hiểu mechanism, không chỉ dùng annotation: vì sao KafkaTemplate xuất hiện tự động? Vì sao K8s Secret override được application.yml? Vì sao Custom Starter cần @ConditionalOnMissingBean? File này deep-dive từ AutoConfiguration.imports đến custom starter production-ready.

📖 5 quy tắc đọc file này — KHÔNG đọc một mạch

  1. Mở 1 Spring Boot project thật khi đọc. Mỗi module có lab tương ứng — đừng đọc lý thuyết. Run app, gọi /actuator/conditions, đọc output cho từng concept.
  2. Bật --debug ít nhất 1 lần. java -jar app.jar --debug → console in full auto-config report. Phải tự mắt thấy "DataSourceAutoConfiguration matched / KafkaAutoConfiguration not matched" để hiểu cơ chế.
  3. Thử override 1 auto-configured bean. Tạo @Bean DataSource trong code → quan sát log "DataSourceAutoConfiguration did not match: existing bean defined". Đây là cách @ConditionalOnMissingBean hoạt động thực sự.
  4. Test property override với env var. Set SERVER_PORT=9090 trước khi run → app start trên port 9090, override application.yml. Hiểu được relaxed binding bằng thực nghiệm.
  5. Build custom starter ít nhất 1 lần. Lab 3 hướng dẫn. Khi tự build được starter + dùng nó từ project khác → đã master Boot.

1. Vì sao Spring Boot ra đời? CORE

Spring Boot KHÔNG phải framework mới — nó là opinionated convention trên Spring Framework. Hiểu lý do ra đời = hiểu "tại sao auto-config tồn tại" và "tại sao starter là pattern hay".

📜 Lịch sử Spring Boot — từ pain point đến de-facto standard

  • 2004–2012 — Spring Framework dominant Java backend, nhưng config nặng: XML files dài 500 dòng, Java config tay cho mỗi bean, dependency version conflicts thường xuyên. Một Spring MVC project mới mất 2–3 ngày setup cơ bản.
  • 2013, July — Pivotal team announce Spring Boot 0.5 (alpha). Triết lý: "Take an opinionated view of building production-ready Spring applications". Kiểu Rails / Play Framework cho Java.
  • 2014, April — Spring Boot 1.0 GA. 3 trụ: auto-configuration, starter dependencies, embedded server (Tomcat trong jar). Project setup từ 2 ngày → 5 phút với Spring Initializr (start.spring.io).
  • 2015–2017 — Spring Boot 1.x dominate. Microservices boom (Netflix OSS) đẩy adoption. Actuator + Spring Cloud thành standard cho cloud-native Java.
  • 2018, March — Spring Boot 2.0: Reactive (WebFlux), Java 8 baseline, HikariCP default, Micrometer cho metrics.
  • 2022, November — Spring Boot 3.0: Java 17 baseline, Jakarta EE 9+ (javax → jakarta), native compilation với GraalVM AOT. spring.factories → AutoConfiguration.imports.
  • 2024–2026 — Spring Boot 3.2+: virtual threads (Project Loom), improved observability, AOT cho cold-start dưới 100ms. Vẫn dominate Java backend với 70%+ market share trong web services.

Verifiable: Spring Boot release notes — spring.io/blog. Spring Initializr — start.spring.io. AutoConfiguration source — github.com/spring-projects/spring-boot.

3 thứ Spring Boot thêm so với Spring thuần

Tính năngTrước Spring BootVới Spring Boot
Configuration XML 500 dòng + Java config tay cho DataSource, JPA, MVC, Security Auto-configuration tự động kích hoạt theo classpath. Override khi cần.
Dependencies Tự pick version từng lib (Spring 4.3 + Hibernate 5.2 + Jackson 2.9 — có compatible?) Starter POMs bundle tested versions. Add 1 dependency = đủ stack.
Deployment WAR + external Tomcat / WebLogic. Config server-specific. Embedded Tomcat / Netty / Undertow trong jar. java -jar app.jar chạy ngay.

2. Bug onboard X Bank — 4 câu hỏi CORE

Bạn vừa join dự án X Bank, được giao setup một microservice mới. Bạn copy config từ service cũ:

# application.yml
spring:
  datasource:
    url: jdbc:postgresql://db-host:5432/xbank_db
    username: ${DB_USERNAME}
    password: ${DB_PASSWORD}

server:
  port: 8080

Service start OK, nhưng khi gọi API → Connection refused. Bạn kiểm tra:

# Environment variables đã được set
echo $DB_USERNAME  # → xbank_user
echo $DB_PASSWORD  # → (có giá trị)

# Nhưng log Spring Boot hiện:
# HikariPool: Failed to obtain connection - Connection refused: db-host/192.168.1.100:5432
# DataSourceHealthContributor: DOWN

Sau đó bạn thêm config Kafka nhưng service bị crash khi start:

NoSuchBeanDefinitionException: No qualifying bean of type 'KafkaTemplate'

Trong khi service cũ (copy từ đó) có KafkaTemplate hoạt động tốt.

❌ 4 câu hỏi buộc bạn tư duy

  1. Spring Boot resolve ${DB_USERNAME} từ đâu? Thứ tự ưu tiên của các property sources là gì? (→ Section 4)
  2. Tại sao service cũ có KafkaTemplate mà service mới thì không? (→ Section 3)
  3. DataSourceHealthContributor: DOWN xuất hiện mà không cần config — ai tạo ra nó? (→ Section 6)
  4. Làm thế nào để debug xem Spring Boot đã auto-configure những gì? (→ Section 6, /actuator/conditions)
Chi phí onboard sai: Bug "copy config nhưng không hoạt động" mất trung bình 4–8 giờ debug cho mỗi junior mới join — tổng quy mô X Bank 10 microservices × 20 junior/năm = ~1200 giờ/năm lãng phí. Hiểu auto-config + property sources giảm xuống còn 1 giờ/onboard.

3. Auto-configuration — "Convention over Configuration" CORE

Vì sao CORE: Auto-config là trái tim của Spring Boot. Hiểu = debug được mọi bug "bean này từ đâu ra / không xuất hiện". Không hiểu = chỉ là user copy-paste từ Stack Overflow.

Auto-config KHÔNG phải magic — là systematic convention

Khi bạn add dependency spring-boot-starter-data-jpa:
  pom.xml / build.gradle
      ↓
  spring-boot-autoconfigure.jar
      ↓
  META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
  (Spring Boot 3.x; trước đó là spring.factories trong Boot 2.x)
      ↓
  List của 140+ AutoConfiguration classes

Ví dụ: DataSourceAutoConfiguration
  @ConditionalOnClass({ DataSource.class, EmbeddedDatabaseType.class })
  @ConditionalOnMissingBean(type = "io.r2dbc.spi.ConnectionFactory")
  @Import({ DataSourcePoolMetadataProvidersConfiguration.class, ... })
  public class DataSourceAutoConfiguration {
    // Chỉ tạo DataSource bean NẾU:
    // 1. DataSource.class có trong classpath (→ JDBC driver được add)
    // 2. CHƯA có bean DataSource nào được define (→ không override của bạn)
  }

Flow đầy đủ khi Spring Boot start

1. @SpringBootApplication
       ↓
   @EnableAutoConfiguration
       ↓
2. AutoConfigurationImportSelector
       ↓
   Đọc spring/AutoConfiguration.imports → list 140+ classes
       ↓
3. Filter qua @Conditional annotations:
   @ConditionalOnClass        — class có trong classpath?
   @ConditionalOnMissingBean  — bean chưa được define?
   @ConditionalOnProperty     — property có set không?
   @ConditionalOnWebApplication — đây có phải web app?
       ↓
4. Register BeanDefinitions cho những class pass điều kiện
       ↓
5. Create beans theo lifecycle thông thường (→ #05 Spring Core)

Vì sao KafkaTemplate không xuất hiện ở service mới?

Answer: Service cũ có spring-kafka trong pom.xml → KafkaAutoConfiguration được activate (vì @ConditionalOnClass(KafkaTemplate.class) match) → KafkaTemplate bean được tạo. Service mới KHÔNG có dependency này → @ConditionalOnClass fail → auto-config skip → KHÔNG có bean. Fix: add spring-kafka hoặc spring-boot-starter-kafka.

4 loại @Conditional thường gặp

AnnotationTrigger khiUse case
@ConditionalOnClassClass có trong classpathAuto-config chỉ chạy nếu có dependency tương ứng (KafkaAutoConfiguration cần KafkaTemplate.class)
@ConditionalOnMissingBeanBean chưa được define"Default if user chưa override" — pattern back-off
@ConditionalOnPropertyProperty set giá trị nhất địnhToggle feature: banking.audit.enabled=true kích hoạt audit
@ConditionalOnWebApplicationLà web app (Servlet hoặc Reactive)Tracing filter chỉ chạy cho web app, không cho batch job

📌 Đáp án Pause & Predict từ Section 1

Câu hỏi: Add spring-boot-starter-data-redis mà không config gì — service start được không? RedisTemplate có available?
Câu trả lời: CÓ, service start được. RedisAutoConfiguration sẽ activate (vì RedisTemplate.class có trong classpath). Spring Boot dùng default config: localhost:6379. Bean RedisTemplate được tạo và inject được. NHƯNG khi gọi method (vd redisTemplate.opsForValue().set()), nếu KHÔNG có Redis running → RedisConnectionFailureException. Auto-config chỉ setup, không guarantee connectivity. Đây là gotcha onboarding hay gặp.

🔨 Thử Phá Vỡ: Auto-config có lỗ hổng nào?

  • Surprise bean. Add dependency vô tình → auto-config thêm bean conflict với code của bạn. Spring 5+ default spring.main.allow-bean-definition-overriding=false → fail-fast khi conflict.
  • Hidden config. Junior không biết bean từ đâu ra — debug 30 phút. Fix: tự kiểm tra /actuator/beans hoặc /actuator/conditions.
  • Order issue. Auto-config chạy theo thứ tự không xác định. Custom auto-config phải explicit @AutoConfigureAfter / @AutoConfigureBefore nếu cần.
  • Test slowness. Full @SpringBootTest load tất cả auto-config → 10–30 giây. Fix: dùng slice annotations (@WebMvcTest, @DataJpaTest) → chỉ load relevant auto-config.
  • Production vs Dev divergence. Auto-config dùng default localhost cho Redis, RabbitMQ → dev OK, production cần config explicit → quên = bug ở môi trường thật.

4. Property Sources — 18 nguồn, thứ tự ưu tiên CORE

Vì sao CORE: 90% bug "config không áp dụng" / "secret không inject" đều từ hiểu sai property source priority. Trong K8s deployment, biết đúng thứ tự = debug được 5 phút thay vì 5 giờ.

Bảng 18 sources — thứ tự ưu tiên (cao → thấp)

Cao nhất (override tất cả):
   1. Devtools global settings
   2. @TestPropertySource (trong test)
   3. @SpringBootTest properties
   4. Command line arguments (--server.port=9090)  ← debug
   5. SPRING_APPLICATION_JSON (inline JSON trong env var)
   6. ServletConfig / ServletContext init params

Thường dùng nhất trong production:
   7.  JNDI (java:comp/env)
   8.  Java System properties (-Dserver.port=9090)
   9.  OS Environment variables (DB_USERNAME)         ← K8s Secret inject vào đây
  10.  application-{profile}.properties (bên ngoài jar)
  11.  application.properties (bên ngoài jar)
  12.  @PropertySource
  13.  application-{profile}.properties (trong jar)   ← Config mặc định
  14.  application.properties (trong jar)

Thấp nhất:
  15-18. Default values

K8s deployment — vì sao env var thắng application.yml?

# K8s Secret inject vào container như env var
env:
- name: SPRING_DATASOURCE_PASSWORD   # Spring Boot convert: . → _ , uppercase
  valueFrom:
    secretKeyRef:
      name: db-secret
      key: password

# → OS Environment Variable (source #9)
# → Override spring.datasource.password trong application.yml (source #13-14)
# → Production password KHÔNG được commit vào git, mà inject từ K8s Secret

Relaxed Binding — tại sao DB_USERNAME map được vào spring.datasource.username

Spring Boot auto-convert giữa 4 format đại diện cho cùng 1 property:

spring.datasource.username       ← dot-separated lowercase (canonical)
SPRING_DATASOURCE_USERNAME       ← env var (uppercase + underscore)
spring.datasource.user-name      ← kebab-case
springDatasourceUsername         ← camelCase

Tất cả đều bind vào CÙNG 1 property. Quy tắc convert env var:
  • Uppercase
  • Dot (.)         → Underscore (_)
  • Dash (-)        → Underscore (_)
  • Index [0]       → _0_

3 trade-off khi chọn property source

SourceKhi nào dùngTrade-off
application.yml (trong jar)Default config, safe values, dev localCommit vào git → KHÔNG được chứa secret
application-{profile}.yml (ngoài jar)Per-environment overrideCần CI/CD copy file vào container — phức tạp hơn env var
OS Environment variableSecret, K8s ConfigMap/Secret injectMọi process trên container thấy → memory dump có thể leak

Banking-grade rule: KHÔNG commit secret vào git ở bất kỳ source nào (application.yml, .properties, .env). Secret luôn inject runtime: K8s Secret → env var (đơn giản) hoặc Vault → Spring Cloud Vault (banking standard).

5. @ConfigurationProperties vs @Value CORE

2 cách bind property vào code — chọn sai khiến config khó test, khó validate, khó refactor.

⚠️ @Value (giới hạn dùng)

@Service
public class FeeCalc {
    @Value("${fee.rate}")
    private BigDecimal rate;

    @Value("${fee.min:1000}")
    private BigDecimal min;

    @Value("${fee.max:500000}")
    private BigDecimal max;
}

// Cons:
// - Scatter khắp code
// - Không validate (kiểu, range)
// - Khó test (cần Spring context
//   hoặc reflection)
// - Default value che giấu
//   missing config

✅ @ConfigurationProperties

@ConfigurationProperties("fee")
@Validated
@Getter @Setter
public class FeeProperties {
    @NotNull @DecimalMin("0.0")
    private BigDecimal rate;

    @NotNull @Positive
    private BigDecimal min = new BigDecimal("1000");

    @NotNull @Positive
    private BigDecimal max = new BigDecimal("500000");
}

// Pros:
// - Centralized config class
// - JSR-303 validation
// - IDE autocomplete + rename
// - Testable: new FeeProperties()
// - Hierarchical nested config

Bảng quyết định

Tình huốngGiải phápLý do
1 property đơn giản, không validation@ValueOverhead thấp, đủ cho simple case
Nhiều properties liên quan@ConfigurationPropertiesType-safe, validate, IDE support
Config thay đổi theo môi trườngProfile + yamlRõ ràng, không cần if/else trong code

⚠️ 5 edge cases khi dùng @ConfigurationProperties

  1. Phải có setter (hoặc constructor binding). Field private + không setter = không bind được. Lombok @Setter hoặc Java 17 record + @ConstructorBinding.
  2. Phải register class. Cách 1: @EnableConfigurationProperties(FeeProperties.class) ở config class. Cách 2: @ConfigurationPropertiesScan ở main app. Cách 3: @Component trên POJO (không khuyến nghị — mix concern).
  3. Validation chỉ kích hoạt với @Validated. Quên annotation này = JSR-303 silent ignored.
  4. Nested config phải new. private CoreBanking coreBanking = new CoreBanking(); — nếu không init, nested field null khi bind.
  5. List / Map bind từ env var phức tạp. Indexed binding qua underscore (BANKING_ALLOWED_IPS_0_=10.0.0.1).

6. Actuator — Production-ready features out-of-box CORE

Custom HealthIndicator

@Component("coreBankingHealth")
public class CoreBankingHealthIndicator implements HealthIndicator {

    private final CoreBankingClient client;

    public CoreBankingHealthIndicator(CoreBankingClient client) {
        this.client = client;
    }

    @Override
    public Health health() {
        try {
            PingResponse response = client.ping();
            if (response.isOk()) {
                return Health.up()
                        .withDetail("version", response.getVersion())
                        .withDetail("latencyMs", response.getLatencyMs())
                        .build();
            } else {
                return Health.down().withDetail("reason", response.getError()).build();
            }
        } catch (Exception e) {
            return Health.down().withDetail("error", e.getMessage()).withException(e).build();
        }
    }
}

Liveness vs Readiness — phân biệt

ProbeCâu hỏiKhi fail
Liveness (/actuator/health/liveness)Process còn alive không?K8s restart pod
Readiness (/actuator/health/readiness)Có sẵn sàng nhận traffic?K8s remove pod khỏi LB (không restart)

Quy tắc: Liveness chỉ check process; Readiness check dependencies (DB, Kafka). Sai = K8s restart loop khi DB down tạm thời.
