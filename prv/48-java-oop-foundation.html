#48 · Junior Foundation

Java OOP Foundation — Vì sao senior reject PR junior

Junior viết "Service" với public static method khắp nơi, gọi nhau chằng chịt. Code chạy được nhưng senior reject vì "đây là procedural code đội lốt OOP". File này biến từ biết syntax class/interface → hiểu why của OOP + SOLID + 5 anti-patterns junior hay vướng.

CORE ★ 1 tuần đọc + refactor 1 project thật Domain: E-commerce

⚠️ Cảnh báo "hiểu giả"

  • Biết "4 trụ cột OOP" — quote được Encapsulation/Inheritance/Polymorphism/Abstraction nhưng code service vẫn full static
  • Hiểu interface — luôn dùng abstract class khi 1 implementation, lý do "để dễ extend sau"
  • Biết SOLID — Single Responsibility = "class chỉ làm 1 thứ" (sai — nên là "class chỉ có 1 lý do để thay đổi")
  • Override equals mà quên hashCode → object "biến mất" khỏi HashMap
🚨 3 cấp — bạn đang ở đâu?

L1 (Recognize): viết được class + extends + implements. Code chạy nhưng class 500 dòng, static khắp nơi.
L2 (Explain): chọn abstract vs interface theo case. Apply SRP + DIP. Nhận ra anti-pattern khi review PR.
L3 (Apply): design domain model rich (không anemic). Refactor procedural → OOP có tactical reason. SOLID là phản xạ, không lookup.

1. Bài toán thực tế — Procedural Code Đội Lốt OOP CORE ★

📖 Tình huống — Junior viết "OrderService" cho E-commerce, senior reject PR
// ❌ Code junior — chạy được nhưng senior reject ngay
public class OrderService {
    public static double TAX_RATE = 0.1;
    public static List<Order> orders = new ArrayList<>();

    public static void createOrder(String customerId, List<Item> items) {
        double total = 0;
        for (Item i : items) total += i.price * i.qty;
        total = total + total * TAX_RATE;

        Order o = new Order();
        o.id = UUID.randomUUID().toString();
        o.customerId = customerId;
        o.items = items;
        o.total = total;
        o.status = "PENDING";
        orders.add(o);
    }

    public static void payOrder(String orderId) {
        for (Order o : orders) {
            if (o.id.equals(orderId)) {
                o.status = "PAID";
                return;
            }
        }
    }
}

public class Order {
    public String id;
    public String customerId;
    public List<Item> items;
    public double total;
    public String status;
}

5 vấn đề senior thấy ngay:

  1. Static method + static state = procedural. Không inject được, không test được, không thread-safe.
  2. Public field = no encapsulation. Bất kỳ code nào cũng set o.total = -100 mà object không biết.
  3. Anemic domain model: Order chỉ có data, mọi logic ngoài service. Đáng lẽ order.pay() mới là OOP.
  4. String status thay vì enum → typo "PEDNING" không catch lúc compile.
  5. Magic number 0.1 tax rate → không config được, không multi-region.

Refactor đúng OOP:

public enum OrderStatus { PENDING, PAID, SHIPPED, CANCELLED }

public class Order {
    private final String id;
    private final String customerId;
    private final List<OrderItem> items;
    private Money total;
    private OrderStatus status;

    // Constructor enforce invariant (không tạo Order với items rỗng)
    public Order(String customerId, List<OrderItem> items, TaxPolicy taxPolicy) {
        if (items == null || items.isEmpty()) {
            throw new IllegalArgumentException("Order must have items");
        }
        this.id = UUID.randomUUID().toString();
        this.customerId = customerId;
        this.items = List.copyOf(items);  // immutable
        this.total = taxPolicy.applyTo(calculateSubtotal());
        this.status = OrderStatus.PENDING;
    }

    // Behavior thuộc về Order, không phải Service
    public void pay() {
        if (status != OrderStatus.PENDING) {
            throw new IllegalStateException("Cannot pay order in status " + status);
        }
        this.status = OrderStatus.PAID;
    }

    public void cancel() {
        if (status == OrderStatus.SHIPPED) {
            throw new IllegalStateException("Cannot cancel shipped order");
        }
        this.status = OrderStatus.CANCELLED;
    }

    private Money calculateSubtotal() {
        return items.stream()
            .map(OrderItem::lineTotal)
            .reduce(Money.ZERO, Money::add);
    }

    // Getters only — no setters (immutable where possible)
    public String getId() { return id; }
    public OrderStatus getStatus() { return status; }
    public Money getTotal() { return total; }
}

@Service
public class OrderService {
    private final OrderRepository orderRepo;
    private final TaxPolicy taxPolicy;

    public OrderService(OrderRepository orderRepo, TaxPolicy taxPolicy) {
        this.orderRepo = orderRepo;
        this.taxPolicy = taxPolicy;
    }

    public Order create(String customerId, List<OrderItem> items) {
        Order order = new Order(customerId, items, taxPolicy);
        return orderRepo.save(order);
    }
}

Khác biệt: Enum thay String. Field private + immutable. Logic pay/cancel trong domain. TaxPolicy inject (testable). Constructor enforce invariant. Service mỏng — chỉ orchestrate.

2. 4 Trụ Cột OOP — Hiểu Why CORE ★

Trụ cột #1 — Encapsulation: "Class là con thú có boundary"

Why: object phải kiểm soát state của chính nó. Bên ngoài muốn đổi state → phải đi qua method, để class verify + maintain invariant.

// ❌ No encapsulation
public class Account {
    public double balance;     // ai cũng có thể set âm
}
acc.balance = -1000;           // OK syntax, sai semantic

// ✅ Encapsulation
public class Account {
    private double balance;

    public void withdraw(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("amount must be positive");
        if (amount > balance) throw new InsufficientFundsException(balance, amount);
        balance -= amount;
    }
}

Rule: field private + getter selective + behavior method. Setter là code smell — thường có nghĩa "tôi cần đổi state mà không có rule" = no encapsulation.

Trụ cột #2 — Inheritance: "is-a", KHÔNG "has code-a"

Why: tái sử dụng behavior. Nhưng chỉ dùng khi quan hệ thật là "is-a".

// ✅ "is-a": PremiumCustomer IS A Customer
public class Customer { ... }
public class PremiumCustomer extends Customer {
    public double discount() { return 0.15; }
}

// ❌ "has code-a": Order EXTENDS HashMap để dùng code Map
public class Order extends HashMap<String, Object> { ... }   // SAI
// Order KHÔNG phải là một Map. Order CHỨA fields.
// Effective Java: "Favor composition over inheritance"
❌ Khi nào KHÔNG dùng inheritance
  • Chỉ để tái sử dụng code → dùng composition (helper class) hoặc utility
  • Hierarchy > 3 cấp — fragile, đổi parent break tất cả con
  • Subclass override gần hết method parent → wrong abstraction
  • Subclass thêm precondition / yếu postcondition → vi phạm Liskov (xem SOLID)

Trụ cột #3 — Polymorphism: "Cùng API, khác behavior"

// 1 interface, nhiều implementation
public interface ShippingCalculator {
    Money calculate(Order order);
}

public class StandardShipping implements ShippingCalculator {
    public Money calculate(Order order) { return Money.of(30_000); }
}
public class ExpressShipping implements ShippingCalculator {
    public Money calculate(Order order) { return order.weight().multiply(50_000); }
}
public class PickupAtStore implements ShippingCalculator {
    public Money calculate(Order order) { return Money.ZERO; }
}

// Code gọi không cần biết loại nào
Money fee = shippingCalc.calculate(order);   // polymorphism

Why: code gọi không cần thay đổi khi thêm SameDayShipping. Đây là Open-Closed Principle (xem SOLID).

Trụ cột #4 — Abstraction: "Hide complexity, expose intent"

// ❌ Lộ implementation detail
public class OrderRepository {
    public void save(Order o, Connection conn, boolean autoCommit, int isolationLevel) { ... }
}
// Caller phải biết Connection, transaction internals

// ✅ Abstract level intent
public interface OrderRepository {
    Order save(Order order);
    Optional<Order> findById(String id);
    List<Order> findByCustomer(String customerId);
}
// Caller chỉ care "save order" — không care JDBC vs JPA vs MongoDB

Rule: API public expose what, hide how. Đổi implementation (JDBC → JPA) → caller không đổi.

3. Concept Junior Phải Master CORE ★

3.1 this / super / static / final

public class PremiumCustomer extends Customer {
    private final int loyaltyPoints;     // final = immutable sau constructor

    public PremiumCustomer(String name, int points) {
        super(name);                      // gọi constructor parent
        this.loyaltyPoints = points;      // this để phân biệt với param
    }

    @Override
    public Money discount() {
        Money base = super.discount();    // gọi method parent
        return base.add(Money.of(50_000));
    }

    public static PremiumCustomer fromOrder(Order o) {   // static factory
        return new PremiumCustomer(o.getCustomer().getName(), o.loyaltyPoints());
    }
}
  • this — reference object hiện tại
  • super — reference parent class (gọi method/constructor)
  • static — thuộc class, không thuộc instance
  • final: final field (immutable), final method (không override được), final class (không extends được)

3.2 Abstract Class vs Interface — Khi nào dùng cái nào

Tiêu chíAbstract ClassInterface
Multiple inheritance❌ chỉ 1 parent✅ implement nhiều

Modern rule: default interface. Chỉ abstract class khi cần share state/constructor logic giữa nhiều subclass.

3.3 Constructor & Initialization Order

// Thứ tự khi new ChildClass():
// 1. Static field + static block của Parent (1 lần)
// 2. Static field + static block của Child (1 lần)
// 3. Instance field + instance block của Parent
// 4. Constructor Parent
// 5. Instance field + instance block của Child
// 6. Constructor Child

Bẫy: gọi method overridable trong constructor parent → version Child chạy khi Child field chưa init → NPE.

3.4 Access Modifier — 4 Cấp

Rule: default private. Mở rộng khi cần. Public method = API contract, mỗi public method = lời hứa với caller.

3.5 Inner Class vs Static Nested Class

public class Outer {
    private int x;

    // Inner class (non-static): có reference ngầm tới Outer instance
    class Inner {
        int getX() { return x; }   // truy cập field Outer
    }

    // Static nested: KHÔNG có reference Outer → ít memory + dễ test
    static class Helper {
        // KHÔNG access x
    }
}

Default static nested. Inner class gây memory leak (giữ Outer alive) — chỉ dùng khi thật sự cần.

4. SOLID Cho Junior CORE ★

S — Single Responsibility Principle

Sai diễn giải: "class chỉ làm 1 thứ". Đúng: "class chỉ có 1 lý do để thay đổi" (Robert Martin).

// ❌ 3 responsibility: business + email + log
public class OrderService {
    public void create(Order o) {
        validate(o);
        save(o);
        sendEmail(o);           // ← thay đổi khi email template đổi
        writeAuditLog(o);       // ← thay đổi khi audit format đổi
    }
}

// ✅ Tách
public class OrderService {
    private final OrderValidator validator;
    private final OrderRepository repo;
    private final OrderEvents events;     // publish event, listener handle email + log

    public void create(Order o) {
        validator.validate(o);
        repo.save(o);
        events.publish(new OrderCreated(o));
    }
}

O — Open-Closed Principle

Class open for extension, closed for modification. Thêm tính năng = thêm class mới, không sửa class cũ.

// ❌ Mỗi loại discount = sửa method
public class DiscountCalculator {
    public Money calculate(Order o, String type) {
        if (type.equals("PERCENT")) return ...;
        else if (type.equals("FLAT")) return ...;
        else if (type.equals("BOGO")) return ...;   // ← sửa class mỗi lần thêm
    }
}

// ✅ Strategy pattern
public interface DiscountStrategy {
    Money apply(Order order);
}
// Thêm BOGO = tạo class BogoDiscount implements DiscountStrategy, không sửa code cũ

L — Liskov Substitution Principle

Subclass phải thay được parent mà không break behavior. Vi phạm = "Penguin extends Bird" rồi override fly() throw exception → mọi code dùng Bird.fly() đột nhiên fail.

Rule: subclass không thêm precondition, không yếu postcondition. Nếu phải vi phạm → abstraction sai từ đầu.

I — Interface Segregation Principle

"Caller không bị bắt depend method họ không dùng". Tránh "fat interface".

// ❌ Fat interface
interface Worker { void work(); void eat(); void sleep(); }
class Robot implements Worker {
    public void work() { ... }
    public void eat() { throw new UnsupportedOperationException(); }   // ← code smell
    public void sleep() { throw new UnsupportedOperationException(); }
}

// ✅ Tách
interface Workable { void work(); }
interface Eatable { void eat(); }
interface Sleepable { void sleep(); }
class Human implements Workable, Eatable, Sleepable { ... }
class Robot implements Workable { ... }

D — Dependency Inversion Principle

High-level depend abstraction, không depend low-level concrete. Đây là nền tảng của DI (Spring).

// ❌ High-level (OrderService) depend low-level (MySQLOrderRepo)
public class OrderService {
    private MySQLOrderRepo repo = new MySQLOrderRepo();   // hard-coded
}

// ✅ Depend interface, inject implementation
public class OrderService {
    private final OrderRepository repo;     // abstraction
    public OrderService(OrderRepository repo) { this.repo = repo; }
}
// Test dùng InMemoryOrderRepo, prod dùng JpaOrderRepo

5. 5 Anti-Patterns Phổ Biến CORE ★

❌ #1 — God Class (1000+ dòng, 30+ method)

Symptom: UtilService, BusinessManager, OrderService 2000 dòng. Mọi developer touch đều conflict merge.

Fix: tách theo SRP. OrderCreator, OrderValidator, OrderPricer, OrderShipper.

❌ #2 — Anemic Domain Model
// ❌ Domain object chỉ là DTO, mọi logic ngoài
public class Order { public String status; public double total; /* getters/setters */ }

public class OrderService {
    public void pay(Order o) {
        if (o.status.equals("PENDING")) o.status = "PAID";   // logic Order ở Service
    }
}

// ✅ Rich domain model
public class Order {
    private OrderStatus status;
    public void pay() {
        if (status != OrderStatus.PENDING) throw new IllegalStateException(...);
        this.status = OrderStatus.PAID;
    }
}

Test phân biệt: nếu xóa hết Service và đặt logic vào domain, business rule còn enforce không? Anemic = không. Rich = có.

❌ #3 — Constructor 10 tham số
// ❌
new Order(id, customerId, items, total, tax, discount, shipping, currency, ...);

// ✅ Builder
Order order = Order.builder()
    .customerId("CUST_1")
    .addItem(item1)
    .discount(percentDiscount(10))
    .shipping(standardShipping())
    .build();

// ✅ Hoặc parameter object
new Order(orderRequest, taxPolicy);
❌ #4 — Cyclic Dependency

Symptom: A → B → C → A. Spring throw BeanCurrentlyInCreationException. Đổi A break B và C.

Fix:

  • Tách interface chung lên parent module
  • Dùng event-driven (A publish event, C subscribe — không direct call)
  • Lazy injection (@Lazy Spring) — band-aid, không fix root
❌ #5 — Thiếu equals/hashCode khi làm Map key

Override equals mà quên hashCode (hoặc ngược lại) → object "biến mất" khỏi HashMap/HashSet vì 2 object equal có hashCode khác → vào 2 bucket khác.

// ✅ Java 16+ record auto-generate equals + hashCode đúng contract
public record OrderKey(String id, int version) { }

// ✅ Hoặc IDE generate (IntelliJ Cmd+N → equals and hashCode)
// ❌ Tự viết — dễ quên 1 field, dễ sai contract

🎯 3 Cấp Độ Mastery CORE ★

4 Trụ cột OOP

L1 — Recognize

Quote được tên 4 trụ cột. Code class có private field + getter/setter. Vẫn dùng static method khắp nơi.

Symptom: service class 500 dòng, public field, no constructor injection.

L2 — Explain

Hiểu why của encapsulation (invariant + boundary). Phân biệt "is-a" vs "has code-a". Dùng interface + polymorphism cho varying behavior.

L3 — Apply

Design rich domain model (behavior thuộc domain, không Service). Refactor anemic → rich. Pull request bao gồm test cho invariant.

SOLID

L1

Nhớ tên 5 chữ. Hiểu mơ hồ. Code review không catch được vi phạm SOLID.

L2

SRP = "1 lý do thay đổi". OCP = strategy pattern. DIP = constructor injection. Catch được vi phạm trong PR đồng nghiệp.

L3

SOLID là phản xạ. Thấy "if/else type" → reach Strategy. Thấy 3 class share state → tách interface. Refactor legacy code từ procedural → SOLID-compliant.

Abstract vs Interface choice

L1

Luôn dùng abstract class. Lý do "có thể thêm state sau".

L2

Default interface. Abstract class chỉ khi cần share state/constructor. Hiểu default method Java 8+.

L3

Design plugin interface với sealed interface (Java 17+). Cân nhắc Java record cho immutable data carrier.

🚨 Symptom → Debug Recipes CORE ★

🚨 Recipe #1 — "NPE trong constructor parent khi tạo subclass"
public class Parent {
    public Parent() { init(); }                   // gọi method overridable
    protected void init() { /* default */ }
}
public class Child extends Parent {
    private List<String> data = new ArrayList<>();
    @Override protected void init() {
        data.add("x");                            // ← NPE! data chưa init
    }
}
new Child();   // NPE

Root cause: init order — constructor Parent chạy trước instance field của Child. Override method chạy version Child khi field Child chưa khởi tạo.

Fix: KHÔNG gọi overridable method trong constructor. Hoặc final method, hoặc lazy init.

🚨 Recipe #2 — "Object biến mất khỏi HashSet"

Symptom: Add object vào Set, vài line sau contains trả false.

3 nguyên nhân:

  1. Quên override hashCode (đã override equals)
  2. Mutate field dùng trong hashCode sau khi add
  3. equals + hashCode dùng field khác nhau

Fix: dùng record (Java 16+) hoặc IDE generate. Field hashCode phải immutable.

🚨 Recipe #3 — "Spring BeanCurrentlyInCreationException"

Symptom: startup fail với "Requested bean is currently in creation: Is there an unresolvable circular reference?"

Root cause: Cyclic dependency. A → B → A (constructor injection).

Fix theo độ ưu tiên:

  1. Best: refactor — tách common logic vào C, A và B đều depend C
  2. Decent: chuyển sang event-driven (A publish event, B handle)
  3. Band-aid: @Lazy 1 bên — hide problem, không fix

🎤 Interview Framework CORE ★

Tier 1 — Surface

Q: Abstract class vs Interface — khi nào dùng?

Default interface (Java modern). Abstract class khi cần share state/constructor giữa nhiều subclass. Interface = "can-do" (capability), abstract = "is-a". Java 8+ interface có default method nên ranh giới mờ — chọn interface nếu chỉ define contract, abstract nếu cần partial implementation + state.

Tier 2 — Deep Dive

Q: Single Responsibility Principle — "1 thứ" hay "1 lý do"?

Đúng là "1 lý do để thay đổi" (Robert Martin). Ví dụ Employee với calculatePay() + save() + printReport() = 3 responsibility vì 3 stakeholder khác nhau (HR, IT, Finance) yêu cầu thay đổi. Tách theo audience.

Tier 3 — Architecture

Q: Phân biệt Anemic vs Rich Domain Model — kèm ví dụ

Anemic: domain object chỉ có getter/setter, mọi logic ở Service. Service trở thành "transaction script" — procedural.

Rich: behavior thuộc domain (order.pay(), account.withdraw()). Service chỉ orchestrate.

Test phân biệt: nếu xóa Service và dùng domain trực tiếp, business rule còn enforce không? Anemic = không (logic ở Service). Rich = có (logic trong domain).

Khi nào Anemic OK: DTO truyền giữa layer, project CRUD đơn giản. Project có business logic phức tạp → bắt buộc Rich (DDD).
