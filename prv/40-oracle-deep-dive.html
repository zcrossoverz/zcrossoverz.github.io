Oracle Deep Dive — SGA/PGA, UNDO/SCN, Tuning

Nhiều dev "dùng Oracle" nhưng không biết SGA khác PGA, vì sao có ORA-01555, hay ROWNUM khác ROW_NUMBER(). File này dạy kiến trúc Oracle đủ để tune query chậm và đọc AWR, không chỉ viết SQL chạy được — trên domain ShopFlow, một sàn e-commerce chạy Oracle cho hệ thống đơn hàng.

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Oracle với PostgreSQL gần như nhau, biết SQL là đủ."
Thực tế: MVCC khác cơ bản: Oracle update in-place + version cũ vào UNDO segment (→ ORA-01555), PostgreSQL giữ version trong table (→ cần VACUUM). REPEATABLE READ Oracle không có; SERIALIZABLE Oracle chỉ là Snapshot Isolation. Mang giả định PG sang Oracle = bug.
Tưởng hiểu: "ROWNUM dùng để phân trang, WHERE ROWNUM BETWEEN 11 AND 20 lấy trang 2."
Thực tế: ROWNUM gán sau khi lọc, trước ORDER BY → ROWNUM > 10 luôn rỗng (row đầu phải là 1). Phân trang đúng cần subquery + ROW_NUMBER() hoặc OFFSET FETCH (12c+).
Tưởng hiểu: "Query chậm thì thêm index là xong."
Thực tế: Phải đọc execution plan + AWR/ASH để biết nghẽn ở đâu (full scan, sai index, bind variable peeking, contention). Thêm index bừa có thể làm chậm INSERT/UPDATE và optimizer chọn sai.

1. Bài toán thực tế — query 8s giờ cao điểm CORE ★

ShopFlow chạy Oracle. Trang "đơn hàng của tôi" bình thường 200ms, giờ cao điểm vọt 8s. Dev thêm index nhưng không đỡ. Có lúc job report ném ORA-01555: snapshot too old. AWR cho thấy "buffer busy waits" và "log file sync" cao.

⏸ Pause & Predict — trước khi đọc lời giải
Viết ra giấy:
  1. ORA-01555 nghĩa là gì? Nó liên quan UNDO thế nào, và vì sao long-running report dễ gặp?
  2. "Thêm index không đỡ" — những nguyên nhân nào khác ngoài thiếu index khiến query chậm giờ peak?
  3. Để biết nghẽn thật, bạn đọc report nào của Oracle và nhìn chỉ số gì trước?
Đáp án: ORA-01555 = UNDO bị ghi đè trước khi report đọc xong snapshot (mục 3); chậm có thể do full scan/sai plan/contention/bind peeking (mục 7); đọc AWR/ASH nhìn top wait events + SQL by elapsed (mục 7). Đừng đọc tiếp khi chưa giải thích ORA-01555.

2. SGA + PGA CORE ★

  • SGA (System Global Area): chia sẻ giữa mọi session — buffer cache (data block), shared pool (SQL/plan), redo log buffer.
  • PGA (Program Global Area): riêng mỗi session — sort, hash join workarea. Sort lớn vượt PGA → tràn xuống temp tablespace (chậm).
Vì sao quan trọng

Query chậm có thể do PGA nhỏ (sort to disk) hoặc buffer cache miss (đọc disk). Biết SGA/PGA để đọc đúng chỉ số trong AWR.

3. MVCC: UNDO + SCN CORE ★

Oracle update in-place, ghi giá trị cũ vào UNDO segment. SCN (System Change Number) là đồng hồ logic: mỗi query đọc theo SCN của mình; nếu row bị đổi sau SCN đó → đọc lại version cũ từ UNDO (read consistency).

L1 — Biết tên

Biết "Oracle có MVCC, không lock đọc". Symptom hiểu giả: không giải thích được ORA-01555 hay vai trò UNDO/SCN.

L2 — Explain được

Read consistency: query thấy snapshot tại SCN bắt đầu; version cũ dựng từ UNDO. ORA-01555 = UNDO chứa version cũ bị ghi đè (UNDO_RETENTION ngắn / UNDO nhỏ) trước khi query dài đọc xong → "snapshot too old".

L3 — Apply được

Tune UNDO_RETENTION + kích thước UNDO tablespace; tránh report chạy quá dài giữ snapshot; tách OLTP/reporting. Đọc đúng error để biết chỉnh UNDO chứ không phải index.

4. Row lock qua ITL SUPPLEMENTARY

Oracle lock ở mức row, ghi thông tin lock vào ITL (Interested Transaction List) trong header của data block — không có lock manager riêng, không leo thang lock. "Writers don't block readers, readers don't block writers". Lock contention biểu hiện qua wait event (enq: TX).

5. Redo log & recovery SUPPLEMENTARY

Mọi thay đổi ghi redo (để recover) trước khi data file cập nhật (write-ahead). Commit = flush redo (đảm bảo durability). "log file sync" cao trong AWR = nghẽn ở việc ghi redo lúc commit (đĩa redo chậm / commit quá nhiều giao dịch nhỏ).

6. MERGE, analytic, pagination đúng CORE ★

-- Upsert chuẩn Oracle
MERGE INTO orders t USING (SELECT :id id, :total total FROM dual) s
ON (t.id = s.id)
WHEN MATCHED THEN UPDATE SET t.total = s.total
WHEN NOT MATCHED THEN INSERT (id, total) VALUES (s.id, s.total);

-- Pagination ĐÚNG (ROWNUM gán trước ORDER BY → phải bọc subquery)
SELECT * FROM (
  SELECT o.*, ROW_NUMBER() OVER (ORDER BY created_at DESC) rn FROM orders o
) WHERE rn BETWEEN 11 AND 20;     -- hoặc OFFSET 10 ROWS FETCH NEXT 10 ROWS ONLY (12c+)

7. Index, partition, AWR/ASH SUPPLEMENTARY

  • Index: B-tree (mặc định, high cardinality), bitmap (low cardinality, đọc nhiều ghi ít — KHÔNG cho OLTP vì bitmap lock rộng), composite (theo thứ tự cột trong WHERE).
  • Partition: chia table lớn theo range (ngày)/list/hash → partition pruning chỉ quét phần cần.
  • AWR/ASH workflow: AWR (snapshot định kỳ) → top wait events + SQL by elapsed time; ASH (active session history) → "lúc 14:00 session chờ gì". Tìm SQL tốn nhất → đọc plan → sửa.

⚠️ Anti-patterns CORE ★

🔧 Counter-example — khi "tối ưu" làm chậm hơn / sai
  • Bitmap index cho OLTP: bitmap lock cả nhóm row khi update → contention nặng. Bitmap chỉ cho data warehouse đọc nhiều.
  • Thêm index bừa: mỗi index làm chậm INSERT/UPDATE và có thể khiến optimizer chọn sai plan. Đo trước, index theo query thật.
  • PL/SQL SELECT INTO không bắt NO_DATA_FOUND/TOO_MANY_ROWS: lỗi runtime bất ngờ. Luôn handle exception.
  • Hardcode literal thay bind variable: mỗi giá trị tạo một SQL mới → library cache bùng nổ (hard parse). Dùng bind variable.
  • SELECT * trên table lớn: kéo cột thừa, phá index-only scan.

Quy tắc: mọi tối ưu dựa trên execution plan + AWR thực đo, không cảm tính. Oracle khác PostgreSQL ở nhiều giả định — đừng mang nguyên xi.

🎯 3-tier interview framework

Tầng 1 — SurfaceMid gate

Q: SGA và PGA khác gì?

SGA chia sẻ mọi session (buffer cache, shared pool, redo buffer). PGA riêng mỗi session (sort, hash workarea). Sort vượt PGA → temp tablespace (chậm). Biết để đọc đúng chỉ số AWR.

Tầng 2 — Deep DiveSenior gate

Q: Giải thích ORA-01555 và cách khắc phục.

Query dài đọc theo SCN cũ, cần version cũ từ UNDO; nhưng UNDO bị ghi đè (retention ngắn / UNDO nhỏ / nhiều DML) → "snapshot too old". Fix: tăng UNDO_RETENTION + UNDO tablespace, rút ngắn query, tách reporting khỏi OLTP.

Tầng 3 — ArchitectureSenior+

Q: Query đột ngột chậm giờ cao điểm, workflow chẩn đoán?

AWR top wait events + SQL by elapsed → ASH xem session lúc đó chờ gì → execution plan của SQL nghi vấn (full scan? sai index? bind peeking?) → kiểm tra contention (enq: TX, buffer busy) / log file sync. Sửa theo bằng chứng: index đúng, bind variable, tách reporting, partition. Không đoán.

🔧 Lab bắt buộc — đọc plan & reproduce (2 bài)

Lab 1 — Pagination sai. Viết SELECT ... WHERE ROWNUM > 10 → quan sát trả rỗng. Sửa bằng subquery ROW_NUMBER() hoặc OFFSET FETCH. Tiêu chí: giải thích ROWNUM gán trước ORDER BY nên row đầu phải là 1.

Lab 2 — Plan thay đổi vì thiếu index. Bảng orders lớn; chạy EXPLAIN PLAN cho query lọc theo customer_id chưa index → FULL TABLE SCAN. Thêm index → INDEX RANGE SCAN, đo thời gian. Sau đó thêm điều kiện làm index vô hiệu (function trên cột) → quan sát plan đổi lại. Tiêu chí: đọc được plan và biết vì sao index bị bỏ qua.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. SGA vs PGA khác gì? Sort lớn ảnh hưởng cái nào?
  2. ORA-01555 xảy ra thế nào? Liên quan UNDO/SCN ra sao?
  3. Vì sao WHERE ROWNUM > 10 luôn rỗng? Phân trang đúng làm sao?
  4. Bitmap index vì sao không dùng cho OLTP?
  5. Workflow đọc AWR/ASH để tìm query chậm là gì?
Bí câu nào = quay lại đúng section. Trôi cả 5 = tune được Oracle, không chỉ viết SQL.
