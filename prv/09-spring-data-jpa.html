Spring Data JPA & Hibernate — N+1, Lazy/Eager, Lock, Cache

Junior dùng JPA repository mỗi ngày. Khi dashboard load 30s, không biết "DB không có vấn đề gì" thực ra là 500 query. Khi 2 user cùng update balance → race condition. File này bóc N+1, dirty checking, lock strategy, L2 cache + 5 anti-pattern junior hay vướng.

CORE ★ 1 tuần đọc + fix N+1 production Domain: E-commerce

⚠️ Cảnh báo "hiểu giả"

  • Dùng @OneToMany mỗi ngày — không biết default EAGER fetch là bom hẹn giờ
  • Biết "lazy loading" — không nhận ra LazyInitializationException khi access ngoài transaction
  • Hiểu "JOIN FETCH" — không biết kết hợp với pagination = HHH000104 warning + in-memory pagination
  • Dùng save() — không hiểu Hibernate dirty checking tự auto-update entity khi commit (không cần gọi save lại)
🚨 3 cấp

L1: CRUD repository, default fetch. Endpoint chậm = "DB issue".
L2: Detect + fix N+1. Hiểu LAZY vs EAGER. Dùng @EntityGraph hoặc JOIN FETCH. Optimistic lock cho update.
L3: Tune L2 cache, design read model riêng cho dashboard, dùng projection DTO native. Hiểu Hibernate Session lifecycle qua transaction boundary.

1. Bài toán — Dashboard 30s, "DB không có vấn đề" CORE ★

📖 Tình huống — Order dashboard chậm
// Endpoint: list 100 order với customer + items
@GetMapping("/orders")
public List<OrderDto> list() {
    return orderRepo.findAll().stream()
        .map(o -> new OrderDto(
            o.getId(),
            o.getCustomer().getName(),     // ← lazy load customer
            o.getItems().stream()           // ← lazy load items
                .map(i -> i.getProduct().getName())   // ← lazy load product
                .toList()
        ))
        .toList();
}

Đo: 1 request = 30 giây. DB CPU 15%, app CPU 80%. "DB không có vấn đề".

Bật Hibernate SQL log:

spring:
  jpa:
    show-sql: true
    properties:
      hibernate.format_sql: true
SELECT * FROM orders                          ← 1 query
SELECT * FROM customers WHERE id = ?          ← 100 queries (1/order)
SELECT * FROM order_items WHERE order_id = ?  ← 100 queries
SELECT * FROM products WHERE id = ?           ← 500+ queries (5 item/order trung bình)
-------
TỔNG: 700+ queries cho 1 request

Root cause: N+1. Lazy load mỗi association = 1 query thêm. Network roundtrip 1ms × 700 = 700ms minimum. Với connection pool wait + JDBC overhead → 30s.

Fix (1 dòng):

@EntityGraph(attributePaths = {"customer", "items.product"})
List<Order> findAll();

1 query JOIN. Latency 30s → 200ms.

⏸ Pause & Predict
  1. Vì sao endpoint chậm 30s mà DB CPU chỉ 15%? App CPU làm gì?
  2. JOIN FETCH 3 association: customer + items + product. Risk gì với 100 order × 5 item × 1 product?
  3. Nếu thêm pagination .limit(20) vào endpoint — JOIN FETCH có work không?

2. N+1 Problem — Internals CORE ★

// Entity
@Entity
class Order {
    @Id Long id;
    @ManyToOne(fetch = LAZY) Customer customer;        // default LAZY cho @ManyToOne là EAGER!
    @OneToMany(mappedBy = "order") List<OrderItem> items;  // default LAZY
}

// Repo
orderRepo.findAll();    // 1 query: SELECT * FROM orders
// Khi access o.getCustomer().getName() → Hibernate fire query SELECT customer WHERE id=?
// 100 order = 100 query thêm
⚠️ Default fetch type — học thuộc
AnnotationDefault
@OneToManyLAZY ✅
@ManyToManyLAZY ✅
@ManyToOneEAGER ⚠️ (bom hẹn giờ)
@OneToOneEAGER ⚠️

Rule: luôn explicit fetch = LAZY cho mọi association. EAGER mặc định cho ManyToOne/OneToOne là 1 trong những default tệ nhất của JPA.

3. Hibernate Session + L1/L2 Cache CORE ★

Session = Hibernate's unit of work, tương đương EntityManager trong JPA. Lifecycle = transaction.

@Transactional method:
  ┌─ Session open
  │   ├─ L1 cache (per-session): entity loaded lưu key=id
  │   ├─ Dirty checking: track changes auto-flush at commit
  │   └─ Lazy load proxy: fire query khi access
  └─ Session close (commit)
         └─ Lazy proxy access SAU đây → LazyInitializationException

L1 Cache — Per Session

@Transactional
public void demo() {
    Order o1 = orderRepo.findById(1L).get();   // SELECT
    Order o2 = orderRepo.findById(1L).get();   // L1 HIT — không query
    assert o1 == o2;                            // same instance!
}

L2 Cache — Cross-Session

// Entity
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
@Entity class Country { ... }   // reference data, ít đổi

// Config
spring:
  jpa:
    properties:
      hibernate:
        cache:
          use_second_level_cache: true
          region.factory_class: org.hibernate.cache.jcache.JCacheRegionFactory

Khi dùng: reference data (country, currency, config) — đọc nhiều, update rất hiếm.

Khi KHÔNG dùng: data có write concurrent từ nhiều JVM (sẽ stale), data thay đổi thường xuyên (invalidation overhead).

4. Optimistic vs Pessimistic Lock CORE ★

Optimistic Lock — @Version

@Entity
class Account {
    @Id Long id;
    BigDecimal balance;
    @Version Long version;   // Hibernate tự manage
}

// Hibernate add WHERE version=? vào UPDATE
// UPDATE accounts SET balance=?, version=version+1 WHERE id=? AND version=?
// Nếu 0 row affected → OptimisticLockException (ai đó đã update trước)

Khi dùng: conflict hiếm (e.g., user edit profile). Read-heavy, write occasional. Không block DB.

Khi KHÔNG dùng: high contention (mọi user update cùng hot row). Conflict rate cao → retry storm.

Pessimistic Lock — SELECT FOR UPDATE

@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT a FROM Account a WHERE a.id = :id")
Account findByIdForUpdate(@Param("id") Long id);

// Generates: SELECT ... FOR UPDATE
// DB lock row, các tx khác wait đến khi commit/rollback

Khi dùng: high-value transaction (transfer tiền, inventory deduct). Conflict thường xuyên. Cần guarantee no lost update.

Bẫy: giữ lock lâu = block khác. Timeout deadlock cao.

🎯 Decision guide
  • User profile, settings → optimistic (low conflict)
  • Inventory deduction, payment, balance → pessimistic hoặc atomic SQL (UPDATE ... SET stock = stock - 1 WHERE stock >= 1)
  • Cart update → optimistic + retry 3 lần

5. Dirty Checking — Auto Update CORE ★

@Transactional
public void changeName(Long id, String newName) {
    Order o = orderRepo.findById(id).get();
    o.setNote(newName);
    // KHÔNG cần gọi orderRepo.save(o)
    // Hibernate compare snapshot lúc load vs lúc commit
    // Khác nhau → tự generate UPDATE
}

Cơ chế: khi entity load qua Session, Hibernate lưu snapshot. Cuối transaction, compare current vs snapshot, nếu khác → fire UPDATE chỉ field thay đổi.

⚠️ Bẫy
  • Detached entity (ngoài session) → dirty checking KHÔNG hoạt động. Phải merge() để attach lại
  • Field collection (List, Set) — modify in place mới detect, replace reference mới không
  • Performance: dirty checking O(n) trên field count. Entity 50 field × 10K entity loaded = chậm

6. 4 Strategies Fix N+1 CORE ★

Strategy 1 — @EntityGraph (đơn giản nhất)

@EntityGraph(attributePaths = {"customer", "items.product"})
List<Order> findByStatus(OrderStatus status);
// 1 query LEFT JOIN

Strategy 2 — JOIN FETCH (JPQL)

@Query("SELECT o FROM Order o JOIN FETCH o.customer JOIN FETCH o.items i JOIN FETCH i.product")
List<Order> findAllWithDetails();

Strategy 3 — Batch fetching

@OneToMany(fetch = LAZY)
@BatchSize(size = 50)   // Hibernate gom 50 id, IN query 1 lần
List<OrderItem> items;
// 1 query orders + 1 query (IN 50 ids) thay vì N query

Strategy 4 — DTO projection (best cho list view)

interface OrderListView {
    Long getId();
    String getCustomerName();
    int getItemCount();
}

@Query("""
    SELECT o.id AS id,
           o.customer.name AS customerName,
           SIZE(o.items) AS itemCount
    FROM Order o
""")
List<OrderListView> findAllForList();
// Chỉ load fields cần, không load full entity → nhanh hơn JOIN FETCH
⚠️ Khi nào dùng cái nào
  • List view (dashboard) → DTO projection (chỉ load field cần)
  • Detail view (1 entity + đầy đủ relationship) → @EntityGraph hoặc JOIN FETCH
  • Multiple collection → @BatchSize (tránh Cartesian product)
  • Pagination → KHÔNG dùng JOIN FETCH (in-memory pagination). Dùng @EntityGraph trên query đã paginate.

7. 5 Anti-Patterns CORE ★

❌ #1 — EAGER mặc định trên @OneToMany
// ❌ Mỗi lần load Customer = load all orders
@OneToMany(fetch = EAGER) List<Order> orders;

// ✅ LAZY + fetch theo nhu cầu
@OneToMany(fetch = LAZY) List<Order> orders;
❌ #2 — JOIN FETCH + pagination
// ❌ Warning HHH000104, in-memory pagination
@Query("SELECT o FROM Order o JOIN FETCH o.items")
List<Order> findAll(Pageable p);   // load ALL rồi paginate in JVM = OOM

Fix: Query 2 phase — query 1 paginate IDs, query 2 fetch với JOIN FETCH theo IDs đó.

❌ #3 — LazyInitializationException
// ❌ Controller access ngoài @Transactional
@GetMapping("/orders/{id}")
public Order get(Long id) {
    Order o = orderService.findById(id);    // tx kết thúc
    o.getItems().size();                     // LazyInitializationException
    return o;
}

// ✅ DTO mapping trong service (trong tx)
@GetMapping("/orders/{id}")
public OrderDto get(Long id) {
    return orderService.findById(id);   // service trả DTO đã map
}

Anti-fix tệ: spring.jpa.open-in-view: true — extend tx tới controller. Tiện nhưng giấu bug + lock connection cả request lifetime.

❌ #4 — Cartesian product (JOIN FETCH nhiều collection)
// ❌ JOIN FETCH 2 collection cùng level
@Query("SELECT o FROM Order o JOIN FETCH o.items JOIN FETCH o.payments")
List<Order> findAll();
// Order có 5 item + 3 payment → 1 order = 15 row trong result
// 100 order = 1500 row → memory + parse chậm

Fix: JOIN FETCH 1 collection + @BatchSize cho cái còn lại. Hoặc 2 query riêng.

❌ #5 — Không pessimistic lock cho high-value transaction
// ❌ Race condition: 2 user cùng update inventory
Inventory inv = repo.findById(productId).get();
inv.setStock(inv.getStock() - quantity);
repo.save(inv);
// 2 tx cùng read stock=10, cùng set stock=8 → lost update

// ✅ Atomic SQL hoặc pessimistic lock
@Modifying
@Query("UPDATE Inventory i SET i.stock = i.stock - :qty WHERE i.id = :id AND i.stock >= :qty")
int decreaseStock(@Param("id") Long id, @Param("qty") int qty);
// Return 0 nếu không đủ stock → throw business exception

🎯 3 Cấp Mastery CORE ★

N+1 detection + fix

L1

Repository CRUD. Default fetch. "Slow endpoint = DB issue".

L2

Bật SQL log → đếm query. Dùng @EntityGraph hoặc JOIN FETCH. Hiểu bẫy pagination + Cartesian.

L3

DTO projection cho list view. @BatchSize cho many collection. Build read model riêng (CQRS) cho dashboard phức tạp.

Lock strategy

L1

Không biết lock. Update kiểu read-modify-write naive.

L2

@Version cho optimistic. Pessimistic cho high-contention. Atomic SQL khi possible.

L3

Hybrid: optimistic + retry 3 lần, fallback pessimistic. Monitor deadlock rate. Design partition để giảm contention.

🚨 Symptom → Debug Recipes CORE ★

🚨 Recipe #1 — "Endpoint chậm dù DB không tải"
spring.jpa.show-sql: true
spring.jpa.properties.hibernate.format_sql: true
logging.level.org.hibernate.SQL: DEBUG
# Reproduce → đếm query/request. > 5 query = nghi N+1
🚨 Recipe #2 — "LazyInitializationException trong production"

Access lazy field ngoài transaction. Fix: map sang DTO trong service. Không dùng open-in-view: true.

🚨 Recipe #3 — "OptimisticLockException rate cao"

Symptom: log show OptimisticLockException 30%+ của update.

Fix:

  1. Conflict thực sự high → switch sang pessimistic lock hoặc atomic SQL
  2. Hoặc retry 3 lần với exponential backoff (Spring Retry)
  3. Hoặc redesign: nếu nhiều user update cùng hot row → partition (split aggregate)
🚨 Recipe #4 — "Deadlock detected"

2 tx lock 2 row theo thứ tự ngược. Fix: luôn lock theo thứ tự cố định (ORDER BY id). Hoặc SELECT FOR UPDATE NOWAIT + retry.

🎤 Interview Framework CORE ★

Q: N+1 problem là gì?

1 query parent + N query con cho lazy load association. Vd: load 100 order, mỗi order access customer → 1 + 100 = 101 query. Fix: @EntityGraph, JOIN FETCH, @BatchSize, DTO projection.

Q: Optimistic vs Pessimistic lock — khi nào dùng?

Optimistic (@Version) — conflict hiếm, read-heavy. Không block DB. Pessimistic (SELECT FOR UPDATE) — high contention, cần guarantee no lost update. Block DB, deadlock risk. Cart update = optimistic + retry. Inventory deduct = atomic SQL hoặc pessimistic.

Q: Design read model cho dashboard analytics e-commerce

  1. Read vs write model tách (CQRS lite): repo read trả DTO, không entity
  2. DTO projection cho list view — chỉ field cần
  3. Materialized view hoặc denormalized table cho aggregate (revenue, top customer)
  4. L2 cache cho reference data ít đổi
  5. Read replica route dashboard query, master cho write
  6. JPA only cho transactional write. Dashboard query dùng jOOQ / native SQL — kiểm soát plan
  7. Monitor: slow query log + pg_stat_statements alert top SQL by time
