Cẩm nang ôn phỏng vấn Backend Java

Tài liệu này giải thích từng chủ đề theo cách dành cho người mới: bắt đầu từ khái niệm, ví dụ đơn giản, cách dùng trong dự án thật, lỗi thường gặp và cách trả lời khi phỏng vấn.

0. Cách dùng tài liệu này mà không bị ngợp

Bạn không cần biết toàn bộ mọi thứ ở mức chuyên gia. Với khoảng 3 năm kinh nghiệm, mục tiêu thực tế là: hiểu bản chất phần mình ghi trong CV, giải thích được bằng lời của mình và kể được một ví dụ thực tế.

Mức 1 — Biết khái niệm

Giải thích được “nó là gì” và “dùng để làm gì”. Ví dụ: Redis là kho dữ liệu in-memory, thường dùng làm cache.

Mức 2 — Biết áp dụng

Nói được cách dùng trong Spring Boot hoặc dự án. Ví dụ: dùng cache-aside với TTL để giảm query DB.

Mức 3 — Biết trade-off

Nói được khi nào không nên dùng, lỗi nào có thể xảy ra và cách xử lý. Đây là mức khiến câu trả lời có chiều sâu.

Công thức học mỗi chủ đề: Định nghĩa → ví dụ → dùng ở đâu trong dự án → lỗi từng gặp → cách sửa → một câu trả lời phỏng vấn 30–60 giây.

Ba nguyên tắc quan trọng

  1. Không nói đã làm production nếu thực tế chỉ mới đọc hoặc làm thử.
  2. Không cố thuộc lòng định nghĩa sách giáo khoa; hãy diễn giải bằng ví dụ nghiệp vụ.
  3. Với những phần rất rộng như virtual threads, microservices hay Kubernetes, chỉ cần chắc phần cốt lõi và nói đúng phạm vi kinh nghiệm.

2. Giới thiệu dự án — phần quan trọng nhất

Nhiều cuộc phỏng vấn không bắt đầu bằng thuật toán mà bắt đầu bằng câu: “Hãy giới thiệu dự án gần nhất và phần bạn phụ trách.” Câu trả lời này quyết định người phỏng vấn sẽ đào sâu vào đâu.

Cấu trúc trả lời 6 phần

  1. Bối cảnh: Dự án phục vụ ai, giải quyết vấn đề gì.
  2. Kiến trúc: Monolith hay microservice, có những hệ thống nào giao tiếp.
  3. Tech stack: Java, Spring Boot, Oracle, Redis, Docker, K8S...
  4. Vai trò cá nhân: Bạn trực tiếp làm module/API nào.
  5. Vấn đề khó: Một lỗi hoặc bài toán cụ thể.
  6. Kết quả: Tối ưu bao nhiêu, giảm lỗi gì, cải thiện quy trình ra sao.
Mẫu nói: “Dự án của em là hệ thống ACH Gateway kết nối core banking với NAPAS. Em phụ trách backend Java Spring Boot, chủ yếu xử lý API giao dịch, trạng thái message và đối soát. Hệ thống sử dụng Oracle, Redis và triển khai trên OpenShift. Một bài toán em từng xử lý là tránh ghi nhận giao dịch trùng khi bên ngoài retry. Em kết hợp business key duy nhất ở database, kiểm tra idempotency và log correlation ID để truy vết.”

Chuẩn bị ba câu chuyện theo STAR

Tối ưu hiệu năng

S: API chậm.
T: Giảm latency.
A: Xem execution plan, sửa query/index, giảm dữ liệu load.
R: Nêu kết quả đo được.

Lỗi production

S: Giao dịch lỗi/tồn đọng.
T: Tìm root cause.
A: Dùng log, trace ID, SQL, thread dump.
R: Khôi phục và ngăn tái diễn.

Yêu cầu gấp

S: Deadline ngắn.
T: Đảm bảo đúng nghiệp vụ.
A: Chia task, test case, review rủi ro.
R: Giao đúng hạn, không phát sinh lỗi nghiêm trọng.

Những con số nên nhớ

Không cần bịa số. Chỉ cần nhớ dữ liệu thật hoặc khoảng tương đối: số lượng service, số API bạn làm, số bản ghi/ngày, thời gian response trước và sau, số người trong team, tần suất deploy.

3. Java Core từ cơ bản

3.1 Class và Object

Class là bản thiết kế; object là một thực thể được tạo từ bản thiết kế đó.

public class BankAccount {
    private String accountNo;
    private BigDecimal balance;

    public void deposit(BigDecimal amount) {
        balance = balance.add(amount);
    }
}

BankAccount account = new BankAccount(); // object

3.4 Overload và Override

  • Overload: cùng tên method nhưng khác danh sách tham số; quyết định ở compile time.
  • Override: class con viết lại method của class cha/interface; quyết định ở runtime.

3.5 equals() và hashCode()

HashMap và HashSet dùng hashCode() để chọn bucket, sau đó dùng equals() để so sánh chính xác. Nếu hai object được coi là bằng nhau theo equals(), chúng phải có cùng hash code.

@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Customer other)) return false;
    return Objects.equals(customerId, other.customerId);
}

@Override
public int hashCode() {
    return Objects.hash(customerId);
}

3.7 HashMap hoạt động thế nào?

Khi gọi put(key, value), Java tính hash của key để xác định bucket. Nếu bucket đã có phần tử, Java so sánh key bằng hash và equals(). Khi collision nhiều, từ Java 8 bucket có thể chuyển từ linked list sang tree để cải thiện hiệu năng trong điều kiện phù hợp.

Câu hỏi: HashMap có thread-safe không?
Không. Nếu nhiều thread cùng sửa HashMap, có thể mất dữ liệu hoặc trạng thái không nhất quán. Dùng ConcurrentHashMap hoặc đồng bộ phù hợp.

3.8 Exception

  • Checked exception: compiler bắt buộc xử lý hoặc khai báo, ví dụ IOException.
  • Unchecked exception: kế thừa RuntimeException, thường biểu thị lỗi lập trình hoặc business validation.
  • Error: lỗi nghiêm trọng của JVM như OutOfMemoryError; thường không nên catch để tiếp tục như bình thường.
public class BusinessException extends RuntimeException {
    private final String code;

    public BusinessException(String code, String message) {
        super(message);
        this.code = code;
    }
}

Try-with-resources giúp tự đóng tài nguyên:

try (InputStream in = Files.newInputStream(path)) {
    // đọc file
}

3.9 Stack, Heap và Garbage Collector

  • Stack: biến local, frame của method, mỗi thread có stack riêng.
  • Heap: object được cấp phát, dùng chung giữa các thread.
  • Metaspace: metadata của class.
  • Garbage Collector: thu hồi object không còn được tham chiếu.
Java vẫn có memory leak khi object không còn hữu ích nhưng vẫn bị giữ tham chiếu, ví dụ static List tăng mãi, cache không TTL, ThreadLocal không remove hoặc listener không unregister.

4. SOLID và Design Pattern

SOLID không phải luật cứng. Đây là các nguyên tắc giúp code dễ thay đổi, dễ test và giảm phụ thuộc.

4.1 SOLID

S — Single Responsibility

Một class chỉ nên có một lý do để thay đổi. Không để một service vừa validate, vừa gọi DB, vừa gửi email, vừa build báo cáo.

O — Open/Closed

Mở cho mở rộng, đóng cho sửa đổi. Thêm phương thức thanh toán mới bằng class mới thay vì sửa một chuỗi if/else rất dài.

L — Liskov Substitution

Class con phải có thể thay thế class cha mà không phá hành vi mong đợi.

I — Interface Segregation

Interface nên nhỏ và đúng mục đích. Không ép class implement method nó không cần.

D — Dependency Inversion

Business logic phụ thuộc abstraction thay vì implementation cụ thể.

public interface PaymentProcessor {
    PaymentResult process(PaymentRequest request);
}

@Service
public class NapasPaymentProcessor implements PaymentProcessor {
    public PaymentResult process(PaymentRequest request) {
        // xử lý NAPAS
    }
}

Ví dụ Strategy + Factory đơn giản
public interface TransferHandler {
    String type();
    void handle(TransferRequest request);
}

@Component
class InternalTransferHandler implements TransferHandler {
    public String type() { return "INTERNAL"; }
    public void handle(TransferRequest request) { /* ... */ }
}

@Component
class TransferHandlerFactory {
    private final Map<String, TransferHandler> handlers;

    TransferHandlerFactory(List<TransferHandler> list) {
        handlers = list.stream().collect(Collectors.toMap(
            TransferHandler::type,
            Function.identity()
        ));
    }

    TransferHandler get(String type) {
        return Optional.ofNullable(handlers.get(type))
            .orElseThrow(() -> new IllegalArgumentException("Unsupported type"));
    }
}

5. Spring Boot và MVC

5.1 Spring Boot là gì?

Spring Framework cung cấp IoC, DI, transaction, web MVC và nhiều module. Spring Boot giúp cấu hình nhanh hơn nhờ auto-configuration, starter dependency và embedded server.

5.2 Luồng một HTTP request

  1. Request vào web server.
  2. Đi qua Filter, ví dụ authentication filter.
  3. DispatcherServlet tìm controller phù hợp.
  4. Interceptor có thể chạy trước/sau controller.
  5. Controller parse request và gọi service.
  6. Service xử lý nghiệp vụ, gọi repository/client.
  7. Response được Jackson serialize thành JSON.
Nguyên tắc: Controller mỏng, Service giữ business logic, Repository phụ trách truy cập dữ liệu. Không để mọi thứ trong Controller.

5.3 Dependency Injection

Thay vì class tự tạo dependency bằng new, Spring tạo và tiêm dependency vào. Điều này giảm coupling và giúp test dễ hơn.

@Service
public class TransferService {
    private final TransferRepository repository;

    public TransferService(TransferRepository repository) {
        this.repository = repository;
    }
}

Constructor injection được ưu tiên vì dependency rõ ràng, field có thể final và test thuận tiện.

5.4 Các annotation phổ biến

AnnotationÝ nghĩa
@RestControllerController trả dữ liệu, thường là JSON
@ServiceĐánh dấu business service
@RepositoryData access layer, có exception translation
@ComponentBean Spring chung
@Configuration + @BeanKhai báo bean bằng Java config

5.5 Bean scope và thread safety

Mặc định bean Spring là singleton: cả ứng dụng dùng chung một instance. Singleton không tự động thread-safe. Nếu bean chứa mutable field dùng chung, nhiều request có thể ghi đè nhau.

@Service
public class BadService {
    private String currentUser; // nguy hiểm vì nhiều request dùng chung
}

Service nên stateless: dữ liệu theo request nằm trong biến local hoặc truyền qua tham số.

5.6 DTO, Entity và Mapper

  • Entity: mô hình dữ liệu persistence.
  • Request DTO: dữ liệu client gửi vào.
  • Response DTO: dữ liệu API trả ra.
  • Mapper: chuyển đổi giữa các model.

Không nên expose entity trực tiếp vì dễ lộ field, dính lazy loading, vòng lặp JSON và phụ thuộc chặt API vào DB schema.

5.7 Validation và xử lý lỗi

    @ExceptionHandler(BusinessException.class)
    ResponseEntity<ErrorResponse> handle(BusinessException ex) {
        return ResponseEntity.badRequest()

5.8 @Transactional và Spring Proxy

Spring thường tạo proxy quanh bean. Khi method được gọi qua proxy, Spring bắt đầu transaction, gọi method thật, sau đó commit hoặc rollback.

Self-invocation: Một method trong cùng class gọi method khác có @Transactional có thể bypass proxy, nên transaction không chạy như kỳ vọng.

Mặc định Spring rollback với RuntimeException và Error; checked exception cần cấu hình nếu muốn rollback.

6. JPA, Hibernate và Oracle Database

6.1 JPA và Hibernate khác nhau thế nào?

JPA là đặc tả API cho ORM. Hibernate là một implementation phổ biến của JPA. Spring Data JPA giúp viết repository thuận tiện hơn.

6.2 Entity lifecycle

Trạng tháiÝ nghĩa
TransientObject mới tạo, chưa được persistence context quản lý
ManagedĐược quản lý; thay đổi có thể tự được ghi xuống DB
DetachedTừng managed nhưng context đã đóng hoặc object bị detach
RemovedĐược đánh dấu để xóa

6.3 Persistence Context và Dirty Checking

Trong transaction, khi lấy entity từ DB, Hibernate giữ entity trong persistence context. Nếu sửa field của entity managed, Hibernate phát hiện thay đổi và tạo SQL UPDATE lúc flush/commit.

@Transactional
public void updateStatus(Long id, String status) {
    Transfer transfer = repository.findById(id).orElseThrow();
    transfer.setStatus(status); // không cần gọi save trong nhiều trường hợp
}

6.4 Lazy và Eager

  • Lazy: dữ liệu quan hệ chỉ tải khi truy cập.
  • Eager: tải ngay cùng entity chính.

Eager không phải lúc nào cũng tốt vì có thể tải dư dữ liệu. Lazy cũng có rủi ro LazyInitializationException nếu truy cập quan hệ khi session đã đóng.

6.5 N+1 Query

Ví dụ: lấy 100 order bằng 1 query, sau đó mỗi order lại query customer riêng → tổng 101 query.

Cách xử lý:

  • Fetch join.
  • EntityGraph.
  • DTO projection.
  • Batch fetching.
  • Thiết kế query theo đúng use case thay vì load graph object lớn.
@Query("select o from Order o join fetch o.customer where o.status = :status")
List<Order> findWithCustomer(@Param("status") String status);

6.8 Optimistic Lock và Pessimistic Lock

Optimistic lock phù hợp khi xung đột ít. Dùng cột version; nếu bản ghi đã bị người khác cập nhật, update thất bại.

@Version
private Long version;

Pessimistic lock khóa bản ghi ngay khi đọc để ngăn transaction khác sửa. Phù hợp khi xung đột cao nhưng có thể gây chờ và deadlock.

6.9 Index

Index giống mục lục sách: giúp tìm dữ liệu nhanh hơn nhưng tốn dung lượng và làm insert/update chậm hơn vì phải cập nhật index.

Composite index

Index (status, created_at) thường hữu ích cho query lọc theo status và thời gian. Thứ tự cột rất quan trọng vì database thường dùng tốt phần tiền tố bên trái của index.

Dùng function lên cột, ép kiểu sai, wildcard đầu chuỗi như LIKE '%abc' hoặc select phần lớn bảng có thể khiến optimizer không dùng index.

6.10 Cách tối ưu query

  1. Đo query nào chậm, không tối ưu theo cảm giác.
  2. Xem execution plan.
  3. Kiểm tra số bản ghi, độ chọn lọc và join condition.
  4. Chỉ lấy cột cần thiết.
  5. Thêm/sửa index hợp lý.
  6. Tránh N+1.
  7. Phân trang đúng cách.
  8. Đo lại trước và sau.

7. Login, JWT và Security

7.1 Authentication và Authorization

  • Authentication: Xác minh bạn là ai.
  • Authorization: Sau khi biết bạn là ai, kiểm tra bạn được làm gì.

7.2 Luồng login dùng JWT

  1. Client gửi username/password qua HTTPS.
  2. Server tìm user và verify password hash.
  3. Server tạo access token, có thể kèm refresh token.
  4. Client gửi access token trong header Authorization.
  5. Security filter verify chữ ký, expiration, issuer/audience và tạo Authentication.
  6. Controller hoặc method security kiểm tra role/permission.

7.3 JWT gồm gì?

JWT thường có ba phần: Header, Payload và Signature. Payload chỉ được encode Base64URL, không mặc định được mã hóa. Vì vậy không đặt password, private key hoặc dữ liệu nhạy cảm vào payload.

Access token nên sống ngắn. Refresh token sống lâu hơn, cần quản lý chặt, rotation và khả năng thu hồi.

7.4 Logout với JWT

Vì JWT thường stateless, server không nhất thiết lưu access token. Một số cách logout:

  • Client xóa token; đơn giản nhưng token bị đánh cắp vẫn còn hiệu lực đến khi hết hạn.
  • Blacklist token trong Redis đến hết TTL.
  • Thu hồi refresh token; access token ngắn hạn tự hết hạn.
  • Dùng token version để vô hiệu hóa hàng loạt token của user.

7.6 OWASP — hiểu theo tình huống

Broken Access Control

User A đổi ID trên URL và xem dữ liệu User B. Cách tránh: kiểm tra quyền trên chính resource, không chỉ kiểm tra role.

Injection

Không nối chuỗi SQL từ input. Dùng parameterized query/JPA binding.

Security Misconfiguration

Không bật debug trên production, không expose actuator nhạy cảm, không dùng default password.

Cryptographic Failure

Không lưu plaintext password, không hard-code key, dùng TLS và thuật toán chuẩn.

SSRF

Không cho user tùy ý nhập URL để server gọi nội bộ. Cần allowlist và chặn private IP.

Logging Failure

Có audit log và cảnh báo, nhưng phải mask token, password, số tài khoản nhạy cảm.

8. Async, Sync, đa luồng và Concurrent Request

8.1 Sync và Async

  • Synchronous: caller chờ công việc hoàn tất rồi mới đi tiếp.
  • Asynchronous: caller gửi việc và có thể làm việc khác; kết quả đến sau qua future, callback hoặc message.

Async không đồng nghĩa với chạy song song. Một tác vụ có thể async nhưng vẫn chỉ chạy trên một thread/event loop.

8.2 Concurrency và Parallelism

  • Concurrency: nhiều công việc cùng tồn tại và được tiến hành xen kẽ.
  • Parallelism: thực sự chạy đồng thời trên nhiều core.

8.3 Race condition

Hai thread cùng đọc balance = 100, mỗi thread cộng 10 và ghi 110. Kết quả đúng phải là 120 nhưng bị mất một lần cập nhật.

class Counter {
    private int value;
    public void increment() { value++; } // không atomic
}

Có thể xử lý bằng synchronized, AtomicInteger, lock, database atomic update hoặc optimistic lock tùy phạm vi.

8.4 synchronized, volatile và Atomic

Công cụGiải quyếtKhông giải quyết
synchronizedMutual exclusion + visibilityCó thể giảm throughput nếu lock lớn
volatileVisibility và ordering của một biếnKhông làm phép tăng value++ thành atomic
AtomicIntegerCAS cho thao tác atomic đơn giảnKhông thay thế transaction phức tạp nhiều biến

8.5 Deadlock

Thread A giữ lock 1 chờ lock 2; thread B giữ lock 2 chờ lock 1. Cả hai chờ nhau mãi.

Cách giảm:

  • Luôn lấy lock theo cùng thứ tự.
  • Giữ lock trong thời gian ngắn.
  • Không gọi network/DB lâu khi đang giữ lock.
  • Dùng tryLock với timeout khi phù hợp.

8.6 Thread Pool

Tạo thread có chi phí. Thread pool tái sử dụng thread và giới hạn số lượng công việc chạy đồng thời.

  • corePoolSize: số thread lõi.
  • maximumPoolSize: số thread tối đa.
  • queue: nơi chờ.
  • rejection policy: xử lý khi pool và queue đầy.
Queue không giới hạn có thể làm memory tăng mạnh. Pool quá lớn có thể gây context switching và làm DB/external API quá tải.

8.7 CompletableFuture

CompletableFuture<User> userFuture =
    CompletableFuture.supplyAsync(() -> userClient.getUser(id), executor);

CompletableFuture<Account> accountFuture =
    CompletableFuture.supplyAsync(() -> accountClient.getAccount(id), executor);

return userFuture.thenCombine(accountFuture, UserAccountResponse::new)
    .orTimeout(2, TimeUnit.SECONDS)
    .exceptionally(ex -> fallback());

Cần executor riêng, timeout, exception handling và tránh dùng common pool mù quáng trong hệ thống server.

8.8 @Async

@Async chạy method qua executor. Tương tự @Transactional, self-invocation trong cùng class có thể không đi qua proxy. Ngoài ra transaction và security context không tự động truyền theo cách bạn mong muốn.

8.9 Virtual Threads

Virtual thread là thread nhẹ do JVM quản lý, phù hợp với lượng lớn tác vụ blocking I/O. Nó giúp mô hình code tuần tự dễ đọc hơn mà vẫn chịu được nhiều request chờ I/O.

  • Phù hợp: HTTP call, DB call, file I/O.
  • Không làm CPU-bound nhanh hơn.
  • Connection pool DB vẫn là giới hạn thật.
  • Không nên tạo hàng nghìn connection chỉ vì có hàng nghìn virtual thread.

8.10 Concurrent request trong nghiệp vụ

Ví dụ user bấm thanh toán hai lần hoặc hệ thống đối tác retry cùng request.

Các lớp bảo vệ:

  1. Idempotency key hoặc business reference duy nhất.
  2. Unique constraint trong database.
  3. Kiểm tra trạng thái trước khi xử lý.
  4. Optimistic/pessimistic lock nếu cần.
  5. Redis lock chỉ là lớp hỗ trợ, không nên là bảo vệ duy nhất.
CREATE UNIQUE INDEX uk_transfer_external_ref
ON transfer(external_reference);

9. Redis và Caching

9.1 Redis là gì?

Redis là data store in-memory, truy cập rất nhanh. Redis có thể dùng làm cache, session store, distributed lock, rate limit, counter, queue/stream và nhiều mục đích khác.

9.2 Data type cơ bản

TypeVí dụ dùng
StringCache JSON, token blacklist, counter
HashLưu object dạng field-value
ListQueue đơn giản
SetTập không trùng
Sorted SetLeaderboard, lịch theo score
StreamEvent stream/consumer group

9.3 Cache-aside

  1. Ứng dụng tìm dữ liệu trong Redis.
  2. Nếu có, trả cache hit.
  3. Nếu không, query database.
  4. Ghi kết quả vào Redis với TTL.
  5. Trả kết quả.
public Customer getCustomer(String id) {
    Customer cached = cache.get("customer:" + id);
    if (cached != null) return cached;

    Customer customer = repository.findById(id).orElseThrow();
    cache.set("customer:" + id, customer, Duration.ofMinutes(10));
    return customer;
}

9.4 Cache invalidation

Khi dữ liệu DB thay đổi, cache cũ có thể stale. Cách thường dùng là update DB rồi xóa cache. Request sau sẽ load dữ liệu mới.

Nếu xóa cache trước rồi update DB, một request khác có thể đọc DB cũ và ghi lại cache cũ. Thứ tự và transaction cần được cân nhắc.

9.6 Distributed lock

Cách cơ bản: SET lock-key unique-token NX PX 30000. Khi unlock phải kiểm tra token đúng owner và nên dùng Lua script để thao tác atomic.

Lock có timeout, network partition và failover. Với giao dịch quan trọng, cần thêm unique constraint, idempotency và state machine ở database.

10. Microservices

10.1 Microservice là gì?

Hệ thống được chia thành các service nhỏ hơn theo domain/business capability, có thể phát triển và deploy độc lập. Microservice không chỉ là “nhiều project Spring Boot”.

10.2 Lợi ích và chi phí

Lợi íchChi phí
Deploy độc lậpGiao tiếp mạng có thể lỗi
Scale từng serviceKhó debug end-to-end
Team ownership rõDistributed transaction phức tạp
Cô lập failure tốt hơn nếu thiết kế đúngVận hành, monitoring, security khó hơn

10.3 Giao tiếp đồng bộ và bất đồng bộ

  • REST/gRPC: cần phản hồi ngay, dễ hiểu nhưng tạo coupling theo thời gian.
  • Message broker: giảm coupling, chịu spike tốt nhưng eventual consistency và duplicate message.

10.4 Reliability pattern

  • Timeout: không chờ vô hạn.
  • Retry: chỉ retry lỗi tạm thời; có backoff và giới hạn.
  • Circuit breaker: tạm dừng gọi dependency đang lỗi liên tục.
  • Bulkhead: tách resource để một dependency không làm sập toàn hệ thống.
  • Rate limit: bảo vệ hệ thống khỏi quá tải.
Retry có thể tạo duplicate. Chỉ retry operation idempotent hoặc có idempotency key.

10.5 Distributed Transaction

Không thể dễ dàng dùng một transaction ACID qua nhiều database/service. Hai hướng phổ biến:

  • Saga: mỗi bước có local transaction và compensating action nếu bước sau thất bại.
  • Transactional Outbox: ghi business data và event vào cùng một DB transaction; worker phát event sau, tránh trạng thái “DB commit nhưng message chưa gửi”.

10.6 Idempotent Consumer

Message broker có thể giao lại message. Consumer nên lưu message ID đã xử lý hoặc thiết kế update có điều kiện để xử lý lặp không gây hậu quả.

10.7 Observability trong microservice

Mỗi request cần correlation ID/trace ID. Logs, metrics và traces phải giúp nối luồng từ API Gateway qua nhiều service.

11.3 Liveness, Readiness và Startup Probe

  • Readiness fail: Pod tạm thời không nhận traffic nhưng không nhất thiết restart.
  • Liveness fail: Kubernetes restart container.
  • Startup probe: cho app thời gian khởi động trước khi liveness kiểm tra.
Không để liveness phụ thuộc trực tiếp vào DB ngoài. Nếu DB chập chờn, toàn bộ pod có thể restart hàng loạt và làm tình hình tệ hơn.

11.4 Request và Limit

  • Request: lượng tài nguyên scheduler dùng để xếp pod.
  • Limit: trần tài nguyên. Memory vượt limit có thể bị OOMKilled; CPU vượt limit bị throttling.

11.5 Rolling Update

Deployment tạo pod mới dần, chỉ đưa vào nhận traffic khi readiness pass, sau đó dừng pod cũ. App cần backward-compatible với database/API trong thời gian hai version cùng chạy.

11.6 OpenShift

OpenShift xây trên Kubernetes và bổ sung công cụ vận hành, bảo mật, route, image stream và quy trình enterprise. OpenShift thường hạn chế chạy container bằng root thông qua security policy/SCC.

16. Memory Leak, High CPU và xử lý sự cố

16.1 Memory leak trong Java

GC chỉ thu hồi object không còn reachable. Nếu object vẫn được tham chiếu dù không còn hữu ích, GC không thể xóa.

Nguyên nhân thường gặp

  • Static collection tăng không giới hạn.
  • Cache không TTL hoặc eviction.
  • ThreadLocal không remove trong thread pool.
  • Listener/callback không unregister.
  • Đọc file/stream nhưng không đóng.
  • Giữ object graph lớn trong session hoặc queue.

16.2 Quy trình điều tra memory

  1. Xem memory usage và GC pattern.
  2. Kiểm tra heap có tăng liên tục sau full GC không.
  3. Lấy heap dump đúng thời điểm.
  4. Dùng Eclipse MAT/JProfiler/VisualVM để xem dominator tree và retained size.
  5. Xác định object nào giữ nhiều bộ nhớ và đường tham chiếu đến GC root.
  6. Sửa root cause, load test và so sánh trước/sau.

16.3 High CPU

Nguyên nhân có thể là vòng lặp vô hạn, regex xấu, serialization lớn, GC liên tục, thread contention, retry storm hoặc query/logic CPU-bound.

Quy trình điều tra

  1. Xác định process/pod có CPU cao.
  2. Lấy nhiều thread dump cách nhau vài giây.
  3. Tìm thread RUNNABLE lặp lại ở cùng stack.
  4. Map native thread ID nếu cần.
  5. Kiểm tra GC, request rate và dependency.
  6. Dùng Java Flight Recorder/profiler để thấy hot method.

16.4 Connection pool cạn

Dấu hiệu: request chờ connection, timeout tăng nhưng DB có thể chưa quá tải. Nguyên nhân: query chậm, transaction dài, connection leak hoặc pool nhỏ/traffic tăng.

Cách xử lý: metric pool, slow query, timeout hợp lý, đóng resource, transaction ngắn và không gọi external API khi đang giữ DB transaction.

16.5 Mẫu trả lời phỏng vấn

“Khi CPU tăng cao, em không đoán ngay nguyên nhân. Em kiểm tra metric theo pod, lấy thread dump nhiều lần để tìm thread nóng lặp lại, đối chiếu trace/log, sau đó dùng JFR nếu cần. Nếu CPU đi kèm GC cao thì em kiểm tra allocation và heap; nếu thread bị BLOCKED nhiều thì kiểm tra contention. Sau khi sửa em load test và theo dõi p95, CPU và GC trước/sau.”

17.4 Pagination

Offset pagination dễ dùng nhưng có thể chậm với offset lớn và dữ liệu thay đổi. Keyset/cursor pagination dùng cột sắp xếp ổn định, phù hợp dataset lớn.

18. Bộ câu hỏi phỏng vấn kèm hướng trả lời

1. Hãy giới thiệu dự án gần nhất

Nói theo 6 phần: nghiệp vụ, kiến trúc, tech stack, vai trò, vấn đề khó, kết quả. Không kể lan man toàn bộ dự án.

2. Spring Bean singleton có thread-safe không?

Không tự động thread-safe. Singleton chỉ nói có một instance. Nếu bean stateless thì thường ổn; nếu có mutable shared field thì có race condition.

3. @Transactional hoạt động thế nào?

Spring dùng proxy để mở transaction trước khi gọi method và commit/rollback sau đó. Self-invocation có thể bypass proxy. Mặc định rollback với RuntimeException.

4. N+1 query là gì?

Một query lấy danh sách, sau đó mỗi phần tử phát sinh thêm query cho relation. Xử lý bằng fetch join, EntityGraph, DTO projection hoặc batch fetching.

5. Vì sao index làm query nhanh hơn?

Index tạo cấu trúc tìm kiếm riêng giúp giảm số block phải đọc. Đổi lại tốn storage và làm thao tác ghi chậm hơn.

6. JWT có mã hóa payload không?

Thông thường không. Payload chỉ encode, ai có token cũng có thể đọc. Signature chỉ đảm bảo token không bị sửa.

7. 401 và 403 khác nhau thế nào?

401 là chưa xác thực hoặc credential không hợp lệ. 403 là đã xác thực nhưng không có quyền.

8. Làm sao chống giao dịch trùng?

Dùng business reference/idempotency key, unique constraint ở DB, kiểm tra state, transaction và có thể lock. Không chỉ dựa vào Redis lock.

9. volatile có làm value++ thread-safe không?

Không. value++ gồm đọc, cộng, ghi; volatile chỉ đảm bảo visibility/order, không đảm bảo toàn bộ chuỗi thao tác atomic.

10. Khi nào dùng virtual thread?

Khi có nhiều tác vụ blocking I/O và muốn giữ code tuần tự dễ đọc. Không giúp CPU-bound nhanh hơn và vẫn bị giới hạn bởi DB pool/external service.

11. Cache-aside là gì?

Đọc cache trước; miss thì đọc DB và ghi cache. Khi update thường update DB rồi xóa cache để lần đọc sau nạp mới.

12. Redis bị down thì hệ thống nên thế nào?

Tùy Redis là cache hay source of truth. Nếu chỉ là cache, có fallback về DB có giới hạn, timeout ngắn, circuit breaker và tránh tạo stampede.

13. Retry có nguy hiểm gì?

Có thể khuếch đại tải và tạo duplicate. Cần retry lỗi tạm thời, backoff, jitter, giới hạn số lần và idempotency.

14. Liveness và Readiness khác nhau?

Readiness quyết định có nhận traffic không. Liveness quyết định container có cần restart không.

15. Merge và Rebase?

Merge giữ lịch sử nhánh và có thể tạo merge commit. Rebase viết lại commit lên base mới để lịch sử thẳng hơn; tránh rebase shared history nếu không thống nhất.

16. Java có GC sao vẫn memory leak?

GC chỉ xóa object không reachable. Nếu object không còn hữu ích nhưng vẫn bị static collection/cache/listener giữ tham chiếu, nó không được thu hồi.

17. Cách điều tra API chậm?

Xem p95/p99, trace, slow query, external API latency, thread/connection pool, CPU/GC, payload và cache hit rate. Đo trước rồi mới tối ưu.

18. HashMap và ConcurrentHashMap khác gì?

HashMap không thread-safe. ConcurrentHashMap hỗ trợ concurrent access với cơ chế đồng bộ/CAS phân tán hơn, nhưng compound operation vẫn cần dùng API atomic như compute.

19. SHA có phải encryption không?

Không. SHA là hash một chiều. Encryption có key và có thể decrypt; hash dùng để tạo digest.

20. Vì sao không gọi external API trong transaction dài?

Vì transaction giữ DB connection/lock trong lúc chờ network, dễ cạn pool, tăng contention và deadlock. Nên rút ngắn transaction hoặc thiết kế outbox/state machine.

19. Cheat sheet trước giờ phỏng vấn

Java/Spring

  • Constructor injection, service stateless.
  • HashMap: hash → bucket → equals.
  • @Transactional chạy qua proxy.
  • RuntimeException rollback mặc định.
  • N+1: fetch join/EntityGraph/projection.

DB/Concurrency

  • Index giúp đọc nhanh, làm ghi tốn hơn.
  • Optimistic lock khi conflict ít.
  • Idempotency + unique constraint chống duplicate.
  • volatile không làm ++ atomic.
  • Transaction ngắn, không giữ lock khi gọi network.

Security/Redis

  • 401 chưa xác thực; 403 thiếu quyền.
  • JWT payload không mặc định mã hóa.
  • SHA là hash; AES/RSA là encryption.
  • Cache-aside, TTL và invalidation.
  • Redis lock không thay thế DB constraint.

K8S/Production

  • Readiness ngắt traffic; liveness restart.
  • OOMKilled do memory limit.
  • High CPU: metrics → thread dump → JFR.
  • Memory leak: heap dump → dominator tree.
  • Logs + metrics + traces + correlation ID.
