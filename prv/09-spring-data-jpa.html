JPA Là ORM, Không Phải Performance Shield

Spring Data JPA giúp tạo repository nhanh, Hibernate giúp map object sang table, nhưng endpoint nhanh hay chậm vẫn phụ thuộc vào SQL cuối cùng được chạy. Nếu không nhìn được query shape, bạn sẽ tưởng code Java sạch là đủ.

Nguyên tắc học: mỗi repository method phải trả lời được 3 câu: chạy bao nhiêu SQL, load bao nhiêu row/column, entity có còn attached trong transaction không?

1. Bài Toán Thực Tế - Catalog Dashboard Load 25 Giây

Admin dashboard hiển thị 200 sản phẩm, mỗi sản phẩm gồm tên brand, 3 review mới nhất và số lượng tồn kho. DBA báo không có slow query nào quá 15ms, nhưng API vẫn mất 25 giây.

@Service
public class CatalogDashboardService {
    @Transactional(readOnly = true)
    public List<ProductCard> loadDashboard(String categoryId) {
        List<Product> products = productRepository.findByCategoryId(categoryId);

        return products.stream()
            .map(product -> {
                Brand brand = product.getBrand(); // có thể trigger query
                List<Review> reviews = reviewRepository
                    .findTop3ByProductIdOrderByCreatedAtDesc(product.getId());
                Stock stock = stockRepository.findBySku(product.getSku());
                return ProductCard.of(product, brand, reviews, stock);
            })
            .toList();
    }
}
1 query products
200 queries brand/review/stock pattern
Tổng 401-601 round-trips nhỏ, mỗi query nhanh nhưng tổng network + pool wait rất lớn.
Tại sao DBA không thấy slow query?

Vì từng query riêng lẻ nhanh. N+1 là latency do số lượng round-trip, không nhất thiết do một câu SQL chậm.

2. Mental Model

2.1 N+1 Problem

N+1 là: 1 query lấy N parent rows, rồi N query phụ để lấy association hoặc dữ liệu liên quan cho từng parent.

N càng lớn, endpoint càng chậm tuyến tính dù mỗi query nhỏ.

2.2 Hibernate Session, L1 Cache, L2 Cache

CacheScopeBật mặc định?Dùng cho
L1 cacheEntityManager/Session hiện tạiLuôn bậtĐảm bảo cùng ID trong 1 transaction là cùng object instance.
L2 cacheSessionFactory, share toàn appTắt mặc địnhReference data ít đổi: category, country, shipping method.
Query cacheQuery result idsTắt mặc địnhQuery lặp lại, data ít đổi, invalidation rõ.

L1 cache không giải quyết N+1 review query vì mỗi product có query khác nhau. L2 cache cũng không nên dùng để che query shape xấu; trước hết phải sửa fetch/projection.

2.3 Dirty Checking

Trong transaction, Hibernate giữ snapshot của entity. Khi flush/commit, nó so sánh entity hiện tại với snapshot để generate update.

@Transactional
public void renameProduct(String productId, String newName) {
    Product product = productRepository.findById(productId).orElseThrow();
    product.renameTo(newName);
    // không bắt buộc gọi save(); Hibernate dirty checking sẽ flush UPDATE khi commit
}

Dirty checking tiện nhưng có cost nếu bạn load hàng chục nghìn entity chỉ để read. Với read path lớn, dùng @Transactional(readOnly = true) và DTO projection.

2.4 Optimistic vs Pessimistic Lock

LockCơ chếDùng khiTrade-off
Optimistic@Version, update kèm WHERE version=?Conflict hiếm: product description, profile, document editCần retry/handle conflict.
PessimisticSELECT ... FOR UPDATEConflict thường: giữ ghế cuối, decrement stock flash saleGiữ lock, giảm throughput, có deadlock risk.

2.5 Entity Lifecycle: transient, managed, detached, removed

Rất nhiều bug JPA đến từ việc không biết entity hiện đang ở trạng thái nào. Hibernate chỉ dirty-check entity managed trong persistence context. Entity detached thay đổi field không tự update DB nếu không merge/save lại đúng cách.

StateNghĩaVí dụ
TransientObject mới, chưa gắn DB/sessionnew Product() trước khi save.
ManagedĐang nằm trong persistence contextEntity load trong @Transactional.
DetachedTừng managed nhưng session đã đóngEntity trả ra ngoài transaction.
RemovedĐược đánh dấu delete khi flushrepository.delete(entity).

API layer nên trả DTO, không trả entity detached. Nếu controller serialize entity, lazy association có thể gây LazyInitializationException hoặc kéo thêm query ngoài ý muốn nếu OSIV bật.

2.6 Flush vs Commit

flush gửi SQL pending xuống database nhưng transaction vẫn chưa commit. commit mới xác nhận thay đổi. Hibernate có thể flush trước commit, hoặc trước một query cần thấy thay đổi pending để đảm bảo consistency.

Hiểu sai

"Gọi save() là DB đã commit" hoặc "flush xong là không rollback được".

Hiểu đúng

save() làm entity managed/pending. Flush gửi SQL. Commit xác nhận. Exception trước commit vẫn rollback.

Trong batch import lớn, flush/clear theo chunk giúp tránh persistence context giữ quá nhiều entity managed và làm dirty checking chậm dần.

3. Production-Grade Implementation

3.1 JOIN FETCH Cho Many-to-One

public interface ProductRepository extends JpaRepository<Product, String> {
    @Query("""
        select p
        from Product p
        join fetch p.brand
        where p.categoryId = :categoryId
        """)
    List<Product> findByCategoryWithBrand(String categoryId);
}

3.2 DTO Projection Cho Dashboard

public record ProductCardRow(
    String productId,
    String productName,
    String brandName,
    int availableQuantity
) {}

public interface ProductDashboardRepository extends Repository<Product, String> {
    @Query("""
        select new com.example.ProductCardRow(
            p.id, p.name, b.name, s.availableQuantity
        )
        from Product p
        join p.brand b
        join Stock s on s.sku = p.sku
        where p.categoryId = :categoryId
        """)
    List<ProductCardRow> findCards(String categoryId);
}

Projection chỉ lấy cột cần thiết, không tạo full entity graph, không dirty-check. Đây là lựa chọn tốt cho dashboard/report.

3.6 Pessimistic Lock Cho Stock

public interface StockRepository extends JpaRepository<Stock, String> {
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("select s from Stock s where s.sku = :sku")
    Optional<Stock> findBySkuForUpdate(String sku);
}

@Transactional(timeout = 5)
public void reserve(String sku, int quantity) {
    Stock stock = stockRepository.findBySkuForUpdate(sku).orElseThrow();
    stock.reserve(quantity);
}

3.7 EntityGraph: Fetch Plan Theo Use Case

@EntityGraph cho phép định nghĩa association cần fetch cho một repository method mà không viết JPQL fetch join dài. Nó hợp với case fetch graph vừa phải, nhất là many-to-one/one-to-one hoặc một vài association rõ ràng.

public interface ProductRepository extends JpaRepository<Product, String> {
    @EntityGraph(attributePaths = {"brand", "category"})
    Optional<Product> findDetailedById(String id);
}

Đừng dùng EntityGraph như cách kéo toàn bộ object graph. Với dashboard/list/report, DTO projection vẫn thường rõ và nhanh hơn.

3.8 Bulk Update Bypass Persistence Context

JPQL bulk update/delete chạy trực tiếp trên database và bypass entity managed trong persistence context. Nếu trong cùng transaction bạn đang giữ entity cũ, object trong memory có thể stale.

@Modifying(clearAutomatically = true, flushAutomatically = true)
@Query("""
    update Product p
    set p.status = :status
    where p.categoryId = :categoryId
    """)
int updateStatusByCategory(String categoryId, ProductStatus status);
Bẫy stale state

Sau bulk update, đừng tiếp tục tin entity đã load trước đó vẫn phản ánh DB. Dùng clearAutomatically, tách transaction, hoặc reload entity cần dùng tiếp.

3.9 Query Count Test: Chống N+1 Tái Phát

N+1 dễ quay lại sau refactor. Ngoài review code, nên có integration test đếm query cho endpoint quan trọng bằng datasource-proxy/p6spy hoặc helper nội bộ.

@Test
void dashboardShouldNotTriggerNPlusOne() {
    queryCounter.clear();

    List<ProductCard> cards = service.loadDashboard("camera");

    assertThat(cards).hasSize(200);
    assertThat(queryCounter.selectCount()).isLessThanOrEqualTo(5);
}

Con số threshold phụ thuộc use case, nhưng test phải bảo vệ shape chính: không từ 3 query nhảy thành 403 query mà CI vẫn xanh.

5. Trade-offs & Anti-Patterns

AP1

EAGER Mặc Định

@ManyToOne và @OneToOne mặc định EAGER. Trong app lớn, nên explicit LAZY và fetch đúng theo use case.

AP2

Trả Entity Ra Controller

Entity có lazy association, bidirectional graph, field nội bộ và lifecycle attached/detached. API nên trả DTO.

AP3

L2 Cache Cho Data Hay Đổi

L2 cache hợp với reference data. Với stock, price realtime, availability, cache sai có thể làm user thấy trạng thái cũ.

AP4

Open Session In View Để Che Lazy Error

OSIV giữ session mở tới view/controller serialization, làm lazy load xảy ra ngoài service boundary và giấu N+1. Tốt hơn là fetch/map DTO rõ trong transaction.

6. Interview Framework

SurfaceN+1 là gì?

Là pattern 1 query lấy N parent rows, sau đó N query phụ để load association hoặc data liên quan từng parent. Từng query có thể nhanh nhưng tổng round-trip lớn khiến endpoint chậm.

DeepL1 cache khác L2 cache?

L1 cache thuộc một EntityManager/Session, luôn bật, sống trong transaction/request. L2 cache thuộc SessionFactory, optional, share toàn application. L1 phục vụ identity map và dirty checking; L2 chỉ nên dùng cho data ít đổi và invalidation rõ.

ArchitectureThiết kế read path cho catalog lớn

Tách write model entity khỏi read model projection. API list/dashboard dùng DTO projection/native query nếu cần window function. Detail page fetch aggregate có kiểm soát. Report nặng chạy trên read replica hoặc async export. Theo dõi query count và index theo query thực tế.

Checklist Thành Thạo

  • Nhìn code loop + repository/lazy access và chỉ ra N+1.
  • Chọn được JOIN FETCH, EntityGraph, BatchSize hoặc DTO projection theo use case.
  • Giải thích được L1 cache, dirty checking và LazyInitializationException.
  • Biết vì sao JOIN FETCH collection với pagination nguy hiểm.
  • Viết được @Version optimistic lock và @Lock(PESSIMISTIC_WRITE).
  • Phân biệt entity transient, managed, detached, removed và hệ quả với dirty checking.
  • Biết bulk update bypass persistence context và cần clear/reload sau đó.
  • Viết được query count test để ngăn N+1 quay lại.
