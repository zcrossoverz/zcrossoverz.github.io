#06 - Spring Boot: Auto-configuration, Actuator, Starter, Property Binding

Spring Boot là lớp convention trên Spring Framework. Bạn không chỉ cần biết "Boot tự cấu hình", mà phải trace được vì sao một bean xuất hiện hoặc không xuất hiện, property được resolve từ đâu, và actuator nào an toàn cho production.

Boot Không Thay Spring, Boot Giảm Boilerplate

Spring Framework cung cấp IoC, AOP, transaction, web, data. Spring Boot thêm auto-configuration, starter dependencies, embedded server, externalized configuration, actuator và một cách đóng gói service nhất quán.

Câu trả lời phỏng vấn tốt không dừng ở "add starter là chạy". Hãy nói được: starter kéo dependency nào, auto-config class nào được load, conditional nào pass/fail, bean nào được tạo, và cách override bean mặc định mà không fork framework.

2. Mental Model

2.1 Auto-configuration

Auto-configuration là một danh sách class cấu hình được Boot import tự động, rồi lọc qua điều kiện. Điều kiện phổ biến: class có trong classpath, property có bật, bean chưa tồn tại, app là servlet hay reactive.

Không có magic: nếu điều kiện fail, bean không được register.
@AutoConfiguration
@ConditionalOnClass(KafkaTemplate.class)
@ConditionalOnProperty(prefix = "spring.kafka", name = "bootstrap-servers")
public class KafkaAutoConfiguration {
    @Bean
    @ConditionalOnMissingBean
    KafkaTemplate<String, Object> kafkaTemplate(ProducerFactory<String, Object> factory) {
        return new KafkaTemplate<>(factory);
    }
}

Nếu service không có dependency spring-kafka hoặc không có spring.kafka.bootstrap-servers, KafkaTemplate sẽ không xuất hiện.

2.2 Property Binding

Boot merge nhiều nguồn config theo thứ tự ưu tiên. Những nguồn hay gặp nhất trong daily work:

Ưu tiênNguồnVí dụGhi nhớ

CaoJava system props-Dspring.profiles.active=stagingHay dùng trong JVM args.
CaoOS env varsSPRING_DATASOURCE_PASSWORDOverride application.yml trong container.

Relaxed binding cho phép notification.provider.timeout-ms, NOTIFICATION_PROVIDER_TIMEOUT_MS, và notification.provider.timeoutMs map vào cùng field.

2.3 Config Precedence: Lỗi Deploy Hay Nằm Ở Đây

Khi cùng một key xuất hiện ở nhiều nơi, Boot chọn source có ưu tiên cao hơn. Đây là nguyên nhân phổ biến của lỗi "local đúng, staging sai": biến môi trường trong container override YAML trong jar, hoặc command line override profile file.

Dễ gây hiểu nhầm
notification:
  provider:
    timeout-ms: 3000

Nhưng deployment lại set NOTIFICATION_PROVIDER_TIMEOUT_MS=30000.

Debug có evidence
curl internal-host/actuator/env/notification.provider.timeout-ms
java -jar app.jar --debug

Xem value cuối cùng và source nào đóng góp value đó.

Với secret hoặc endpoint bắt buộc, tránh default giả như ${PROVIDER_URL:http://localhost} trong production. Hãy validate bằng @NotBlank, fail startup sớm, và để deployment cung cấp config thật.

2.4 Actuator

Actuator biến service thành thứ có thể vận hành: health, metrics, info, mappings, beans, loggers, conditions. Nhưng actuator cũng có thể lộ thông tin nếu expose quá rộng.

EndpointDùng đểProduction
/actuator/healthLiveness/readinessCó thể public nội bộ.
/actuator/metricsJVM, HTTP, Hikari, custom metricsCho Prometheus/internal.
/actuator/conditionsDebug auto-configChỉ internal, có auth.
/actuator/envXem resolved propertiesRất nhạy, hạn chế mạnh.
/actuator/heapdumpDebug memoryKhông expose rộng.

2.5 Custom Starter

Starter hữu ích khi nhiều service cần cùng một năng lực: correlation ID, error response chuẩn, WebClient logging, common metrics, feature flags. Starter tốt không ép service dùng implementation duy nhất; nó cung cấp default có thể override.

3. Production-Grade Implementation

3.1 Type-safe Configuration

@ConfigurationProperties(prefix = "notification")
@Validated
public class NotificationProperties {
    @NotNull
    private Provider provider = new Provider();

    @NotNull
    private Retry retry = new Retry();

    public static class Provider {
        @NotBlank
        private String baseUrl;

        @Min(100)
        @Max(30000)
        private int timeoutMs = 3000;
    }

    public static class Retry {
        @Min(0)
        @Max(10)
        private int maxAttempts = 3;

        @Min(10)
        private long backoffMs = 200;
    }
}
@SpringBootApplication
@ConfigurationPropertiesScan
public class NotificationApplication {
    public static void main(String[] args) {
        SpringApplication.run(NotificationApplication.class, args);
    }
}

Dùng @ConfigurationProperties thay vì rải @Value khắp code để config có schema, validation, autocomplete và test dễ hơn.

3.3 Custom Health Indicator

@Component("providerHealth")
public class ProviderHealthIndicator implements HealthIndicator {
    private final NotificationProviderClient client;

    public ProviderHealthIndicator(NotificationProviderClient client) {
        this.client = client;
    }

    @Override
    public Health health() {
        try {
            ProviderStatus status = client.ping();
            if (status.available()) {
                return Health.up()
                    .withDetail("provider", status.name())
                    .withDetail("latencyMs", status.latencyMs())
                    .build();
            }
            return Health.down().withDetail("reason", status.reason()).build();
        } catch (Exception ex) {
            return Health.down(ex).build();
        }
    }
}

3.4 Starter Skeleton Có Override Point

@AutoConfiguration
@ConditionalOnWebApplication(type = ConditionalOnWebApplication.Type.SERVLET)
@EnableConfigurationProperties(PlatformWebProperties.class)
public class PlatformWebAutoConfiguration {
    @Bean
    @ConditionalOnMissingBean
    public CorrelationIdFilter correlationIdFilter(PlatformWebProperties props) {
        return new CorrelationIdFilter(props.getCorrelationHeader());
    }

    @Bean
    @ConditionalOnMissingBean
    public ApiErrorHandler apiErrorHandler() {
        return new ApiErrorHandler();
    }
}
# META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
com.example.platform.web.PlatformWebAutoConfiguration

Rule: mọi bean default trong starter gần như luôn cần @ConditionalOnMissingBean. Nếu service cần custom correlation header, custom error handler hoặc custom metric tag, họ chỉ cần tự define bean.

3.5 Liveness vs Readiness

Trong container/orchestrator, health không chỉ là "process còn sống". Liveness trả lời process có cần restart không. Readiness trả lời service đã sẵn sàng nhận traffic chưa. Lẫn hai khái niệm này có thể gây restart loop hoặc route traffic vào instance chưa warm up.

ProbeNên báo DOWN khiKhông nên báo DOWN khi
LivenessApp deadlock, event loop chết, state không thể phục hồiDependency tạm chậm vài giây
ReadinessDB/messaging/cache bắt buộc chưa sẵn sàng, app đang warm upMột optional provider phụ bị lỗi
 
management:
  endpoint:
    health:
      probes:
        enabled: true
      group:
        readiness:
          include: readinessState,db,providerHealth
        liveness:
          include: livenessState

3.6 Metrics & Tracing Vừa Đủ Cho Daily Work

Boot tích hợp Micrometer để xuất metrics nhất quán. Senior không cần tự viết monitoring platform, nhưng phải biết service cần tối thiểu HTTP latency, error rate, JVM, datasource pool, executor, custom business counter và correlation ID trong log.

@Service
public class NotificationSender {
    private final Counter sentCounter;
    private final Timer providerTimer;

    public NotificationSender(MeterRegistry registry) {
        this.sentCounter = Counter.builder("notification.sent")
            .description("Notifications sent successfully")
            .register(registry);
        this.providerTimer = Timer.builder("notification.provider.latency")
            .publishPercentileHistogram()
            .register(registry);
    }

    public void send(Notification notification) {
        providerTimer.record(() -> doSend(notification));
        sentCounter.increment();
    }
}
Metric cardinality

Không dùng userId, requestId, email, documentId làm metric tag. Cardinality cao làm backend metrics phình to và dashboard chậm. Tag nên là dimension hữu hạn như provider, status, errorType, region.

5. Trade-offs & Anti-Patterns

AP1

Expose * Actuator

Không expose toàn bộ actuator trong production. env, heapdump, threaddump, shutdown đều cần kiểm soát chặt.

AP2

Default Secret Trong Config

${API_KEY:demo} có thể che lỗi thiếu secret. Với config bắt buộc, hãy để thiếu là fail startup hoặc validate bằng @NotBlank.

AP3

Starter Quá Nhiều Trách Nhiệm

Một starter gom JWT, database, logging, tracing, client, scheduler sẽ coupling mạnh. Tách theo capability: platform-web-starter, platform-observability-starter, platform-client-starter.

6. Interview Framework

SurfaceSpring Boot khác Spring Framework ở đâu?

Boot không thay Spring. Boot thêm auto-configuration, starter dependency management, embedded server, externalized config và actuator để build/deploy service nhanh hơn với convention rõ ràng.

DeepAuto-configuration hoạt động thế nào?

Boot đọc danh sách auto-configuration từ AutoConfiguration.imports, rồi filter qua @ConditionalOnClass, @ConditionalOnProperty, @ConditionalOnMissingBean, web app type và nhiều condition khác. Class pass mới register BeanDefinition. Debug bằng /actuator/conditions hoặc --debug.

ArchitectureThiết kế custom starter cho 20 service

Tách module autoconfigure và starter, dùng @ConfigurationProperties cho customization, dùng @ConditionalOnMissingBean cho override, expose versioning rõ ràng, viết integration tests bằng ApplicationContextRunner, và tránh kéo dependency không cần thiết vào mọi service.

Checklist Thành Thạo

  • Trace được path từ starter dependency tới auto-config class và bean được tạo.
  • Giải thích được @ConditionalOnMissingBean là cơ chế override, không phải decoration.
  • Biết property source nào thắng khi cùng key xuất hiện ở env var và application.yml.
  • Viết được @ConfigurationProperties có nested config và validation.
  • Biết endpoint actuator nào nên expose, endpoint nào phải bảo vệ.
  • Phân biệt liveness và readiness để tránh restart loop hoặc route traffic sai.
  • Biết tạo metric Micrometer cơ bản và tránh high-cardinality tags.
