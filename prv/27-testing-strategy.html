Testing Strategy — Pyramid, Testcontainers, Contract, Non-Brittle

"80% coverage" là con số gây hiểu giả nguy hiểm nhất trong testing — nó đo dòng code chạy qua, không đo hành vi được khẳng định. File này giải thích kỹ vì sao có test pyramid, vì sao Testcontainers thắng H2, contract testing chặn breaking change ra sao, và thế nào là test "brittle" — đọc một lần là đủ để trả lời phỏng vấn. Domain minh hoạ: ShopFlow — OrderService của sàn e-commerce.

🎯 Mục tiêu
Viết test không brittle, dùng Testcontainers, hiểu contract test
📅 Thời lượng
1 tuần · Sau #9 JPA, #7 Transaction
📚 Prereq
JUnit 5 · Mockito · Spring Boot test · Docker
🔥 Độ ưu tiên
🟠 Cao — senior sở hữu chất lượng, không chỉ feature

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Coverage 80% nghĩa là code được test kỹ, ít bug."
Thực tế: Coverage chỉ đo dòng code được chạy qua trong lúc test, không đo có assertion ý nghĩa nào kiểm tra kết quả. Một test gọi method rồi không assert gì vẫn nâng coverage 100% cho method đó. P1 vẫn lọt vì test phủ rộng nhưng nông — chạy qua nhưng không khẳng định đúng/sai.
Tưởng hiểu: "Integration test với H2 in-memory cho nhanh là đủ."
Thực tế: H2 chạy ở "compatibility mode" mô phỏng PostgreSQL nhưng không bằng — JSONB, window function, native query, ON CONFLICT, hành vi constraint đều khác. Test pass với H2 nhưng prod dùng PostgreSQL → bug bị che tới khi lên production.
Tưởng hiểu: "Mock càng nhiều càng tốt vì test chạy nhanh và độc lập."
Thực tế: Nếu mock cả class đang test thì bạn đang test cái mock, không test code thật → false confidence. Chỉ mock dependency (repository, external API); để class cần kiểm chạy logic thật.

🗺️ Roadmap — nếu chỉ có X giờ thì học gì

Thứ tự ưu tiên
1 giờ (CORE ★): Coverage đo gì (2.1), test pyramid + lý do (2.2), unit test isolation đúng (2.3, 3.1), test behavior không implementation (2.7). Gốc để viết test không brittle.
+1 giờ (CORE ★): Integration với Testcontainers vs H2 (2.4, 3.2), @SpringBootTest slices cho tốc độ (2.6, 3.3).

1. Bài toán thực tế — 80% coverage vẫn lọt P1 CORE ★

ShopFlow có 80% test coverage nhưng bug P1 vẫn lên production đều đặn. Đào sâu thấy bốn triệu chứng: (1) CI chạy test mất 45 phút, chặn merge cả buổi; (2) thêm một feature làm vỡ 50 test cũ — các test này check cách code làm (gọi setter nào, gọi hàm nào) thay vì kết quả, nên đổi cấu trúc nội bộ là đỏ hàng loạt; (3) contract giữa OrderService và NotificationService vỡ âm thầm vì NotificationService đổi field JSON mà OrderService không biết, chỉ phát hiện khi chạy thật; (4) integration test dùng H2 nên một native query đúng cú pháp PostgreSQL pass ở test nhưng fail ở prod.

⏸ Pause & Predict — trước khi đọc lời giải
Viết ra giấy:
  1. 80% coverage mà vẫn lọt P1 — coverage đo cái gì, KHÔNG đo cái gì? Một loại test nào làm coverage cao mà giá trị thấp?
  2. Vì sao "thêm 1 feature làm vỡ 50 test"? Đặc điểm chung của 50 test đó là gì (gợi ý: chúng check cái gì)?
  3. H2 pass nhưng PostgreSQL fail — cho 2 ví dụ cụ thể về khác biệt dialect khiến bug lọt.
Đáp án ở 2.1 (coverage ≠ quality), 2.7 (test implementation = brittle), 2.4 (H2 vs PostgreSQL: JSONB, native query, ON CONFLICT, constraint). Đừng đọc tiếp khi chưa trả lời câu 2.

5 câu Senior phải trả lời

  1. Coverage tốt nghĩa là gì? (không chỉ 80%)
  2. Vì sao test pyramid quan trọng? Tỉ lệ Unit:Integration:E2E?
  3. Vì sao Testcontainers (PostgreSQL) tốt hơn H2? Khác biệt thật?
  4. Contract testing (Pact) là gì? Chặn breaking change API thế nào?
  5. Viết test không brittle thế nào? (test behavior, không implementation)

2.1 Coverage đo gì & KHÔNG đo gì CORE ★

Code coverage là tỉ lệ phần trăm code được thực thi trong khi chạy test (line coverage = % dòng chạy qua; branch coverage = % nhánh if/else đi qua). Vấn đề căn bản: coverage đo "code có chạy không", không đo "kết quả có được kiểm tra không". Một test gọi service.placeOrder(req) rồi kết thúc mà không có một assert nào vẫn làm coverage của placeOrder lên 100% — nhưng nó không phát hiện được bất kỳ bug nào, vì không khẳng định kết quả phải là gì.

Vì vậy "80% coverage" chỉ nói "80% code từng được chạy ít nhất một lần trong test suite", chứ không nói "80% hành vi đã được kiểm chứng đúng". Đó là lý do coverage cao mà vẫn lọt P1. Coverage tốt = kết hợp ba thứ: (a) phủ đủ nhánh quan trọng (cả happy path lẫn error path), (b) mỗi test có assertion ý nghĩa kiểm tra đúng kết quả/hành vi, và (c) test đó thật sự bắt được bug nếu code sai. Thước đo (c) chính xác nhất không phải coverage mà là mutation testing: công cụ cố tình đổi code (đổi > thành >=, xoá một dòng) rồi xem test có đỏ không — nếu test vẫn xanh nghĩa là nó không thật sự kiểm cái đó.

Cách nói trong phỏng vấn

"Coverage là điều kiện cần chứ không đủ. Tôi dùng nó như sàn (vd không để dưới 70% cho code nghiệp vụ) nhưng không coi nó là mục tiêu — vì tối ưu coverage dễ tạo test rỗng. Thước đo chất lượng thật là mutation score và việc test có phủ error path không."

2.2 Test pyramid — vì sao có hình tháp CORE ★

Test pyramid là một heuristic về tỉ lệ giữa ba loại test, dựa trên ba thuộc tính đối nghịch của mỗi loại: tốc độ, độ tin cậy của tín hiệu (test đỏ thì có đúng là bug không), và phạm vi bắt lỗi.

        E2E (ít)        — toàn hệ thống, chậm nhất (~1 phút/test), brittle nhất, confidence cao nhất
     Integration (vừa)  — vài component + DB thật, tốc độ trung bình (~vài giây/test)
   Unit (nhiều)         — 1 class, mock dependency, nhanh (~mili-giây), deterministic, chạy song song

Tỉ lệ tham chiếu: ~70% Unit : 20% Integration : 10% E2E

Vì sao nhiều unit: unit test nhanh (hàng nghìn test chạy trong vài giây) và deterministic (không phụ thuộc mạng/DB nên không "flaky"). Chúng cho vòng phản hồi ngắn — sửa code, chạy lại, biết ngay sai ở đâu vì mỗi test chỉ kiểm một class. Đây là nơi bắt phần lớn bug logic, nên làm nhiều.

Vì sao vừa phải integration: integration test cắm DB/component thật nên bắt được loại bug mà unit không thấy — sai cú pháp SQL, hành vi transaction, vi phạm constraint, mapping ORM. Nhưng nó chậm hơn (phải dựng DB) và phức tạp hơn để setup, nên chỉ dùng cho các luồng có giá trị tích hợp thật, không phủ mọi nhánh ở đây.

Vì sao ít E2E: E2E chạy cả hệ thống qua nhiều service nên cho confidence cao nhất ("luồng đặt hàng từ đầu tới cuối hoạt động"), nhưng cũng chậm nhất, brittle nhất (một thay đổi nhỏ ở UI/flow làm đỏ), và khi đỏ thì khó biết lỗi ở đâu trong 8 service. Nên chỉ dành cho vài critical path.

🔧 Counter-example — inverted pyramid (kim tự tháp ngược)

Nếu đảo ngược — 50% là E2E: CI mất hàng giờ → feedback chậm → dev bỏ chạy test cục bộ; không kham nổi nhiều E2E nên ít test tổng → bug lọt; E2E flaky (đỏ ngẫu nhiên do mạng/timing) làm đội mất niềm tin vào test, bắt đầu "retry cho xanh" — che mất bug thật. Quy tắc vàng: đẩy mỗi assertion xuống tầng thấp nhất bắt được bug đó. Một quy tắc tính giá nên test bằng unit; đừng dựng cả flow E2E chỉ để kiểm "giảm 10% có đúng không".

2.3 Unit test — "isolation" nghĩa là gì CORE ★

Unit test kiểm một đơn vị logic (thường là một class) trong cô lập với phần còn lại của hệ thống. "Cô lập" đạt được bằng mock: thay mọi dependency thật (repository, external API, clock) bằng một đối tượng giả mà bạn lập trình để trả về giá trị định trước. Nhờ đó test không chạm DB/mạng → nhanh và deterministic, và bạn kiểm soát hoàn toàn input để test từng nhánh.

Cấu trúc chuẩn là AAA: Arrange (dựng mock, set giá trị trả về), Act (gọi method cần test), Assert (kiểm kết quả). Mấu chốt: chỉ mock dependency, KHÔNG mock class đang test — nếu mock luôn OrderService thì test chạy logic của mock chứ không phải của OrderService, vô nghĩa.

Một sai lầm phổ biến nữa là test code của framework. Đừng viết test kiểm "Spring có inject bean không" hay "Hibernate có lưu được không" — đó là việc của vendor, họ đã test rồi. Unit test của bạn chỉ nên kiểm logic nghiệp vụ của bạn.

✅ Test logic của OrderService (behavior)

@Test
void appliesDiscountToOrderTotal() {
    var repo    = mock(OrderRepository.class);   // mock dependency
    var pricing = mock(PricingService.class);
    when(pricing.discountFor(1000)).thenReturn(50);

    var service = new OrderService(repo, pricing); // class THẬT đang test
    Order o = service.place(1000);

    assertEquals(950, o.netTotal());   // assert KẾT QUẢ (behavior)
    verify(repo).save(any());
}

❌ Test framework (vô nghĩa)

@SpringBootTest                       // nạp cả context chỉ để...
@Test
void orderServiceIsAutowired() {
    assertNotNull(orderService);      // ...test Spring DI, không test logic
}
Đừng quên error path

Phần lớn bug production trốn ở nhánh lỗi, không phải happy path. Hãy test các khẳng định kiểu "khi pricing fail thì repository.save KHÔNG được gọi" (verify(repo, never()).save(any())) — đảm bảo không lưu một phần dữ liệu hỏng. Code đầy đủ ở 3.1.

2.4 Integration test: Testcontainers vs H2 CORE ★

Integration test kiểm tương tác giữa nhiều thành phần với một DB thật — để bắt loại bug mà unit (vốn mock DB) không bao giờ thấy: SQL sai cú pháp, hành vi transaction/rollback, vi phạm constraint, mapping ORM lệch. Câu hỏi là dùng DB nào trong test.

Cám dỗ là dùng H2 — một DB in-memory chạy ngay trong JVM, không cần Docker, khởi động tức thì. Nhưng H2 chỉ mô phỏng PostgreSQL qua "compatibility mode", không phải PostgreSQL thật. Hệ quả: nhiều thứ hành xử khác đi và bug bị che — test xanh nhưng prod đỏ. Testcontainers giải bằng cách spin một container PostgreSQL thật (đúng version prod) trong Docker khi test chạy, rồi dọn sau khi xong. Chậm hơn H2 vài giây nhưng phản ánh đúng production.

Khác biệtH2 (compatibility mode)PostgreSQL thật (Testcontainers)
Native query / dialectCú pháp khác → query đúng-prod có thể fail, query sai-prod có thể passĐúng như prod
JSONB, array, window functionHỗ trợ hạn chế hoặc hành vi khácĐầy đủ
INSERT ... ON CONFLICT (upsert)Không có / cú pháp khác (H2 dùng MERGE)Như prod
Constraint / unique violationLoại exception & message khácĐúng DataIntegrityViolationException như prod

Mẹo tốc độ: Testcontainers cho phép tái dùng một container singleton cho cả test suite (thay vì dựng lại mỗi class) và chạy các test song song — nhờ đó rút thời gian từ 45 phút xuống ~10 phút mà vẫn dùng DB thật.

L1 — Biết tên

Biết "Testcontainers chạy DB thật trong Docker". Symptom hiểu giả: vẫn dùng H2 cho tốc độ, không nêu được khác biệt cụ thể.

L2 — Explain được

Nêu được H2 compatibility mode khác PostgreSQL ở: native query, JSONB, window function, ON CONFLICT, hành vi constraint. Giải thích vì sao "pass với H2 = false confidence" (bug bị che tới prod).

L3 — Apply được

Thiết kế suite: Testcontainers cho repository/integration, reuse container singleton + chạy song song để nhanh, tách stage CI (unit trước cho feedback, integration sau), dùng cùng version PostgreSQL với prod.

2.5 Contract testing (Pact) — chặn breaking change SUPPLEMENTARY

Vấn đề: trong microservices, OrderService (consumer) gọi NotificationService (provider) và kỳ vọng một cấu trúc JSON cụ thể. Khi provider đổi API (đổi tên field, xoá field), consumer vỡ — nhưng unit test mỗi bên đều xanh (vì đã mock lẫn nhau), và chỉ phát hiện khi deploy cả hai vào môi trường integration, lúc đó đã muộn và khó truy lỗi.

Contract testing giải bằng cách biến "kỳ vọng" thành một hợp đồng kiểm được tự động, theo mô hình consumer-driven: consumer viết ra "khi tôi gọi /notify với payload X, tôi mong response Y"; công cụ (Pact) ghi kỳ vọng này thành một file contract. Sau đó ở phía provider, một test nạp file contract và verify rằng provider thật sự trả đúng Y. Nếu provider lỡ đổi API phá vỡ kỳ vọng, test verify của provider sẽ đỏ ngay trong CI của provider — trước khi merge, không cần chạy cả hai service cùng lúc.

Consumer (OrderService): "gọi /notify {customerId, message} → mong {status:'sent'}"
   → Pact ghi file contract
Provider (NotificationService): test nạp contract → verify provider trả đúng {status:'sent'}
   → provider đổi field 'status' → 'state' → test verify ĐỎ trong CI provider → chặn merge
Giá trị cốt lõi

Contract test bắt breaking change sớm (CI từng service) và rẻ (không cần dựng cả hệ). Đây chính là thứ ngăn "API đổi, consumer vỡ âm thầm" ở phần 1. Code đầy đủ ở 3.4.

2.6 @SpringBootTest slices — vì sao nhanh hơn CORE ★

Khi bạn dùng @SpringBootTest đầy đủ, Spring nạp toàn bộ application context — mọi bean, mọi config, kết nối DB, web server... — chỉ để test một controller. Việc nạp context này tốn vài giây mỗi class test, cộng dồn thành phút. Slice annotation giải bằng cách chỉ nạp phần context cần thiết cho loại test đó:

SliceChỉ nạp gìĐể test cái gì
@WebMvcTestTầng web: controller + MockMvc + Jackson + validation (service được mock)routing, JSON mapping, validation, error handling — KHÔNG đụng service/DB
@DataJpaTestTầng JPA: entity, repository, Hibernate + DB (Testcontainers)query, mapping, constraint — KHÔNG nạp controller
@JsonTestChỉ Jacksonserialize/deserialize JSON

Vì mỗi slice nạp ít bean hơn nhiều so với full context, nó khởi động nhanh gấp khoảng 10 lần — và còn giúp test tập trung: @WebMvcTest chỉ thấy tầng web nên test của bạn buộc phải kiểm đúng thứ thuộc tầng web, không lẫn lộn.

2.7 Behavior vs implementation — gốc của test brittle CORE ★

Một test brittle (giòn) là test đỏ khi bạn refactor mà hành vi không đổi. Nguyên nhân số một: test kiểm cách code làm (implementation) thay vì kết quả code tạo ra (behavior). Ví dụ test verify(service).setStatus("PAID") đang khẳng định "phải gọi đúng method setStatus" — nếu bạn refactor đổi tên method nội bộ hoặc đổi cách cập nhật trạng thái (dù kết quả vẫn là order PAID), test sẽ đỏ. Đây là lý do "thêm 1 feature làm vỡ 50 test" ở phần 1: 50 test đó khoá chặt vào chi tiết nội bộ.

Cách đúng: assert kết quả quan sát được từ bên ngoài — trạng thái cuối, giá trị trả về, hiệu ứng phụ quan trọng. Test order.status() == PAID không quan tâm bạn đạt được nó bằng cách nào, nên cho phép refactor tự do mà vẫn bảo vệ hành vi. Quy tắc: test nên là rào chắn cho refactor, không phải xiềng xích.

3.1 Unit test với Mockito — đầy đủ CORE ★

class OrderServiceTest {
    private OrderService service;
    private OrderRepository repository;
    private PricingService pricing;
    private InventoryService inventory;

    @BeforeEach
    void setUp() {
        repository = mock(OrderRepository.class);
        pricing    = mock(PricingService.class);
        inventory  = mock(InventoryService.class);
        service    = new OrderService(repository, pricing, inventory);
    }

    @Test
    void placesOrderAndSavesWithDiscount() {           // happy path
        when(inventory.reserve("SKU1", 2)).thenReturn(true);
        when(pricing.discountFor(1000)).thenReturn(50);
        Order result = service.place("C1", "SKU1", 2, 1000);
        assertEquals(950, result.netTotal());          // assert behavior
        verify(repository).save(argThat(o -> o.netTotal() == 950));
    }

    @Test
    void rejectsWhenStockUnavailable() {                // error path
        when(inventory.reserve("SKU1", 2)).thenReturn(false);
        assertThrows(OutOfStockException.class,
            () -> service.place("C1", "SKU1", 2, 1000));
        verify(repository, never()).save(any());        // CRITICAL: không lưu một phần
    }

    @Test
    void doesNotSaveWhenPricingFails() {                // error path
        when(inventory.reserve(any(), anyInt())).thenReturn(true);
        when(pricing.discountFor(anyLong())).thenThrow(new PricingServiceException());
        assertThrows(PricingServiceException.class,
            () -> service.place("C1", "SKU1", 2, 1000));
        verify(repository, never()).save(any());
    }
}

3.2 Integration test với Testcontainers CORE ★

@Testcontainers                                    // tự quản lifecycle container
@SpringBootTest
class OrderIntegrationTest {
    @Container
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:15");

    @DynamicPropertySource                          // trỏ datasource vào container
    static void props(DynamicPropertyRegistry reg) {
        reg.add("spring.datasource.url", postgres::getJdbcUrl);
        reg.add("spring.datasource.username", postgres::getUsername);
        reg.add("spring.datasource.password", postgres::getPassword);
    }

    @Autowired OrderService service;
    @Autowired OrderRepository orderRepo;
    @Autowired AuditLogRepository auditRepo;
    @Autowired TestEntityManager em;

    @Test
    void placingOrderWritesAuditEntry() {           // test cả flow: service → repo → DB → audit
        Customer c = em.persistAndFlush(new Customer("C1"));
        Order o = service.place(c.getId(), "SKU1", 2, 1000);
        assertThat(orderRepo.findById(o.getId())).isPresent();
        assertThat(auditRepo.findByOrderId(o.getId())).hasSize(1);
    }

    @Test
    void rollsBackOnUniqueConstraintViolation() {   // H2 KHÔNG bắt đúng case này
        em.persistAndFlush(new Customer("DUP"));
        assertThrows(DataIntegrityViolationException.class,
            () -> em.persistAndFlush(new Customer("DUP")));   // unique constraint thật
        em.clear();
        assertThat(orderRepo.findByCustomerCode("DUP")).hasSize(0);
    }
}

3.3 @DataJpaTest · @WebMvcTest SUPPLEMENTARY

@DataJpaTest                                       // chỉ nạp JPA + DB (Testcontainers)

5. 3-tier interview framework

Tầng 1 — SurfaceJunior gate

Q: Giải thích test pyramid.

Nhiều unit (nhanh, deterministic, mock dependency) : một số integration (chậm hơn, DB thật, bắt bug SQL/transaction) : ít E2E (chậm nhất, brittle, toàn hệ thống). ~70:20:10. Lý do: đẩy mỗi assertion xuống tầng thấp nhất bắt được bug → feedback nhanh + chi phí CI hợp lý. Inverted pyramid làm CI chậm, flaky, khó debug.

Tầng 2 — Deep DiveSenior gate

Q: Test integration mất 45 phút, tối ưu xuống 10 phút.

Testcontainers reuse singleton container (không tạo lại mỗi class) + chạy song song; dùng @DataJpaTest/@WebMvcTest slice thay full @SpringBootTest (nạp ít context); tách CI stage (unit trước cho feedback nhanh, integration sau); fixtures dùng chung tránh INSERT lặp; chỉ critical path mới integration, còn lại unit.

Tầng 3 — ArchitectureSenior+

Q: 5 microservice, 80% coverage vẫn P1. Thiết kế chiến lược chất lượng.

Coverage ≠ quality. Thêm contract test (Pact — bắt breaking API sớm), mutation testing (đo test có thật sự bắt bug), chaos testing (giết pod ngẫu nhiên — verify resilience), canary deploy (1% traffic bắt bug trước full rollout), observability (không test hết được mọi scenario thì quan sát prod), property-based test (sinh 10K input tìm edge case). Coverage chỉ là sàn.

🔧 Lab bắt buộc — viết & chứng minh (2 bài)

Lab 1 — Brittle vs behavior. Viết một test check verify(service).setStatus("PAID") (implementation). Refactor đổi tên method nội bộ → test đỏ dù hành vi không đổi. Viết lại test chỉ assert kết quả quan sát được (order.status() == PAID), refactor lại → test vẫn xanh. Tiêu chí: giải thích vì sao test thứ hai cho phép refactor tự do.

Lab 2 — H2 che bug. Viết một native query/upsert dùng cú pháp PostgreSQL (INSERT ... ON CONFLICT hoặc jsonb). Chạy test với H2 → pass nhầm hoặc lỗi khó hiểu; chạy với Testcontainers PostgreSQL → phản ánh đúng prod. Tiêu chí: chỉ ra đúng một câu SQL mà H2 và PostgreSQL hành xử khác nhau.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. Vì sao "80% coverage" không đảm bảo ít bug? Coverage đo gì, không đo gì? Mutation testing đo gì?
  2. Tỉ lệ pyramid và lý do mỗi tầng nhiều/ít (theo tốc độ/confidence/phạm vi)? Inverted pyramid hại ở đâu?
  3. Cho 2 khác biệt cụ thể H2 vs PostgreSQL khiến bug lọt. Vì sao gọi là "false confidence"?
  4. Contract test (Pact) chặn breaking change API thế nào, mà không cần chạy cả 2 service?
  5. "Test behavior, không test implementation" nghĩa là gì? Cho ví dụ một test brittle và bản sửa.
Bí câu nào = quay lại đúng section. Trôi cả 5 không nhìn tài liệu = nắm chắc testing mức senior.
