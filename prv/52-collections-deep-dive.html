Collections Deep Dive — Internals + Production Choice

Junior dùng ArrayList + HashMap cho mọi thứ. Khi data lên 1M+, latency tăng 100×, không biết tại sao. File này bóc internals (B-tree node, hash collision, treeify) + recipe chọn đúng collection cho từng access pattern.

CORE ★ 1 tuần đọc + JMH benchmark Domain: E-commerce / SaaS

⚠️ Cảnh báo "hiểu giả"

  • Biết HashMap O(1) — nhưng không biết khi nào degrade thành O(log n) (Java 8+ treeify khi bucket > 8)
  • Dùng LinkedList vì nghĩ "insert nhanh hơn ArrayList" — không biết LinkedList iterate cache-unfriendly, thường chậm hơn ArrayList trong thực tế
  • Biết ConcurrentHashMap thread-safe — không phân biệt với Collections.synchronizedMap() hay HashMap + lock external
  • Biết equals/hashCode contract — quên override khi dùng POJO làm Map key → bug silent (object "biến mất" khỏi Map)
🚨 3 cấp — bạn đang ở đâu?

L1 (Recognize): dùng ArrayList + HashMap cho mọi case. Không nghĩ về complexity.
L2 (Explain): chọn collection theo access pattern. Hiểu hash collision, load factor, treeify threshold. Implement equals/hashCode đúng.
L3 (Apply): JMH benchmark trước decision. Tune initialCapacity + loadFactor. Biết khi nào ConcurrentHashMap, khi nào LongAdder, khi nào CopyOnWriteArrayList.

1. Bài toán thực tế — Sai collection = bug performance ẩn CORE ★

📖 Câu chuyện #1 — "Service chậm dần sau 4 giờ"

Bối cảnh: Recommendation service. Sau restart 4h, p99 latency từ 50ms → 800ms. Restart fix tạm. CPU + heap bình thường.

// Code junior viết
private Map<String, Product> cache = new HashMap<>();

public Product get(String customSku) {
    return cache.get(customSku);
}

// Mỗi giờ, async job add 50K product mới vào cache

Root cause: customSku là object custom, hashCode trả về cùng giá trị cho 80% sku → mọi entry rơi vào 1 bucket → Java 8+ treeify thành Red-Black tree → lookup O(log n) thay vì O(1). Sau 200K entry, bucket có 160K node, lookup 17 step thay vì 1.

Fix: Override hashCode() đúng — distribute đều theo các field. JFR profile show HashMap.getNode + TreeNode.find chiếm 60% CPU.

📖 Câu chuyện #2 — "LinkedList chậm hơn ArrayList trên 1M items"

Bối cảnh: Junior chọn LinkedList để "insert nhanh". Benchmark: iterate 1M item sum total.

ArrayList:  18 ms
LinkedList: 142 ms  ← chậm 8×!

Root cause: ArrayList = mảng liên tiếp trong memory → CPU cache prefetch hiệu quả. LinkedList = nodes scattered khắp heap → mỗi node.next = cache miss → fetch từ RAM (100× chậm hơn cache).

Bài học: "Insert nhanh" chỉ đúng khi insert vào giữa với iterator đã ở đó. Hầu hết case backend không cần — ArrayList tốt hơn cho mọi access pattern khác.

📖 Câu chuyện #3 — "Object 'biến mất' khỏi HashSet"
class Order {
    private String id;
    private OrderStatus status;
    // equals + hashCode dùng cả id + status
}

Set<Order> orders = new HashSet<>();
Order o = new Order("ORD_1", PENDING);
orders.add(o);
o.setStatus(PAID);            // ← mutate field dùng trong hashCode
orders.contains(o);           // false! object "biến mất"

Root cause: hashCode thay đổi sau khi add → object nằm sai bucket → contains tìm bucket mới, không có.

Rule: field dùng trong hashCode() phải immutable. Hoặc dùng chỉ id (không bao giờ đổi) làm key.

2. List Implementations — Internals CORE ★

2.1 ArrayList — Dynamic Array

[A][B][C][D][_][_][_][_]    ← array, size=4, capacity=8
                ^^^^^^^^^^      empty slots

Internals: mảng Object[]. Khi full → tạo array mới capacity × 1.5 → copy. add() amortized O(1), worst O(n) khi grow.

Khi nào nhanh: random access get(i) O(1), iterate (cache-friendly), append cuối.

Khi nào chậm: add(0, x) hoặc remove(0) = O(n) vì phải shift cả mảng.

2.2 LinkedList — Doubly Linked List

[null|A|→] ⇄ [←|B|→] ⇄ [←|C|null]

Internals: mỗi node = 3 pointer (prev, next, value) = 24 byte overhead/element (so với 4 byte ArrayList).

Khi nào nhanh: insert/remove ở giữa với iterator đã ở đó.

Khi nào chậm: random access get(i) O(n) — phải traverse. Iterate cache-unfriendly.

❌ Khi nào KHÔNG dùng LinkedList
  • Dev không biết khi nào dùng — default ArrayList. 95% case ArrayList thắng.
  • Cần FIFO queue — dùng ArrayDeque (mảng tròn) thay vì LinkedList. ArrayDeque nhanh hơn 2-5×.
  • Memory-sensitive — 1M Long element: ArrayList ~12MB, LinkedList ~32MB.

Joshua Bloch trong Effective Java: "Tôi đã viết LinkedList. Đừng dùng nó."

2.3 CopyOnWriteArrayList — Thread-safe Snapshot

Mỗi add/remove tạo array MỚI (copy toàn bộ). Đọc lock-free. Use case: read-heavy, write-rare (event listener, config).

KHÔNG dùng cho write-heavy — copy 1M element/write sẽ chết.

3. Map Implementations — Internals CORE ★

3.1 HashMap — Hash Table với Treeify (Java 8+)

Index 0: [K1|V1|→] [K9|V9|→null]    ← linked list (chain)
Index 1: null
Index 2: [K2|V2|→null]
Index 3: TreeNode root → RB-tree    ← treeify khi bucket > 8
Index 4: null
...
Index 15:[K8|V8|→null]

Internals:

  • Default capacity 16 buckets, load factor 0.75 → resize khi size > 12
  • Resize = double capacity, rehash all entries — cực đắt
  • Treeify (Java 8+): khi bucket có > 8 nodes + tổng size > 64 → convert linked list → Red-Black tree. Lookup từ O(n) bucket → O(log n).
  • Untreeify: khi bucket < 6, convert ngược.

3.2 Khi nào HashMap fail

🚨 3 nguyên nhân HashMap chậm bất ngờ
  1. Bad hashCode: trả cùng số cho nhiều key → tất cả vào 1 bucket. Treeify giảm xuống O(log n) nhưng vẫn chậm hơn O(1).
  2. Quá nhiều resize: không set initialCapacity. Add 1M entry = resize ~16 lần, rehash từng phần.
  3. Mutable key: object thay đổi hashCode sau khi put → "biến mất" khỏi map.

3.3 Pre-size HashMap — Production rule

// ❌ Default 16, resize 16 lần khi đạt 1M
Map<String, Order> cache = new HashMap<>();

// ✅ Pre-size theo expected size / load factor
Map<String, Order> cache = new HashMap<>(1_300_000);
// Java tự round lên 2^n = 2097152, capacity 2M → không resize

3.4 TreeMap — Red-Black Tree

Internals: self-balancing BST. Mọi operation O(log n). Keys luôn sorted.

Khi dùng: cần iterate theo thứ tự, range query (subMap(from, to), headMap, tailMap), ceiling/floor key.

Khi KHÔNG dùng: không cần ordering → HashMap nhanh hơn 5-10×.

3.5 LinkedHashMap — Insertion Order

HashMap + doubly linked list track thứ tự insert (hoặc access order). Iterate predictable. Use case: LRU cache, JSON output cần thứ tự field.

// LRU cache 100 entry
Map<K,V> lru = new LinkedHashMap<>(16, 0.75f, true) {  // accessOrder=true
    protected boolean removeEldestEntry(Map.Entry<K,V> e) {
        return size() > 100;
    }
};

3.6 ConcurrentHashMap — Production Thread-safe

Java 8+ dùng CAS + node-level locking. KHÔNG còn segment lock như Java 7. Read gần như lock-free, write lock chỉ 1 bucket.

API quan trọng:

map.computeIfAbsent(key, k -> loadFromDb(k));    // atomic, không double-load
map.compute(key, (k, v) -> v == null ? 1 : v + 1);  // atomic update
map.merge(key, 1, Integer::sum);                    // atomic counter
// KHÔNG dùng: get + check null + put → race condition
⚠️ ConcurrentHashMap vs alternatives
CầnDùng
Thread-safe map generalConcurrentHashMap
Counter concurrentConcurrentHashMap + LongAdder (high contention) hoặc AtomicLong (low)
Wrap HashMap thuầnKHÔNG dùng Collections.synchronizedMap — lock toàn map, kém ConcurrentHashMap 10×
Cache với evictionCaffeine (built-in TTL, max size, async refresh)

💡 Top-K pattern với PriorityQueue
// Top 5 đơn hàng mới nhất từ stream 1M element
// O(n log k) thay vì O(n log n) nếu sort all
PriorityQueue<Order> heap = new PriorityQueue<>(
    Comparator.comparing(Order::getCreatedAt));
for (Order o : orders) {
    heap.offer(o);
    if (heap.size() > 5) heap.poll();   // bỏ min, giữ 5 lớn nhất
}

6. JMH Benchmark — Đo trước khi optimize ADVANCED

JMH (Java Microbenchmark Harness) = framework benchmark chuẩn của OpenJDK. Loại bỏ JIT warmup, dead code elimination, OS noise.

@BenchmarkMode(Mode.AverageTime)
@OutputTimeUnit(TimeUnit.MICROSECONDS)
@State(Scope.Benchmark)
public class MapBenchmark {

    @Param({"100", "10000", "1000000"})
    private int size;

    private HashMap<Integer, String> hashMap;
    private TreeMap<Integer, String> treeMap;

    @Setup
    public void setup() {
        hashMap = new HashMap<>(size);
        treeMap = new TreeMap<>();
        for (int i = 0; i < size; i++) {
            hashMap.put(i, "v" + i);
            treeMap.put(i, "v" + i);
        }
    }

    @Benchmark
    public String hashMapGet() {
        return hashMap.get(size / 2);
    }

    @Benchmark
    public String treeMapGet() {
        return treeMap.get(size / 2);
    }
}

Rule: KHÔNG bao giờ optimize collection mà không benchmark. System.nanoTime() manual = sai 90% case vì JIT, warmup, GC noise.

7. 7 Anti-Patterns Phổ Biến CORE ★

❌ #1 — Mutate field trong hashCode

Mutate field dùng trong hashCode → object lạc bucket → "biến mất". Field hashCode phải immutable, hoặc dùng id-only.

❌ #2 — Quên override hashCode khi override equals
// equals đúng, hashCode mặc định (Object.hashCode) sai
// → 2 object "equal" có hashCode khác → vào 2 bucket khác
@Override public boolean equals(Object o) { ... }
// quên @Override public int hashCode() { ... }

Contract: a.equals(b) → a.hashCode() == b.hashCode(). Vi phạm = HashMap/HashSet fail silent.

❌ #3 — ConcurrentModificationException khi remove during iterate
// ❌
for (Order o : orders) {
    if (o.isExpired()) orders.remove(o);  // CME!
}

// ✅ Iterator.remove() hoặc removeIf
orders.removeIf(Order::isExpired);

// ✅ Iterator explicit
Iterator<Order> it = orders.iterator();
while (it.hasNext()) {
    if (it.next().isExpired()) it.remove();
}
❌ #4 — Arrays.asList() return fixed-size
List<String> list = Arrays.asList("a", "b");
list.add("c");   // UnsupportedOperationException!

// ✅ Wrap thành ArrayList
List<String> list = new ArrayList<>(Arrays.asList("a", "b"));
❌ #5 — Synchronized wrapper thay vì Concurrent
// ❌ Lock toàn map mỗi op
Map<K,V> m = Collections.synchronizedMap(new HashMap<>());

// ✅ Lock từng bucket
Map<K,V> m = new ConcurrentHashMap<>();

Benchmark: dưới contention, ConcurrentHashMap nhanh hơn 5-10×.

❌ #6 — Boxing trong hot loop
// ❌ Auto-box Integer mỗi entry
Map<Integer, Integer> counter = new HashMap<>();

// ✅ Dùng IntIntHashMap (Eclipse Collections) hoặc int[] nếu key range hạn chế
❌ #7 — Default HashMap không pre-size
// ❌ Add 1M entry → resize 16 lần
Map<String, Order> cache = new HashMap<>();
orders.forEach(o -> cache.put(o.getId(), o));

// ✅ Pre-size
Map<String, Order> cache = new HashMap<>(orders.size() * 4 / 3);

8. Decision Tree — Chọn Collection CORE ★

Cần Map (key-value)?
├── Cần ordering theo key sorted? → TreeMap
├── Cần insertion order / LRU? → LinkedHashMap
├── Thread-safe? → ConcurrentHashMap
├── Key là enum? → EnumMap (nhanh + ít memory)
└── Default → HashMap (pre-size!)

Cần Set (unique)?
├── Sorted? → TreeSet
├── Insertion order? → LinkedHashSet
├── Thread-safe read-heavy? → CopyOnWriteArraySet
├── Element là enum? → EnumSet
└── Default → HashSet

Cần List?
├── Random access + iterate? → ArrayList (gần như luôn)
├── FIFO/LIFO queue? → ArrayDeque
├── Thread-safe read-heavy? → CopyOnWriteArrayList
├── LinkedList? → gần như không bao giờ. Default ArrayList.

Cần Queue?
├── FIFO/LIFO general? → ArrayDeque
├── Lấy min/max (top-K)? → PriorityQueue
├── Producer-consumer fixed? → ArrayBlockingQueue
├── Producer-consumer unbounded? → LinkedBlockingQueue
└── High-throughput non-blocking? → ConcurrentLinkedQueue

🎯 3 Cấp Độ Mastery CORE ★

Map choice + tuning

L1 — Recognize

Dùng HashMap cho mọi case. Không pre-size. Không hiểu hash collision.

Symptom: service chậm dần sau hours, không biết nguyên nhân.

L2 — Explain

Phân biệt HashMap/TreeMap/LinkedHashMap/Concurrent. Pre-size khi biết expected size. Hiểu treeify, load factor. Implement equals/hashCode đúng.

L3 — Apply

JMH benchmark trước decision. Tune initialCapacity + loadFactor cho production. Biết khi nào LongAdder vs AtomicLong vs ConcurrentHashMap counter. Dùng Caffeine cho cache.

equals / hashCode contract

L1

Biết phải override cả 2. Dùng IDE generate. Không hiểu contract sâu.

L2

Hiểu contract: a.equals(b) → a.hashCode() == b.hashCode(). Field hashCode phải immutable. Dùng Java 17 record khi possible.

L3

Design POJO làm cache key — immutable + hashCode pre-compute (cache hashCode trong constructor). Recipe debug "object biến mất khỏi HashSet" là phản xạ.

🚨 Symptom → Debug Recipes CORE ★

🚨 Recipe #1 — "HashMap lookup chậm dần"

Symptom: service cold start lookup 1ms. Sau vài giờ load 50ms. Heap OK, GC OK.

# JFR profile show HashMap.getNode + TreeNode.find chiếm CPU
# → Bad hashCode hoặc bucket distribution lệch

# Check hashCode distribution
jcmd PID GC.class_histogram | grep -i node
# TreeNode count cao bất thường → bucket bị treeify

# Fix: improve hashCode để distribute đều
# Hoặc pre-size map đủ lớn để giảm collision rate
🚨 Recipe #2 — "OutOfMemoryError nhưng heap không full"

Symptom: OOM với Requested array size exceeds VM limit.

Root cause: Resize HashMap/ArrayList rất lớn. Resize ArrayList 1B element = cần array size 1.5B (capacity × 1.5) — vượt Integer.MAX_VALUE.

Fix: partition data thành multiple collection nhỏ, hoặc dùng external storage (DB, file).

🚨 Recipe #3 — "ConcurrentModificationException trong production"

Symptom: Stack trace java.util.ConcurrentModificationException từ ArrayList iterator.

Debug:

  1. Có thread khác modify list không? → đổi sang CopyOnWriteArrayList hoặc synchronized block
  2. Cùng thread modify trong iterator? → dùng removeIf hoặc Iterator.remove()
  3. Có recursive call modify? → break thành 2 pass (collect, then remove)
🚨 Recipe #4 — "GC pause cao bất thường khi dùng HashMap"

Symptom: GC log show Full GC mỗi 30 phút, mỗi lần pause 5-10s. Heap dump show HashMap$Node chiếm 60% old gen.

Root cause: Resize HashMap promote Node objects sang old gen → mỗi resize tạo wave Node mới → fragmentation old gen.

Fix: Pre-size đủ lớn để chỉ resize 1 lần lúc init. Hoặc dùng Caffeine (off-heap option).

🎤 Interview Framework CORE ★

Tier 1 — Surface

Q: ArrayList vs LinkedList — chọn cái nào?

Default ArrayList. ArrayList nhanh hơn cho random access (O(1) vs O(n)) và iterate (cache-friendly). LinkedList chỉ thắng khi insert/remove ở giữa với iterator đã ở đó — case hiếm trong backend. Joshua Bloch khuyên "đừng dùng LinkedList". Cần FIFO queue → ArrayDeque.

Tier 2 — Deep Dive

Q: HashMap trong Java 8 khác Java 7 thế nào?

Java 7: linked list trong bucket. Worst case O(n) khi nhiều collision. Java 8: treeify — khi bucket có > 8 nodes và size > 64, convert linked list → Red-Black tree. Worst case xuống O(log n). Untreeify khi bucket < 6. Default capacity vẫn 16, load factor 0.75.

Tier 3 — Architecture

Q: Design cache cho service 1M req/s, 100K key hot

  1. Don't use raw HashMap: không có eviction, OOM tiềm năng
  2. Caffeine: W-TinyLFU eviction, async refresh, max size + TTL, stat dashboard
  3. Pre-size ~1.3× max entries để tránh resize trong production
  4. Concurrent: Caffeine built-in thread-safe (CAS-based)
  5. Distributed? nếu cần share giữa instance → Redis layer 2, Caffeine layer 1 (multi-tier)
  6. Eval: hit rate dashboard, p99 latency, capacity plan growth

🧠 Feynman Test — 5 câu

  1. Treeify trong HashMap Java 8 — khi nào trigger, lợi gì?
  2. Tại sao mutate field dùng trong hashCode = bug?
  3. ArrayList vs LinkedList: tại sao ArrayList iterate nhanh hơn dù cùng O(n)?
  4. ConcurrentHashMap vs synchronizedMap — khác cơ chế lock?
  5. Top-K với PriorityQueue: tại sao O(n log k) tốt hơn sort all O(n log n)?
Pass: 5/5 trong < 10 phút.
