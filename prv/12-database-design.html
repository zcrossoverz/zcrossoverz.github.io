Tổng Quan — Vì Sao Schema Design Critical

Senior khác junior ở 2 điểm cốt lõi với database:

  1. Predict cost trước khi schema deploy: senior nhìn schema biết "query này sẽ chết khi 100M rows". Junior phải đợi sập mới biết.
  2. Trade-off awareness: senior biết "denormalize tăng read speed, mất consistency, cần invalidation strategy". Junior chỉ thấy "đỡ JOIN cho nhanh".
📜 Lịch sử normalization — Từ Codd đến NoSQL
1970 — Edgar F. Codd (IBM Research) publish paper "A Relational Model of Data for Large Shared Data Banks" — introduce relational model. Trước đó: hierarchical (IMS) và network (CODASYL) database, data redundancy cực cao.

1971-1974 — Codd define 1NF, 2NF, 3NF (Boyce-Codd Normal Form 1974). Mỗi level giảm 1 loại anomaly: insert anomaly, update anomaly, delete anomaly.

1979-1980 — Oracle (Oracle v2), IBM DB2 thương mại hóa relational DB. Normalization thành standard practice.

1990s-2000s — OLAP (Online Analytical Processing) emerge, denormalized schema (star schema, snowflake) cho data warehouse. Trade-off: read-heavy reporting cần denormalized, write-heavy OLTP cần normalized.

2009-2010 — Eric Brewer formulate CAP theorem (2000 conjecture, 2002 proof bởi Gilbert & Lynch). NoSQL movement (MongoDB, Cassandra, DynamoDB) intentionally denormalize cho scale.

2013 — Daniel Abadi extend CAP thành PACELC: even when no partition, có trade-off giữa Latency và Consistency.

2020s — Hybrid systems (Spanner, CockroachDB) push boundary CAP, achieve "external consistency" với atomic clock + Paxos. Nhưng cho 99% app, classical relational DB + read replica vẫn đúng.

Insight: Normalization là default đúng. Denormalization là optimization có cost. Đừng denormalize trước khi đo performance — phổ biến nhất là junior denormalize "for performance" rồi tạo bug consistency.

1. Bài Toán Thực Tế — Order Platform Của ShopVN Sập Khi Scale

Sàn ShopVN có table orders tăng từ 100M rows (2024) → 500M rows (2026). Query simple từng take 100ms giờ take 30 giây. 3 engineer trong team đưa 3 giải pháp mâu thuẫn nhau:

💬 Cuộc tranh luận trong team
  • Engineer A (junior): "Add more RAM. Caching sẽ fix it."
  • Engineer B (mid): "Add more indexes. Không có index = full table scan."
  • Engineer C (senior): "Partition by date. Chúng ta keep 2 năm data, query thường chỉ recent 3 tháng."

3 đều có lý nhưng không ai hoàn toàn đúng. Câu trả lời thật sự là combination + biết trade-off.

Schema hiện tại
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    customer_id BIGINT NOT NULL,
    shop_id BIGINT NOT NULL,
    total_amount DECIMAL(15,2) NOT NULL,
    status VARCHAR(20),         -- PENDING/PROCESSING/SHIPPED/DELIVERED/CANCELLED
    payment_method VARCHAR(20),
    created_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP NOT NULL,
    shipping_address TEXT,
    customer_name VARCHAR(100),  -- ⚠️ denormalized
    customer_email VARCHAR(200), -- ⚠️ denormalized
    internal_notes TEXT,
    FOREIGN KEY (customer_id) REFERENCES customers(id),
    FOREIGN KEY (shop_id) REFERENCES shops(id)
);
-- 500M rows. ~2KB/row → 1TB table.
3 query patterns chính
-- Query A (80% traffic): order history của 1 customer, recent 30 days
SELECT * FROM orders
WHERE customer_id = ? AND created_at > DATE_SUB(NOW(), INTERVAL 30 DAY)
ORDER BY created_at DESC LIMIT 20;

-- Query B (15% traffic): doanh thu của 1 shop trong năm
SELECT shop_id, SUM(total_amount), COUNT(*)
FROM orders
WHERE shop_id = ? AND YEAR(created_at) = 2026
GROUP BY shop_id;

-- Query C (5% traffic): analytics — all orders last year
SELECT * FROM orders
WHERE created_at > DATE_SUB(NOW(), INTERVAL 365 DAY)
LIMIT 100000;
⏸ Pause & Predict — BẮT BUỘC dừng
Trước khi đọc tiếp, tự trả lời (viết ra giấy):
  1. Normalization: Có bug ngay trong schema không? Denormalized customer_name, customer_email sẽ gây vấn đề gì khi customer rename?
  2. Partitioning vs Indexing: Query A search 30 ngày trong 500M rows. Partition by created_at (monthly) hay composite index (customer_id, created_at) hiệu quả hơn? Vì sao?
  3. Read replica problem: Nếu add read replica + route reads sang đó, vấn đề gì xảy ra khi transaction write to primary rồi immediately query replica?
  4. Sharding trap: Nếu shard by customer_id, query B (theo shop_id) sẽ fail như thế nào? Nếu 1 shard slow, query timeout không?
  5. CAP for e-commerce: Banking chọn CP. E-commerce nên chọn CP hay AP? Vì sao? Edge case nào nguy hiểm nếu chọn sai?
Nếu trả lời được 4/5 → solid foundation, đọc lướt. Nếu <3 → đọc kỹ.

2. Normalization — 1NF, 2NF, 3NF CORE

★ CORE — Foundation. Junior thường skip đi vì "đơn giản", senior interview hỏi 100%. Hiểu cơ chế (anomaly nào được fix) quan trọng hơn memorize definition.

1NF — Atomic Values

Rule: mỗi cell chứa 1 value không thể chia nhỏ. Không có array, list, JSON với multi-value trong column.

❌ Không 1NF — Multi-value column
CREATE TABLE orders (
  id BIGINT,
  customer_id BIGINT,
  product_ids VARCHAR(500),  -- CSV: "P001,P003,P007"
  quantities VARCHAR(200)     -- CSV: "2,1,5"
);

-- Query khó: tìm orders chứa P003?
SELECT * FROM orders
WHERE product_ids LIKE '%P003%';
-- Index không help, full scan.
-- Bug: LIKE '%P003%' match cả 'P0030', 'P0031', ...
✅ 1NF — Atomic, separate table
CREATE TABLE orders (
  id BIGINT,
  customer_id BIGINT
);

CREATE TABLE order_items (
  order_id BIGINT,
  product_id VARCHAR(20),
  quantity INT,
  PRIMARY KEY (order_id, product_id),
  FOREIGN KEY (order_id) REFERENCES orders(id)
);

-- Query rõ ràng, indexable
SELECT * FROM orders o
JOIN order_items oi ON o.id = oi.order_id
WHERE oi.product_id = 'P003';
🎯 Vì sao 1NF matter?
  • Queryability: SQL designed cho atomic values. Multi-value columns biến SQL thành string parsing nightmare.
  • Index ineffective: B-Tree index trên CSV column không help cho "find rows containing X".
  • Type safety: quantity phải là INT. CSV string nhồi nhét types → bug khi parse.
  • Update khó: thay đổi 1 quantity = parse string, modify, serialize lại. Race condition risk.

2NF — Full Functional Dependency on PK

Rule: nếu PK là composite, mọi non-key column phải depend on toàn bộ PK, không phải subset. (1NF cộng thêm rule này = 2NF.)

❌ Không 2NF
CREATE TABLE order_items (
  order_id BIGINT,
  product_id VARCHAR(20),
  quantity INT,
  product_name VARCHAR(200),   -- ❌ depends only on product_id
  product_category VARCHAR(50), -- ❌ depends only on product_id
  PRIMARY KEY (order_id, product_id)
);

-- Problem: nếu product P001 rename
-- → phải UPDATE 1M+ rows order_items
-- → quên 1 row = data inconsistent
✅ 2NF
CREATE TABLE order_items (
  order_id BIGINT,
  product_id VARCHAR(20),
  quantity INT,
  PRIMARY KEY (order_id, product_id)
);

CREATE TABLE products (
  id VARCHAR(20) PRIMARY KEY,
  name VARCHAR(200),
  category VARCHAR(50)
);

-- Product rename: UPDATE 1 row in products
UPDATE products SET name = 'New Name'
WHERE id = 'P001';

3NF — No Transitive Dependency

Rule: non-key column không được depend on non-key column khác (transitive). Mọi non-key chỉ depend trực tiếp lên PK.

❌ Không 3NF
CREATE TABLE orders (
  id BIGINT PRIMARY KEY,
  customer_id BIGINT,
  shop_id BIGINT,
  shop_name VARCHAR(200),      -- ❌ depends on shop_id
  shop_city VARCHAR(50),       -- ❌ depends on shop_id
  shop_phone VARCHAR(20)       -- ❌ depends on shop_id
);

-- Shop "ABC Store" thay đổi phone:
-- Phải UPDATE 50K+ orders rows
-- Hoặc một số rows còn old phone → inconsistent
✅ 3NF
CREATE TABLE orders (
  id BIGINT PRIMARY KEY,
  customer_id BIGINT,
  shop_id BIGINT,
  FOREIGN KEY (shop_id) REFERENCES shops(id)
);

CREATE TABLE shops (
  id BIGINT PRIMARY KEY,
  name VARCHAR(200),
  city VARCHAR(50),
  phone VARCHAR(20)
);

-- Update shop info: UPDATE 1 row in shops
-- Orders tự reference qua shop_id, luôn current
🎯 3 anomalies mà 3NF fix
  1. Update anomaly: shop rename → phải update 50K rows. Forget 1 = inconsistent.
  2. Insert anomaly: thêm shop mới chưa có order → không thể insert vào orders table.
  3. Delete anomaly: xóa last order của shop → mất luôn shop info nếu chỉ ở orders table.
🧠 Misconception — "JOIN luôn chậm, denormalize tốt hơn"
"3NF có nhiều JOIN, denormalize bỏ JOIN sẽ luôn nhanh hơn"
2 vấn đề với mindset này: (1) JOIN với proper index thường rất nhanh (~1-5ms cho 2-3 table). PostgreSQL/MySQL optimizer giỏi với JOIN. (2) Denormalize tạo update overhead và data inconsistency risk. Trade off thường KHÔNG worth cho OLTP. Chỉ denormalize có chiến lược (xem section 3). Default: 3NF. Denormalize là exception, không phải rule.

3. Denormalization Có Chiến Lược CORE

★ CORE — Khi nào và làm sao break normalization an toàn.

Denormalization worth khi:

  • Read-heavy: column được read 1000× hơn write
  • Rarely changes: column update < 1×/tháng
  • Critical path latency: 5ms JOIN giảm xuống 1ms direct read tạo khác biệt UX
  • Có invalidation strategy: khi source change, có mechanism update denormalized copies

Pattern 1 — Denormalize "Stable" Column

-- Customer's home country rất ít thay đổi (move overseas hiếm)
-- Tax calculation cần country mỗi order → JOIN customers mỗi lần expensive

ALTER TABLE orders ADD COLUMN customer_country CHAR(2);
-- Khi insert order: copy customer.country
INSERT INTO orders (..., customer_country) VALUES (..., 'VN');

-- Khi customer move (rare):
BEGIN;
UPDATE customers SET country = 'US' WHERE id = ?;
-- Async job update orders eventually (acceptable lag cho cold orders)
COMMIT;

Pattern 2 — CQRS Read Model (xem #17)

-- Write side: 3NF orders + customers
-- Read side: denormalized order_view với customer info joined-in

CREATE MATERIALIZED VIEW order_view AS
SELECT
  o.id, o.total_amount, o.status, o.created_at,
  c.name AS customer_name, c.email AS customer_email,
  s.name AS shop_name
FROM orders o
JOIN customers c ON o.customer_id = c.id
JOIN shops s ON o.shop_id = s.id;

-- Refresh nightly hoặc on event
REFRESH MATERIALIZED VIEW CONCURRENTLY order_view;

Pattern 3 — Event-Driven Sync (xem #18)

// Customer updates email → publish event
@EventListener
public void on(CustomerEmailChangedEvent event) {
    // Update denormalized copies async
    orderRepo.updateCustomerEmail(event.customerId(), event.newEmail());
}
🎯 Quy tắc 80/20

80% application: stay với 3NF + proper index. JOIN nhanh, consistency simple.
20% high-traffic / latency-critical path: denormalize có chiến lược.
Đừng denormalize trước khi đo. Setup monitoring → identify bottleneck thật → denormalize chỉ chỗ đó. Premature denormalization là root cause của nhiều consistency bug.

🔬 Thử Phá Vỡ — Denormalize price vào order_items

"Tôi sẽ copy product price vào order_items để không cần JOIN products khi calculate total." Hãy nghĩ: đây là tốt hay xấu?

Click để xem analysis

Thực ra đây là pattern ĐÚNG cho 1 lý do quan trọng nhưng vì sao khác với những trường hợp denormalize khác:

Vì sao đúng: price tại thời điểm mua phải frozen forever. Nếu chỉ FK đến products.price, khi shop tăng giá ngày mai → reports show historical orders với giá MỚI → wrong revenue calculation.

Đây không phải "denormalization cho speed" mà là "data semantic correctness". order_items.price không phải "copy" của products.price — chúng là 2 concepts khác nhau:

  • products.price = "giá HIỆN TẠI của product này"
  • order_items.price = "giá khi customer đã mua"

Insight: nhiều cases "denormalization" thực ra là 2 fields semantic khác nhau happens to có cùng value tại 1 thời điểm. Đừng merge chúng nhân danh "normalization". Tax rate khi tính tax = same pattern.

4. Partitioning CORE

★ CORE — Scale lên 100M+ rows mà không phải sharding. Single biggest improvement cho time-series data.

Partitioning = chia 1 table thành nhiều physical partition trong cùng 1 database. Query planner skip partition không liên quan = partition pruning.

3 Types — Range, List, Hash

Range Partitioning (phổ biến nhất cho time-series)

-- PostgreSQL syntax (modern, declarative)
CREATE TABLE orders (
    id BIGSERIAL,
    customer_id BIGINT NOT NULL,
    shop_id BIGINT NOT NULL,
    total_amount DECIMAL(15,2),
    status VARCHAR(20),
    created_at TIMESTAMP NOT NULL,
    PRIMARY KEY (id, created_at)  -- Partition key MUST be in PK
) PARTITION BY RANGE (created_at);

-- Create partitions per month
CREATE TABLE orders_2026_01 PARTITION OF orders
    FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');
CREATE TABLE orders_2026_02 PARTITION OF orders
    FOR VALUES FROM ('2026-02-01') TO ('2026-03-01');
-- ... etc

-- Default partition cho rows nằm ngoài range
CREATE TABLE orders_default PARTITION OF orders DEFAULT;

Hash Partitioning (distribute evenly)

CREATE TABLE user_events (
    user_id BIGINT,
    event_type VARCHAR(50),
    event_data JSONB,
    occurred_at TIMESTAMP
) PARTITION BY HASH (user_id);

CREATE TABLE user_events_p0 PARTITION OF user_events FOR VALUES WITH (MODULUS 8, REMAINDER 0);
CREATE TABLE user_events_p1 PARTITION OF user_events FOR VALUES WITH (MODULUS 8, REMAINDER 1);
-- ... 8 partitions total

-- Query: user_id=12345 → hash(12345) % 8 → partition p3
SELECT * FROM user_events WHERE user_id = 12345;
-- Planner queries only p3, not all 8

List Partitioning (discrete categories)

CREATE TABLE orders_by_region (
    id BIGSERIAL,
    region VARCHAR(20)
) PARTITION BY LIST (region);

CREATE TABLE orders_north PARTITION OF orders_by_region
    FOR VALUES IN ('HANOI', 'HAIPHONG', 'NAMDINH');
CREATE TABLE orders_central PARTITION OF orders_by_region
    FOR VALUES IN ('DANANG', 'HUE', 'QUYNHON');
CREATE TABLE orders_south PARTITION OF orders_by_region
    FOR VALUES IN ('HCMC', 'CANTHO', 'VUNGTAU');

Partition Pruning — Magic Behind Performance

🎯 Vì sao partition pruning critical với time-series?
  • Locality of access: 95% queries hit recent data (last 30-90 days). Old data hiếm khi query nhưng vẫn cần keep.
  • Partition pruning = automatic optimization: developer không cần biết partition exists. Query với WHERE created_at > X tự skip irrelevant partitions.
  • Archive cheap: drop old partition = O(1) operation. DROP TABLE orders_2020; trong <1s thay vì DELETE FROM orders WHERE created_at < '2021' mất hàng giờ + bloat.
  • Index riêng per partition: smaller indexes → faster lookups within partition.
🎯 Edge Cases — Partitioning Production Reality
  • Query without partition key → SCAN ALL PARTITIONS: WHERE customer_id = X không có created_at filter → planner phải scan mọi partition. Partition pruning defeated. Solution: train team luôn include partition key trong WHERE.
  • Partition key trong PK: PostgreSQL/MySQL require partition column trong PK. Conflict với existing PRIMARY KEY (id) — phải đổi thành (id, created_at). Side effect: FK reference phức tạp hơn.
  • UNIQUE constraint cross-partition: UNIQUE (email) không enforce được nếu email không trong partition key. Workaround: separate unique table, hoặc trigger-based check.
  • Partition explosion: forget to create future partitions → INSERT fail với "no partition exists". Auto-create partition với pg_partman hoặc cron job.
  • Default partition trap: rows match default partition → khi tạo partition mới phải DETACH default, move rows, ATTACH lại. Downtime nếu không careful.
  • Cross-partition JOIN expensive: JOIN với non-partitioned table OK, nhưng JOIN 2 partitioned tables có thể cause Cartesian explosion. Verify EXPLAIN.
  • Foreign key cross-partition: order_items reference orders.id — nếu orders partitioned, FK constraint có thể fail hoặc slow. Solution: include partition key trong FK, hoặc remove FK + enforce ở app level.
  • VACUUM mỗi partition độc lập: cần monitor + tune per partition. Old partitions không cần vacuum often.
  • Backup strategy: backup per partition vs full table. Per-partition cho incremental backup nhanh hơn (recent partitions change, old stable).

5. Sharding ADVANCED

ADVANCED — Premature sharding là root cause của nhiều system disaster. Chỉ áp dụng khi đã tối đa hóa partitioning + read replica + caching.

Sharding = chia data across multiple databases/servers (vs partitioning chỉ trong 1 DB). Mục đích: scale beyond 1 server.

Shard Key Selection — Quyết Định Sống Còn

❌ Shard by sequential ID
int shard = (int)(orderId % 4);
// Order ID auto-increment
// Order 1-100 → shard 1,2,3,0,1,...
// Order 1M-1.001M → shard 0,1,2,3,...
// BUT: new orders luôn đi vào shard cuối
// → Hot shard, contention
✅ Shard by content hash
int shard = Math.abs(customerId.hashCode()) % 4;
// Hash distributes evenly
// Customer 12345 → consistent shard
// Customer 12346 → potentially different shard
// → Load balanced

Multi-Shard Fan-Out Trap

// Sharded by customer_id
// Query: "Top 100 orders by total_amount in last 24h" → KHÔNG có customer_id filter
// → Must query ALL 4 shards

@Service
public class OrderQueryService {
    public List<Order> topOrdersLast24h() {
        List<CompletableFuture<List<Order>>> futures = new ArrayList<>();
        for (int shard = 0; shard < 4; shard++) {
            ShardConnection conn = shardManager.get(shard);
            futures.add(CompletableFuture.supplyAsync(() ->
                conn.query("SELECT * FROM orders WHERE created_at > ? ORDER BY total_amount DESC LIMIT 100",
                    Instant.now().minus(Duration.ofHours(24)))));
        }

        // Wait for ALL shards (or timeout)
        try {
            List<Order> merged = CompletableFuture.allOf(futures.toArray(new CompletableFuture[0]))
                .thenApply(v -> futures.stream()
                    .map(CompletableFuture::join)
                    .flatMap(List::stream)
                    .sorted(comparing(Order::totalAmount).reversed())
                    .limit(100)
                    .toList())
                .get(5, TimeUnit.SECONDS);
            return merged;
        } catch (TimeoutException e) {
            // 1 shard slow → entire query fail
            throw new ShardQueryTimeoutException("Fan-out query timeout", e);
        }
    }
}
// Latency = MAX of all shards
// If shard 2 has 5s latency và 3 shards khác 50ms → total 5s
// 1 shard down → entire query fail
🔥 Sharding cost reality check
  • Operational complexity 10×: 4 databases backup, monitoring, schema migration, version sync
  • Cross-shard transaction impossible: phải dùng SAGA (xem #13)
  • Query không có shard key = expensive fan-out: nếu > 20% queries fan-out, sharding losses outweigh benefits
  • Re-sharding painful: thay đổi shard count = data migration weeks/months
  • Cost > benefit thường xuyên: 90% cases có thể avoid bằng partitioning + read replica + caching

Rule: chỉ shard khi single DB đã hit hardware limits (CPU 100%, IOPS saturated) và mọi optimization khác đã thử. Đừng shard "for the future".

6. Read Replica CORE

★ CORE — Pattern phổ biến nhất để scale reads. Phải hiểu replication lag + read-after-write trap.

Setup

Application
  ├─ Write requests → PostgreSQL Primary (master)
  ├─ Read requests → PostgreSQL Replica (1+ replicas)
  └─ Async replication: Primary WAL → Replica (typical lag 10-500ms)

Read-After-Write Trap — Bug Phổ Biến Nhất

❌ Sai — Read sau write từ replica
@Transactional
public OrderResponse placeOrder(Cart cart) {
  // Write to PRIMARY
  Order order = orderService.create(cart);
  orderRepo.save(order);

  // Read from REPLICA (lag 100ms!)
  // Replica chưa replicate xong → return null
  Order saved = orderRepo.findById(order.getId());
  return OrderResponse.from(saved); // NPE!
}
✅ Đúng — Read after write từ Primary
@Transactional  // Routes to PRIMARY
public OrderResponse placeOrder(Cart cart) {
  Order order = orderService.create(cart);
  orderRepo.save(order);

  // Same transaction = PRIMARY
  // Hoặc explicit primary route
  Order saved = orderRepo.findById(order.getId());
  return OrderResponse.from(saved); // OK
}

@Transactional(readOnly = true)  // Routes to REPLICA
public List<Order> orderHistory(Long customerId) {
  // Stale data tolerable cho history
  return orderRepo.findByCustomerId(customerId);
}

3 Strategies Cho Read-After-Write

  1. Route writes + immediate reads to Primary: dùng @Transactional với readOnly flag. Spring AOP auto-route.
  2. Sticky session: user sau write → next N seconds routes to primary. Cookie hoặc header tracking.
  3. Wait for replication: write trả về LSN (log sequence number), replica catch up to LSN trước khi read. Highest consistency, highest latency.

Implementation — AbstractRoutingDataSource

@Component
public class RoutingDataSource extends AbstractRoutingDataSource {
    private static final ThreadLocal<String> routingKey =
        ThreadLocal.withInitial(() -> "primary");

    @Override
    protected Object determineCurrentLookupKey() {
        return routingKey.get();
    }

    public static void routeToReplica() { routingKey.set("replica"); }
    public static void routeToPrimary() { routingKey.set("primary"); }
    public static void clear() { routingKey.remove(); }
}

@Aspect
@Component
public class TransactionRoutingAspect {
    @Before("@annotation(org.springframework.transaction.annotation.Transactional)")
    public void route(JoinPoint jp) {
        Method m = ((MethodSignature)jp.getSignature()).getMethod();
        Transactional tx = m.getAnnotation(Transactional.class);
        if (tx != null && tx.readOnly()) {
            RoutingDataSource.routeToReplica();
        } else {
            RoutingDataSource.routeToPrimary();
        }
    }

    @After("@annotation(org.springframework.transaction.annotation.Transactional)")
    public void cleanup(JoinPoint jp) {
        RoutingDataSource.clear();
    }
}
🎯 Edge Cases — Read Replica Production
  • Replication lag spike: heavy write burst → replica fall behind 10+ seconds. Reads return very stale data. Monitor lag, alert > 1s.
  • Replica down: failover to primary (graceful degradation) hoặc reject reads. App must handle both modes.
  • Replica promoted to primary: failover scenario. Connection pool cần re-discover topology. Pgpool/HAProxy/AWS RDS handle automatically.
  • Read-after-write within session: user X submit form, redirect, GET → có thể hit replica với stale data. Sticky session by user.
  • Cross-region replica: lag 100ms-1s do network. Acceptable cho dashboard, NOT cho transactional reads.
  • Replica thread pool exhaustion: heavy reporting query trên replica → block other reads. Separate replica cho analytics vs OLTP.
  • Schema migration: ALTER TABLE on primary có thể block replication 10+ minutes. Test migration on replica first.
  • Cache invalidation timing: invalidate cache sau write to primary. Nếu cache reload từ replica → load stale value back into cache. Wait for replication lag trước invalidate.

7. CAP Theorem & PACELC SUPPLEMENTARY

SUPPLEMENTARY — Framework để argue về distributed DB. System design interview hỏi nhiều.

CAP — Brewer's Theorem

Trong distributed system, chỉ có thể chọn 2 trong 3 khi network partition xảy ra:

  • Consistency: mọi node thấy cùng data tại cùng thời điểm
  • Availability: mọi request nhận response (kể cả stale)
  • Partition tolerance: system continue work despite network split

P không phải lựa chọn — network partition sẽ xảy ra (cable đứt, switch fail, region down). Vì vậy real choice là C hay A:

SystemChoiceBehavior khi partition
PostgreSQL (single primary)CPReplica từ chối write nếu partition khỏi primary → consistency preserved, availability mất
Cassandra (default)APMỗi node accept writes → eventual consistency, luôn available
MongoDB (modern, w=majority)CPWrite require majority quorum, minority partition không write được
DynamoDBAP (tunable)Configurable consistency level

PACELC — Extended CAP

Daniel Abadi (2013): "CAP only applies when partition. But even without partition, có trade-off giữa Latency và Consistency."

If Partition (P), choose Availability (A) or Consistency (C)
Else (E) — normal operation, choose Latency (L) or Consistency (C)

Examples:
- PostgreSQL: PC/EC — consistency in both cases
- DynamoDB: PA/EL — availability under partition, low latency normally
- MongoDB (read preference primary): PC/EC
- MongoDB (read preference nearest): PC/EL
🎯 E-commerce nên chọn CAP nào?

Phụ thuộc operation:

  • Order placement / payment: CP. Better fail than double-charge or oversell. User retry vs lifetime support ticket.
  • Inventory count display: AP. Stale "5 in stock" OK; better show stale than blank page.
  • Search: AP. Outdated results acceptable; key UX = responsiveness.
  • Cart: AP. Best-effort sync across devices; conflicts rare và resolvable.

Insight: CAP không phải decision cho toàn system, mà cho mỗi feature. Modern architecture: critical paths CP (payment), UX paths AP (search, browse).

8. Materialized View SUPPLEMENTARY

SUPPLEMENTARY — Tool cho reporting / analytics. Khi nào worth.

Materialized view = pre-computed query result lưu như table. Refresh định kỳ. Trade computation upfront cho fast reads.

-- Reporting query slow: total revenue per shop per day
CREATE MATERIALIZED VIEW shop_daily_revenue AS
SELECT
    shop_id,
    DATE(created_at) AS order_date,
    COUNT(*) AS order_count,
    SUM(total_amount) AS revenue,
    AVG(total_amount) AS avg_order_value
FROM orders
WHERE status = 'DELIVERED'
GROUP BY shop_id, DATE(created_at);

-- Add index trên MV cho query nhanh
CREATE INDEX idx_mv_shop_date ON shop_daily_revenue(shop_id, order_date);

-- Refresh nightly (cron job)
REFRESH MATERIALIZED VIEW CONCURRENTLY shop_daily_revenue;
-- CONCURRENTLY: no lock, slower refresh nhưng reads vẫn work
🎯 Worth khi nào?
  • Reporting query slow: aggregation trên 100M+ rows, take seconds-minutes
  • Stale acceptable: hourly/daily refresh OK cho dashboard
  • Read 1000× hơn write: ROI cao
  • Stable schema: query không thay đổi often

KHÔNG worth: query thay đổi liên tục, real-time data required, write-heavy table (refresh contention).

9. Anti-Patterns CORE

#1 Premature sharding

Team với 5M rows quyết định shard "for future". Operational complexity 10× tăng. 3 năm sau vẫn 10M rows (under-estimate growth). Wasted complexity, slower development.

🎯 Khi nào thực sự cần shard?

Khi single DB hit hardware limits: CPU 100%, IOPS saturated, không thể scale up. Optimize partitioning + replica + caching trước. Sharding là last resort.

#2 Partitioning without query update

Partition by date, nhưng query SELECT * FROM orders WHERE customer_id = X không include date → scan ALL partitions. Pruning defeated, performance same hoặc tệ hơn.

#3 Denormalize without invalidation strategy

Copy customer_name vào orders. Customer rename → 50K rows trong orders vẫn old name. Reports show wrong stats. Fix: event-driven invalidation hoặc async sync job.

#4 Sharding by sequential ID

Hot shard với new data. Load không balanced. Fix: hash-based shard key.

#5 Ignoring replication lag

Read sau write từ replica → stale data → user thấy "thao tác fail" mặc dù success. Fix: route critical reads to primary.

#6 Cross-shard JOIN

Sharded data, JOIN 2 sharded tables → fan-out × fan-out = Cartesian explosion. Solution: denormalize, hoặc materialize JOIN result.

#7 No partition cleanup

Partitions tích lũy mãi. 5 năm sau có 60+ partitions, metadata overhead. Fix: scheduled DROP old partitions, archive to cold storage.

#8 3NF queries on reporting tables

4-JOIN query trên 500M rows table = 2 phút. Fix: materialized view denormalized cho reporting.

9b. Symptom → Debug Recipes CORE ★

Bug schema design thường không lộ ngay — chúng xuất hiện khi data scale, traffic skew, hoặc team mới join read replica. Đây là cách nhận diện.

🚨 Recipe #1 — User report "đơn hàng vừa tạo bỗng biến mất"

Symptom: User tạo order, redirect sang trang "Đơn của tôi" trong 200ms → không thấy order. F5 sau 2 giây thấy. Reproduce không ổn định, 5-10% requests.

Log/Metric cần xem:

  • pg_stat_replication.replay_lag (Postgres) hoặc MySQL SHOW SLAVE STATUS \G — xem Seconds_Behind_Master
  • App log: routing datasource log để biết request đi master hay replica
  • APM trace: span của write request và read request có overlap thời gian không

Root cause: Read-after-write trên read replica. Write commit master xong, app redirect → next read đi replica nhưng replication lag 500ms-2s nên data chưa có.

3 cách fix theo trade-off:

  1. Sticky session theo write window — sau write, force read về master trong N giây. Code đơn giản, nhưng master load tăng
  2. Causal consistency token — write trả về LSN/GTID, read truyền token kèm. Replica đợi đến LSN đó mới serve. Chuẩn nhất nhưng phức tạp
  3. Read-your-own-writes cookie — set cookie timestamp khi write, read kiểm cookie và chọn master nếu vừa write trong 5s. Compromise tốt cho web

Anti-fix: KHÔNG tăng pool replica để "lag giảm" — lag không liên quan pool, liên quan WAL apply speed và network.

🚨 Recipe #2 — Partition table chậm thay vì nhanh sau khi partition

Symptom: Team partition orders theo created_at (range monthly), expect query nhanh hơn. Thực tế chậm hơn 30%.

Log/Metric cần xem:

  • EXPLAIN ANALYZE query — đếm số partition planner phải scan (line Subplans Removed hoặc list partition trong plan)
  • pg_stat_user_tables trên parent table và từng partition — xem table nào bị scan

3 nguyên nhân:

  1. Query không có partition key trong WHERE: WHERE customer_id = ? không có created_at → planner phải scan TẤT CẢ partition. Sửa: thêm AND created_at >= NOW() - INTERVAL '30 days'
  2. Function trên partition key: WHERE EXTRACT(month FROM created_at) = 5 defeats partition pruning. Sửa: WHERE created_at >= '2026-05-01' AND created_at < '2026-06-01'
  3. Quá nhiều partition: 5 năm × 12 tháng = 60 partition. Planner overhead lookup metadata > benefit. Fix: archive partition cũ vào cold storage

Debug command: SET enable_partition_pruning = on; EXPLAIN (ANALYZE, BUFFERS) ... — nếu thấy "Subplans Removed: 0" tức là không prune được partition nào.

🚨 Recipe #3 — Hot shard / hot partition khi traffic tăng

Symptom: Sharded 8 nodes by user_id. Tổng traffic chia đều theo lý thuyết. Thực tế 1 shard CPU 95%, 7 shard kia 20%.

Log/Metric cần xem:

  • Per-shard CPU, QPS, query latency — vẽ heatmap
  • Top query per shard: pg_stat_statements filter theo shard
  • Hot key analysis: SELECT user_id, COUNT(*) FROM orders GROUP BY user_id ORDER BY 2 DESC LIMIT 10

Nguyên nhân (theo độ phổ biến):

  1. Power user / VIP account — 1 user chiếm 30% requests (admin tool, bot, enterprise customer). Fix: dedicated shard cho VIP, hoặc sub-shard theo (user_id, region)
  2. Sequential ID nếu shard theo modulo của auto-increment → mọi insert mới đi cùng shard. Fix: shard by hash(uuid) hoặc reshard với consistent hashing
  3. Time-based hot nếu data mới được query nhiều — shard theo (user_id, time bucket) thay vì pure user_id

Anti-fix: KHÔNG add cache trước khi xác định hot key — nếu cache cùng key bị hot, cache stampede sẽ amplify problem.

🎯 3 cấp độ Mastery cho schema design

L1 — Recognize (Junior)

Biết 1NF/2NF/3NF, biết "denormalize cho performance", biết khái niệm sharding/partitioning. Khi gặp slow query, thường suggest "thêm read replica".

Symptom của L1: gợi ý sharding khi vấn đề thực ra là missing index.

L2 — Explain (Mid)

Biết trade-off giữa normalization vs denormalization, hiểu read replica lag, biết partition pruning yêu cầu gì. Khi đề xuất shard, biết hỏi "shard key gì? hot key thế nào?".

Symptom của L2: đặt câu hỏi về access pattern trước khi chọn schema.

L3 — Apply (Senior)

Đo replication lag percentile, biết khi nào nên/không nên shard (rule of thumb: dưới 500GB chưa cần shard), thiết kế causal consistency cho read-after-write. Recipe debug ở trên là phản xạ.

Symptom của L3: pull request schema migration có rollback plan + data backfill strategy.

11. Interview Framework

Tầng 1 — SurfaceMid gate
Q: Khác biệt giữa 3NF và denormalization? Khi nào denormalize?
"3NF: mọi non-key column depend trực tiếp lên PK, không có transitive dependency. Tránh data anomalies. Denormalization: cố ý lặp data cho performance.

Denormalize khi: (1) column được read > 1000× hơn write, (2) column rarely changes (< 1×/month), (3) JOIN cost ảnh hưởng critical path latency, (4) có invalidation strategy (event-driven hoặc async job).

Default: 3NF. Denormalize là exception, không phải rule. Premature denormalization tạo consistency bug nhiều hơn solve performance."
Q: Partition pruning là gì, vì sao useful?
"Partition pruning: query planner skip scanning partitions không liên quan dựa trên WHERE conditions. Ví dụ: orders table partition by year. Query WHERE created_at > '2026-01-01' → planner chỉ scan 2026 partition (không scan 2023/2024/2025).

Useful vì: (1) scan 1-2 partitions thay vì all → I/O ít hơn nhiều, (2) smaller index per partition → faster lookups, (3) old partitions có thể archive/drop easily.

Pitfall: nếu query KHÔNG có partition key trong WHERE → planner phải scan TẤT CẢ partitions → pruning defeated. Team phải train để luôn include partition key."
Tầng 2 — Deep DiveMid → Senior
Q: 500M rows orders table, queries slow. 3 options: add indexes, partition by date, add read replicas. Choose nào và vì sao?
"Multi-layer approach, theo thứ tự ROI:

1. Index trước (1-2 tuần, lowest risk, highest immediate impact)
• Run EXPLAIN ANALYZE trên top queries để identify missing indexes
• Composite index theo query pattern, ví dụ (customer_id, status, created_at DESC)
• Có thể giảm latency 100× cho queries selective
• Trade-off: write latency +5-10%, storage +1-2GB

2. Partition by date (3-4 tuần, medium risk)
• Range partition by month/quarter
• Recent queries (last 30 days) chỉ scan 1-2 partitions
• Old data drop dễ: DROP TABLE orders_2023
• Trade-off: schema migration painful, queries phải include date filter, FK reference phức tạp

3. Read replica (2-3 tuần)
• Route SELECT queries cho replica
• Giảm load primary cho writes critical
• Trade-off: replication lag 100ms-1s, read-after-write trap

Order matters: index nhanh nhất implement và biggest immediate impact. Partition cần schema rework. Replica cần infrastructure setup. Layer build trên nhau.

Monitoring: setup pg_stat_statements, alert khi p95 > SLA. Iterate based data."
Q: Denormalize customer_name trong orders cho performance. Customer rename → ensure consistency thế nào?
"3 approaches, trade-off khác nhau:

Option 1 — Async batch update:
• Customer update name → save
• Async job hourly: UPDATE orders SET customer_name = ? WHERE customer_id = ?
• Pros: simple, không ảnh hưởng write latency
• Cons: inconsistency window 1 hour. Reports trong window show mix old/new names.

Option 2 — Event-driven (recommended cho most cases):
• Customer service publish CustomerNameChangedEvent sau update
• OrderService listen, async update denormalized rows
• Outbox pattern guarantee delivery (xem #18)
• Pros: near real-time, decoupled
• Cons: complex setup, eventual consistency

Option 3 — Don't denormalize, use materialized view:
• Keep 3NF orders table
• Materialized view JOIN orders + customers, refresh nightly
• Pros: consistency trong MV, simple invalidation (full refresh)
• Cons: MV stale until refresh, full refresh expensive

Recommendation: nếu nguyên nhân denormalize là OLTP read performance, dùng Option 2. Nếu cho reporting, dùng Option 3 (MV). Option 1 chỉ khi không có event infrastructure."
Tầng 3 — ArchitectureSenior
Q: Design database architecture cho ShopVN platform. Handle 500M orders, scale to 1B. Consistency required (no oversell). Queries: by-customer (80%), by-shop (15%), analytics (5%).
"Multi-tier database architecture:

Tier 1 — OLTP (Primary + Replica)
• PostgreSQL primary cho writes (orders, payment, inventory)
• 3 read replicas cho reads (geographically distributed)
• Replication lag < 500ms (monitor + alert)
• Read-after-write: routes to primary via @Transactional(readOnly=false)

Tier 2 — Schema design
• 3NF: orders, customers, shops, products tables separate
• Composite indexes per query pattern (xem #11)
• Range partition orders by month → auto-prune recent queries
• Denormalize selectively: customer_country trong orders (rarely changes, used for tax), product_price_snapshot trong order_items (frozen at purchase)

Tier 3 — Caching
• Redis cho hot data: customer profile (5min TTL), product info (10min TTL)
• Cache invalidation event-driven: CustomerUpdatedEvent → evict cache
• Cache-aside pattern, không write-through (avoid stale write)

Tier 4 — Analytics (separate)
• ClickHouse hoặc BigQuery cho analytics queries (5% traffic)
• ETL nightly từ OLTP → analytics DB
• Materialized views cho top dashboards (refresh hourly)
• Không ảnh hưởng OLTP performance

Consistency strategy:
• Order placement: CP — better fail than oversell (use SERIALIZABLE isolation hoặc optimistic locking)
• Inventory display: AP — show stale OK
• Cart sync across devices: AP với conflict resolution

Sharding decision:
• 500M rows: chưa cần shard. Partition by month đủ.
• 1B rows + hardware saturated: shard by customer_id (most query có customer filter)
• Fan-out queries (by-shop) chấp nhận 2-3× latency
• Cross-shard transactions dùng SAGA

Roadmap:
• Phase 1 (1 tháng): optimize indexes, setup pg_stat_statements monitoring
• Phase 2 (2 tháng): partition by month, archive old partitions
• Phase 3 (1 tháng): read replica + routing
• Phase 4 (2 tháng): caching layer + invalidation
• Phase 5 (3 tháng nếu reach hardware limit): sharding

Expected outcome: p95 query < 50ms hot data, < 500ms cold. Handle 10× current load."
