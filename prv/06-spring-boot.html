Spring Boot — Auto-Configuration đến Production

Junior dùng @SpringBootApplication mỗi ngày nhưng khi gặp No qualifying bean found hoặc auto-configuration mismatch = không biết bắt đầu từ đâu. File này bóc auto-config internals + actuator + custom starter + 5 anti-pattern junior hay vướng.

CORE ★ 1 tuần đọc + build 1 starter thật Domain: E-commerce / SaaS

⚠️ Cảnh báo "hiểu giả"

  • Dùng @SpringBootApplication — không biết nó = @Configuration + @EnableAutoConfiguration + @ComponentScan
  • Biết auto-config — không đọc nổi --debug output để biết bean nào loaded vs skipped
  • Tạo starter cho team — không hiểu spring.factories (Boot 2) vs AutoConfiguration.imports (Boot 3)
  • Dùng Actuator — chỉ enable /health, không biết /loggers, /threaddump, /heapdump cứu mạng production
🚨 3 cấp — bạn đang ở đâu?

L1 (Recognize): chạy được Boot app, dùng application.yml basic.
L2 (Explain): đọc auto-config output, override config qua profile, dùng Actuator full set, viết test với @SpringBootTest hiểu context loading.
L3 (Apply): build custom starter cho team, design conditional config, optimize startup time, debug production qua Actuator endpoint.

1. Bài toán thực tế CORE ★

📖 Tình huống — "Local OK, staging fail boot"

Bối cảnh: Junior deploy service mới lên staging. Boot crash với:

APPLICATION FAILED TO START
Description:
  Field redisTemplate in com.shop.CartService required a bean of type
  'org.springframework.data.redis.core.RedisTemplate' that could not be found.

Action:
  Consider defining a bean of type RedisTemplate in your configuration.

Local Mac chạy ngon. Tại sao staging fail?

Root cause: dependency spring-boot-starter-data-redis bị mark <scope>test</scope> nhầm trong pom. Local có Redis local nên IDE load qua classpath khác. Staging deploy jar production → starter không trong classpath → auto-config không kick in → RedisTemplate bean không tạo.

Debug đúng:

# 1. Run với --debug để xem auto-config report
java -jar app.jar --debug | grep -A 5 "Redis"

# Output: "RedisAutoConfiguration: did not match"
# Reason: @ConditionalOnClass(RedisOperations.class) → class not on classpath

# 2. Verify jar có Redis starter không
unzip -l app.jar | grep redis
📖 Tình huống — "Auto-config override không work"

Junior define @Bean DataSource custom. Nhưng Boot vẫn auto-create thêm 1 HikariDataSource default → 2 bean conflict.

// ❌
@Bean
public DataSource dataSource() {
    return new MyCustomDataSource();
}
// Boot auto-config detect "có DataSource trên classpath" → vẫn tạo

Fix: Spring Boot có @ConditionalOnMissingBean — nếu app define rồi thì skip. Vấn đề thường là user define bean với tên khác, Boot không match.

// ✅ Match name "dataSource" để Boot auto-config skip
@Bean(name = "dataSource")
public DataSource dataSource() { ... }

// Hoặc exclude auto-config
@SpringBootApplication(exclude = DataSourceAutoConfiguration.class)
⏸ Pause & Predict
  1. Auto-config dùng cơ chế gì để biết "có nên tạo bean không"? (Hint: 4 annotation chính)
  2. Tình huống 2 — nếu bạn dùng @Primary trên bean custom, Boot có skip auto-config không?
  3. --debug khác logging.level.root=DEBUG như nào?

2. Mental Model — Auto-Configuration CORE ★

2.1 @SpringBootApplication = 3 annotation

@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@SpringBootConfiguration  // = @Configuration
@EnableAutoConfiguration  // ← magic
@ComponentScan(excludeFilters = {...})
public @interface SpringBootApplication { }

2.2 Auto-configuration discovery

Spring Boot 3 đọc META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports trong mọi jar. File này list class auto-config:

org.springframework.boot.autoconfigure.web.servlet.WebMvcAutoConfiguration
org.springframework.boot.autoconfigure.data.redis.RedisAutoConfiguration
org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration
...

(Boot 2 dùng META-INF/spring.factories — đổi format ở Boot 2.7).

2.3 Conditional annotation — quyết định bean tạo hay không

AnnotationTạo bean khi…
@ConditionalOnClass(Foo.class)Class Foo có trên classpath
@ConditionalOnMissingClassClass KHÔNG có
@ConditionalOnBean(Bar.class)Bean Bar đã tồn tại trong context
@ConditionalOnMissingBeanBean CHƯA define — đây là cách Boot không override user bean
@ConditionalOnProperty("foo.enabled")Property set (=true mặc định)
@ConditionalOnWebApplicationLà web app (có DispatcherServlet)

2.4 Auto-config example — Redis

@AutoConfiguration
@ConditionalOnClass({ RedisOperations.class })   // chỉ khi spring-data-redis trên classpath
@EnableConfigurationProperties(RedisProperties.class)
public class RedisAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean(name = "redisTemplate")  // skip nếu user đã define
    public RedisTemplate<Object, Object> redisTemplate(
            RedisConnectionFactory factory) {
        RedisTemplate<Object, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);
        return template;
    }
}

2.5 Debug auto-config — --debug output

============================
CONDITIONS EVALUATION REPORT
============================

Positive matches:
-----------------
  RedisAutoConfiguration matched:
    - @ConditionalOnClass found required class 'RedisOperations'

  RedisAutoConfiguration#redisTemplate matched:
    - @ConditionalOnMissingBean (names: redisTemplate)

Negative matches:
-----------------
  KafkaAutoConfiguration:
    Did not match:
      - @ConditionalOnClass did not find required class 'KafkaTemplate'

Đọc report: Positive = bean được tạo. Negative = skip + lý do. Đây là tool đầu tiên debug "tại sao bean không có".

3. Production Config — Profile + Externalize CORE ★

3.1 Profile

# application.yml (default)
spring:
  application:
    name: order-service

---
# application-dev.yml
spring:
  config:
    activate:
      on-profile: dev
  datasource:
    url: jdbc:postgresql://localhost:5432/orders
    username: dev
    password: dev

---
# application-prod.yml
spring:
  config:
    activate:
      on-profile: prod
  datasource:
    url: ${DB_URL}              # env var
    username: ${DB_USER}
    password: ${DB_PASS}

Activate: --spring.profiles.active=prod hoặc env SPRING_PROFILES_ACTIVE=prod.

3.2 Property precedence (cao → thấp)

  1. Command line args (--server.port=9000)
  2. SPRING_APPLICATION_JSON env var
  3. OS environment variables
  4. External application.yml (config dir)
  5. Profile-specific application-{profile}.yml
  6. Default application.yml trong jar
  7. @PropertySource

→ Production thường dùng env var. Local dev dùng application-dev.yml.

3.3 @ConfigurationProperties — Type-safe binding

@ConfigurationProperties(prefix = "shop.payment")
@Validated
public record PaymentProperties(
    @NotBlank String stripeKey,
    @NotNull Duration timeout,
    @Min(1) int retryMax
) {}

// application.yml:
// shop:
//   payment:
//     stripe-key: ${STRIPE_KEY}
//     timeout: 5s
//     retry-max: 3

// Inject như normal bean
@Service
public class PaymentService {
    public PaymentService(PaymentProperties props) { ... }
}

Tốt hơn @Value: type-safe, validated, dễ test, IDE auto-complete.

3.4 Secret management — KHÔNG commit secret vào git

  • Dev: .env file (gitignored), load qua spring-dotenv
  • Staging/Prod: AWS Secrets Manager, HashiCorp Vault, K8s Secret + spring-cloud-vault
  • KHÔNG: commit application-prod.yml với DB password vào git

4. Actuator + Observability CORE ★

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,prometheus,loggers,threaddump,heapdump,env
  endpoint:
    health:
      show-details: when-authorized       # KHÔNG always (leak info)
      probes:
        enabled: true                     # liveness + readiness cho K8s
  metrics:
    tags:
      application: ${spring.application.name}
      env: ${ENV:dev}

Endpoint cứu mạng production

EndpointUse case
/actuator/healthK8s liveness + readiness probe
/actuator/loggersĐổi log level runtime KHÔNG restart (POST với {"configuredLevel":"DEBUG"})
/actuator/threaddumpLấy thread dump không cần ssh server
/actuator/heapdumpHeap dump cho MAT (file lớn, dùng cẩn thận)
/actuator/metrics/hikaricp.connections.activeConnection pool real-time
/actuator/prometheusPrometheus scrape format
/actuator/envXem effective config (mask secret)
⚠️ Security cho Actuator
  • KHÔNG expose * public — leak info, attack surface
  • Chỉ expose /health + /info + /prometheus public; các endpoint khác behind auth
  • Dùng management.server.port tách port internal (chỉ K8s service mesh access)

5. Custom Starter — Cho team ADVANCED

Khi team có 10+ microservice cùng cần config (audit log, rate limit, MDC filter) → tạo custom starter, mọi service add 1 dependency là xong.

my-audit-starter/
├── pom.xml
└── src/main/
    ├── java/com/shop/audit/
    │   ├── AuditAutoConfiguration.java
    │   ├── AuditProperties.java
    │   └── AuditInterceptor.java
    └── resources/META-INF/spring/
        └── org.springframework.boot.autoconfigure.AutoConfiguration.imports
@AutoConfiguration
@ConditionalOnProperty(prefix = "shop.audit", name = "enabled", havingValue = "true", matchIfMissing = true)
@EnableConfigurationProperties(AuditProperties.class)
public class AuditAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    public AuditInterceptor auditInterceptor(AuditProperties props) {
        return new AuditInterceptor(props);
    }
}
// AutoConfiguration.imports
com.shop.audit.AuditAutoConfiguration

Service sử dụng:

<dependency>
    <groupId>com.shop</groupId>
    <artifactId>my-audit-starter</artifactId>
</dependency>
shop:
  audit:
    enabled: true
    log-level: INFO

6. 5 Anti-Patterns CORE ★

❌ #1 — Expose Actuator * public

management.endpoints.web.exposure.include=* + không auth = leak env var, heap dump, config. Tấn công recon dễ.

❌ #2 — @SpringBootTest cho mọi test

Load full context cho unit test = chậm 10-50s/test. Dùng @WebMvcTest, @DataJpaTest, @JsonTest slice test thay vì full context.

❌ #3 — Component scan cả com.*

@ComponentScan(basePackages = "com") → scan mọi thư viện 3rd party. Startup chậm 30s+. Default @SpringBootApplication chỉ scan package của nó trở xuống — giữ vậy.

❌ #4 — Hard-code secret trong application.yml commit git
# ❌
spring:
  datasource:
    password: SuperSecret123!     # commit vào git → leak vĩnh viễn

Lỡ commit: rotate ngay + dùng BFG Repo-Cleaner xóa history. Fix: env var hoặc Vault.

❌ #5 — Spring Boot 2 → 3 không đọc migration guide

Major changes: javax.* → jakarta.*, spring.factories → AutoConfiguration.imports, Java 17 minimum. Upgrade mò = boot fail không hiểu tại sao.

🚨 Symptom → Debug Recipes CORE ★

🚨 Recipe #1 — "No qualifying bean of type X"

Symptom: Boot fail với NoSuchBeanDefinitionException.

java -jar app.jar --debug | grep -A 10 "X\|XAutoConfiguration"
# Tìm Negative match → lý do skip
# Thường: @ConditionalOnClass not found (starter chưa add)
# Hoặc: @ConditionalOnProperty không match (config missing)
🚨 Recipe #2 — "Multiple beans found"
expected single matching bean but found 2: dataSource, customDataSource

Fix: @Primary trên bean default + @Qualifier ở injection point cần bean khác.

🚨 Recipe #3 — "Startup chậm 30s+"
java -jar app.jar -Ddebug --spring.boot.run.profiles=dev
# Look for: "Started Application in X seconds"
# Bottleneck thường:
# - ComponentScan quá rộng
# - JPA scan toàn bộ entity quá nhiều
# - Eager init beans nặng
# Fix: lazy init + selective auto-config + native image (GraalVM)
🚨 Recipe #4 — "Profile prod load nhưng property dev vẫn dùng"
# Kiểm property hiệu lực qua Actuator
curl http://localhost:8080/actuator/env/spring.datasource.url
# response shows: PropertySource origin + value

Thường do precedence — env var override file, hoặc profile chưa active.

🎤 Interview Framework CORE ★

Q: @SpringBootApplication gồm gì?

= @SpringBootConfiguration (= @Configuration) + @EnableAutoConfiguration + @ComponentScan. EnableAutoConfiguration kích hoạt cơ chế đọc AutoConfiguration.imports (Boot 3) hoặc spring.factories (Boot 2) trong mọi jar.

Q: Auto-config quyết định tạo bean như nào?

4 conditional annotation chính: @ConditionalOnClass (class trên classpath), @ConditionalOnMissingBean (user chưa define), @ConditionalOnProperty (config set), @ConditionalOnWebApplication. Run --debug để xem CONDITIONS EVALUATION REPORT.

Q: Design custom starter cho 20 microservice team

Mục tiêu: cross-cutting concern (audit, MDC, rate limit) — mọi service chỉ add 1 dependency.

  1. Module độc lập với @AutoConfiguration
  2. AutoConfiguration.imports declare class
  3. @ConditionalOnProperty(matchIfMissing=true) để default enable nhưng có thể disable
  4. @ConditionalOnMissingBean cho phép service override
  5. Type-safe @ConfigurationProperties với @Validated
  6. Versioning semver + migration guide
  7. Test với ApplicationContextRunner verify conditional logic
