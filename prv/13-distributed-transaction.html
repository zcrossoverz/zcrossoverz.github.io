Vì Sao Distributed Transaction Khó?

Trong monolith với 1 database, ACID transaction được DB engine guarantee:

  • Atomicity — all-or-nothing
  • Consistency — invariants preserved
  • Isolation — concurrent TX không thấy nhau
  • Durability — committed data không mất

Trong microservices với nhiều DB (mỗi service own DB riêng), không có concept "transaction span 2 DBs". Mỗi service commit riêng. Khi step 2 fail sau step 1 đã commit → data inconsistency.

📜 Lịch sử — Từ 2PC đến SAGA
1978 — Jim Gray publish "Notes on Database Operating Systems" mô tả Two-Phase Commit (2PC) cho distributed databases. Mục đích: ACID across multiple nodes.

1980s-1990s — XA standard (X/Open) define 2PC interface, distributed databases (Oracle TPM, IBM TXSeries) implement. Banks dùng cho cross-database transactions.

1987 — Hector Garcia-Molina và Kenneth Salem publish "Sagas" paper tại Princeton. Mô tả pattern cho long-lived transactions. Insight: thay vì hold locks duration entire transaction (như 2PC), break thành sequence of local transactions + compensating actions.

2000s — SOA, then microservices movement bắt đầu. 2PC blocking nature + SPOF coordinator → industry move away. SAGA pattern revive cho distributed systems.

2014 — Chris Richardson popularize Saga trong microservices context. Distinguish choreography vs orchestration. Outbox pattern emerge (Gunnar Morling, Debezium team) để solve dual-write.

2016+ — Camunda, Temporal, Conductor (Netflix) build mature SAGA orchestrators. Industry standard cho complex distributed workflow.

Modern — 2PC dùng rất hạn chế (chỉ trong same-vendor DB cluster). SAGA + Outbox + Idempotency là combo standard cho microservices distributed transaction.

Insight quan trọng: 2PC không sai về lý thuyết — nó sai cho microservices context vì giả định coordinator luôn alive và network reliable. SAGA chấp nhận eventual consistency để có availability.

1. Bài Toán Thực Tế — E-commerce Checkout Bị "Tiền Mất Hàng Không"

Sàn ShopVN đã tách monolith thành microservices. Checkout flow đụng 3 services:

  • OrderService (PostgreSQL): orders table
  • InventoryService (PostgreSQL riêng): stock table
  • PaymentService (MongoDB): payment_intents collection

Business requirement: khi user click "Place Order", atomic cần:

  1. Tạo order trong OrderService
  2. Trừ stock trong InventoryService
  3. Charge customer card trong PaymentService
❌ Code junior viết — chạy được local, fail production
@RestController
public class CheckoutController {
    private final OrderService orderService;
    private final InventoryService inventoryService;
    private final PaymentService paymentService;

    @PostMapping("/checkout")
    @Transactional  // ⚠️ Junior tưởng @Transactional cover hết
    public OrderResponse checkout(@RequestBody CheckoutRequest req) {
        // Step 1: Create order (local DB transaction OK)
        Order order = orderService.create(req.customerId(), req.items());

        // Step 2: Reserve inventory (REMOTE HTTP call)
        try {
            inventoryService.reserve(req.items());
            // → POST http://inventory-service/api/reserve
            // → InventoryService trừ stock + COMMIT local DB của nó
        } catch (TimeoutException e) {
            // Order đã tạo nhưng stock chưa trừ
            // Rollback @Transactional chỉ rollback OrderService DB
            // → Order vẫn tồn tại (trong tx này, sẽ rollback)
            // → Nhưng nếu retry thì có thể stock đã trừ rồi?!
            throw e;
        }

        // Step 3: Charge payment (REMOTE call)
        try {
            paymentService.charge(req.customerId(), req.total());
            // → Charge thành công, tiền đã trừ thẻ customer
        } catch (PaymentFailedException e) {
            // ⚠️ Order created, inventory reserved, BUT payment failed
            // @Transactional rollback OrderService → order biến mất
            // BUT inventory đã reserve → stock locked
            // BUT payment có thể đã charge (network unclear)
            // → DATA INCONSISTENCY MASSIVE
            throw e;
        }

        return OrderResponse.from(order);
    }
}
🔥 Scenarios production sẽ gặp
  1. Step 2 timeout nhưng InventoryService thực ra processed → next retry trừ stock 2 lần
  2. Step 3 fail sau step 1+2 commit → order ghost (created in some DB, not all), stock locked không bao giờ release
  3. Step 3 timeout — không biết payment thực sự success hay fail → manual investigation
  4. OrderService crash giữa step 2 và step 3 → inventory reserved forever, không có ai release
  5. Customer click 2 lần → 2 orders, 2× stock deduction, 2× charge
⏸ Pause & Predict — BẮT BUỘC dừng
Trước khi đọc giải pháp, tự trả lời:
  1. Vì sao @Transactional KHÔNG đủ ở đây? Nó cover gì và không cover gì?
  2. Nếu propose 2PC (XA transaction) — vấn đề gì khi 1 service crash giữa phase 2?
  3. Compensation cho step "charge payment" là gì? Đơn giản như "refund" không?
  4. Khi customer click 2 lần (cùng cart, trong 2 giây), làm sao prevent 2 orders/charges?
  5. Step 3 timeout — bạn assume payment success hay fail? Vì sao?
Distributed TX dễ trả lời sai vì có nhiều giả định ẩn. Đáp án ở sections dưới.

2. Dual Write Problem CORE

★ CORE — Foundation. Hiểu vì sao "save DB rồi publish event" KHÔNG atomic.

Dual write: write to 2 systems khác nhau trong 1 logical operation. Vì 2 systems không share transaction, có thể fail giữa chừng.

❌ Dual write (sai pattern)
@Transactional
public void placeOrder(Cart cart) {
  Order order = Order.create(cart);
  orderRepo.save(order);  // Commit DB OK

  // ❌ KAFKA publish NGOÀI transaction
  kafkaTemplate.send("orders",
    new OrderPlacedEvent(order.id()));
  // Nếu network fail tại đây:
  // - DB: order saved ✓
  // - Kafka: event lost ✗
  // - Downstream services không biết
}
✅ Outbox pattern
@Transactional
public void placeOrder(Cart cart) {
  Order order = Order.create(cart);
  orderRepo.save(order);

  // ✅ Lưu event vào OUTBOX table
  // CÙNG transaction với business data
  outboxRepo.save(OutboxEvent.of(
    "OrderPlacedEvent",
    serialize(new OrderPlacedEvent(...))
  ));
  // Atomic: cả 2 commit cùng nhau hoặc rollback cùng nhau
}
// Async poller đọc outbox → publish Kafka
// Retry until success
🎯 Vì sao dual write KHÔNG thể atomic?

2 systems khác nhau (PostgreSQL, Kafka) không support distributed transaction qua network. Nếu chỉ có 1 system commit, system kia có thể fail ở mọi point:

  • DB commit, app crash trước Kafka publish → event lost forever
  • DB commit, Kafka network fail → có thể retry nhưng app phải có logic retry chính xác
  • Kafka publish, DB commit fail (constraint violation) → event published cho "order" chưa tồn tại

Outbox pattern fix bằng cách chỉ write to 1 system (DB) atomic với event. Sau đó async process đọc event và publish to Kafka. Xem chi tiết section 5.

3. Two-Phase Commit (2PC) — Tại Sao Industry Bỏ CORE

★ CORE — Phải hiểu vì sao 2PC fail trong microservices. Câu hỏi senior interview phổ biến.

2PC là "the classical solution" cho distributed transaction. Textbook dạy nhiều. Nhưng industry tránh dùng trong microservices. Hiểu vì sao là dấu hiệu senior.

How 2PC Works

4 Lý Do Industry Bỏ 2PC

  1. SPOF (Single Point of Failure): Coordinator down → tất cả services trong limbo. Restart coordinator phải recover state — possible nhưng complex.
  2. Blocking protocol: services hold lock từ phase 1 đến phase 2 (có thể seconds). Throughput thấp, latency cao. Acceptable cho banking trong 1990s, không acceptable cho web-scale modern.
  3. Network partition: nếu coordinator không thể reach 1 service trong phase 2, không biết service đó commit hay abort. "In-doubt transaction" phải resolve manually.
  4. Heterogeneous systems: 2PC require XA support. Modern systems (MongoDB, Redis, Kafka, REST APIs) không support XA. Chỉ work với XA-compliant relational DBs.
🧠 Misconception — "2PC vẫn dùng được nếu coordinator có HA"
"Coordinator HA (active-passive backup) sẽ fix SPOF. 2PC OK."
HA coordinator giảm nhưng KHÔNG eliminate problem. Trong khoảng thời gian failover (10-60s), services trong "prepared" state vẫn hold locks. Worst case: locks hold cho đến khi new coordinator restore + replay log. Latency spike, throughput collapse. Plus: HA coordinator add operational complexity (consensus, leader election) — và bạn vẫn còn các vấn đề khác (blocking, network partition). Thực tế: 2PC chỉ dùng trong same-vendor cluster (Oracle RAC, SQL Server Always On) — nơi infrastructure tightly integrated. Không cho cross-service microservices.

4. SAGA Pattern CORE

★ CORE — Pattern thay thế 2PC cho microservices. Phải master cả 2 styles.

SAGA: sequence of local transactions. Mỗi step commit local. Nếu step N fail, run compensating transactions cho steps 1..N-1 theo thứ tự ngược.

Style 1: Choreography (Event-driven, decentralized)

// OrderService - step 1
@Transactional
public void placeOrder(Cart cart) {
    Order order = Order.create(cart);
    orderRepo.save(order);
    outboxRepo.save(OutboxEvent.of(new OrderPlacedEvent(order.id(), cart.items())));
}

// InventoryService - listen OrderPlaced, do step 2
@KafkaListener(topics = "order-events")
@Transactional
public void on(OrderPlacedEvent e) {
    try {
        inventoryRepo.reserve(e.items());  // Local TX
        outboxRepo.save(OutboxEvent.of(new StockReservedEvent(e.orderId())));
    } catch (OutOfStockException ex) {
        // Failed - publish compensation trigger
        outboxRepo.save(OutboxEvent.of(new StockReservationFailedEvent(e.orderId(), ex.reason())));
    }
}

// PaymentService - listen StockReserved, do step 3
@KafkaListener(topics = "order-events")
@Transactional
public void on(StockReservedEvent e) {
    try {
        paymentRepo.charge(e.customerId(), e.amount());
        outboxRepo.save(OutboxEvent.of(new PaymentChargedEvent(e.orderId())));
    } catch (PaymentDeclinedException ex) {
        outboxRepo.save(OutboxEvent.of(new PaymentFailedEvent(e.orderId(), ex.reason())));
    }
}

// OrderService - listen success
@KafkaListener(topics = "order-events")
public void on(PaymentChargedEvent e) {
    orderRepo.markCompleted(e.orderId());
}

// COMPENSATION listeners
// InventoryService listen for payment failure → release stock
@KafkaListener(topics = "order-events")
@Transactional
public void on(PaymentFailedEvent e) {
    inventoryRepo.release(e.orderId());  // Compensate step 2
}

// OrderService listen for stock failure → cancel order
@KafkaListener(topics = "order-events")
@Transactional
public void on(StockReservationFailedEvent e) {
    orderRepo.cancel(e.orderId(), e.reason());
}

Style 2: Orchestration (Central coordinator)

@Service
public class CheckoutOrchestrator {
    private final OrderService orderService;
    private final InventoryService inventoryService;
    private final PaymentService paymentService;

    public OrderResult execute(CheckoutRequest req) {
        Order order = null;
        InventoryReservation reservation = null;

        try {
            // Step 1
            order = orderService.create(req);

            // Step 2
            reservation = inventoryService.reserve(order.items());

            // Step 3
            Payment payment = paymentService.charge(req.customerId(), order.total());

            // Success: mark complete
            orderService.markCompleted(order.id(), payment.id());
            return OrderResult.success(order, payment);

        } catch (Exception e) {
            // Compensate in REVERSE order
            if (reservation != null) {
                try {
                    inventoryService.release(reservation.id());
                } catch (Exception releaseEx) {
                    log.error("CRITICAL: Failed to release inventory {}", reservation.id(), releaseEx);
                    // Alert ops — manual cleanup needed
                }
            }
            if (order != null) {
                orderService.cancel(order.id(), e.getMessage());
            }
            throw new CheckoutFailedException(e);
        }
    }
}

Choreography vs Orchestration

Khía cạnhChoreographyOrchestration
Flow visibility❌ Phân tán across services✅ Rõ ràng 1 nơi
Add new stepThêm 1 listener mớiSửa orchestrator
Debugging❌ Phải trace events cross services✅ Single log stream
Coupling✅ Loose, services biết events thôi⚠️ Orchestrator depend mọi services
Scale✅ Natural (each service independent)⚠️ Orchestrator có thể bottleneck
Complex workflow❌ Khó (chains, branches)✅ Tốt (visual BPMN với Camunda)
Failure recovery⚠️ Distributed, khó audit✅ State machine persistent
🎯 Khi chọn cái nào?
  • Choreography: workflow 2-3 steps, services autonomous, low coupling priority. Event-driven architecture natural.
  • Orchestration: workflow 5+ steps, complex compensation logic, audit trail critical, có Camunda/Temporal infrastructure.
  • Hybrid: choreography cho main flow, orchestration cho complex sub-flow. Phổ biến nhất trong production lớn.
🎯 Edge Cases — SAGA Production Reality
  • Compensation cũng fail: refund payment fail (card expired). Phải có "manual compensation queue" cho ops xử lý.
  • Partial compensation: step 4 fail, compensate step 3 success, compensate step 2 fail → state mixed. Saga must track granular state.
  • Out-of-order events: StockReservedEvent arrives BEFORE OrderPlacedEvent (Kafka rebalance, multi-partition). Consumer phải buffer / wait.
  • Duplicate events: at-least-once delivery → consumer process event 2 lần. Idempotency essential (xem section 6).
  • Long-running saga timeout: payment service slow 10s, saga stuck. Add timeout per step + automatic compensation.
  • Compensation cho irreversible operation: email đã gửi customer "Order confirmed" — không thể "unsend". Solution: send compensating email "Order cancelled".
  • Concurrent sagas trên cùng resource: 2 sagas reserve cùng product cuối cùng. Race condition. Solution: optimistic locking trên inventory.
  • Saga state persistence: orchestrator crash → mất saga progress. Persist state to DB after each step. Camunda/Temporal handle automatically.
  • Compensation order matters: refund trước release inventory hay sau? Generally: reverse order of execution. Document explicit.
  • Schema evolution events: OrderPlacedEvent add field mới. Old consumers process events new format → potentially break. Versioning + backward compatibility.

5. Outbox Pattern CORE

★ CORE — Foundation cho reliable event publishing. Combine với SAGA cho complete distributed TX solution.

Outbox pattern solve dual write problem: thay vì write 2 systems (DB + Kafka), chỉ write 1 (DB) atomic.

Implementation

-- Schema
CREATE TABLE outbox_events (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    aggregate_type VARCHAR(100) NOT NULL,
    aggregate_id VARCHAR(100) NOT NULL,
    event_type VARCHAR(100) NOT NULL,
    payload JSONB NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    published_at TIMESTAMP NULL,
    INDEX idx_unpublished (published_at, created_at) WHERE published_at IS NULL
);
@Service
public class OrderService {
    @Transactional  // SAME transaction cho cả 2 saves
    public Order placeOrder(PlaceOrderCommand cmd) {
        Order order = Order.create(cmd);
        orderRepo.save(order);  // Business write

        // Save event in outbox — same transaction
        OutboxEvent event = OutboxEvent.builder()
            .aggregateType("Order")
            .aggregateId(order.id().value())
            .eventType("OrderPlacedEvent")
            .payload(json.writeValueAsString(new OrderPlacedEvent(order)))
            .build();
        outboxRepo.save(event);

        return order;
    }
}

@Component
@Slf4j
public class OutboxPoller {
    private final OutboxEventRepository outboxRepo;
    private final KafkaTemplate<String, String> kafka;

    @Scheduled(fixedDelay = 1000)  // Poll mỗi 1s
    @Transactional
    public void publishPending() {
        // Lock rows để tránh duplicate publish (multiple poller instances)
        List<OutboxEvent> pending = outboxRepo.findUnpublishedForUpdate(100);

        for (OutboxEvent event : pending) {
            try {
                kafka.send(topicFor(event), event.aggregateId(), event.payload()).get();
                event.markPublished();
                outboxRepo.save(event);
            } catch (Exception e) {
                log.error("Failed to publish event {}, will retry", event.id(), e);
                // Don't update published_at → retry next cycle
            }
        }
    }

    private String topicFor(OutboxEvent event) {
        return event.aggregateType().toLowerCase() + "-events";
    }
}
🎯 Vì sao Outbox guarantee reliability?
  • Atomic: business data + event lưu cùng DB transaction. Cả 2 commit hoặc rollback. Không có "data saved but event lost".
  • Reliable delivery: poller retry cho đến khi publish success. Crash giữa chừng? Restart, poller resume.
  • Decoupled từ Kafka availability: Kafka down 10 phút → service vẫn ship orders. Events tích trong outbox, publish khi Kafka recover.
  • Audit trail: outbox table = log of mọi events. Replay được history.

Alternative: CDC (Change Data Capture) với Debezium

Debezium read DB transaction log (PostgreSQL WAL, MySQL binlog) → tự động publish changes to Kafka. Không cần app code implement polling.

DB transaction commit → write to WAL
Debezium → read WAL → parse changes → publish to Kafka topic
Downstream consumers → process events

Pros: no app code changes, very high throughput
Cons: infrastructure complex (Debezium + Kafka Connect), events include raw DB changes (might need transform)

6. Idempotency CORE

★ CORE — Foundation cho retry-safe consumer. Without idempotency, distributed TX = bug magnet.

Idempotency: process cùng request/event N lần → kết quả giống lần đầu.

Kafka/RabbitMQ deliver at-least-once → message có thể arrive 2+ lần. Consumer phải idempotent.

@Component
@Slf4j
public class StockReservationConsumer {
    private final InventoryRepository inventoryRepo;
    private final ProcessedEventRepository processedEventRepo;

    @KafkaListener(topics = "order-events")
    @Transactional
    public void on(OrderPlacedEvent event) {
        // Check idempotency FIRST
        if (processedEventRepo.exists(event.eventId())) {
            log.info("Event {} already processed, skip", event.eventId());
            return;
        }

        // Process
        try {
            inventoryRepo.reserve(event.items(), event.orderId());

            // Mark processed — cùng transaction với business logic
            processedEventRepo.save(new ProcessedEvent(event.eventId(), Instant.now()));
        } catch (OutOfStockException e) {
            // Don't mark processed if business logic fails — allow retry
            // But publish failure event (separate transaction)
            // Note: this is tricky — see edge cases
            throw e;
        }
    }
}
CREATE TABLE processed_events (
    event_id VARCHAR(64) PRIMARY KEY,  -- Unique constraint = atomic dedup
    processed_at TIMESTAMP NOT NULL,
    consumer_name VARCHAR(100) NOT NULL,
    INDEX idx_processed_at (processed_at)  -- For cleanup
);
🎯 Vì sao dedup check + business logic phải atomic (cùng transaction)?

Nếu tách:

  1. Business logic commit (stock reserved)
  2. Crash trước khi save ProcessedEvent
  3. Kafka retry → check exists = false → reserve again → stock reserved 2 lần

Đặt cả 2 trong cùng @Transactional: hoặc cả 2 commit (next retry safe) hoặc cả 2 rollback (next retry sẽ work đúng).

Idempotency cho HTTP APIs (Idempotency-Key)

Client gửi Idempotency-Key: uuid-xyz header. Server cache response by key. Retry với same key → return cached response.

@PostMapping("/api/orders")
public OrderResponse placeOrder(
        @RequestBody PlaceOrderRequest req,
        @RequestHeader("Idempotency-Key") String idempotencyKey) {

    Optional<OrderResponse> cached = idempotencyStore.get(idempotencyKey);
    if (cached.isPresent()) {
        return cached.get();  // Retry detected
    }

    Order order = orderService.place(req);
    OrderResponse response = OrderResponse.from(order);
    idempotencyStore.put(idempotencyKey, response, Duration.ofHours(24));
    return response;
}

7. Eventual Consistency SUPPLEMENTARY

SUPPLEMENTARY — Trade-off awareness. Hiểu khi acceptable.

SAGA = eventual consistency. Giữa lúc step 2 commit và step 3 commit, system trong inconsistent state. User có thể thấy:

  • Order created nhưng "Processing" status (chưa charge)
  • Stock "Reserved" nhưng order chưa visible (eventual)
  • Refund pending sau cancellation (chưa actual refund)
🎯 Khi nào eventual consistency acceptable?
  • User UX tolerant: e-commerce "We're processing your order" — user expect delay
  • Window ngắn: inconsistency window < 5s thường acceptable
  • Compensable: nếu inconsistency cause issue, có ability rollback/notify
  • Audit trail: log every state transition để debug

KHÔNG acceptable: financial settlement (money MUST balance), inventory accuracy critical (luxury items không thể oversell), compliance reports.

8. Anti-Patterns CORE

#1 @Transactional thinking it covers distributed

Spring @Transactional chỉ cover LOCAL DB transaction. KHÔNG cover remote HTTP calls. Junior assume rollback works → data inconsistent.

#2 Synchronous saga (call services sync, no events)

Orchestrator call service A, wait response, call B, wait, ... Mọi service must available. Latency cumulative. Một service down → entire saga fail.

#3 No compensation logic

Implement happy path, không có compensation. Step N fail → previous steps stuck "in progress" forever.

#4 Compensation throws exception → ignored

Refund fail → log error → continue. Now money taken but order cancelled. Customer support nightmare. Solution: alerting + manual queue.

#5 No idempotency in consumer

Kafka retry → process event 2 lần → stock trừ 2 lần. Đã cover ở section 6.

#6 Publish event NGOÀI transaction (dual write)

Save DB → publish Kafka. Nếu publish fail → event lost, downstream không biết. Fix: Outbox pattern.

#7 Saga state không persistent

Orchestrator in-memory state. Restart → lose progress. Solution: persist state to DB after each step, hoặc dùng Camunda/Temporal.

#8 Long-running locks across services

Service A lock row, wait service B, B wait C, C wait A → deadlock cross-service. Hard to detect/recover. Avoid locks across service boundary.

8b. Symptom → Debug Recipes CORE ★

Distributed transaction lỗi không crash app — lỗi xuất hiện dưới dạng data inconsistency phát hiện sau X ngày/tuần. Đây là cách bắt lỗi trước khi customer report.

🚨 Recipe #1 — Customer "đã trả tiền nhưng đơn hàng không có"

Symptom: Cuối tháng reconcile: payment service có 1247 transaction status PAID, order service chỉ có 1241 ORDER_CREATED. Lệch 6 đơn, ~12 triệu VND. Không log error nào.

Log/Metric cần xem:

  • Saga state table: SELECT * FROM saga_instances WHERE state NOT IN ('COMPLETED','COMPENSATED') AND updated_at < NOW() - INTERVAL '10 minutes' — đây là saga "treo"
  • Outbox table: số record WHERE processed_at IS NULL tăng dần = relay bị dừng
  • Kafka consumer lag trên topic order-events, payment-events
  • Distributed trace (Jaeger/Tempo): tìm trace có span payment.charge nhưng không có order.create

3 nguyên nhân theo độ phổ biến:

  1. Outbox relay crash giữa chừng — payment commit DB (PAID + outbox row), nhưng relay process die trước khi publish. Recover: restart relay với WHERE processed_at IS NULL ORDER BY created_at
  2. Order service consume event nhưng throw exception, không retry — consumer phải có DLQ + alert khi DLQ > 0. Check Kafka __consumer_offsets và DLQ topic
  3. Idempotency key bị duplicate detect SAI — order service thấy event đã xử lý (cache), skip nhưng cache stale. Check idempotency table có entry cho event đó không

Recovery procedure:

-- 1. Tìm payment đã PAID nhưng order chưa có
SELECT p.id, p.amount, p.created_at
FROM payment_db.payments p
LEFT JOIN order_db.orders o ON o.payment_id = p.id
WHERE p.status = 'PAID' AND o.id IS NULL
  AND p.created_at < NOW() - INTERVAL '5 minutes';

-- 2. Re-publish event hoặc trigger compensation manual
-- (KHÔNG insert order trực tiếp vì bypass business logic)
🚨 Recipe #2 — Saga compensation throw exception, log "silent"

Symptom: Inventory reservation thành công, payment fail → cần compensate (release reservation). Compensation call inventory service nhưng inventory đã restart, response 503. Saga "đứng" ở state COMPENSATING. Không alert.

Log/Metric cần xem:

  • Saga state: SELECT state, COUNT(*) FROM saga_instances GROUP BY state — alert nếu COMPENSATING > 0 lâu hơn 5 phút
  • Saga step retry log: mỗi compensation phải có retry count + last_attempt_at
  • Dead letter queue cho compensation events

Anti-pattern cần tránh: try { compensate(); } catch(Exception e) { log.error(e); } — exception bị nuốt, saga đứng. Đúng là phải retry với exponential backoff, sau N lần fail → escalate đến human ops queue.

Pattern đúng (state machine):

@Scheduled(fixedDelay = 30000)
public void retryStuckCompensations() {
    var stuck = sagaRepo.findByStateAndUpdatedBefore(
        SagaState.COMPENSATING, Instant.now().minusSeconds(60));
    for (var saga : stuck) {
        if (saga.getCompensationAttempts() > 5) {
            alertOps(saga); // escalate
            saga.setState(SagaState.COMPENSATION_FAILED);
        } else {
            try {
                compensationService.compensate(saga);
                saga.setState(SagaState.COMPENSATED);
            } catch (Exception e) {
                saga.incrementAttempts();
                saga.setLastError(e.getMessage());
            }
            sagaRepo.save(saga);
        }
    }
}
🚨 Recipe #3 — "Duplicate order" sau khi user click 2 lần

Symptom: User click "Đặt hàng" 2 lần vì lag UI → 2 order, 2 payment. Customer report. Reproduce: tạo 2 request gần nhau với cùng cart.

Log/Metric cần xem:

  • Access log nginx/load balancer: tìm 2 request POST /orders cùng user_id trong < 2s
  • Idempotency key table: kiểm có entry duplicate không
  • Order create endpoint: có check Idempotency-Key header không

Root cause + Fix tier:

  1. Client-side debounce (Tier 1, không đủ) — disable button sau click. Vẫn fail với network retry, browser back/forward
  2. Idempotency-Key header (Tier 2, chuẩn) — client gen UUID per intent, server lưu (key, response) → request 2 cùng key trả lại response cũ
  3. Business-level uniqueness (Tier 3, defense-in-depth) — unique constraint (user_id, cart_hash, created_within_5min)

Note: Idempotency key TTL phải đủ dài (24h) để cover retry sau network blip. Lưu trong Redis với expire, hoặc Postgres với cleanup job.

10. Interview Framework

Tầng 1 — SurfaceMid gate
Q: Vì sao 2PC ít dùng trong microservices?
"4 lý do chính:
1. SPOF: coordinator down → all services trong limbo state, hold locks.
2. Blocking protocol: hold locks từ phase 1 đến phase 2 → throughput thấp, latency cao.
3. Network partition: không reach 1 service → in-doubt transaction, không biết commit hay abort.
4. Heterogeneous systems: require XA. Modern systems (MongoDB, Kafka, REST APIs) không support XA.

Industry move sang SAGA pattern: chấp nhận eventual consistency để có availability."
Q: Outbox pattern là gì, solve vấn đề gì?
"Outbox solve dual write problem: khi cần save DB + publish event, không có distributed transaction. Pattern: thay vì publish event trực tiếp, save event vào outbox table trong CÙNG DB transaction với business data. Async poller (hoặc Debezium CDC) đọc outbox, publish to Kafka, mark as published.

Guarantee: atomic save (DB transaction), reliable delivery (retry until success), audit trail (outbox = event log).

Trade-off: thêm DB write overhead (~5-10%), thêm 1 component (poller), latency publish ~1-2s vs immediate."
Tầng 2 — Deep DiveMid → Senior
Q: E-commerce checkout: tạo order + reserve inventory + charge payment. Design distributed transaction.
"SAGA pattern + Outbox + Idempotency:

Architecture:
• 3 services own own DB
• Orchestrator: CheckoutSaga (Camunda BPMN cho visual workflow)
• Kafka cho event delivery

Happy flow:
1. CheckoutSaga.start(cart) — persist saga state (PENDING)
2. Call OrderService.create() — local TX commit, outbox event
3. Call InventoryService.reserve() — local TX commit, outbox event
4. Call PaymentService.charge() — local TX commit, outbox event
5. CheckoutSaga.complete() — mark COMPLETED, publish OrderCompleted event

Failure paths với compensation:
• If reserve fails: cancel order (compensate step 2)
• If charge fails: release inventory + cancel order (compensate step 3, 2)
• Each compensation has retry logic + alert if fail repeatedly

Idempotency:
• Client send Idempotency-Key header → CheckoutSaga check before start
• Mỗi service operation idempotent qua event ID dedup

Outbox: mọi services dùng outbox để publish events. Kafka deliver to subscribers. Consumers idempotent.

Observability: traceId cross saga steps. Saga state visible via dashboard. Alert nếu saga stuck > 5min."
Q: Compensation cho operation đã send email/SMS — không thể "unsend". Handle thế nào?
"3 strategies:

1. Send compensating notification: gửi email mới 'Order cancelled, refund processing'. Don't try to 'unsend' original — admit it happened, communicate update.

2. Delay irreversible operations: implement 'soft commit' phase. Email scheduler queue email 5 phút sau order placed. Trong window này nếu saga fail → cancel queued email. Trade-off: delay legitimate email.

3. Make operation reversible: virtual money đã credit → debit back. Promo code đã redeem → mark unused. Designed reversibility từ đầu.

Real-world hybrid: critical notifications (order confirmation, payment receipt) gửi sau khi saga COMPLETED. Promotional/marketing notifications (cross-sell) có thể gửi ngay trong saga vì compensable.

Document explicitly: trong design doc liệt kê operations cho saga, mark mỗi cái 'reversible' / 'compensable' / 'irreversible'. Force team think về consequences."
Tầng 3 — ArchitectureSenior
Q: Design distributed transaction architecture cho ride-sharing app (book trip cần: reserve driver, charge payment, allocate vehicle, notify driver). Handle failure scenarios, scale, observability.
"SAGA orchestration với Camunda BPMN:

Components:
• TripOrchestratorService — Camunda BPMN engine cho workflow visibility
• DriverService, PaymentService, VehicleService, NotificationService — independent microservices
• Each service own PostgreSQL + outbox table
• Kafka cluster cho event bus
• Debezium CDC publish outbox → Kafka topics
• Redis cho idempotency cache

Workflow (Camunda BPMN):
1. ReserveDriver — call DriverService.reserve(pickup, vehicleType)
2. ChargePayment — call PaymentService.preAuthorize(rider, estimatedFare)
3. AllocateVehicle — call VehicleService.allocate(driverId, tripId)
4. NotifyDriver — publish DriverAssignmentEvent → DriverApp consume
5. Mark trip CONFIRMED, publish TripBookedEvent

Compensation flows:
• ChargePayment fail → ReleaseDriver (compensate step 1)
• AllocateVehicle fail → RefundPayment + ReleaseDriver
• NotifyDriver fail (timeout) → retry 3× với exponential backoff. After 3rd fail → mark COMPLETED with warning + manual notify ops

Scale considerations:
• Each service horizontally scalable (independent DBs)
• Kafka partitioned by riderId → maintain ordering per user
• Outbox poller multiple instances với row-level locking (SKIP LOCKED)
• Idempotency cache Redis cluster 10K ops/sec capacity

Failure handling:
• Per-step timeout (3-5s): force compensation
• Service circuit breaker: fast-fail when downstream sick
• Saga state persistent in Camunda — survive orchestrator restart
• Dead letter queue for poison events → manual ops review

Observability:
• Jaeger distributed tracing — traceId across all services
• Saga dashboard show in-flight sagas + state distribution
• Prometheus metrics: saga success rate, compensation rate, p95 duration
• Alert: compensation rate > 5% (something wrong), saga p95 > 10s (degradation)

Audit/Compliance:
• All events persist in outbox table + Kafka topic (replay-able)
• Saga history (Camunda) audit trail of every step + decision
• Compensating events also persisted — full record of attempted + rolled-back operations

Edge cases:
• Driver accept request offline (network partition) — eventual consistency, sync khi reconnect
• Customer cancel during saga — graceful interrupt + compensation
• Concurrent bookings for same driver — DriverService optimistic lock

Total: foundational pattern stack. Mỗi component có alternative (in-house orchestrator vs Camunda vs Temporal, polling vs Debezium, etc.) — choice depend team expertise + ops capability."
