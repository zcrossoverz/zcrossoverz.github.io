Testing Strategy — Pyramid, Testcontainers, Contract, Non-Brittle

"80% coverage" là con số gây hiểu giả nguy hiểm nhất trong testing — nó đo dòng code chạy qua, không đo hành vi được khẳng định. File này phân biệt test tốt (test behavior, real DB, contract) với test giả an toàn (test implementation, H2, mock cả class under test), trên domain ShopFlow — OrderService của sàn e-commerce.

🎯 Mục tiêu
Viết test không brittle, dùng Testcontainers, hiểu contract test
📅 Thời lượng
1 tuần · Sau #9 JPA, #7 Transaction
📚 Prereq
JUnit 5 · Mockito · Spring Boot test
🔥 Độ ưu tiên
🟠 Cao — senior sở hữu chất lượng, không chỉ feature

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Coverage 80% nghĩa là code được test kỹ, ít bug."
Thực tế: Coverage chỉ đo dòng code chạy qua, không đo assertion có ý nghĩa. Một test gọi method rồi không assert gì vẫn tăng coverage. P1 vẫn lọt vì coverage cao mà test toàn check implementation detail.
Tưởng hiểu: "Integration test với H2 in-memory cho nhanh là đủ."
Thực tế: H2 có dialect SQL khác PostgreSQL (JSONB, window function, native query, constraint behavior). Test pass với H2 nhưng prod dùng PostgreSQL → bug bị che tới khi lên production.
Tưởng hiểu: "Mock càng nhiều càng tốt vì test chạy nhanh và độc lập."
Thực tế: Mock cả class under test = test đang test cái mock, không test code thật → false confidence. Chỉ mock dependency, để class cần test chạy logic thật.

🗺️ Roadmap — nếu chỉ có X giờ thì học gì

Thứ tự ưu tiên
1 giờ (CORE ★): Test pyramid + tỉ lệ và lý do (2), Unit test isolation đúng — chỉ mock dependency (3), test behavior chứ không implementation (anti-patterns). Đây là gốc để viết test không brittle.
+1 giờ (CORE ★): Integration test với Testcontainers vs H2 (4), @SpringBootTest slices cho tốc độ (6). Hỏi nhiều trong phỏng vấn senior.
+1 giờ (SUPPLEMENTARY): Contract testing Pact (5), test data builder (7).
Khi cần (ADVANCED): property-based testing, chaos testing, canary deploy, mutation testing để đo chất lượng test thật.

1. Bài toán thực tế — 80% coverage vẫn lọt P1 CORE ★

ShopFlow có 80% test coverage nhưng bug P1 vẫn lên production. CI chạy test mất 45 phút, chặn merge. Thêm một feature làm vỡ 50 test cũ — test brittle, check implementation detail thay vì behavior. Contract giữa OrderService và NotificationService vỡ âm thầm (API đổi, consumer chưa cập nhật). Integration test dùng H2, nhưng dialect khác PostgreSQL nên bug bị che tới production.

⏸ Pause & Predict — trước khi đọc lời giải
Viết ra giấy:
  1. 80% coverage mà vẫn lọt P1 — coverage đo cái gì, KHÔNG đo cái gì? Một loại test nào làm coverage cao mà giá trị thấp?
  2. Vì sao "thêm 1 feature làm vỡ 50 test"? Đặc điểm chung của 50 test đó là gì (gợi ý: chúng check cái gì)?
  3. H2 pass nhưng PostgreSQL fail — cho 2 ví dụ cụ thể về khác biệt dialect khiến bug lọt.
Đáp án ở mục 2 (coverage ≠ quality), anti-patterns (test implementation = brittle), mục 4 (H2 vs PostgreSQL: JSONB, native query, constraint). Đừng đọc tiếp khi chưa trả lời câu 2.

5 câu Senior phải trả lời

  1. Vì sao test pyramid quan trọng? Tỉ lệ Unit:Integration:E2E?
  2. Vì sao Testcontainers (PostgreSQL) tốt hơn H2? Khác biệt thật?
  3. Contract testing (Pact) là gì? Chặn breaking change API thế nào?
  4. "Coverage tốt" nghĩa là gì? (không chỉ 80%)
  5. Viết test không brittle thế nào? (test behavior, không implementation)

2. Test pyramid + tỉ lệ CORE ★

        E2E (ít)        — toàn hệ thống, chậm nhất, brittle nhất, confidence cao nhất
     Integration (vừa)  — nhiều component, DB thật, tốc độ trung bình
   Unit (nhiều)         — 1 class, mock dependency, nhanh, deterministic

Tỉ lệ tham chiếu: 70% Unit : 20% Integration : 10% E2E

WHY pyramid: unit nhanh (feedback <1s, parallel) nên làm nhiều; integration bắt bug thật (SQL, transaction) nên vừa phải; E2E chậm (1 phút/test) nên chỉ dành cho critical path.

🔧 Counter-example — inverted pyramid (kim tự tháp ngược)

Nếu 50% là E2E: CI mất 2 giờ, feedback chậm, không kham nổi nhiều E2E → ít test → bug lọt. E2E cũng brittle nhất (một thay đổi nhỏ ở UI/flow làm đỏ hàng loạt) và khó debug (lỗi ở đâu trong 8 service?). Quy tắc: đẩy assertion xuống tầng thấp nhất bắt được bug — đừng test một quy tắc tính giá bằng E2E khi unit test đủ.

3. Unit test — isolation đúng CORE ★

Test 1 class cô lập, mock mọi dependency (repository, external API). Nhanh, deterministic. KHÔNG test code của framework (đừng test Spring DI hay Hibernate — tin vendor).

✅ Test logic của OrderService

@Test
void appliesDiscountToOrderTotal() {
    var repo = mock(OrderRepository.class);
    var pricing = mock(PricingService.class);
    when(pricing.discountFor(1000)).thenReturn(50);

    var service = new OrderService(repo, pricing);
    Order o = service.place(1000);

    assertEquals(950, o.netTotal());   // behavior: total sau giảm giá
    verify(repo).save(any());
}

❌ Test framework (vô nghĩa)

@SpringBootTest
@Test
void orderServiceIsAutowired() {
    // đang test Spring DI, không test logic OrderService
    assertNotNull(orderService);
}
Test cả error path

Đừng chỉ test happy path. Test "khi pricing fail thì repository.save KHÔNG được gọi" (verify(repo, never()).save(any())) — đảm bảo không lưu một phần. Đây là chỗ bug thật hay trốn.

4. Integration test: Testcontainers vs H2 CORE ★

Integration test kiểm tra tương tác giữa component với DB thật. Testcontainers spin một PostgreSQL thật trong Docker → bắt được bug dialect mà H2 bỏ sót.

@Testcontainers
@SpringBootTest
class OrderIntegrationTest {
    @Container
    static PostgreSQLContainer<?> pg = new PostgreSQLContainer<>("postgres:15");

    @Autowired OrderService service;
    @Autowired OrderRepository repo;

    @Test
    void placingOrderWritesAuditEntry() {
        Order o = service.place(new Customer("C1"), 1000);
        // side-effect thật: audit log được ghi cùng transaction
        assertThat(repo.findById(o.getId())).isPresent();
        assertThat(auditRepo.findByOrderId(o.getId())).hasSize(1);
    }
}

L1 — Biết tên

Biết "Testcontainers chạy DB thật trong Docker". Symptom hiểu giả: vẫn dùng H2 cho tốc độ, không nêu được khác biệt cụ thể.

L2 — Explain được

Nêu được H2 khác PostgreSQL ở: native query, JSONB, window function, unique/foreign-key constraint behavior, kiểu dữ liệu. Giải thích vì sao "pass với H2 = false confidence".

L3 — Apply được

Thiết kế suite: dùng Testcontainers cho repository/integration, reuse container giữa test (singleton) để nhanh, tách stage CI (unit trước cho feedback, integration sau), chạy container song song để rút thời gian từ 45→10 phút.

5. Contract testing (Pact) SUPPLEMENTARY

Consumer (OrderService) kỳ vọng Provider (NotificationService) trả JSON cụ thể.

Consumer test: "Gọi /notify với payload X, tôi mong response Y"
   → Pact ghi lại kỳ vọng (contract)
Provider test: "Tôi có thoả mãn contract này không?" (verify)
   → Cả hai pass = API an toàn
   → Provider đổi API phá contract → contract test FAIL sớm trong CI,
     KHÔNG cần chạy cả hai service cùng lúc
Giá trị

Contract test bắt breaking change ngay trong CI của từng service, trước khi tới integration environment. Đây là thứ ngăn "API đổi, consumer vỡ âm thầm" ở mục 1.

6. @SpringBootTest slices — test nhanh hơn SUPPLEMENTARY

Full @SpringBootTest nạp toàn bộ context (chậm). Slice chỉ nạp phần cần:

SliceNạp gìTest cái gì
@WebMvcTestController + MockMvcrouting, JSON mapping, validation, error handling
@DataJpaTestJPA/Hibernate + DBquery, mapping, constraint
@JsonTestJacksonserialize/deserialize

Mỗi slice nạp ít context hơn → nhanh gấp ~10 lần full context.

7. Test data builder SUPPLEMENTARY

✅ Builder — đọc được, linh hoạt

Order o = anOrder()
    .withTotal(5000)
    .withDiscount(200)
    .forCustomer("C9")
    .build();

❌ Constructor 12 tham số

Order o = new Order(5000, 200, "C9", "C1",
    LocalDate.now(), "PENDING", null, false,
    true, 0, null, "SALE");  // brittle, khó đọc

🎯 3-tier interview framework

Tầng 1 — SurfaceJunior gate

Q: Giải thích test pyramid.

Nhiều unit (nhanh) : một số integration (chậm hơn, DB thật) : ít E2E (chậm nhất). Tỉ lệ ~70:20:10. Lý do: feedback nhanh + confidence, chi phí CI hợp lý.

Tầng 2 — Deep DiveSenior gate

Q: Test integration mất 45 phút, tối ưu.

Testcontainers reuse singleton + chạy song song; dùng @DataJpaTest slice thay full context; tách CI (unit trước cho feedback, integration sau); fixtures dùng chung tránh INSERT lặp; chỉ critical path mới integration. Mục tiêu ~10 phút.

Tầng 3 — ArchitectureSenior+

Q: 5 microservice, 80% coverage vẫn P1. Thiết kế chiến lược chất lượng.

Coverage ≠ quality. Thêm contract test (bắt breaking API), chaos test (giết pod ngẫu nhiên), canary deploy (1% traffic), observability (không test hết được thì quan sát prod), property-based test (sinh 10K input tìm edge case), mutation test để đo test có thật sự bắt bug.

🔧 Lab bắt buộc — viết & chứng minh (2 bài)

Lab 1 — Brittle vs behavior. Viết một test check verify(service).setStatus("PAID") (implementation). Refactor đổi tên method nội bộ → test đỏ dù hành vi không đổi. Viết lại test chỉ assert kết quả quan sát được (order.status() == PAID), refactor lại → test vẫn xanh. Tiêu chí: giải thích vì sao test thứ hai cho phép refactor tự do.

Lab 2 — H2 che bug. Viết một native query dùng cú pháp PostgreSQL (vd jsonb hoặc ON CONFLICT). Chạy test với H2 → pass hoặc lỗi khó hiểu; chạy với Testcontainers PostgreSQL → phản ánh đúng prod. Tiêu chí: chỉ ra đúng một câu SQL mà H2 và PostgreSQL hành xử khác nhau.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. Vì sao "80% coverage" không đảm bảo ít bug? Coverage đo gì?
  2. Tỉ lệ pyramid và lý do mỗi tầng nhiều/ít? Inverted pyramid hại ở đâu?
  3. Cho 2 khác biệt cụ thể H2 vs PostgreSQL khiến bug lọt.
  4. Contract test (Pact) chặn breaking change API thế nào, mà không cần chạy cả 2 service?
  5. "Test behavior, không test implementation" nghĩa là gì? Cho ví dụ một test brittle và bản sửa.
Bí câu nào = quay lại đúng section. Trôi cả 5 = nắm chắc testing mức senior.
