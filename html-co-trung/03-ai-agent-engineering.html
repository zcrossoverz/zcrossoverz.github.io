1. Bản chất — Agent ≠ Chatbot CORE ★

TypeFlow
ChatbotHỏi → Trả lời → Xong
AgentHỏi → Suy nghĩ → Gọi tool → Quan sát → Suy nghĩ → ... → Trả lời cuối

AI Agent = cổ trùng đặc biệt: có thể suy nghĩ ra bước cần làm, gọi tool, quan sát kết quả, lặp đến khi đạt mục tiêu. Nghề Nhân là build các agent này cho doanh nghiệp.

2. 9 khái niệm cốt lõi CORE ★

#1 Function Calling / Tool Use

Cho phép LLM gọi hàm bên ngoài thay vì chỉ trả text. Đây là bước nối giữa LLM và thế giới thật.

  1. Bạn cho LLM biết: "Có 3 tool: get_weather(city), book_meeting(time, person), send_email(to, subject, body)"
  2. User hỏi: "Đặt lịch họp với John ngày mai 3pm"
  3. LLM trả JSON: {"tool":"book_meeting","args":{"time":"tomorrow 3pm","person":"John"}}
  4. Code của bạn gọi function thật → đặt lịch
  5. LLM nhận kết quả → trả lời user

OpenAI: "function calling". Anthropic: "tool use". Cùng concept.

#2 RAG — Retrieval-Augmented Generation

Cho LLM trả lời dựa trên dữ liệu riêng không có trong training data (FAQ công ty, docs riêng, code repo).

📐 Setup (offline)

Documents → chunk (200-1000 từ) → embed → vector DB

🔍 Query (online)

Question → embed → semantic search top 3-5 chunks → đưa vào prompt → LLM trả lời

Khi nào KHÔNG dùng RAG: data nhỏ < 50KB (cho hết vào context), hoặc data thay đổi liên tục (không kịp index).

#3 Vector Embedding + Vector DB

Embedding: biến text thành vector số 1500 chiều sao cho text nghĩa giống → vector gần nhau.

"tôi yêu mèo"   → [0.23, -0.45, 0.67, ..., 0.12]
"tôi thích mèo" → [0.21, -0.43, 0.65, ..., 0.11]  ← gần
"đường về quê"  → [-0.55, 0.78, ..., -0.34]       ← xa

Provider: OpenAI text-embedding-3-small ($0.02/1M tokens), Voyage AI (chất lượng cao domain-specific), Cohere embed.

Vector DB: managed (Pinecone, Weaviate Cloud) hoặc self-host (Qdrant, Milvus, ChromaDB, pgvector).

💡 Đề xuất cho Nhân

Bắt đầu với pgvector (Postgres extension) — bạn đã biết Postgres. Không học DB mới. Production ready, sub-100ms search trên 1M vectors.

#4 Reranking + #5 Chunking Strategy

Reranker: sau vector search top 20, model rerank xếp lại sao cho chunk thực sự phù hợp lên đầu. Vector nhanh nhưng không chính xác hoàn toàn → rerank chậm hơn nhưng chính xác hơn → combo = nhanh + chính xác.

Provider: Cohere Rerank, Jina Reranker.

Chunking ảnh hưởng RỄ chất lượng RAG:

  • Quá nhỏ (50 từ): mất context
  • Quá lớn (5000 từ): vector "loãng", search không trỏ đúng
  • Theo ý: chia ở paragraph/header — giữ nghĩa

Tools: LangChain Text Splitters, LlamaIndex parsers.

#6 Prompt Engineering

Không phải "tip & trick" — là discipline. 6 kỹ thuật chính:

  1. Few-shot examples: 2-3 ví dụ trong prompt → LLM bắt chước
  2. Chain of Thought: "think step by step" — chính xác hơn
  3. Role prompting: "Bạn là chuyên gia X..."
  4. Output format: yêu cầu JSON/XML schema cụ thể → dễ parse
  5. Negative examples: "đừng làm như thế này"
  6. System vs User: tách system (bối cảnh) vs user (câu hỏi)

Đọc: Anthropic prompt engineering guide + OpenAI cookbook.

#7 MCP — Model Context Protocol

MCP = chuẩn Anthropic công bố cuối 2024, cho LLM kết nối nguồn dữ liệu/tool theo cách chuẩn hoá. Trước: mỗi integration custom code. Có MCP: tool exposes 1 MCP server → bất kỳ LLM nào support đều dùng được. USB cho AI tools.

🎯 Vì sao quan trọng cho Nhân (2026)

Rất ít dev hiểu MCP lúc này. Đây là edge của edge — rành MCP + build MCP servers = top 5% AI builder. Charge premium.

#8 Eval + #9 Observability

Eval ≠ test code:

  • Code test: input → output đúng/sai (deterministic)
  • AI eval: input → output đúng nhiều cách → phải đo: relevance, accuracy, safety, latency, cost

Tools: Promptfoo (open source, local), DSPy (auto-optimize prompt), Langfuse (observability + eval), Phoenix/Arize.

🚨 Quy tắc vàng

Production AI không có eval = code clean nhưng deploy là chết. Enterprise hỏi "how do you measure quality?" → không trả lời = mất deal.

Observability: log mỗi LLM call (prompt, response, latency, cost), trace tool chain, alert cost/latency spike, dashboard tổng hợp.
