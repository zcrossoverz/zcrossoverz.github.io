CI/CD Pipeline — Deploy An Toàn, Rollback Trong Giây

"Có pipeline tự build & deploy" là CI/CD ở mức bề mặt. Lên thật: "blue-green và canary khác nhau ở rủi ro nào?", "vì sao đổi schema DB và đổi code phải tách 2 lần deploy?", "feature flag không xoá thành nợ kỹ thuật ra sao?" — là lộ hiểu giả. File này giải thích cơ chế deployment strategy, expand-contract migration, và vì sao tách deploy khỏi release — đọc một lần là hiểu. Domain: ShopFlow — checkout-api của sàn e-commerce.

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Deploy = release. Push code lên là user dùng feature mới."
Thực tế: Deploy (đưa code lên server, code đã chạy nhưng feature còn ẩn) và release (bật feature cho user thấy) là 2 việc tách được bằng feature flag. Tách ra cho phép: deploy bất cứ lúc nào (kể cả code chưa hoàn thiện nhưng tắt cờ), bật feature theo lịch ra mắt, và tắt ngay khi lỗi mà không cần rollback.
Tưởng hiểu: "Đổi schema DB và đổi code cứ deploy chung một lần cho gọn."
Thực tế: Trong lúc rolling update, pod chạy code cũ và pod chạy code mới tồn tại đồng thời vài phút. Nếu bạn drop/rename cột cùng lúc, pod cũ (vẫn đọc cột cũ) sẽ vỡ. Phải expand-contract: thêm cột tương thích ngược trước, deploy code, rồi mới xoá cột ở lần deploy sau.
Tưởng hiểu: "Canary và blue-green na ná nhau, đều deploy an toàn."
Thực tế: Khác nhau ở "blast radius" (vùng ảnh hưởng khi có lỗi). Blue-green chuyển 100% traffic tức thì sang version mới (rollback nhanh nhưng nếu lỗi thì mọi user dính ngay). Canary cho version mới hứng 1%→10%→100% dần theo metric (lỗi chỉ chạm một ít user, nhưng cần đo để quyết định promote).

1. Bài toán thực tế — deploy thứ Sáu = nín thở CORE ★

ShopFlow deploy bằng tay, mỗi lần ~30 phút downtime, cả team thức canh. Một lần đổi schema (rename cột) làm pod cũ vỡ giữa rolling update → lỗi 500 hàng loạt. Rollback phải build lại image từ đầu, mất 40 phút. Và mỗi feature mới phải canh deploy đúng đêm ra mắt vì không có cách nào "đưa code lên trước, bật sau". Cả ba vấn đề đều có lời giải đã được ngành đúc kết.

⏸ Pause & Predict — trước khi đọc lời giải
Viết ra giấy:
  1. "Rename cột làm pod cũ vỡ giữa rolling update" — vì sao? Pod cũ và pod mới có chạy đồng thời không? Cách migration nào tránh được?
  2. "Feature phải deploy đúng đêm ra mắt" — kỹ thuật nào cho phép deploy trước, bật sau?
  3. Muốn rollback trong <1 phút thay vì build lại 40 phút, cần chuẩn bị gì từ trước (gợi ý: artifact)?
Đáp án: rolling update chạy song song code cũ+mới → cần expand-contract migration (2.5); feature flag tách deploy/release (2.4); rollback nhanh = giữ sẵn artifact/image version cũ để switch về, không build lại (2.6).

2.1 CI vs Continuous Delivery vs Continuous Deployment SUPPLEMENTARY

Ba khái niệm là ba mức tự động hoá nối tiếp, hay bị gọi lẫn:

  • Continuous Integration (CI): mỗi lần push, hệ thống tự build + chạy test, và lập trình viên merge code vào nhánh chính thường xuyên (trunk-based). Mục tiêu: phát hiện xung đột/lỗi sớm, khi thay đổi còn nhỏ và dễ sửa.
  • Continuous Delivery (CD): hệ thống luôn ở trạng thái có thể deploy bất cứ lúc nào — mọi commit qua được pipeline là một artifact sẵn sàng — nhưng việc đẩy lên production vẫn cần một người bấm nút.
  • Continuous Deployment: bước tiếp theo — bỏ luôn nút bấm: commit nào pass toàn bộ pipeline thì tự động lên production. Cần độ tin cậy test rất cao + cơ chế canary/rollback tự động.

2.2 Pipeline stages + fail-fast CORE ★

Pipeline là một chuỗi giai đoạn (stage), mỗi stage là một cổng phải qua mới đến stage sau:

checkout → build → unit test → lint/static analysis → package (image)
        → integration test → security scan → deploy staging → smoke test
        → [approval gate] → deploy prod (canary) → monitor → promote / rollback

Nguyên tắc fail-fast: xếp các stage rẻ & nhanh trước, đắt & chậm sau. Unit test (vài giây) chạy trước integration test (vài phút) chạy trước E2E. Lý do: nếu một thay đổi sai về logic, unit test bắt nó trong vài giây và pipeline dừng ngay — không tốn tài nguyên/thời gian chạy tiếp các stage đắt. Một stage đỏ chặn mọi stage sau, cho dev feedback nhanh nhất có thể.

2.3 Blue-green / Canary / Rolling — khác nhau ở blast radius CORE ★

Ba chiến lược đưa version mới ra production, khác nhau ở bao nhiêu user dính lỗi nếu version mới hỏng (blast radius) và rollback nhanh thế nào:

Chiến lượcCách hoạt độngBlast radius / rollback
RollingThay pod cũ bằng pod mới dần dần (mặc định K8s)Code cũ + mới chạy chung trong lúc rollout (→ cần tương thích, xem 2.5); rollback = rollout undo
Blue-GreenDựng môi trường mới (green) song song môi trường cũ (blue), test xong thì chuyển 100% traffic tức thìNếu lỗi thì mọi user dính ngay; nhưng rollback = chuyển traffic về blue trong vài giây. Tốn gấp đôi tài nguyên (2 môi trường)
CanaryCho version mới hứng % traffic nhỏ trước (1%), theo dõi metric, tăng dần 10%→50%→100% nếu ổnLỗi chỉ chạm ít user (chính là ý nghĩa "canary" — con chim thử khí độc trong mỏ); cần observability tốt để quyết định promote/rollback theo metric

L1 — Biết tên

Biết tên 3 chiến lược. Symptom hiểu giả: không nói được khác biệt blast radius hay khi nào chọn cái nào.

L2 — Explain được

Blue-green = switch toàn bộ (rollback nhanh nhưng blast radius lớn, tốn 2× tài nguyên); canary = phơi bày dần dựa trên metric (bắt lỗi sớm trên ít user, cần observability); rolling = mặc định nhưng phải lo tương thích cũ/mới.

L3 — Apply được

Chọn theo ngữ cảnh: thay đổi rủi ro cao + có metric tốt → canary; cần rollback tức thì + đủ tài nguyên → blue-green; thay đổi nhỏ thường ngày → rolling. Ghép feature flag để giảm rủi ro thêm một lớp.

2.4 Feature flags — tách deploy khỏi release CORE ★

Feature flag (cờ tính năng) là một điều kiện bật/tắt trong code: if (flags.isOn("new-checkout")) { ... } else { ... }. Nó tách hai việc trước đây dính liền — deploy (đưa code mới lên, code đã chạy nhưng nằm sau cờ tắt) và release (bật cờ cho user thấy). Khi tách được, bạn có ba siêu năng lực: (1) merge & deploy code chưa hoàn thiện nhưng để cờ tắt (trunk-based development — không cần nhánh feature sống lâu); (2) bật feature cho một % nhỏ user trước (rollout dần), theo dõi rồi mở rộng; (3) khi feature lỗi, tắt cờ để khôi phục ngay lập tức — không cần redeploy hay rollback.

🔧 Counter-example — feature flag thành nợ kỹ thuật

Flag mạnh nhưng không miễn phí. Lạm dụng biến nó thành gánh nặng:

  • Flag không xoá: để mãi sau khi đã rollout 100% → codebase đầy if(flag) chết; mỗi flag nhân đôi số đường code phải đọc & test.
  • Flag lồng nhau: 5 flag độc lập = 32 tổ hợp trạng thái, không ai test hết → bug tổ hợp khó tái hiện.
  • Dùng flag thay config có cấu trúc: biến mọi thứ thành flag → mất khả năng suy luận hệ thống đang ở trạng thái nào.

Quy tắc: flag "release" là tạm thời — gắn ngày hết hạn và task dọn ngay khi rollout xong. Flag dài hạn (kill-switch, entitlement theo gói) thì phân loại rõ ràng và quản lý riêng.

2.5 DB migration backward-compatible (expand-contract) CORE ★

Gốc của vấn đề: rolling update (và canary, blue-green trong lúc chuyển) đều có giai đoạn code cũ và code mới cùng chạy trên cùng một database. Nếu schema thay đổi không tương thích ngược (drop/rename cột), pod chạy code cũ — vẫn truy vấn cột cũ — sẽ vỡ ngay khi cột biến mất. Vì vậy mọi thay đổi schema phải đi qua các bước tương thích ngược, gọi là expand-contract:

Đổi tên cột full_name → display_name (KHÔNG drop ngay):
1. EXPAND   : thêm cột display_name; code mới ghi CẢ HAI cột, đọc display_name (fallback full_name)
              → lúc này code cũ vẫn đọc full_name OK, code mới đọc display_name OK
2. MIGRATE  : backfill display_name từ full_name cho dữ liệu cũ (chạy batch)
3. (deploy phiên bản code chỉ còn dùng display_name) → đợi mọi pod cũ biến mất
4. CONTRACT : sau khi chắc KHÔNG pod nào còn đọc full_name → drop full_name (ở lần deploy SAU)
Anti-pattern + symptom

Triệu chứng: ngay sau deploy, log đầy column "full_name" does not exist phát từ các pod CŨ chưa kịp thay. Nguyên nhân: drop cột cùng lúc deploy code mới. Fix: tách bước drop (contract) sang lần deploy sau, khi đã chắc không còn pod cũ; dùng Liquibase/Flyway với changeset có version & rollback (3.2).

2.6 Rollback theo loại thay đổi SUPPLEMENTARY

"Rollback" không phải một thao tác duy nhất — độ khó phụ thuộc loại thay đổi. Mấu chốt để rollback nhanh là giữ sẵn artifact cũ (image version trước) để switch về, thay vì build lại từ source (mất hàng chục phút):

Loại thay đổiCách rollback
Code (stateless)kubectl rollout undo / switch blue-green / tắt feature flag — vài giây
DB schema (đã expand, chưa contract)Rollback code an toàn vì schema vẫn tương thích ngược
DB schema (đã contract/drop)KHÔNG rollback dễ — đây chính là lý do tách bước contract sang lần sau
Data migration (đã đổi dữ liệu)Cần compensating script; phải backup trước khi chạy

3.1 GitHub Actions workflow SUPPLEMENTARY

name: ci-cd
on: { push: { branches: [main] } }
jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with: { distribution: temurin, java-version: '21', cache: maven }
      - run: mvn -B verify                       # build + unit + integration (nhanh trước)
      - run: docker build -t shopflow/checkout-api:${{ github.sha }} .
      - name: Security scan (fail nếu CVE nặng)
        run: trivy image --exit-code 1 --severity CRITICAL shopflow/checkout-api:${{ github.sha }}
      - run: docker push shopflow/checkout-api:${{ github.sha }}
  deploy-prod:
    needs: build-test
    environment: production                       # GitHub environment = approval gate
    runs-on: ubuntu-latest
    steps:
      - name: Bump image tag trong GitOps repo    # ArgoCD sẽ tự sync về cluster
        run: yq -i '.image.tag="${{ github.sha }}"' manifests/checkout/values.yaml && git commit -am ci && git push

3.2 Liquibase changeset (expand-contract) SUPPLEMENTARY

# db/changelog/2026-05-add-display-name.yaml  (bước EXPAND — release N)
databaseChangeLog:
  - changeSet:
      id: add-display-name
      author: nhan
      changes:
        - addColumn:
            tableName: users
            columns: [{ column: { name: display_name, type: varchar(255) } }]
      rollback:
        - dropColumn: { tableName: users, columnName: display_name }
# release N: code ghi cả full_name + display_name, đọc display_name fallback full_name
# release N+1: backfill xong → changeset CONTRACT drop full_name (KHÔNG cùng release với expand)

3.3 Feature flag + GitOps SUPPLEMENTARY

// rollout dần ổn định: % theo hash userId → cùng user luôn vào cùng nhánh (không nhảy qua lại)
boolean newCheckoutOn(String userId) {
  if (!flags.isEnabled("new-checkout")) return false;
  int pct = flags.rolloutPercent("new-checkout");        // 5 → 100 tăng dần
  return Math.floorMod(userId.hashCode(), 100) < pct;
}
// lỗi → set rolloutPercent=0 ở flag service → TẮT ngay, KHÔNG redeploy/rollback

GitOps (ArgoCD): trạng thái mong muốn của cluster được khai báo trong một Git repo; ArgoCD liên tục so sánh cluster thật với Git và tự đồng bộ về đúng Git. Lợi ích: mọi thay đổi có audit trail (commit), và rollback = git revert — quay về một trạng thái đã biết.

5. 3-tier interview framework

Tầng 1 — SurfaceJunior gate

Q: CI khác CD thế nào?

CI = mỗi push build+test+merge thường xuyên (bắt lỗi sớm). Continuous Delivery = luôn ở trạng thái deploy được, nhưng bấm nút mới deploy. Continuous Deployment = pass pipeline thì tự động deploy production (cần test tin cậy cao + canary/rollback tự động).

Tầng 2 — Deep DiveSenior gate

Q: Deploy đổi schema không downtime, có rollback.

Expand-contract: thêm cột tương thích ngược → deploy code đọc/ghi cả hai → backfill → deploy code chỉ dùng cột mới → drop cột ở lần sau. Trong lúc rolling, code cũ+mới cùng chạy nên schema phải tương thích ngược; code rollback an toàn vì cột cũ còn đó. Dùng Flyway/Liquibase versioned + rollback changeset.

Tầng 3 — ArchitectureSenior+

Q: Thiết kế progressive delivery cho thay đổi rủi ro cao.

Canary 1%→10%→50%→100% gắn metric gate (error rate, p99, business KPI); auto-rollback nếu vượt ngưỡng (Argo Rollouts/Flagger); bọc feature flag để tắt nhanh không cần rollback; GitOps để audit + revert; approval gate cho bước 100%.

🔧 Lab bắt buộc — thực hành deploy an toàn (2 bài)

Lab 1 — Expand-contract migration. Với bảng users(full_name), đổi tên sang display_name qua 4 bước expand-contract bằng Liquibase. Giữa bước, chạy đồng thời "code cũ" (đọc full_name) và "code mới" (đọc display_name) → chứng minh cả hai chạy được. Tiêu chí: giải thích vì sao KHÔNG drop ở bước 1.

Lab 2 — Feature flag tắt nhanh. Bọc một feature sau flag (config/env). Bật cho "10% request" bằng hash userId. Giả lập lỗi → tắt flag (không redeploy) và xác nhận hết lỗi tức thì. Sau đó viết task "dọn flag". Tiêu chí: nói được khác biệt deploy vs release.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. Deploy và release khác gì? Feature flag tách chúng thế nào, cho 3 lợi ích?
  2. Blue-green vs canary khác nhau ở blast radius và rollback ra sao?
  3. Vì sao drop cột cùng lúc deploy code gây lỗi (pod cũ+mới cùng chạy)? Expand-contract sửa thế nào?
  4. Làm sao rollback trong giây thay vì build lại 40 phút?
  5. Khi nào feature flag trở thành nợ kỹ thuật?
Bí câu nào = quay lại đúng section. Trôi cả 5 không nhìn tài liệu = nắm chắc CI/CD mức senior.
