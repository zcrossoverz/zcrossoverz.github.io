Camunda & BPMN — Workflow Engine Đúng Cách

BPMN nhìn như "vẽ flowchart cho đẹp" — hiểu giả ở đây làm workflow thành mớ rối khó sửa. Phải hiểu process engine quản state & persistence ra sao, user task khác service task, vì sao service task phải idempotent. File này dạy Camunda cho ShopFlow — quy trình duyệt hoàn tiền (refund approval) của sàn e-commerce.

🎯 Mục tiêu
Mô hình quy trình có người + tự động, bền với crash, dễ sửa
📅 Thời lượng
1 tuần · Sau #18 Event-Driven, #17 Microservices
📚 Prereq
Spring Boot · async · state machine
🔥 Độ ưu tiên

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "BPMN chỉ là sơ đồ để giải thích cho business, code vẫn viết tay."
Thực tế: Với process engine (Camunda), BPMN là executable — engine chạy sơ đồ thật, lưu state mỗi bước vào DB, recover sau crash, đợi người/timer. Coi nó là "tài liệu" → bỏ phí giá trị chính: state persistence + visibility.
Tưởng hiểu: "Service task chỉ là gọi một hàm, chạy lại cũng không sao."
Thực tế: Engine có thể retry service task khi crash/timeout → nếu task không idempotent (vd gọi hoàn tiền) sẽ hoàn 2 lần. Service task BẮT BUỘC idempotent vì engine đảm bảo at-least-once.
Tưởng hiểu: "Nhồi business logic vào expression trong BPMN cho gọn."
Thực tế: Logic phức tạp trong expression XML = không test được, không debug, không version đẹp. BPMN giữ luồng; logic để trong code (delegate/external task) test được.

1. Bài toán thực tế — refund approval rối CORE ★

ShopFlow xử lý hoàn tiền: nếu < 500K tự duyệt; ≥ 500K cần quản lý duyệt tay; quá hạn 48h chưa duyệt thì tự escalate; sau duyệt gọi PSP hoàn tiền. Bản tự code bằng if-else + cron + flag trạng thái → khó theo dõi đơn đang ở bước nào, crash giữa chừng mất state, gọi hoàn tiền 2 lần khi retry.

⏸ Pause & Predict — trước khi đọc lời giải
Viết ra giấy:
  1. Quy trình này có "chờ người duyệt" (có thể vài ngày) và "chờ timer 48h" — vì sao tự code bằng flag + cron dễ mất state khi service restart?
  2. Bước "gọi PSP hoàn tiền" bị retry sau crash → hoàn 2 lần. Thuộc tính nào của task ngăn được?
  3. Process engine cho bạn cái gì mà if-else + DB flag không cho (gợi ý: 2 thứ)?
Đáp án: engine lưu state mỗi bước vào DB nên restart không mất (mục 2); service task phải idempotent (anti-patterns); engine cho persistence của long-running state + visibility (đơn đang ở bước nào). Đừng đọc tiếp khi chưa thấy giá trị "state bền".

2. BPMN elements + process engine CORE ★

BPMN cơ bản: start/end event, task (việc cần làm), gateway (rẽ nhánh: exclusive = một đường, parallel = nhiều đường), sequence flow (mũi tên), timer/message event. Process engine đọc BPMN, tạo process instance cho mỗi đơn, di chuyển token qua các bước, và lưu state vào DB tại mỗi điểm chờ (wait state) → crash rồi restart vẫn tiếp tục đúng chỗ.

L1 — Biết tên

Vẽ được sơ đồ BPMN. Symptom hiểu giả: không giải thích được engine lưu state ở đâu hay vì sao recover được sau crash.

L2 — Explain được

Engine persist token + biến process vào DB tại mỗi wait state (user task, timer, message). Long-running (chờ người duyệt 3 ngày) không giữ thread — chỉ là row trong DB. Restart → load lại, tiếp tục.

L3 — Apply được

Thiết kế quy trình: tách wait state rõ ràng, service task idempotent, dùng external task worker để scale; chọn dùng engine khi quy trình dài + có người + cần audit/visibility; tự code khi luồng ngắn đơn giản.

4. Timer + error handling SUPPLEMENTARY

Timer event: chờ một khoảng (48h) hoặc tới mốc → tự escalate/timeout mà không cần cron riêng. Error handling: phân biệt business error (PSP từ chối hoàn — rẽ nhánh BPMN xử lý) vs technical error (timeout — engine retry với backoff, hết retry → incident cần người can thiệp).

5. Message correlation SUPPLEMENTARY

Khi PSP gửi callback "đã hoàn tiền cho refund #R123", engine phải route message tới đúng process instance đang chờ. Correlation dùng business key (refundId) để khớp message ↔ instance. Sai correlation → message rơi vào instance khác hoặc bị bỏ.

⚠️ Anti-patterns CORE ★

🔧 Counter-example — khi dùng BPMN engine là SAI / dùng sai
  • Service task không idempotent: engine retry → hoàn tiền/gửi mail 2 lần. Mọi service task phải idempotent (key).
  • Task quá mịn: chia 30 task cho việc tuyến tính đơn giản → sơ đồ rối, mỗi bước một transaction, chậm. Engine hợp với quy trình có wait state & người, không phải mọi for-loop.
  • Business logic trong expression XML: không test/debug được. Để trong code.
  • Dùng engine cho luồng ngắn không có người: over-engineering — một REST call đồng bộ đơn giản không cần workflow engine.

Quy tắc: chọn workflow engine khi quy trình dài, có bước con người, cần audit/visibility, nhiều rẽ nhánh & timer. Luồng ngắn thuần code → đừng dùng.

🎯 3-tier interview framework

Tầng 1 — SurfaceMid gate

Q: User task vs service task?

User task = engine tạo việc cho con người & chờ họ complete (wait state, có thể lâu). Service task = engine gọi code tự động; phải idempotent vì engine có thể retry.

Tầng 2 — Deep DiveSenior gate

Q: Vì sao dùng Camunda thay vì tự code refund flow?

Quy trình dài (chờ người duyệt, timer 48h) → engine persist state vào DB, recover sau crash, không giữ thread; visibility (đơn ở bước nào); audit; rẽ nhánh + escalation khai báo được. Tự code = flag + cron dễ mất state, khó theo dõi.

Tầng 3 — ArchitectureSenior+

Q: Khi nào KHÔNG dùng workflow engine?

Luồng ngắn, đồng bộ, không có người, ít rẽ nhánh → code thường rẻ & rõ hơn. Engine thêm hạ tầng (DB engine, học BPMN). Dùng khi quy trình dài + người + audit + nhiều biến thể; cân nhắc orchestration nhẹ (Saga) cho microservices thuần tự động.

🔧 Lab bắt buộc — build & phá (2 bài)

Lab 1 — Refund process. Vẽ & chạy một BPMN: start → exclusive gateway (<500K tự duyệt / ≥500K user task) → service task gọi "PSP refund" (mock) → end. Thêm timer boundary 48h trên user task → escalate. Tiêu chí: kill app giữa lúc chờ duyệt, khởi động lại, chứng minh instance vẫn ở đúng bước.

Lab 2 — Idempotency của service task. Làm service task "PSP refund" non-idempotent (cộng tiền mỗi lần gọi). Ép nó throw sau khi đã hoàn → engine retry → hoàn 2 lần. Sửa bằng idempotency key. Tiêu chí: giải thích at-least-once của engine.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. Process engine lưu state ở đâu? Vì sao recover được sau crash?
  2. User task vs service task khác gì? Cái nào là wait state?
  3. Vì sao service task BẮT BUỘC idempotent?
  4. Message correlation giải quyết vấn đề gì khi PSP callback về?
  5. Khi nào KHÔNG nên dùng workflow engine?
Bí câu nào = quay lại đúng section. Trôi cả 5 = biết khi nào dùng & dùng đúng engine.
