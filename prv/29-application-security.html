Application Security — OWASP, Injection, CSRF, IDOR

Security là nơi hiểu giả dễ chết người nhất: "có dùng prepared statement", "có HTTPS" nghe đủ — nhưng "vì sao parameterized query chặn injection ở tầng nào?", "CSRF cần token cho form nhưng KHÔNG cần cho JWT API — vì sao?", "IDOR khác broken auth chỗ nào?" thì lộ. File này dạy cơ chế tấn công và phòng thủ kèm code thật, trên domain ShopFlow — tài khoản & đơn hàng của sàn e-commerce.

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Dùng prepared statement là an toàn SQL injection. Escape input cũng được."
Thực tế: Parameterized query an toàn vì tách câu lệnh khỏi dữ liệu — DB nhận query đã compile + data là tham số, data không bao giờ được parse như SQL. Tự escape thì luôn sót case (Unicode, encoding). Nhưng nối chuỗi vào tên bảng/cột/ORDER BY (không tham số hoá được) vẫn injectable.
Tưởng hiểu: "CSRF với CORS là một, đều về cross-origin."
Thực tế: CSRF = trình duyệt tự gửi cookie/session khi bị dụ submit form sang site bạn → cần CSRF token. CORS = cơ chế trình duyệt cho phép/chặn JS đọc response cross-origin. JWT trong header (không phải cookie) thì KHÔNG bị CSRF (trình duyệt không tự gắn custom header).
Tưởng hiểu: "Đã login (authentication) là an toàn truy cập."
Thực tế: Authentication (bạn là ai) ≠ authorization (bạn được làm gì). IDOR: user A login hợp lệ nhưng đổi orderId trên URL để xem đơn của user B. Phải check ownership ở server, không tin ID client gửi.

1. Bài toán thực tế — rò rỉ & chiếm đơn CORE ★

ShopFlow gặp 3 sự cố: (1) một query search nối chuỗi input → kẻ tấn công dump cả bảng user; (2) user đổi ?orderId=1002 trên URL xem được đơn người khác; (3) form đổi email bị site khác submit hộ qua cookie → tài khoản bị chiếm.

⏸ Pause & Predict — trước khi đọc lời giải
Đặt tên đúng từng lỗ hổng và cách chặn:
  1. Sự cố (1) tên là gì? Vì sao "escape dấu nháy" không đủ mà parameterized query mới đúng?
  2. Sự cố (2) tên là gì? Vì sao "đã login" không ngăn được? Check gì ở server?
  3. Sự cố (3) tên là gì? Vì sao JWT trong header sẽ không bị, nhưng session cookie thì bị?
Đáp án: (1) SQL Injection — parameterized tách lệnh khỏi data (2.2); (2) IDOR — check ownership server, không tin ID client (2.5); (3) CSRF — cần token; cookie tự gửi còn JWT header thì không (2.4). Đừng đọc tiếp khi chưa đặt đúng 3 tên.

2.1 OWASP Top 10 — cốt lõi SUPPLEMENTARY

Một số mục hay gặp trong backend e-commerce: Broken Access Control (gồm IDOR), Injection (SQL/NoSQL/command), Cryptographic Failures (PII không mã hoá), Security Misconfiguration, Identification & Authentication Failures (thiếu rate limit, session yếu), SSRF. Học theo cơ chế chứ không thuộc danh sách.

2.2 SQL Injection — vì sao parameterized fix CORE ★

❌ Nối chuỗi — injectable

"SELECT * FROM users WHERE email='" + input + "'"
// input = x' OR '1'='1' --  → trả mọi user (1=1 luôn đúng)

✅ Parameterized — tách lệnh khỏi data

jdbc.query("SELECT * FROM users WHERE email = ?", input);
// query compile TRƯỚC; input là DATA, DB không parse nó như SQL
Vẫn injectable dù dùng PreparedStatement

Tham số chỉ thay cho giá trị, không thay cho tên bảng/cột/ORDER BY. "ORDER BY " + sortColumn vẫn injectable → phải whitelist sortColumn trong tập cho phép. ORM mostly safe (JPA dùng parameterized mặc định) nhưng createNativeQuery() + nối chuỗi = vulnerable.

Second-order injection

Data từ nguồn "an toàn" (DB) nhưng khi lấy ra dùng lại (log message, render HTML) mà không sanitize → injection/XSS bậc hai. Sanitize tại điểm dùng, không chỉ tại điểm nhập.

2.3 XSS — 3 biến thể + 4 tầng chặn SUPPLEMENTARY

LoạiCơ chế
ReflectedScript trong URL phản chiếu vào trang: /search?q=<script>...</script>
StoredScript lưu vào DB (tên/ review) → hiện cho mọi người, vd admin xem danh sách → trigger
DOM-basedJS client ghi input vào DOM qua innerHTML (parse HTML); dùng textContent thì an toàn
 
4 tầng phòng thủ XSS (defense in depth):
1. Input validation   — whitelist ký tự cho phép
2. Output encoding     — convert "<" → "<" khi RENDER (OWASP Java Encoder) ← tầng chính
3. Content-Security-Policy header — trình duyệt không chạy inline script dù bị inject
4. HttpOnly cookie flag — JavaScript không đọc được cookie (chống đánh cắp session)

Cốt lõi: XSS được chặn chủ yếu ở output encoding theo ngữ cảnh (HTML/JS/URL), không chỉ ở input validation.

2.4 CSRF vs CORS + SameSite CORE ★

CSRF attack (lợi dụng cookie tự gửi):
1. Victim đang login ShopFlow (trình duyệt giữ session cookie)
2. Victim vào attacker.com — trang này có form ẩn POST tới shopflow.com/account/email
3. Form auto-submit; trình duyệt TỰ ĐỘNG kèm session cookie (same-origin cho cookie)
4. ShopFlow nhận request "đã xác thực" → đổi email nạn nhân

CSRF token chặn:
- Form chứa token ngẫu nhiên lưu ở server session (không ở cookie)
- attacker.com gửi được form NHƯNG không đọc được token (cross-origin) → thiếu token → fail

L1 — Biết tên

Biết "CSRF cần token". Symptom hiểu giả: nghĩ CSRF và CORS là một, hoặc bật CSRF token cho REST API stateless dùng JWT (thừa).

L2 — Explain được

CSRF lợi dụng việc trình duyệt tự gửi cookie. Token chống vì site khác không đoán được. CORS là cơ chế khác: kiểm soát JS đọc response cross-origin — không phải để chống CSRF. JWT header không bị CSRF.

L3 — Apply được

Session cookie → CSRF token hoặc SameSite. JWT header → không cần CSRF token. Chọn cơ chế chống theo cách lưu credential.

SameSite cookie:
  Strict — không bao giờ gửi cookie cho cross-site request
  Lax    — gửi cho cross-site nhưng chỉ safe method (GET)  ← default hợp lý
  None   — gửi mọi nơi (bắt buộc Secure = HTTPS)

2.5 IDOR — đừng tin client ID CORE ★

// ❌ tin orderId từ URL → user xem đơn người khác
Order o = orderRepo.findById(orderId).orElseThrow();

// ✅ check ownership: ràng buộc theo user đang đăng nhập
Order o = orderRepo.findByIdAndCustomerId(orderId, currentUser.getId())
                   .orElseThrow(NotFoundException::new);   // trả 404, không lộ tồn tại

Fix phải ở backend (service layer) — không fix ở frontend (attacker bypass) hay log (đã bị khai thác). Trả 404 chứ không 403 để không tiết lộ resource có tồn tại.

3.1 Controller: parameterized + validation CORE ★

@RestController
@RequestMapping("/orders")
class OrderController {
  private final OrderService service;

  @PostMapping
  ResponseEntity<OrderResponse> create(
      @Valid @RequestBody OrderRequest req,            // Bean Validation server-side
      @AuthenticationPrincipal UserPrincipal user) {
    OrderResponse r = service.place(user.id(), req);   // dùng user từ token, KHÔNG từ body
    return ResponseEntity.status(CREATED).body(r);
  }

  @GetMapping("/{id}")
  OrderResponse get(@PathVariable Long id, @AuthenticationPrincipal UserPrincipal user) {
    return service.getOwned(id, user.id());            // check ownership chống IDOR
  }
}

record OrderRequest(
    @NotBlank String sku,
    @Min(1) @Max(999) int quantity,                    // validate biên server-side
    @Pattern(regexp = "[A-Z0-9-]{0,16}") String coupon // whitelist ký tự
) {}

Repository dùng JPA/parameterized; nếu cần native query động (sort), whitelist cột: Set.of("created_at","total").contains(sortCol) trước khi ghép.

3.2 Security headers filter SUPPLEMENTARY

@Component
class SecurityHeadersFilter extends OncePerRequestFilter {
  @Override protected void doFilterInternal(HttpServletRequest req,
      HttpServletResponse res, FilterChain chain) throws ... {
    res.setHeader("Content-Security-Policy", "default-src 'self'");   // chặn inline script
    res.setHeader("X-Content-Type-Options", "nosniff");
    res.setHeader("X-Frame-Options", "DENY");                        // chống clickjacking
    res.setHeader("Strict-Transport-Security", "max-age=31536000");  // HSTS, ép HTTPS
    res.setHeader("Referrer-Policy", "no-referrer");
    chain.doFilter(req, res);
  }
}
// Cookie: Set-Cookie: session=...; HttpOnly; Secure; SameSite=Lax

3.3 PII masking + rate limit SUPPLEMENTARY

// PII masking — KHÔNG log email/thẻ/địa chỉ đầy đủ
static String maskEmail(String e) {           // a***@mail.com
  int at = e.indexOf('@');
  return at <= 1 ? "***" : e.charAt(0) + "***" + e.substring(at);
}
static String maskCard(String c) {            // **** **** **** 1234
  return "**** **** **** " + c.substring(c.length() - 4);
}
log.info("order placed customer={} card={}", maskEmail(email), maskCard(card));

// Rate limit login chống brute-force / credential stuffing (Redis token bucket)
boolean allow(String ip, String account) {
  return bucketFor("login:ip:" + ip).tryConsume(1)        // theo IP
      && bucketFor("login:acct:" + account).tryConsume(1); // theo account
}
// vượt N lần → exponential backoff + captcha; khoá tạm account khi nghi credential stuffing

4. Anti-patterns CORE ★

🔧 Counter-example — khi "phòng thủ" lại tạo lỗ hổng / sai chỗ
  • Tự escape thay parameterized: blacklist ký tự luôn sót (Unicode, comment, encoding). Phòng thủ đúng tầng là tách lệnh/data, không lọc input.
  • Verbose error: trả stack trace/SQL ra client = chỉ đường cho attacker. Log chi tiết server, trả message chung cho client.
  • Validate ở client là đủ: client validation chỉ cho UX; attacker bỏ qua client → mọi check bảo mật phải ở server.
  • Bật CSRF token cho JWT API stateless: thừa, gây hiểu sai mô hình; ngược lại quên SameSite/HttpOnly cho session cookie mới là lỗ hổng.

Quy tắc: chặn ở đúng tầng (parameterized cho injection, output encoding cho XSS, ownership check cho IDOR), và không tin bất cứ gì từ client.

5. 3-tier interview framework

Tầng 1 — SurfaceJunior gate

Q: Chặn SQL injection thế nào?

Parameterized query/PreparedStatement — tách câu lệnh khỏi dữ liệu, data không bao giờ parse như SQL. Whitelist cho phần không tham số hoá được (tên cột, ORDER BY). Không tự escape.

Tầng 2 — Deep DiveSenior gate

Q: Thiết kế API xử lý PII an toàn.

TLS bắt buộc; authorization theo ownership (chống IDOR); PII mã hoá at-rest, masking trong log; rate limit; input validation server-side + output encoding; security headers (CSP/HSTS/X-Frame); cookie HttpOnly+Secure+SameSite; audit log truy cập PII; least-privilege DB account.

Tầng 3 — ArchitectureSenior+

Q: CSRF vs CORS — giải thích sự nhầm lẫn phổ biến.

CSRF: trình duyệt tự gửi cookie → cần token/SameSite; ảnh hưởng app dùng session cookie. CORS: cơ chế trình duyệt cho JS đọc response cross-origin — KHÔNG phải chống CSRF. JWT header không bị CSRF (trình duyệt không tự gắn custom header). Nhầm hai cái → bật sai cơ chế, vừa thừa vừa hở.

🔧 Lab bắt buộc — khai thác & vá (2 bài)

Lab 1 — SQL Injection. Viết một endpoint search nối chuỗi input vào SQL. Khai thác bằng ' OR '1'='1 để dump bảng. Vá bằng parameterized query, thử lại → vô hại. Thêm case ORDER BY động → chứng minh vẫn injectable, vá bằng whitelist. Tiêu chí: giải thích tách lệnh/data.

Lab 2 — IDOR. Tạo endpoint GET /orders/{id} chỉ findById. Login user A, đổi id sang đơn của user B → xem được (lỗ hổng). Vá bằng findByIdAndCustomerId(currentUser), trả 404. Tiêu chí: phân biệt authentication vs authorization.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. Vì sao parameterized query chặn SQL injection? Trường hợp nào vẫn injectable?
  2. IDOR là gì? Vì sao "đã login" không ngăn được? Check gì ở server?
  3. CSRF vs CORS khác nhau ra sao? Vì sao JWT header không bị CSRF? SameSite làm gì?
  4. XSS chặn chủ yếu ở đâu — input hay output? 4 tầng phòng thủ là gì?
  5. Vì sao mọi check bảo mật phải ở server dù client đã validate?
