API Design — Vì Sao Quan Trọng?

API là contract giữa systems. Một khi public, rất khó thay đổi mà không break users. Khác với code internal có thể refactor tự do, API design mistake = bị "kẹt" với decision đó nhiều năm. Vì vậy senior phải:

  • Hiểu HTTP semantics đủ sâu để chọn đúng method (vì sao GET không được modify state?)
  • Anticipate edge cases (network retry, race condition, scale)
  • Plan cho evolution (versioning strategy ngay từ đầu, không phải afterthought)
  • Defend từ abuse (rate limiting, idempotency)
📜 Lịch sử ra đời của REST
1990s — Web ban đầu chỉ có HTML form + CGI. APIs giữa systems dùng SOAP/XML-RPC — phức tạp, verbose, dùng custom protocol over HTTP.

2000 — Roy Fielding viết PhD dissertation tại UC Irvine, mô tả REST (Representational State Transfer). Đặt câu hỏi: "Web đã scale lên hàng tỷ document — kiến trúc nào enable điều đó?"

Insight cốt lõi của Fielding: kiến trúc Web work tốt vì uniform interface (HTTP verbs có semantics rõ), stateless (server không nhớ context), cacheable (proxy/CDN tận dụng được).

2005-2010 — REST dần thay SOAP cho public APIs (Twitter, Facebook, AWS). Lý do: simpler, HTTP-native, browser-friendly.

2010s — REST thành de-facto standard. GraphQL (2015) và gRPC (2016) emerge như alternative cho specific use cases (frontend flexibility, internal RPC).

Bài học: REST không phải "the only way" — chọn vì philosophy: leverage HTTP semantics, decouple client/server, evolve over time. Hiểu philosophy quan trọng hơn memorize "RESTful rules".

1. Bài Toán Thực Tế — Music Streaming App "TuneFlow"

TuneFlow là app streaming nhạc (kiểu Spotify) đã launch 2 năm. 4 vấn đề thực gặp trong production:

Issue #1 — Mobile app crash khi load playlist lớn

Mobile gọi GET /api/playlists/{id}/songs. Playlist của user power "Workout Mix 2025" có 10,000 bài. API trả về JSON 8MB trong 12 giây. Mobile parse JSON OOM → crash.

Issue #2 — Double charge khi upgrade Premium

User upgrade Premium qua mobile. Network flaky → request reach server, charge thành công, response timeout. User không thấy gì → bấm "Upgrade" lần nữa. Server xử lý lần 2 → user bị charge 2 lần. Customer support ngập complaints.

Issue #3 — Update API break mobile cũ

Team add field currency vào response price. Mobile app v3.2 (cũ) hardcode parse JSON theo position, không theo name → crash. Hơn 30% user dùng version cũ → 30% userbase bị down 2 ngày trong khi đợi mobile release fix.

Issue #4 — 1 user lạm dụng → ảnh hưởng mọi user khác

Script kiddie phát hiện GET /api/search?q=X không có auth check, viết bot search 1000 req/sec. Server overload, mọi user khác slow. Rate limit có nhưng global (10 req/sec total), không per-user → 1 user starve toàn bộ.

2. REST + HTTP Method Semantics

REST = Representational State Transfer. 5 constraints chính:

  1. Client-Server: tách biệt. Server không biết client UI, client không biết server DB.
  2. Stateless: mỗi request chứa đủ info để xử lý. Server không lưu session state.
  3. Cacheable: response label được "cache" hay không. Tận dụng CDN/browser cache.
  4. Uniform Interface: resources identify qua URI, manipulate qua representations.
  5. Layered System: client không assume connect trực tiếp (qua proxy, LB).

HTTP Method Semantics — Bảng phải thuộc

MethodSafe?Idempotent?Cacheable?Use case
GET✅✅✅Đọc data, no side effect
POST❌❌⚠️ rarelyTạo resource, action có side effect
PUT❌✅❌Replace toàn bộ resource
PATCH❌❌ (technically)❌Update một phần resource
DELETE❌✅❌Remove resource
🧠 Misconception Check — Safe vs Idempotent
"Idempotent = không thay đổi state". Junior thường confuse safe và idempotent.
Safe = không có side effect (GET). Idempotent = gọi N lần kết quả giống nhau (cùng end state). DELETE idempotent nhưng KHÔNG safe — gọi DELETE lần đầu xóa, lần 2 vẫn "đã xóa" (cùng state) → idempotent. PUT thay đổi state nhưng vẫn idempotent — replace toàn bộ resource, gọi N lần cùng result.
🧠 Misconception — PATCH có idempotent không?
"PATCH idempotent vì update partial vẫn ra same result"
Technically KHÔNG idempotent trong general case. Ví dụ: PATCH /counter { "op": "increment" } — gọi N lần counter tăng N lần. Nếu PATCH dùng để set value ({"name": "John"}) thì idempotent. Spec HTTP không guarantee — depends on operation. Best practice: design PATCH operations idempotent (set-based, không increment-based).
🔬 Thử Phá Vỡ — POST cho mọi thứ thì sao?

"Tôi sẽ dùng POST cho mọi endpoint, đơn giản hơn." — Liệt kê 3 vấn đề sẽ gặp.

Click để xem 3 vấn đề (sau khi bạn đã nghĩ)
  1. Mất caching: GET cacheable, browser/CDN/proxy đều cache được. POST không cache → mọi request hit server → load tăng 10-100×.
  2. Mất idempotency tự nhiên: GET idempotent → retry safe. POST không → cần idempotency key thủ công cho mọi endpoint, code phức tạp hơn.
  3. Mất semantics: monitoring tool, API gateway, SDK đều dựa HTTP verbs. POST cho mọi thứ → log nhìn rối (POST /search? POST /list-users?), audit khó (POST có thể là read hoặc write?).

3. Idempotency Key — Tránh Duplicate Khi Retry

Vấn đề

Mạng flaky. Client POST /api/subscriptions/upgrade. Request reach server, charge thành công, response timeout. Client retry → server xử lý lần 2 → user bị charge 2 lần.

@RestController
@RequestMapping("/api/v1/subscriptions")
public class SubscriptionController {
    private final UpgradeSubscriptionUseCase upgradeUseCase;
    private final IdempotencyStore idempotencyStore;

    @PostMapping("/upgrade")
    public ResponseEntity<UpgradeResponse> upgrade(
            @RequestBody UpgradeRequest request,
            @RequestHeader("Idempotency-Key") String idempotencyKey) {

        // Check cache trước
        Optional<UpgradeResponse> cached = idempotencyStore.get(idempotencyKey);
        if (cached.isPresent()) {
            // Retry detected — return cached response
            return ResponseEntity.ok()
                .header("Idempotency-Replay", "true")  // Indicate this is a replay
                .body(cached.get());
        }

        // First-time request — process
        UpgradeResponse response = upgradeUseCase.execute(request);

        // Cache result với TTL 24h
        idempotencyStore.put(idempotencyKey, response, Duration.ofHours(24));

        return ResponseEntity.ok(response);
    }
}

@Service
public class RedisIdempotencyStore implements IdempotencyStore {
    private final RedisTemplate<String, String> redis;
    private final ObjectMapper json;

    @Override
    public Optional<UpgradeResponse> get(String key) {
        String value = redis.opsForValue().get("idem:" + key);
        if (value == null) return Optional.empty();
        try {
            return Optional.of(json.readValue(value, UpgradeResponse.class));
        } catch (Exception e) { return Optional.empty(); }
    }

    @Override
    public void put(String key, UpgradeResponse value, Duration ttl) {
        try {
            redis.opsForValue().set("idem:" + key, json.writeValueAsString(value), ttl);
        } catch (Exception e) { throw new RuntimeException(e); }
    }
}
🎯 Edge Cases — Production Reality

Code trên work cho happy path. Production gặp 7 edge cases sau, mỗi cái cần handle riêng:

  • Race condition giữa check và put: 2 retry arrive cùng lúc, cả 2 check cache không có → cả 2 process. Fix: dùng SETNX (set-if-not-exists) atomic, hoặc database unique constraint trên idempotency key.
  • Server crash giữa "process success" và "cache put": business logic commit nhưng cache chưa update. Retry → process lại. Fix: cache write trong cùng transaction với business write, hoặc dùng Saga pattern.
  • Idempotency-Key collision: 2 client gen cùng UUID (rất hiếm nhưng có thể với UUID v1 timestamp-based). Fix: dùng UUID v4 (random) hoặc combine với userId scope.
  • TTL expire trong retry window: client retry sau 25h (offline period dài), cache đã expire → process lại. Fix: TTL phải > max retry window expected (thường 48-72h).
  • Different request body, same Idempotency-Key: client gửi key X cho request A, sau đó gửi key X cho request B khác. Server return response của A → user thấy sai. Fix: cache phải verify request hash, mismatch → reject với 422 Unprocessable Entity.
  • Idempotency cho long-running operations: request 1 đang process 30s, retry arrive lúc 5s. Cache chưa có. Fix: ngay khi nhận request, put "IN_PROGRESS" marker. Retry thấy marker → return 409 Conflict hoặc 202 + polling URL.
  • GDPR — xóa user, idempotency cache vẫn còn data: user data đã xóa khỏi DB chính nhưng cache còn response. Fix: cache invalidation on user delete, hoặc encrypt cache với user-derived key (crypto-shredding).

4. Pagination Strategies

⏸ Pause & Predict
Trước khi đọc, tự design API cho: "Get 50 most recent posts của user X trên social platform, scroll infinite". Câu hỏi:
  1. URL endpoint và query params bạn sẽ chọn?
  2. Nếu trong lúc user scroll, có 10 posts mới được tạo, behavior nên thế nào?
  3. Nếu user muốn jump đến page 100 directly, có support không? Nếu không, vì sao?

3 Strategies — Offset, Cursor, Keyset

StrategyComplexityStable under insert?Jump to page N?Khi nào dùng
Offset (?offset=100&limit=50)O(offset + limit)❌✅Small dataset (<10K), admin UI
Cursor (?cursor=abc&limit=50)O(limit)✅❌Infinite scroll, feed, timeline

📜 Vì sao Cursor pagination phổ biến cho modern APIs?
Era 1 (1990s-2000s) — Web app dùng offset pagination vì SQL có LIMIT/OFFSET built-in. Đơn giản, ai cũng hiểu.

Era 2 (2010s, Twitter, Facebook) — Social feed thay đổi liên tục. Offset pagination tạo bug: user scroll page 1, thấy post A B C; trong lúc đó posts mới được tạo, push offset xuống; user load page 2 → thấy A B C lại (duplicate) hoặc miss posts (gap). Twitter implement cursor pagination dùng since_id và max_id để fix.

Era 3 (2015+) — Cursor pagination thành standard cho:
  • Public APIs (Facebook Graph, GitHub API, Stripe API)
  • Infinite scroll UIs (Instagram, TikTok, Twitter)
  • Event streams (Kafka offset is essentially cursor)

Insight: cursor pagination evolve không phải vì "performance" (mặc dù có) — chính vì stable iteration semantics. Người dùng không tolerate duplicate/missed items trong feed.

Implementation — Cursor Pagination cho Music Streaming Playlist

@RestController
@RequestMapping("/api/v1/playlists")
public class PlaylistController {
    private final PlaylistService playlistService;

    @GetMapping("/{playlistId}/songs")
    public ResponseEntity<PagedSongs> getSongs(
            @PathVariable String playlistId,
            @RequestParam(defaultValue = "50") int limit,
            @RequestParam(required = false) String cursor) {

        // Validate limit để tránh abuse
        if (limit < 1 || limit > 200) {
            throw new BadRequestException("limit must be between 1 and 200");
        }

        // Decode cursor (base64-encoded timestamp + ID)
        SongCursor cursorObj = cursor != null ? SongCursor.decode(cursor) : null;

        // Fetch limit + 1 để check hasMore
        List<Song> songs = playlistService.getSongsAfter(playlistId, cursorObj, limit + 1);

        boolean hasMore = songs.size() > limit;
        if (hasMore) songs = songs.subList(0, limit);

        // Build next cursor từ last item
        String nextCursor = hasMore
            ? SongCursor.encode(songs.get(songs.size() - 1))
            : null;

        return ResponseEntity.ok(new PagedSongs(
            songs.stream().map(SongDto::from).toList(),
            nextCursor,
            hasMore,
            new PaginationLinks(
                String.format("/api/v1/playlists/%s/songs?cursor=%s", playlistId, cursor),
                hasMore ? String.format("/api/v1/playlists/%s/songs?cursor=%s", playlistId, nextCursor) : null
            )
        ));
    }
}

// Cursor encodes timestamp + ID để uniquely identify position
public record SongCursor(Instant addedAt, String songId) {
    public static String encode(Song song) {
        String raw = song.addedAt().toString() + "|" + song.id();
        return Base64.getUrlEncoder().encodeToString(raw.getBytes());
    }

    public static SongCursor decode(String encoded) {
        String raw = new String(Base64.getUrlDecoder().decode(encoded));
        String[] parts = raw.split("\\|");
        return new SongCursor(Instant.parse(parts[0]), parts[1]);
    }
}

// Repository query với keyset (more efficient than OFFSET)
@Repository
public class SongRepository {
    public List<Song> findAfterCursor(String playlistId, SongCursor cursor, int limit) {
        String sql;
        if (cursor == null) {
            sql = "SELECT * FROM playlist_songs WHERE playlist_id = ? " +
                  "ORDER BY added_at DESC, song_id DESC LIMIT ?";
            return jdbc.query(sql, mapper, playlistId, limit);
        } else {
            // Keyset condition: tuple comparison
            sql = "SELECT * FROM playlist_songs WHERE playlist_id = ? " +
                  "AND (added_at, song_id) < (?, ?) " +
                  "ORDER BY added_at DESC, song_id DESC LIMIT ?";
            return jdbc.query(sql, mapper, playlistId, cursor.addedAt(), cursor.songId(), limit);
        }
    }
}
🎯 Vì sao cursor là (timestamp, id) chứ không chỉ timestamp?

Nếu 2 songs add cùng millisecond (rất phổ biến với batch add từ external playlist), cursor chỉ timestamp → ambiguous: WHERE timestamp < T loại bỏ cả 2 hoặc giữ cả 2. Tuple (timestamp, id) với secondary sort theo ID → strictly ordered, không bao giờ duplicate hay miss.

🔬 Thử Phá Vỡ — Cursor pagination với "Sort by popularity"

User muốn sort songs theo ?sort=popularity_desc (popularity là field play_count update realtime). Code trên có work không? Nếu không, vì sao?

Click để xem analysis

KHÔNG work. Cursor pagination assume sort field stable. Nhưng play_count tăng liên tục → cursor (play_count=1000, id=X) hôm nay có thể trỏ đến song khác ngày mai.

Hậu quả: scroll page 2 thấy song đã thấy ở page 1 (duplicate), hoặc miss songs giữa.

Fix options:

  1. Snapshot popularity at query time (cache popularity score trong session 5 phút) → stable trong session
  2. Pagination dùng (popularity_at_snapshot, id) thay vì current popularity
  3. Hoặc thay sort: "popular last 7 days" — pre-calculate daily, immutable trong day
  4. Accept some inconsistency — UX message "Results may shift as new data arrives"

Insight: cursor pagination chỉ work với append-only ordered fields (timestamp, ID, immutable counters). Mutable sort fields cần snapshot.

🎯 Edge Cases — Pagination Production Reality
  • Empty page với hasMore=true: filter loại bỏ hết items page này. Client tưởng đã hết, dừng scroll → miss. Fix: tiếp tục fetch internally cho đến khi đủ N items hoặc hết DB.
  • Cursor invalid sau schema change: thêm column mới, cursor encoding cũ break. Fix: version cursor (v1:base64data), reject old version với 400 + force client re-fetch.
  • User permission change giữa pages: user lose access đến playlist khi đang scroll. Page 1 fetched OK, page 2 → 403. Fix: re-validate permission mỗi page, return 403 nicely với guidance.
  • Cursor leak data position: cursor base64 timestamp → reverse engineer biết "song ID = 123 added at 2024-01-15". Fix: encrypt cursor (HMAC + secret), client không decode được.
  • Bot scrape toàn bộ dataset: cursor pagination cho phép walk through everything. Fix: limit max pages (sau N pages reject), hoặc rate limit hơn cho cursor requests.
  • Network retry duplicate cursor: client gọi page 2 với cursor C, timeout, retry. Idempotent vì GET — nhưng analytics đếm 2 hits. Fix: deduplicate ở analytics layer dùng request ID.

5. API Versioning

🧠 Misconception — "Add field mới là backward-compatible"
"Thêm field currency vào response không break client cũ vì client cũ chỉ ignore field mới"
Phụ thuộc client implementation. JSON parser strict mode (như Jackson FAIL_ON_UNKNOWN_PROPERTIES = true) sẽ throw exception khi gặp field unknown. Mobile app cũ dùng generated code (Protobuf, OpenAPI codegen) parse theo schema cố định cũng break. "Add field is backward-compatible" chỉ đúng nếu client tolerant — không phải luôn. Safe assumption: thay đổi response = breaking change unless proven otherwise.

4 Versioning Strategies

1. URI Path:        /api/v1/songs    →  /api/v2/songs       ← Explicit, dễ test
2. Query param:     /api/songs?v=1   →  /api/songs?v=2      ← Easy forget default
3. Header:          Accept-Version: 2                        ← Clean URL, hidden
4. Content nego:    Accept: application/vnd.tuneflow.v2+json ← RESTful, complex
🎯 Recommendation cho 90% case: URI Path Versioning

Lý do thực tế:

  • Visible: nhìn URL biết ngay version. Browser test dễ.
  • Logging/analytics: filter by version trong access log đơn giản (path-based).
  • Cache friendly: CDN cache theo URL — v1 và v2 cache độc lập tự nhiên.
  • Documentation: Swagger/OpenAPI generate per-path dễ hiểu hơn per-header.

Trade-off: code duplication. Nhưng đây chính là điểm — mỗi version có lifecycle riêng, deprecation timeline riêng, không nên share code.

Implementation — V1 và V2 song song

// V1 (deprecated, maintained 2 năm)
@RestController
@RequestMapping("/api/v1/songs")
@Deprecated  // Annotation cho IDE warn
public class SongControllerV1 {

        ResponseEntity.BodyBuilder builder = ResponseEntity.ok();
        // Sunset header — báo client v1 sẽ retire

// V2 (current) — thêm artist, album, price với currency
@RestController
@RequestMapping("/api/v2/songs")
public class SongControllerV2 {
    @GetMapping("/{id}")
    public ResponseEntity<SongResponseV2> getSong(@PathVariable String id) {
        SongDto song = songService.findById(id);
        return ResponseEntity.ok(new SongResponseV2(

            song.title(),
            song.durationSeconds(),
            new ArtistRef(song.artistId(), song.artistName()),

🎯 Edge Cases — Versioning Production
  • V1 vs V2 cùng update DB → race condition: V1 PUT song, V2 PUT song cùng lúc, 2 schemas khác → last write wins, fields của loser bị lost. Fix: optimistic locking với version field, hoặc converge schema (V1 endpoint internally update full schema với defaults).
  • V1 client gửi field tên cũ, V2 expect field mới: rename name → title. Nếu client cũ POST V2 endpoint với name field → silently ignored. Fix: V2 phải reject explicit unknown fields (Jackson FAIL_ON_UNKNOWN_PROPERTIES) hoặc transform request through anti-corruption layer.
  • Database supports cả 2 versions = schema bloat: thêm cột mới cho V2, V1 không hiểu. Nullable cột → invariant không enforce. Fix: tách microservice per version (extreme), hoặc accept temporary nullable + migration plan để retire V1.
  • Mobile app stuck ở V1: enterprise apps không update được (approval cycle 6+ tháng). V1 retire sẽ break critical users. Fix: support window ≥ 2 năm từ V2 launch, multiple deprecation warnings (header, email, dashboard).
  • Caching giữa V1 và V2 confusion: CDN cache /api/v1/songs/123 và /api/v2/songs/123 độc lập (good), nhưng nếu shared underlying DB query → invalidation cần care. Fix: cache key bao gồm version.
  • Documentation drift: V1 docs outdated vì team focus V2. Confusion cho V1 maintenance users. Fix: auto-generate docs từ code (Swagger/OpenAPI), mỗi PR check docs build.

6. Rate Limiting

3 Algorithms phổ biến

Algorithm 1

Token Bucket

Bucket có N tokens, refill X tokens/sec. Mỗi request consume 1 token. Hết tokens → reject. Allow bursts đến bucket size. Phổ biến nhất (AWS, GCP).

Algorithm 2

Fixed Window

Đếm requests trong 1 window cố định (00:00-01:00, 01:00-02:00). Đơn giản nhưng có boundary spike: 60 req ở 00:59 + 60 req ở 01:00 = 120 req trong 1 phút.

Algorithm 3

Sliding Window

Đếm requests trong rolling window N giây trước hiện tại. Chính xác nhất, không có boundary effect. Implement phức tạp (sorted set Redis).

Bonus

Leaky Bucket

Tương tự token bucket nhưng output rate constant — smooth traffic. Dùng cho bandwidth shaping, không phổ biến cho HTTP rate limit.

📜 Vì sao Sliding Window phổ biến hơn cho modern APIs?
Early 2000s — Apache mod_evasive, simple counter per IP. Fixed window. Boundary spike phổ biến nhưng tolerable.

2010s — DoS attackers học spike boundary trick → fixed window không đủ. Cloudflare, AWS WAF chuyển sang sliding window.

Redis as kingmaker — Sliding window complex để implement (cần ordered set). Redis ZADD/ZREMRANGEBYSCORE đủ nhanh để làm trên scale lớn → sliding window thành affordable.

Modern preference — Sliding window log (chính xác, tốn memory) cho high-value endpoints; token bucket cho general traffic.

Practical insight: chọn algorithm dựa trên (1) traffic pattern (bursty hay smooth?), (2) tolerance cho boundary effect, (3) Redis memory budget. Không có "best", có "best for your case".

Implementation — Token Bucket với Redis

@Component
public class TokenBucketRateLimiter {
    private final StringRedisTemplate redis;
    private final DefaultRedisScript<Long> refillScript;

    public TokenBucketRateLimiter(StringRedisTemplate redis) {
        this.redis = redis;
        // Lua script atomic: check + decrement + refill
        this.refillScript = new DefaultRedisScript<>(LUA_TOKEN_BUCKET, Long.class);
    }

    /**
     * @return true nếu allow, false nếu rate-limited
     */
    public boolean tryAcquire(String userId, int capacity, int refillPerSec) {
        String key = "rl:user:" + userId;
        Long result = redis.execute(refillScript,
            List.of(key),
            String.valueOf(capacity),
            String.valueOf(refillPerSec),
            String.valueOf(System.currentTimeMillis())
        );
        return result != null && result == 1;
    }

    private static final String LUA_TOKEN_BUCKET = """
        local key = KEYS[1]
        local capacity = tonumber(ARGV[1])
        local refillPerSec = tonumber(ARGV[2])
        local now = tonumber(ARGV[3])

        local bucket = redis.call('HMGET', key, 'tokens', 'lastRefill')
        local tokens = tonumber(bucket[1]) or capacity
        local lastRefill = tonumber(bucket[2]) or now

        -- Refill tokens based on elapsed time
        local elapsed = (now - lastRefill) / 1000
        local newTokens = math.min(capacity, tokens + elapsed * refillPerSec)

        if newTokens < 1 then
            redis.call('HMSET', key, 'tokens', newTokens, 'lastRefill', now)
            redis.call('EXPIRE', key, 3600)
            return 0  -- Reject
        end

        redis.call('HMSET', key, 'tokens', newTokens - 1, 'lastRefill', now)
        redis.call('EXPIRE', key, 3600)
        return 1  -- Allow
        """;
}

@Component
public class RateLimitFilter extends OncePerRequestFilter {
    private final TokenBucketRateLimiter limiter;

    @Override
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res, FilterChain chain)
            throws ServletException, IOException {
        String userId = extractUserIdFromJwt(req);
        if (userId == null) {
            // Anonymous — limit by IP
            userId = "ip:" + req.getRemoteAddr();
        }

        // 100 requests / second sustained, burst 200
        boolean allowed = limiter.tryAcquire(userId, 200, 100);
        if (!allowed) {
            res.setStatus(429);
            res.setHeader("Retry-After", "1");
            res.setHeader("X-RateLimit-Limit", "100");
            res.setHeader("X-RateLimit-Remaining", "0");
            res.getWriter().write("{\"error\":\"RATE_LIMIT_EXCEEDED\",\"retryAfter\":1}");
            return;
        }
        chain.doFilter(req, res);
    }
}
🎯 Vì sao Lua script atomic critical?

Nếu implement bằng nhiều Redis commands (GET → compute → SET), 2 threads cùng request:

  1. Thread A: GET tokens = 1
  2. Thread B: GET tokens = 1 (cùng lúc, chưa decrement)
  3. Thread A: SET tokens = 0, allow
  4. Thread B: SET tokens = 0, allow

→ Cả 2 đều pass dù chỉ có 1 token. Lua script run atomic trong Redis single-thread → race condition không thể xảy ra.

🔬 Thử Phá Vỡ — Rate limit theo IP với CGNAT

"Tôi rate limit per IP để bảo vệ API." Hãy nghĩ tình huống IP-based rate limit fail.

Click để xem
  1. CGNAT (Carrier-Grade NAT): hàng nghìn user mobile cùng public IP. Limit per-IP = limit per ngàn người. 1 attacker dùng hết quota → tất cả user của carrier đó bị block.
  2. Corporate/School network: 1 office 500 nhân viên, 1 public IP. Limit thấp = mọi người block nhau.
  3. Mobile NAT chuyển IP: user di chuyển → IP đổi (4G → wifi → 4G). Rate limit reset → user bypass.
  4. VPN/Tor: user dễ đổi IP để bypass.
  5. IPv6: mỗi device unique IPv6 → IP-based vô nghĩa với users IPv6.

Fix: layer multiple identifiers — per user (JWT), per API key, per IP (fallback for anonymous). Whitelist trusted IPs (office, partners). Adaptive limit (lower cho new accounts, higher cho verified).

🎯 Edge Cases — Rate Limiting Production
  • Distributed rate limit consistency: 3 servers behind LB, mỗi server rate limit local → user gửi 100 req qua 3 servers vẫn pass dù limit là 100. Fix: central store (Redis), atomic ops.
  • Redis down → fail open hay fail closed? Fail open = không có rate limit, attack passes. Fail closed = mọi user block. Production thường: fail open with alerting (better UX, monitoring catches issue).
  • Clock skew giữa servers: sliding window dùng System.currentTimeMillis(). Server clocks lệch nhau vài giây → inconsistent windows. Fix: dùng Redis server time (TIME command) làm source of truth.
  • Rate limit headers gây cache hit/miss inconsistent: response chứa X-RateLimit-Remaining đổi mỗi request. CDN cache response → wrong header serve cho user khác. Fix: Cache-Control: private cho rate-limited endpoints.
  • Burst capacity bị abuse: bucket size 200, refill 100/s → attacker burst 200 lúc 0s, đợi 1s, burst 200 lần nữa = 400 req trong 1s. Fix: nếu cần strict, dùng sliding window thay token bucket.
  • Differential limit cho operations khác nhau: GET song cheap, POST upload song expensive. Single limit unfair. Fix: weighted tokens (GET cost 1, POST cost 10).

7. HATEOAS — Hypermedia As The Engine Of Application State

⏸ Pause & Predict
Nhìn 2 response JSON dưới, predict ưu/nhược điểm:
// A. Plain REST
{ "id": "song-123", "title": "Yesterday", "albumId": "album-456" }

// B. HATEOAS
{
  "id": "song-123", "title": "Yesterday",
  "_links": {
    "self":   { "href": "/songs/song-123" },
    "album":  { "href": "/albums/album-456" },
    "lyrics": { "href": "/songs/song-123/lyrics" },
    "play":   { "href": "/play", "method": "POST" }
  }
}
Câu hỏi:
  1. Khi backend đổi URL pattern (ví dụ /albums/X thành /library/albums/X), client nào break?
  2. Khi user state thay đổi (chưa Premium thì play link absent), client UI thế nào?
  3. Response size khác nhau ra sao? Mobile data plan tốn không?
🧠 Misconception — HATEOAS phổ biến trong REST APIs
"REST = HATEOAS. Mọi REST API nên có _links field."
Roy Fielding (cha đẻ REST) khẳng định HATEOAS là required để thật sự "RESTful". Nhưng thực tế < 5% public APIs implement HATEOAS (Twitter, Facebook, GitHub, Stripe đều không). Lý do: complexity vs benefit. Client mobile hardcode URLs work tốt hơn parse _links runtime. HATEOAS shine với public APIs có evolution rất nhanh hoặc generic clients (browser tự navigate). Cho 90% case, REST level 2 (verbs + resources, no HATEOAS) đủ.
📜 Richardson Maturity Model — Đo độ "RESTful"
Leonard Richardson (2008) propose 4 level maturity cho REST APIs:

Level 0 — Swamp of POX: 1 URL, POST tất cả. SOAP-style. (Không RESTful)
Level 1 — Resources: nhiều URLs cho resources. Nhưng vẫn POST mọi thứ.
Level 2 — HTTP Verbs: GET/POST/PUT/DELETE đúng semantics. 90% modern APIs ở đây.
Level 3 — HATEOAS: response chứa links để navigate. Đây mới là "true REST" theo Fielding.

Insight: Industry chọn Level 2 vì cost/benefit. Level 3 cost cao (server impl + client parse logic), benefit nhỏ với typed clients (mobile, SDK). Hiểu mature model giúp bạn argue về architecture decisions, không phải để áp dụng cứng nhắc.

8. Error Response Format — Consistent Across API

Error format inconsistent → client phải parse mỗi endpoint differently → bug + bad UX. Standard hóa:

{
  "error": {
    "code": "INSUFFICIENT_FUNDS",
    "message": "Cannot complete upgrade: payment method rejected",
    "details": {
      "paymentMethod": "card_visa_1234",
      "declineReason": "insufficient_funds"
    },
    "timestamp": "2026-05-20T14:32:15.123Z",
    "path": "/api/v1/subscriptions/upgrade",
    "traceId": "abc-123-def",
    "documentation": "https://docs.tuneflow.com/errors/insufficient_funds"
  }
}
🎯 Mỗi field có purpose riêng — không phải decoration
  • code: machine-readable, client switch/case xử lý. KHÔNG dùng message để switch — i18n sẽ break.
  • message: human-readable cho support/log. KHÔNG show thẳng cho end-user (có thể leak internal info).
  • details: contextual data để debug. Validation errors có field-level breakdown.
  • timestamp: cho support correlate với log timeline.
  • path: endpoint gây error (dynamic routing có thể che mất).
  • traceId: critical cho distributed tracing — copy vào support ticket.
  • documentation: link self-service để user/dev tự fix.

Implementation với Spring @ControllerAdvice:

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(InsufficientFundsException.class)
    public ResponseEntity<ApiErrorResponse> handle(InsufficientFundsException e, HttpServletRequest req) {
        return ResponseEntity.badRequest().body(ApiErrorResponse.builder()
            .code("INSUFFICIENT_FUNDS")
            .message("Cannot complete upgrade: payment method rejected")
            .detail("paymentMethod", e.getPaymentMethodId())
            .detail("declineReason", e.getDeclineReason())
            .path(req.getRequestURI())
            .traceId(MDC.get("traceId"))
            .documentation("https://docs.tuneflow.com/errors/insufficient_funds")
            .build());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)  // Validation
    public ResponseEntity<ApiErrorResponse> handleValidation(MethodArgumentNotValidException e, HttpServletRequest req) {
        Map<String, String> fieldErrors = e.getBindingResult().getFieldErrors().stream()
            .collect(Collectors.toMap(FieldError::getField, FieldError::getDefaultMessage));
        return ResponseEntity.badRequest().body(ApiErrorResponse.builder()
            .code("VALIDATION_FAILED")
            .message("Request validation failed")
            .detail("fields", fieldErrors)
            .path(req.getRequestURI())
            .traceId(MDC.get("traceId"))
            .build());
    }

    @ExceptionHandler(Exception.class)  // Fallback
    public ResponseEntity<ApiErrorResponse> handleUnexpected(Exception e, HttpServletRequest req) {
        log.error("Unexpected error", e);
        return ResponseEntity.internalServerError().body(ApiErrorResponse.builder()
            .code("INTERNAL_ERROR")
            .message("An unexpected error occurred. Please retry or contact support.")
            // ⚠️ KHÔNG leak stack trace ra response (security risk)
            .path(req.getRequestURI())
            .traceId(MDC.get("traceId"))
            .build());
    }
}

8b. Symptom → Debug Recipes CORE ★

Lỗi API design ít crash app — chúng manifest dưới dạng mobile app cũ break, customer charge trùng, hoặc 1 user bot phá hệ thống cho mọi người. Đây là cách trace ngược.

🚨 Recipe #1 — Mobile app phiên bản cũ đột nhiên crash sau deploy backend

Symptom: Deploy version mới của API endpoint /v1/playlists lúc 10h sáng. 30 phút sau, Sentry báo mobile app v3.2 crash rate 40% (nhưng app v3.5 ổn). Customer phàn nàn.

Log/Metric cần xem:

  • Access log: filter User-Agent chứa "TuneFlow/3.2" và 5xx response
  • Sentry crash: kiểm stack trace — thường là NullPointerException trên field cũ bị rename, hoặc JsonParseException trên format mới
  • API gateway: diff response payload trước/sau deploy bằng tcpdump hoặc proxy log

3 nguyên nhân thường gặp:

  1. Field rename trong response — đổi track_count → trackCount. Mobile cũ deserialize fail. Fix: giữ cả 2 field 3 tháng, deprecation warning, monitor adoption
  2. Field bắt buộc thành optional / ngược lại — thêm field bắt buộc mới, mobile cũ gửi request không có field này → 400 Bad Request. Fix: chỉ thêm optional field, không thêm required trong cùng version
  3. Enum value mới — thêm "status": "PROCESSING" mới. Mobile cũ chỉ handle "ACTIVE/INACTIVE", crash khi switch. Fix: design enum mở từ đầu, default case

Pattern phòng ngừa:

// Backward-compatible response
public class PlaylistResponse {
    private int trackCount;       // new name
    private int track_count;       // deprecated, kept for v3.2 clients

    // Set both in serialization
    public int getTrack_count() { return this.trackCount; }
}

Anti-fix: KHÔNG force user update — luôn có user không update app trong 6 tháng. Plan compatibility window từ design phase.

🚨 Recipe #2 — Customer "tôi bị charge 2 lần cho cùng đơn hàng"

Symptom: 5-10 ticket/tuần customer charge trùng. Stripe webhook log show 2 payment_intent cho cùng user trong 10s. Reproduce bằng cách click rapid 2 lần trên web hoặc retry trên mobile khi mạng yếu.

Log/Metric cần xem:

  • Nginx/load balancer log: tìm 2 request POST /upgrade cùng user_id trong < 5s
  • Idempotency key table: kiểm có entry cho cả 2 request không, hay request 2 không gửi key
  • Mobile SDK telemetry: retry strategy có persist Idempotency-Key giữa các retry không

Root cause tree:

  1. Mobile không gửi Idempotency-Key — backend tạo key mới mỗi request → 2 request = 2 transaction. Fix: bắt buộc header này ở API gateway, reject nếu thiếu
  2. Mobile gửi key khác nhau mỗi retry — sai design: key phải gen 1 lần per intent, không phải per request. Fix: SDK persist key đến khi nhận response 2xx hoặc 4xx terminal
  3. Backend lưu key nhưng TTL ngắn — TTL 5 phút, retry sau 6 phút → key đã expire → treat as new. Fix: TTL ≥ 24h cho payment
  4. Race condition tại backend — 2 request đến đồng thời, cùng check "key chưa tồn tại" → cùng INSERT. Fix: INSERT ... ON CONFLICT DO NOTHING RETURNING, hoặc unique constraint + catch DuplicateKey

Debug command:

-- Tìm duplicate charge
SELECT user_id, COUNT(*), array_agg(idempotency_key)
FROM payments
WHERE created_at > NOW() - INTERVAL '1 day'
  AND status = 'CHARGED'
GROUP BY user_id, amount, ROUND(EXTRACT(EPOCH FROM created_at) / 60)
HAVING COUNT(*) > 1;
🚨 Recipe #3 — 1 user "phá" rate limit cho mọi user khác

Symptom: Endpoint /api/search đột nhiên trả 429 Too Many Requests cho mọi user dù QPS chung chỉ 100/s. Reproduce: 1 bot account spam 5000 req/s.

Log/Metric cần xem:

  • Rate limiter metric: phân biệt theo dimension (per-user, per-IP, per-API-key, global)
  • Top consumer: SELECT user_id, COUNT(*) FROM access_log WHERE created_at > NOW() - INTERVAL '5 min' GROUP BY user_id ORDER BY 2 DESC LIMIT 10
  • Redis token bucket key: KEYS "ratelimit:*" (chỉ scan, không KEYS production), xem có key nào TTL gần hết và count cao

Root cause: Rate limit chỉ apply global (toàn endpoint), không per-user. 1 user spam → tổng QPS vượt limit → mọi user khác bị 429.

Pattern đúng (multi-tier limit):

@Component
public class RateLimitFilter {
    public boolean allow(Request req) {
        // Tier 1: per-user limit (chống abuse)
        if (!check("user:" + req.userId, 100, Duration.ofMinutes(1))) {
            return false; // reject riêng user này
        }
        // Tier 2: per-endpoint global (chống ddos)
        if (!check("global:" + req.endpoint, 10000, Duration.ofMinutes(1))) {
            return false; // reject all
        }
        // Tier 3: per-IP (chống bypass user limit)
        if (!check("ip:" + req.ip, 500, Duration.ofMinutes(1))) {
            return false;
        }
        return true;
    }
}

Symptom của thiết kế đúng: Khi 1 user spam, chỉ user đó nhận 429. Global metric không spike. Other users unaffected.

🎯 3 cấp độ Mastery cho API Design

L1 — Recognize (Junior)

Biết REST verbs (GET/POST/PUT/DELETE), biết "đừng dùng GET cho mutate". Tạo endpoint trả về tất cả data, không pagination. Versioning nghĩ "thêm /v2/ là xong".

Symptom của L1: endpoint /getUserOrders POST trả 500 records.

L2 — Explain (Mid)

Phân biệt cursor vs offset pagination, hiểu idempotency key tại sao cần TTL. Biết 4 cách versioning trade-off. Implement rate limiting per-endpoint.

Symptom của L2: thiết kế API có pagination, error format consistent, idempotency cho mutation.

L3 — Apply (Senior)

Design backward-compatible từ ngày 1 (additive change only). Multi-tier rate limit (user + endpoint + IP). Sunset deprecated endpoint có plan 6-12 tháng. Recipe 3 ở trên là phản xạ.

Symptom của L3: PR API change có migration guide cho mobile + deprecation timeline + capacity plan.

9. Anti-Patterns

#1 GET cho operations modify state

// ❌
GET /api/users/123/delete
GET /api/cart/add?productId=456

// Vấn đề: browser prefetch link → tự xóa user; bot crawl → tự add cart

#2 No pagination (return all)

@GetMapping("/posts")
public List<Post> all() { return postRepo.findAll(); }
// 1M posts → 100MB JSON → mobile OOM, server slow

#3 Expose internal IDs (sequential)

GET /api/orders/1001  → exists
GET /api/orders/1002  → exists  (attacker enumerates)
GET /api/orders/1003  → exists  (data leak)

// Fix: UUID/hashids, không sequential

#4 Inconsistent error format

Endpoint A: { "error": "..." }
Endpoint B: { "errorCode": "...", "message": "..." }
Endpoint C: { "errors": ["..."] }
Endpoint D: returns 500 with HTML page

// Client phải write 4 parsing logic. Pain.

#5 Breaking changes trong minor version

v2.0: { "balance": 100.5 }      // number
v2.1: { "balance": "100.50" }   // string — minor bump, breaking!

// Mobile parseDouble crash → user app freeze

#6 No rate limiting

1 bad actor đủ để DoS toàn bộ service. Even legitimate misbehaved client (infinite loop) cũng có thể.

#7 Verb in URL

// ❌
POST /api/users/createUser
POST /api/orders/cancelOrder/123

// ✅ Resource + verb separation
POST /api/users               (verb in HTTP method)
DELETE /api/orders/123        (verb = DELETE)

#8 No documentation versioning

API v2 release, docs vẫn show v1. Client confused. Fix: docs deploy cùng API release, hosted per-version.

11. Interview Framework

Tầng 1 — SurfaceMid gate
Q: Giải thích cursor pagination. Vì sao không dùng offset?
"Offset scan rows từ đầu đến offset+limit (O(N) cho page xa). Cursor dùng (sort_field, id) làm position, query WHERE (sort_field, id) < cursor LIMIT N — O(limit) với index. Cursor cũng stable: nếu insert/delete xảy ra giữa pages, không shift offset → no duplicate/miss. Trade-off: client không jump-to-page-N, chỉ next/prev. Phù hợp infinite scroll, feed, không phù hợp paginated table có 'Go to page 50' button."
Q: Idempotency key là gì, vì sao cần cho POST?
"Client generate UUID gửi qua header Idempotency-Key. Server cache response by key (Redis, TTL 24h). Lần đầu: process + cache. Retry (cùng key): return cached response thay vì process lại. Vì sao cần: POST không idempotent natively. Network timeout → client retry → server process lần 2 (double charge). Idempotency key cho phép client retry safely. Spec: header chuẩn IETF (đang draft), Stripe/Shopify đã implement."
Tầng 2 — Deep DiveMid → Senior
Q: Design pagination cho social feed (Instagram-like) — 10K followers, mỗi follower có 100 posts.
"Architecture:
1. Storage: posts table với index (author_id, created_at DESC, post_id DESC).
2. Feed generation: 2 approaches:
Pull model: query WHERE author_id IN (followed_ids) ORDER BY created_at DESC LIMIT 50. Slow với 10K follows.
Push (fanout-on-write): khi follower post, write copy vào feed table của tất cả followers. Fast read, expensive write.
Hybrid: celebrities (millions of followers) pull; regular pull. Twitter dùng approach này.

3. Pagination: cursor (created_at, post_id) tuple. Stable, O(limit) — critical cho infinite scroll UX không bị duplicate.

4. Response: chunk 50 items, include next_cursor + hasMore. Pre-fetch next page khi user scroll 80%.

5. Caching: Redis cache top 100 items mỗi user 5 phút. CDN cache page 1 (most accessed).

6. Edge cases: new post arrives during scroll (handle with 'X new posts' indicator, không insert silently), post deleted (return null entry hoặc skip), permission revoked (filter out)."
Q: Rate limit per user 100 req/min. 1 user gửi 500 → handle thế nào? Tránh starvation user khác?
"Token bucket per user: mỗi user có bucket riêng, 100 tokens, refill 100/min. User A consume hết → reject A. User B bucket independent → không bị ảnh hưởng.

Implementation: Redis HMSET cho mỗi userId (tokens, lastRefill). Lua script atomic check + decrement. Key có TTL (cleanup inactive users).

Response cho rate-limited request: HTTP 429 + headers Retry-After: 60, X-RateLimit-Reset: <timestamp>. Client backoff exponential.

Fairness across endpoints: weighted tokens — search cost 10, getProfile cost 1. Prevent 1 expensive endpoint drain all tokens.

Anti-abuse: nếu user hit limit liên tục (cumulative), escalate: lower limit, require captcha, temporary ban. Detect pattern với analytics, không hardcode.

Graceful degradation: limit hit không return 500. Return 429 với clear message + retry guidance + documentation link."
Tầng 3 — ArchitectureSenior
Q: TuneFlow đã release v1 cho 50,000 mobile apps. Cần add field preview_url cho song response để mobile có thể preview 30s. Plan rollout không break old apps.
"Strategy: URI versioning + parallel deployment + sunset timeline.

Phase 1 — Build (week 1-2):
Create /api/v2/songs/{id} endpoint với schema mới (include preview_url). v1 endpoint giữ nguyên, không touch.
Internal services backfill preview_url cho mọi songs (async job, vài giờ).

Phase 2 — Test (week 3):
Deploy v2 to staging, mobile QA test build với v2.
Load test v2 — verify performance không degrade.
Internal alpha với engineering team.

Phase 3 — Rollout (week 4):
Mobile app release v2 update (call /api/v2/) — push qua app store.
v1 endpoint thêm headers: Sunset: Sat, 31 Dec 2026 23:59:59 GMT, Deprecation: true, Link: </api/v2/songs/X>; rel='successor-version'.
Email + in-app notification cho users dùng mobile cũ: 'Update app để tính năng X'.

Phase 4 — Monitor (week 5-52):
Dashboard: % traffic v1 vs v2 theo ngày.
Khi v1 traffic < 1% → engineer notify last few users → set sunset date 30 ngày sau.
At sunset: v1 trả 410 Gone với clear message.

Risk mitigation:
Enterprise users (B2B partners): outreach personally, extended deprecation window.
Have rollback plan: nếu v2 có bug critical, route v2 traffic về v1-compatible handler tạm thời.

Cost analysis: maintaining 2 versions ~6 tháng tốn engineering effort, nhưng breaking 50K users tốn nhiều hơn (support cost, churn). Cost reasonable."
