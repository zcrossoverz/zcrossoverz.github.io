FAANG System Design HLD — 4S framework + 5 designs walkthrough

File này KHÁC #37: #37 là framework chung, file này là FAANG-specific bar với 4S framework + 5 design kinh điển walkthrough đầy đủ. Đa số candidate VN đọc Alex Xu Vol 1 cảm thấy "hiểu" — vào mock thì lộ ngay không estimate được capacity, không justify được trade-off, không phân biệt được L4 và L5 expectation. File này dạy cả 3 + Pause/Predict để self-check.

⚠️ Cảnh báo "hiểu giả" — System Design dễ false confidence nhất

2. S1 — Scenario (5 phút) CORE

Quy tắc: 5 phút clarify TRƯỚC khi vẽ gì. 3 trục: FR (features), NFR (scale/latency), Capacity (DAU/QPS/storage). Capacity numbers PHẢI viết ra giấy, không tính nhẩm.

3 trục clarifying

① Functional Requirements (FR)

- Core features bắt buộc (vs nice-to-have)?
- User flow (đi từ đâu đến đâu)?
- Read-heavy or write-heavy?
- Real-time or eventual?
- Mobile + web hay chỉ 1?

② Non-Functional Requirements (NFR)

- Number of users? (DAU)
- Latency target? (P99 < 100ms? 1s?)
- Availability? (99.9% = 8.76h downtime/year, 99.99% = 52 min)
- Consistency model? (strong / eventual / session)
- Geo distribution? (single region / global)

③ Capacity estimation

DAU × actions/day = total requests/day
÷ 86,400 sec = average QPS
× peak factor (3-5×) = peak QPS

Storage: records/day × bytes/record × retention years
       × replication factor (3) + backup

Capacity cheatsheet — memorize

Time:
  1 day        = 86,400 sec   ≈ 100K
  1 month      = 2.6M sec
  1 year       = 31.5M sec

Data sizes (memorize):
  ASCII char   = 1 byte
  UTF-8 char   = 1-4 bytes (avg 2)
  int          = 4 bytes
  long/double  = 8 bytes
  UUID         = 16 bytes (binary), 36 bytes (string)
  Timestamp    = 8 bytes

Common content:
  Tweet         ~280 chars × 2 = 560 bytes
  Email         ~50 chars × 2  = 100 bytes
  Web page HTML ~100 KB
  Image (JPEG)  ~500 KB - 5 MB
  Video minute  ~10 MB

Network & latency:
  1 Gbps      = 125 MB/s
  Round-trip US-EU = 100 ms
  Disk read   = 10 ms (HDD), 0.1 ms (SSD), 0.05 ms (NVMe)
  RAM read    = 100 ns
  L1 cache    = 1 ns

4. S3 — Storage (10 phút) CORE

3 câu hỏi phải trả lời:

  1. DB choice? SQL vs NoSQL vs Graph vs Time-series
  2. Schema? Tables + columns + relationships
  3. Sharding strategy? By userId? geographic? hash?

Decision matrix — DB choice

NeedDB ChoiceVì sao
ACID transactions, complex queriesPostgreSQL, MySQLMature, JOIN, isolation
Massive scale, eventual consistencyCassandra, DynamoDBHorizontal scale, write-optimized
Document/JSONMongoDB, DocumentDBSchema flexibility
Graph relationshipsNeo4jMulti-hop queries
Time-series metricsInfluxDB, TimescaleDBCompression, downsampling
Full-text searchElasticsearchInverted index, ranking
CacheRedis, MemcachedIn-memory, low latency
Object/BlobS3, GCSCheap, durable, scale-out
Counter / leaderboardRedis Sorted SetO(log n) range queries

Sharding strategies

Hash sharding:
  shard = hash(key) % N
  + Uniform distribution
  - Range query phải scatter-gather
  - Resharding khó (rehash toàn bộ)

Range sharding:
  shard theo key range (userId 0-1M shard 1, 1M-2M shard 2)
  + Range query dễ
  - Hot spot nếu key skewed

Consistent hashing:
  shard = hash(key) → ring position → nearest node
  + Resharding chỉ move 1/N data
  + Phổ biến trong Cassandra, Redis Cluster

Geo sharding:
  shard theo geographic region
  + Low latency cho user
  - Cross-region query expensive

5. S4 — Scale (15 phút) HIGH-SIGNAL

Đây là trục phân biệt L4 và L5. L4: identify 2-3 bottleneck. L5: multiple trade-off level + alternatives. Articulate trade-off = senior signal.

Bottleneck checklist

1. Read amplification?       → Add cache (Redis), CDN
2. Write amplification?      → Async (queue), batch, sharding
3. Hot key (skewed traffic)? → Consistent hashing, request coalescing
4. Single point of failure?  → Replicate, multi-AZ
5. Database overload?        → Read replicas, sharding
6. Network bandwidth?        → Compression, edge serving (CDN)
7. Latency?                   → Geo-distributed, edge cache

Trade-offs to articulate

- Strong vs Eventual consistency
- Read vs Write throughput (master-master vs master-slave)
- Latency vs Cost (cache TTL, replication factor)
- Storage cost vs Compute cost
- CAP theorem: choose 2 of 3 (network partition luôn xảy ra → choose CP or AP)
- PACELC: in case of Partition, CP/AP; Else, Latency vs Consistency

📌 Đáp án Pause & Predict — Instagram capacity

Câu hỏi từ S1: Instagram 500M DAU, 2 photo/tuần upload, 20 feed reads/day. 
Writes (upload):
  500M × 2/tuần / 7 ngày = 143M uploads/day
  ÷ 86,400 sec = 1,650/sec average
  Peak ×3 = 5,000 uploads/sec

Reads (feed):
  500M × 20 / 86,400 = 116K reads/sec average
  Peak ×3 = 348K reads/sec (cần CDN heavy)

Storage 5 năm photos:
  143M × 500KB × 365 × 5 = 130 PB
  → Cần tier storage: hot (S3 Standard) + cold (S3 Glacier)

Bandwidth read peak:
  348K reads × 100KB = 35 GB/sec
  → CDN bắt buộc

Cache hot photos:
  Pareto 10% × 143M × 365 × 5 = 26B photos
  Cache 1% top hot: 260M × 500KB = 130 TB
  → Multi-tier cache cluster Redis
±50% OK. Off 1 magnitude = practice thêm.

D1. URL Shortener — Walkthrough đầy đủ CORE

S1: Scenario

FR:

  • POST /shorten?longURL=... → returns shortURL (e.g., bit.ly/abc123)
  • GET /abc123 → redirect to longURL
  • Optional: custom alias, expiration, click analytics

NFR:

  • 100M URLs created/day → 100M / 86,400 ≈ 1,150 writes/sec
  • Read:Write = 100:1 → 115K reads/sec
  • Latency: P99 < 100ms (redirect critical)
  • Availability: 99.9%

Capacity:

100M URLs/day × 365 × 5 years = 180B URLs total
Each record: 7-byte short + 200-byte long + 8-byte timestamp ≈ 250 bytes
Total storage: 180B × 250 = 45 TB (manageable, no need to delete)

S2: Service

                        ┌── ShortCode Generator (ZooKeeper or counter)
Client → CDN → LB → URL Service ──── Redis (hot URLs, LRU)
                        └── DB (Cassandra/MySQL)
                              ↓
                          Click events → Kafka → BigQuery (analytics)

APIs:

POST /api/shorten
  Request:  { longUrl: "https://...", customAlias?: string }
  Response: { shortUrl: "https://bit.ly/abc123", expiry: timestamp }

GET /:shortCode
  Response: 301 redirect to longUrl
            (async publish click event to Kafka)

S3: Storage

Table: urls (Cassandra)
  short_code (PK):   string (7 chars, base62)
  long_url:          string
  user_id:           string
  created_at:        timestamp
  expiry:            timestamp?

Click counts (separate table or Redis counter):
  short_code → counter (Redis INCR)
  Async flush daily to BigQuery for analytics

Index: short_code PK = O(1) lookup by short

Short code generation — 3 approach

Option 1: Hash longURL

MD5(longURL)[0:7]
+ Deterministic
- Collisions possible
- Custom alias conflict
- Same URL → same short
  (đôi khi không muốn)

Option 2: Counter + base62

id = ZooKeeper.next()
shortCode = base62(id)
7 chars = 62^7 = 3.5T unique
+ No collision
+ Sequential
- Predictable (security?)

Option 3 (hybrid): Counter + random salt. Đảm bảo unique + unpredictable.

S4: Scale

Bottleneck 1: Read 115K req/s on DB
  → Redis cache (LRU, TTL 24h)
    Cache hit rate ~95% → DB load = 6K reads/s, manageable
    Cache size: top 10M URLs × 250 bytes = 2.5 GB (1 Redis instance đủ)

Bottleneck 2: Write 1.1K req/s + counter contention
  → Pre-allocate range: mỗi app server lấy 1000 IDs từ ZooKeeper
    Fallback ZooKeeper khi exhausted
  → No bottleneck cho 100K writes/sec

Bottleneck 3: Storage 45TB
  → Shard by short_code (consistent hashing N nodes)
    Hot nodes cho popular URLs → replicate factor 3

Bottleneck 4: Analytics writes overwhelm DB
  → Async via Kafka. App publishes click event,
    Spark/Flink aggregates daily into BigQuery

Bottleneck 5: Geo latency
  → CDN at edge, cache redirect for 1 hour
    Edge writes go to nearest region, async replicate

Final architecture

                                ┌─ Redis Cluster ─┐
Client ─→ CDN ─→ LB ─→ App ─────┤                  │
                                └─ Cassandra ──────┤
                                          ↓        │
                                  Kafka ←─┘         │
                                          ↓        │
                                  Spark → BigQuery │
                                                    │
ID Gen ←─ ZooKeeper ←──────────────────────────────┘

Trade-offs để discuss

  • Cassandra vs MySQL: Cassandra wins write-heavy + horizontal scale, sacrifice strong consistency. URL shortener tolerate eventual OK.
  • Counter vs Hash: Counter cleaner; predictable URLs cho phép enumeration attack. Trade-off cho startup OK, enterprise dùng hybrid.
  • Cache TTL 24h: Trade-off freshness vs hit rate. URL hiếm update → TTL dài OK.

D2. Twitter Feed — Famous Fanout Problem HIGH-SIGNAL

S1: Scenario

FR:

  • Post tweet (280 chars text, optional image/video)
  • Follow/unfollow user
  • Home feed: chronological tweets from followees
  • Trending topics (optional)

NFR:

  • 200M DAU
  • Avg user follows 200, posts 2 tweets/day
  • Tweet writes: 200M × 2 / 86,400 ≈ 4,600 writes/sec
  • Feed reads: 200M × 10 reads/day / 86,400 ≈ 23K reads/sec
  • Latency: feed P99 < 200ms

Capacity:

Tweets/year: 200M × 2 × 365 = 146B/year
Per tweet ~1KB → 146 TB/year, 730 TB cho 5 năm

User graph: 200M users × 200 follows × 16 bytes = 640 GB
  (feasible — single multi-shard DB cluster)

S2: Service

                          ┌─ Tweet Service
                          ├─ User Service
Client → LB → Gateway ────┼─ Follow Service
                          ├─ Feed Service ─── Redis (timeline cache)
                          └─ Search Service ── Elasticsearch

S3: Storage

Tweets (Cassandra):
  tweet_id (PK):    long (snowflake)
  user_id:          long
  content:          string (280 chars)
  media_url?:       string
  created_at:       timestamp

Follows (wide-column or graph):
  follower_id (PK), followee_id (CK)

User (PostgreSQL hoặc DynamoDB):
  user_id, name, email, bio, profile_url

Timelines (Redis):
  user:{userId}:home_timeline → LIST of tweet_ids
  (LPUSH new tweets, LTRIM oldest, keep ~800)

S4: Scale — Fanout problem (trục chính)

Approach 1: Pull (read-time fanout)

User opens feed → Service queries:
  - Get list of followees from DB
  - For each, get latest tweets from Cassandra
  - Merge sorted by time, top N
  - Return

+ Cheap write: chỉ insert tweet
- Expensive read: 200 followee queries/feed open
  Slow cho popular users (Twitter active user nhiều follow)

Approach 2: Push (write-time fanout)

User posts tweet → Service:
  - Insert tweet into Cassandra
  - For each follower, push tweet_id vào Redis list:
    user:{follower}:home_timeline LPUSH

+ Cheap read: single Redis fetch
- Expensive write: celebrity 100M followers = 100M Redis writes
  Đây là "Justin Bieber problem"

Approach 3: Hybrid (industry standard)

Default: Push fanout via Kafka workers cho regular users
But: celebrity users (> X followers, vd 1M) dùng Pull

Feed service merges:
  - User's pushed tweets (from regular users họ follow)
  - Pull-fetched tweets từ celebrities

Result: 95% writes light (push), 5% reads heavier (pull cho celeb)
This là approach Twitter, Facebook thực tế dùng.

Trade-offs

  • Push vs Pull: Latency at read (push) vs write (pull). Hybrid để optimize cả 2.
  • Cassandra vs DynamoDB: Similar capability, choose by ecosystem (Amazon → DynamoDB).
  • Redis timeline depth: Cache 800 tweets (typical scroll). Hơn = memory expensive, ít user scroll deep.
  • Eventual consistency OK: Tweet xuất hiện chậm 1-2s acceptable. Read-your-write quan trọng cho UX.

Follow-up L4/L5

L4 follow-up: Viral tweet

"Tweet viral, cần handle?"

Approach: Trending detection via stream processing (Flink), boost cache TTL, increase read replicas dynamically.

L5 follow-up: Recommendation + ranking

"Feed ranked không phải chronological?"

Approach: ML model scoring (engagement, recency, social). Feature store, online inference at edge. Offline batch update daily, online refine real-time.

D3. Uber / Ride Sharing — Geospatial & Real-time HIGH-SIGNAL

S1: Scenario

FR:

  • Driver ping location every 4 seconds
  • Rider request ride A → B, matched with nearest driver
  • Real-time ETA updates
  • Trip completion + payment

NFR:

  • 1M concurrent drivers, 5M concurrent riders
  • Driver pings = 1M / 4 = 250K writes/sec (location updates)
  • Match request = 100K/sec peak
  • P99 match latency < 1s

S2: Service

                       ┌─ Location Service (sharded by geohash)
                       ├─ Match Service
Client → Gateway ──────┼─ Trip Service
                       ├─ Payment Service
                       └─ Notification Service (push, SMS)

S3: Storage

Driver location — Redis Geo commands

GEOADD drivers <lon> <lat> "<driver_id>"
GEORADIUS drivers <rider_lon> <rider_lat> 5 km WITHCOORD
  → Returns drivers trong 5km, sorted by distance

Trips — Cassandra

Table: trips
  trip_id (PK):    UUID
  rider_id, driver_id, start_loc, end_loc
  status: REQUESTED | MATCHED | IN_PROGRESS | COMPLETED
  fare, payment_id
  created_at, completed_at

S4: Scale

Sharding by Geohash

World divided into 6-char geohash cells (~1km × 1km)
Each cell handled by 1 server (or shard).

Driver location update → goes to right cell server.
Match request → query 9 cells (3×3 grid around rider).

→ Massive parallelism, no global lock.
→ Each server handles ~250K / số cells ops/sec → manageable.

Geohash advantages:
+ Hierarchical (zoom in/out by char length)
+ Prefix string distance ~ physical distance
+ Range queries trên contiguous space

Real-time updates với WebSocket

Driver app ↔ Location Service (WebSocket persistent)
Rider app  ↔ Trip Service (WebSocket)

Match flow:
  1. Rider request → Trip Service
  2. Trip Service query Match Service (geohash query)
  3. Match Service GEORADIUS → top 5 drivers
  4. Trip Service publishes match event qua Kafka
  5. Driver Service forwards match event → driver app via WebSocket
  6. Driver accept/reject within 15s
  7. Confirmed → trip MATCHED, both sides notified

Trade-offs

  • Update frequency 4s vs 1s: 1s = 4× server load. 4s đủ accuracy cho matching, smoother UX.
  • Geohash vs R-tree: Geohash đơn giản, distribute dễ. R-tree linh hoạt query complex shape — Uber dùng Geohash + custom indexing.
  • Strong consistency cho trip state: Mandatory (không double-book driver). Use distributed lock hoặc CAS.
  • Surge pricing: ML model query geohash density → pricing multiplier. Real-time update qua Kafka.

D4. WhatsApp / Chat App CORE

S1: Scenario

FR:

  • 1-on-1 chat (text, image, video)
  • Group chat (up to 256 members)
  • Online/offline status
  • Read receipts, delivery receipts
  • E2E encryption

NFR:

  • 2B users, 1B DAU
  • 100B messages/day = 1.2M msg/sec average
  • Peak 3× = 3.5M msg/sec
  • P99 latency < 500ms

S2: Service

                            ┌─ Auth Service
Client (mobile) → LB → GW ──┼─ Chat Service (WebSocket)
                            ├─ Storage Service
                            └─ Push Notification (offline users)

Persistent: mỗi client → 1 WebSocket connection tới chat server

S3: Storage

Messages (Cassandra, partitioned by chat_id):
  chat_id (PK), message_id (CK by time)
  sender_id, content_encrypted, sent_at, received_at, read_at

User connections (Redis):
  online:{user_id} → "server_id_42"
  (Which chat server holds this user's WebSocket)

S4: Scale

Connection routing

1B online users / 100K connections per server = 10K servers needed
Each user → consistent hashing → which server holds connection
Redis lookup: online:{user_id} → server_id

Message delivery flow

1. User A sends msg to User B
2. A's chat server stores msg vào Cassandra
3. Lookup B's connection → "server_id_42"
4. Forward msg via internal RPC tới server 42
5. Server 42 pushes msg over WebSocket tới B's app
6. B nhận → send ACK back → server stores received_at
7. B đọc → send read receipt → server stores read_at

If B offline:
  → Server 42 lookup fail → trigger push notification (APNS/FCM)
  → When B online lại → fetch missed messages từ Cassandra

Group chat fanout

1 sender → 256 receivers = 256 deliveries
Approach: Kafka topic per group
  → fanout workers consume messages from topic
  → mỗi worker chịu N members, forward via WebSocket
Throttle group: 256 max prevents abuse

Trade-offs

  • E2E encryption: Server không đọc message → no server-side search, no spam filter. Trade-off security cho features.
  • Delivery guarantee: At-least-once via ACK từ receiver. Duplicate detection bằng message_id.
  • Storage: Messages encrypted, key ở user device only. Backup encrypted (iCloud, Google Drive).
  • Online presence: Real-time vs eventual. Real-time (1s heartbeat) = expensive. Eventual (30s) UX vẫn OK.

D5. YouTube / Video Streaming CORE

S1: Scenario

FR:

  • Upload video
  • Watch video (adaptive streaming)
  • Search, recommendation
  • Like, comment, subscribe

NFR:

  • 2B DAU, 500 hours of video uploaded per minute
  • Watch: 1B hours/day → ~35M concurrent streams
  • Storage growing: 500h/min × 60 × 24 × 365 ≈ 260M hours/year
  • Per hour ~600 MB → ~150 PB/year, EB-scale total

S2: Service

Upload flow:
Client → Upload Service → Object Storage (S3)
                       → Encoding Service (multiple resolutions)
                       → Metadata Service (DB)
                       → CDN Origin Push

Watch flow:
Client → CDN Edge → Origin (cache miss) → S3
                → Manifest (HLS/DASH) → segment URLs

S3: Storage

Video metadata (DynamoDB):
  video_id, title, description, uploader, tags,
  duration, resolutions[], thumbnail_url

Video files (S3):
  s3://yt-videos/{video_id}/manifest.m3u8
  s3://yt-videos/{video_id}/720p_segment_001.ts
  s3://yt-videos/{video_id}/720p_segment_002.ts
  s3://yt-videos/{video_id}/1080p_segment_001.ts
  ...

User data (DynamoDB):
  watch_history, subscriptions, liked_videos

Comments (Cassandra):
  Partitioned by video_id, sorted by time

S4: Scale

Encoding pipeline

Raw upload → Kafka job queue → Encoding workers (FFmpeg)
  - Transcode to 360p, 480p, 720p, 1080p, 4K
  - Generate HLS manifest (.m3u8) + segments (.ts) every 6s
  - Generate thumbnails (multiple frames)
  → Push to CDN origin

CDN strategy

Multi-tier caching:
  Edge (POP)  → small, hot videos, ~1 sec từ user
  Regional    → ~50% library, ~10ms
  Origin (S3) → 100% library, ~100ms

Cache eviction: LRU + popularity-based prefetch (ML)
ML predicts hot videos overnight → pre-warm edge cache

Recommendation pipeline

Offline batch: Spark on logs daily → user-video affinity scores
Online serving: Redis với sorted set per user
Real-time signals: Kafka stream → update partial scores

Inference: 2-tower neural net (user embedding × video embedding)
Re-rank top 1000 candidates trong ~10ms

Trade-offs

  • Adaptive bitrate (HLS/DASH): Client picks resolution based on network. Smooth UX cho cả 4G và fiber.
  • Storage tier: Hot (S3) / Warm (S3-IA) / Cold (Glacier). 80% videos không xem trong 30 ngày → Glacier (90% cheaper).
  • Live streaming: Separate path với low-latency HLS, ~3s delay (vs VOD 30s+).
  • Copyright detection: Content ID scan upload. Async pipeline. Trade-off: false positive vs miss piracy.

6. L4 vs L5 vs L6 — Khác biệt thực tế HIGH-SIGNAL

L4 expectation

  • Vẽ được box-and-arrow đầy đủ trong 30-45 phút
  • Identify 2-3 bottleneck, suggest solution
  • Capacity estimation đúng order of magnitude
  • Hiểu trade-off cơ bản (SQL vs NoSQL, push vs pull)

L5 expectation = L4 + sâu hơn

  • Multiple trade-off levels: "If we choose Cassandra, we sacrifice X for Y. But if we use Spanner, we get Z but cost W."
  • LLD on 1 critical service: schema, API contract, error handling
  • Migration story: "How do we evolve from system today to this?"
  • Cross-team: "How does this interact with team B's service?"

L6 expectation = L5 + organizational

  • Define entire system across 5+ teams
  • ROI analysis (cost-benefit)
  • Hire & grow plan
  • Multi-year roadmap (1-3 năm)

⚠️ 7 anti-patterns trong FAANG System Design mock CORE

❌ #1: Skip Scenario, jump straight to Service

Symptom: "I'll use 3 services and Cassandra and Kafka..." — không clarify gì.

Fix: "First let me clarify: how many users? read/write ratio?" 5 phút clarify TRƯỚC khi vẽ.

❌ #2: "Just use Kafka for everything"

Symptom: Throw Kafka at every async problem.

Fix: Justify: "Kafka because we have async + replay + consumer groups. vs SQS simpler if don't need replay. vs RabbitMQ better cho low throughput + complex routing."

❌ #3: No capacity numbers

Symptom: "Lots of users, lots of data" — vague.

Fix: "100M DAU, 1KB per record, 86,400s/day → 1.2K writes/sec." Show math.

❌ #4: Magic auto-scaling

Symptom: "K8s sẽ auto-scale" — waving hands.

Fix: "Read replicas based on CPU > 70%, scale 2-10 replicas. Cold start time 30s, pre-warm khi traffic predict tăng."

❌ #5: No trade-off discussion

Symptom: "I chose Cassandra" — không justify.

Fix: "I chose Cassandra over MySQL for write throughput, sacrificing strong consistency. Acceptable here because feed eventually consistent OK for users."

❌ #6: Micro-optimize early

Symptom: Spend 20 phút trên bloom filter implementation trong 45-min round.

Fix: "I'd add bloom filter for cache prefilter, classic LRU otherwise." Mention optimization, don't implement đầy đủ.

❌ #7: No diagram

Symptom: Talk-only design, không vẽ.

Fix: Vẽ box-and-arrow rõ ràng (Excalidraw / paper / Google Doc). Interviewer cần visual để follow.

🎯 3-tier interview framework

Tier 1 — Junior (pre-L4)

Q1: 4S framework là gì? Time budget mỗi step?

Trả lời: "Scenario (5'), Service (10'), Storage (10'), Scale (15'), Q&A (5'). Total 45 phút. Scenario = clarify FR/NFR/capacity. Service = box-arrow architecture + top 5 APIs. Storage = DB choice + schema + sharding. Scale = bottleneck + trade-off."

Q2: Capacity estimation cho 100M DAU app, 10 actions/user/day?

Trả lời: "100M × 10 / 86,400 = ~12K avg QPS. Peak 3× = 36K QPS. Bandwidth: 12K × 500 bytes = 6 MB/sec đơn giản. Storage: depends actions — 1KB/action × 1B/day = 1 TB/day = 365 TB/year. Add replication 3× = 1 PB/year."

Tier 2 — Mid (L4 bar)

Q3: Twitter Feed fanout — Push vs Pull, choose nào?

"Hybrid. Push fanout (write-time) cho regular users — cheap reads at cost of celebrity write problem. Pull fanout (read-time) cho celebrity users (> 1M followers) — avoid 100M Redis writes per tweet.

Feed Service merges 2 sources khi user open timeline: pushed tweets từ regular followees + pull-fetched tweets từ celebs họ follow. Sort by time, return top 800 cached trong Redis.

Trade-off: hybrid phức tạp implement hơn pure push hoặc pure pull. Industry standard (Twitter, Facebook) chấp nhận complexity vì 95% latency tốt + cost manageable."

Q4: Uber driver location update — design 250K writes/sec?

"Shard by geohash. World chia thành cells ~1km × 1km bằng 6-char geohash. Mỗi cell có 1 server (hoặc shard) chịu trách nhiệm.

Driver location update 4s → routed to cell server based on driver's current geohash. Redis GEO commands trên cell server: GEOADD drivers lon lat driver_id. Mỗi cell server chỉ chịu drivers trong cell đó → load distribute.

Match request: query 9 cells (3×3 grid around rider). GEORADIUS trả top N nearest. Parallel queries, fast (~10ms total).

Trade-off: Geohash đơn giản hơn R-tree, distribute dễ. Hot cells (downtown SF) cần replicate. Trip state PHẢI strong consistency (không double-book) — dùng distributed lock cho transition state."

Tier 3 — Senior (L5 bar)

Q5: Design YouTube từ scratch cho 2B DAU. Migration story từ 100M DAU current state.

Phút 1 — Scenario:
  FR: upload, watch, search, recommend, like/comment
  NFR: 2B DAU, 500h/min upload, 35M concurrent streams
  Capacity: ~150 PB/year storage, EB total

Phút 2 — Service (high-level):
  Upload path: Client → Upload Service → S3 → Encoding (FFmpeg) → CDN push
  Watch path: Client → CDN Edge → Origin S3 → manifest m3u8 + segments
  Metadata: DynamoDB. Comments: Cassandra. Recommendation: ML pipeline.

Phút 3 — Storage details:
  Videos: S3 multi-tier (Hot/Warm/Glacier based on access pattern ML)
  Metadata: DynamoDB sharded by video_id
  Watch history: Kafka stream → batch insert Cassandra (cheap writes)
  Recommendation: Redis Sorted Set per user, refresh daily

Phút 4 — Scale + bottleneck:
  Encoding throughput: 500h/min × 5 resolutions = 2500h/min encoding
    → Distributed workers, Kafka job queue, FFmpeg
  CDN: 35M concurrent × 1 Mbps avg = 35 Tbps egress
    → Multi-tier cache (edge popular, regional medium, origin all)
    → Pre-warm cache cho expected viral videos
  Storage: cheap S3 Standard for hot 7-day, IA cho 30-day, Glacier 365+
    → 80% videos cold → Glacier saves 90% cost vs Standard

Phút 5 — Migration story L5:
  Current 100M DAU → Target 2B DAU (20× growth)
  Phase 1 (3 tháng): scale current architecture, add replicas
  Phase 2 (6 tháng): migrate to multi-region active-active
    → DNS routing closest region, async replication
  Phase 3 (6 tháng): Encoding pipeline refactor
    → Move from monolithic FFmpeg cluster to serverless (Lambda)
    → Cost reduce 40% via spot instances
  Phase 4 (ongoing): ML recommendation upgrade
    → Migrate from collaborative filtering to 2-tower neural net
    → A/B test, 5% rollout per week
  Trade-off: 18 tháng total migration, business cần ship features parallel
    → Strangler pattern: new architecture alongside old, gradual migration
