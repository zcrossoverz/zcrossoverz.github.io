Cryptography — Hash, Encrypt, Sign, TLS Đúng Cách

Crypto là nơi "có mã hoá" che giấu vô số lỗi chí mạng: hash mật khẩu bằng SHA-256 (sai!), dùng AES-ECB (lộ pattern), hardcode key. Lên thật phải phân biệt hashing vs encryption, symmetric vs asymmetric, hiểu vì sao bcrypt chậm là tính năng chứ không phải bug. File này dạy cơ chế, trên domain ShopFlow — mật khẩu & PII (email, địa chỉ, token) của sàn e-commerce.

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Hash mật khẩu bằng SHA-256 là an toàn vì SHA-256 mạnh."
Thực tế: SHA-256 thiết kế để nhanh → attacker thử hàng tỉ hash/giây (GPU). Mật khẩu cần hàm chậm có chủ đích + salt: bcrypt/scrypt/Argon2. "Chậm" ở đây là tính năng chống brute-force, không phải nhược điểm.
Tưởng hiểu: "Hashing và encryption đều biến data thành chuỗi khó đọc nên dùng cái nào cũng được."
Thực tế: Hashing một chiều (không giải ngược) — dùng cho mật khẩu, kiểm tra toàn vẹn. Encryption hai chiều (giải ngược bằng key) — dùng cho PII cần đọc lại (email, địa chỉ). Hash mật khẩu rồi cần "giải ra" = bạn đã hiểu sai.
Tưởng hiểu: "AES là mã hoá mạnh nên dùng AES kiểu gì cũng an toàn."
Thực tế: Mode quan trọng ngang thuật toán. AES-ECB mã hoá mỗi block độc lập → block giống nhau ra ciphertext giống nhau → lộ pattern (ảnh penguin nổi tiếng). Dùng AES-GCM (có IV + authentication tag) để vừa bí mật vừa chống sửa đổi.

1. Bài toán thực tế — lưu mật khẩu & PII CORE ★

ShopFlow cần lưu: (1) mật khẩu user, (2) email/địa chỉ giao hàng (PII cần đọc lại để giao), (3) token tích hợp đối tác. Một dev đề xuất "hash hết bằng SHA-256 cho an toàn". Một dev khác "mã hoá AES rồi base64 là xong".

⏸ Pause & Predict — trước khi đọc lời giải
Viết ra giấy:
  1. Mỗi loại data (mật khẩu / email-địa chỉ / token) nên dùng hashing hay encryption? Vì sao khác nhau?
  2. Vì sao SHA-256 cho mật khẩu là sai, dù SHA-256 "mạnh"? Dùng gì thay?
  3. "AES rồi base64" — base64 có phải mã hoá không? AES thiếu gì để an toàn (gợi ý: mode + IV)?
Đáp án: mật khẩu → hash chậm + salt (bcrypt); email/địa chỉ cần đọc lại → encryption hai chiều (AES-GCM); token → encryption hoặc vault. SHA-256 quá nhanh → brute-force; base64 chỉ là encoding, không bảo mật; AES cần GCM mode + IV ngẫu nhiên. Đừng đọc tiếp khi chưa map đúng 3 loại data.

2. Symmetric — AES-256-GCM CORE ★

Một key dùng để mã hoá và giải mã. Nhanh → dùng cho khối lượng lớn (PII, file). GCM cho cả bí mật (confidentiality) và toàn vẹn/chống sửa (authentication tag). Mỗi lần mã hoá dùng IV (nonce) ngẫu nhiên, không tái sử dụng.

// IV ngẫu nhiên mỗi lần; lưu IV cạnh ciphertext (IV không bí mật)
byte[] iv = randomBytes(12);
Cipher c = Cipher.getInstance("AES/GCM/NoPadding");
c.init(ENCRYPT_MODE, key, new GCMParameterSpec(128, iv));
byte[] ct = c.doFinal(plaintext);   // ct kèm auth tag → sửa 1 bit là giải mã fail

3. Asymmetric — RSA / ECDSA SUPPLEMENTARY

Cặp key: public (chia sẻ) + private (giữ bí mật). Chậm hơn symmetric → KHÔNG mã hoá data lớn trực tiếp, mà dùng để: trao đổi khoá (mã hoá session key), và chữ ký số. Đây là lý do TLS dùng asymmetric để bắt tay rồi chuyển sang symmetric cho data.

L1 — Biết tên

Biết "symmetric 1 key, asymmetric 2 key". Symptom hiểu giả: không nói được khi nào dùng cái nào, hoặc tính mã hoá file 1GB bằng RSA.

L2 — Explain được

Symmetric nhanh, vấn đề là chia sẻ key an toàn. Asymmetric giải bài chia sẻ key (mã hoá bằng public, chỉ private giải) nhưng chậm → chỉ dùng cho key exchange + signature. Hybrid = asymmetric trao session key, symmetric mã hoá data.

L3 — Apply được

Thiết kế envelope encryption: data key (symmetric) mã hoá data; KMS/master key (có thể asymmetric) mã hoá data key. Rotation master key không cần mã hoá lại toàn bộ data. Chọn ECDSA cho chữ ký nhẹ hơn RSA.

4. Hashing vs encryption — bcrypt cho mật khẩu CORE ★

HashingEncryption
ChiềuMột chiều (không giải ngược)Hai chiều (giải bằng key)
Dùng choMật khẩu, toàn vẹn (checksum)PII cần đọc lại (email, địa chỉ)
Ví dụ đúngbcrypt / Argon2 + saltAES-256-GCM
Ví dụ SAISHA-256/MD5 cho mật khẩuAES-ECB, hardcode key

Salt = chuỗi ngẫu nhiên thêm vào mỗi mật khẩu trước hash → hai user cùng mật khẩu ra hash khác → vô hiệu rainbow table. bcrypt nhúng salt + cost factor (số vòng) ngay trong chuỗi hash.

5. Digital signature SUPPLEMENTARY

Ký bằng private key, verify bằng public key → chứng minh (a) đúng người gửi (authenticity), (b) data không bị sửa (integrity), (c) không chối được (non-repudiation). Cơ chế: hash data → mã hoá hash bằng private key = chữ ký; verify = giải chữ ký bằng public key, so với hash tính lại.

6. TLS / mTLS handshake SUPPLEMENTARY

TLS handshake (đơn giản hoá):
1. Client hello (cipher suites hỗ trợ)
2. Server gửi certificate (chứa public key, ký bởi CA)
3. Client verify cert qua chuỗi CA tin cậy
4. Trao đổi khoá (ECDHE) → cả hai có session key (symmetric)
5. Từ đây dùng symmetric mã hoá data (nhanh)
mTLS: thêm bước client cũng gửi cert để server xác thực client (service-to-service)

⚠️ Anti-patterns CORE ★

🔧 Counter-example — "tự làm crypto" gần như luôn sai
  • Roll your own crypto: tự chế thuật toán/giao thức → gần như chắc có lỗ hổng tinh vi (timing attack, padding oracle). Dùng thư viện đã được kiểm định (JCA, Bouncy Castle, libsodium).
  • AES-ECB: mỗi block độc lập → lộ pattern. Luôn dùng GCM (hoặc CBC + HMAC) với IV ngẫu nhiên.
  • SHA-256/MD5 cho mật khẩu: quá nhanh → brute-force. Dùng bcrypt/Argon2.
  • Hardcode key trong code/repo: lộ qua Git, decompile. Dùng KMS/secret manager + rotation.
  • Bỏ qua verify certificate chain: trustAll = vô hiệu TLS, mở MITM.

Quy tắc: đừng phát minh; chọn nguyên thuỷ chuẩn (bcrypt cho mật khẩu, AES-GCM cho PII, TLS cho transport), quản key qua KMS, để thư viện lo chi tiết.

🎯 3-tier interview framework

Tầng 1 — SurfaceJunior gate

Q: Khi nào symmetric, khi nào asymmetric?

Symmetric (AES): nhanh, một key, mã hoá khối lượng lớn (PII, file), vấn đề là chia sẻ key. Asymmetric (RSA/ECDSA): hai key, chậm, dùng cho trao đổi khoá + chữ ký. Thực tế hybrid: asymmetric trao session key, symmetric mã hoá data.

Tầng 2 — Deep DiveSenior gate

Q: Giải thích TLS handshake.

Client hello → server gửi cert (public key, ký bởi CA) → client verify chain → ECDHE trao session key → chuyển sang symmetric mã hoá data. Asymmetric chỉ dùng để bắt tay vì chậm; data dùng symmetric. mTLS thêm client cert.

Tầng 3 — ArchitectureSenior+

Q: Thiết kế mã hoá end-to-end cho PII.

Envelope encryption: data key (AES-GCM) mã hoá PII; master key trong KMS mã hoá data key. Rotation master không cần mã hoá lại data. Per-field encryption cho cột nhạy cảm; crypto-shredding (xoá key) cho GDPR delete; audit truy cập key; TLS cho transit.

🔧 Lab bắt buộc — làm đúng & thấy cái sai (2 bài)

Lab 1 — bcrypt vs SHA-256. Hash 100K mật khẩu bằng SHA-256 và bằng bcrypt (cost 12), đo thời gian. Chứng minh SHA-256 nhanh gấp hàng nghìn lần → attacker brute-force dễ. Verify: hai user cùng mật khẩu ra hash bcrypt khác nhau (do salt). Tiêu chí: giải thích vì sao "chậm" là tính năng.

Lab 2 — ECB lộ pattern. Mã hoá một ảnh bitmap đơn giản (hoặc data lặp) bằng AES-ECB và AES-GCM. Quan sát ECB giữ pattern (block giống → ciphertext giống), GCM thì không. Tiêu chí: giải thích vì sao ECB nguy hiểm và vai trò IV trong GCM.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. Hashing vs encryption khác gì? Loại data nào dùng cái nào?
  2. Vì sao SHA-256 cho mật khẩu là sai? bcrypt "chậm" giúp gì? Salt làm gì?
  3. Symmetric vs asymmetric — vì sao TLS dùng cả hai?
  4. Vì sao AES-ECB nguy hiểm? GCM thêm gì?
  5. Chữ ký số chứng minh 3 điều gì? Cơ chế ký/verify?
Bí câu nào = quay lại đúng section. Trôi cả 5 = không tự bắn vào chân khi làm crypto.
