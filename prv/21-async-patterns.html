#21 — Async Patterns

Async không có nghĩa là “ném sang thread khác cho nhanh”. Senior backend phải biết khi nào dùng message queue, khi nào dùng job queue, khi nào chỉ cần `@Async`, và làm sao không mất việc khi service crash giữa chừng.

Tổng quan & Vì sao quan trọng CORE

Trong e-commerce, request đặt hàng hoặc upload sản phẩm không nên chờ mọi side effect: resize ảnh, gửi email, cập nhật search index, tính recommendation, sync warehouse view. Nhưng nếu chỉ quăng vào thread pool trong cùng JVM, app restart là mất việc. Nếu chỉ publish message sau khi commit DB, network lỗi có thể làm database đã đổi nhưng event không ra ngoài.

Async pattern tốt không bắt đầu từ câu “dùng Kafka hay RabbitMQ?”. Nó bắt đầu từ câu: work này có cần durability không, có cần ordering không, có cần user thấy trạng thái không, có thể retry bao nhiêu lần, và duplicate có gây hỏng dữ liệu không?

📜 Lịch sử ra đời của async patterns
Batch era — Hệ thống xử lý nền ban đầu thường chạy theo batch ban đêm: export, import, report, cleanup. Đơn giản nhưng latency cao và khó phản ứng theo event.

Message-oriented middleware — Các message broker xuất hiện để tách producer và consumer, giúp hệ thống chịu spike tốt hơn và retry độc lập hơn.

Microservices era — Khi mỗi business capability có database riêng, vấn đề mới là atomicity giữa local transaction và publish event. Outbox pattern trở thành cách phổ biến để không mất event sau DB commit.

Insight: async không chỉ để “nhanh hơn”. Nó là cách kiểm soát coupling, durability, retry và failure isolation.

1. Bài Toán Thực Tế — ShopSphere upload sản phẩm bị timeout CORE

ShopSphere cho seller tạo sản phẩm mới. Sau khi seller submit, hệ thống cần lưu product, resize 6 kích thước ảnh, scan nội dung ảnh, cập nhật search index, gửi notification cho follower và warm cache cho trang category. Junior code làm tất cả trong request path.

Synchronous implementation — một request ôm quá nhiều side effect
@PostMapping("/products")
public ProductResponse create(@RequestBody CreateProductRequest request) {
    Product product = productService.create(request);
    imageService.resizeAll(product.id());       // CPU + IO
    moderationService.scan(product.id());       // external API
    searchClient.index(product.id());           // network
    followerNotifier.notify(product.id());      // fan-out
    cacheWarmer.warm(product.categoryId());     // best effort
    return ProductResponse.from(product);
}
🔥 Hậu quả production

Latency p95 tăng từ 300ms lên 12s. Khi moderation API chậm, seller không tạo được product dù dữ liệu chính có thể đã hợp lệ. Khi request timeout nhưng server vẫn chạy tiếp, user retry tạo duplicate. Nếu app crash sau khi lưu product nhưng trước search index, product tồn tại trong database nhưng không tìm kiếm được.

⏸ Pause & Predict (BẮT BUỘC dừng)
Hãy chia 5 side effect trên thành 3 nhóm: phải đồng bộ, async durable, async best-effort.
  1. Cái nào phải hoàn thành trước khi trả response?
  2. Cái nào cần retry sau crash?
  3. Cái nào fail thì chỉ log/metric, không ảnh hưởng business state?
Nếu bạn gom tất cả vào một queue, bạn đang bỏ qua semantics của từng loại work.

2. Async Boundary — Không phải work nào cũng nên async CORE

★ CORE — Bắt buộc nắm chắc: Câu hỏi “khi nào async?” xuất hiện nhiều hơn câu hỏi “dùng Kafka thế nào?”.

Boundary đúng thường là: request path chỉ làm phần cần thiết để xác nhận command thành công và ghi state bền vững. Các work có thể làm sau được đưa sang queue/job với trạng thái rõ ràng. Nhưng async cũng làm hệ thống phức tạp hơn: eventual consistency, duplicate, retry storm, monitoring, replay tool, và user experience khi trạng thái chưa cập nhật.

Loại workVí dụPattern phù hợpCần chú ý
Critical state changeLưu product, validate ownerSynchronous DB transactionAtomic, fail fast
Business event`ProductCreated` cho search/recoOutbox + message queueAt-least-once, idempotent consumer
Long-running taskResize video/imageJob queueStatus, retry, visibility timeout
Best-effortWarm cache, analytics pingAsync fire-and-observeMetric, no business dependency
🧠 Misconception Check — “Async là nhanh hơn”
"Em chuyển sang async để API nhanh hơn, nên không cần quan tâm lỗi bên dưới."
Async chỉ chuyển thời điểm và nơi xảy ra lỗi. Nếu work đó quan trọng, bạn phải có durability, retry, DLQ, idempotency và monitoring. API nhanh hơn nhưng data âm thầm lệch thì đó là lỗi nặng hơn timeout.
❌ Dùng `@Async` để che latency
@Async
public void indexProduct(String productId) {
    searchClient.index(productId);
}
✅ Durable job/event có trạng thái
productOutboxRepository.save(new OutboxEvent(
    UUID.randomUUID(),
    "ProductCreated",
    product.id(),
    objectMapper.writeValueAsString(payload)
));
💡 Rule of thumb

Nếu mất work sau app restart là chấp nhận được, `@Async` có thể đủ. Nếu mất work làm sai business state, cần durable queue/job/outbox. Nếu user cần theo dõi tiến độ, cần job state machine chứ không chỉ message event.

3. Message Queue — Event đã xảy ra CORE

Message queue/event stream phù hợp khi service A muốn thông báo “điều này đã xảy ra” để nhiều consumer độc lập phản ứng. Ví dụ `ProductCreated` có thể được search-service, recommendation-service, audit-service, notification-service đọc theo consumer group riêng.

⏸ Pause & Predict — Event hay Command?
Với các tên sau, cái nào là event, cái nào là command?
  1. `ProductCreated`
  2. `ResizeProductImage`
  3. `SellerSuspended`
  4. `SendFollowerNotification`
Event là sự thật đã xảy ra. Command là yêu cầu làm việc.
❌ Event đặt tên như command
public record IndexProductMessage(String productId) {}
✅ Event nói sự thật domain
public record ProductCreated(
    UUID eventId,
    String productId,
    String sellerId,
    Instant occurredAt
) {}
🎯 Vì sao tên event quan trọng?

Nếu producer publish `IndexProductMessage`, producer đang biết consumer nào làm gì, coupling tăng. Nếu publish `ProductCreated`, producer chỉ công bố sự thật domain, consumer tự quyết định có index, notify hay update recommendation không.

❌ Consumer không idempotent
@KafkaListener(topics = "product-events")
void on(ProductCreated event) {
    searchClient.index(event.productId());
    notificationClient.notifyFollowers(event.sellerId());
}
✅ Tách consumer theo responsibility
@KafkaListener(topics = "product-events", groupId = "search-indexer")
void on(ProductCreated event) {
    if (inbox.markIfNew(event.eventId())) {
        searchIndexer.upsert(event.productId());
    }
}

4. Job Queue — Work cần làm, có trạng thái CORE

Job queue phù hợp với command/background work có lifecycle riêng: resize ảnh, transcode video, generate monthly seller report, import catalog CSV. Khác với event, job cần status để user/admin biết đang queued, running, succeeded, failed hay cần retry.

❌ Job không có lease/claim atomic
Job job = jobRepository.findFirstByStatus("QUEUED");
job.setStatus("RUNNING");
jobRepository.save(job);
✅ Claim job bằng atomic update
UPDATE media_job
SET status = 'RUNNING',
    locked_by = :workerId,
    locked_until = now() + interval '2 minutes'
WHERE id = (
  SELECT id FROM media_job
  WHERE status = 'QUEUED'
  ORDER BY created_at
  FOR UPDATE SKIP LOCKED
  LIMIT 1
)
RETURNING *;
❌ Retry ngay lập tức gây storm
catch (Exception ex) {
    job.setStatus("QUEUED");
    jobRepository.save(job);
}
✅ Retry có backoff và max attempts
catch (RetryableException ex) {
    job.scheduleNextAttempt(
        Backoff.exponentialWithJitter(job.attempts()),
        clock.instant()
    );
}
🔬 Thử Phá Vỡ — Job queue resize ảnh

Câu hỏi: tìm 3 cách làm job resize ảnh bị chạy trùng hoặc mất job.

Click để xem analysis (sau khi đã nghĩ)
  1. Worker crash sau khi set RUNNING: nếu không có `locked_until`, job kẹt vĩnh viễn. Cần lease timeout để worker khác claim lại.
  2. Hai worker cùng select job: nếu claim không atomic, cả hai cùng xử lý. Cần `FOR UPDATE SKIP LOCKED` hoặc atomic compare-and-set.
  3. Output không idempotent: resize trùng ghi hai object path khác nhau, database trỏ lung tung. Cần deterministic output key hoặc upsert theo `(product_id, size)`.

Insight: job queue không chỉ là bảng `jobs`. Nó là state machine có locking, lease, retry, dedupe và observability.

5. Outbox Pattern — DB commit xong thì event không được mất CORE

Outbox giải bài toán atomicity giữa local database transaction và message publish. Thay vì save product rồi gọi Kafka trực tiếp, service ghi product và outbox row trong cùng DB transaction. Một relay/poller đọc outbox và publish ra broker. Nếu app crash sau commit, outbox row vẫn còn để publish lại.

❌ Publish sau DB commit có lỗ mất event
@Transactional
public Product create(CreateProductRequest request) {
    Product product = productRepository.save(Product.from(request));
    kafkaTemplate.send("product-events", ProductCreated.from(product));
    return product;
}
✅ Product + outbox trong cùng transaction
@Transactional
public Product create(CreateProductRequest request) {
    Product product = productRepository.save(Product.from(request));
    outboxRepository.save(OutboxEvent.productCreated(product));
    return product;
}
Outbox table tối thiểu
CREATE TABLE outbox_event (
  id UUID PRIMARY KEY,
  aggregate_type VARCHAR(80) NOT NULL,
  aggregate_id VARCHAR(120) NOT NULL,
  event_type VARCHAR(120) NOT NULL,
  payload JSONB NOT NULL,
  status VARCHAR(20) NOT NULL,
  attempts INT NOT NULL DEFAULT 0,
  next_attempt_at TIMESTAMPTZ NOT NULL,
  created_at TIMESTAMPTZ NOT NULL,
  published_at TIMESTAMPTZ
);

CREATE INDEX idx_outbox_ready
ON outbox_event (status, next_attempt_at, created_at);
🎯 Vì sao outbox không đảm bảo exactly-once?

Relay có thể publish thành công rồi crash trước khi mark `PUBLISHED`; lần sau nó publish lại. Vì vậy outbox thường cho at-least-once delivery. Consumer vẫn phải idempotent. Outbox giải “không mất event sau DB commit”, không giải “không bao giờ duplicate”.

❌ Mark published trước khi broker ack
event.markPublished();
outboxRepository.save(event);
kafkaTemplate.send(topic, event.key(), event.payload());
✅ Chỉ mark sau khi publish thành công
kafkaTemplate.send(topic, event.key(), event.payload()).get();
event.markPublished(clock.instant());
outboxRepository.save(event);
🎯 Edge Cases — Production Reality

Outbox/job code chạy happy path chưa đủ. Production cần cover:

  • Race condition: nhiều relay cùng publish một row; fix bằng `FOR UPDATE SKIP LOCKED` hoặc status transition atomic.
  • Network failure: publish thành công nhưng response mất; relay retry tạo duplicate; fix bằng event id và idempotent consumer.
  • Scale limit: outbox table 100M rows làm poll chậm; fix bằng partition theo thời gian, archive, index đúng.
  • Security: payload outbox chứa email/số điện thoại; fix bằng data minimization, encryption, retention cleanup.
  • Data integrity: event payload không khớp schema version consumer; fix bằng schema version và compatibility tests.
  • Time/timezone: `next_attempt_at` dùng local time gây retry sai sau DST/timezone; fix dùng `Instant`/UTC.
  • Concurrent access: retry manual và relay tự động cùng publish; fix bằng lock hoặc command replay idempotent.
  • Poison event: một event payload lỗi retry mãi; fix bằng max attempts, dead status, operator workflow.
  • Backpressure: broker chậm làm outbox backlog tăng; fix bằng metric backlog age, rate limit producer và alert theo age.

6. Scheduler, Batch & `@Async` SUPPLEMENTARY

`@Scheduled`, Quartz, Spring Batch và `@Async` đều liên quan background work nhưng không thay thế nhau. `@Async` chạy trong process hiện tại. `@Scheduled` trigger theo thời gian. Quartz quản lý job scheduling durable hơn. Spring Batch phù hợp xử lý dữ liệu lớn theo chunk với restartability.

Công cụDùng khiKhông dùng khi
@AsyncBest-effort, request không cần chờ, mất work chấp nhận đượcWork quan trọng cần retry sau crash
@ScheduledCleanup, polling nhẹ, periodic health taskCluster nhiều node nhưng không có lock
QuartzJob có lịch cụ thể, durable trigger, pause/resumeHigh-throughput event stream
Spring BatchImport/export lớn, chunk processing, restartLow-latency event reaction
❌ Scheduled job chạy trên mọi pod
@Scheduled(fixedDelay = 60000)
void sendDailyDigest() {
    digestService.sendToAllUsers();
}
✅ Distributed lock hoặc single scheduler
@Scheduled(fixedDelay = 60000)
void sendDailyDigest() {
    lockService.runOnce("daily-digest", () ->
        digestService.enqueueMissingDigests()
    );
}
⚠️ Batch cần restartability

Nếu import 5 triệu sản phẩm từ CSV và fail ở dòng 3.2 triệu, senior answer không phải “run lại từ đầu”. Cần chunk checkpoint, skip policy có giới hạn, idempotent upsert và report lỗi theo dòng.

7. Production Playbook — Chọn pattern, vận hành queue/job/outbox ADVANCED

Async workflow production cần nhiều hơn một annotation hoặc một broker. Bạn cần quyết định pattern bằng tiêu chí rõ ràng, thiết kế state machine, định nghĩa retry policy, đặt metric và có runbook khi backlog tăng. Đây là phần giúp bạn trả lời interview ở tầng architecture thay vì chỉ kể tên pattern.

SUPPLEMENTARY Retry policy theo loại lỗi

Retry tốt bắt đầu từ taxonomy lỗi. Với e-commerce media processing, lỗi “file chưa replicate xong” có thể retry; lỗi “file format không hỗ trợ” không nên retry; lỗi “virus scanner timeout” retry có backoff; lỗi “image quá lớn vượt limit” đi thẳng failed và hiển thị cho seller.

❌ Một catch-all cho mọi lỗi
catch (Exception ex) {
    job.retryLater();
}
✅ Phân loại retryable/non-retryable
catch (UnsupportedMediaTypeException ex) {
    job.failPermanently("UNSUPPORTED_FORMAT", ex.getMessage());
} catch (TransientStorageException ex) {
    job.retryWithBackoff(clock.instant());
} catch (Exception ex) {
    job.deadLetter("UNKNOWN_ERROR", sanitize(ex));
}

CORE Outbox relay: runbook khi backlog tăng

Outbox backlog tăng có thể do broker chậm, relay chết, query poll thiếu index, payload lỗi schema, hoặc downstream consumer bị kẹt. Đừng xóa outbox row thủ công trước khi biết nguyên nhân.

  1. Đo backlog age: row cũ nhất chưa publish bao lâu, không chỉ count.
  2. Kiểm tra relay health: số relay instance, lock contention, poll query time, publish latency.
  3. Kiểm tra broker: publish timeout, authorization error, topic missing, partition unavailable.
  4. Phân loại lỗi payload: nếu schema lỗi, chuyển DEAD và tạo migration/replay plan.
  5. Rate limit: khi relay hồi lại, backlog publish quá nhanh có thể làm consumer downstream quá tải.
Outbox relay loop có claim, publish, mark
while (running) {
    List<OutboxEvent> batch = outboxRepository.claimReady("relay-1", 100);
    for (OutboxEvent event : batch) {
        try {
            publisher.publish(event.topic(), event.key(), event.payload()).get();
            outboxRepository.markPublished(event.id(), clock.instant());
        } catch (RetryablePublishException ex) {
            outboxRepository.scheduleRetry(event.id(), backoff(event.attempts()));
        } catch (NonRetryablePublishException ex) {
            outboxRepository.markDead(event.id(), sanitize(ex));
        }
    }
}

ADVANCED Backpressure cho async pipeline

Async thường làm caller nhanh hơn nhưng có thể dồn nợ vào queue. Queue không phải nơi giấu overload. Nếu worker xử lý 500 jobs/phút mà producer tạo 2,000 jobs/phút liên tục, backlog sẽ tăng đến khi storage đầy hoặc SLA bị phá. Backpressure cần xuất hiện ở API, scheduler, producer hoặc worker scaling.

SignalNghĩaAction
Backlog count tăngProduce > consumeScale worker nếu downstream chịu được, hoặc rate limit producer
Backlog age tăngSLA async bị đe dọaAlert theo business priority
Dead jobs tăngLỗi logic/schema/inputDừng retry, fix code/data, replay có kiểm soát
Retry attempts tăngDownstream/storage bất ổnBackoff dài hơn, circuit breaker, kiểm tra dependency
Worker CPU thấp nhưng backlog caoWorker chờ IO/lockProfile wall-clock, DB/downstream metrics
✅ Câu trả lời senior khi thiết kế async workflow

“Em sẽ tách command path khỏi background work bằng durable boundary. Với event cho nhiều service, dùng outbox + message broker. Với long-running work cần progress, dùng job queue có state machine, lease, retry/backoff, dead state. Em sẽ thiết kế idempotency trước, sau đó đặt metrics backlog age, retry rate, dead job count và runbook replay.”

7b. 3 Cấp Độ Mastery Cho Async CORE ★

Async dễ implement sai vì test happy path luôn pass. Bug chỉ lộ khi crash, retry, hoặc lệch DB. 3 trục dưới phân biệt rõ junior/mid/senior.

🎯 L1 / L2 / L3 cho 3 chủ đề CORE

Async Boundary — Khi nào async vs sync

L1 — Recognize

Biết "async = không block". Dùng @Async hoặc CompletableFuture mỗi khi nghe "chậm thì async". Không phân biệt user-facing latency vs background work.

Symptom: mọi method dài hơn 100ms đều thêm @Async, không nghĩ về durability hay retry.

L2 — Explain

Phân biệt 3 loại work: user-blocking (sync), user-deferred (async + ack ngay), background (queue). Biết @Async mất khi JVM restart, message queue thì không. Hiểu happy path async dễ, failure path khó.

Symptom: trong design, nói "thanh toán phải sync vì user cần ack, gửi email confirm thì async qua queue".

L3 — Apply

Quyết định async boundary dựa trên 3 câu hỏi: (1) user có cần kết quả ngay không? (2) work có durability requirement không? (3) failure cần retry hay accept loss? Map ra @Async / Job Queue / Message Queue / Outbox tương ứng. Recipe ở Section 7 là phản xạ.

Symptom: design doc có "async decision matrix" liệt kê từng workflow + chọn primitive nào + tại sao.

Outbox Pattern — Dual-write & Atomicity

L1 — Recognize

Biết "outbox để publish event đáng tin cậy". Đôi khi nhầm outbox với DLQ.

Symptom: code có orderRepo.save(); kafkaTemplate.send(); trong cùng method, nghĩ là đủ.

L2 — Explain

Hiểu dual-write problem: 2 system không thể commit atomically. Outbox = ghi event vào table cùng transaction với business data, relay đọc table và publish. Biết CDC (Debezium) là alternative không cần relay code.

Symptom: code review từ chối publish() trong @Transactional mà chưa có outbox hoặc @TransactionalEventListener.

L3 — Apply

Build outbox với (event_id UUID, payload JSONB, created_at, processed_at, version). Relay là separate process có advisory lock để không double-publish khi multi-instance. Idempotency consumer-side dùng event_id. Cleanup processed rows. Recipe debug "tiền mất hàng không" là phản xạ.

Symptom: production có metric "outbox_pending_count" + alert khi > 100 và stagnant 5 phút.

Retry, Backoff & Dead Letter

L1 — Recognize

Biết "retry khi fail". Implement retry bằng for(int i=0; i<3; i++) try{...}. Không có backoff, không có DLQ.

Symptom: consumer retry 3 lần liên tiếp trong 50ms, vẫn fail, message bị skip silently.

L2 — Explain

Hiểu exponential backoff + jitter. Phân biệt retryable vs non-retryable error (4xx vs 5xx). Biết DLQ để không mất message. Biết retry budget để không spam downstream.

Symptom: code retry có backoff (Spring Retry, Resilience4j) + decision logic "có nên retry hay đẩy DLQ".

L3 — Apply

Build retry strategy: short retry inline (3 lần, exponential), nếu vẫn fail thì publish retry topic với delay tăng dần (5min, 1h, 24h), cuối cùng DLQ. DLQ có alert + replay tool + ownership. Recipe debug "DLQ tăng" là phản xạ. Biết khi nào KHÔNG retry (idempotent guarantee chưa có, hoặc gây cascading failure).

Symptom: service có "retry stages dashboard" show count theo từng stage + DLQ owner + last replay timestamp.

Interview Drill Bank — Async workflow tình huống CORE

Luyện phần này bằng cách nói thành tiếng. Mỗi câu phải có boundary, failure mode và recovery. Nếu câu trả lời chỉ là “dùng queue”, chưa đủ senior.

CORE Q1. Khi nào `@Async` đủ, khi nào không đủ?

Answer mẫu: `@Async` đủ cho work best-effort trong cùng process, ví dụ warm cache hoặc log enrichment mà mất cũng chấp nhận được. Nó không đủ cho work business quan trọng vì app restart, thread pool reject, hoặc process crash có thể làm mất task. Nếu work cần retry sau crash, cần durable job queue/message queue/outbox với trạng thái và idempotency.

Signal: phải nhắc đến durability và process boundary, không chỉ “@Async chạy background”.

CORE Q2. Message queue khác job queue thế nào?

Answer mẫu: Message queue/event stream phù hợp thông báo sự kiện đã xảy ra cho nhiều consumer, ví dụ `ProductCreated`. Job queue phù hợp work cần làm có lifecycle, ví dụ `ResizeProductImage`, với status queued/running/succeeded/failed, retry, lease và progress. Event thường là fact; job thường là command. Nếu user cần xem tiến độ, job queue hợp hơn.

Signal: câu trả lời tốt dùng “event vs command” và “state machine”.

CORE Q3. Outbox pattern giải quyết atomicity thế nào?

Answer mẫu: Thay vì save DB rồi publish broker trong hai operation rời, service ghi business row và outbox row trong cùng local transaction. Relay đọc outbox và publish sau. Nếu service crash sau commit, outbox row vẫn còn. Outbox giải vấn đề mất event sau DB commit, nhưng không đảm bảo exactly-once vì relay có thể publish duplicate sau crash.

Signal: phải nói rõ “không mất event” và “vẫn có duplicate”.

SUPPLEMENTARY Q4. Thiết kế job resize ảnh product như thế nào?

Answer mẫu: Tạo job với unique key `(productId, imageId, size)`, status, attempts, `locked_by`, `locked_until`, `next_attempt_at`. Worker claim atomic bằng `FOR UPDATE SKIP LOCKED`, resize ra deterministic object key, upsert output metadata. Retry lỗi transient với backoff, fail permanent nếu format không hỗ trợ, dead-letter sau max attempts. API trả job status để seller biết ảnh đang xử lý.

Signal: interviewer muốn nghe lease/atomic claim/idempotent output.

CORE Q5. Làm sao tránh duplicate side effect khi retry?

Answer mẫu: Đặt idempotency key theo logical work, không theo attempt. Với job, unique `job_key`. Với event, unique `eventId` trong inbox/processed table. Với side effect, dùng deterministic output path hoặc upsert/unique constraint. Retry có thể chạy lại cùng work nhiều lần, nên side effect phải biến “chạy lại” thành no-op hoặc update cùng record.

Signal: phải đưa được mechanism cụ thể ở storage layer, không chỉ nói “check duplicate”.

SUPPLEMENTARY Q6. Scheduler chạy nhiều pod sẽ gây lỗi gì?

Answer mẫu: Nếu `@Scheduled` chạy trên mọi pod, mỗi instance có thể enqueue/gửi cùng một job, tạo duplicate digest, duplicate cleanup hoặc double processing. Cần distributed lock, leader election, dedicated scheduler instance, hoặc thiết kế job enqueue idempotent bằng unique key. Với Quartz clustered mode, dùng database lock/trigger state đúng cách.

Signal: câu trả lời tốt nhắc đến cluster, không chỉ local scheduler.

ADVANCED Q7. Outbox backlog tăng, bạn debug thế nào?

Answer mẫu: Xem backlog age và count, relay health, poll query time, lock contention, publish latency, broker error, schema error và retry/dead count. Nếu broker hồi phục sau outage, rate limit relay để không làm consumer quá tải. Không xóa row thủ công. Nếu payload poison, mark dead với reason rồi tạo replay/migration plan.

Signal: phải có runbook theo triệu chứng, không chỉ “scale relay”.

CORE Q8. Async có làm hệ thống eventually consistent. Bạn xử lý UX thế nào?

Answer mẫu: UI/API phải hiển thị state trung gian rõ: product created nhưng image processing, search indexing pending. Với work user quan tâm, expose `GET /jobs/{id}` hoặc field trạng thái. Command path chỉ xác nhận phần đã durable. Không trả “xong hết” khi các side effect async chưa chạy. Notification/search có thể trễ, nhưng user phải thấy trạng thái hợp lý.

Signal: senior answer nối backend semantics với user experience.

ADVANCED Q9. Spring Batch khác job queue thường ở đâu?

Answer mẫu: Spring Batch phù hợp xử lý dữ liệu lớn theo step/chunk, có job repository, checkpoint, restartability, skip/retry policy và partitioning. Job queue thường xử lý nhiều work item độc lập theo worker. Import 1M sản phẩm từ CSV nên dùng batch/chunk để commit từng phần và restart từ checkpoint. Resize từng ảnh độc lập thường dùng job queue.

Signal: biết chọn công cụ theo workload shape.

SUPPLEMENTARY Q10. Queue backlog tăng có phải cứ scale worker?

Answer mẫu: Không. Scale worker chỉ đúng nếu bottleneck là worker compute và downstream còn capacity. Nếu worker chờ DB/storage, scale làm downstream quá tải hơn. Cần xem backlog age, worker CPU, job duration, retry rate, DB/downstream latency, dead jobs. Có thể cần rate limit producer, backpressure API, tối ưu job, hoặc tách priority queue.

Signal: tránh reflex scale; phải đọc saturation.

Async Cookbook — 8 workflow thường gặp SUPPLEMENTARY

Cookbook này giúp bạn mapping requirement sang pattern nhanh hơn. Khi interview đưa case, hãy nhận diện workload shape trước: event broadcast, long-running job, scheduled batch, fan-out notification, hoặc consistency bridge.

CORE Recipe 1 — Product created cần search index

Pattern: transactional outbox + message queue + idempotent consumer.

Why: product DB là source-of-truth. Search index là read model async. Nếu DB commit xong mà event mất, product không tìm được. Outbox bảo vệ event; consumer upsert index theo `productId` và dedupe theo `eventId`.

Trap: đừng index trực tiếp trong transaction tạo product vì search latency làm request chậm và rollback semantics không rõ.

CORE Recipe 2 — Resize ảnh/video sau upload

Pattern: durable job queue có status, lease, retry và deterministic output.

Why: user cần biết processing đang chạy, job có thể mất vài giây/phút, worker có thể chết giữa chừng. Output path nên deterministic như `products/{productId}/images/{imageId}/size-{size}.webp` để retry không tạo object rác.

Trap: đừng dùng `@Async` nếu seller cần chắc chắn ảnh sẽ được xử lý sau restart.

SUPPLEMENTARY Recipe 3 — Gửi follower notification

Pattern: event consumer + fan-out job queue.

Why: `ProductCreated` có thể kích hoạt notification, nhưng gửi cho 1 triệu follower không nên làm trong Kafka listener trực tiếp. Listener enqueue fan-out jobs theo shard/follower batch, ack nhanh, worker gửi notification có rate limit và retry.

Trap: gửi notification trực tiếp trong consumer làm `max.poll.interval.ms` bị vượt và consumer rebalance.

SUPPLEMENTARY Recipe 4 — Import catalog CSV lớn

Pattern: Spring Batch hoặc batch job với chunk checkpoint.

Why: file lớn cần restart từ checkpoint, skip invalid row có policy, commit theo chunk, report lỗi theo dòng. Queue từng row có thể overhead lớn và khó tạo report tổng thể.

Trap: import lại từ đầu sau fail dễ tạo duplicate nếu không upsert theo `(sellerId, sku)`.

CORE Recipe 5 — Sync read model cho social timeline

Pattern: event stream + projection consumer, đôi khi dùng rebuild/replay.

Why: timeline/read model có thể eventually consistent. Consumer cần version guard để event cũ không overwrite state mới. Khi projection bug, có thể rebuild từ event log hoặc source-of-truth snapshot.

Trap: nếu consumer không idempotent, replay để rebuild sẽ nhân đôi item trong timeline.

ADVANCED Recipe 6 — Outbox relay bằng polling hay CDC?

Polling dễ implement, kiểm soát batch/retry trong app, đủ tốt cho nhiều hệ thống. Nhược điểm là query/poll overhead, cần index/cleanup tốt.

CDC như Debezium đọc database log và publish change gần real-time, giảm custom poller nhưng thêm hạ tầng, schema/connector vận hành, và cần hiểu transaction log semantics.

Decision: team nhỏ, throughput vừa, cần kiểm soát nhanh: polling. Throughput cao, nhiều table, muốn stream DB changes chuẩn hóa: CDC.

SUPPLEMENTARY Recipe 7 — Priority jobs

Pattern: queue/job table có priority + fair scheduling.

Why: thumbnail cho product hot cần nhanh hơn report offline. Nhưng priority tuyệt đối có thể starve low-priority jobs. Thiết kế worker lấy theo priority nhưng có quota hoặc aging.

Trap: một queue chung FIFO làm job lớn phía trước chặn mọi job nhỏ phía sau.

ADVANCED Recipe 8 — Multi-step workflow

Pattern: state machine hoặc orchestration nhẹ, không chỉ nhiều queue nối nhau.

Why: upload media có các bước scan, transcode, thumbnail, publish. Mỗi bước có retry/failure riêng. Nếu chỉ publish message bước sau mà không lưu workflow state, debug sẽ khó khi step 3 fail. State machine giúp biết workflow đang ở đâu và resume thế nào.

Trap: choreograph quá nhiều event mà không có observability sẽ biến workflow thành “mê cung event”.

📖 Câu nhớ nhanh

Async design tốt là thiết kế trạng thái và failure trước, chọn broker/framework sau. Nếu không nói được work nằm ở state nào khi crash, pattern chưa đủ production-ready.

Self-Test Scenarios — Thiết kế async workflow bằng tay CORE

Hãy tự trả lời trước khi đọc gợi ý. Mục tiêu là luyện khả năng chọn đúng pattern, không phải nhớ tên pattern.

CORE Scenario 1 — Product created nhưng search index trễ

Bối cảnh: seller tạo product thành công, 5 phút sau vẫn không search được. DB có product, Kafka không thấy event tương ứng.

Tự trả lời: bug có thể nằm ở đâu? Pattern nào ngăn mất event?

Gợi ý: nếu service save DB rồi publish event ngoài transaction, crash/network lỗi có thể làm mất event. Dùng outbox: product + outbox row cùng transaction, relay publish sau. Thêm metric outbox backlog age và audit event id để trace product → event → index.

CORE Scenario 2 — Worker chết giữa resize ảnh

Bối cảnh: job status RUNNING mãi, ảnh không xuất hiện. Worker pod bị restart trong lúc xử lý.

Tự trả lời: job table cần field gì để recover?

Gợi ý: cần `locked_by`, `locked_until`, status, attempts, next_attempt_at. Worker khác reclaim job sau lease timeout. Output phải deterministic/upsert để retry không tạo duplicate. UI nên hiển thị processing/retry thay vì im lặng.

SUPPLEMENTARY Scenario 3 — Notification fan-out làm consumer lag

Bối cảnh: một seller lớn đăng sản phẩm, consumer `product-events` gửi notification trực tiếp cho 2 triệu follower. Consumer lag tăng, rebalance liên tục.

Gợi ý: Kafka listener nên ack nhanh sau khi enqueue fan-out jobs theo shard/batch. Worker notification xử lý rate-limited, retry/DLQ riêng. Long-running fan-out không nằm trong poll loop. Dùng idempotency key cho notification `(eventId, followerId)`.

ADVANCED Scenario 4 — Outbox relay publish duplicate

Bối cảnh: relay publish thành công rồi crash trước mark published. Sau restart, event được publish lại. Search index ổn nhưng notification gửi hai lần.

Gợi ý: outbox là at-least-once. Consumer notification phải dedupe theo `eventId` hoặc `(eventId, recipientId)`. Với external push provider, gửi idempotency key nếu hỗ trợ. Nếu không hỗ trợ, lưu send record trước/atomic với state để tránh gửi lại.

CORE Scenario 5 — Batch import fail ở 70%

Bối cảnh: import 1M products từ CSV, fail ở dòng 700k. Team run lại từ đầu và tạo duplicate SKU.

Gợi ý: dùng Spring Batch/chunk checkpoint, upsert theo `(sellerId, sku)`, skip policy có giới hạn, restart từ checkpoint. Report invalid rows. Không rely vào “file nhỏ nên chạy lại được”.

SUPPLEMENTARY Scenario 6 — Queue chung bị job nặng chặn

Bối cảnh: cùng một job queue xử lý resize ảnh nhỏ và transcode video dài. Khi nhiều video lớn vào queue, ảnh sản phẩm mới mất 30 phút mới có thumbnail.

Gợi ý: tách queue/pool theo workload hoặc priority, giới hạn concurrency cho job nặng, dùng fair scheduling/aging để tránh starvation. Metric cần có backlog age theo job type, không chỉ tổng backlog. Một queue FIFO duy nhất thường không đủ khi job duration phân bố quá lệch.

ADVANCED Scenario 7 — Workflow nhiều bước khó debug

Bối cảnh: media upload đi qua scan, transcode, thumbnail, publish. Mỗi step publish message cho step sau. Khi thumbnail không có, team mất 2 giờ tìm xem workflow kẹt ở đâu.

Gợi ý: cần workflow state hoặc correlation id xuyên suốt. Mỗi step ghi status, attempt, error, timestamps. Event choreography vẫn dùng được, nhưng phải có observability để query “mediaId này đang ở step nào”. Với workflow critical, cân nhắc orchestration/state machine thay vì chỉ nối queue mù.

CORE Scenario 8 — API trả success nhưng async step fail

Bối cảnh: API tạo product trả 201 ngay sau khi ghi DB. Sau đó moderation async fail permanent vì ảnh vi phạm policy. Seller phàn nàn vì tưởng product đã published.

Gợi ý: response phải nói đúng state, ví dụ product ở `PENDING_REVIEW` hoặc `PROCESSING`, không phải `PUBLISHED`. Async workflow cần state visible cho user, notification khi fail, và command path không hứa điều chưa hoàn tất. Đây là UX consequence của eventual consistency.

SUPPLEMENTARY Scenario 9 — Dead jobs không có owner

Bối cảnh: `DEAD` jobs tăng mỗi ngày nhưng không team nào xử lý vì API vẫn trả nhanh.

Gợi ý: async failure vẫn là product failure. Dead queue cần dashboard, alert, owner, replay/fix workflow và SLA. Nếu không, queue chỉ biến lỗi realtime thành dữ liệu bẩn tích lũy.

Calibration: nếu bạn không chỉ được ai chịu trách nhiệm khi job chết, hệ thống async đó chưa production-ready dù throughput có cao. Một queue khỏe trên dashboard hạ tầng nhưng không có owner nghiệp vụ vẫn có thể tạo incident âm thầm trong nhiều tuần.

Interview Framework CORE

Tầng 1 — Surface Pattern selection
Q: Khác nhau giữa `@Async`, message queue và job queue?
`@Async` chạy trong cùng process, không durable. Message queue phù hợp publish event đã xảy ra cho nhiều consumer. Job queue phù hợp work cần làm có lifecycle, status, retry và ownership. Chọn pattern theo durability, ordering, retry, user-visible status và failure handling.
Tầng 2 — Deep Dive Outbox semantics
Q: Outbox pattern giải quyết vấn đề gì và không giải quyết vấn đề gì?
Outbox giải atomicity giữa DB commit và event publish bằng cách ghi event trong cùng DB transaction. Nó không đảm bảo exactly-once delivery; relay có thể publish duplicate nếu crash sau publish trước khi mark published. Consumer vẫn phải idempotent.
Tầng 3 — Architecture Workflow design
Q: Thiết kế workflow upload product có resize ảnh, scan nội dung, search index và notification.
Request path validate và lưu product draft. Ghi outbox `ProductCreated`. Search/reco consume event idempotent. Resize/scan là job queue có status, lease, retry, dead state. Notification có thể consume event riêng. UI hiển thị trạng thái processing. Metrics gồm backlog age, retry count, dead jobs và search lag.
