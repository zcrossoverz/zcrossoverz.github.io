1. Bài toán thực tế — Token leak production CORE ★

📖 Tình huống — E-commerce API bị gọi trái phép

Bối cảnh: Shop API có 5K user. Tuần này phát hiện 50 account đặt order với địa chỉ giao lạ, charge card thật. Customer phàn nàn.

Debug:

  1. Audit log show: order tạo bởi đúng user_id, nhưng từ IP lạ (US, RU)
  2. User claim không login lúc đó
  3. JWT log: token hợp lệ, signature đúng, chưa expire

Root cause: 3 nguyên nhân kết hợp:

# 1. Log token vào ELK
log.info("Auth request: Authorization={}", request.getHeader("Authorization"));
# → ELK index public dashboard → attacker quét → có 5K token

# 2. Access token TTL = 30 ngày
jwt:
  access-token-ttl: 30d   # ❌ quá dài cho mobile + web

# 3. Không có token rotation hoặc revoke list
# → token leak vẫn valid 30 ngày

Fix immediate (4 giờ):

  1. Rotate JWT signing key → invalidate mọi token
  2. Force re-login mọi user
  3. Xóa Authorization header khỏi ELK index, scrub log history
  4. Alert Stripe để refund unauthorized charge

Fix long-term:

  1. Access token TTL = 15 phút. Refresh token TTL = 7 ngày, single-use rotation
  2. Logger filter strip Authorization, Cookie, X-API-Key headers
  3. Token blacklist (Redis) cho revoke ngay
  4. Device fingerprint + IP anomaly detection
⏸ Pause & Predict
  1. Tại sao access token TTL 30 ngày là tệ? Trade-off nếu 15 phút?
  2. Refresh token rotation hoạt động thế nào? Vì sao "single-use" quan trọng?
  3. Tại sao rotate signing key invalidate mọi token? Có cách nào revoke 1 token cụ thể không?

2. Spring Security Filter Chain CORE ★

Mỗi request đi qua chuỗi filter trước khi đến Controller. Filter nào cũng có thể reject (401/403) hoặc tiếp tục.

Request → SecurityContextPersistenceFilter   (load context từ session)
        → HeaderWriterFilter                  (add security headers)
        → CorsFilter                          (CORS preflight)
        → CsrfFilter                          (CSRF token check, default ON cho session)
        → LogoutFilter
        → UsernamePasswordAuthenticationFilter (form login)
        → BearerTokenAuthenticationFilter      (JWT bearer)
        → ExceptionTranslationFilter           (catch AuthException → 401)
        → AuthorizationFilter                  (check authorization rules)
        → DispatcherServlet                    (route to Controller)

Debug filter chain:

logging:
  level:
    org.springframework.security: DEBUG
# Sẽ log mọi filter visit + decision

Custom filter — JWT example:

@Component
public class JwtAuthFilter extends OncePerRequestFilter {
    private final JwtService jwt;

    @Override
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res, FilterChain chain)
            throws ServletException, IOException {
        String header = req.getHeader("Authorization");
        if (header != null && header.startsWith("Bearer ")) {
            String token = header.substring(7);
            try {
                Claims claims = jwt.validate(token);
                var auth = new UsernamePasswordAuthenticationToken(
                    claims.getSubject(), null,
                    parseAuthorities(claims.get("roles", List.class)));
                SecurityContextHolder.getContext().setAuthentication(auth);
            } catch (JwtException e) {
                // Log nhưng KHÔNG block — để filter sau quyết định
                log.debug("Invalid JWT: {}", e.getMessage());
            }
        }
        chain.doFilter(req, res);
    }
}

3. JWT — Stateless Authentication CORE ★

JWT = JSON Web Token. 3 phần encode base64url: header.payload.signature.

eyJhbGciOiJIUzI1NiJ9          ← header  {"alg":"HS256"}
.eyJzdWIiOiJVU1JfMSIsImV4cCI6MTcwMDAwMDAwMH0  ← payload {"sub":"USR_1","exp":...}
.SflKxwRJSMeKKF2QT4fwpMeJf36...  ← signature HMAC(header+payload, secret)

3 claim chuẩn (RFC 7519)

  • sub — subject (user id)
  • iat — issued at (timestamp)
  • exp — expiration
  • iss — issuer (service nào tạo)
  • aud — audience (service nào được dùng)
  • jti — JWT ID (cho revoke list)

Stateless vs Session

JWT StatelessSession-based
StorageClient (cookie/localStorage)Server (Redis/DB)
ScaleDễ horizontal scaleSticky session hoặc shared store
RevokeKhó (cần blacklist)Dễ (xóa session)
Payload size~200-500 byte/request~32 byte session id
XSS risklocalStorage = cao. HTTP-only cookie = thấpHTTP-only cookie = thấp
⚠️ Bẫy JWT junior
  • JWT KHÔNG phải encryption — payload base64, ai cũng decode được. KHÔNG đặt password/PII trong payload.
  • Signature ≠ encryption — chỉ verify integrity. Want encrypt: dùng JWE thay JWT.
  • HS256 cần shared secret giữa services. RS256 (asymmetric) tốt hơn cho microservices — chỉ issuer cần private key.
  • alg:none attack: JWT spec cho phép "alg":"none" — đảm bảo verify reject. Modern lib default reject.

4. OAuth2 vs JWT SUPPLEMENTARY

JWT và OAuth2 KHÔNG cùng tầng:

  • JWT = token format (cách encode credential)
  • OAuth2 = authorization framework (cách obtain + dùng access token)

OAuth2 thường dùng JWT làm access token, nhưng cũng có thể dùng opaque token.

OAuth2 4 grant types

  • Authorization Code — web app + user login qua provider (Google, GitHub)
  • Client Credentials — service-to-service, không user (vd: microservice gọi internal API)
  • Password — deprecated, không nên dùng
  • Implicit — deprecated, không nên dùng

OIDC (OpenID Connect) = OAuth2 + ID Token (info user). Dùng khi cần biết "ai đăng nhập" không chỉ "có quyền gì".

6. Production Config CORE ★

@Configuration
@EnableWebSecurity
@EnableMethodSecurity   // bật @PreAuthorize
public class SecurityConfig {

    private final JwtAuthFilter jwtFilter;

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(csrf -> csrf.disable())                           // stateless API
            .sessionManagement(s -> s.sessionCreationPolicy(STATELESS))
            .cors(c -> c.configurationSource(corsSource()))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**", "/actuator/health").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class)
            .exceptionHandling(e -> e
                .authenticationEntryPoint(jsonAuthEntryPoint())
                .accessDeniedHandler(jsonAccessDeniedHandler())
            )
            .build();
    }

    @Bean
    public CorsConfigurationSource corsSource() {
        CorsConfiguration c = new CorsConfiguration();
        c.setAllowedOrigins(List.of("https://shop.example.com"));   // explicit, KHÔNG "*"
        c.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE"));
        c.setAllowedHeaders(List.of("Authorization", "Content-Type"));
        c.setAllowCredentials(true);                                 // OK vì allowedOrigins explicit
        c.setMaxAge(3600L);
        UrlBasedCorsConfigurationSource src = new UrlBasedCorsConfigurationSource();
        src.registerCorsConfiguration("/api/**", c);
        return src;
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder(12);   // cost 12 → ~250ms/hash (anti-brute force)
    }
}

7. Method Security với SpEL CORE ★

@Service
public class OrderService {

    // Role-based
    @PreAuthorize("hasRole('ADMIN')")
    public void deleteOrder(String id) { ... }

    // Ownership check — chỉ owner xem được
    @PreAuthorize("#userId == authentication.principal")
    public List<Order> getUserOrders(String userId) { ... }

    // Combine
    @PreAuthorize("hasRole('SUPPORT') or #order.customerId == authentication.principal")
    public void updateOrderNote(@P("order") Order order, String note) { ... }

    // Post check (sau khi method run, check kết quả)
    @PostAuthorize("returnObject.customerId == authentication.principal")
    public Order findById(String id) { ... }

    // Filter list
    @PostFilter("filterObject.customerId == authentication.principal")
    public List<Order> findAll() { ... }
}
⚠️ Bẫy
  • @PreAuthorize chỉ work qua Spring proxy → self-invocation bị bypass (giống @Transactional)
  • @PostFilter trên list lớn = chậm (filter sau khi load all)
  • SpEL parse runtime → typo không catch lúc compile. Dùng test cover authorization.

8. Refresh Token Flow ADVANCED

Login:
  POST /auth/login {email, password}
  → access_token (TTL 15min, in body)
     refresh_token (TTL 7d, in HttpOnly Secure cookie, single-use)

Access:
  GET /api/orders   Authorization: Bearer <access_token>
  → 200

Token expire:
  GET /api/orders   Authorization: Bearer <expired>
  → 401

Refresh:
  POST /auth/refresh   Cookie: refresh_token=<old>
  → access_token (new)
     refresh_token (NEW — rotate)
  Old refresh_token marked used. Reuse = security incident → logout all sessions.

Logout:
  POST /auth/logout
  → blacklist access_token (Redis TTL = remaining ttl)
     delete refresh_token from DB

Refresh rotation single-use = nếu attacker dùng stolen refresh trước user → user dùng refresh hợp lệ sẽ fail → detect → force logout. Phòng MITM.

9. 5 Anti-Patterns CORE ★

❌ #1 — Log token / sensitive header
// ❌
log.info("Request: {}", request.getHeaders());   // includes Authorization
log.debug("Token: {}", token);

// ✅ Filter ra
Set<String> SENSITIVE = Set.of("authorization", "cookie", "x-api-key");
headers.entrySet().stream()
    .filter(e -> !SENSITIVE.contains(e.getKey().toLowerCase()))
    .forEach(...);
❌ #2 — CORS allowedOrigins("*") + allowCredentials(true)

Spec CORS cấm combo này. Browser sẽ reject. Đa số junior copy code, không hiểu — production fail khi browser thật vào.

Fix: list explicit origin: setAllowedOrigins(List.of("https://shop.example.com")).

❌ #3 — ROLE_ prefix không nhất quán
// Spring quy ước: hasRole("ADMIN") → check "ROLE_ADMIN" trong authorities
@PreAuthorize("hasRole('ADMIN')")          // check ROLE_ADMIN
@PreAuthorize("hasAuthority('ADMIN')")     // check ADMIN literal

// Khi tạo authority:
new SimpleGrantedAuthority("ROLE_ADMIN")    // dùng với hasRole
new SimpleGrantedAuthority("ADMIN")         // dùng với hasAuthority

// ❌ Mix → @PreAuthorize fail silent vì naming mismatch
❌ #4 — Hard-code secret trong code
// ❌
private static final String JWT_SECRET = "my-super-secret-key";

// ✅ Env var hoặc Vault
@Value("${jwt.secret}")
private String secret;
// application-prod.yml: jwt.secret: ${JWT_SECRET}

Lỡ commit secret: rotate ngay, BFG Repo-Cleaner để xóa history, scan git log có còn không.

❌ #5 — Không validate JWT claims đầy đủ
// ❌ Chỉ check signature + expiration
jwt.parse(token).getBody();

// ✅ Validate full
Claims c = jwt.parse(token).getBody();
if (!"shop-api".equals(c.getIssuer())) throw new InvalidTokenException();
if (!c.getAudience().contains("shop-mobile")) throw new InvalidTokenException();
if (blacklist.contains(c.getId())) throw new RevokedTokenException();

🚨 Recipe #2 — "CORS preflight fail"

Symptom: Browser console Access blocked by CORS policy. curl OK vì không gửi Origin header.

3 nguyên nhân:

  1. allowedOrigins không match origin client (case-sensitive, scheme + port matter)
  2. allowedMethods không có OPTIONS (preflight)
  3. allowedHeaders không có Authorization

Debug: curl -v -X OPTIONS -H "Origin: https://x.com" -H "Access-Control-Request-Method: POST" ...

🚨 Recipe #3 — "@PreAuthorize không work"

3 nguyên nhân:

  1. Quên @EnableMethodSecurity
  2. Self-invocation (this.method)
  3. Method private hoặc final — proxy không override được
🚨 Recipe #4 — "Token leak — phản ứng emergency"
  1. Rotate signing key NGAY (mọi token cũ invalid)
  2. Force re-login mọi user (clear refresh DB)
  3. Scrub log: ELK API delete documents có Authorization header
  4. Audit: identify endpoint nào log token, fix
  5. Notify user nếu có data leak

🎤 Interview Framework CORE ★

Q: JWT 3 phần là gì?

Header (alg + type) + Payload (claims) + Signature (HMAC hoặc RSA sign over header+payload). Base64url encode, ngăn bằng dấu chấm. Payload không encrypt — chỉ signature verify integrity.

Q: Khác stateful session vs stateless JWT?

Session lưu server, scale cần sticky/Redis. Revoke dễ. JWT lưu client, stateless dễ scale, revoke khó (cần blacklist). Production thường combine: JWT short-lived + refresh trong server store có thể revoke.

Q: Design auth cho 20 microservice + mobile + web + 3rd-party API

  1. Identity provider: Keycloak hoặc Auth0 — central
  2. Token format: JWT RS256 (asymmetric — service chỉ cần public key)
  3. JWKS endpoint: service tự fetch + cache public keys, support rotation
  4. Mobile/Web: Authorization Code + PKCE flow → access token (15min) + refresh (7d, single-use rotation)
  5. Service-to-service: Client Credentials flow
  6. 3rd-party: API key qua gateway (rate limit + scope)
  7. Revoke: blacklist short-lived (Redis TTL = remaining), refresh DB delete
  8. Audit: log mọi auth event (no token content) + correlate qua traceId
