#07 - Spring Transaction: Isolation, Propagation, Rollback Rules

Transaction là boundary của dữ liệu đúng. Trong phỏng vấn, người ta không hỏi @Transactional để xem bạn nhớ annotation; họ hỏi để biết bạn có hiểu commit/rollback, lock, proxy, exception và side effect hay không.

Trọng tâm
Data correctness
Bẫy chính
Checked exception, self-invocation
Liên quan
#5 AOP, #9 JPA, #13 Outbox
Ví dụ
E-commerce checkout

Transaction Là Unit Of Work

Khi người dùng checkout, hệ thống phải tạo order, giữ hàng tồn kho, ghi payment intent, và phát event cho service giao hàng. Nếu chỉ một phần thành công, dữ liệu sẽ lệch: order tồn tại nhưng stock chưa giữ, hoặc stock bị trừ nhưng order fail.

Spring giúp quản lý transaction bằng AOP proxy. Nhưng proxy chỉ giải quyết boundary trong cùng application và cùng resource manager. Khi chạm sang service khác, message broker, email, external API, bạn đã rời khỏi local transaction và cần pattern khác như outbox, saga hoặc retry.

Kiến Trúc Học Chống "Hiểu Giả"

1

Vẽ boundary

Luôn chỉ ra transaction bắt đầu ở method nào, commit/rollback ở đâu.

2

Vẽ timeline

Propagation chỉ rõ TX nào suspend, TX nào join, TX nào commit trước.

3

Phân loại exception

Runtime rollback mặc định, checked thì không, trừ khi khai báo.

4

Tách side effect

Email, Kafka, HTTP call không nằm trong DB transaction. Đừng giả vờ chúng rollback cùng DB.

1. Bài Toán Thực Tế - Checkout Tạo Order Nhưng Inventory Bị Lệch

Một sàn bán vé workshop gặp lỗi: khách checkout fail ở bước thanh toán, nhưng ghế vẫn bị giữ. Trường hợp khác, order đã tạo nhưng event gửi fulfillment mất, khiến kho không chuẩn bị hàng.

Sai: transaction dài và side effect lẫn lộn
@Service
public class CheckoutService {
    @Transactional
    public OrderResult checkout(CheckoutRequest request) throws PaymentDeclinedException {
        Order order = orderRepository.save(Order.from(request));
        inventoryClient.reserve(request.items()); // HTTP call ngoài DB TX
        paymentGateway.charge(request.payment()); // external side effect
        kafkaTemplate.send("order-created", order.id()); // có thể publish trước commit
        return OrderResult.success(order.id());
    }
}
Đúng hướng: DB change + outbox cùng transaction
@Service
public class CheckoutService {
    @Transactional(rollbackFor = Exception.class, timeout = 10)
    public OrderResult checkout(CheckoutRequest request) {
        Order order = orderRepository.save(Order.pending(request));
        stockReservationRepository.saveAll(order.reservations());
        outboxRepository.save(OutboxEvent.orderCreated(order));
        return OrderResult.accepted(order.id());
    }
}
Tư duy đúng

Transaction nên bao quanh phần dữ liệu cần atomic trong một database. External side effect nên chạy sau commit hoặc qua outbox. Nếu gửi message trước commit, consumer có thể đọc order chưa commit hoặc nhận event cho order đã rollback.

2.2 Isolation Levels

Isolation là mức "tường ngăn" giữa các transaction. Mức càng mạnh càng ít anomaly nhưng càng tốn lock/throughput.

READ_UNCOMMITTEDGần như khôngDirty readHiếm dùng trong app business.
READ_COMMITTEDDirty readNon-repeatable, phantomQuery thông thường, default của nhiều DB.
REPEATABLE_READDirty + non-repeatablePhantom tùy DBRead-modify-write cần ổn định hơn.
SERIALIZABLEGần serial executionThroughput giảm, retry nhiềuReport hoặc invariant rất nghiêm.
Lost update trong inventory: kết quả cuối cùng stock=0 nhưng có 2 order được xác nhận.

2.3 Propagation

Propagation trả lời câu hỏi: method B được gọi khi method A đang có transaction thì B join hay tạo transaction riêng?

PropagationHành viDùng khi
REQUIREDJoin TX hiện có, chưa có thì tạo mớiDefault cho service method.
REQUIRES_NEWSuspend TX cũ, tạo TX mới độc lậpOperation log, retry marker, audit kỹ thuật phải lưu riêng.
NESTEDSavepoint trong TX hiện tạiBatch import: lỗi 1 item rollback item đó.
MANDATORYBắt buộc caller đã có TXEnforce method chỉ được gọi trong unit-of-work.
SUPPORTSCó TX thì join, không có thì chạy thườngRead method linh hoạt.
NOT_SUPPORTEDSuspend TX, chạy ngoài TXGenerate file/report nặng không cần lock.
REQUIRED: checkout TX_1 begin -> reserve() joins TX_1 -> commit/rollback cùng nhau
REQUIRES_NEW: checkout TX_1 suspend -> operationLog TX_2 begin -> TX_2 commit -> TX_1 resume

2.4 Rollback Rules

Bẫy phỏng vấn kinh điển

Spring rollback mặc định với RuntimeException và Error. Checked exception không rollback nếu bạn không khai báo rollbackFor.

@Transactional
public void reserveSeat(String showId, String seatNo) throws SeatUnavailableException {
    seatRepository.markHeld(showId, seatNo);
    throw new SeatUnavailableException(seatNo); // checked exception
    // Nếu SeatUnavailableException extends Exception, TX có thể commit!
}

Fix bằng cách dùng domain exception extends RuntimeException hoặc khai báo @Transactional(rollbackFor = SeatUnavailableException.class).

2.5 readOnly, Flush Và Dirty Checking

@Transactional(readOnly = true) không phải "cấm tuyệt đối mọi write" ở mọi database. Ý nghĩa thực dụng là nói với Spring/Hibernate rằng flow này chỉ đọc: giảm dirty checking/flush không cần thiết và giúp driver/database tối ưu nếu hỗ trợ. Đừng dựa vào nó như security rule.

Khái niệmNghĩaBẫy
Dirty checkingHibernate so entity managed với snapshot để update khi flushLoad nhiều entity chỉ để đọc vẫn có cost.
FlushĐẩy SQL pending xuống DB trước commit hoặc trước query cần consistencyFlush không đồng nghĩa commit.
CommitDB xác nhận transaction thành côngException sau flush vẫn rollback được nếu TX chưa commit.
readOnly=trueHint cho transaction/providerKhông thay thế phân quyền write.

Với read path lớn như dashboard/report, ưu tiên DTO projection hoặc native query phù hợp thay vì load hàng nghìn entity managed rồi để Hibernate dirty-check.

3. Production-Grade Implementation

3.1 Checkout Boundary Gọn Và Rõ

@Service
public class CheckoutService {
    private final OrderRepository orderRepository;
    private final StockReservationRepository reservationRepository;
    private final OutboxRepository outboxRepository;

    @Transactional(
        isolation = Isolation.READ_COMMITTED,
        rollbackFor = Exception.class,
        timeout = 10
    )
    public OrderResult checkout(CheckoutRequest request) {
        Order order = Order.pending(request.customerId(), request.items());
        orderRepository.save(order);

        List<StockReservation> reservations = request.items().stream()
            .map(item -> StockReservation.hold(order.id(), item.sku(), item.quantity()))
            .toList();
        reservationRepository.saveAll(reservations);

        outboxRepository.save(OutboxEvent.orderCreated(order.id()));
        return OrderResult.accepted(order.id());
    }
}

3.2 Operation Log Với REQUIRES_NEW

@Service
public class OperationLogService {
    private final OperationLogRepository repository;

    @Transactional(propagation = Propagation.REQUIRES_NEW, timeout = 3)
    public void record(String action, String entityId, String status) {
        repository.save(OperationLog.of(action, entityId, status, Instant.now()));
    }
}

Chỉ dùng REQUIRES_NEW khi log này thật sự phải commit độc lập. Nếu log là một phần của business consistency, hãy để nó join transaction chính.

3.3 Publish Event Sau Commit

@Service
public class OrderApplicationService {
    private final ApplicationEventPublisher events;

    @Transactional
    public void confirmOrder(String orderId) {
        Order order = orderRepository.findById(orderId).orElseThrow();
        order.confirm();
        events.publishEvent(new OrderConfirmedEvent(orderId));
    }
}

@Component
public class OrderEventPublisher {
    private final KafkaTemplate<String, Object> kafkaTemplate;

    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void onOrderConfirmed(OrderConfirmedEvent event) {
        kafkaTemplate.send("order-confirmed", event.orderId(), event);
    }
}

3.4 Batch Import Với NESTED

@Service
public class CatalogImportService {
    private final ProductRowProcessor rowProcessor;

    @Transactional
    public ImportSummary importRows(List<ProductRow> rows) {
        ImportSummary summary = new ImportSummary();
        for (ProductRow row : rows) {
            try {
                rowProcessor.process(row);
                summary.success(row.sku());
            } catch (Exception ex) {
                summary.failed(row.sku(), ex.getMessage());
            }
        }
        return summary;
    }
}

@Service
public class ProductRowProcessor {
    @Transactional(propagation = Propagation.NESTED)
    public void process(ProductRow row) {
        productRepository.save(Product.from(row));
    }
}

3.5 TransactionTemplate: Khi Annotation Không Đủ Rõ

@Transactional tốt cho boundary cố định ở method public. TransactionTemplate hữu ích khi bạn cần transaction nhỏ nằm trong một method lớn hơn, cần bắt lỗi quanh block cụ thể, hoặc muốn code thể hiện rõ commit boundary.

@Service
public class ImportCoordinator {
    private final TransactionTemplate transactionTemplate;
    private final ProductRepository productRepository;

    public ImportResult importOne(ProductRow row) {
        return transactionTemplate.execute(status -> {
            try {
                Product product = Product.from(row);
                productRepository.save(product);
                return ImportResult.success(product.id());
            } catch (ValidationException ex) {
                status.setRollbackOnly();
                return ImportResult.rejected(row.sku(), ex.getMessage());
            }
        });
    }
}

Không dùng TransactionTemplate để né hiểu biết về propagation. Nó chỉ là cách biểu diễn boundary khác, vẫn chịu cùng rule về connection, flush, isolation và rollback.

3.6 After Commit Và Outbox: Đừng Publish Event Mồ Côi

@TransactionalEventListener(AFTER_COMMIT) phù hợp khi publish event trong cùng JVM sau commit và lỗi publish có thể được xử lý riêng. Với message broker cần retry bền vững, outbox table an toàn hơn: ghi event vào DB trong transaction, worker sau commit đọc outbox và publish.

CáchDùng khiRủi ro
@TransactionalEventListenerSide effect nhẹ, cùng process, không cần retry bềnProcess chết sau commit trước khi publish.
Outbox tableEvent/message phải publish eventuallyCần worker, retry, idempotency, cleanup.
Publish trong transactionHầu như tránhMessage có thể ra ngoài dù DB rollback.

3.7 @Async + @Transactional: Thread Khác, Context Khác

Transaction context gắn với thread. Nếu một method @Transactional gọi @Async, phần async chạy ở thread khác và không tự tham gia transaction của caller. Đây là bẫy daily work khi gửi notification, export report hoặc gọi provider sau khi lưu DB.

Sai: tưởng async join TX caller
@Transactional
public void confirm(String orderId) {
    order.confirm();
    notificationService.sendAsync(orderId); // thread khác
}
Đúng hơn: sau commit hoặc outbox
@Transactional
public void confirm(String orderId) {
    order.confirm();
    outboxRepository.save(OutboxEvent.orderConfirmed(orderId));
}

5. Trade-offs & Anti-Patterns

AP1

Long Transaction

Không gọi external API, generate file lớn hoặc sleep trong transaction. Transaction dài giữ connection, giữ lock, tăng timeout và deadlock.

AP2

REQUIRES_NEW Vô Tội Vạ

Mỗi REQUIRES_NEW có thể cần connection riêng và phá atomicity với caller. Dùng khi dữ liệu phải commit độc lập, không dùng như cách "chắc ăn".

AP3

Publish Message Trong Transaction

Message broker không rollback cùng DB. Dùng outbox hoặc event listener sau commit để tránh event mồ côi.

6. Interview Framework

SurfaceREQUIRED khác REQUIRES_NEW?

REQUIRED join transaction hiện có hoặc tạo mới nếu chưa có. REQUIRES_NEW suspend transaction hiện có và tạo transaction độc lập. Commit/rollback của TX mới không phụ thuộc TX caller.

DeepKhi nào @Transactional không rollback?

Khi exception là checked và không khai báo rollbackFor; khi method không đi qua proxy; khi annotation đặt trên private/final method; khi exception bị catch rồi không rethrow; hoặc khi object không phải Spring bean.

ArchitectureCheckout + message broker cần consistency thế nào?

Local DB transaction chỉ bao quanh order/reservation/outbox row. Sau commit, outbox publisher gửi message với retry. Consumer xử lý idempotent. Nếu cần rollback cross-service, dùng saga và compensating action thay vì cố dùng local @Transactional.

Checklist Thành Thạo

  • Vẽ được boundary transaction cho một flow checkout.
  • Giải thích được 4 isolation levels bằng anomaly: dirty read, non-repeatable read, phantom, lost update.
  • Vẽ timeline REQUIRED vs REQUIRES_NEW, gồm suspend/resume.
  • Biết checked exception không rollback mặc định và cách fix.
  • Biết vì sao message/email/API call không nên nằm trực tiếp trong DB transaction.
  • Phân biệt flush và commit, và biết readOnly=true là optimization hint chứ không phải security rule.
  • Dùng được TransactionTemplate khi cần boundary transaction nhỏ và rõ trong code.
  • Nhận ra bẫy @Async chạy khác thread nên không giữ transaction context của caller.
