📖 5 quy tắc đọc file này — KHÔNG đọc một mạch

  1. 1 pattern / 0.5–1 tuần. Đọc Mental Model + Template, sau đó solve 5–7 bài drill. KHÔNG đọc cả file trong 1 ngày — overload + illusion of competence.
  2. Tay code trước, mắt đọc sau. Mỗi template trong file, tự viết lại 1 lần KHÔNG nhìn. Nếu bí, đối chiếu rồi xoá và viết lại lần 2.
  3. Đo bằng pattern recognition test. Mỗi cuối tuần, lấy 10 bài Medium random từ LeetCode → đoán pattern trong 60s, không cần solve. ≥ 80% accuracy là đạt.
  4. Skip phần CORE/SUPP/ADV theo level hiện tại. Lần đầu chỉ làm CORE (12 patterns), vòng 2 thêm SUPP (4 patterns), vòng 3 thêm ADV (4 patterns).
  5. Mỗi pattern phải pass 5 tiêu chí hiểu sâu trước khi qua pattern tiếp theo: (1) vấn đề gốc, (2) tại sao pattern này ra đời, (3) không có pattern → bug gì, (4) ví dụ sai trông thế nào, (5) edge case nguy hiểm nhất.

⏸️ Pause & Predict

Trước khi đọc tiếp: nếu chỉ học 1 trong 20 patterns, bạn nghĩ pattern nào "cover" nhiều bài nhất? Viết câu trả lời và ước lượng % bài LeetCode mà pattern đó áp dụng được. Đáp án ở cuối phần này.

Tại sao "20 patterns" không phải con số tuỳ tiện?

Mỗi bài LC = 1 combination của: cấu trúc dữ liệu + thao tác + ràng buộc time/space. Khi liệt kê đầy đủ, tổ hợp có thể có hàng triệu — nhưng không gian "câu hỏi hay" mà interviewer hỏi rất hẹp. Lý do:

  • Interviewer chỉ có 45 phút → bài phải codable trong 25–30 phút. Loại bỏ các bài đòi hỏi thuật toán quá đặc thù (Suffix Tree, FFT, Network Flow).
  • Bài phải có signal phân biệt được senior vs junior. Bài quá dễ (1 vòng for) không signal được; bài quá khó (research-level) không fair. Sweet spot = các pattern có ít nhất 1 trick non-obvious.
  • Bài phải có variants để hỏi follow-up. Pattern "Sliding Window" có thể biến thành 50+ bài khác nhau bằng cách đổi constraint → interviewer thích vì có chiều sâu probe.
Đáp án Pause & Predict: Pattern phổ biến nhất là Hashing (~25–30% bài LC Easy/Medium dùng HashMap/HashSet ở dạng nào đó). Tuy nhiên, "cover" theo nghĩa chỉ cần pattern này để solve thì Two Pointers + Sliding Window mới là cặp đôi cover nhiều bài đơn lẻ nhất — vì chúng thay thế brute force O(n²) bằng O(n) cho rất nhiều bài array/string.

2. Vấn đề thực: 3000 bài → 20 mental models CORE

❌ Hiểu lầm phổ biến của junior

"Tôi cần solve 500 bài LC để được FAANG." — Sai. Nhiều người solve 1000+ bài mà vẫn fail vì họ nhớ solution thuộc lòng chứ không internalize pattern. Ngược lại, có người chỉ solve 150 bài (NeetCode 150) nhưng pass Google L4 vì họ deeply hiểu 20 patterns và transfer được sang bài lạ.

Số bài KHÔNG quan trọng bằng chất lượng pattern recognition.

Tại sao "pattern thinking" ra đời?

Trước 2015, interview prep là "đọc CLRS + làm hết Algorithms (Sedgewick) + thuộc thuật toán". Cách này quá nặng — bạn học algorithm (Dijkstra, KMP, A*) nhưng không học cách nhận diện khi nào dùng. Pattern thinking lật ngược: dạy bạn signal phrase trong đề bài → trigger template phù hợp.

❌ Học theo "thuật toán"

  • Đọc Dijkstra trên Wikipedia, hiểu pseudocode
  • Implement Dijkstra trong môi trường vô trùng
  • Gặp bài LC 743 (Network Delay) → không nhận ra là Dijkstra
  • Bí 30 phút, đọc solution, "à thì ra"
  • 2 tuần sau gặp bài variant → lại bí

✅ Học theo "pattern"

  • Học signal: "đồ thị có trọng số + tìm đường ngắn nhất từ A đến tất cả" → Dijkstra
  • Học template: PriorityQueue<int[]>, dist[] array, while loop pop min
  • Solve 5 bài Dijkstra variants liên tiếp trong 1 tuần
  • Tháng sau gặp LC 787 (Cheapest Flights) → "có K stops constraint, đây là Dijkstra modified" — nhận ra trong 30 giây

🔨 Thử Phá Vỡ: "20 patterns" có lỗ hổng nào?

  • Bài Hard ngoài 20 patterns: ~5–10% bài Hard yêu cầu kiến thức ngoài rổ này — Segment Tree, KMP, Manacher, Suffix Array, Maximum Flow, Bitmask DP. Nếu target Google L5+/Meta E5+, cần học thêm.

  • Pattern dạy "tay nghề chiến thuật" nhưng không dạy "tư duy hệ thống": Bài Hard thật sự (LC 4 Median of Two Sorted Arrays, LC 1until8 Trapping Rain Water II) đòi insight non-pattern. 20 patterns chỉ là sàn cơ bản.

3. UMPIRE — khung tư duy gọi pattern ra CORE

UMPIRE là khung phỏng vấn 6 bước (đã chi tiết trong #43 Mental Model). Ở đây tóm tắt để bạn quy chiếu mỗi pattern:

U — Understand   : Lặp lại đề, hỏi clarify (input range, edge case, output format)
M — Match        : MAP đề về pattern (← đây là phần file này dạy)
P — Plan         : Pseudocode, complexity dự kiến, sketch trên bảng
I — Implement    : Code (theo template pattern đã match)
R — Review       : Trace 1 example nhỏ + 1 edge case
E — Evaluate     : Verify time/space, đề xuất optimization

Mỗi pattern dưới đây có mục "Signal nhận pattern" — đây chính là bước M của UMPIRE.

Pattern 1 · Hash Map / Hash Set CORE

Vì sao đây là CORE: Hashing xuất hiện trong ~25–30% bài LC. Junior fail phỏng vấn không phải vì không biết HashMap — mà vì không reflex "thấy duplicate / lookup / count → HashMap" trong 10 giây đầu.

Signal nhận pattern (60s test)

  • "Find pair / triplet sum to X" → put complement vào map
  • "First non-repeating character" → count với map
  • "Group anagrams / by some key" → key → list
  • "Contains duplicate / distinct elements" → Set
  • "Top K frequent" → map count + heap

Mental Model — "Trade space for time"

Brute force của tất cả các bài này là O(n²): lặp 2 vòng, kiểm tra điều kiện. HashMap đổi O(n) space để giảm xuống O(n) time bằng cách memoize những element đã thấy.

Brute: for each i: for each j > i: check pair(i, j)        → O(n²)
Hash:  for each i: lookup complement in HashMap O(1)         → O(n)

Key insight: thay vì search 2 lần tuyến tính, lưu lần 1 + lookup O(1) lần 2.

Template Java

public int[] twoSum(int[] nums, int target) {
    Map<Integer, Integer> seen = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int complement = target - nums[i];
        if (seen.containsKey(complement)) {
            return new int[]{seen.get(complement), i};
        }
        seen.put(nums[i], i);
    }
    return new int[]{};
}

Walkthrough — LC 1 Two Sum (UMPIRE đầy đủ)

U: Input nums = [2,7,11,15], target = 9. Output indices. Có đúng 1 solution, không reuse cùng index 2 lần. Clarify: array có thể có duplicate? (Có) Negative? (Có) Sorted? (Không).

M: "find pair sum to X" → reflex Hash Map. Brute O(n²) → memoize complement.

P: Map<value, index>. Duyệt 1 lần. Mỗi element: hỏi map có target - nums[i] không, nếu có return; nếu không put vào map.

I: Như template trên.

R: Trace [2,7,11,15], target=9 → i=0 put(2,0); i=1 complement=9-7=2, map có → return [0,1]. ✓

E: Time O(n), space O(n). Không thể better — vì chỉ đọc 1 lần qua array đã đủ.

Variant — LC 49 Group Anagrams (Medium)

public List<List<String>> groupAnagrams(String[] strs) {
    Map<String, List<String>> groups = new HashMap<>();
    for (String s : strs) {
        char[] chars = s.toCharArray();
        Arrays.sort(chars);
        String key = new String(chars);              // sorted string = anagram fingerprint
        groups.computeIfAbsent(key, k -> new ArrayList<>()).add(s);
    }
    return new ArrayList<>(groups.values());
}
// O(n × k log k) where k = max string length
// Variant tối ưu: dùng int[26] làm key → O(n × k)

Variant — LC 347 Top K Frequent (Medium)

public int[] topKFrequent(int[] nums, int k) {
    Map<Integer, Integer> freq = new HashMap<>();
    for (int n : nums) freq.merge(n, 1, Integer::sum);

    PriorityQueue<int[]> heap = new PriorityQueue<>((a, b) -> a[1] - b[1]);  // min-heap
    for (var entry : freq.entrySet()) {
        heap.offer(new int[]{entry.getKey(), entry.getValue()});
        if (heap.size() > k) heap.poll();
    }
    int[] result = new int[k];
    for (int i = k - 1; i >= 0; i--) result[i] = heap.poll()[0];
    return result;
}
// O(n log k) — kết hợp Hashing + Heap

⚠️ Edge cases phải tự nhớ (junior hay quên)

  1. Array rỗng / 1 phần tử: Two Sum với 1 phần tử → return [] (không có pair).
  2. Duplicate giá trị, không duplicate index: [3,3], target=6 → return [0,1] (cùng giá trị, khác index — hợp lệ).
  3. Cùng phần tử dùng 2 lần: [3,2,4], target=6 → KHÔNG được dùng [0,0] dù 3+3=6. Vì sao template không bug? Vì put() chạy SAU containsKey() → khi i=0 chưa có gì trong map.
  4. Negative numbers: [-3,4,3,90], target=0 → [0,2]. HashMap handle negative tự nhiên — nhưng Bit operations thì không.
  5. Integer == Integer vs .equals(): Java cache Integer từ -128 đến 127. Integer a = 200; Integer b = 200; a == b → false. Khi compare value từ HashMap, LUÔN dùng .equals() hoặc unbox về int.
  6. HashMap key là custom object: Phải override equals() và hashCode() theo contract. Quên = collision/lookup fail. Test bằng cách put 2 object "logically equal" → assertEquals size = 1.
  7. Hash collision attack (DoS): Production server nhận untrusted input → attacker craft input gây collision → O(n) lookup degrade về O(n) (chain dài). Java 8+ chuyển bucket thành tree khi chain > 8 → giảm xuống O(log n). LC không test cái này nhưng phỏng vấn senior có hỏi.
  8. Concurrent modification: Duyệt HashMap đồng thời thêm/xoá → ConcurrentModificationException. Phỏng vấn: dùng Iterator.remove() hoặc collect vào list rồi xoá sau.
  9. HashMap chứa null key: Cho phép 1 null key (Hashtable thì không). Đề bài có giá trị null → cần handle riêng hoặc dùng sentinel.
  10. Memory blow-up khi key range lớn: Nếu n = 10⁹ → HashMap nuốt RAM. Khi đó cần switch sang Bloom filter / disk-based map.
Pitfall #1: Junior thường viết map.get(key) == 1 với key trả về Integer → bug khi value > 127. LUÔN dùng .equals() hoặc int v = map.get(key);.
Pitfall #2: Quên check containsKey trước khi get → NullPointerException khi auto-unbox Integer null về int.
Pitfall #3: Custom object làm key mà không override hashCode → mỗi instance hash khác nhau, lookup luôn fail.

Pattern 2 · Prefix Sum CORE

Vì sao CORE: Range query sum/count là 1 trong những thao tác phổ biến nhất (analytics, sliding-window-by-sum, subarray problems). Prefix sum chuyển query từ O(n) xuống O(1).

Signal nhận pattern

  • "Sum of subarray from i to j" — range sum query
  • "Subarray sum equals K" — đếm số subarray
  • "Number of subarrays with property X" — khi property = sum-based
  • "Equilibrium index" — chia mảng thành 2 nửa cùng sum

Mental Model

prefix[i] = sum of nums[0..i-1]
sum(i..j) = prefix[j+1] - prefix[i]   ← O(1) range query

Invariant quan trọng: prefix[0] = 0 (empty prefix — neutral element của phép cộng)

Template

int[] prefix = new int[n + 1];
for (int i = 0; i < n; i++) prefix[i + 1] = prefix[i] + nums[i];

// Range sum [l..r] (inclusive) = prefix[r+1] - prefix[l]

Walkthrough — LC 560 Subarray Sum Equals K (Medium)

Insight then-and-now: Brute O(n²): duyệt mọi (i, j), tính sum, check = k. Cải thiện: dùng prefix → vẫn O(n²). Insight thật: nếu prefix[j] - prefix[i] = k thì prefix[i] = prefix[j] - k → trong khi duyệt, hỏi HashMap đã thấy prefix - k bao nhiêu lần.

public int subarraySum(int[] nums, int k) {
    Map<Integer, Integer> prefixCount = new HashMap<>();
    prefixCount.put(0, 1);  // ⚠️ empty prefix — base case

    int prefix = 0, count = 0;
    for (int n : nums) {
        prefix += n;
        count += prefixCount.getOrDefault(prefix - k, 0);
        prefixCount.merge(prefix, 1, Integer::sum);
    }
    return count;
}
// O(n) time, O(n) space — combo Prefix Sum + Hashing

Variant — LC 238 Product Except Self

Prefix * Suffix pattern. Không được dùng phép chia.

public int[] productExceptSelf(int[] nums) {
    int n = nums.length;
    int[] result = new int[n];
    result[0] = 1;
    for (int i = 1; i < n; i++) result[i] = result[i-1] * nums[i-1];  // left product

    int right = 1;
    for (int i = n - 1; i >= 0; i--) {
        result[i] *= right;
        right *= nums[i];
    }
    return result;
}
// O(n), O(1) extra space (output array không tính)

⚠️ Edge cases Prefix Sum

  1. Quên prefix[0] = 0: với LC 560, không put(0, 1) → subarray bắt đầu từ index 0 không được đếm. Bug silent.
  2. Integer overflow: n = 10⁵, mỗi nums[i] = 10⁹ → tổng vượt Integer.MAX_VALUE. Dùng long.
  3. Off-by-one trong index: Sum [l..r] inclusive = prefix[r+1] - prefix[l] (KHÔNG phải prefix[r] - prefix[l-1] khi l=0).
  4. 2D prefix sum: sum(r1, c1, r2, c2) = ps[r2+1][c2+1] - ps[r1][c2+1] - ps[r2+1][c1] + ps[r1][c1]. Phải vẽ ra giấy 1 lần để nhớ inclusion-exclusion.
  5. Negative numbers + sum: Khi sum có thể giảm, không thể dùng two-pointer sliding window — phải dùng prefix + map.
  6. Modifiable array: Nếu array thay đổi liên tục, prefix sum O(n) update — chuyển sang Fenwick Tree (BIT) O(log n).
  7. Bài "count subarrays divisible by K": Variant prefix sum modulo K. Map count of prefix % K.

Pattern 3 · Two Pointers CORE

Vì sao CORE: Cùng với Hashing, Two Pointers là cặp đôi giảm O(n²) → O(n) phổ biến nhất. Mảng sorted + tìm pair/triplet → reflex 2-pointer.

Signal nhận pattern

  • "Sorted array" + "find pair" hoặc "find triplet"
  • "Palindrome check"
  • "Remove duplicates in-place" (in O(1) extra space)
  • "Reverse string/array"
  • "Container with most water" / "trapping rain water"

Mental Model — 3 variants

1. Opposite ends:  left=0, right=n-1, di chuyển vào giữa
                    → Two Sum II, 3Sum, palindrome, container
2. Fast & slow:    cùng start, khác speed
                    → LinkedList cycle, remove duplicates
3. Same direction: 2 pointer maintain window/group
                    → Sliding window (sẽ tách thành P4 riêng)

Template — Opposite Ends

int left = 0, right = nums.length - 1;
while (left < right) {
    if (condition_met) {
        // record / return
    }
    if (need_smaller_sum) right--;
    else left++;
}

Walkthrough — LC 15 3Sum (Medium, classic FAANG)

Insight: Cho mảng nums, tìm tất cả triplet (a, b, c) sao cho a + b + c = 0, không trùng lặp. Brute O(n³). Cải thiện: sort + fix 1 + two-pointer cho 2 cái còn lại = O(n²).

public List<List<Integer>> threeSum(int[] nums) {
    Arrays.sort(nums);  // ⚠️ phải sort trước
    List<List<Integer>> result = new ArrayList<>();

    for (int i = 0; i < nums.length - 2; i++) {
        if (i > 0 && nums[i] == nums[i - 1]) continue;  // skip duplicate
        if (nums[i] > 0) break;                          // optimization: positive → no sum 0

        int left = i + 1, right = nums.length - 1;
        while (left < right) {
            int sum = nums[i] + nums[left] + nums[right];
            if (sum == 0) {
                result.add(List.of(nums[i], nums[left], nums[right]));
                // skip duplicates trong 2 vế còn lại
                while (left < right && nums[left] == nums[left + 1]) left++;
                while (left < right && nums[right] == nums[right - 1]) right--;
                left++; right--;
            } else if (sum < 0) left++;
            else right--;
        }
    }
    return result;
}
// O(n²) time, O(1) extra space (output không tính, sort in-place)

Variant — LC 11 Container With Most Water

public int maxArea(int[] height) {
    int left = 0, right = height.length - 1, max = 0;
    while (left < right) {
        int area = Math.min(height[left], height[right]) * (right - left);
        max = Math.max(max, area);
        if (height[left] < height[right]) left++;   // move pointer của bên thấp hơn
        else right--;
    }
    return max;
}
// Vì sao luôn move bên thấp? — Vì giữ bên thấp + thu hẹp width chỉ làm area giảm.
// Bên thấp giới hạn area → chỉ có cơ hội tăng khi bỏ bên thấp đi.

⚠️ Edge cases Two Pointers

  1. Quên skip duplicate trong 3Sum: Output sẽ có triplet trùng. Test với [0,0,0,0] → kỳ vọng 1 triplet [0,0,0], lỗi sẽ ra 4.
  2. left < right vs left <= right: Phụ thuộc bài. Two Sum II dùng < (vì cần 2 index khác nhau). Palindrome dùng < (gặp giữa thì thôi).
  3. Mảng chưa sorted: 2-pointer kiểu opposite-ends chỉ work khi sorted. Nếu không, sort trước (O(n log n)) hoặc dùng HashMap.
  4. Move sai pointer: Cần sum nhỏ hơn → move left (tăng giá trị nhỏ). Junior hay nhầm move right.
  5. Triplet không trùng nhưng giá trị trùng: [-2,0,1,1,2] → [-2,0,2] và [-2,1,1] đều unique. Đừng over-skip.
  6. Integer overflow trong sum: nums[i] + nums[left] + nums[right] với giá trị gần Integer.MAX_VALUE → overflow. Dùng long nếu range cho phép.
  7. Fast/slow pointer null check: LinkedList cycle → fast.next.next mà fast.next == null → NullPointerException. Phải check fast != null && fast.next != null trong while.
  8. "Remove duplicates in-place" trả về length: Output là k đầu tiên, phần còn lại tuỳ ý. Đề không test phần đuôi — đừng confuse.

Pattern 4 · Sliding Window CORE

Vì sao CORE: "Longest/shortest substring/subarray với property X" là họ bài kinh điển của Meta/Google. Sliding window là biến thể nâng cao của Two Pointers — junior phải có template reflex.

Signal nhận pattern

  • "Longest substring/subarray with property X"
  • "Maximum sum of subarray size k" (fixed window)
  • "Minimum window substring containing T" (variable window)
  • "Number of substrings with K distinct characters"
  • "Smallest subarray sum >= target"

Mental Model

Maintain window [left, right] thoả mãn constraint.
- Loop: expand right
- Khi vi phạm constraint: shrink left
- Track best ở mỗi step (hoặc cuối)

2 variants:
- Fixed size: window luôn = k
- Variable size: expand/shrink dynamic theo constraint

Template — Variable Window

int left = 0, result = 0;
WindowState state = ...;  // HashMap, count[], sum, ...

for (int right = 0; right < n; right++) {
    state.add(arr[right]);                 // expand

    while (state.violatesConstraint()) {
        state.remove(arr[left]);           // shrink
        left++;
    }

    result = Math.max(result, right - left + 1);  // record valid window
}

Walkthrough — LC 3 Longest Substring Without Repeating

public int lengthOfLongestSubstring(String s) {
    Map<Character, Integer> lastIndex = new HashMap<>();
    int left = 0, maxLen = 0;

    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        // Nếu ký tự đã xuất hiện trong window hiện tại → jump left
        if (lastIndex.containsKey(c) && lastIndex.get(c) >= left) {
            left = lastIndex.get(c) + 1;
        }
        lastIndex.put(c, right);
        maxLen = Math.max(maxLen, right - left + 1);
    }
    return maxLen;
}
// O(n) — mỗi ký tự visit tối đa 2 lần (1 lần expand, 1 lần shrink)

Walkthrough — LC 76 Minimum Window Substring (Hard, FAANG favorite)

public String minWindow(String s, String t) {
    if (s.length() < t.length()) return "";

    int[] need = new int[128];                       // ASCII full range
    for (char c : t.toCharArray()) need[c]++;
    int required = t.length();                       // tổng số ký tự (kể cả duplicate) cần match

    int left = 0, minLen = Integer.MAX_VALUE, minStart = 0;

    for (int right = 0; right < s.length(); right++) {
        // Nếu ký tự này còn cần (need[c] > 0) → giảm required
        if (need[s.charAt(right)]-- > 0) required--;

        while (required == 0) {                      // window đủ
            if (right - left + 1 < minLen) {
                minLen = right - left + 1;
                minStart = left;
            }
            // Shrink: nếu ký tự bị bỏ ra là "cần" → required++
            if (need[s.charAt(left)]++ == 0) required++;
            left++;
        }
    }
    return minLen == Integer.MAX_VALUE ? "" : s.substring(minStart, minStart + minLen);
}
// O(n + m), m = len(t)

Variant — LC 424 Longest Repeating Character Replacement

public int characterReplacement(String s, int k) {
    int[] count = new int[26];
    int left = 0, maxCount = 0, maxLen = 0;
    for (int right = 0; right < s.length(); right++) {
        maxCount = Math.max(maxCount, ++count[s.charAt(right) - 'A']);
        // Window valid khi (windowSize - maxCount) <= k
        if (right - left + 1 - maxCount > k) {
            count[s.charAt(left) - 'A']--;
            left++;
        }
        maxLen = Math.max(maxLen, right - left + 1);
    }
    return maxLen;
}

⚠️ Edge cases Sliding Window

  1. Update result trong khi shrink: Sai vị trí thì result sai. Convention: record SAU khi shrink xong, hoặc trong while loop khi window đã valid (như LC 76).
  2. Quên cập nhật state khi shrink: Trong LC 76, quên need[leftChar]++ → required đếm sai → infinite loop hoặc kết quả lệch.
  3. Char counter dùng int[26] vs HashMap: a-z chỉ → int[26] nhanh hơn nhiều (cache locality). Unicode → HashMap.
  4. Window có thể trống: "subarray sum >= target" với target = 0 → trả về 1 hay 0? Đọc kỹ đề.
  5. Fixed window vs variable window: Fixed (size = k) thì shrink khi window > k; variable thì shrink khi violate constraint. Đừng nhầm.
  6. Negative numbers + sum: Sliding window classic KHÔNG work khi có negative — vì shrink left không đảm bảo sum giảm. Phải chuyển sang prefix sum + map.
  7. Empty string / target dài hơn source: Return "" cho LC 76. Check early.
  8. String contains duplicate trong target: LC 76 với t = "aabc" cần đếm 2 'a'. need[] array tự handle, nhưng nếu dùng Set thì sai.

Pattern 5 · Stack (incl. Monotonic Stack) CORE

Vì sao CORE: Stack có 2 use case: (1) parens/expression — kinh điển, (2) monotonic stack — trick mạnh giảm O(n²) → O(n) cho bài "next greater/smaller". Senior interview rất hay test monotonic.

Signal nhận pattern

  • "Valid parenthesis / matching brackets" → Stack
  • "Next greater/smaller element" → Monotonic Stack
  • "Daily temperatures" / "stock span" → Monotonic Stack
  • "Largest rectangle in histogram" → Monotonic Stack
  • "Evaluate RPN expression" → Stack
  • "Implement Min Stack / Max Stack" → Stack auxiliary

Mental Model — Monotonic Stack

Stack giữ tính chất monotonic (increasing OR decreasing).
Khi push element vi phạm → pop những phần tử ra, mỗi pop là cơ hội tính answer.
Mỗi element được push/pop đúng 1 lần → tổng O(n).

Template — Next Greater Element

int[] result = new int[n];
Arrays.fill(result, -1);
Deque<Integer> stack = new ArrayDeque<>();  // ⚠️ ArrayDeque, KHÔNG dùng java.util.Stack

for (int i = 0; i < n; i++) {
    while (!stack.isEmpty() && nums[stack.peek()] < nums[i]) {
        result[stack.pop()] = nums[i];
    }
    stack.push(i);
}
// Phần tử còn trong stack: không có next greater → giữ -1

Walkthrough — LC 739 Daily Temperatures

Cho mảng nhiệt độ temps, trả về result[i] = số ngày phải đợi để có nhiệt độ ấm hơn ngày i. Brute O(n²). Monotonic stack giảm về O(n).

public int[] dailyTemperatures(int[] temps) {
    int n = temps.length;
    int[] result = new int[n];
    Deque<Integer> stack = new ArrayDeque<>();  // lưu index

    for (int i = 0; i < n; i++) {
        while (!stack.isEmpty() && temps[stack.peek()] < temps[i]) {
            int prev = stack.pop();
            result[prev] = i - prev;             // số ngày từ prev đến i
        }
        stack.push(i);
    }
    return result;
}

Walkthrough — LC 84 Largest Rectangle in Histogram (Hard)

Insight: Với mỗi bar i, tìm leftmost và rightmost bar có chiều cao >= height[i]. Width = right - left + 1. Brute O(n²). Monotonic increasing stack → O(n).

public int largestRectangleArea(int[] heights) {
    Deque<Integer> stack = new ArrayDeque<>();
    int maxArea = 0;

    for (int i = 0; i <= heights.length; i++) {
        int h = (i == heights.length) ? 0 : heights[i];  // sentinel: flush stack cuối
        while (!stack.isEmpty() && heights[stack.peek()] > h) {
            int height = heights[stack.pop()];
            int width = stack.isEmpty() ? i : i - stack.peek() - 1;
            maxArea = Math.max(maxArea, height * width);
        }
        stack.push(i);
    }
    return maxArea;
}

Variant — LC 20 Valid Parentheses (Easy gate)

public boolean isValid(String s) {
    Deque<Character> stack = new ArrayDeque<>();
    Map<Character, Character> pairs = Map.of(')', '(', ']', '[', '}', '{');
    for (char c : s.toCharArray()) {
        if (pairs.containsValue(c)) stack.push(c);
        else if (stack.isEmpty() || stack.pop() != pairs.get(c)) return false;
    }
    return stack.isEmpty();
}

⚠️ Edge cases Stack

  1. Dùng java.util.Stack (legacy synchronized): Chậm hơn ArrayDeque ~3x. Phỏng vấn dùng Stack → trừ điểm "Java knowledge".
  2. Push value thay vì index trong monotonic stack: Mất thông tin position → không tính được width/distance.
  3. Quên check empty trước peek/pop: NoSuchElementException. Luôn while (!stack.isEmpty() && ...).
  4. Quên sentinel cuối array trong LC 84: Bar cuối không được pop khỏi stack → miss kết quả. Trick: dùng i <= n + giá trị 0.
  5. ArrayDeque không cho null element: Push null → NullPointerException. Hashtable cũ cũng vậy.
  6. Comparator equality trap: stack.pop() != pairs.get(c) với Character > 127 → bug. Nhưng ASCII trong LC 20 an toàn vì < 127.
  7. Monotonic increasing vs decreasing: Next greater → decreasing stack (pop khi gặp lớn hơn). Largest rectangle → increasing stack (pop khi gặp nhỏ hơn). Junior hay nhầm.
  8. Stack overflow với recursion sâu: Đôi khi đề tương đương dùng stack thủ công thay vì recursion (DFS tree với input 10^5 depth).

Pattern 6 · Binary Search CORE

Vì sao CORE: Không chỉ search sorted array — binary search trên answer space (BS on answer) là kỹ thuật ưa thích của Google/Meta cho bài optimization có monotonic property. Bạn phải viết được template KHÔNG bug.

Signal nhận pattern

  • "Sorted array" + "find target" → BS cổ điển
  • "Rotated sorted array" → BS variant
  • "Find peak / valley" → BS trên slope
  • "Search 2D matrix" → BS với index flattening
  • "Min/max X to make Y feasible" → BS on answer (Koko bananas, ship packages)

Mental Model

Invariant: target nằm trong [left, right]
Mỗi iteration loại nửa space → O(log n)

Template chuẩn (KHÔNG bug):
  left = 0, right = n - 1
  while (left <= right):
      mid = left + (right - left) / 2   ⚠️ tránh overflow
      if found: return
      elif need go right: left = mid + 1
      else: right = mid - 1

Template — Find Exact Target

public int binarySearch(int[] arr, int target) {
    int left = 0, right = arr.length - 1;
    while (left <= right) {
        int mid = left + (right - left) / 2;     // tránh (l+r) overflow
        if (arr[mid] == target) return mid;
        if (arr[mid] < target) left = mid + 1;
        else right = mid - 1;
    }
    return -1;
}

Template — Find Leftmost (lower_bound)

// Trả về index nhỏ nhất sao cho arr[i] >= target
public int lowerBound(int[] arr, int target) {
    int left = 0, right = arr.length;            // ⚠️ right = n (không phải n-1)
    while (left < right) {                       // ⚠️ <, không phải <=
        int mid = left + (right - left) / 2;
        if (arr[mid] < target) left = mid + 1;
        else right = mid;                        // ⚠️ mid, không phải mid-1
    }
    return left;
}

Walkthrough — LC 33 Search in Rotated Sorted Array

public int search(int[] nums, int target) {
    int left = 0, right = nums.length - 1;
    while (left <= right) {
        int mid = left + (right - left) / 2;
        if (nums[mid] == target) return mid;

        if (nums[left] <= nums[mid]) {           // left half sorted
            if (target >= nums[left] && target < nums[mid]) right = mid - 1;
            else left = mid + 1;
        } else {                                  // right half sorted
            if (target > nums[mid] && target <= nums[right]) left = mid + 1;
            else right = mid - 1;
        }
    }
    return -1;
}

Walkthrough — LC 875 Koko Eating Bananas (BS on answer)

Insight: Tìm tốc độ ăn k nhỏ nhất để ăn xong piles trong h giờ. Answer space = [1, max(piles)]. Hàm canFinish(k) monotonic (k tăng → có thể finish) → BS trên k.

public int minEatingSpeed(int[] piles, int h) {
    int left = 1, right = Arrays.stream(piles).max().getAsInt();
    while (left < right) {
        int mid = left + (right - left) / 2;
        long hours = 0;
        for (int p : piles) hours += (p + mid - 1) / mid;  // ceil(p / mid)
        if (hours <= h) right = mid;             // có thể nhỏ hơn
        else left = mid + 1;
    }
    return left;
}
// O(n log max(piles))

⚠️ Edge cases Binary Search

  1. Overflow (left + right) / 2: n ~ 2×10⁹ → left+right tràn int. Dùng left + (right - left) / 2.
  2. < vs <=: Exact target template dùng <= + right = mid - 1. Lower bound dùng < + right = mid. Nhầm = bug off-by-one hoặc infinite loop.
  3. Infinite loop khi left = mid: Nếu mid = left (khi right - left = 1), không tiến → loop vô tận. Fix: mid = left + (right - left + 1) / 2 (round up) khi cần.
  4. Array có duplicate: LC 81 (Search Rotated II) — worst case O(n) vì không thể quyết định half nào sorted khi nums[left] == nums[mid].
  5. BS on answer cần monotonic: Phải prove hàm feasibility(x) monotonic. Nếu không, BS sai.
  6. Long vs int trong sum: Koko bananas: hours phải là long vì tổng có thể vượt int (10⁹ * 10⁴ piles).
  7. Empty array: Return -1 / 0 / -1 — đọc đề. Check arr.length == 0 trước khi vào loop.
  8. Search 2D matrix flat index: mid_row = mid / cols; mid_col = mid % cols. Junior hay nhầm mid % rows.
  9. Find peak (LC 162): Compare arr[mid] với arr[mid+1], không phải neighbor cả 2 bên — vì invariant peak luôn tồn tại trong half cao hơn.
  10. Median of 2 sorted arrays (LC 4 Hard): BS trên partition index, không phải value. Edge case empty array / phân chia lệch.

Pattern 7 · Linked List (Fast & Slow Pointer) CORE

Vì sao CORE: Linked list là gateway để interviewer test pointer manipulation. Reverse, cycle, middle là 3 reflex bắt buộc — fail = signal "không hiểu pointer".

Signal nhận pattern

  • "Detect cycle in linked list" → Floyd's tortoise & hare
  • "Find middle" → fast/slow
  • "Reverse linked list" → 3-pointer (prev, curr, next)
  • "Merge two sorted lists" → dummy + iteration
  • "Remove Nth from end" → fast pointer headstart n
  • "Palindrome linked list" → find middle + reverse half

Mental Model

Fast & slow (Floyd):
- slow đi 1 step, fast đi 2 step
- Cycle: fast bắt kịp slow trong vòng
- Middle: fast đến cuối → slow ở middle

Reverse: 3 pointer (prev, curr, next), invariant: prev = đã reverse phần đầu

Template — Reverse Linked List

public ListNode reverse(ListNode head) {
    ListNode prev = null, curr = head;
    while (curr != null) {
        ListNode next = curr.next;       // lưu next TRƯỚC khi đổi
        curr.next = prev;                // đảo pointer
        prev = curr;                     // dịch prev
        curr = next;                     // dịch curr
    }
    return prev;
}

Template — Detect Cycle (Floyd)

public boolean hasCycle(ListNode head) {
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
        if (slow == fast) return true;
    }
    return false;
}
// Vì sao work? Nếu có cycle, fast catch up slow với rate 1 step/iteration trong vòng.

Walkthrough — LC 19 Remove Nth From End

Trick: Dummy node + fast headstart n. Khi fast đến cuối, slow ở node TRƯỚC node cần xoá → slow.next = slow.next.next.

public ListNode removeNthFromEnd(ListNode head, int n) {
    ListNode dummy = new ListNode(0, head);   // ⚠️ dummy để handle xoá head
    ListNode fast = dummy, slow = dummy;
    for (int i = 0; i < n; i++) fast = fast.next;
    while (fast.next != null) { fast = fast.next; slow = slow.next; }
    slow.next = slow.next.next;
    return dummy.next;
}

Variant — LC 21 Merge Two Sorted Lists

public ListNode mergeTwoLists(ListNode l1, ListNode l2) {
    ListNode dummy = new ListNode(0), tail = dummy;
    while (l1 != null && l2 != null) {
        if (l1.val <= l2.val) { tail.next = l1; l1 = l1.next; }
        else { tail.next = l2; l2 = l2.next; }
        tail = tail.next;
    }
    tail.next = (l1 != null) ? l1 : l2;
    return dummy.next;
}

⚠️ Edge cases Linked List

  1. Quên dummy node: Khi modify head (xoá, insert ở đầu), không dummy → phải code 2 branch (head/non-head). Dummy = bug-free.
  2. NullPointerException fast.next.next: Phải check fast != null && fast.next != null trong while.
  3. Mất reference head: Modify head pointer trong loop → mất gốc. Luôn lưu ListNode result = head trước.
  4. Even vs odd length khi find middle: Fast/slow: even (2,4) → slow = node 3 (second middle); odd (1,3,5) → slow = node 3. Test cả 2.
  5. Cycle detect — find entry point (LC 142): Sau khi gặp, reset slow về head, cả 2 đi 1 step → gặp tại entry. Phải prove được (math: 2(a+b) = a+b+kc).

Walkthrough — LC 124 Binary Tree Max Path Sum (Hard)

Mind-bender: Path có thể đi QUA node (left → node → right) HOẶC chỉ đi xuống 1 nhánh. Tách 2 khái niệm:

  • through = path đi qua node (có thể là answer)
  • return value = path TỪ node ĐI XUỐNG 1 nhánh (để parent dùng)
class Solution {
    private int maxSum = Integer.MIN_VALUE;
    public int maxPathSum(TreeNode root) { maxGain(root); return maxSum; }

    private int maxGain(TreeNode node) {
        if (node == null) return 0;

        // Chỉ lấy gain dương (negative thì bỏ — không lấy)
        int leftGain = Math.max(maxGain(node.left), 0);
        int rightGain = Math.max(maxGain(node.right), 0);

        // Path THROUGH node (candidate answer)
        int through = node.val + leftGain + rightGain;
        maxSum = Math.max(maxSum, through);

        // Return: path FROM node đi xuống 1 nhánh
        return node.val + Math.max(leftGain, rightGain);
    }
}

public List<Integer> rightSideView(TreeNode root) {
    List<Integer> result = new ArrayList<>();
    if (root == null) return result;
    Queue<TreeNode> queue = new ArrayDeque<>();
    queue.offer(root);
    while (!queue.isEmpty()) {
        int size = queue.size();
        for (int i = 0; i < size; i++) {
            TreeNode n = queue.poll();
            if (i == size - 1) result.add(n.val);   // last in level = rightmost

            if (n.right != null) queue.offer(n.right);
        }
    }
    return result;

class Trie {

    private boolean isEnd;

    public void insert(String word) {
        Trie node = this;
        for (char c : word.toCharArray()) {
            int i = c - 'a';
            if (node.children[i] == null) node.children[i] = new Trie();
            node = node.children[i];

        node.isEnd = true;
    }

        Trie node = find(word);
        return node != null && node.isEnd;
    }

        return find(prefix) != null;
    }

    private Trie find(String s) {
        Trie node = this;
        for (char c : s.toCharArray()) {
            int i = c - 'a';
            if (node.children[i] == null) return null;
            node = node.children[i];
        }
        return node;
    }
}

Insight 2 heaps: small half (max-heap) + big half (min-heap), kích thước chênh tối đa 1. Median = root của heap lớn hơn, hoặc trung bình 2 root nếu bằng nhau.

class MedianFinder {
    PriorityQueue<Integer> small = new PriorityQueue<>(Collections.reverseOrder()); // max
    PriorityQueue<Integer> large = new PriorityQueue<>();                              // min

    public void addNum(int num) {
        small.offer(num);
        large.offer(small.poll());                       // balance: số lớn chuyển sang large
        if (large.size() > small.size()) small.offer(large.poll());
    }

    public double findMedian() {
        return small.size() > large.size() ? small.peek() : (small.peek() + large.peek()) / 2.0;
    }
}
// addNum O(log n), findMedian O(1)

Walkthrough — LC 23 Merge K Sorted Lists (Hard)

public ListNode mergeKLists(ListNode[] lists) {
    PriorityQueue<ListNode> heap = new PriorityQueue<>((a, b) -> a.val - b.val);
    for (ListNode l : lists) if (l != null) heap.offer(l);

    ListNode dummy = new ListNode(0), tail = dummy;
    while (!heap.isEmpty()) {
        ListNode min = heap.poll();
        tail.next = min;
        tail = tail.next;
        if (min.next != null) heap.offer(min.next);
    }
    return dummy.next;
}
// O(N log k), N = total nodes, k = số list

  1. PriorityQueue default min-heap: Junior hay nhầm là max. Cần max → Collections.reverseOrder() hoặc comparator (a,b)->b-a.
  2. Comparator overflow: (a, b) -> a - b với Integer.MAX_VALUE → tràn → sort sai. Dùng Integer.compare(a, b).
  3. Heap không sorted khi iterate: Chỉ root đảm bảo nhỏ nhất. for (Integer n : heap) trả về thứ tự internal — KHÔNG sorted.
  4. Mỗi op O(log n): Không phải O(1). Tổng n ops = O(n log n), không phải O(n).
  5. Remove arbitrary element: O(n) vì phải linear search. Cần update/delete by key → dùng TreeMap hoặc indexed heap.
  6. Stable vs unstable: PriorityQueue không stable (cùng priority không giữ insert order). Top K frequent words (LC 692) cần tie-break lexicographic.
  7. Heap of objects với mutable field: Modify field sau khi offer → heap invariant phá vỡ. Re-offer hoặc dùng immutable.
  8. "Lazy deletion" trong outdated entries: Dijkstra/LC 1834 — pop ra entry cũ thì skip thay vì update.
  9. Memory: heap size = n khi không bounded: Top-K phải bound size = k, không offer hết rồi pop k lần (O(n log n)).

private void backtrack(List<Integer> path, int[] nums, int start, List<List<Integer>> result) {
    result.add(new ArrayList<>(path));     // ⚠️ COPY, không add path trực tiếp
    for (int i = start; i < nums.length; i++) {
        path.add(nums[i]);                  // choose
        backtrack(path, nums, i + 1, result);
        path.remove(path.size() - 1);       // unchoose
    }
}

private void backtrack(List<Integer> path, int[] nums, boolean[] used, List<List<Integer>> result) {

    for (int i = 0; i < nums.length; i++) {

        used[i] = true; path.add(nums[i]);

public boolean exist(char[][] board, String word) {
    for (int i = 0; i < board.length; i++)
        for (int j = 0; j < board[0].length; j++)
            if (dfs(board, word, i, j, 0)) return true;
    return false;
}
private boolean dfs(char[][] board, String word, int i, int j, int idx) {
    if (idx == word.length()) return true;
    if (i < 0 || i >= board.length || j < 0 || j >= board[0].length
        || board[i][j] != word.charAt(idx)) return false;

    char tmp = board[i][j];
    board[i][j] = '#';     // mark visited (in-place — tiết kiệm space)
    boolean found = dfs(board, word, i+1, j, idx+1) || dfs(board, word, i-1, j, idx+1)
                 || dfs(board, word, i, j+1, idx+1) || dfs(board, word, i, j-1, idx+1);
    board[i][j] = tmp;     // backtrack
    return found;
}

Walkthrough — LC 127 Word Ladder (BFS shortest path)

public int ladderLength(String beginWord, String endWord, List<String> wordList) {
    Set<String> dict = new HashSet<>(wordList);
    if (!dict.contains(endWord)) return 0;

    Queue<String> queue = new ArrayDeque<>();
    queue.offer(beginWord);
    int level = 1;

    while (!queue.isEmpty()) {
        int size = queue.size();
        for (int i = 0; i < size; i++) {
            String word = queue.poll();
            if (word.equals(endWord)) return level;

            char[] chars = word.toCharArray();
            for (int j = 0; j < chars.length; j++) {
                char orig = chars[j];
                for (char c = 'a'; c <= 'z'; c++) {
                    chars[j] = c;
                    String next = new String(chars);
                    if (dict.remove(next)) queue.offer(next);   // remove = mark visited
                }
                chars[j] = orig;
            }
        }
        level++;
    }
    return 0;
}

public int dijkstra(int n, int[][] edges, int start, int end) {
    Map<Integer, List<int[]>> graph = new HashMap<>();

        graph.computeIfAbsent(e[0], k -> new ArrayList<>()).add(new int[]{e[1], e[2]});

    PriorityQueue<int[]> heap = new PriorityQueue<>((a, b) -> a[1] - b[1]);
    int[] dist = new int[n];
    Arrays.fill(dist, Integer.MAX_VALUE);
    dist[start] = 0;
    heap.offer(new int[]{start, 0});

    while (!heap.isEmpty()) {
        int[] curr = heap.poll();
        int node = curr[0], d = curr[1];
        if (d > dist[node]) continue;   // lazy skip outdated entry
        for (int[] next : graph.getOrDefault(node, List.of())) {
            int nd = d + next[1];
            if (nd < dist[next[0]]) {
                dist[next[0]] = nd;
                heap.offer(new int[]{next[0], nd});
            }
        }
    }
    return dist[end] == Integer.MAX_VALUE ? -1 : dist[end];
}
// O((V + E) log V)

  • "Redundant connection / valid tree"

Template Union-Find (với path compression + union by rank)

class UnionFind {
    int[] parent, rank;
    int components;

    UnionFind(int n) {
        parent = new int[n]; rank = new int[n]; components = n;
        for (int i = 0; i < n; i++) parent[i] = i;
    }

    int find(int x) {
        if (parent[x] != x) parent[x] = find(parent[x]);   // path compression
        return parent[x];
    }

    boolean union(int x, int y) {
        int px = find(x), py = find(y);
        if (px == py) return false;                         // đã chung component
        if (rank[px] < rank[py]) parent[px] = py;           // union by rank
        else if (rank[px] > rank[py]) parent[py] = px;
        else { parent[py] = px; rank[px]++; }
        components--;
        return true;
    }
}
// Mỗi op gần O(1) — α(n) inverse Ackermann

⚠️ Edge cases Advanced Graph

  1. Dijkstra fail với negative weights: Dùng Bellman-Ford O(VE) hoặc SPFA.
  2. Dijkstra cần lazy skip: Heap có thể chứa entry cũ (đã updated). Skip nếu d > dist[node].
  3. Cheapest Flight với K stops (LC 787): Dijkstra modify hoặc Bellman-Ford bounded — careful với pruning logic.
  4. Union-Find phải path compression + union by rank: Thiếu = degrade về O(n) per op.
  5. Find recursive overflow: n=10⁵ skewed → stack overflow. Iterative find với 2 pass.
  6. Bellman-Ford detect negative cycle: Sau n-1 relaxations, nếu vẫn relax được → có negative cycle.
  7. Floyd-Warshall all-pairs shortest path: O(V³). LC 1334 — không phổ biến nhưng nên biết.
  8. MST (Minimum Spanning Tree): Kruskal (sort edges + Union-Find) hoặc Prim (Dijkstra-like). LC 1135.

Signal

  • "Number of ways to..."

Mental Model

dp[i] = answer cho first i elements (hoặc ending at i)
Recurrence: dp[i] dùng dp[i-1], dp[i-2], ...

Top-down (memoization): recursion + cache
Bottom-up (tabulation): iterative fill array → thường hiệu quả hơn

Walkthrough — LC 70 Climbing Stairs (gateway)

Insight: dp[n] = số cách lên n bậc. Bậc cuối từ bậc n-1 (bước 1) hoặc bậc n-2 (bước 2) → dp[n] = dp[n-1] + dp[n-2] (Fibonacci!).

public int climbStairs(int n) {
    if (n <= 2) return n;
    int prev2 = 1, prev1 = 2;
    for (int i = 3; i <= n; i++) {
        int curr = prev1 + prev2;
        prev2 = prev1; prev1 = curr;
    }
    return prev1;
}
// O(n) time, O(1) space

Walkthrough — LC 198 House Robber

public int rob(int[] nums) {
    int prev2 = 0, prev1 = 0;
    for (int n : nums) {
        int curr = Math.max(prev1, prev2 + n);   // rob hoặc skip
        prev2 = prev1; prev1 = curr;
    }
    return prev1;
}

Walkthrough — LC 300 Longest Increasing Subsequence

// O(n²) DP straightforward
public int lengthOfLIS(int[] nums) {
    int n = nums.length;
    int[] dp = new int[n];
    Arrays.fill(dp, 1);
    int max = 1;
    for (int i = 1; i < n; i++) {
        for (int j = 0; j < i; j++)
            if (nums[j] < nums[i]) dp[i] = Math.max(dp[i], dp[j] + 1);
        max = Math.max(max, dp[i]);
    }
    return max;
}

// O(n log n) — patience sort trick
public int lengthOfLISOptimal(int[] nums) {
    List<Integer> tails = new ArrayList<>();
    for (int n : nums) {
        int idx = Collections.binarySearch(tails, n);
        if (idx < 0) idx = -(idx + 1);
        if (idx == tails.size()) tails.add(n);
        else tails.set(idx, n);
    }
    return tails.size();
}

⚠️ Edge cases 1D DP

  1. Quên base case: dp[0] = ? dp[1] = ? Sai = toàn bộ recurrence sai.
  2. Off-by-one giữa array index và DP index: dp[0] có thể mean "0 phần tử" hoặc "phần tử thứ 0". Phải định nghĩa rõ.

  4. Memoize sai key: Object key không override equals/hashCode → cache miss.
  5. Coin Change unbounded: Outer loop coin, inner loop amount → mỗi coin dùng nhiều lần. Đảo lại → 0/1 knapsack (mỗi coin 1 lần).
  6. Word Break false positive: Boolean DP dp[i] = true nếu s[0..i] break được. Phải check tất cả dp[j] (j<i) + s[j..i] in dict.
  7. House Robber II (LC 213 — circular): Split thành 2 cases — rob first / rob last. Max của 2.
  8. Decode Ways (LC 91): "06" không decode được — phải check 2-digit leading 0.

Vì sao CORE: LCS, Edit Distance, Knapsack là 3 archetype Hard nhất nhưng được hỏi vô số lần. Pass = signal mạnh "biết DP". Senior phải biết space optimize O(mn) → O(min(m,n)).

Walkthrough — LC 1143 Longest Common Subsequence

Recurrence: dp[i][j] = LCS của text1[0..i-1] và text2[0..j-1]. Nếu char giống → dp[i-1][j-1]+1. Nếu khác → max(dp[i-1][j], dp[i][j-1]).

public int longestCommonSubsequence(String text1, String text2) {
    int m = text1.length(), n = text2.length();
    int[][] dp = new int[m + 1][n + 1];
    for (int i = 1; i <= m; i++) {
        for (int j = 1; j <= n; j++) {
            if (text1.charAt(i-1) == text2.charAt(j-1)) dp[i][j] = dp[i-1][j-1] + 1;
            else dp[i][j] = Math.max(dp[i-1][j], dp[i][j-1]);
        }
    }
    return dp[m][n];
}
// O(mn) time, O(mn) space — có thể optimize về O(n) bằng 2 hàng

Walkthrough — LC 72 Edit Distance (Hard)

public int minDistance(String w1, String w2) {
    int m = w1.length(), n = w2.length();
    int[][] dp = new int[m + 1][n + 1];
    for (int i = 0; i <= m; i++) dp[i][0] = i;     // delete i chars
    for (int j = 0; j <= n; j++) dp[0][j] = j;     // insert j chars

    for (int i = 1; i <= m; i++) {
        for (int j = 1; j <= n; j++) {
            if (w1.charAt(i-1) == w2.charAt(j-1)) dp[i][j] = dp[i-1][j-1];
            else dp[i][j] = 1 + Math.min(dp[i-1][j-1],          // replace
                              Math.min(dp[i-1][j],              // delete
                                       dp[i][j-1]));            // insert
        }
    }
    return dp[m][n];
}

Walkthrough — LC 416 Partition Equal Subset Sum (0/1 knapsack)

public boolean canPartition(int[] nums) {
    int sum = Arrays.stream(nums).sum();
    if (sum % 2 != 0) return false;
    int target = sum / 2;

    boolean[] dp = new boolean[target + 1];
    dp[0] = true;
    for (int n : nums) {
        for (int j = target; j >= n; j--) {        // ⚠️ duyệt từ phải qua trái
            dp[j] |= dp[j - n];
        }
    }
    return dp[target];
}
// Optimized 1D từ 2D O(n × target) — chỉ giữ 1 hàng

⚠️ Edge cases 2D DP

  1. Quên base case dp[0][*] và dp[*][0]: Edit Distance phải initialize i và j base.
  2. Off-by-one giữa string index và DP index: dp[i][j] dùng text.charAt(i-1) — vì dp 1-based, char 0-based.
  3. Space optimization wrong direction: Knapsack 0/1 phải duyệt j từ phải qua trái → tránh dùng item 2 lần.
  4. LCS với reconstruction (output substring): Phải backtrack từ dp[m][n] về dp[0][0].
  5. Edit Distance 3 ops: Replace, delete, insert — đừng quên op nào.
  6. Knapsack unbounded: Cho phép dùng item nhiều lần → duyệt j từ TRÁI sang PHẢI.
  7. Min Path Sum (LC 64): dp[0][0] = grid[0][0], không phải 0.
  8. Dungeon Game (LC 174): DP từ ĐÍCH về BẮT ĐẦU vì health ràng buộc forward không tính được.
  9. Memoize với 2D key: Map<Pair, Integer> — Pair phải có hashCode/equals đúng.

public int canCompleteCircuit(int[] gas, int[] cost) {
    int total = 0, tank = 0, start = 0;
    for (int i = 0; i < gas.length; i++) {
        int diff = gas[i] - cost[i];
        total += diff; tank += diff;
        if (tank < 0) { start = i + 1; tank = 0; }   // restart từ i+1
    }
    return total < 0 ? -1 : start;
}
// Proof: nếu total >= 0, luôn có solution. Nếu fail tại i, restart từ i+1.

Walkthrough — LC 56 Merge Intervals

public int[][] merge(int[][] intervals) {
    Arrays.sort(intervals, (a, b) -> a[0] - b[0]);   // sort by start
    List<int[]> result = new ArrayList<>();
    int[] curr = intervals[0];
    result.add(curr);
    for (int i = 1; i < intervals.length; i++) {
        if (intervals[i][0] <= curr[1]) {            // overlap
            curr[1] = Math.max(curr[1], intervals[i][1]);
        } else {
            curr = intervals[i];
            result.add(curr);
        }
    }
    return result.toArray(new int[result.size()][]);
}

Walkthrough — LC 253 Meeting Rooms II (FAANG classic)

Insight: Sort theo start. Dùng min-heap lưu end times. Nếu meeting tiếp theo start >= earliest end → reuse room (pop). Else thêm room mới.

public int minMeetingRooms(int[][] intervals) {
    Arrays.sort(intervals, (a, b) -> a[0] - b[0]);
    PriorityQueue<Integer> heap = new PriorityQueue<>();
    for (int[] meet : intervals) {
        if (!heap.isEmpty() && heap.peek() <= meet[0]) heap.poll();   // room rảnh
        heap.offer(meet[1]);
    }
    return heap.size();
}
// O(n log n)

Walkthrough — LC 435 Non-overlapping Intervals (Greedy)

public int eraseOverlapIntervals(int[][] intervals) {
    Arrays.sort(intervals, (a, b) -> a[1] - b[1]);   // sort by END (greedy choice)
    int end = Integer.MIN_VALUE, count = 0;
    for (int[] i : intervals) {
        if (i[0] >= end) end = i[1];
        else count++;
    }
    return count;
}

Useful tricks

// GCD (Euclidean)
int gcd(int a, int b) { return b == 0 ? a : gcd(b, a % b); }

// LCM
int lcm(int a, int b) { return a / gcd(a, b) * b; }   // a/gcd TRƯỚC để tránh overflow

// Fast exponentiation với modulo
long pow(long base, long exp, long mod) {
    long result = 1;
    base %= mod;
    while (exp > 0) {
        if ((exp & 1) == 1) result = result * base % mod;
        base = base * base % mod;
        exp >>= 1;
    }
    return result;
}

// Prime sieve (Eratosthenes)
boolean[] sieve(int n) {
    boolean[] composite = new boolean[n + 1];
    for (int i = 2; i * i <= n; i++)
        if (!composite[i])
            for (int j = i * i; j <= n; j += i) composite[j] = true;
    return composite;
}

Walkthrough — LC 50 Pow(x, n)

public double myPow(double x, int n) {
    long N = n;                                     // ⚠️ long vì -Integer.MIN_VALUE overflow
    if (N < 0) { x = 1 / x; N = -N; }
    double result = 1;
    while (N > 0) {
        if ((N & 1) == 1) result *= x;
        x *= x;
        N >>= 1;
    }
    return result;
}
// O(log n)

  3. Negative modulo trong Java: -7 % 3 = -1 (không phải 2). Cần ((a % m) + m) % m để dương.

⚠️ Anti-patterns junior hay rơi vào

Đây là 10 lỗi tư duy (không phải lỗi cú pháp) mà junior lặp lại khi học pattern. Đọc và self-check.

❌ Anti-pattern 1: "Solve càng nhiều bài càng tốt"

Người học solve 500 bài/tháng nhưng pattern recognition không tăng. Lý do: solve nhanh → không có thời gian internalize → nhớ solution thuộc lòng. Đến bài variant lại bí.

Fix: Sau mỗi bài, viết 3 dòng note "pattern + signal + key insight". Nếu bí 2 tuần sau làm lại bài đó → bạn CHƯA internalize.

❌ Anti-pattern 2: "Đọc solution trước khi try"

Đọc solution → não nhận diện "tôi hiểu" (illusion of competence) → không tự generate được. Phải bí ít nhất 30 phút trước khi đọc.

❌ Anti-pattern 3: "Optimize trước khi correct"

Cố viết solution O(n) ngay từ đầu → bí, miss edge case, miss complexity. Phỏng vấn 45 phút → fail.

Fix: Viết brute force CHẠY ĐÚNG trước, communicate "đây là brute O(n²), tôi sẽ optimize". Interviewer nhiều khi chấp nhận brute nếu bạn analyse được bottleneck.

❌ Anti-pattern 4: "Bỏ qua edge case vì 'phỏng vấn không test'"

Empty, single element, all duplicates — đây là 3 case interviewer check signal "engineering rigor". Bỏ qua = signal junior.

Fix: Mỗi bài, BẮT BUỘC nói ra mental list edge cases trước khi code. Section "Edge cases" mỗi pattern trên là material.

❌ Anti-pattern 5: "Im lặng khi code"

Interviewer mất sight tư duy → đánh giá "không biết". Khi bí cũng im → interviewer không cứu được.

Fix: Verbalize liên tục: "Tôi nghĩ approach này..." "Tôi đang băn khoăn liệu có thể..." "Nếu input là X thì Y có hold không?". Im < 30s.

❌ Anti-pattern 6: "Memorize template mà không hiểu invariant"

Học thuộc binary search template nhưng không hiểu vì sao left + (right - left) / 2, vì sao <= vs <. Đến variant lệch nhẹ → fail.

❌ Anti-pattern 7: "Skip complexity analysis"

Code chạy ra answer → submit → AC → next bài. Nhưng phỏng vấn LUÔN hỏi "what's time complexity?". Không nói được = -1 round.

Fix: Sau mỗi bài, ép mình viết Big-O time + space. So sánh với optimal. Hiểu tại sao có thể (hoặc không thể) better.

❌ Anti-pattern 8: "Không revise pattern cũ"

Tuần 1 học Hashing, tuần 5 vô Tree DFS — quên hết Hashing. Đến phỏng vấn random bài Hashing → bí.

Fix: Spaced repetition. Mỗi pattern, 2-3 bài review sau 1 tuần, 1 bài sau 1 tháng. Dùng Anki hoặc spreadsheet track.

❌ Anti-pattern 9: "Solve mà không trace example"

Viết code xong submit luôn. AC thì OK, WA thì panic. Phỏng vấn — interviewer hỏi "trace với input [2,7,11,15]" → bí.

Fix: Bước R trong UMPIRE — trace 1 example nhỏ + 1 edge case. Viết state ra bảng. Vẽ trace giúp catch off-by-one trước khi submit.

❌ Anti-pattern 10: "Phỏng vấn = recall = pattern matching"

Suy nghĩ "tôi chỉ cần match đề về pattern" → khi gặp bài novel (kết hợp 2 pattern, hoặc twist) → block.

Fix: Pattern là scaffold, không phải solution. Sau khi match pattern, vẫn cần adapt. Bài hay là bài kết hợp 2-3 pattern (VD: Trie + DFS, Heap + Sliding Window). Practice loại bài này.

🔧 Build & drill — 3 exercises để biến hiểu thành phản xạ

Đọc 20 patterns không đủ — phải practice with deliberate friction. 3 exercise dưới đây là chìa khoá biến "hiểu" thành "reflex".

🔨 Exercise 1 — Pattern Recognition Drill (1 giờ/tuần)

Mục tiêu: Đoán đúng pattern trong < 60s cho ít nhất 80% bài random.

Cách làm:

  1. Vào leetcode.com/problemset/, filter Medium + Frequency High.
  2. Mỗi ngày random 5 bài. KHÔNG đọc constraints / examples kỹ.
  3. Đọc đề trong 60s → guess pattern + viết key insight (1-2 dòng).
  4. Sau đó đọc solution để verify. Đúng thì ✓, sai thì note xuống vì sao miss.
  5. Track tỉ lệ: 10/15/20 bài. Goal: ≥ 80% sau 4 tuần.

Acceptance criteria:

  • ≥ 80% accuracy với 20 bài random ở tuần 4.
  • Recognize được multi-pattern bài (VD: "đây là Trie + DFS" thay vì chỉ "Trie").
  • Note miss patterns ≤ 5 dòng — tự rút insight, không copy solution.

Time: 15-20 phút/ngày × 4 tuần = 7-10 giờ.

🔨 Exercise 2 — Template Reflex (2 tuần)

Mục tiêu: Viết được template từng pattern KHÔNG nhìn cheatsheet, ≤ 5 phút.

Cách làm:

  1. Lập spreadsheet 20 patterns × 3 cột: Template, Time-to-write (s), Bugs Count.
  2. Mỗi sáng, chọn 3 patterns random. Mở editor trống, viết template từ đầu.
  3. Time bản thân. Sau đó compile + chạy với test case nhỏ. Đếm bug.
  4. Goal: tất cả 20 patterns viết được trong ≤ 5 phút, 0 bug.
  5. Pattern nào > 5 phút hoặc > 1 bug → revise tuần kế.

Acceptance: 18/20 patterns đạt ≤ 5 phút + 0 bug. 2 patterns khó (Dijkstra, Edit Distance) có thể ≤ 10 phút.

Why it works: Phỏng vấn không cho thời gian "hồi tưởng template". Reflex giải phóng 80% bandwidth cho phần insight + edge cases.

🔨 Exercise 3 — End-to-end Mock (4 tuần cuối prep)

Mục tiêu: Solve 1 random Medium trong 25 phút end-to-end với UMPIRE, narrate liên tục.

Cách làm:

  1. Chuẩn bị: bảng trắng / blank doc, timer, camera ghi hình bản thân.
  2. Mỗi 2 ngày, chọn 1 random Medium chưa solve. Set timer 35 phút.
  3. Narrate ra MIỆNG (không phải gõ comment): U → M → P → I → R → E.
  4. Sau khi xong (hoặc hết giờ), xem lại video. Đếm: im > 30s, edge case miss, complexity không nói.
  5. Tuần 1: 8/10 bài hoàn thành trong 35 phút. Tuần 4: 9/10 bài hoàn thành trong 25 phút + 5 phút buffer.

Acceptance:

  • 9/10 mock đạt: code đúng + edge case nhận ra + complexity nói đúng.
  • Tổng im lặng < 10% thời gian.
  • Pattern recognition trong 60s đầu (verbalize "Tôi nghĩ đây là...").

Bonus: Mock với người thật trên Pramp.com (free) hoặc Interviewing.io (paid). Real human pressure khác camera 1 mình.

🎯 3-tier interview framework — pattern mastery rubric

Khi sẵn sàng phỏng vấn FAANG, bạn cần đạt cả 3 tier. Tier 1 là sàn, Tier 3 là signal cho L5+/E5+.

Tier 1 — Pattern Recognition (Junior signal)

Mục tiêu: Cho 20 problem random, đoán đúng pattern ≥ 16/20.

Cách test: Mỗi đề đọc trong 60s → trả lời "đây là pattern nào". Bonus: nêu 1-2 invariant chính.

Checklist: Cover toàn bộ 20 patterns với accuracy ≥ 80%.

Khi đạt: Pass được phone screen Easy/Medium. L3/L4 Google, E3 Meta.

Tier 2 — Template Reflex (Mid-level signal)

Mục tiêu: Viết được template từng pattern KHÔNG cheatsheet, modify cho variant trong 5 phút.

Cách test:

  • Cho random pattern → viết template, time-to-write ≤ 5 phút, 0 bug.
  • Cho variant (VD: "Two Sum nhưng allow trùng index") → adapt template trong 5 phút.
  • Recognize pattern trong 60s, plan trong 5 phút, code trong 15 phút.

Khi đạt: Pass được onsite Medium consistently. L4/L5 Google, E4 Meta, mid Amazon.

Tier 3 — End-to-End Solve (Senior signal)

Mục tiêu: Solve 1 random Medium trong 25 phút end-to-end với UMPIRE, narrate liên tục.

Cách test:

  • Random Medium chưa từng thấy → 25 phút end-to-end.
  • Narrate liên tục, im < 30 giây (mỗi lúc).

  • Complexity tự verify đúng + đề xuất optimization (nếu có).

Edge-Case Lab Bank — 5 lab buộc gặp bug typical CORE ★

3 exercise trên là build kỹ năng. 5 lab dưới là "trap" — bug bạn SẼ gặp lần đầu, đến lần thứ 5 sẽ thành phản xạ. Code trên IDE/LeetCode, KHÔNG nhìn solution trước.

🧪 Lab #1 — Two Sum với duplicate values (Hash Map trap)

Problem: Two Sum nhưng input có duplicate: nums = [3, 3], target = 6. Trả index của 2 element. Edge case: chỉ có 1 cặp, không có nhiều cặp.

Trap: Nếu put cả 2 vào map trước, key=3 sẽ bị overwrite, index thứ nhất mất. Phải check map trong cùng loop (one-pass), không phải two-pass.

Symptom của bug: Output là [1, 1] thay vì [0, 1]. Khi reproduce với [3, 2, 4], target=6 thì pass nhưng [3, 3] fail.

Bonus: Sau khi pass, modify để trả tất cả pair (Two Sum II), so sánh complexity.

🧪 Lab #2 — Sliding Window với "longest substring" có Unicode/emoji

Problem: Longest substring without repeating chars. Input là "abcabcbb" → 3. Edge: input "🙂🙂a" hoặc "café".

Trap: Java String.length() tính theo char (UTF-16 code unit), không phải code point. Emoji là surrogate pair (2 chars). Window slide bằng index sẽ tách emoji ra.

Symptom: Trả về kết quả sai cho input có emoji. Hoặc ArrayIndexOutOfBoundsException khi access char giữa surrogate pair.

Fix: Dùng codePointAt + iterate by code point, hoặc convert thành int[] code points trước.

Bài học: Real interview tại Apple/Meta có hỏi follow-up "what if input is Unicode?" — interviewer test edge case thinking.

🧪 Lab #3 — Binary Search với integer overflow

Problem: Search trong sorted array size 1.5 tỷ. nums.length = 1_500_000_000. Tìm target.

Trap: int mid = (left + right) / 2 overflow khi left + right > Integer.MAX_VALUE (~2.1 tỷ). Mid trở thành số âm, nums[mid] throw exception.

Symptom: ArrayIndexOutOfBoundsException: -123456 sau vài iteration. Không reproduce với small array.

Fix: int mid = left + (right - left) / 2. Luôn safe.

Bonus: Cùng bug xuất hiện trong merge sort, quick sort partition. Verify cả 3 cùng pattern.

🧪 Lab #4 — Tree DFS với recursion stack overflow

Problem: Validate BST cho tree skewed (mỗi node chỉ có left child). Height = N = 100_000.

Trap: Recursive DFS depth = N. Java default stack ~512KB → ~50K frames trước StackOverflowError. Test case lớn sẽ crash.

Symptom: StackOverflowError khi N > 50_000. Reproduce: build skewed tree bằng for(i=0;i<100000;i++) node = new TreeNode(i, node, null);

Fix: Convert sang iterative DFS dùng explicit Stack. Hoặc tăng stack -Xss4m nhưng đây là band-aid, không phải fix proper.

Bài học: Interview hỏi follow-up "what if tree depth 1M?" — test understanding của recursion limit.

🧪 Lab #5 — Dynamic Programming với negative/zero edge

Problem: Maximum subarray sum. Input có thể all negative [-3, -1, -2] → expect -1 (subarray phải có ≥1 element).

Trap: Init maxSum = 0 rồi maxSum = max(maxSum, currentSum) → trả về 0 cho all-negative input, sai. Phải init maxSum = Integer.MIN_VALUE hoặc nums[0].

Symptom: Test case [-1] trả 0 thay vì -1. [5, 4, -1, 7, 8] pass nhưng [-3, -1, -2] fail.

Fix: maxSum = nums[0]; currentSum = nums[0]; start từ index 1.

Pattern chung: DP base case sai là bug #1 trong DP problems. Always check edge: empty, single element, all negative, all same.

🎓 Tự test Feynman — giải thích cho người không biết code

Đóng file này và thử giải thích các câu sau cho 1 sinh viên năm 1 KHÔNG biết LeetCode:

  2. "Khi nào HashMap thay được Two Pointers?" — Khi mảng KHÔNG sorted. Two Pointers cần invariant ordering. HashMap trade space cho time, không cần sorted.
  3. "Vì sao Greedy không phải bài nào cũng work?" — Greedy cần exchange argument / proof. Coin Change [1,3,4] amount 6 → greedy chọn 4+1+1=3, optimal 3+3=2. Greedy fail vì local choice không lead global.
  4. "DP top-down vs bottom-up khác gì?" — Top-down = recursion + cache, lazy (chỉ compute state cần). Bottom-up = iterative fill, eager (compute hết). Bottom-up nhanh hơn (no recursion overhead) nhưng cần biết order.
  5. "Vì sao BFS dùng được cho shortest path unweighted nhưng KHÔNG cho weighted?" — BFS giả định mỗi edge = 1 step. Khi weighted, "level" không monotonic theo distance → cần Dijkstra với priority queue.
