Testing Strategy — Pyramid, Testcontainers, Contract, Non-Brittle

"80% coverage" là con số gây hiểu giả nguy hiểm nhất trong testing — nó đo dòng code chạy qua, không đo hành vi được khẳng định. File này phân biệt test tốt (test behavior, real DB, contract) với test giả an toàn (test implementation, H2, mock cả class under test), kèm code thật cho từng loại, trên domain ShopFlow — OrderService của sàn e-commerce.

🎯 Mục tiêu
Viết test không brittle, dùng Testcontainers, hiểu contract test
📅 Thời lượng
1 tuần · Sau #9 JPA, #7 Transaction
📚 Prereq
JUnit 5 · Mockito · Spring Boot test · Docker
🔥 Độ ưu tiên
🟠 Cao — senior sở hữu chất lượng, không chỉ feature

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Coverage 80% nghĩa là code được test kỹ, ít bug."
Thực tế: Coverage chỉ đo dòng code chạy qua, không đo assertion có ý nghĩa. Một test gọi method rồi không assert gì vẫn tăng coverage. P1 vẫn lọt vì coverage cao mà test toàn check implementation detail thay vì behavior.
Tưởng hiểu: "Integration test với H2 in-memory cho nhanh là đủ."
Thực tế: H2 có dialect SQL khác PostgreSQL (JSONB, window function, native query, ON CONFLICT, constraint behavior). Test pass với H2 nhưng prod dùng PostgreSQL → bug bị che tới khi lên production.
Tưởng hiểu: "Mock càng nhiều càng tốt vì test chạy nhanh và độc lập."
Thực tế: Mock cả class under test = test đang test cái mock, không test code thật → false confidence. Chỉ mock dependency (repository, external API), để class cần test chạy logic thật.

🗺️ Roadmap — nếu chỉ có X giờ thì học gì

Thứ tự ưu tiên
1 giờ (CORE ★): Test pyramid + tỉ lệ & lý do (2.1), Unit test isolation đúng — chỉ mock dependency (2.2, 3.1), test behavior không implementation (4). Đây là gốc để viết test không brittle.
+1 giờ (CORE ★): Integration test với Testcontainers vs H2 (2.3, 3.2), @SpringBootTest slices cho tốc độ (2.6, 3.3). Hỏi nhiều trong phỏng vấn senior.
+1 giờ (SUPPLEMENTARY): Contract testing Pact (2.4, 3.4), test data builder (2.5).
Khi cần (ADVANCED): property-based testing, mutation testing (đo chất lượng test), chaos testing, canary deploy.

1. Bài toán thực tế — 80% coverage vẫn lọt P1 CORE ★

ShopFlow có 80% test coverage nhưng bug P1 vẫn lên production. CI chạy test mất 45 phút, chặn merge. Thêm một feature làm vỡ 50 test cũ — test brittle, check implementation detail thay vì behavior. Contract giữa OrderService và NotificationService vỡ âm thầm (API đổi, consumer chưa cập nhật). Integration test dùng H2, nhưng dialect khác PostgreSQL nên bug bị che tới production.

⏸ Pause & Predict — trước khi đọc lời giải
Viết ra giấy:
  1. 80% coverage mà vẫn lọt P1 — coverage đo cái gì, KHÔNG đo cái gì? Một loại test nào làm coverage cao mà giá trị thấp?
  2. Vì sao "thêm 1 feature làm vỡ 50 test"? Đặc điểm chung của 50 test đó là gì (gợi ý: chúng check cái gì)?
  3. H2 pass nhưng PostgreSQL fail — cho 2 ví dụ cụ thể về khác biệt dialect khiến bug lọt.
Đáp án ở 2.1 (coverage ≠ quality), 4 (test implementation = brittle), 2.3 (H2 vs PostgreSQL: JSONB, native query, ON CONFLICT, constraint). Đừng đọc tiếp khi chưa trả lời câu 2.

5 câu Senior phải trả lời

  1. Vì sao test pyramid quan trọng? Tỉ lệ Unit:Integration:E2E?
  2. Vì sao Testcontainers (PostgreSQL) tốt hơn H2? Khác biệt thật?
  3. Contract testing (Pact) là gì? Chặn breaking change API thế nào?
  4. "Coverage tốt" nghĩa là gì? (không chỉ 80%)
  5. Viết test không brittle thế nào? (test behavior, không implementation)

2.1 Test pyramid + tỉ lệ CORE ★

        E2E (ít)        — toàn hệ thống, chậm nhất (~1 phút/test), brittle nhất, confidence cao nhất
     Integration (vừa)  — nhiều component, DB thật (Testcontainers), tốc độ trung bình (~10s)
   Unit (nhiều)         — 1 class, mock dependency, nhanh (~1s), deterministic, parallel

Tỉ lệ tham chiếu: 70% Unit : 20% Integration : 10% E2E

WHY pyramid:

  • Unit = feedback nhanh (<1s), CI rẻ (chạy song song), nên làm nhiều — bắt lỗi logic sớm.
  • Integration = bắt bug thật (SQL syntax, transaction semantics, constraint), tốc độ trung bình, làm vừa phải.
  • E2E = bắt lỗi workflow (feature A làm vỡ feature B), chậm, brittle → chỉ dành cho critical path.
🔧 Counter-example — inverted pyramid (kim tự tháp ngược)

Nếu 50% là E2E: CI mất 2 giờ, feedback chậm, không kham nổi nhiều E2E → ít test → bug lọt. E2E cũng brittle nhất (một thay đổi nhỏ ở UI/flow làm đỏ hàng loạt) và khó debug (lỗi ở đâu trong 8 service?). Quy tắc: đẩy assertion xuống tầng thấp nhất bắt được bug — đừng test một quy tắc tính giá bằng E2E khi unit test đủ.

2.2 Unit test — isolation đúng CORE ★

Test 1 class cô lập, mock mọi dependency (repository, external API, config). Nhanh, deterministic. KHÔNG test code của framework (đừng test Spring DI hay Hibernate — tin vendor đã test).

✅ Test logic của OrderService (behavior)

@Test
void appliesDiscountToOrderTotal() {
    var repo    = mock(OrderRepository.class);
    var pricing = mock(PricingService.class);
    when(pricing.discountFor(1000)).thenReturn(50);

    var service = new OrderService(repo, pricing);
    Order o = service.place(1000);

    assertEquals(950, o.netTotal());   // behavior: total sau giảm giá
    verify(repo).save(any());
}

❌ Test framework (vô nghĩa)

@SpringBootTest
@Test
void orderServiceIsAutowired() {
    // đang test Spring DI, không test logic OrderService
    assertNotNull(orderService);
}
Đừng chỉ test happy path

Test cả error path: "khi pricing fail thì repository.save KHÔNG được gọi" (verify(repo, never()).save(any())) — đảm bảo không lưu một phần. Đây là chỗ bug thật hay trốn. Code đầy đủ ở 3.1.

2.3 Integration test: Testcontainers vs H2 CORE ★

Integration test kiểm tra tương tác giữa component với DB thật. Testcontainers spin một PostgreSQL thật trong Docker → bắt được bug dialect mà H2 bỏ sót.

Khác biệtH2 (in-memory)PostgreSQL thật (prod)
Native query / dialectCú pháp khác → có thể pass nhầmĐúng cú pháp prod
JSONB, array, window fnHỗ trợ hạn chế/khácĐầy đủ
ON CONFLICT upsertCú pháp khác (MERGE)Như prod
Constraint / unique violationHành vi/exception khácĐúng DataIntegrityViolationException như prod

L1 — Biết tên

Biết "Testcontainers chạy DB thật trong Docker". Symptom hiểu giả: vẫn dùng H2 cho tốc độ, không nêu được khác biệt cụ thể.

L2 — Explain được

Nêu được H2 khác PostgreSQL ở: native query, JSONB, window function, ON CONFLICT, unique/foreign-key constraint behavior, kiểu dữ liệu. Giải thích vì sao "pass với H2 = false confidence".

L3 — Apply được

Thiết kế suite: dùng Testcontainers cho repository/integration, reuse container singleton để nhanh, tách stage CI (unit trước cho feedback, integration sau), chạy container song song để rút thời gian từ 45 → 10 phút.

2.4 Contract testing (Pact) SUPPLEMENTARY

Consumer (OrderService) kỳ vọng Provider (NotificationService) trả JSON cụ thể.

Consumer test: "Gọi /notify với payload X, tôi mong response Y"
   → Pact ghi lại kỳ vọng (contract file)
Provider test: "Tôi có thoả mãn contract này không?" (verify trên provider)
   → Cả hai pass = API an toàn
   → Provider đổi API phá contract → contract test FAIL sớm trong CI,
     KHÔNG cần chạy cả hai service cùng lúc
Giá trị

Contract test bắt breaking change ngay trong CI của từng service, trước khi tới integration environment. Đây là thứ ngăn "API đổi, consumer vỡ âm thầm" ở mục 1. Code đầy đủ ở 3.4.

2.5 Test data builder SUPPLEMENTARY

✅ Builder — đọc được, linh hoạt

Order o = anOrder()
    .withTotal(5000)
    .withDiscount(200)
    .forCustomer("C9")
    .build();

❌ Constructor 12 tham số — brittle

Order o = new Order(5000, 200, "C9", "C1",
    LocalDate.now(), "PENDING", null, false,
    true, 0, null, "SALE");  // khó đọc, sai thứ tự dễ
public class OrderBuilder {
    private String customer = "C123";
    private BigDecimal total = new BigDecimal("1000");
    private BigDecimal discount = new BigDecimal("50");
    private OrderStatus status = OrderStatus.COMPLETED;

    public static OrderBuilder anOrder() { return new OrderBuilder(); }
    public OrderBuilder withTotal(long t)    { this.total = BigDecimal.valueOf(t); return this; }
    public OrderBuilder withDiscount(long d) { this.discount = BigDecimal.valueOf(d); return this; }
    public OrderBuilder forCustomer(String c){ this.customer = c; return this; }
    public Order build() { return new Order(customer, total, discount, status); }
}

3.1 Unit test với Mockito — đầy đủ CORE ★

class OrderServiceTest {
    private OrderService service;
    private OrderRepository repository;
    private PricingService pricing;
    private InventoryService inventory;

    @BeforeEach
    void setUp() {
        repository = mock(OrderRepository.class);
        pricing    = mock(PricingService.class);
        inventory  = mock(InventoryService.class);
        service    = new OrderService(repository, pricing, inventory);
    }

    @Test
    void placesOrderAndSavesWithDiscount() {
        // Arrange
        when(inventory.reserve("SKU1", 2)).thenReturn(true);
        when(pricing.discountFor(1000)).thenReturn(50);
        // Act
        Order result = service.place("C1", "SKU1", 2, 1000);
        // Assert — behavior, không phải implementation
        assertEquals(950, result.netTotal());
        assertEquals(50, result.discount());
        verify(repository).save(argThat(o -> o.netTotal() == 950));
    }

    @Test
    void rejectsWhenStockUnavailable() {            // error path
        when(inventory.reserve("SKU1", 2)).thenReturn(false);
        assertThrows(OutOfStockException.class,
            () -> service.place("C1", "SKU1", 2, 1000));
        verify(repository, never()).save(any());     // CRITICAL: không lưu một phần
    }

    @Test
    void doesNotSaveWhenPricingFails() {            // error path
        when(inventory.reserve(any(), anyInt())).thenReturn(true);
        when(pricing.discountFor(anyLong())).thenThrow(new PricingServiceException());
        assertThrows(PricingServiceException.class,
            () -> service.place("C1", "SKU1", 2, 1000));
        verify(repository, never()).save(any());
    }
}

3.2 Integration test với Testcontainers CORE ★

@Testcontainers                                    // tự quản lifecycle container
@SpringBootTest
class OrderIntegrationTest {
    @Container
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:15")
        .withDatabaseName("shop_test");

    @DynamicPropertySource
    static void props(DynamicPropertyRegistry reg) {
        reg.add("spring.datasource.url", postgres::getJdbcUrl);
        reg.add("spring.datasource.username", postgres::getUsername);
        reg.add("spring.datasource.password", postgres::getPassword);
    }

    @Autowired OrderService service;
    @Autowired OrderRepository orderRepo;
    @Autowired AuditLogRepository auditRepo;
    @Autowired TestEntityManager em;

    @Test
    void placingOrderWritesAuditEntry() {          // test cả flow: service → repo → DB → audit
        Customer c = em.persistAndFlush(new Customer("C1"));
        Order o = service.place(c.getId(), "SKU1", 2, 1000);

        assertThat(orderRepo.findById(o.getId())).isPresent();
        var audits = auditRepo.findByOrderId(o.getId());
        assertThat(audits).hasSize(1);
        assertThat(audits.get(0).getAction()).isEqualTo("ORDER_CREATED");
    }

    @Test
    void rollsBackOnUniqueConstraintViolation() {   // H2 KHÔNG bắt đúng case này
        em.persistAndFlush(new Customer("DUP"));
        assertThrows(DataIntegrityViolationException.class,
            () -> em.persistAndFlush(new Customer("DUP")));   // unique constraint
        em.clear();
        assertThat(em.getEntityManager()
            .createQuery("SELECT c FROM Customer c WHERE c.code='DUP'", Customer.class)
            .getResultList()).hasSize(1);            // chỉ 1, bản trùng không persist
    }
}

3.3 @DataJpaTest · @WebMvcTest SUPPLEMENTARY

@DataJpaTest                                       // chỉ nạp JPA + DB (Testcontainers)
class OrderRepositoryTest {
    @Autowired TestEntityManager em;
    @Autowired OrderRepository repo;

    @Test
    void findsOrdersByCustomerAndDateRange() {
        Customer c = em.persistAndFlush(new Customer("C1"));
        em.persistAndFlush(new Order(c, 1000, LocalDate.of(2026, 3, 1)));
        em.persistAndFlush(new Order(c, 500,  LocalDate.of(2026, 4, 1)));

        var result = repo.findByCustomerAndDateBetween(
            c, LocalDate.of(2026, 3, 1), LocalDate.of(2026, 3, 31));

        assertThat(result).hasSize(1);
        assertThat(result.get(0).getTotal()).isEqualByComparingTo("1000");
    }
}

@WebMvcTest(OrderController.class)                 // chỉ controller + MockMvc, service mock
class OrderControllerTest {
    @Autowired MockMvc mvc;
    @Autowired ObjectMapper json;
    @MockBean OrderService service;

    @Test
    void createReturns201() throws Exception {
        when(service.place(any())).thenReturn(new OrderResponse("O123", 1000, 50));
        mvc.perform(post("/orders").contentType(APPLICATION_JSON)
                .content(json.writeValueAsString(new OrderRequest("C1", "SKU1", 2))))
            .andExpect(status().isCreated())
            .andExpect(jsonPath("$.id").value("O123"))
            .andExpect(jsonPath("$.total").value(1000));
    }

    @Test
    void validatesNegativeQuantity() throws Exception {   // test Spring validation
        mvc.perform(post("/orders").contentType(APPLICATION_JSON)
                .content(json.writeValueAsString(new OrderRequest("C1", "SKU1", -2))))
            .andExpect(status().isBadRequest());
    }
}

3.4 Contract test (Pact) — consumer + provider SUPPLEMENTARY

// ── Consumer side (OrderService): định nghĩa kỳ vọng ──
@ExtendWith(PactConsumerTestExt.class)
@PactTestFor(providerName = "NotificationService")
class NotificationPactTest {
    @Pact(consumer = "OrderService")
    RequestResponsePact notifyContract(PactDslWithProvider b) {
        return b.given("notification service is up")
            .uponReceiving("notify customer of order")
            .path("/notify").method("POST")
            .body(new PactDslJsonBody().stringType("customerId", "C1")
                                       .stringType("message", "Order placed"))
            .willRespondWith().status(200)
            .body(new PactDslJsonBody().stringType("status", "sent"))
            .toPact();
    }
    @Test
    void verifiesConsumerCanCall(MockServer mock) {
        String res = notificationClient.notify("C1", "Order placed", mock.getUrl());
        assertThat(res).contains("sent");
    }
}

// ── Provider side (NotificationService): verify thoả contract ──
@Provider("NotificationService")
@PactFolder("pacts")
class NotificationProviderTest {
    @TestTemplate
    @ExtendWith(PactVerificationInvocationContextProvider.class)
    void verify(PactVerificationContext ctx) { ctx.verifyInteraction(); }
}

5. 3-tier interview framework

Tầng 1 — SurfaceJunior gate

Q: Giải thích test pyramid.

Nhiều unit (nhanh, mock dependency) : một số integration (chậm hơn, DB thật) : ít E2E (chậm nhất, toàn hệ thống). Tỉ lệ ~70:20:10. Lý do: feedback nhanh + confidence, chi phí CI hợp lý. Inverted pyramid (nhiều E2E) làm CI chậm, brittle, khó debug.

Tầng 2 — Deep DiveSenior gate

Q: Test integration mất 45 phút, tối ưu xuống 10 phút.

Testcontainers reuse singleton container (không tạo lại mỗi test class) + chạy song song; dùng @DataJpaTest/@WebMvcTest slice thay full @SpringBootTest; tách CI stage (unit trước cho feedback nhanh, integration sau); fixtures dùng chung tránh INSERT lặp; chỉ critical path mới integration, còn lại unit. Đo và parallelize theo thời gian thật.

Tầng 3 — ArchitectureSenior+

Q: 5 microservice, 80% coverage vẫn P1. Thiết kế chiến lược chất lượng.

Coverage ≠ quality. Thêm contract test (Pact — bắt breaking API), mutation testing (đo test có thật sự bắt bug hay chỉ chạy qua), chaos testing (giết pod ngẫu nhiên — verify resilience), canary deploy (1% traffic bắt bug trước full rollout), observability (không test hết được mọi scenario thì quan sát prod), property-based test (sinh 10K input tìm edge case). Coverage chỉ là sàn, không phải mục tiêu.

🔧 Lab bắt buộc — viết & chứng minh (2 bài)

Lab 1 — Brittle vs behavior. Viết một test check verify(service).setStatus("PAID") (implementation). Refactor đổi tên method nội bộ → test đỏ dù hành vi không đổi. Viết lại test chỉ assert kết quả quan sát được (order.status() == PAID), refactor lại → test vẫn xanh. Tiêu chí: giải thích vì sao test thứ hai cho phép refactor tự do.

Lab 2 — H2 che bug. Viết một native query/upsert dùng cú pháp PostgreSQL (INSERT ... ON CONFLICT hoặc jsonb). Chạy test với H2 → pass nhầm hoặc lỗi khó hiểu; chạy với Testcontainers PostgreSQL → phản ánh đúng prod. Tiêu chí: chỉ ra đúng một câu SQL mà H2 và PostgreSQL hành xử khác nhau.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. Vì sao "80% coverage" không đảm bảo ít bug? Coverage đo gì?
  2. Tỉ lệ pyramid và lý do mỗi tầng nhiều/ít? Inverted pyramid hại ở đâu?
  3. Cho 2 khác biệt cụ thể H2 vs PostgreSQL khiến bug lọt.
  4. Contract test (Pact) chặn breaking change API thế nào, mà không cần chạy cả 2 service?
  5. "Test behavior, không test implementation" nghĩa là gì? Cho ví dụ một test brittle và bản sửa.
Bí câu nào = quay lại đúng section. Trôi cả 5 = nắm chắc testing mức senior.
