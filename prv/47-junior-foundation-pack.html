1. Bài Toán Thực Tế — Một API lọc user activity chạy tốt ở 1k records, sập ở 1M records CORE

Social platform cần tìm user có tổng watch time vượt ngưỡng. Junior viết code chạy đúng với test nhỏ, nhưng production dataset 1M events làm endpoint timeout.

Code junior viết — chạy được nhưng dễ gây lỗi
public List<String> findPowerUsers(List<Activity> activities) {
    List<String> result = new ArrayList<>();
    Map<String, Long> sums = new HashMap<>();
    for (int i = 0; i < activities.size(); i++) {
        Activity a = activities.get(i);
        if (sums.containsKey(a.userId())) {
            sums.put(a.userId(), sums.get(a.userId()) + a.seconds());
        } else {
            sums.put(a.userId(), a.seconds());
        }
    }
    for (String userId : sums.keySet()) {
        if (sums.get(userId) > 3600) result.add(userId);
    }
    return result;
}
🔥 Hậu quả

Code đúng output nhưng thiếu fluency: nhiều lookup thừa, không nghĩ đến list implementation, không guard null/empty, không đo complexity. Ở production, những điểm nhỏ này cộng lại thành latency lớn.

⏸ Pause & Predict (BẮT BUỘC dừng)
Trước khi đọc tiếp, tự trả lời:
  1. Nếu input là LinkedList, vòng for index có còn ổn không?
  2. containsKey + get + put tốn mấy lookup?
  3. Nếu userId null hoặc seconds âm thì API nên làm gì?
Nếu trả lời còn mơ hồ, đọc kỹ phần concept và tự chạy lại ví dụ.

2. Collections choice CORE

★ CORE: Junior backend dùng collection mỗi ngày.

Collection không chỉ là nơi chứa data. Nó là quyết định về access pattern, memory, ordering, uniqueness và thread-safety.

Rule đơn giản: cần contains nhanh thì nghĩ Set/Map; cần index access thì ArrayList; cần stack/queue thì ArrayDeque; cần concurrency thì dùng collection concurrent thay vì tự synchronized bừa.

🎯 Vì sao phải nắm?

Chọn sai collection thường không fail ngay, chỉ chậm dần khi data lớn.

❌ LinkedList + contains cho dedupe
List<String> seen = new LinkedList<>();
if (seen.contains(id)) return;
seen.add(id);
✅ HashSet đúng access pattern
Set<String> seen = new HashSet<>();
if (!seen.add(id)) return;
🧠 Misconception Check — Collections choice
"LinkedList nhanh hơn vì add/remove O(1)."
Chỉ đúng khi bạn đã có node/iterator ở vị trí đó. Tìm vị trí hoặc contains vẫn O(n), cache locality cũng kém ArrayList.

3. Big-O bằng cảm giác chạy thật CORE

★ CORE: Không có Big-O, bạn không dự đoán được code khi data tăng.

Big-O không phải bài toán trường lớp. Nó là cách hỏi: nếu data tăng 10 lần, code tăng bao nhiêu lần?

Junior cần nhìn thấy nested loop, lookup trong loop, sort, recursion và nhận ra cost chính.

🎯 Vì sao phải nắm?

Interview coding và production performance đều cần estimate nhanh.

❌ Nested loop tìm duplicate
for (User a : users) {
  for (User b : users) {
    if (!a.id().equals(b.id()) && a.email().equals(b.email())) return true;
  }
}
✅ Set giảm O(n²) thành O(n)
Set<String> emails = new HashSet<>();
for (User u : users) {
  if (!emails.add(u.email())) return true;
}
🧠 Misconception Check — Big-O bằng cảm giác chạy thật
"O(n) lúc nào cũng nhanh."
O(n) với n=100M vẫn rất lớn; constant factor, IO và memory cũng quan trọng. Big-O là khởi đầu, không phải toàn bộ performance.

4. equals/hashCode CORE

★ CORE: HashMap/HashSet đúng hay sai nằm ở contract này.

Nếu object dùng làm key mà equals/hashCode sai, lookup sẽ mất hoặc duplicate kỳ lạ.

Record giúp giảm lỗi cho value object, nhưng bạn vẫn phải hiểu identity nào là business identity.

🎯 Vì sao phải nắm?

Bug này khó debug vì code nhìn có vẻ hợp lý.

❌ Key mutable làm HashSet mất dấu
UserKey key = new UserKey("u1");
set.add(key);
key.setValue("u2");
set.contains(key); // false-ish behavior
✅ Value object immutable
public record UserKey(String value) {
  public UserKey {
    Objects.requireNonNull(value);
  }
}
🧠 Misconception Check — equals/hashCode
"Override equals là đủ."
Hash-based collection dùng cả equals và hashCode. Hai object equals phải có cùng hashCode.

5. Generics PECS SUPPLEMENTARY

SUPPLEMENTARY: Đọc được signature Java/Spring mà không sợ wildcard.

PECS: Producer Extends, Consumer Super. Nếu collection chỉ để đọc, dùng extends. Nếu chỉ để add T vào, dùng super.

Không cần dùng wildcard mọi nơi. Hãy dùng khi API thật sự cần linh hoạt type hierarchy.

🎯 Vì sao phải nắm?

Nhiều API framework dùng wildcard; không hiểu sẽ copy code mù.

❌ API quá cứng
double sum(List<Number> numbers) { return 0; }
// Không nhận List<Integer>
✅ Producer dùng extends
double sum(List<? extends Number> numbers) {
  return numbers.stream().mapToDouble(Number::doubleValue).sum();
}

6. Recursion mental model ADVANCED

ADVANCED: Nền cho tree, graph và divide-and-conquer.

Đừng trace mọi stack frame. Hãy tin recursive call giải đúng input nhỏ hơn, rồi tập trung vào base case và combine step.

Nếu recursion không giảm input hoặc base case sai, StackOverflowError chỉ là vấn đề thời gian.

🎯 Vì sao phải nắm?

LeetCode tree/DFS và nhiều parser/backtracking cần tư duy này.

❌ Base case thiếu
int height(Node n) {
  return 1 + Math.max(height(n.left), height(n.right));
}
✅ Base case rõ
int height(Node n) {
  if (n == null) return 0;
  return 1 + Math.max(height(n.left), height(n.right));
}

🎯 Edge Cases — Production Reality SUPPLEMENTARY

🎯 Edge Cases — Production Reality

Happy path chưa đủ. Junior cần học cách hỏi: input lạ, data lớn, concurrency, security và vận hành thì sao?

  • Empty/null input: API collection nên document null policy; thường nhận empty list, reject null.
  • Scale 1M items: Chọn O(n) thay vì O(n²), pre-size collection khi biết size.
  • Concurrent access: HashMap không thread-safe; dùng ConcurrentHashMap hoặc external lock rõ.
  • Security: Không log raw user data khi debug collection payload.
  • Data integrity: equals/hashCode sai làm dedupe sai.
  • Time/timezone: So sánh time bằng Instant, không sort string date.
  • Memory pressure: Set 10M ids có thể ăn nhiều GB; cân nhắc streaming/batch.
🔬 Thử Phá Vỡ — HashSet dedupe activity ids

Câu hỏi: tìm 3 cách làm đoạn code hoặc workflow trên fail trong production.

Click để xem analysis (sau khi đã nghĩ)
  1. id null làm hash/dedupe policy mơ hồ.
  2. ActivityId mutable sau khi add vào set làm lookup sai.
  3. Set giữ mãi mọi id làm memory tăng không giới hạn.

Insight: Foundation tốt luôn hỏi data lớn, input bẩn và vòng đời object.

Luyện nói thành tiếng. Mỗi câu nên trả lời trong 1-3 phút, có ví dụ và trade-off.

CORE Q1. ArrayList khác LinkedList thế nào?

Answer mẫu: ArrayList backed by array, random access O(1), add cuối amortized O(1), insert đầu/giữa O(n). LinkedList node-based, get(i) O(n), thêm/xóa khi có node O(1) nhưng cache locality kém.

Signal: Có Big-O và memory/locality, không chỉ định nghĩa.

CORE Q2. HashSet dựa vào gì để dedupe?

Answer mẫu: Dựa vào hashCode để tìm bucket và equals để xác nhận equality. Contract: nếu a.equals(b) thì hashCode phải bằng nhau.

Signal: Nhắc đủ cả hai method.

SUPPLEMENTARY Q3. PECS là gì?

Answer mẫu: Producer Extends, Consumer Super. Đọc từ collection dùng ? extends T; ghi T vào collection dùng ? super T.

Signal: Có ví dụ đọc/ghi.

CORE Q4. Big-O của nested loop?

Answer mẫu: Thường O(n²), nhưng phải xem loop inner chạy bao nhiêu và operation bên trong có O(1) thật không.

Signal: Không trả lời máy móc.

ADVANCED Q5. Recursion cần kiểm tra gì?

Answer mẫu: Base case, input nhỏ hơn sau mỗi call, combine step đúng và stack depth có chấp nhận được không.

Signal: Nhắc StackOverflow/iterative alternative.

CORE Q6. Nếu code chạy đúng ở local, vì sao senior vẫn có thể reject?

Answer mẫu: Vì local happy path chỉ chứng minh một input nhỏ chạy được. Senior review failure mode: input bẩn, data lớn, concurrency, rollback, observability, security và maintainability.

Signal: Bạn hiểu code review không phải bắt bẻ style.

CORE Q7. Một câu trả lời junior tốt khác gì câu trả lời thuộc lòng?

Answer mẫu: Nó có ví dụ cụ thể, trade-off, và nói được khi nào cách đó sai. Thuộc lòng thường chỉ có định nghĩa.

Signal: Có thể tự tạo ví dụ mới.

SUPPLEMENTARY Q8. Bạn sẽ viết checklist học topic này thế nào?

Answer mẫu: Chia thành concept, code example, edge case, exercise, Feynman explanation và source verification.

Signal: Có cách học có thể lặp lại.

SUPPLEMENTARY Q9. Khi gặp output lạ, bước đầu tiên là gì?

Answer mẫu: Đóng băng hiện trạng: input, command, log, version, branch, environment. Sau đó thu hẹp bằng reproduction nhỏ.

Signal: Không sửa mù.

ADVANCED Q10. Làm sao biết mình đã hiểu thật?

Answer mẫu: Bạn có thể phá code, sửa code, giải thích mechanism, và dạy người khác tránh lỗi tương tự.

Signal: Hiểu thật gắn với hành động.

ADVANCED Q11. Nếu phải mentor junior khác, bạn bắt đầu từ đâu?

Answer mẫu: Bắt đầu từ lỗi thật họ vừa gặp, vẽ mental model, sửa một ví dụ nhỏ, rồi cho họ tự làm biến thể.

Signal: Biết dạy bằng feedback loop.

Self-Test Scenarios SUPPLEMENTARY

Đọc bối cảnh, tự viết đáp án, rồi mới xem gợi ý. Đây là phần chống đọc hiểu giả mạnh nhất.

CORE Scenario 1 — Endpoint timeout với 1M events

Bối cảnh: Code dùng nested loop dedupe activity.

Tự trả lời: Bạn giảm complexity thế nào?

Gợi ý: Dùng Set/Map để chuyển lookup sang O(1), đo memory và test với dataset lớn.

CORE Scenario 2 — HashMap mất key

Bối cảnh: Object key bị mutate sau khi put.

Tự trả lời: Vì sao get trả null?

Gợi ý: Hash bucket tính từ hashCode cũ, object đổi field làm hashCode mới khác.

SUPPLEMENTARY Scenario 3 — Generics compile fail

Bối cảnh: Method nhận List<Number> nhưng caller có List<Integer>.

Tự trả lời: Sửa signature thế nào?

Gợi ý: Nếu chỉ đọc number, dùng List<? extends Number>.

CORE Scenario 4 — Code review bị hỏi 'vì sao?'

Bối cảnh: Bạn chọn một cách implement nhưng chỉ trả lời được 'em thấy người ta hay làm vậy'.

Tự trả lời: Làm sao nâng câu trả lời?

Gợi ý: Nói problem, constraint, trade-off, alternative và lý do chọn. Nếu chưa nói được, bạn mới copy pattern.

CORE Scenario 5 — Unit test pass nhưng integration fail

Bối cảnh: Test nhỏ mock hết dependency, lên môi trường thật fail vì data/permission/path khác.

Tự trả lời: Thiếu loại kiểm chứng nào?

Gợi ý: Cần test gần reality hơn: integration, contract, command transcript, hoặc environment checklist.

SUPPLEMENTARY Scenario 6 — Bug chỉ xảy ra khi chạy song song

Bối cảnh: Hai request hoặc hai dev thao tác cùng resource làm state lệch.

Tự trả lời: Bạn sẽ tìm race thế nào?

Gợi ý: Tìm shared mutable state, check-then-act, non-atomic operation, lock/transaction boundary.

SUPPLEMENTARY Scenario 7 — Log không đủ debug

Bối cảnh: Production chỉ có dòng 'failed' mà không có id, input, stack trace.

Tự trả lời: Log tối thiểu cần gì?

Gợi ý: Event name, correlation id, business id, outcome, duration, exception cause, không chứa secret.

ADVANCED Scenario 8 — Tự viết runbook cho lỗi đã gặp

Bối cảnh: Bạn vừa được senior cứu một lỗi trong 5 phút.

Tự trả lời: Biến nó thành tài sản học thế nào?

Gợi ý: Ghi symptom, command, expected output, root cause, fix, rollback và prevention checklist.

Interview Framework CORE

Tầng 1 — SurfaceJunior gate
Q: Khi nào dùng ArrayList, HashSet, HashMap?
ArrayList cho ordered list + random access; HashSet cho uniqueness/contains nhanh; HashMap cho lookup key-value nhanh.
Tầng 2 — Deep DiveCơ chế bên trong
Q: HashMap worst-case vì sao không luôn O(1)?
Collision nhiều làm bucket dài; Java 8 treeify bucket khi đủ điều kiện để giảm worst-case. Nhưng hash tốt và key immutable vẫn là nền.
Tầng 3 — PracticalÁp dụng vào project
Q: Review code junior xử lý 1M events, bạn nhìn gì?
Complexity, collection choice, null/empty policy, equality, memory, logging và test data lớn.
