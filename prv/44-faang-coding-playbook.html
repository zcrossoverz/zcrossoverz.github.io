FAANG Coding Playbook — Pass round dù solve ít LC, fail dù solve nhiều

Bạn solve 200 LeetCode, pattern recognition trong 60 giây, code clean — vẫn fail FAANG onsite. Lý do: coding round KHÔNG phải cuộc thi giải bài. Là đánh giá đa chiều theo rubric 5 trục mà interviewer ghi vào feedback form. Junior chỉ tối ưu trục 1 (Problem Solving) → fail. Senior tối ưu cả 5. File này show rubric thật, feedback form thật, walkthrough 45-phút mock đầy đủ — không phải tips chung chung.

🎯 Mục tiêu
Hiểu rubric 5 trục + Walk-through 1 mock + Strategic loop choice
📅 Thời lượng
Tháng 4 trong 6-month roadmap · ~80 LC Medium đã solve trước
📚 Prereq
#42 Patterns + #43 UMPIRE đã master
🔥 Độ ưu tiên
🔥 Critical — kỹ năng tách bạn khỏi 200-LC-fail crowd

⚠️ Cảnh báo "hiểu giả" — đặc trưng FAANG prep

Vì sao FAANG coding round dễ hiểu giả:
  • "Solve được = pass" là sai. Solve optimal nhưng silent + không test = "no signal" = no hire.
  • Rubric không public. Đa số dev không biết 5 trục đánh giá → tối ưu sai chỗ.
  • YouTube mock không show feedback form. Bạn thấy mock "perfect" nhưng không biết interviewer ghi gì.
  • Blog "tips" generic. "Communicate clearly" không actionable. Cần concrete: "im < 30s, narrate mỗi 30-60s, edge case 5-7 tự liệt kê".

1. Vì sao solve 200 LC vẫn fail? CORE

Hiện tượng phổ biến: candidate VN solve 200-500 LC, recognize pattern nhanh, code clean trong 25 phút — vẫn No Hire FAANG. Lý do căn bản:

Coding round KHÔNG phải cuộc thi giải bài. Là đánh giá đa chiều mà interviewer điền vào feedback form theo rubric cụ thể. Nếu bạn không biết rubric, bạn không biết tối ưu cái gì.

4 hiểu lầm gốc

  1. "Solve optimal = pass." Sai. Bạn có thể solve optimal mà silent (no signal) hoặc không test (no verification) → vẫn fail.
  2. "Pattern recognition là tất cả." Sai. Recognition là 25% (trục 1). Còn 75% rubric khác.
  3. "Code đẹp là plus." Code đẹp là bắt buộc (trục 2 — 25%). Không có là minus, có là baseline.
  4. "FAANG interview như LeetCode contest." Sai. LeetCode contest đo speed solo. FAANG đo collaboration + communication + judgment.

Mục tiêu file này

Show bạn chính xác rubric, feedback form, walkthrough. Sau khi đọc:

  • Biết 5 trục đánh giá + tỷ trọng
  • Tự fill feedback form sau mỗi mock
  • Có script 45-phút để adapt
  • Biết khác biệt Google vs Meta vs Amazon
  • Có recovery plan nếu reject

3. Rubric 5 trục — Đây là kim chỉ nam HIGH-SIGNAL

Đây là section quan trọng nhất file. Đọc kỹ. Nếu chỉ nhớ 1 thứ từ file này, nhớ 5 trục + tỷ trọng. Mọi anti-pattern, mọi rule đều derive từ rubric.

Trục 1 — Problem Solving (25%)

Interviewer đánh giá:
  - Có recognize pattern không?
  - Brute force trước → BUD analysis → optimal?
  - Self-driven exploration hay phải hint nhiều?

Junior fail vì:
  Code thẳng optimal, không show thinking process.
  Khi hỏi "vì sao chọn approach này?" → lúng túng.

Strong signal:
  "Khi gặp bài chưa từng gặp, brute force trước, complexity,
   BUD analysis tự nhận ra duplicated work, derive optimal
   mà không cần hint."

Trục 2 — Coding (25%)

Interviewer đánh giá:
  - Code có clean không (variable names, structure)?
  - Code chạy đúng trong thời gian?
  - Off-by-one, edge cases handle đúng?

Junior fail vì:
  Dùng `i, j, x, temp` chung chung, indent lung tung,
  không có guard clause cho null/empty.

Strong signal:
  "Code đọc như đọc văn — biến tên describe purpose,
   helper function khi cần, không over-engineer,
   edge case guard ở đầu."

Trục 3 — Communication (25%)

Interviewer đánh giá:
  - Có narrate liên tục không?
  - Có ask clarifying question không?
  - Có handle hint từ interviewer không?

Junior fail vì:
  Im lặng 5-10 phút khi code, không cập nhật interviewer.

Strong signal:
  "Mỗi 30-60 giây có 1 update verbal. Hint từ interviewer
   được absorb và integrate ngay. Tôi cảm thấy như đang
   pair programming với senior dev."

Trục 4 — Verification (15%)

Interviewer đánh giá:
  - Có tự test code với example?
  - Có liệt kê edge cases?
  - Có tự tìm bug trước khi interviewer chỉ ra?

Junior fail vì:
  Code xong nói "done", không trace.

Strong signal:
  "Tự trace code với example đã vẽ. Liệt kê 5-7 edge cases
   đầu mình. Tự tìm bug 'integer overflow ở dòng 12'
   trước khi tôi hỏi."

Trục 5 — Knowledge Depth (10%)

Interviewer đánh giá:
  - Có hiểu data structure internals?
  - Có biết alternatives + trade-offs?
  - Có suggest follow-up optimization?

Junior fail vì:
  Dùng HashMap mà không giải thích được Java 8 tree fallback,
  không biết khi nào dùng TreeMap.

Strong signal:
  "Khi tôi hỏi 'why HashMap?', candidate trả lời được
   Java 8 internals, alternatives như TreeMap/LinkedHashMap,
   và trade-offs cụ thể."
Tổng kết tỷ trọng:
  • Problem Solving + Coding + Communication = 75% (3 trục chủ chốt)
  • Verification (15%) tách candidate "OK" khỏi "Strong"
  • Knowledge Depth (10%) tách "Hire" khỏi "Strong Hire"
Tối ưu: dành 75% effort cho 3 trục đầu, 25% cho 2 trục sau.

⏸️ Pause & Predict

Trước đọc Section 4 (feedback form): viết ra giấy 3 mock gần nhất của bạn (hoặc 3 LC bạn solve gần nhất nếu chưa mock). Tự score 5 trục mỗi cái — 1-5. Trục nào weakest? Đó là focus 2 tuần tới.

Cách đọc form ngược

Mục tiêu của candidate là tối đa hóa số check ở:

  • "Self-derived optimal" (không phải "after hints")
  • "Strong signal — narrated continuously"
  • "Self-tested with example + edge cases"
  • "Engaged + suggested alternative"
  • "STRONG HIRE"
Insight về "NOTES" field: Đây là phần hiring committee đọc kỹ nhất. Check-box là baseline; notes là "signal description". Strong note: "Recognized DP/Kadane's quickly, derived optimal from brute force using BUD reasoning." — concrete, specific. Weak note: "Candidate seemed nervous, solved the problem." — vague, no signal.

Cách candidate giúp interviewer ghi note mạnh: say things that are quotable. Khi nói "Tôi sẽ apply BUD analysis" + "Optimal là Kadane's vì..." → interviewer dễ quote → note mạnh.

6. L4 vs L5 vs L6 bar SUPP

L4 (SDE2 / E4 / SWE3) — "senior contributor" entry

  • Coding bar: Solve Medium trong 25-30 phút, Hard với 1-2 hints. Clean code, edge cases mostly correct. Complexity self-derived.
  • System design bar: HLD cho known problem (URL shortener, simple chat). Identify bottlenecks. Suggest 1-2 trade-offs.
  • Behavioral: Concrete examples of ownership trong 6-12 tháng project. Conflict resolution peer level.

L5 (Senior / E5 / SWE4) — "tech lead"

  • Coding bar: Solve Hard trong 30 phút cleanly. Handle 2-3 follow-ups deeply. Spot edge cases preemptively.
  • System design bar: HLD + LLD cho new problem. Multiple trade-off levels (consistency vs availability). Capacity estimation rigorous.
  • Behavioral: Lead 2+ engineers, mentor juniors. Cross-team negotiation. Strategic decision making.

L6 (Staff / E6 / SWE5) — "principal contributor"

  • Coding bar: Ít emphasis (vẫn solid). Design new approach cho novel problem.
  • System design bar: Design entire system across teams. Org-level trade-offs. Migrations, evolutions.
  • Behavioral: Lead 5-10 engineers, multi-team initiatives. Set technical direction cho org. Hire và grow team.
Strategic note: Junior 3 năm KHÔNG nên target L5+ trực tiếp. Lý do: L5+ đòi hỏi leadership signal (lead engineers, drive org decisions) — 3 năm khó có substantive evidence. Apply L4, promote lên L5 trong 2-3 năm. Salary L4 FAANG đã ~$200K+ total comp — đủ tốt.

7. Mock 45 phút end-to-end — Walkthrough thật HIGH-SIGNAL

Setup: Phone screen Google L4. Interviewer = senior engineer 6 năm. Tool = Google Doc share screen. Bài: LC 53 Maximum Subarray.

Phút 0:00-2:00 — Intro

Interviewer (Sarah): "Hi, I'm Sarah, software engineer at Google for 6 years.
                       Today we'll do 1 coding problem in 45 minutes.
                       Tell me about yourself briefly, then we'll start."

Candidate: "Hi Sarah, I'm Nhan. 3 years backend Java, working in e-commerce
            building order/payment platforms. Looking forward to today."

Đừng dài dòng. 30-60 giây enough. Quá dài = mất thời gian cho bài.

Phút 2:00-3:00 — Đề bài

Interviewer: "Given a list of integers, find the contiguous subarray with
              the largest sum. Return that sum.
              Example: [-2,1,-3,4,-1,2,1,-5,4] should return 6 (subarray [4,-1,2,1])."

Phút 3:00-5:00 — Understand + Match

Candidate: "Just to confirm — I need to find the contiguous subarray
            (not subsequence) with maximum sum. Return only the sum,
            not the indices? Can the array be empty? Can elements be negative?"

Interviewer: "Yes contiguous. Return sum only. Array is non-empty. Elements
              can be negative or zero."

Candidate: "Got it. Let me also check edge cases:
            - All negative: e.g., [-3,-1,-2] → return -1 (best single element)
            - Single element: [-5] → return -5
            - All positive: [1,2,3] → return 6 (whole array)

            And constraint on length? 10^5? Need O(n) or O(n log n)?"

Interviewer: "Length up to 10^5. Aim for efficient."

Candidate: "OK. This problem has the signal of either DP or sliding window —
            'maximum subarray sum'. Let me think both."

What just happened: Pattern recognition signal sent. Edge case 3 cái nêu trước. Constraint hỏi → biết complexity target. 2 phút này worth its weight in gold.

Phút 5:00-10:00 — Plan (brute → optimal)

Candidate: "Brute force: 2 nested loops, compute sum of every (i, j) pair,
            track max. O(n²) time, O(1) space. Works but won't scale to 10^5
            (10^10 operations).

            BUD analysis:
            - Bottleneck: nested loop
            - Duplicated work: computing sum from scratch each pair.
              Could use prefix sum to compute (i,j) sum in O(1).
              That gives O(n²) still but cleaner.

            But there's a smarter pattern. Kadane's algorithm:
            - Maintain currentSum = best sum ending at index i
            - Either: extend previous (currentSum + nums[i])
                      or: start fresh from nums[i]
            - currentSum = max(nums[i], currentSum + nums[i])
            - Track maxSoFar

            O(n) time, O(1) space. Optimal."

Interviewer: "Sounds good. Code it up."

Senior signal: Brute force first → BUD → derive optimal. Show thinking, không jump to optimal. Interviewer note "self-derived, didn't memorize".

Phút 10:00-25:00 — Implement (narrate liên tục)

public int maxSubArray(int[] nums) {
    if (nums == null || nums.length == 0) {
        throw new IllegalArgumentException("Array must be non-empty");
    }

    int currentSum = nums[0];
    int maxSoFar = nums[0];

    for (int i = 1; i < nums.length; i++) {
        // Extend or restart
        currentSum = Math.max(nums[i], currentSum + nums[i]);
        maxSoFar = Math.max(maxSoFar, currentSum);
    }

    return maxSoFar;
}

Narration suốt khi code (quotable cho interviewer):

"I'll guard the edge case empty array first..."
"Initialize currentSum and maxSoFar with nums[0] — handles single element..."
"Loop from i=1..."
"At each step, either extend by adding nums[i] to currentSum, or restart
 with just nums[i] — whichever is larger..."
"Update maxSoFar with the running max..."
"Return maxSoFar."

Phút 25:00-32:00 — Review + Test

Candidate: "Let me trace with the example:
   nums = [-2, 1, -3, 4, -1, 2, 1, -5, 4]

   i=0: currentSum=-2, maxSoFar=-2
   i=1 (1):  currentSum=max(1, -2+1=-1)=1; maxSoFar=max(-2,1)=1
   i=2 (-3): currentSum=max(-3, 1-3=-2)=-2; maxSoFar=1
   i=3 (4):  currentSum=max(4, -2+4=2)=4; maxSoFar=4
   i=4 (-1): currentSum=max(-1, 4-1=3)=3; maxSoFar=4
   i=5 (2):  currentSum=max(2, 3+2=5)=5; maxSoFar=5
   i=6 (1):  currentSum=max(1, 5+1=6)=6; maxSoFar=6  ← target answer ✓
   i=7 (-5): currentSum=max(-5, 6-5=1)=1; maxSoFar=6
   i=8 (4):  currentSum=max(4, 1+4=5)=5; maxSoFar=6

   Return 6. Correct!"

Candidate: "Edge cases:
   - All negative [-3,-1,-2]:
     i=0: cur=-3, max=-3
     i=1: cur=max(-1,-3-1=-4)=-1; max=-1
     i=2: cur=max(-2,-1-2=-3)=-2; max=-1 ✓

   - Single [-5]: returns -5 from initialization ✓

   - All positive [1,2,3]: cur grows to 6, max=6 ✓

   I think it handles all cases."

Phút 32:00-35:00 — Complexity + Follow-up

Candidate: "Time: O(n), single pass.
            Space: O(1), only 2 variables.
            This is optimal — we have to look at each element at least once."

Interviewer: "Nice. Follow-up: what if we needed to return the actual
              subarray, not just the sum?"

Candidate: "We track start and end indices. When currentSum 'restarts'
            (i.e., nums[i] > currentSum + nums[i]), update tempStart = i.
            When maxSoFar updates, lock start = tempStart, end = i.
            [code modification]
            Same O(n) complexity, just 2 extra variables."

8. 7 rules during interview CORE

Rule 1 — First 3 minutes is gold

Đừng vội code. Clarify, vẽ example, identify pattern. Bước này quyết định 50% kết quả. Skip = jump-to-design lạc hướng.

Rule 2 — Speak before code

Mỗi action có narration:

  • "Tôi sẽ thử brute force trước..."
  • "Đây là dòng tôi cần handle null..."
  • "Tôi đang nghĩ pattern sliding window phù hợp vì..."

Rule 3 — When stuck, talk it out

Im lặng = no signal. Khi bí:

"Hmm, tôi đang nghĩ giữa 2 hướng A và B. A có vẻ work nhưng O(n²),
 B chưa rõ. Để tôi try B thêm xíu... [explore]"

Show thinking. Interviewer có thể hint dựa trên direction bạn đang đi. Silent = không có data để hint.

Rule 4 — Acknowledge hints gracefully

❌ Defeated

"Oh, đúng rồi, sao tôi
 không nghĩ ra..."

(Self-deprecation signal weak)

✅ Engaged

"Good direction! Let me
 think about that... yes,
 that simplifies the
 left pointer logic..."

(Absorb + integrate)

Rule 5 — Code as if for production

  • Variable names: seenIndices không m
  • Indentation consistent
  • Helper function nếu logic > 15 lines
  • Edge case guard ở đầu (null, empty)

Rule 6 — Self-test BEFORE interviewer asks

Code xong, immediately trace với example. Đợi interviewer hỏi "did you test?" → đã mất signal Verification.

Rule 7 — Always have follow-up answer ready

Mọi optimal solution có "what if" — chuẩn bị mental list:

- "What if input streamed?"      → online algorithm
- "What if duplicates?"           → handle accordingly
- "What if sorted?"               → can we use BS / two pointers?
- "What if multi-thread?"         → concurrent data structure
- "What if input không fit memory?" → external sort, streaming

9. 5 failure modes & fix CORE

❌ Mode 1: "Optimal-first syndrome"

Code thẳng optimal, không brute force. Khi hỏi "why this?", lúng túng.

Fix: Practice bao giờ cũng start brute force. Note ra "Brute first" làm khẩu hiệu.

❌ Mode 2: "Silent debugger"

Khi bug, mặt căng thẳng, sửa code im lặng 5 phút.

Fix: "Let me trace this carefully... at line 12 I see issue..." Narrate debug process.

❌ Mode 3: "We did" instead of "I did"

Behavioral kể "we deployed", "team decided" — interviewer không biết bạn làm gì cụ thể.

Fix: "I designed... I implemented... I led..." — own your contribution.

❌ Mode 4: "Over-engineering"

Code 30 dòng cho bài 10 dòng. Add validation, logger, custom exception.

Fix: "In production I'd add X, Y. For interview, focus algorithm." State trade-off clearly.

❌ Mode 5: "No questions for me"

Khi interviewer hỏi reverse Q, "I don't have any". → Signal weak interest, no preparation.

Fix: Chuẩn bị 5 câu cho mọi loop. 3 mức: team/role, technical, career growth.

🔨 Failure mode nguy hiểm nhất?

Mode 2 (Silent debugger). Vì: dù solve đúng cuối cùng, 5 phút im lặng = "no signal" cho Communication (25%). Interviewer feedback "lost candidate during debugging, hard to follow". Mất trục lớn nhất.

12. After-reject recovery SUPP

Recovery process

1. Cool-off 1-2 ngày — không học bài mới, đừng hate self
2. Recall what happened, write down 3-5 issues
3. Map issues to rubric trục nào (problem solving / communication / ...)
4. Plan 1-2 tuần focus fix issue đó
5. Apply lại sau 6-12 tháng (most companies có policy)

  • Google: 6-12 months
  • Meta: 12 months
  • Amazon: 6 months for same level
  • Apple: Depends on team
  • Microsoft: 6-12 months

Trong cooldown, apply company khác. Đừng lụi tinh thần. Quan trọng nhất: map feedback vào rubric để biết focus fix gì.

🎯 3-tier interview framework

Tier 1 — Junior pre-interview check

Q1: 5 trục rubric là gì? Tỷ trọng?

Trả lời: "Problem Solving 25%, Coding 25%, Communication 25%, Verification 15%, Knowledge Depth 10%. 3 trục đầu = 75% — focus đây. Verification tách 'OK' khỏi 'Strong'. Knowledge Depth tách 'Hire' khỏi 'Strong Hire'."

Q2: Loop chuẩn cho L4 gồm gì?

Trả lời: "1 phone screen + 4-5 onsite. Onsite: 2 coding + 1 system design + 1 behavioral + 1 optional. Sau đó hiring committee debrief, vote Hire/No-Hire/Strong Hire."

Tier 2 — Mid-level (sau 4 mock)

Q3: Khi bí giữa interview, làm gì?

"KHÔNG im lặng — silent = no signal cho Communication (25%). Nói: 'Tôi đang nghĩ giữa 2 hướng A và B. A có vẻ work nhưng O(n²), B chưa rõ. Để tôi try B thêm.' Show thinking. Interviewer có thể hint dựa trên direction. Nếu hint, acknowledge gracefully + integrate ngay, không self-deprecate."

Q4: Vì sao brute force trước optimal mặc dù sẽ code optimal?

"3 lý do: ① Tạo baseline để derive optimal có context (BUD analysis). ② Show problem-solving process — interviewer note 'self-derived, not memorized'. ③ Nếu bí ở optimal, có fallback brute force để partial credit. Skip brute force = junior signal 'optimal-first syndrome'."

Tier 3 — Senior (sau 8+ mock)

Q5: Strategy cho 6-month FAANG prep, target Google L4. Bạn còn 0 LC solve.

Phút 1 — Phase 1: Foundation (Tháng 1)
  - Java basics + 30 LC Easy
  - Big-O reflex, equals/hashCode, exception
  - Pattern: warm up, không vướng syntax khi interview

Phút 2 — Phase 2: Pattern Mastery (Tháng 2-3)
  - 18 patterns × 5-7 problems = 90-130 Medium
  - Mỗi pattern theo UMPIRE: Understand → Match → Plan → Implement → Review → Evaluate
  - Track Notion: bài, time, mistakes, pattern weak

Phút 3 — Phase 3: Mock + Speed (Tháng 4)
  - 2 mock/tuần (Pramp + Interviewing.io)
  - Focus rubric: 5 trục self-assessment sau mỗi mock
  - Solve 1 Hard/tuần với UMPIRE recorded

Phút 4 — Phase 4: System Design (Tháng 5)
  - 5 design kinh điển (URL shortener, Twitter, Uber, WhatsApp, YouTube)
  - 2 mock/tuần system design với senior

Phút 5 — Phase 5: Behavioral + Final (Tháng 6)
  - 12-15 STAR stories cover Googleyness + 16 Amazon LP
  - 1 mock behavioral/tuần
  - Apply Google sau cùng — đã pass 5-10 loop khác trước

Trade-off:
  + Front-load grinding (Phase 2-3) = solid foundation
  + Last 2 tháng polish soft skill = differentiator
  - Sacrifice: 6 tháng giảm side activity
  Risk mitigation: bookmark recovery plan nếu reject — apply company khác
  trong cooldown 12 tháng.

🚨 Symptom → Failure Mode Recipes — Self-diagnosis sau mỗi mock

Sau mỗi mock, replay recording + đối chiếu với 5 symptom dưới. Mỗi symptom có root cause + fix cụ thể.

  1. Symptom: Interviewer phải prompt "what are you thinking?"
    → Root: Im lặng quá lâu khi code (> 30s).
    → Fix: Practice solo nhưng nói thành tiếng. Record bản thân. Mục tiêu mọi 15s phải có 1 câu narrate.
  2. Symptom: Solve được nhưng nói complexity chậm hoặc sai
    → Root: Tính complexity sau khi xong, không phải lúc code.
    → Fix: "Code-as-you-think complexity" — mỗi khi viết loop/recursion, thì thầm "this adds O(n)".
  3. Symptom: Test pass nhưng interviewer chưa hài lòng
    → Root: Không trace bằng concrete input.
    → Fix: Sau khi code xong, chọn 1 input có nontrivial → trace từng bước trên giấy (hoặc nói ra).
  4. Symptom: Bí khi hỏi follow-up "what if 1B input?"
    → Root: Không có framework cho scalability question.
    → Fix: Học template — "Can't fit memory → streaming + chunking + merge". "Too slow single node → distributed/parallel". "Need real-time → cache + index".
  5. Symptom: Skip clarification, code 5 phút sau mới hỏi
    → Root: Vội muốn show "có pattern recognition".
    → Fix: Force quy tắc: 60s đầu CHỈ clarify, không gõ phím. Time-box.

🧠 Feynman test

🎓 Giải thích cho candidate chưa biết FAANG prep — < 30s mỗi câu

  1. "Vì sao solve 200 LC vẫn fail FAANG?" — Coding round không phải cuộc thi giải bài. Là đánh giá đa chiều theo rubric 5 trục: Problem Solving 25%, Coding 25%, Communication 25%, Verification 15%, Knowledge 10%. Junior chỉ tối ưu trục 1 → fail.
  2. "Vì sao brute force trước optimal dù sẽ code optimal?" — Show thinking process (Problem Solving). BUD analysis derive optimal từ brute force. Có fallback nếu optimal bí. Skip brute = 'optimal-first syndrome' junior signal.

  4. "Khác biệt L4 và L5 bar coding?" — L4: Medium 25-30 phút, Hard với 1-2 hints. L5: Hard 30 phút cleanly, handle 2-3 follow-up deeply, spot edge case preemptively. Junior 3 năm apply L4, không L5+ trực tiếp.
