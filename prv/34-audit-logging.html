Audit Logging — Immutable, Tamper-Evident, Non-Repudiable

Audit logging là kỹ năng bị under-rated trong backend interview — ai cũng "biết" log action ra DB, nhưng khi probe "DBA của bạn xoá row trong audit_log thì làm sao phát hiện?" hoặc "S3 Object Lock GOVERNANCE vs COMPLIANCE khác gì?" thì lộ ngay. File này dạy bạn vừa cơ chế (hash chain, WORM, non-repudiation), vừa compliance reality (HIPAA, GDPR, SOX, PCI-DSS) — và ép bạn phân biệt audit "đúng nghĩa" với log scatter mà 90% project gọi nhầm là audit.

⚠️ Cảnh báo "hiểu giả" trong Audit Logging

Vì sao audit log dễ hiểu nhầm:
  • "Audit log" bị dùng chung với "application log" / "access log" — 3 thứ khác nhau hoàn toàn về requirement.
  • Pattern "lưu vào bảng audit_log" nghe đơn giản → não thưởng cho cảm giác hiểu. Nhưng nếu DBA có DELETE permission, audit log đó vô giá trị về mặt pháp lý.
  • Buzzword "blockchain audit" / "tamper-proof" được dùng marketing → dev nhầm là cần distributed consensus, thực ra hash chain đơn giản đủ.
  • Compliance regulations (GDPR, HIPAA, SOX, PCI-DSS) phức tạp — đa số dev không đọc spec gốc, chỉ nghe lại từ "compliance officer".

1. Bài toán: Insider sửa audit log CORE

Một SaaS platform quản lý hồ sơ bệnh nhân (healthcare). Nhân viên IT cấp cao tên A có DB superuser. Vì lý do cá nhân, A truy cập trái phép hồ sơ của 1 celebrity:

[T+0]   A đăng nhập với account của mình
[T+10s] A query SELECT * FROM patients WHERE name = 'Celebrity X'
        → Hệ thống tự động ghi audit:
          INSERT INTO audit_log (user_id, action, entity_id, timestamp)
          VALUES ('userA', 'VIEW_PATIENT', 'pat_999', '2026-05-23 14:00:10')

[T+5min] A nhận ra mình đã để dấu vết, mở DB tool:
         DELETE FROM audit_log
         WHERE user_id = 'userA'
           AND timestamp BETWEEN '2026-05-23 14:00:00' AND '2026-05-23 14:00:30';

[T+10min] A cũng modify các log khác để cover thêm:
          UPDATE audit_log SET action = 'LIST_PATIENTS'
          WHERE user_id = 'userA' AND id IN (...);

[T+1 month] Celebrity X phát hiện hồ sơ bị leak, kiện platform
[T+1 month + 1 day] Compliance officer query audit log
                    → Không có dấu vết userA truy cập
                    → "Audit log clean, không có dấu hiệu vi phạm"

Hệ quả thực

  • Platform mất khả năng prove "ai đã làm" — pháp lý không có evidence
  • HIPAA fine (Mỹ): tối đa $1.5M/year per violation category. EU GDPR: tối đa €20M hoặc 4% global revenue.
  • Mất niềm tin của customer — celebrity có thể public hoá vụ việc
  • Insider threat: nhân viên có quyền cao tiếp tục lạm dụng

2. Audit log vs Application log — 3 tính chất phải có CORE

"Audit log" bị dùng nhầm rất phổ biến. Để tránh hiểu giả, định nghĩa rõ:

Loại logMục đíchTính chất
Application log Debug, troubleshoot Mutable, có thể rotate/delete. Format đa dạng (text, JSON).
Access log Phân tích traffic, security Append-only nhưng có thể rotate. Standard format (CLF, JSON).
Audit log Compliance, forensic, legal evidence 3 tính chất bắt buộc — xem dưới.

3 tính chất bắt buộc của Audit log

  1. Immutable — Sau khi ghi, không thể UPDATE hoặc DELETE. Cả admin, cả DBA, cả CEO không xoá được.
  2. Tamper-evident — Nếu ai cố sửa, phải phát hiện được. Hash chain là cơ chế phổ biến.
  3. Non-repudiable — Người thực hiện action không thể phủ nhận. Cần auth + signature + trusted timestamp.
Test self-check: Project bạn có "audit_log table" nhưng admin DB có thể DELETE row? Đó là activity log, KHÔNG phải audit log. Trong compliance audit, log đó vô giá trị.

Mục đích thật của audit log

  1. Compliance — regulatory requirement. HIPAA (healthcare), GDPR (EU personal data), SOX (financial reporting), PCI-DSS (payment). Mỗi cái có retention rule khác nhau.
  2. Forensic — sau incident. Investigation: ai access data, khi nào, từ đâu. Audit phải đủ chi tiết để reconstruct timeline.
  3. Legal evidence. Trong toà, audit log phải đủ chứng minh "user A đã làm X lúc T" — đòi hỏi cả 3 tính chất ở trên.
  4. Insider threat detection. Detect bất thường: user bình thường access 1000 record/ngày đột nhiên access 100K.

3. Hash chaining — Detect tampering HIGH-SIGNAL

L2 check: Sau section này phải explain được "Vì sao dùng HMAC thay SHA256" + "DBA bypass được hash chain không?" trong < 30 giây.

Cơ chế hash chain

Entry 1: { data: "User A view patient_999 at T1" }
  hash_1 = HMAC(data_1 + "GENESIS", secret_key)

Entry 2: { data: "User A view patient_888 at T2", prev_hash: hash_1 }
  hash_2 = HMAC(data_2 + hash_1, secret_key)

Entry 3: { data: "User B login at T3", prev_hash: hash_2 }
  hash_3 = HMAC(data_3 + hash_2, secret_key)

Nếu attacker sửa Entry 2:
  → Recompute hash_2' với data mới
  → Nhưng hash_2' ≠ hash_2 lưu trong Entry 3.prev_hash
  → Toàn bộ chain từ Entry 2 trở đi INVALID khi verify
  → Tampering detected!

Nếu attacker muốn fix:
  → Phải recompute hash_2, hash_3, hash_4, ... đến cuối chain
  → Phải biết secret_key của HMAC → KHÔNG bypass được nếu key được bảo vệ

Vì sao HMAC thay SHA256?

❌ SHA256 trần

hash = SHA256(audit_data)

// Ai cũng compute được — không cần key
// DBA sửa data → recompute SHA256 →
// chain vẫn valid → KHÔNG detect được tampering

✅ HMAC-SHA256

hash = HMAC_SHA256(audit_data, secret_key)

// Chỉ người có secret_key compute được
// Secret_key lưu trong HSM / KMS,
// không cho DBA access
// → DBA sửa data nhưng KHÔNG re-sign được
// → Verify job phát hiện invalid hash

Implementation runnable

@Service
@Slf4j
public class AuditChainService {

    private final AuditEventRepository repo;
    private final HmacService hmacService;      // wraps HSM/KMS call
    private final ReentrantLock chainLock = new ReentrantLock();

    /**
     * Append event với hash chain.
     * Lưu ý: getLastHash + insert phải atomic — nếu không thì 2 thread
     * cùng đọc prev_hash, chain sẽ branch.
     *
     * Dùng app-level lock đơn giản cho single-instance.
     * Multi-instance: dùng DB advisory lock hoặc serializable transaction.
     */
    @Transactional(isolation = Isolation.SERIALIZABLE)
    public AuditEvent append(AuditEventRequest req) {
        chainLock.lock();
        try {
            String prevHash = repo.getLastHash().orElse("GENESIS");

            String dataToSign = buildDataString(req, prevHash);
            String hmacHash = hmacService.computeHmac(dataToSign);

            AuditEvent event = AuditEvent.builder()
                .eventId(UUID.randomUUID().toString())
                .userId(req.getUserId())
                .action(req.getAction())
                .entityType(req.getEntityType())
                .entityId(req.getEntityId())
                .beforeState(req.getBeforeState())
                .afterState(req.getAfterState())
                .ipAddress(req.getIpAddress())
                .eventTime(req.getEventTime())
                .serverTime(Instant.now())      // trust server, not client
                .previousHash(prevHash)
                .currentHash(hmacHash)
                .build();

            repo.save(event);
            return event;
        } finally {
            chainLock.unlock();
        }
    }

    /**
     * Verify integrity của chain trong khoảng [from, to].
     * Chạy scheduled job (daily/weekly) hoặc on-demand.
     * Phát hiện 2 loại tampering:
     *   1. Event bị sửa: recompute HMAC ≠ stored hash
     *   2. Event bị xoá: prev_hash mismatch giữa các event liền kề
     */
    public AuditIntegrityReport verifyChain(Instant from, Instant to) {
        List<AuditEvent> events = repo.findInTimeRange(from, to);
        List<String> violations = new ArrayList<>();

        String expectedPrevHash = events.isEmpty() ? null
            : repo.getHashBefore(events.get(0).getServerTime()).orElse("GENESIS");

        for (AuditEvent event : events) {
            // Check 1: prev_hash trỏ đúng event trước
            if (!event.getPreviousHash().equals(expectedPrevHash)) {
                violations.add("Chain break at " + event.getEventId()
                    + ": expected prev " + expectedPrevHash
                    + " but stored " + event.getPreviousHash());
            }

            // Check 2: recompute hash xem có khớp
            String recomputed = hmacService.computeHmac(
                buildDataString(event, event.getPreviousHash()));
            if (!recomputed.equals(event.getCurrentHash())) {
                violations.add("Tampered event: " + event.getEventId());
            }

            expectedPrevHash = event.getCurrentHash();
        }

        return AuditIntegrityReport.builder()
            .totalEvents(events.size())
            .violations(violations)
            .intact(violations.isEmpty())
            .verifiedAt(Instant.now())
            .build();
    }

    private String buildDataString(AuditEventRequest r, String prevHash) {
        // Canonical serialization — thứ tự field cố định
        return String.join("|",
            r.getUserId(), r.getAction(), r.getEntityType(), r.getEntityId(),
            String.valueOf(r.getEventTime().toEpochMilli()), prevHash);
    }
}

Giới hạn của hash chain — DBA bypass?

🔨 Câu hỏi probe senior

Nếu DBA có quyền DELETE + access HSM secret key, có bypass được hash chain không?

Câu trả lời: CÓ. Nếu DBA có cả 2:

  1. DELETE/UPDATE rows trong audit DB
  2. HMAC secret để re-sign chain

→ DBA recompute toàn bộ hash từ point tampering đến cuối → chain looks valid.

Defence: Phải tách quyền: DBA không có HSM access, HSM admin không có DB access. Plus: replicate audit ra WORM storage bên ngoài DBA control (xem Section 4). Hash chain detect tampering ở DB-level; WORM ngăn tampering ở storage-level.

4. WORM storage — 4 tầng immutability HIGH-SIGNAL

WORM = Write Once Read Many. Audit muốn thực sự immutable phải có WORM ở storage layer, không chỉ application layer.

4 tầng immutability — từ yếu đến mạnh

TầngCơ chếAi bypass được?
1. Application Soft delete, append-only API, không expose DELETE endpoint DBA hoặc anyone với DB access bypass dễ dàng
2. Database Revoke UPDATE/DELETE permission, trigger ngăn delete DB SUPERUSER bypass — bypass trigger, regrant permission
3. OS/Filesystem chattr +i immutable flag, append-only filesystem OS root bypass được (chattr -i)
4. Hardware/Cloud WORM AWS S3 Object Lock COMPLIANCE, Azure Immutable Blob, write-once tape KHÔNG AI bypass trong retention period — kể cả AWS root account

AWS S3 Object Lock — 2 modes phải biết

GOVERNANCE mode:
  - Admins với s3:BypassGovernanceRetention permission có thể delete
  - Dùng cho: business data cần "soft immutable" cho compliance
  - Tradeoff: linh hoạt, nhưng cần policy chặt cho ai có bypass permission

COMPLIANCE mode:
  - KHÔNG AI delete được trong retention period
  - Kể cả AWS root account của bạn
  - Kể cả AWS support
  - Dùng cho: legal hold, regulatory data (SEC 17a-4, FINRA, CFTC)
  - Tradeoff: rigid — đặt nhầm retention 10 năm = chờ 10 năm
Lưu ý production: COMPLIANCE mode là "irrevocable" — bạn KHÔNG đặt retention period dài hơn cần thiết. Đặt nhầm 50 năm với 1 TB data = ràng buộc lưu trữ 50 năm, không refund được. Tốt nhất: prototype với GOVERNANCE trước, chỉ switch sang COMPLIANCE khi process đã stable.

Tiered storage strategy (cost-aware)

┌──────────────────────────────────────────────────────────────┐
│ Hot tier   │ PostgreSQL append-only         │ 0-90 ngày      │
│ (active)   │ Hash chained                   │ Fast query     │
│            │ Cost: high                     │ Real-time ops  │
├──────────────────────────────────────────────────────────────┤
│ Warm tier  │ S3 Standard + Object Lock      │ 90 ngày - 3yr  │
│            │ Parquet format (compressed)    │ Athena query   │
│            │ GOVERNANCE mode                │ Compliance     │
├──────────────────────────────────────────────────────────────┤
│ Cold tier  │ S3 Glacier + Object Lock       │ 3yr - 10yr     │
│            │ COMPLIANCE mode (truly WORM)   │ 3-5h retrieval │
│            │ Cost: ~$0.004/GB/month         │ Legal/audit    │
└──────────────────────────────────────────────────────────────┘

Tiết kiệm cost: S3 Glacier ~ 1/6 cost S3 Standard.
Long-term audit 10 năm dùng Glacier — không ai query real-time cần.

5. Non-repudiation stack CORE

"Non-repudiation" = người làm action không thể phủ nhận đã làm. Cần 5 lớp chứng cứ:

LayerCâu trả lờiCơ chế
1. AuthenticationAi login?JWT, session, MFA — proof of identity
2. AuthorizationQuyền gì?RBAC/ABAC logs — proof of permission tại thời điểm
3. Action loggingĐã làm gì?Audit event với before/after state
4. Digital signatureUser đã approve?Ký bằng private key — chỉ user có key đó
5. Timestamp Authority (TSA)Chính xác lúc nào?RFC 3161 TSA service ký timestamp — không dùng server clock
Vì sao server clock không đủ: Server clock có thể bị admin chỉnh (NTP skew, manual set). TSA là third-party trusted service ký timestamp bằng cert chuẩn — không thể backdate. Đối với legal evidence quan trọng (chữ ký hợp đồng, transaction sealed), dùng TSA bắt buộc.

6. Audit Event Schema — WHO / WHAT / WHEN / WHERE CORE

@Entity
@Table(name = "audit_events",
       indexes = {
           @Index(name = "idx_user_time", columnList = "userId,eventTime"),
           @Index(name = "idx_entity", columnList = "entityType,entityId")
       })
@Immutable          // Hibernate — ngăn update khi đã trong session
public class AuditEvent {

    @Id
    private String eventId;             // UUID v4

    // WHO
    private String userId;
    private String userEmail;
    private String sessionId;
    private String ipAddress;
    private String userAgent;

    // WHAT
    private String action;              // VIEW_PATIENT, UPDATE_ORDER, EXPORT_DATA
    private String entityType;          // Patient, Order, Document
    private String entityId;

    // STATE CHANGE — quan trọng cho audit "ai đã sửa gì"
    @Column(columnDefinition = "jsonb")
    @Type(JsonBinaryType.class)
    private JsonNode beforeState;       // State trước (null nếu là CREATE/READ)
    @Column(columnDefinition = "jsonb")
    @Type(JsonBinaryType.class)
    private JsonNode afterState;        // State sau (null nếu là DELETE/READ)

    // CONTEXT
    private String requestId;           // Correlation ID
    private String traceId;             // OpenTelemetry
    private String serviceId;           // Service nào tạo event

    // WHEN
    private Instant eventTime;          // Business time (caller-supplied)
    private Instant serverTime;         // Server-stamped — KHÔNG trust client

    // INTEGRITY
    private String previousHash;
    private String currentHash;

    // OUTCOME
    private String result;              // SUCCESS / FAILURE / DENIED
    private String errorCode;
    private String errorMessage;

    // protected default constructor for JPA
    protected AuditEvent() {}
    // ... builder, getters ONLY (no setters — immutable)
}

Quy tắc thiết kế schema

  1. Before/After state quan trọng cho UPDATE. Chỉ log "user A updated patient_999" KHÔNG đủ — cần biết field nào, từ giá trị gì sang giá trị gì.
  2. 2 timestamp: eventTime (business) + serverTime (truth). Client có thể gửi eventTime sai (delay, manipulation). serverTime là server-stamped, dùng làm thứ tự thật.
  3. Result phải có FAILURE và DENIED. Failed attempts cũng audit — brute force / unauthorized attempt phải trace được.
  4. NO setter public, immutable. JPA @Immutable + Lombok không gen setter. Constructor private, factory AuditEvent.create().

❌ Schema anti-pattern

  • Lưu sensitive data plain trong before/after state. Password, CVV, SSN, medical record number — không log raw. Mask/redact: "password": "[REDACTED]", hoặc tokenize.
  • Không có serverTime, chỉ có eventTime. Client có thể fake. Audit thứ tự không đáng tin.
  • Result thiếu DENIED. Chỉ log SUCCESS/FAILURE → mất permission-check failures, không detect được unauthorized access attempts.

7. Audit Service implementation CORE

Service đầy đủ kết hợp hash chain + async replication ra Kafka cho immutable backup:

@Service
@Slf4j
public class AuditService {

    private final AuditChainService chainService;     // hash chain logic ở Section 3
    private final KafkaTemplate<String, AuditEvent> kafkaTemplate;
    private final AlertService alertService;

    @Value("${audit.kafka-topic}")
    private String auditTopic;

    /**
     * Main entry point — gọi từ AOP aspect hoặc trực tiếp service.
     * Strategy:
     *   1. Synchronous: append vào DB với hash chain (đảm bảo integrity)
     *   2. Async: publish ra Kafka cho immutable replication
     *   3. Periodic: dump từ Kafka sang S3 Object Lock
     */
    public AuditEvent recordEvent(AuditEventRequest request) {
        // Mask sensitive fields TRƯỚC khi lưu
        AuditEventRequest masked = maskSensitive(request);

        // Append vào DB với hash chain (synchronous)
        AuditEvent event = chainService.append(masked);

        // Replicate ra Kafka async — không block business operation
        kafkaTemplate.send(auditTopic, event.getEventId(), event)
            .whenComplete((res, ex) -> {
                if (ex != null) {
                    // Kafka replication fail không break DB write,
                    // nhưng phải alert security team
                    log.error("Audit Kafka replication failed: eventId={}",
                              event.getEventId(), ex);
                    alertService.sendCritical(
                        "Audit replication failure",
                        "Event " + event.getEventId() + " not in Kafka");
                }
            });

        return event;
    }

    private AuditEventRequest maskSensitive(AuditEventRequest req) {
        // Redact known sensitive fields
        JsonNode masked = redactFields(req.getBeforeState(),
            "password", "ssn", "cardNumber", "cvv", "privateKey");
        return req.withBeforeState(masked);
    }
}

Verify integrity — scheduled job

@Component
public class AuditIntegrityScheduler {

    private final AuditChainService chainService;
    private final AlertService alertService;
    private final AuditIntegrityReportRepository reportRepo;

    /**
     * Chạy mỗi đêm — verify audit chain của ngày hôm trước.
     * Phát hiện tampering trong vòng 24h.
     */
    @Scheduled(cron = "0 0 3 * * *")  // 3 AM daily
    public void verifyYesterday() {
        Instant yesterday = LocalDate.now().minusDays(1)
            .atStartOfDay(ZoneOffset.UTC).toInstant();
        Instant today = LocalDate.now()
            .atStartOfDay(ZoneOffset.UTC).toInstant();

        AuditIntegrityReport report = chainService.verifyChain(yesterday, today);
        reportRepo.save(report);

        if (!report.isIntact()) {
            // CRITICAL alert — audit tampering detected
            alertService.sendCritical(
                "AUDIT TAMPERING DETECTED",
                "Period: " + yesterday + " to " + today +
                "\nViolations:\n" + String.join("\n", report.getViolations())
            );
        }
    }
}

8. AOP tự động capture HIGH-SIGNAL

Vì sao AOP thay vì gọi auditService.record() ở mỗi service method? Vì developer quên gọi = security gap. AOP cross-cutting concern không pollute business code.

// Annotation đánh dấu method cần audit
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Auditable {
    String action();
    String entityType();
    String entityIdParam() default "";  // tên param chứa entityId
    boolean captureBeforeState() default false;
}

@Aspect
@Component
@Slf4j
public class AuditAspect {

    private final AuditService auditService;
    private final HttpServletRequest httpRequest;
    private final BeforeStateCaptor stateCaptor;
    private final ObjectMapper objectMapper;

    @Around("@annotation(auditable)")
    public Object around(ProceedingJoinPoint pjp, Auditable auditable) throws Throwable {
        String entityId = extractEntityId(pjp, auditable.entityIdParam());

        // Capture state TRƯỚC khi method chạy (cho UPDATE/DELETE)
        JsonNode beforeState = null;
        if (auditable.captureBeforeState()) {
            beforeState = stateCaptor.snapshot(auditable.entityType(), entityId);
        }

        Throwable error = null;
        Object result = null;
        try {
            result = pjp.proceed();
            return result;
        } catch (AccessDeniedException ex) {
            error = ex;
            throw ex;
        } catch (Throwable ex) {
            error = ex;
            throw ex;
        } finally {
            // Luôn ghi audit — kể cả khi method fail
            // Lý do: failed attempts cần audit (brute force, unauthorized)
            try {
                String userId = currentUserId();
                AuditEventRequest req = AuditEventRequest.builder()
                    .userId(userId)
                    .action(auditable.action())
                    .entityType(auditable.entityType())
                    .entityId(entityId)
                    .beforeState(beforeState)
                    .afterState(result != null ? objectMapper.valueToTree(result) : null)
                    .ipAddress(httpRequest.getRemoteAddr())
                    .userAgent(httpRequest.getHeader("User-Agent"))
                    .requestId(MDC.get("requestId"))
                    .eventTime(Instant.now())
                    .result(error == null ? "SUCCESS"
                          : error instanceof AccessDeniedException ? "DENIED" : "FAILURE")
                    .errorMessage(error != null ? error.getMessage() : null)
                    .build();

                auditService.recordEvent(req);
            } catch (Exception auditEx) {
                // Audit fail là CRITICAL — nhưng phải quyết định fail-open hay fail-closed
                // Xem Section "Fail-open vs fail-closed"
                log.error("AUDIT FAILURE - eventId would have been for action={}",
                          auditable.action(), auditEx);
                // CHỌN fail-closed cho action quan trọng:
                if (isCriticalAction(auditable.action()) && error == null) {
                    throw new AuditFailureException(
                        "Audit failed for critical operation — rolling back");
                }
            }
        }
    }
}

Usage

@Service
public class PatientService {

    @Auditable(action = "VIEW_PATIENT", entityType = "Patient",
               entityIdParam = "patientId")
    public Patient getPatient(String patientId) {
        return patientRepo.findById(patientId).orElseThrow();
    }

    @Auditable(action = "UPDATE_PATIENT", entityType = "Patient",
               entityIdParam = "patientId", captureBeforeState = true)
    public Patient updatePatient(String patientId, PatientUpdate update) {
        Patient p = patientRepo.findById(patientId).orElseThrow();
        p.applyUpdate(update);
        return patientRepo.save(p);
    }
}
Lợi ích AOP approach: ① Business code clean — không có audit call lẫn vào. ② Không thể quên — annotation = audit guaranteed. ③ Consistent format — mọi audit event có cùng schema, tự động ip/userAgent/requestId.

⚠️ 5 anti-patterns kinh điển CORE

❌ #1: Audit log trong cùng DB với business data

Symptom: DBA có quyền DELETE cả audit_log và business_data → bypass dễ. Backup restore → wipe audit. Single point of failure.

Fix: Separate audit DB (khác schema ít nhất, khác instance lý tưởng). Revoke UPDATE/DELETE từ app user. Replicate ra Kafka + WORM storage.

❌ #2: Không capture before/after state

// ❌ Chỉ log action — không evidence
"User A updated patient_999 at 14:00"
// Sửa cái gì? Không biết.

// ✅ Log đủ state
"User A updated patient_999 at 14:00
 BEFORE: {diagnosis: 'flu',     prescription: 'rest'}
 AFTER:  {diagnosis: 'covid-19', prescription: 'paxlovid'}"

Symptom: Khi investigate, không thể reconstruct timeline. Forensic team phải đoán mò.

❌ #3: Audit blocking trong hot path

// ❌ Audit sync trong transaction — chậm, risk timeout
@Transactional
public void doBusinessOp() {
    update();
    auditService.recordEvent(...);  // DB write blocking
    externalApi.call();              // Slow audit → toàn bộ slow
}

// ✅ Outbox pattern — audit event trong cùng transaction,
// nhưng publish async sau commit
@Transactional
public void doBusinessOp() {
    update();
    outboxRepo.save(AuditOutboxEntry.from(event)); // same tx, fast
}
// Background processor đọc outbox → publish to audit pipeline

❌ #4: Không alert khi audit system fail

Symptom: Audit Kafka down → silent fail. Attacker có window để cover tracks. Compliance officer phát hiện sau 3 tháng.

Fix: Monitor audit pipeline health như business critical: write lag, success rate, integrity verify job. Alert nếu lag > threshold. Cho critical ops (payment, medical record update) cân nhắc fail-closed.

❌ #5: Sensitive data không encrypt/mask

// ❌ Audit log = data breach waiting to happen
{
  "action": "RESET_PASSWORD",
  "beforeState": {"hash": "$2a$10$RealBcryptHash..."},
  "afterState":  {"newPassword": "User'sActualPassword123"}
}
// Audit log leak = mọi password leak

// ✅ Mask/redact sensitive
{
  "action": "RESET_PASSWORD",
  "beforeState": {"hash": "[REDACTED]"},
  "afterState":  {"newPassword": "[REDACTED]"}
}

Quy tắc: Whitelist field log, blacklist mọi field nghi vấn. Sensitive fields known: password, ssn, cardNumber, cvv, privateKey, accessToken, refreshToken.

🔨 Anti-pattern nguy hiểm nhất?

#5 (Sensitive data unmasked). Vì: audit log thường lưu lâu (5-10 năm), backup nhiều nơi, access nhiều team → 1 leak = catastrophic. Không như password hash (chỉ valid khi DB live), audit dump leak vĩnh viễn — mọi password trong đó cracked sớm muộn.

⚖️ Fail-open vs Fail-closed — Trade-off senior HIGH-SIGNAL

Audit system down. Có nên block business operation không?

ModeBehavior khi audit failTrade-off
Fail-open Cho phép operation tiếp tục, log error + Availability cao
− Có window không audit, attacker có thể abuse
Fail-closed Reject operation, throw exception + Security strict — không action nào không audit
− Availability giảm — audit fail = service fail

Quy tắc thực tế

  • Critical financial/medical ops: fail-closed. Payment, prescription, contract sign → audit phải work hoặc không cho làm.
  • Read operations: fail-open OK thường. View patient list — audit fail không nên block.
  • Bulk operations: fail-closed. Export 10K records, send 100K email — phải audit.
  • Default mặc định: fail-open cho dev/staging, fail-closed cho production sensitive ops.
Hybrid approach: Có audit pipeline 2 lớp — local DB (sync, fail-closed cho critical) + Kafka replication (async, fail-open OK). Local DB write phải succeed; Kafka publish có thể retry. Cân bằng giữa availability và auditability.

🎯 3-tier interview framework

Tier 1 — Surface (Junior)

Q1: Audit log là gì? Khác application log như thế nào?

Trả lời: "Audit log là bản ghi compliance — ai làm gì, với data nào, lúc nào, từ đâu. Khác application log ở 3 tính chất: immutable (không sửa/xoá được), tamper-evident (sửa thì detect được), non-repudiable (không phủ nhận được). Application log để debug, có thể delete/rotate. Audit log để compliance/legal, không bao giờ xoá trong retention period."

Q2: Cho ví dụ regulation yêu cầu audit log.

Trả lời: "GDPR Article 30 — EU personal data processing. HIPAA — US healthcare data, 6 năm retention. SOX — US financial reporting, 7 năm. PCI-DSS 10.7 — payment card processing, 1 năm online + archive. Mỗi cái có retention requirement riêng, design storage tiered theo đó."

Tier 2 — Deep Dive (Mid vs Senior)

Q3: Implement tamper-evident audit log. DBA có bypass được không?

"Implement hash chain: mỗi event chứa HMAC-SHA256 của data + hash của event trước, dùng secret key. Verify job chạy daily phát hiện 2 loại tampering: ① event bị sửa (recompute HMAC ≠ stored hash), ② event bị xoá (prev_hash mismatch giữa các event liền kề).

Vì sao HMAC thay SHA256: SHA256 ai cũng compute, DBA sửa data → recompute → chain valid. HMAC cần secret key — DBA không có key thì không re-sign được.

DBA bypass: nếu DBA có cả DB access và HSM secret key, vẫn bypass được — re-sign toàn bộ chain. Defence: tách quyền — DBA không có HSM access. Plus replicate ra WORM storage external (S3 Object Lock COMPLIANCE mode) bên ngoài DBA control. Hash chain detect ở DB-level; WORM ngăn ở storage-level."

Q4: Compliance audit 10 năm retention. Cost optimize thế nào?

"Tiered storage: ① Hot tier (PostgreSQL append-only, hash chain) cho 0-90 ngày — fast query cho operations. ② Warm tier (S3 Standard + Object Lock GOVERNANCE, Parquet) cho 90 ngày - 3 năm — Athena SQL query cho compliance team. ③ Cold tier (S3 Glacier Deep Archive + Object Lock COMPLIANCE) cho 3-10 năm — truly immutable.

Cost: S3 Glacier Deep Archive ~$0.001/GB/month vs S3 Standard ~$0.023/GB/month = tiết kiệm ~95%. Trade-off: retrieval Glacier 3-5 giờ, nhưng 10-year data không cần real-time.

Important: Glacier COMPLIANCE mode irrevocable — đặt retention 10 năm = ràng buộc 10 năm thật. Start với GOVERNANCE mode, switch COMPLIANCE chỉ khi process stable."

Tier 3 — Architecture (Senior)

Q5: Design audit system cho healthcare SaaS, 50M events/ngày, HIPAA compliance, audit query SLA 2 giờ.

Phút 1 — Scope & estimation:
  50M events/day = 580 events/sec average, peak ~2000/sec.
  Size: ~1 KB/event × 50M = 50 GB/day = ~18 TB/year.
  10-year retention = ~180 TB cold storage.

Phút 2 — Write path:
  Application → @Auditable AOP → AuditService
    → 1) Local DB (Postgres append-only, hash chained) — sync
    → 2) Kafka audit topic — async replicate
  Kafka consumer → S3 Object Lock (GOVERNANCE mode for warm tier)
  Lifecycle policy → tự động chuyển S3 → Glacier Deep Archive sau 3 năm

Phút 3 — Read path (2h SLA):
  Compliance query → AWS Athena trên S3 Parquet
  Partition: year/month/day/userId-bucket → query 1 user trong 1 ngày
    chỉ scan 1 partition (~MB), không phải toàn 50 GB
  Pre-built query templates: "all access by user X", "all access to patient Y"

Phút 4 — Integrity & non-repudiation:
  Scheduled verify job (daily 3 AM) — verify chain của ngày trước
  HMAC secret trong AWS KMS, key rotation hàng năm
  Cho legal-critical events: thêm RFC 3161 TSA timestamp
  Alert nếu verify fail — security incident response

Phút 5 — Trade-offs:
  + Tiered storage: cost optimize cho long retention
  + Async Kafka: không block business, có audit replication
  + Athena query: SQL trên S3, không cần data warehouse
  - Async replication có lag — local DB là source of truth ngắn hạn
  - Hash chain bottleneck nếu single-threaded write — dùng partitioned chain
    (chain per tenant) cho horizontal scale
  - Glacier COMPLIANCE mode irrevocable — cẩn thận retention setup

🔨 Lab 1 — Hash chain tamper detector (3 giờ)

Mục tiêu: Tự mắt thấy chain break khi tamper data.

Steps:

  1. Tạo Spring Boot project. AuditEvent entity với prevHash + currentHash.
  2. Implement HMAC-SHA256 với secret key trong app.yml (dev only — production phải HSM/KMS).
  3. Implement AuditChainService.append() với chain logic. Insert 10 sự kiện sequential.
  4. Verify job: AuditIntegrityReport scanning toàn bộ chain. Đầu tiên chạy — phải pass (intact = true).
  5. SQL: UPDATE audit_events SET action = 'XXX' WHERE eventId = '...' — tamper 1 event.
  6. Re-run verify. Phải thấy violation với eventId đó.
  7. Bonus: SQL DELETE FROM audit_events WHERE id = N — xoá event giữa chain. Re-run verify. Phải thấy chain break ở event sau (prev_hash mismatch).

Acceptance: Reproduce 2 scenario tampering (modify + delete) + verify job phát hiện. Commit lên GitHub.

🔨 Lab 2 — AOP audit + Outbox replication (4 giờ)

Mục tiêu: Audit không pollute business code + replicate async.

Steps:

  1. Tạo @Auditable annotation + AuditAspect.
  2. Tạo PatientService với 3 method: viewPatient, updatePatient, deletePatient — annotate @Auditable.
  3. Setup outbox table audit_outbox. AuditAspect lưu vào outbox (cùng transaction với business).
  4. Background job (Scheduled) poll outbox, publish ra Kafka topic audit-events. Mark outbox row published.
  5. Kafka consumer (separate microservice) — đọc audit-events, ghi vào S3 (dùng LocalStack cho local test).
  6. Test scenarios:
    • Normal: viewPatient → outbox + Kafka + S3 (eventually consistent).
    • Failure: updatePatient throw exception → audit vẫn ghi (FAILURE result).
    • Crash: kill Kafka mid-replication → restart → outbox poll lại → eventual delivery.

Acceptance: 3 scenario pass + business code không có audit call nào (chỉ annotation). README mô tả flow.

🧠 Feynman test — Verify hiểu thật

🎓 Giải thích cho dev chưa biết — không nhìn file

Đóng tab. Trả lời 5 câu, < 30 giây mỗi câu:

  1. "Phân biệt audit log với application log?" — Application log để debug, có thể delete/rotate. Audit log để compliance/legal, có 3 tính chất bắt buộc: immutable, tamper-evident, non-repudiable. Không bao giờ xoá trong retention period.
  2. "Vì sao HMAC thay SHA256 trong hash chain?" — SHA256 ai cũng compute. DBA sửa data → recompute SHA256 → chain valid → không detect được. HMAC cần secret key — DBA không có key thì không re-sign được. Secret key lưu HSM/KMS.
  3. "S3 Object Lock GOVERNANCE vs COMPLIANCE khác gì?" — GOVERNANCE: admin với bypass permission có thể delete. COMPLIANCE: KHÔNG AI delete được trong retention period, kể cả AWS root account. COMPLIANCE dùng cho regulatory data thật, GOVERNANCE cho business immutable mềm.
  4. "Vì sao AOP capture audit thay vì gọi auditService trong business code?" — Developer quên gọi = security gap. AOP là cross-cutting concern, đảm bảo mọi method có @Auditable đều được audit. Plus: business code clean, không lẫn audit logic.
  5. "Fail-open vs fail-closed khi audit fail?" — Fail-open: cho phép operation, log error → availability cao nhưng có window không audit. Fail-closed: reject operation → security strict nhưng giảm availability. Critical ops (payment, medical) → fail-closed. Read ops → fail-open OK.

Bí 1 câu → đó là chỗ hiểu giả. Quay lại section tương ứng.
