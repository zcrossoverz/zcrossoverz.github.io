#01 - Java Concurrency: Thread, Executor, CompletableFuture, Race Condition, Deadlock

Concurrency là chủ đề gần như chắc chắn xuất hiện trong phỏng vấn senior. Mục tiêu của bài này không phải nhớ tên class, mà là xây được mô hình: thread chia sẻ heap, stack riêng; shared mutable state gây race; lock sai gây deadlock; thread pool sai gây outage.

Concurrency Là Bài Toán Về Shared State Và Scheduling

Một request đơn giản thường chạy tuần tự nên dễ hiểu. Nhưng backend production có hàng trăm request cùng lúc, thread pool, database connection pool, scheduler, consumer, retry job, callback async. Khi nhiều thread cùng đọc/ghi một vùng dữ liệu, kết quả có thể phụ thuộc vào thứ tự OS scheduler cho thread chạy. Đó là lý do bug concurrency thường khó reproduce: code không đổi, nhưng interleaving giữa thread thay đổi.

Để học chắc, hãy tách bốn lớp: thread là gì, state nằm ở đâu, synchronization tạo guarantee gì, và thread pool điều tiết tải thế nào. Nếu bạn chỉ học API như Thread, ExecutorService, CompletableFuture mà không hiểu bốn lớp này, bạn sẽ dùng đúng cú pháp nhưng sai kiến trúc.

Kiến Trúc Học Chống Hiểu Giả

Layer 1

Interleaving trước API

Trước khi sửa code, hãy vẽ timeline hai thread đọc/ghi cùng state. Đây là cách nhìn ra race condition.

Layer 2

Guarantee trước keyword

synchronized, volatile, AtomicInteger không phải "cho thread-safe". Mỗi thứ cho guarantee khác nhau.

Layer 3

Backpressure trước throughput

Thread pool không chỉ để chạy nhanh hơn. Nó còn để giới hạn tải, queue và từ chối task khi hệ thống quá tải.

Layer 4

Debug bằng evidence

Thread dump, pool metrics, queue depth, blocked time và log thread name đáng tin hơn phỏng đoán.

1. Bài Toán Thực Tế - Flash Sale Bán Quá Số Lượng Tồn Kho

Một hệ thống bán vé workshop mở flash sale 100 ghế. Lúc cao điểm, 2.000 request checkout đến trong vài giây. Sau sự kiện, hệ thống phát hiện đã bán 127 vé cho một workshop chỉ có 100 ghế.

Sai: read-modify-write không atomic
@Service
public class SeatService {
    private int remainingSeats = 100;

    public boolean reserveOne() {
        if (remainingSeats > 0) {
            remainingSeats--;
            return true;
        }
        return false;
    }
}
Đúng hơn trong single JVM
@Service
public class SeatService {
    private final AtomicInteger remainingSeats = new AtomicInteger(100);

    public boolean reserveOne() {
        while (true) {
            int current = remainingSeats.get();
            if (current <= 0) return false;
            if (remainingSeats.compareAndSet(current, current - 1)) {
                return true;
            }
        }
    }
}
Race condition không cần code phức tạp. Chỉ cần shared mutable state + nhiều thread + có write.

Trong production nhiều instance, AtomicInteger vẫn chưa đủ vì mỗi JVM có counter riêng. Lúc đó cần đưa consistency về nơi share giữa instance: database lock, optimistic lock trên row, Redis atomic command, hoặc kiến trúc reservation có idempotency.

2. Nền Tảng Cần Nắm

2.1 Thread Và Memory

Thread là đơn vị thực thi được OS scheduler cấp CPU time. Trong một JVM process, các thread chia sẻ heap nhưng mỗi thread có stack riêng. Local variable primitive nằm trên stack nên an toàn theo thread. Object nằm trên heap; nếu nhiều thread giữ reference tới cùng object và object đó mutable, bạn có shared mutable state.

Muốn hết race condition, loại bỏ ít nhất một yếu tố: shared state, mutability, concurrent access, hoặc write.

2.2 Thread Lifecycle - Đọc Thread Dump Phải Biết State

Khi production treo, bạn thường không có thời gian đọc lại tài liệu. Bạn lấy thread dump và phải hiểu ngay thread đang chạy, chờ lock, ngủ, hay chờ I/O. Vì vậy lifecycle state không phải lý thuyết phụ, nó là ngôn ngữ để debug incident.

Runnable, Callable, Future: ba mức trừu tượng

Runnable là công việc không trả kết quả và không throw checked exception. Callable<T> trả kết quả và có thể throw exception. Future<T> là handle đại diện cho kết quả tương lai, nhưng API cũ khá nghèo: get() block, cancel khó compose, error handling không đẹp.

Runnable
Runnable task = () -> sendEmail(orderId);
executor.execute(task);
Callable + Future
Callable<Price> task = () -> priceClient.getPrice(sku);
Future<Price> future = executor.submit(task);
Price price = future.get(300, TimeUnit.MILLISECONDS);

Trong code hiện đại, CompletableFuture thường tốt hơn Future khi bạn cần compose nhiều bước async. Nhưng nếu chỉ submit task nền đơn giản và cần cancel/get timeout, Future vẫn đủ.

2.4 Synchronization Tools - Chọn Công Cụ Theo Guarantee

Điểm senior nằm ở chỗ không nói chung chung "dùng lock". Bạn phải nói rõ cần guarantee gì: visibility, atomicity, mutual exclusion, condition waiting, fairness, timeout, hay nhiều reader song song.

Công cụGuarantee chínhDùng khiBẫy
volatileVisibility + orderingFlag dừng worker, publish immutable snapshotKhông atomic cho count++
synchronizedMutual exclusion + visibilityCritical section ngắn, invariant nhiều fieldKhông timeout, dễ lock sai object
ReentrantLockExplicit lock, timeout, interruptibleCần tryLock, deadlock avoidance, fairnessPhải unlock trong finally
ReadWriteLockNhiều reader, một writerRead-heavy in-memory registryWriter starvation nếu cấu hình/usage sai
StampedLockOptimistic readRead cực nhiều, write ít, low-level optimizationKhông reentrant, API khó dùng hơn
SemaphoreGiới hạn số permitLimit concurrent calls tới providerQuên release làm hệ thống tự bóp nghẹt
AtomicXxxCAS atomic updateCounter, state transition đơn giảnCAS spin tốn CPU khi contention rất cao

L1 — Biết tên

Đọc bảng trên và nhớ được volatile, synchronized, AtomicInteger "đều giúp thread-safe". Symptom hiểu giả: chọn công cụ theo thói quen, hỏi "tại sao volatile không đủ cho count++" thì ấp úng.

L2 — Explain được

Tách rõ visibility (thread khác có thấy giá trị mới không) khỏi atomicity (thao tác có bị chen ngang không). Giải thích: volatile cho visibility nhưng count++ vẫn race vì gồm 3 bước; AtomicInteger dùng CAS nên atomic mà không cần lock.

L3 — Apply được

Đứng trước một bug thật, chọn đúng guarantee tối thiểu cần: flag dừng worker → volatile; counter thống kê → AtomicLong; invariant nhiều field → synchronized/ReentrantLock; cần timeout để né deadlock → tryLock. Giải thích được trade-off throughput của mỗi lựa chọn.

🔧 Counter-example — khi nào KHÔNG nên thêm lock

Lock không phải lúc nào cũng đúng. Ba trường hợp thêm lock là sai hướng:

  • State vốn không chia sẻ: biến local trong method đã thread-safe sẵn (mỗi thread có stack riêng). Thêm synchronized chỉ làm chậm, không tăng an toàn.
  • Read-only sau khi khởi tạo: immutable object (mọi field final, không setter) không cần đồng bộ — đây là cách rẻ nhất để thread-safe. Bọc lock quanh nó là thừa.
  • Chỉ cần atomic 1 biến: dùng synchronized cho một counter++ đơn giản nặng hơn AtomicInteger (lock vs CAS). Chọn lock ở đây = bỏ phí.

Quy tắc: trước khi thêm lock, hỏi "có thể loại bỏ shared state / mutability thay vì khóa nó không?". Lock là phương án cuối, không phải đầu tiên.

synchronized vs ReentrantLock

synchronized: đơn giản
private final Object lock = new Object();

public void addLine(LineItem item) {
    synchronized (lock) {
        items.add(item);
        total = total.add(item.price());
    }
}
ReentrantLock: có timeout
private final ReentrantLock lock = new ReentrantLock();

public boolean addLine(LineItem item) throws InterruptedException {
    if (!lock.tryLock(200, TimeUnit.MILLISECONDS)) {
        return false;
    }
    try {
        items.add(item);
        total = total.add(item.price());
        return true;
    } finally {
        lock.unlock();
    }
}

Semaphore để giới hạn concurrency tới external provider

Semaphore không bảo vệ shared state như lock thông thường; nó giới hạn số operation cùng lúc. Đây là pattern thực tế khi provider chỉ cho 50 concurrent requests, hoặc khi một bước CPU-heavy không được chạy quá nhiều song song.

public class ProviderClient {
    private final Semaphore permits = new Semaphore(50);

    public ProviderResponse call(ProviderRequest request) {
        boolean acquired = false;
        try {
            acquired = permits.tryAcquire(200, TimeUnit.MILLISECONDS);
            if (!acquired) {
                throw new ProviderBusyException();
            }
            return httpClient.call(request);
        } catch (InterruptedException ex) {
            Thread.currentThread().interrupt();
            throw new OperationInterruptedException(ex);
        } finally {
            if (acquired) {
                permits.release();
            }
        }
    }
}

ReadWriteLock cho registry read-heavy

public class FeatureFlagRegistry {
    private final ReadWriteLock rw = new ReentrantReadWriteLock();
    private Map<String, Boolean> flags = Map.of();

    public boolean isEnabled(String key) {
        rw.readLock().lock();
        try {
            return flags.getOrDefault(key, false);
        } finally {
            rw.readLock().unlock();
        }
    }

    public void reload(Map<String, Boolean> newFlags) {
        rw.writeLock().lock();
        try {
            flags = Map.copyOf(newFlags);
        } finally {
            rw.writeLock().unlock();
        }
    }
}

Trong nhiều case, volatile reference tới immutable map còn đơn giản hơn ReadWriteLock. Đây là judgment quan trọng: lock mạnh hơn không đồng nghĩa thiết kế tốt hơn.

2.5 ThreadPoolExecutor CORE ★

⏸ Pause & Predict
Bạn config new ThreadPoolExecutor(core=10, max=50, queue=LinkedBlockingQueue()) (queue mặc định không giới hạn). Lúc cao điểm có 1.000 task đến cùng lúc:
  1. Pool sẽ chạy bao nhiêu thread? 10, 50, hay khác? Vì sao maxPoolSize=50 gần như vô dụng ở đây?
  2. 1.000 task chưa chạy đang nằm ở đâu? Điều gì xảy ra với memory nếu task tiếp tục đến?
  3. Đổi sang ArrayBlockingQueue(200) + CallerRunsPolicy thì hành vi thay đổi thế nào khi quá tải?
Bẫy kinh điển: với unbounded queue, pool KHÔNG bao giờ vượt corePoolSize → max bị bỏ qua, và queue phình tới OOM. Đáp án đầy đủ ở 2.5 và 3.1.

Thread pool không tạo thread vô hạn. Nó có core threads, queue, max threads và rejected handler. Câu hỏi phỏng vấn quan trọng: khi nào pool tạo thêm thread? Với nhiều queue implementation, pool sẽ queue task trước, chỉ tạo thêm thread khi queue đầy.

Queue không giới hạn che giấu overload cho tới khi heap đầy.
Rule of thumb

CPU-bound: số thread xấp xỉ số core. I/O-bound: có thể nhiều hơn core vì thread chờ network/DB. Nhưng con số đúng phải dựa vào metrics: queue depth, active count, task duration, rejected count.

Thread pool sizing: công thức chỉ là điểm bắt đầu

Công thức phổ biến cho I/O-bound là threads = cores * (1 + waitTime / computeTime). Nếu request mất 100ms, trong đó 80ms chờ I/O và 20ms CPU, ratio là 4, pool có thể bắt đầu quanh cores * 5. Nhưng đây chỉ là điểm xuất phát. Production tuning phải dựa vào metric.

Queue sizeTask chờ xử lýTăng đều nghĩa là capacity thấp hơn input rate

RejectedExecutionHandler không phải chi tiết phụ

Khi pool và queue đều đầy, rejected handler quyết định hệ thống fail thế nào. Đây là điểm kiến trúc, không phải config nhỏ.

  • AbortPolicy: throw exception. Hợp với API cần fail fast và trả 429/503.
  • CallerRunsPolicy: caller tự chạy task. Tạo backpressure tự nhiên nhưng có thể làm HTTP thread chậm.
  • DiscardPolicy: bỏ task im lặng. Hiếm khi an toàn.
  • DiscardOldestPolicy: bỏ task cũ nhất trong queue. Chỉ hợp với dữ liệu realtime có thể bỏ cũ.

2.6 Concurrent Collections - Đừng Tự Lock Khi JDK Đã Có Cấu Trúc Phù Hợp

Concurrency không chỉ là lock. JDK cung cấp collection được thiết kế cho access đồng thời. Dùng đúng collection giúp code rõ hơn và giảm critical section tự viết.

ConcurrentHashMap: atomic theo key

public class ProductViewCounter {
    private final ConcurrentHashMap<String, LongAdder> counters = new ConcurrentHashMap<>();

    public void increment(String productId) {
        counters.computeIfAbsent(productId, ignored -> new LongAdder())
                .increment();
    }

    public long count(String productId) {
        LongAdder counter = counters.get(productId);
        return counter == null ? 0L : counter.sum();
    }
}

computeIfAbsent tránh pattern lỗi: check rồi put không atomic. Tuy vậy function truyền vào computeIfAbsent nên nhanh và không có side effect nặng, vì nó chạy trong cơ chế đồng bộ nội bộ của map.

Producer-consumer với BlockingQueue

public class ThumbnailPipeline {
    private final BlockingQueue<ImageJob> queue = new ArrayBlockingQueue<>(5_000);
    private final ExecutorService workers = Executors.newFixedThreadPool(8);

    public boolean submit(ImageJob job) {
        return queue.offer(job); // false nếu queue đầy, caller có thể trả 429
    }

    public void start() {
        for (int i = 0; i < 8; i++) {
            workers.submit(() -> {
                while (!Thread.currentThread().isInterrupted()) {
                    ImageJob job = queue.take();
                    process(job);
                }
            });
        }
    }
}

Queue bounded là một hình thức bảo vệ heap. Queue unbounded có vẻ "mượt" lúc đầu nhưng che giấu overload cho tới khi OOM.

BlockingQueue Patterns: Backpressure, Batch, Poison Pill

BlockingQueue không chỉ là chỗ để nhét task. Nó là boundary giữa producer và consumer. Nếu boundary này không có capacity rõ ràng, producer có thể đẩy nhanh hơn khả năng xử lý và biến queue thành memory leak.

PatternDùng khiĐiểm cần nhớ
Bounded queue + offerAPI/job submit cần fail fast khi quá tảioffer trả false, caller quyết định retry/429/drop
Bounded queue + putProducer nội bộ có thể bị chặnKhông dùng trên request thread nếu blocking làm hết HTTP workers
drainTo batchConsumer xử lý batch hiệu quả hơn từng itemCần giới hạn batch size để tránh latency spike
Poison pillDừng worker gracefulCần số poison pill tương ứng số consumer
Separate queuesWorkload priority khác nhauTránh job chậm làm nghẽn job nhanh
 
public final class BatchConsumer implements Runnable {
    private final BlockingQueue<ImageJob> queue;
    private final List<ImageJob> buffer = new ArrayList<>(100);

    public void run() {
        while (!Thread.currentThread().isInterrupted()) {
            try {
                ImageJob first = queue.take();
                buffer.add(first);
                queue.drainTo(buffer, 99);
                processBatch(buffer);
                buffer.clear();
            } catch (InterruptedException ex) {
                Thread.currentThread().interrupt();
            } finally {
                buffer.clear();
            }
        }
    }
}

Poison Pill Shutdown

sealed interface WorkItem permits ImageJob, StopSignal {}
record StopSignal() implements WorkItem {}

public void stop(int consumerCount) {
    for (int i = 0; i < consumerCount; i++) {
        queue.offer(new StopSignal());
    }
}

Trong production, poison pill phù hợp với worker nội bộ đơn giản. Với executor service, thường ưu tiên shutdown(), shutdownNow(), deadline shutdown và interrupt handling đúng.

CountDownLatch trong test async

@Test

CompletableFuture giúp compose async tasks. Hãy nhớ ba cặp khái niệm:

CompletableFuture<Product> product = supplyAsync(() -> productClient.get(id), executor);

ForkJoin RecursiveTask Ví Dụ Đúng Domain

public class ImageChecksumTask extends RecursiveTask<long[]> {
    private static final int THRESHOLD = 10_000;
    private final byte[] data;
    private final int from;
    private final int to;

    public ImageChecksumTask(byte[] data, int from, int to) {
        this.data = data;
        this.from = from;
        this.to = to;
    }

    @Override
    protected long[] compute() {
        if (to - from <= THRESHOLD) {
            return checksum(data, from, to);
        }
        int mid = (from + to) >>> 1;
        ImageChecksumTask left = new ImageChecksumTask(data, from, mid);
        ImageChecksumTask right = new ImageChecksumTask(data, mid, to);
        left.fork();
        long[] rightResult = right.compute();
        long[] leftResult = left.join();
        return merge(leftResult, rightResult);
    }
}
Common pool là tài nguyên chung

Nếu một thư viện hoặc một đoạn code dùng common pool cho blocking task, nó có thể ảnh hưởng phần khác của app cũng dùng CompletableFuture.supplyAsync không truyền executor hoặc parallelStream. Với backend production, hãy truyền executor rõ ràng cho async I/O.

2.10 ThreadLocal: Context Tiện Nhưng Dễ Leak

ThreadLocal lưu dữ liệu theo thread, không theo request. Trong thread pool, thread sống lâu và được reuse cho nhiều request. Nếu set context mà không remove, request sau có thể nhìn thấy context cũ hoặc object graph lớn bị giữ lại.

Sai: không clear context
REQUEST_CONTEXT.set(context);
handle(request);
Đúng: clear trong finally
try {
    REQUEST_CONTEXT.set(context);
    handle(request);
} finally {
    REQUEST_CONTEXT.remove();
}

ThreadLocal Leak Trong Thread Pool

Leak không nhất thiết là key ThreadLocal còn sống. Ngay cả khi key bị GC, value vẫn có thể nằm trong ThreadLocalMap của thread cho tới khi map được cleanup. Vì worker thread sống lâu, value lớn có thể bị giữ lâu hơn bạn nghĩ.

public final class RequestScope implements AutoCloseable {
    private static final ThreadLocal<RequestContext> CURRENT = new ThreadLocal<>();

    public static RequestScope open(RequestContext context) {
        CURRENT.set(context);
        return new RequestScope();
    }

    public static Optional<RequestContext> current() {
        return Optional.ofNullable(CURRENT.get());
    }

    @Override
    public void close() {
        CURRENT.remove();
    }
}
try (RequestScope ignored = RequestScope.open(context)) {
    service.handle(request);
}

Pattern AutoCloseable giúp ép người đọc thấy lifecycle của context. Với framework web, tốt hơn là clear ở filter/interceptor cuối request.

2.11 Bonus Sau Java 17: Virtual Threads

Virtual threads không thuộc Java 8-17, nhưng senior Java hiện nay thường được hỏi vì nó thay đổi cách nghĩ về blocking I/O. Virtual thread giúp tạo số lượng lớn thread nhẹ hơn platform thread. Nó không làm code CPU-bound chạy nhanh hơn, không sửa race condition, và không thay thế backpressure.

3.1 Thread Pool Có Backpressure

@Configuration
public class ExecutorConfig {
    @Bean("checkoutExecutor")
    public ThreadPoolTaskExecutor checkoutExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(16);
        executor.setMaxPoolSize(64);
        executor.setQueueCapacity(500);
        executor.setKeepAliveSeconds(60);
        executor.setThreadNamePrefix("checkout-");

CallerRunsPolicy là backpressure đơn giản: khi pool quá tải, caller tự chạy task và bị chậm lại. Với API public, bạn có thể chọn AbortPolicy để fail fast và trả 429/503 qua circuit breaker.

3.1.1 Đặt tên thread và log uncaught exception

Khi đọc thread dump, tên thread là manh mối đầu tiên. Thread tên pool-3-thread-17 không giúp gì; checkout-pricing-17 cho biết nó thuộc flow nào.

public final class NamedThreadFactory implements ThreadFactory {
    private final AtomicInteger seq = new AtomicInteger();
    private final String prefix;

    public NamedThreadFactory(String prefix) {
        this.prefix = prefix;
    }

    @Override
    public Thread newThread(Runnable runnable) {
        Thread thread = new Thread(runnable, prefix + "-" + seq.incrementAndGet());
        thread.setDaemon(false);
        thread.setUncaughtExceptionHandler((t, ex) ->
            log.error("Uncaught exception in {}", t.getName(), ex));

3.1.2 Context propagation: MDC không tự đi qua thread khác

Trong web request, correlation ID thường nằm trong MDC/ThreadLocal. Khi submit task sang executor, thread khác không tự có MDC đó. Nếu không copy context, log async bị mất trace ID.

public class MdcTaskDecorator implements TaskDecorator {
    @Override
    public Runnable decorate(Runnable runnable) {
        Map<String, String> parentContext = MDC.getCopyOfContextMap();
        return () -> {
            Map<String, String> previous = MDC.getCopyOfContextMap();
            try {
                if (parentContext != null) {
                    MDC.setContextMap(parentContext);
                } else {
                    MDC.clear();
                }
                runnable.run();
            } finally {
                if (previous != null) {
                    MDC.setContextMap(previous);
                } else {
                    MDC.clear();
                }
            }
        };
    }
}

public void swapInventory(String skuA, String skuB) {
    List<String> ordered = Stream.of(skuA, skuB).sorted().toList();
    ReentrantLock first = lockRegistry.get(ordered.get(0));
    ReentrantLock second = lockRegistry.get(ordered.get(1));

    boolean lockedFirst = false;
    boolean lockedSecond = false;
    try {
        lockedFirst = first.tryLock(300, TimeUnit.MILLISECONDS);
        if (!lockedFirst) throw new LockTimeoutException(ordered.get(0));

        lockedSecond = second.tryLock(300, TimeUnit.MILLISECONDS);
        if (!lockedSecond) throw new LockTimeoutException(ordered.get(1));

        doSwap(skuA, skuB);
    } catch (InterruptedException ex) {
        Thread.currentThread().interrupt();
        throw new OperationInterruptedException(ex);
    } finally {
        if (lockedSecond) second.unlock();
        if (lockedFirst) first.unlock();
    }
}

@Service
public class ProductPageService {

3.3.1 Fan-out có deadline tổng

Timeout từng dependency là cần thiết, nhưng nhiều flow còn cần deadline tổng. Ví dụ trang sản phẩm phải trả trong 900ms dù mỗi dependency có timeout riêng. Nếu không có deadline tổng, tổng hợp nhiều timeout nhỏ vẫn có thể vượt SLA.

public CompletableFuture<ProductPage> loadPageWithDeadline(String productId) {
    long deadlineNanos = System.nanoTime() + TimeUnit.MILLISECONDS.toNanos(900);

    CompletableFuture<ProductDto> product = callWithRemainingDeadline(
        () -> productClient.get(productId), deadlineNanos);

    CompletableFuture<List<ReviewDto>> reviews = callWithRemainingDeadline(
        () -> reviewClient.latest(productId), deadlineNanos)
        .exceptionally(ex -> List.of());

    CompletableFuture<InventoryDto> inventory = callWithRemainingDeadline(
        () -> inventoryClient.get(productId), deadlineNanos)
        .exceptionally(ex -> InventoryDto.unknown());

    return product.thenCombine(reviews, ProductWithReviews::new)
        .thenCombine(inventory, (left, stock) ->
            new ProductPage(left.product(), left.reviews(), stock));
}

private <T> CompletableFuture<T> callWithRemainingDeadline(
        Supplier<T> supplier,
        long deadlineNanos) {
    long remainingMillis = TimeUnit.NANOSECONDS.toMillis(deadlineNanos - System.nanoTime());
    if (remainingMillis <= 0) {
        return CompletableFuture.failedFuture(new TimeoutException("deadline exceeded"));
    }
    return CompletableFuture.supplyAsync(supplier, catalogExecutor)
        .orTimeout(remainingMillis, TimeUnit.MILLISECONDS);
}

3.3.2 allOf không trả list kết quả

CompletableFuture.allOf chỉ trả CompletableFuture<Void>. Bạn vẫn phải join từng future để lấy result. Pattern dưới đây giúp biến list future thành future list.

public static <T> CompletableFuture<List<T>> sequence(List<CompletableFuture<T>> futures) {
    CompletableFuture<Void> allDone = CompletableFuture.allOf(
        futures.toArray(new CompletableFuture[0])
    );
    return allDone.thenApply(ignored ->
        futures.stream()
            .map(CompletableFuture::join)
            .toList()
    );
}

3.4 Single Writer Pattern

Khi correctness quan trọng hơn parallelism, một cách rất mạnh là single writer: nhiều producer đưa command vào queue, một worker duy nhất mutate state. Actor model, event loop và nhiều consumer framework dựa trên tư tưởng này.

public class InventoryActor implements AutoCloseable {
    private final BlockingQueue<ReserveCommand> mailbox = new ArrayBlockingQueue<>(10_000);
    private final Thread worker;
    private final Map<String, Integer> stock = new HashMap<>();
    private volatile boolean running = true;

    public InventoryActor() {
        worker = new Thread(this::loop, "inventory-actor");
        worker.start();
    }

    public boolean submit(ReserveCommand command) {
        return mailbox.offer(command);
    }

    private void loop() {
        try {
            while (running || !mailbox.isEmpty()) {
                ReserveCommand command = mailbox.poll(100, TimeUnit.MILLISECONDS);
                if (command != null) {
                    reserve(command); // only this thread mutates stock
                }
            }
        } catch (InterruptedException ex) {
            Thread.currentThread().interrupt();
        }
    }
}

4.3 Pool Starvation: Task Trong Pool Chờ Task Cùng Pool

Pool starvation xảy ra khi worker của một pool submit task khác vào chính pool đó rồi block chờ kết quả. Nếu mọi worker đều đang chờ subtask nhưng không còn worker trống để chạy subtask, hệ thống treo dù không có deadlock monitor.

Sai: nested submit rồi get cùng pool
ExecutorService pool = Executors.newFixedThreadPool(8);

pool.submit(() -> {
    Future<Details> details = pool.submit(() -> loadDetails());
    return render(details.get()); // có thể starvation
});
Đúng hơn: compose hoặc tách pool
CompletableFuture<Details> details =
    CompletableFuture.supplyAsync(this::loadDetails, ioExecutor);

return details.thenApply(this::render);

Khi đọc thread dump, pool starvation thường thấy nhiều worker ở FutureTask.get, CompletableFuture.join hoặc queue đang có task nhưng không có worker rảnh. Đây là lý do cần metric queue depth và active count theo từng executor.

4.4 Stress Test Race Condition

Race condition không phải lúc nào cũng xuất hiện trong unit test 1 thread. Hãy ép interleaving bằng nhiều thread, latch start cùng lúc và lặp nhiều vòng.

@Test
void unsafeCounterShouldLoseUpdates() throws InterruptedException {
    UnsafeCounter counter = new UnsafeCounter();
    int threads = 32;
    int increments = 100_000;
    CountDownLatch ready = new CountDownLatch(threads);
    CountDownLatch start = new CountDownLatch(1);
    CountDownLatch done = new CountDownLatch(threads);

    for (int i = 0; i < threads; i++) {
        new Thread(() -> {
            ready.countDown();
            await(start);
            for (int j = 0; j < increments; j++) {
                counter.increment();
            }
            done.countDown();
        }).start();
    }

    ready.await();
    start.countDown();
    done.await();

    assertThat(counter.value()).isLessThan(threads * increments);
}

private static void await(CountDownLatch latch) {
    try {
        latch.await();
    } catch (InterruptedException ex) {
        Thread.currentThread().interrupt();
        throw new AssertionError(ex);
    }
}

4.5 Checklist Khi Thread Pool Saturated

  1. Xác định pool nào saturated qua thread name và metric.
  2. Đo active count, queue size, rejected count, task duration.
  3. Phân loại task CPU-bound hay I/O-bound.
  4. Kiểm tra downstream latency nếu task I/O-bound.
  5. Kiểm tra queue có bounded không; nếu unbounded, đánh giá nguy cơ OOM.
  6. Chọn hành động: scale out, giảm input, tăng pool có kiểm soát, thêm timeout, tối ưu downstream, hoặc tách pool theo workload.

6. Interview Framework

SurfaceThread và process khác nhau thế nào?

Process có memory space riêng. Thread sống trong process và chia sẻ heap với thread khác, nhưng có stack riêng. Chia sẻ heap giúp communication nhanh nhưng tạo nguy cơ race condition.

DeepTại sao count++ không thread-safe?

count++ là read-modify-write: đọc giá trị, cộng một, ghi lại. Hai thread có thể cùng đọc giá trị cũ rồi ghi cùng kết quả mới, làm mất increment. volatile chỉ giải quyết visibility, không làm compound operation atomic. Dùng AtomicInteger.incrementAndGet() hoặc lock.

ArchitectureThiết kế checkout không oversell trong multi-instance

Không dựa vào lock trong JVM. Đưa consistency về database bằng optimistic/pessimistic lock trên row stock, hoặc dùng atomic reservation command ở storage chung. Thêm idempotency key để request retry không tạo reservation đôi. Thread pool chỉ điều tiết tải, không thay thế consistency boundary.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. Vì sao bug concurrency lại "khó reproduce" trong khi code không hề đổi? Giải thích bằng từ interleaving.
  2. Phân biệt race condition và deadlock bằng một ví dụ flash-sale, và nói cách nhận ra mỗi loại trong thread dump.
  3. volatile cho bạn cái gì và KHÔNG cho cái gì? Vì sao nó không cứu được count++?
  4. Khi nào ThreadPoolExecutor tạo thread thứ 11 (với core=10)? Vì sao newFixedThreadPool có thể OOM?
  5. Một service chạy 3 instance. Vì sao synchronized không còn chống được oversell, và bạn đưa consistency về đâu?
Bí một câu = quay lại đúng section đó. Nói trôi cả 5 câu không nhìn tài liệu = nắm chắc bài này.

Checklist Thành Thạo

  • Vẽ được memory model cơ bản: heap shared, stack per-thread.
  • Giải thích được 3 điều kiện tạo race condition và cách loại bỏ từng điều kiện.
  • Phân biệt race condition, deadlock, livelock, starvation bằng dấu hiệu debug.
  • Vẽ được flow ThreadPoolExecutor: core → queue → max → reject.
  • Viết được CompletableFuture fan-out có custom executor, timeout và fallback.
  • Nêu được vì sao lock trong JVM không đủ khi app chạy nhiều instance.
  • Biết khi nào parallelStream nguy hiểm vì dùng common ForkJoinPool.
  • Thiết kế được producer-consumer có bounded queue, batch và shutdown rõ ràng.
  • Nhận ra được ThreadLocal leak trong thread pool và luôn clear bằng finally.
  • Phân biệt deadlock, lock contention và pool starvation khi đọc thread dump.
  • Nói được virtual threads giải quyết chi phí blocking thread, không giải quyết backpressure hay shared state.
