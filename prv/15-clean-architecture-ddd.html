Clean Architecture & DDD — Hiểu cơ chế, không thuộc buzzword

DDD và Clean Architecture là 2 chủ đề dễ tạo "hiểu giả" nhất trong kiến trúc backend. Đọc 1 bài blog là gật gù "à hiểu", nhưng khi code thật vẫn viết Anemic Model, Repository return Page<T> của Spring Data, aggregate phình to thành god object. File này KHÔNG giải thích pattern theo kiểu "có 5 building blocks: A, B, C, D, E" — mà ép bạn dừng lại tự predict, nhận diện khi nào pattern SAI, và chạy code reproduce bug để biết thực sự hiểu hay chỉ recognize chữ.

⚠️ Cảnh báo trước khi đọc — Bẫy "hiểu giả"

"Hiểu giả" (illusion of understanding) là trạng thái: đọc xong tài liệu, cảm thấy hiểu, kể lại được tên các pattern — nhưng KHÔNG explain được cơ chế, KHÔNG nhận ra khi pattern bị áp dụng sai, KHÔNG apply được trong code thật. Đây là failure mode phổ biến nhất khi học DDD qua blog / sách / video AI.

Vì sao DDD đặc biệt dễ tạo hiểu giả:
  • Vocabulary đẹp (Aggregate, Bounded Context, Anti-Corruption Layer) → não thưởng cho việc nhớ tên thay vì hiểu cơ chế
  • Pattern nhìn "hợp lý hiển nhiên" sau khi đọc → ảo giác "tôi biết rồi"
  • Không có lỗi compile khi sai — code Anemic vẫn chạy, vẫn pass test → không có feedback loop
  • Senior nói chuyện DDD bằng tiếng Việt + thuật ngữ tiếng Anh trộn → nhớ tiếng nghe hay, không nhớ logic

📖 5 quy tắc đọc — KHÔNG đọc một mạch

  1. Mọi "Pause & Predict" phải dừng lại viết câu trả lời. Viết ra giấy / Notes app. KHÔNG đọc câu trả lời ngay. Nếu trả lời sai → đọc explanation kỹ hơn. Nếu trả lời đúng → bạn đang ở L2 cho concept đó.
  2. Sau mỗi concept, tự đặt câu hỏi: "Khi nào pattern này SAI?" File có sẵn counter-example, nhưng tự nghĩ trước rồi so sánh. Recognize ≠ understand.
  3. Code phải gõ lại bằng tay, không copy-paste. Não nhớ khác khi tay gõ. Đặc biệt phần Refactor (Section 10) — gõ cả 2 phiên bản (Anemic + Rich) để cảm nhận sự khác biệt.
  4. Mỗi 30 phút đóng tab, tự kể lại đoạn vừa đọc cho... cái cây / con mèo / brick wall. Nếu bí 1 câu → đó là chỗ hiểu giả. Quay lại đọc.
  5. 3 labs cuối file là bắt buộc, không phải tùy chọn. Nếu skip lab → bạn vẫn ở L1. Cứ ghi nhớ điều này.

1. Bài toán thực: E-commerce Order chaos CORE

Một team e-commerce có monolith 3 năm tuổi. OrderController gọi trực tiếp OrderRepository, fetch order, trừ stock, tính discount, lưu transaction — tất cả trong 1 method controller. Khi thêm 1 promotion mới (ví dụ: "Mua 2 tặng 1") phải sửa:

  • Controller (validate input mới)
  • OrderService (logic discount mới)
  • PromotionService (logic apply rule)
  • InventoryService (giảm stock theo qty mới)
  • Order entity (thêm field promotionApplied)
  • Database migration (cột mới)
  • 2 batch job (daily report + reconciliation)

Business logic scatter khắp nơi: controller validate, service orchestrate, entity mutate state của chính nó qua setter public, repository enforce business rules (chỉ cho update nếu status = PENDING). Thêm feature thứ 6 cần touch 15 file và regression test 8 module.

⏸️ Pause & Predict — Trước khi đọc tiếp

Hãy viết ra giấy 2 câu trả lời trước khi đọc Section 2:

  1. Trong code chaos ở trên, dòng nào là "business rule" (có thể thay đổi vì policy thay đổi) vs "technical concern" (có thể thay đổi vì tech thay đổi)?
  2. Nếu bạn phải thêm promotion mới mà KHÔNG sửa 15 file, bạn sẽ tách code ra như thế nào? Vẽ sơ đồ.

Quy tắc: KHÔNG đọc Section 2 trước khi viết xong. Nếu bạn skip bước này → đang để não thưởng cho cảm giác "đã hiểu" mà chưa thực sự nghĩ.

4 câu hỏi buộc tư duy

  1. Vì sao thêm promotion mới phải sửa controller? (Hint: controller biết quá nhiều)
  2. Làm sao isolate "promotion logic" để thay đổi mà không ảnh hưởng order/inventory core?
  3. "Bounded context" là gì và vì sao hệ này cần > 1 context?
  4. Nếu Order trong PromotionContext khác Order trong ShippingContext (khác fields, khác behavior), làm sao tránh "god class" có 50 fields?
Chi phí thực: Một team Vietnamese e-commerce mid-size (~50 dev) báo cáo trung bình 6–8 giờ cho 1 feature nhỏ (vì regression 8 module). Sau khi refactor sang Clean + DDD: 1–2 giờ/feature, regression scope giảm còn 1–2 module. ROI refactor: thu hồi sau 2–3 tháng.

2. Vì sao Clean Architecture & DDD tồn tại? CORE

Quy tắc đọc: Phần này giải thích VÌ SAO, không phải WHAT. Nếu bạn skip phần này và nhảy tới "5 building blocks" — bạn sẽ nhớ tên 5 pattern nhưng KHÔNG biết khi nào áp dụng. Đây chính là hiểu giả cấp độ kinh điển.

Pain point 1: Business logic scatter

Khi không có Domain layer rõ ràng, business rules rò rỉ khắp nơi:

  • Controller validate: if (order.getStatus() != "PENDING") throw ... ← rule "chỉ update đơn pending" ở Controller
  • Service: if (item.getQuantity() > stock.getAvailable()) throw ... ← rule "không vượt stock" ở Service
  • Repository: @Modifying @Query("UPDATE order SET ... WHERE status = 'PENDING'") ← rule lại ở Repository
  • Frontend: validate cùng rule lần thứ 4 ← duplication 4 lớp

Hệ quả: thay đổi 1 rule (vd cho phép update đơn SHIPPED nếu chưa giao) phải sửa 4 chỗ, dễ miss 1 chỗ → bug production.

Pain point 2: Domain phụ thuộc Infrastructure

Khi entity có @Entity, repository return Page<Order>, service inject EntityManager — code business không thể test mà không boot Spring. Mỗi test mất 5–15 giây để load context. Test suite 200 test mất 30 phút. Hệ quả: dev skip test, bug lọt production.

Pain point 3: God Object

Khi không có Bounded Context, Order entity bị mọi feature "đắp" thêm field:

class Order {
    Long id;
    // Promotion context cần:
    PromotionCode code;
    Discount discount;
    // Shipping context cần:
    Address shippingAddress;
    Carrier carrier;
    String trackingNumber;
    // Inventory context cần:
    List<ReservationId> reservations;
    // Reporting context cần:
    BigDecimal totalRevenue;
    Map<String, Object> analyticsTags;
    // ... 50 fields, không ai biết field nào dùng cho ai
}

Hệ quả: load Order = load 50 fields. Update 1 field = lock cả entity. Test 1 feature phải mock 49 field khác.

3. Dependency Rule — Code chỉ depend INWARD CORE

L2 check: Sau section này bạn phải explain được: "Vì sao Domain layer KHÔNG được import org.springframework.*?" Nếu trả lời được trong 30s = L2. Nếu phải đọc lại = vẫn L1.

Câu hỏi tranh cãi — Logger có được dùng trong Domain? 2 phái:
  • Phái strict (Uncle Bob): KHÔNG. Domain communicate qua Domain Event, Application layer subscribe event để log. Lý do: log là cross-cutting concern, không phải business logic.
  • Phái pragmatic: CÓ, vì SLF4J là facade (chỉ là interface), implementation ở runtime. Tương tự cách Domain dùng java.util.Logger.
Quan điểm của file này: Tránh nếu có thể, dùng Domain Event để communicate trạng thái. Khi bí (vd debug deep) thì SLF4J cũng chấp nhận được — không phải tội ác chết người.

Trước khi đọc tiếp Section 4: nếu Domain layer chứa @Entity, hãy nghĩ ra 3 hậu quả cụ thể. Viết ra giấy. Không skip — phần này test L1 vs L2.

Đáp án:

4. Value Object — Immutable, định nghĩa bằng giá trị CORE

⏸️ Pause — Trước explanation

Trả lời 2 câu trước khi đọc tiếp:

  1. Nếu Money là String "100 USD" và Email là String "a@b.com" — code sendEmail(money) compile được hay không?
  2. Khi tạo new Money(amount, currency), nếu amount âm thì ứng xử thế nào? Throw runtime hay return null?

VO trả lời 3 câu hỏi sau

  1. Type safety: Compiler chặn sendEmail(money) vì Money ≠ Email. String không làm được điều này.
  2. Validation tập trung: Logic "amount không âm, currency không null" chỉ viết 1 lần trong constructor. Không scatter khắp Service.
  3. Immutability: Không thể money.setAmount(-999). Muốn thay đổi = tạo Money mới.

Quy tắc nhận diện VO

1 object là Value Object nếu thoả MỌI tiêu chí:

  • Không có identity riêng — 2 instance cùng giá trị = bằng nhau (override equals + hashCode theo content).
  • Immutable — sau constructor, không có setter. Mọi "thay đổi" trả về instance mới.
  • Self-validating — constructor đảm bảo invariant. Không tồn tại instance "invalid".
  • Không phụ thuộc lifecycle. Có thể create/destroy free, không tracking.

Implementation runnable — Money

package com.example.shop.domain;

import java.math.BigDecimal;
import java.math.RoundingMode;
import java.util.Currency;
import java.util.Objects;

public final class Money {
    private final BigDecimal amount;
    private final Currency currency;

    // Private constructor — buộc dùng factory để validate
    private Money(BigDecimal amount, Currency currency) {
        Objects.requireNonNull(amount, "amount required");
        Objects.requireNonNull(currency, "currency required");
        if (amount.signum() < 0) {
            throw new IllegalArgumentException(
                "Money cannot be negative: " + amount + " " + currency
            );
        }
        // Scale chuẩn cho currency (USD/VND = 2 decimal, JPY = 0)
        this.amount = amount.setScale(
            currency.getDefaultFractionDigits(),
            RoundingMode.HALF_UP
        );
        this.currency = currency;
    }

    public static Money of(BigDecimal amount, Currency currency) {
        return new Money(amount, currency);
    }

    public static Money of(double amount, String currencyCode) {
        return new Money(BigDecimal.valueOf(amount), Currency.getInstance(currencyCode));
    }

    public static Money zero(Currency currency) {
        return new Money(BigDecimal.ZERO, currency);
    }

    public Money add(Money other) {
        requireSameCurrency(other);
        return new Money(this.amount.add(other.amount), this.currency);
    }

    public Money subtract(Money other) {
        requireSameCurrency(other);
        BigDecimal result = this.amount.subtract(other.amount);
        if (result.signum() < 0) {
            throw new IllegalStateException(
                "Subtraction would produce negative money: " + this + " - " + other
            );
        }
        return new Money(result, this.currency);
    }

    public Money multiply(int times) {
        if (times < 0) throw new IllegalArgumentException("times must be >= 0");
        return new Money(this.amount.multiply(BigDecimal.valueOf(times)), this.currency);
    }

    public boolean isGreaterThan(Money other) {
        requireSameCurrency(other);
        return this.amount.compareTo(other.amount) > 0;
    }

    private void requireSameCurrency(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new IllegalArgumentException(
                "Currency mismatch: " + this.currency + " vs " + other.currency
            );
        }
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Money)) return false;
        Money money = (Money) o;
        return amount.equals(money.amount) && currency.equals(money.currency);
    }

    @Override
    public int hashCode() {
        return Objects.hash(amount, currency);
    }

    @Override
    public String toString() {
        return amount.toPlainString() + " " + currency.getCurrencyCode();
    }
}

Counter-example — Khi nào VO bị áp dụng SAI?

❌ Sai 1: VO có setter "vì cần update giao diện"

public class Money {
    private BigDecimal amount;
    public void setAmount(BigDecimal a) { this.amount = a; } // ❌ Mất immutability
}

Hệ quả: 2 caller cùng giữ reference đến cùng Money, 1 caller setAmount → caller kia thấy giá trị đổi → race condition. Cộng thêm equals/hashCode broken vì hashCode đổi khi instance đã trong HashMap.

❌ Sai 2: VO không validate trong constructor

public class Email {
    public final String value;
    public Email(String value) {
        this.value = value; // ❌ Cho phép Email("not-an-email")
    }
}

Hệ quả: Tồn tại Email "invalid" lẫn trong system. Mọi caller dùng Email phải validate lại → defensive code scatter → 1 chỗ quên validate = bug.

❌ Sai 3: Dùng VO thay vì Entity

// User có 100K instance, identity bằng email
// → Tạo VO User(email, name, ...) ← SAI
// User có identity (userId), tồn tại qua thời gian, balance có thể đổi
// → Phải là Entity, không phải VO

Quy tắc: Nếu object có lifecycle (created → mutated → archived) và identity cần persist qua thay đổi → Entity. Nếu chỉ là "giá trị" (Money, Email, Address) → VO.

L2 check: Explain được "Vì sao Money phải override equals?" trong 20 giây? OK L2. Nếu vẫn cần Google → đọc lại "Quy tắc nhận diện VO" + tự code lại Money từ đầu không nhìn template.

5. Entity — Có identity persist qua thay đổi CORE

Định nghĩa Entity (không phải @Entity của JPA)

Lưu ý quan trọng: "Entity" trong DDD KHÁC @Entity của JPA. DDD Entity = concept ở Domain. JPA @Entity = annotation ở Infrastructure. Có thể có Customer (DDD Entity) mà không có @Entity Customer (JPA) — vd dùng MongoDB.

Một object là DDD Entity nếu:

  • Có identity unique (CustomerId, OrderId) — identity được assign ở constructor và không bao giờ đổi.
  • Equality dựa trên identity, KHÔNG dựa trên content. customer1.equals(customer2) chỉ khi cùng CustomerId, dù name/email khác.
  • Có lifecycle: created → mutate → (archived/deleted). State đổi nhưng vẫn là cùng entity.
  • Mutable — có behavior thay đổi state. Nhưng KHÔNG có public setter trần. Mọi thay đổi qua method có ý nghĩa business.

Counter-example — Anemic Model (Entity giả)

❌ Anemic — Entity chỉ là data holder

✅ Rich Domain — Entity đóng gói behavior

public class Customer {
    private final CustomerId id;        // identity — final
    private Name name;
    private Email email;
    private CustomerStatus status;
    private Instant deactivatedAt;

    private Customer(CustomerId id, Name name, Email email) {
        this.id = Objects.requireNonNull(id);
        this.name = Objects.requireNonNull(name);
        this.email = Objects.requireNonNull(email);
        this.status = CustomerStatus.ACTIVE;
    }

    public static Customer register(CustomerId id, Name name, Email email) {
        return new Customer(id, name, email);
    }

    // Business method, không phải setter
    public void deactivate() {
        if (this.status != CustomerStatus.ACTIVE) {
            throw new IllegalStateException(
                "Cannot deactivate customer in status " + this.status
            );
        }
        this.status = CustomerStatus.DEACTIVATED;
        this.deactivatedAt = Instant.now();
    }

    public void changeEmail(Email newEmail) {
        if (this.status == CustomerStatus.DEACTIVATED) {
            throw new IllegalStateException("Cannot change email of deactivated customer");
        }
        this.email = newEmail;
    }

    // Identity-based equality
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Customer)) return false;
        return id.equals(((Customer) o).id);
    }

    @Override public int hashCode() { return id.hashCode(); }

    // Getters readonly — KHÔNG có setter public
    public CustomerId getId() { return id; }
    public Email getEmail() { return email; }
    public CustomerStatus getStatus() { return status; }
}
So sánh: Trong Anemic, rule "chỉ ACTIVE mới deactivate được" có thể được enforce ở Service nhưng dễ quên. Trong Rich, rule nằm trong entity — không thể bypass. Caller nào gọi customer.deactivate() đều phải đi qua check. Đây là encapsulation đúng nghĩa.

🔨 Thử Phá Vỡ: Counter-example cho Rich Domain

Khi nào Rich Domain KHÔNG phù hợp?

  • Read-heavy reporting model: Object chỉ để hiển thị (vd SalesReport 30 fields) — Rich không thêm value. Dùng DTO/Projection.
  • Migration / ETL pipeline: Object đi qua transformation, không có business rule. Plain POJO/record đủ.
  • Integration object: Wrap response của 3rd-party API — nó không phải Domain của bạn, không cần Rich.

Quy tắc: Rich Domain cho business entity có rule. POJO cho data carrier.

6. Aggregate Root — Transaction boundary HIGH-SIGNAL

Vì sao HIGH-SIGNAL: Aggregate là concept khó nhất DDD. 90% senior interview probe sâu chỗ này. Hiểu nửa vời = thiết kế aggregate phình to → performance disaster. Hiểu đúng = phân biệt được mid vs senior.

Aggregate trả lời 2 câu hỏi

  1. Boundary lưu atomic nào? — Lưu Order phải lưu toàn bộ OrderItem cùng lúc. Không thể lưu Order trước, OrderItem sau (race condition).
  2. Invariant nào phải đảm bảo cùng nhau? — "Total Order = sum của OrderItem.subtotal". Nếu update 1 OrderItem mà không update Total → invariant vỡ.

Aggregate Root vs các entity trong aggregate

Mỗi Aggregate có đúng 1 Root (entity được expose ra ngoài). Các entity khác trong aggregate là "internal":

Order (Aggregate Root) ────────► được expose, có Repository riêng
  ├── OrderItem (internal)  ──► KHÔNG có Repository riêng, KHÔNG được reference từ ngoài
  ├── ShippingInfo (VO)
  └── DiscountApplied (VO)

Rule cứng:
1. Code ngoài aggregate CHỈ giữ reference đến Root, KHÔNG đến OrderItem
2. Mọi modification đi qua Root: order.addItem(...), order.removeItem(...)
3. Save = save toàn bộ aggregate atomic
4. KHÔNG có repository OrderItemRepository ở Domain
   (nếu cần load OrderItem riêng cho report → đó là read model, không phải aggregate)

Implementation — Order Aggregate

public class Order {
    private final OrderId id;
    private final CustomerId customerId;
    private final List<OrderItem> items;       // internal — không expose mutable list
    private OrderStatus status;
    private Money total;

    private Order(OrderId id, CustomerId customerId) {
        this.id = id;
        this.customerId = customerId;
        this.items = new ArrayList<>();
        this.status = OrderStatus.DRAFT;
        this.total = Money.zero(Currency.getInstance("USD"));
    }

    public static Order create(OrderId id, CustomerId customerId) {
        return new Order(id, customerId);
    }

    // Mọi modification qua method có ý nghĩa business
    public void addItem(ProductId productId, int quantity, Money unitPrice) {
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("Cannot modify items of " + status + " order");
        }
        if (quantity <= 0) {
            throw new IllegalArgumentException("Quantity must be positive");
        }
        OrderItem item = new OrderItem(productId, quantity, unitPrice);
        this.items.add(item);
        recalculateTotal();   // ← invariant: total luôn = sum(items)
    }

    public void removeItem(ProductId productId) {
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("Cannot modify items of " + status + " order");
        }
        this.items.removeIf(item -> item.getProductId().equals(productId));
        recalculateTotal();
    }

    public void submit() {
        if (items.isEmpty()) {
            throw new IllegalStateException("Cannot submit empty order");
        }
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("Order already submitted");
        }
        this.status = OrderStatus.SUBMITTED;
    }

    private void recalculateTotal() {
        this.total = items.stream()
            .map(OrderItem::getSubtotal)
            .reduce(Money.zero(Currency.getInstance("USD")), Money::add);
    }

    // Expose readonly view, KHÔNG return mutable list
    public List<OrderItem> getItems() {
        return Collections.unmodifiableList(items);
    }

    public OrderId getId() { return id; }
    public OrderStatus getStatus() { return status; }
    public Money getTotal() { return total; }
}

// OrderItem — internal entity, package-private
class OrderItem {
    private final ProductId productId;
    private final int quantity;
    private final Money unitPrice;
    private final Money subtotal;

    OrderItem(ProductId productId, int quantity, Money unitPrice) {
        this.productId = productId;
        this.quantity = quantity;
        this.unitPrice = unitPrice;
        this.subtotal = unitPrice.multiply(quantity);
    }

    public ProductId getProductId() { return productId; }
    public Money getSubtotal() { return subtotal; }
}

Counter-example — Aggregate quá lớn

❌ Sai: Customer aggregate chứa tất cả Order

public class Customer {
    private CustomerId id;
    private Name name;
    private List<Order> orders;  // ❌ 1 customer có 500 orders sau 3 năm
}

// Load customer = load 500 orders vào memory
// Update name → JPA dirty check toàn bộ orders → slow + risk update sai
// Test customer phải mock 500 orders

Vì sao sai: Customer và Order là 2 aggregate khác nhau, không invariant nào ràng buộc chúng "phải lưu cùng nhau". Quy tắc: aggregate chỉ chứa entity mà invariant business đòi hỏi phải atomic với root.

Fix: Customer và Order là 2 aggregate riêng. Order reference Customer bằng CustomerId (chỉ ID), không embed Customer object.

❌ Sai: Reference internal entity từ ngoài

// External code:
OrderItem item = orderItemRepository.findById(itemId); // ❌ Bypass Order
item.setQuantity(999);  // ← Vỡ invariant: Order.total không cập nhật

// Đúng:
Order order = orderRepository.findById(orderId);
order.updateItemQuantity(itemId, 999); // ← Order recalculate total

Quy tắc thiết kế Aggregate — checklist

  1. Aggregate đủ nhỏ. Vaughn Vernon rule of thumb: aggregate chứa < vài chục entity. Nếu > 1000 → chắc chắn sai.
  2. Reference giữa aggregates bằng ID, không embed. Order chứa CustomerId, không phải Customer.
  3. 1 transaction = 1 aggregate. Nếu use case cần update 2 aggregate → dùng 2 transaction + Domain Event (eventual consistency) hoặc tách use case khác.
  4. Root là entry point duy nhất. Code ngoài không được giữ reference đến internal entity.
  5. Repository chỉ cho Aggregate Root. Không có OrderItemRepository.

⏸️ Pause & Predict

Trong design "Customer có 500 orders", giả sử bạn đã tách Customer và Order thành 2 aggregate. Use case mới: "Khi customer bị banned, cancel tất cả pending orders của họ". Bạn thiết kế thế nào? Viết ra giấy 2 cách.

Đáp án:
  • Cách 1 (eventual consistency, được khuyến nghị): Customer.ban() publish CustomerBannedEvent. OrderHandler subscribe event, query pending orders của customer, gọi order.cancel() cho từng order. Mỗi cancel = 1 transaction riêng.
  • Cách 2 (strong consistency, hiếm khi cần): Use case mở 1 transaction lớn, load Customer + tất cả pending orders, update tất cả. Có risk lock contention nếu nhiều orders.
Mặc định DDD ưu tiên cách 1. Nếu chọn cách 2 mà không suy nghĩ = vẫn L1.

7. Repository — Đúng nghĩa, không phải DAO trá hình CORE

Câu hỏi probe senior: "Khi nào Repository return Page<T> của Spring Data là sai?" — đây là câu test L2 phổ biến nhất trong interview Spring/DDD.

Repository trả lời 3 câu hỏi

  1. Where: Interface ở Domain layer. Implementation ở Infrastructure.
  2. What: Operate trên Aggregate Root, không trên entity internal.
  3. How: API nhìn như "collection in memory" — save, findById, delete. Không expose query DSL (Specification, Criteria, JPQL).

Interface chuẩn ở Domain

Symptom thực tế:

  • Domain layer KHÔNG test được mà không bootstrap Spring Data → unit test > 5s mỗi cái.
  • Đổi tech (MongoDB, DynamoDB) phải rewrite Repository interface, kéo theo rewrite caller.
  • Pagination semantics (offset, cursor) leak vào use case → use case phải biết "page size 20, sort by createdAt desc" — đây là concern của presentation/infrastructure, không phải business.

Vì sao Page<T> là Infrastructure concern?

Pagination là cách truyền data qua boundary (HTTP, gRPC, DB cursor). Domain không quan tâm "20 record/page" — đó là quyết định của UI hoặc API contract. Domain chỉ quan tâm "list pending orders của customer X".

Cách đúng: Application/Read Model layer xử lý pagination, không phải Domain.

// Application layer
@Service

            .map(this::toSummary)

@Service
public class SubmitOrderUseCase {

        if (order.getItems().isEmpty()) {

Quy tắc đơn giản

Khi viết Use Case, đặt câu hỏi: "Logic này nếu tôi cần dùng ở 1 entry point khác (CLI, batch job, message handler), tôi có copy code không?"

  • Nếu CÓ copy → logic phải ở Domain (Entity/Aggregate), không ở Use Case.
  • Nếu KHÔNG (chỉ 1 entry point này dùng) → có thể ở Use Case, nhưng vẫn nên cân nhắc đẩy xuống Domain.

9. Bounded Context — Cùng từ "Order" có thể khác nghĩa HIGH-SIGNAL

Vì sao HIGH-SIGNAL: Bounded Context là khái niệm strategic DDD — phân biệt người dùng tactical pattern (Entity, VO, Aggregate) với người hiểu DDD đúng nghĩa. Senior interview probe sâu chỗ này.

Định nghĩa

Bounded Context = boundary mà 1 model cụ thể áp dụng. Trong cùng business, 1 từ ("Order", "Customer", "Product") có thể có ý nghĩa khác trong context khác.

Khi Context A cần data từ Context B, KHÔNG import model của B trực tiếp. Tạo ACL adapter dịch model:

Context Map — 5 quan hệ chuẩn

Khi vẽ context map, các quan hệ thường gặp:

  • Customer / Supplier: Context A (customer) phụ thuộc Context B (supplier). B "phục vụ" A — A có ảnh hưởng đến roadmap của B (vd negotiate API changes).
  • Conformist: A phải dùng đúng model của B (vd integrate 3rd-party). A không có ảnh hưởng.
  • Anti-Corruption Layer: A tạo adapter dịch model B → model A. Đắt hơn nhưng bảo vệ A khỏi thay đổi của B.
  • Shared Kernel: 2 context dùng chung 1 subset model (phải đồng thuận). Hiếm khi cần, dễ bị abuse.
  • Open Host Service: Context expose API/event public, các context khác consume tự do (vd Publisher → Subscribers).

🔨 Khi nào KHÔNG cần Bounded Context?

  • Hệ thống < 10K LOC, 1 dev — overhead tách context lớn hơn lợi ích.
  • CRUD admin tool — không có business rule đáng tách.
  • Team duy nhất maintain toàn bộ — không có team boundary để align với context boundary.

Bounded Context có giá trị khi: team boundary rõ ràng, business rule khác biệt giữa các phần, hoặc khi cần scale team (Conway's Law).

10. Refactor: Anemic Model → Rich Domain CORE

Đây là phần "apply" — từ L2 lên L3. Bạn sẽ thấy cùng feature, 2 cách viết. Khuyến nghị: gõ lại cả 2 phiên bản bằng tay để cảm nhận khác biệt.

Feature: Submit Order với business rules

Rules:

  1. Order phải có ≥ 1 item mới submit được
  2. Mỗi item quantity ≥ 1
  3. Total = sum(item.subtotal) — phải khớp
  4. Chỉ DRAFT mới submit được
  5. Sau submit, không thể addItem nữa

Version A — Anemic (trước refactor)

public class Order {

    private BigDecimal total;
    private List<OrderItem> items = new ArrayList<>();

    public void setStatus(String status) { this.status = status; }
    public BigDecimal getTotal() { return total; }

    public void setItems(List<OrderItem> items) { this.items = items; }

// Service:
@Service
public class OrderService {
    public void addItem(Long orderId, Long productId, int qty, BigDecimal price) {
        Order order = repo.findById(orderId).orElseThrow();
        if (!order.getStatus().equals("DRAFT")) {
            throw new IllegalStateException("Cannot modify " + order.getStatus());
        }
        if (qty < 1) throw new IllegalArgumentException("Quantity must be positive");

        OrderItem item = new OrderItem();
        item.setProductId(productId);
        item.setQuantity(qty);
        item.setUnitPrice(price);
        item.setSubtotal(price.multiply(BigDecimal.valueOf(qty)));
        order.getItems().add(item);

        // Recalculate total
        BigDecimal newTotal = BigDecimal.ZERO;
        for (OrderItem it : order.getItems()) {
            newTotal = newTotal.add(it.getSubtotal());
        }
        order.setTotal(newTotal);
        repo.save(order);
    }

    public void submit(Long orderId) {
        Order order = repo.findById(orderId).orElseThrow();
        if (order.getItems().isEmpty()) {
            throw new IllegalStateException("Empty order");
        }
        if (!order.getStatus().equals("DRAFT")) {
            throw new IllegalStateException("Already submitted");
        }
        order.setStatus("SUBMITTED");
        repo.save(order);
    }
}

Symptom của Anemic — tự thử reproduce

⚠️ Test reproduce bug — Anemic phá vỡ thế nào?

  1. Bypass invariant: Caller khác có thể order.setTotal(BigDecimal.ZERO) dù items không rỗng → total sai. Compile được, không có warning.
  2. State illegal: Tạo new Order() với status = null → object tồn tại ở trạng thái không hợp lệ.
  3. Rule scatter: Rule "qty ≥ 1" chỉ ở OrderService.addItem. Nếu có BatchOrderService cũng add item mà quên check → bug.
  4. Test phải mock Service: Để test rule "không submit empty order" phải spin up OrderService + mock repository. Test slow.

// Entity với business methods:
public class Order {
    private final OrderId id;
    private OrderStatus status;
    private Money total;
    private final List<OrderItem> items;

    private Order(OrderId id) {
        this.id = Objects.requireNonNull(id);
        this.status = OrderStatus.DRAFT;
        this.total = Money.zero(Currency.getInstance("USD"));
        this.items = new ArrayList<>();
    }

    public static Order create(OrderId id) {
        return new Order(id);
    }

    public void addItem(ProductId productId, int quantity, Money unitPrice) {
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("Cannot add item to " + status + " order");
        }
        if (quantity < 1) {
            throw new IllegalArgumentException("Quantity must be >= 1");
        }
        items.add(new OrderItem(productId, quantity, unitPrice));
        this.total = recomputeTotal();
    }

    public void submit() {
        if (items.isEmpty()) {
            throw new IllegalStateException("Cannot submit empty order");
        }
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("Order in status " + status + " cannot be submitted");
        }
        this.status = OrderStatus.SUBMITTED;
    }

    private Money recomputeTotal() {
        return items.stream()
            .map(OrderItem::getSubtotal)
            .reduce(Money.zero(Currency.getInstance("USD")), Money::add);
    }

    public OrderId getId() { return id; }
    public OrderStatus getStatus() { return status; }
    public Money getTotal() { return total; }
    public List<OrderItem> getItems() { return Collections.unmodifiableList(items); }
}

  • Service mỏng đi 80%. Logic 30 dòng còn 3 dòng.
  • Mọi entry point gọi order.submit() đều phải qua check rule. Không bypass.
  • Test rule không cần Spring — new Order(...).submit() + assert exception. Sub-second test.
  • Type-safety: Money, OrderId, ProductId không thể nhầm với nhau ở compile time.

11. Domain Events + Outbox Pattern SUPP

Khi 1 aggregate thay đổi state, các context khác cần biết. Cách sai: gọi trực tiếp service khác. Cách đúng: publish Domain Event, downstream listen.

Domain Event là gì

Object immutable mô tả "việc gì đã xảy ra trong domain" — tên ở past tense (OrderSubmitted, CustomerDeactivated, không phải SubmitOrder). Event chứa enough info để downstream xử lý mà KHÔNG cần callback ngược.

// Event là record immutable
public record OrderSubmittedEvent(
    OrderId orderId,
    CustomerId customerId,
    Money total,
    Instant submittedAt
) {}

Aggregate record event

public class Order {
    // ... fields ...
    private final List<Object> domainEvents = new ArrayList<>();

    public void submit() {
        if (items.isEmpty()) throw new IllegalStateException("Empty");
        if (status != OrderStatus.DRAFT) throw new IllegalStateException(...);
        this.status = OrderStatus.SUBMITTED;
        // Record event — chưa publish
        domainEvents.add(new OrderSubmittedEvent(id, customerId, total, Instant.now()));
    }

    public List<Object> getDomainEventsAndClear() {
        List<Object> events = new ArrayList<>(domainEvents);
        domainEvents.clear();
        return events;
    }
}

Vì sao cần Outbox Pattern

❌ Naive publish — bug consistency

@Transactional
public void submit(OrderId id) {
    Order order = repo.findById(id);
    order.submit();
    repo.save(order);                    // DB commit
    kafkaPublisher.publish(events);      // ❌ Kafka down → event mất, DB đã commit
}

Symptom: Order đã SUBMITTED trong DB nhưng Notification service không nhận event → khách không có email confirm. Audit log thiếu. Reconciliation báo lệch.

✅ Outbox Pattern

Idea: lưu event vào bảng outbox trong CÙNG transaction với business data. Async job đọc outbox publish ra Kafka. Atomic guarantee được giữ vì cùng DB transaction.

// Repository:
@Repository
public class JpaOrderRepository implements OrderRepository {
    private final SpringDataOrderRepo orderJpa;
    private final OutboxRepository outboxRepo;
    private final ObjectMapper mapper;

    @Override
    @Transactional
    public void save(Order order) {
        // 1. Save aggregate
        orderJpa.save(toEntity(order));

        // 2. Save events vào outbox (CÙNG TRANSACTION)
        for (Object event : order.getDomainEventsAndClear()) {
            OutboxRow row = new OutboxRow();
            row.setId(UUID.randomUUID().toString());
            row.setAggregateType("Order");
            row.setAggregateId(order.getId().toString());
            row.setEventType(event.getClass().getSimpleName());
            row.setPayload(mapper.writeValueAsString(event));
            row.setCreatedAt(Instant.now());
            row.setStatus(OutboxStatus.PENDING);
            outboxRepo.save(row);
        }
        // 3. Transaction commit → cả 2 atomic
    }
}

// Scheduled job poll outbox:
@Component
public class OutboxPoller {
    @Scheduled(fixedDelay = 1000)
    public void poll() {
        List<OutboxRow> pending = outboxRepo.findPending(100);
        for (OutboxRow row : pending) {
            try {
                kafkaPublisher.publish(row.getEventType(), row.getPayload());
                outboxRepo.markPublished(row.getId());
            } catch (Exception e) {
                // Để retry lần sau, KHÔNG mark published
                log.warn("Publish failed for outbox {}", row.getId(), e);
            }
        }
    }
}
Guarantees của Outbox:
  • At-least-once delivery: Event sẽ publish ít nhất 1 lần (có thể nhiều lần nếu poller retry). Downstream phải idempotent.
  • Eventual consistency: Có lag từ commit đến publish (~ms đến seconds tuỳ poll interval).
  • Trade-off: Phức tạp hơn naive publish, nhưng đảm bảo không mất event.

⚠️ 7 anti-patterns kinh điển CORE

❌ #1: Anemic Domain Model

Symptom: Entity chỉ có getter/setter, mọi logic ở Service. Rule scatter, dễ bypass. Test phải spin up Service.

Fix: Đẩy logic vào Entity/Aggregate. Service mỏng đi 70-80%. Xem Section 10.

❌ #2: Repository return infrastructure type

Symptom: Domain Repository return Page<T>, Specification<T>, Pageable. Domain depend Spring Data.

Fix: Repository return domain object hoặc List. Pagination ở Application layer.

❌ #3: Use Case gọi Use Case

Symptom: SubmitOrderUseCase gọi SendEmailUseCase trực tiếp → coupling. Khi đổi email logic phải test cả 2.

Fix: SubmitOrderUseCase publish OrderSubmittedEvent. EmailHandler subscribe event. Loose coupling.

❌ #4: 1 microservice = 1 bounded context (dogma)

Symptom: Nanoservice. 5 service mỗi cái 200 LOC. Network overhead > business value. Debug nightmare.

Fix: 1 microservice có thể chứa nhiều bounded context nếu ownership rõ. Tách service khi team boundary hoặc scale boundary đòi hỏi.

❌ #5: Over-engineer DDD cho CRUD

Symptom: Form đăng ký 4 field → tạo Aggregate + 3 Value Object + Domain Event + Outbox. 400 LOC cho 4 field.

Fix: DDD cho complex domain. CRUD đơn giản dùng JPA Entity + Service direct, không gồng.

❌ #6: Domain Event mất (Kafka down)

Symptom: Naive publish sau commit. Kafka down → event mất → downstream lệch state. Reconciliation khó.

Fix: Outbox Pattern. Event vào DB cùng transaction, poll publish async. Xem Section 11.

❌ #7: Aggregate phình to (god aggregate)

Symptom: Customer aggregate chứa List<Order> chứa List<OrderItem>... Load 1 customer = load 10K record. OOM.

Fix: Tách aggregate. Reference bằng ID. Aggregate có invariant ràng buộc atomic mới ở chung. Xem Section 6.

🔨 Anti-pattern nào nguy hiểm nhất?

#1 (Anemic). Vì:

  • Khó detect — code chạy được, test pass được.
  • Lan rộng — 1 service Anemic dễ khiến cả team theo style đó.
  • Refactor cost cao — phải sửa cả Entity + Service + Test cùng lúc.
  • Mất luôn lợi ích DDD: rule scatter, không reuse được logic, test slow.

❌ Khi nào KHÔNG dùng DDD + Clean Architecture CORE

Sự thật ít người nói: DDD + Clean Arch là cost đáng kể. Áp dụng sai bối cảnh = over-engineer. File này chống hiểu giả nên phải nói rõ.

5 dấu hiệu KHÔNG dùng

  1. CRUD app < 5 tables, không có business rule phức tạp. Form đăng ký, admin panel, simple blog. Spring Boot + JPA Entity + @RestController đủ.
  2. Prototype / MVP < 3 tháng lifespan. Mục tiêu validate idea, không phải maintain dài hạn. Speed > correctness.
  3. 1 dev solo, không có team boundary. Bounded Context align với team boundary. Không có team thì không có context để align.
  4. Domain expertise chưa đủ. DDD đòi hỏi ubiquitous language với domain expert. Nếu chưa có domain expert hoặc business rule chưa stable → DDD sẽ phải refactor liên tục.
  5. Team chưa quen DDD. Áp dụng DDD trên team chưa hiểu = code "DDD-flavored" nhưng vẫn Anemic + Repository expose Page<T> + Use Case chứa rule. Tệ hơn không có DDD vì giả vờ có.

🎯 3-tier interview framework — DDD & Clean Arch

Tier 1 — Surface (Junior signal, ai cũng phải biết)

Q1: Entity và Value Object khác nhau như thế nào?

Trả lời: "Entity có identity persist qua thay đổi (Customer với CustomerId vẫn là cùng Customer dù name/email đổi), equality by identity. Value Object KHÔNG có identity, định nghĩa bằng content (Money(100, USD) bằng nhau bất kỳ instance nào), immutable, equality by value. Quy tắc: nếu có lifecycle + identity cần persist → Entity. Nếu chỉ là 'giá trị' (Money, Email, Address) → VO."

Q2: Repository trong DDD là gì?

Trả lời: "Interface ở Domain layer, đại diện cho persistence abstraction. API nhìn như 'collection in memory': findById(id), save(aggregate). Implementation (JPA, MongoDB, ...) ở Infrastructure — Domain không biết DB. Repository chỉ operate trên Aggregate Root, không cho entity internal."

Tier 2 — Deep Dive (phân biệt mid vs senior)

Q3: Vì sao Repository return Page<T> của Spring Data là sai?

"Page<T> là type của Spring Data — nó là infrastructure concern. Khi Domain Repository return Page<T>:

① Domain layer depend Spring Data → mất tính framework-agnostic. Đổi sang MongoDB Reactive phải rewrite Repository interface.
② Pagination semantics (offset, cursor, sort) leak vào Domain. Use case phải biết 'page size 20'. Đó không phải business concern.
③ Unit test Domain phải có Spring Data runtime → test slow.

Cách đúng: Domain Repository return List<T> hoặc Optional<T>. Pagination ở Application layer hoặc tách Read Model (CQRS lite) — Query Service return PagedResult<DTO>, không qua Aggregate."

Q4: Thiết kế Order aggregate với rules: (a) items atomic với Order, (b) total = sum(items), (c) chỉ DRAFT mới modify được. Show code.

Approach: Order là Aggregate Root, OrderItem internal entity. Order method addItem/removeItem validate status + recalculate total trong cùng method (invariant). Items expose qua Collections.unmodifiableList. Constructor private, factory Order.create(). Status enum, không phải String. Repository save toàn bộ aggregate atomic (1 DB transaction). Code đầy đủ ở Section 6 của file này.

Tier 3 — Architecture (Senior signal)

Q5: Hệ e-commerce có Order trong 4 context: Sales (cart), Fulfillment (đóng gói), Shipping (vận chuyển), Accounting (doanh thu). Mỗi context cần dữ liệu Order khác nhau. Thiết kế.

Approach trong 5 phút:

Phút 1 — Identify Bounded Contexts:
  Sales Context: Order = cart, fields {items, total, customerId, discountCode}
  Fulfillment Context: Order = pickup list, fields {packingItems, warehouseId, picker}
  Shipping Context: Order = parcel, fields {carrier, tracking, address}
  Accounting Context: Order = revenue entry, fields {revenue, vat, journalId}

Phút 2 — Mỗi context có Order model riêng:
  KHÔNG shared kernel "god Order" 50 fields.
  Mỗi context có Order(orderId, ...riêng) — orderId là correlation key.

Phút 3 — Context Map relationships:
  Sales (Publisher) → publish OrderSubmittedEvent
  Fulfillment (Subscriber) → react: tạo Picking task
  Shipping (Subscriber) → react sau "OrderPicked": tạo Shipment
  Accounting (Subscriber) → react sau "OrderDelivered": ghi revenue entry

Phút 4 — Anti-Corruption Layer:
  Khi Fulfillment cần info từ Sales:
    Fulfillment.SalesContextAdapter — call Sales API → map sang
    Fulfillment.PickableOrder (model riêng), bỏ field không cần.
  Lợi ích: Sales đổi schema không break Fulfillment.

Phút 5 — Trade-offs:
  + Loose coupling: 4 team deploy độc lập
  + Test focused: mỗi context test riêng
  + Eventual consistency: order trạng thái diverge ngắn hạn — khách hàng UX cần xử lý
  - Distributed tracing phức tạp (correlation ID + outbox pattern)
  - Reconciliation cần — daily job verify cả 4 context khớp
  - Nếu team < 5, overhead cao hơn benefit → cân nhắc monolith với module boundary
