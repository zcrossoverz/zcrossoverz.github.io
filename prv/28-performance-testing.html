Performance Testing — Load, Stress, Soak, và Đọc Đúng Số

"Chạy load test, RPS cao là ổn" là hiểu giả nguy hiểm. Lên thật: "load test khác stress/spike/soak ở chỗ nào?", "vì sao báo cáo latency trung bình là sai?", "tăng RPS mà p99 vọt — nghẽn ở đâu?" — là lộ. File này dạy 5 loại test, đọc percentile, và phân tích bottleneck có hệ thống, trên domain ShopFlow — flash sale của sàn e-commerce.

🎯 Mục tiêu
Chọn đúng loại test, đọc p99, tìm bottleneck thật, gate trong CI
📅 Thời lượng

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Load test với stress test giống nhau, đều bắn nhiều request."
Thực tế: Load test = ở tải kỳ vọng xem có đạt SLA. Stress test = đẩy quá giới hạn để tìm điểm gãy & cách gãy. Spike = tải tăng đột ngột. Soak = tải vừa nhưng kéo dài để lộ memory leak. Mỗi loại trả lời câu hỏi khác.
Tưởng hiểu: "Latency trung bình 80ms nghĩa là user thấy nhanh."
Thực tế: Trung bình giấu đuôi. p99 = 2s nghĩa 1% request (có thể là khách VIP, hoặc mọi user vào giờ cao điểm) rất chậm. Luôn đọc p95/p99, không đọc mean.
Tưởng hiểu: "Test trên máy dev nhỏ, nhân lên ước lượng production."
Thực tế: Bottleneck phi tuyến (connection pool, GC, lock contention) không scale tuyến tính. Test môi trường khác production → kết quả vô nghĩa. Cần môi trường giống prod (data size, network, cấu hình).

1. Bài toán thực tế — flash sale sập CORE ★

ShopFlow mở flash sale lúc 12h. Bình thường 200 RPS chạy mượt. Lúc cao điểm 2.000 RPS, checkout p99 vọt từ 150ms lên 9s, rồi 504 hàng loạt. Dev nói "đã load test 1.000 RPS OK" — nhưng test trên staging (DB nhỏ, 1 pod) và chỉ nhìn latency trung bình.

⏸ Pause & Predict — trước khi đọc lời giải
Viết ra giấy:
  1. "Load test 1.000 RPS OK" nhưng prod gãy ở 2.000 — loại test nào lẽ ra phải chạy để biết điểm gãy và cách gãy?
  2. Vì sao "latency trung bình OK" che giấu vấn đề? Số nào nên nhìn?
  3. p99 vọt khi tăng RPS thường do tài nguyên nào cạn trước (gợi ý: không phải CPU)?
Đáp án: cần stress test để tìm breaking point (mục 2); nhìn p95/p99 không phải mean (mục 3); thường nghẽn ở connection pool / thread pool / lock trước CPU (mục 5). Đừng đọc tiếp khi chưa phân biệt load vs stress.

3. Metrics + percentile CORE ★

Đo: throughput (RPS), latency (p50/p95/p99/p99.9), error rate, và tài nguyên (CPU, memory, GC, connection pool). Đọc percentile, không đọc mean.

L1 — Biết tên

Biết "p99 là phân vị 99". Symptom hiểu giả: vẫn báo cáo latency trung bình, không hiểu vì sao mean lừa.

L2 — Explain được

Giải thích mean bị kéo bởi đa số nhanh, giấu 1% chậm; p99=2s nghĩa cứ 100 request có 1 cái 2s — ở 2000 RPS là 20 request/giây bị chậm. Hiểu "coordinated omission" làm tool báo p99 lạc quan.

L3 — Apply được

Đặt SLO theo percentile (p99 < 300ms), gắn perf gate CI fail nếu p99 vượt; phân tích đuôi (tail latency) để tìm GC pause/lock; tránh coordinated omission bằng open-model load.

4. Baseline & môi trường giống prod SUPPLEMENTARY

Trước khi tối ưu, đo baseline (số hiện tại) để biết "tốt hơn" là bao nhiêu. Môi trường test phải giống prod ở: kích thước data (DB 10 triệu row khác 1000 row), cấu hình JVM/pool, network latency, số pod. Test trên môi trường nhỏ rồi nhân lên = sai vì bottleneck phi tuyến.

5. Bottleneck analysis có hệ thống CORE ★

Tăng tải, p99 vọt → kiểm tra theo thứ tự cạn-trước:
1. Connection pool (DB) cạn → request chờ lấy connection (thường nghẽn đầu tiên)
2. Thread pool cạn → request queue, 503/timeout
3. Lock contention → nhiều thread BLOCKED trên monitor
4. GC pause → stop-the-world dài → latency spike chu kỳ
5. Downstream chậm → backpressure ngược lên
6. CPU saturation → cuối cùng (thường không phải thủ phạm đầu)
Quy tắc

Đừng đoán — đo. Kết hợp metric (USE: utilization/saturation), thread dump (đếm BLOCKED), GC log (pause time), và pool metric (active/idle/wait). Bottleneck thật thường là tài nguyên hữu hạn cạn trước CPU.

6. JVM under load SUPPLEMENTARY

  • GC pause tạo tail latency: p50 ổn nhưng p99 nhảy theo chu kỳ GC. Chọn GC phù hợp (G1/ZGC) và đo pause.
  • Warm-up / JIT: JVM chậm lúc đầu (interpreted) rồi nhanh khi JIT compile → bỏ qua phần warm-up khi đo.
  • Heap sizing: heap nhỏ → GC nhiều; quá lớn → pause dài hơn. Đo, không đoán.

7. Gatling + perf gate CI SUPPLEMENTARY

// Gatling DSL (ý tưởng): ramp tới 2000 user, assert p99
setUp(
  scn.injectOpen(rampUsersPerSec(10).to(2000).during(120))
).assertions(
  global.responseTime.percentile4.lt(300),  // p99 < 300ms
  global.failedRequests.percent.lt(1)        // error < 1%
)
// CI gate: assertion fail → pipeline đỏ → chặn deploy

⚠️ Anti-patterns CORE ★

🔧 Counter-example — khi load test cho kết quả ĐÁNH LỪA
  • Đo cả warm-up: gộp phần JVM chưa JIT vào số → latency tệ giả tạo, hoặc cache lạnh → không phản ánh prod.
  • Closed-model + coordinated omission: tool đợi response xong mới gửi tiếp → khi server chậm, tool tự giảm tải → p99 báo lạc quan, giấu mất vấn đề.
  • Data nhỏ: DB 1000 row, mọi query trong cache/index nóng → prod 10 triệu row hành xử khác hẳn.
  • Test một endpoint: prod là hỗn hợp request cạnh tranh tài nguyên; test đơn lẻ không thấy contention.

Quy tắc: môi trường + mô hình tải + dữ liệu phải giống prod, bỏ warm-up, dùng open-model. Sai một trong số đó = con số đẹp nhưng vô nghĩa.

🎯 3-tier interview framework

Tầng 1 — SurfaceJunior gate

Q: Load test khác stress test?

Load = tải kỳ vọng, kiểm tra đạt SLA. Stress = vượt giới hạn để tìm điểm gãy và cách gãy. Thêm spike (đột biến), soak (kéo dài, tìm leak), scalability (thêm tài nguyên).

Tầng 2 — Deep DiveSenior gate

Q: p99 vọt khi tăng RPS, tìm root cause thế nào?

Đo theo thứ tự cạn-trước: connection pool wait → thread pool queue → lock contention (thread dump BLOCKED) → GC pause (GC log) → downstream → CPU cuối. Kết hợp USE metric + thread dump + pool metric. Không đoán, đo.

Tầng 3 — ArchitectureSenior+

Q: Thiết kế perf testing cho hệ chuẩn bị flash sale 10×.

Môi trường prod-like; baseline; stress để tìm breaking point; spike test mô phỏng flash sale; soak 8h tìm leak; perf gate CI theo p99; capacity plan (autoscale ngưỡng); chaos nhẹ (kill pod khi tải). Đọc percentile + tail analysis.

🔧 Lab bắt buộc — đo & đọc đúng (2 bài)

Lab 1 — Mean lừa, p99 thật. Viết một endpoint mà 99% nhanh (10ms) nhưng 1% sleep 1s. Bắn 10K request bằng k6/Gatling. So sánh mean (~20ms, "đẹp") với p99 (~1s). Tiêu chí: giải thích vì sao mean che vấn đề và 1% đó ảnh hưởng ai ở 2000 RPS.

Lab 2 — Tìm bottleneck connection pool. Set HikariCP maximum-pool-size=5, endpoint gọi DB query 100ms. Tăng tải tới khi p99 vọt. Quan sát pool metric (active=5, threads waiting). Tăng pool size hợp lý → p99 giảm. Tiêu chí: chứng minh nghẽn ở pool, không phải CPU.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. 5 loại perf test, mỗi loại trả lời câu hỏi gì?
  2. Vì sao latency trung bình lừa? p99 nói gì mà mean không nói?
  3. Khi tăng RPS, thứ tự tài nguyên cạn thường là gì? Vì sao CPU không phải đầu tiên?
  4. Vì sao test môi trường nhỏ rồi nhân lên là sai?
  5. Coordinated omission làm tool báo p99 sai thế nào?
Bí câu nào = quay lại đúng section. Trôi cả 5 = đọc kết quả perf test không bị lừa.
