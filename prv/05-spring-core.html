Spring Core Là Nền Móng Của Spring

Trong dự án thật, bạn hiếm khi gọi trực tiếp ApplicationContext.getBean(), nhưng mọi thứ bạn dùng đều đi qua nó: controller, service, repository, transaction, security, event, configuration. Nếu không hiểu container, bạn sẽ debug Spring bằng cảm giác: thêm annotation cho tới khi chạy.

Senior không cần thuộc mọi annotation. Senior cần nhìn thấy ranh giới: object nào do Spring quản lý, object nào do mình tự new; method call nào đi qua proxy, method call nào bypass proxy; state nào được share giữa request, state nào chỉ sống trong stack của một request.

1. Bài Toán Thực Tế - Voucher Bị Áp Dụng Sai Trong Marketplace

Một marketplace có service áp voucher cho đơn hàng. Bug chỉ xuất hiện lúc traffic cao: khách A dùng voucher SUMMER10, khách B đôi khi bị báo "voucher đã dùng", dù chưa từng dùng voucher đó.

Sai: Stateful singleton service
@Service
public class VoucherService {
    private Set<String> appliedCodes = new HashSet<>();
    private BigDecimal lastDiscount = BigDecimal.ZERO;

    public DiscountResult apply(Cart cart, String code) {
        if (appliedCodes.contains(code)) {
            return DiscountResult.rejected("Code already used");
        }
        BigDecimal discount = calculate(cart, code);
        appliedCodes.add(code);
        lastDiscount = discount;
        return DiscountResult.accepted(discount);
    }
}
Đúng: Stateless service
@Service
public class VoucherService {
    private final VoucherRepository voucherRepository;

    public DiscountResult apply(Cart cart, String code) {
        Voucher voucher = voucherRepository
            .findActiveByCode(code)
            .orElseThrow(() -> new VoucherNotFoundException(code));

        BigDecimal discount = voucher.calculateDiscount(cart);
        return DiscountResult.accepted(discount);
    }
}
Điểm mấu chốt

@Service mặc định là singleton trong một ApplicationContext. Một instance được mọi request dùng chung. Field mutable như appliedCodes hoặc lastDiscount sẽ trở thành state toàn cục trá hình.

Bug cùng nhóm thường đi kèm 3 lỗi khác: field injection khiến dependency ẩn, @Transactional đặt trên private method không chạy, và method có annotation AOP được gọi nội bộ trong cùng class nên không đi qua proxy.

2. Mental Model - Spring Không Magic, Spring Quản Lý Object Graph

2.1 IoC Container

IoC nghĩa là bạn không tự tạo dependency bằng new. Bạn khai báo object cần gì, container quyết định tạo object theo thứ tự nào, inject vào đâu, và bọc proxy nào nếu cần.

Container đi từ metadata đến instance thật. AOP proxy được tạo trong giai đoạn BeanPostProcessor, không phải khi bạn viết annotation.

Ba khái niệm cần tách rõ:

  • IoC: quyền tạo và nối object graph nằm ở container.
  • DI: cách container đưa dependency vào object, thường qua constructor.
  • Bean: object được container quản lý lifecycle, scope, proxy và dependency.

2.3 Bean Scope

Trong web backend, 90% bean là singleton: controller, service, repository, mapper, policy. Vì vậy rule an toàn là: singleton service không giữ mutable per-request state.

2.4 AOP Proxy

Spring AOP không sửa bytecode của class bạn. Nó tạo proxy đứng trước target. External caller gọi proxy, proxy chạy advice, rồi mới gọi method thật.

Nếu method gọi nội bộ bằng this.otherMethod(), call đi thẳng tới target, không qua proxy.
  • Private method: proxy không intercept được.
  • Final method/class: CGLIB không override được.
  • Self-invocation: gọi trong cùng class bypass proxy.
  • Object tự new: không phải Spring bean nên không có proxy.

2.5 Bean Resolution: Khi Có Nhiều Bean Cùng Type

Khi constructor cần một interface nhưng trong context có nhiều implementation, Spring phải chọn một bean duy nhất. Nếu không có tín hiệu rõ, startup fail với lỗi kiểu NoUniqueBeanDefinitionException. Đây là kiến thức daily work, không phải chi tiết phụ.

Cơ chếDùng khiTrade-off
@QualifierCall site cần implementation cụ thểRõ ràng nhất, nhưng tên bean trở thành contract.
@PrimaryMột implementation là default cho đa số use caseDễ che dependency nếu lạm dụng.
Map<String, Strategy>Cần registry nhiều strategyPhù hợp plugin/strategy pattern.
ObjectProvider<T>Dependency optional/lazy/multipleĐừng dùng để giấu thiết kế phụ thuộc mơ hồ.
 
public interface DiscountPolicy {
    Money apply(Cart cart);
}

@Service("seasonalPolicy")
class SeasonalDiscountPolicy implements DiscountPolicy { ... }

@Service("loyaltyPolicy")
class LoyaltyDiscountPolicy implements DiscountPolicy { ... }

@Service
class CheckoutPricingService {
    private final DiscountPolicy discountPolicy;

    CheckoutPricingService(@Qualifier("seasonalPolicy") DiscountPolicy discountPolicy) {
        this.discountPolicy = discountPolicy;
    }
}

Nếu số policy thay đổi theo business rule, inject Map<String, DiscountPolicy> và chọn theo key trong config sẽ rõ hơn tạo nhiều if/else hoặc dùng @Primary tùy tiện.

2.6 JDK Dynamic Proxy vs CGLIB Proxy

Spring AOP có thể tạo proxy theo interface bằng JDK dynamic proxy hoặc subclass bằng CGLIB. Dù dùng loại nào, giới hạn quan trọng vẫn là call phải đi qua proxy.

Loại proxyCách hoạt độngCần nhớ
JDK dynamic proxyProxy implement interfaceCaller nên depend vào interface.
CGLIB proxyProxy subclass target classKhông override được final class/method.
Self-invocationthis.method() gọi target trực tiếpAdvice không chạy dù proxy type nào.
Object tự newKhông nằm trong contextKhông có DI, lifecycle, AOP.
Không học proxy để thuộc implementation

Điểm phải nắm là hành vi runtime: annotation có cần proxy không, method call có đi qua proxy không, class/method có bị final/private cản không. Đây là thứ quyết định @Transactional, security method advice, metrics advice có chạy hay không.

3. Production-Grade Implementation

3.1 Constructor Injection Là Default

@Service
public class CheckoutService {
    private final CartRepository cartRepository;
    private final VoucherService voucherService;
    private final PriceCalculator priceCalculator;

    public CheckoutService(
            CartRepository cartRepository,
            VoucherService voucherService,
            PriceCalculator priceCalculator) {
        this.cartRepository = Objects.requireNonNull(cartRepository);
        this.voucherService = Objects.requireNonNull(voucherService);
        this.priceCalculator = Objects.requireNonNull(priceCalculator);
    }

    public CheckoutPreview preview(String cartId, String voucherCode) {
        Cart cart = cartRepository.findById(cartId)
            .orElseThrow(() -> new CartNotFoundException(cartId));
        DiscountResult discount = voucherService.apply(cart, voucherCode);
        return priceCalculator.preview(cart, discount);
    }
}

Lợi ích thật của constructor injection: dependency hiện rõ trong signature, field có thể final, unit test không cần Spring context, thiếu dependency thì fail ngay lúc startup.

3.2 Prototype Bean Trong Singleton: Dùng Provider

@Component
@Scope("prototype")
public class CsvImportReportBuilder {
    private final List<String> warnings = new ArrayList<>();

    public void warn(String message) {
        warnings.add(message);
    }

    public ImportReport build() {
        return new ImportReport(List.copyOf(warnings));
    }
}

@Service
public class ProductImportService {
    private final Provider<CsvImportReportBuilder> reportBuilderProvider;

    public ProductImportService(Provider<CsvImportReportBuilder> reportBuilderProvider) {
        this.reportBuilderProvider = reportBuilderProvider;
    }

    public ImportReport importCsv(Path file) {
        CsvImportReportBuilder report = reportBuilderProvider.get();
        // mỗi lần import có builder riêng
        return report.build();
    }
}

3.3 Fix Self-Invocation Bằng Cách Tách Boundary

Sai: gọi nội bộ
@Service
public class OrderService {
    public void place(OrderRequest request) {
        validate(request);
        saveOrder(request); // this.saveOrder(), không qua proxy
    }

    @Transactional
    public void saveOrder(OrderRequest request) {
        orderRepository.save(Order.from(request));
    }
}
Đúng: tách service có proxy riêng
@Service
public class OrderWriter {
    @Transactional
    public void save(OrderRequest request) {
        orderRepository.save(Order.from(request));
    }
}

@Service
public class OrderService {
    private final OrderWriter orderWriter;

    public void place(OrderRequest request) {
        validate(request);
        orderWriter.save(request); // qua proxy
    }
}

3.4 Custom AOP Chỉ Khi Có Cross-Cutting Concern Thật

@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface TraceBusinessAction {
    String value();
}

@Aspect
@Component
public class BusinessActionTraceAspect {
    private final Logger log = LoggerFactory.getLogger(getClass());

    @Around("@annotation(trace)")
    public Object trace(ProceedingJoinPoint jp, TraceBusinessAction trace) throws Throwable {
        long started = System.nanoTime();
        try {
            return jp.proceed();
        } finally {
            long tookMs = TimeUnit.NANOSECONDS.toMillis(System.nanoTime() - started);
            log.info("action={} method={} tookMs={}",
                    trace.value(), jp.getSignature().toShortString(), tookMs);
        }
    }
}

AOP hợp với logging, tracing, security, transaction, metric. Không dùng AOP để giấu business rule vì nó làm flow khó đọc.

5. Trade-offs & Anti-Patterns

AP1

Field Injection

Field injection làm dependency ẩn, khó test, không final, và dễ tạo circular dependency mà bạn chỉ phát hiện muộn. Dùng constructor injection làm default.

AP2

Mutable Field Trong Singleton

Không đặt per-request state như current user, last cart, last discount, temporary result trong service field. Dùng local variable, database, cache có key rõ ràng, hoặc request-scoped object nếu thật sự cần.

AP3

Annotation-Driven Programming

Thấy lỗi thì thêm annotation là dấu hiệu chưa hiểu cơ chế. Hãy hỏi: annotation này được xử lý bởi BeanPostProcessor nào, proxy nào, vào thời điểm nào?

6. Interview Framework

SurfaceIoC và DI khác nhau thế nào?

IoC là nguyên tắc đảo quyền kiểm soát việc tạo và nối object graph cho container. DI là kỹ thuật cụ thể để đưa dependency vào object, thường qua constructor. Spring ApplicationContext là IoC container thực hiện DI.

DeepTại sao @Transactional trên private method không chạy?

Spring transaction dựa trên AOP proxy. Proxy intercept public/protected method call đi từ bên ngoài vào bean. Private method không override được, self-invocation không đi qua proxy, object tự new cũng không có proxy. Vì vậy annotation tồn tại trên code nhưng không có interceptor nào chạy.

ArchitectureThiết kế plugin rule cho voucher mà không restart service?

Tách rule engine khỏi Spring bean chính. Có thể dùng config-driven rules hoặc child ApplicationContext cho plugin code. Cần validate rule mới, atomic swap reference, giữ rule cũ cho in-flight requests, cleanup classloader/context cũ, và có rollback. Nếu rule đơn giản, DSL/config tốt hơn dynamic class loading.

Checklist Thành Thạo

  • Vẽ được startup flow: scan component → BeanDefinition → instantiate → inject → BeanPostProcessor/proxy → ready.
  • Giải thích được tại sao singleton service phải stateless bằng ví dụ race condition cụ thể.
  • Phân biệt constructor, @PostConstruct, @PreDestroy nên dùng cho việc gì.
  • Chỉ ra được 4 trường hợp AOP không chạy: private, final, self-invocation, object không do Spring quản lý.
  • Biết dùng Provider/ObjectFactory khi cần prototype instance trong singleton.
  • Giải quyết được nhiều bean cùng type bằng @Qualifier, @Primary hoặc strategy map.
  • Phân biệt JDK proxy và CGLIB ở mức hành vi ảnh hưởng tới final/private/self-invocation.
