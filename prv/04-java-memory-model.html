#04 - Java Memory Model: happens-before, volatile, synchronized, AtomicXxx

Java Memory Model là phần lý thuyết đứng sau concurrency. Nó trả lời câu hỏi: khi nào một thread chắc chắn thấy write của thread khác? Nếu không có happens-before, code có thể chạy đúng trên máy bạn nhưng sai dưới load hoặc trên CPU nhiều core.

JMM Là Contract Giữa Code Java Và Runtime

Developer thường nghĩ code chạy theo đúng thứ tự mình viết. Trong single-thread, compiler và CPU có thể reorder instruction mà bạn không thấy vì kết quả cuối vẫn giống nhau. Trong multi-thread, reordering và CPU cache có thể làm thread khác thấy trạng thái "lạ": flag đã true nhưng data vẫn null, object reference đã publish nhưng field chưa initialized, counter volatile vẫn mất increment.

JMM không yêu cầu bạn hiểu từng CPU instruction. Nó cho bạn một contract: nếu có happens-before giữa action A và action B, thread thực hiện B phải thấy effect của A. Nếu không có happens-before, kết quả không được đảm bảo.

Cách học JMM đúng

Đừng học JMM như một danh sách keyword. Hãy học theo ba câu hỏi: visibility có được đảm bảo không, ordering có được đảm bảo không, operation có atomic không. Mỗi bug concurrency sâu thường là thiếu một trong ba guarantee này.

1

Thread B có chắc thấy write của Thread A không?

2

Thread B có thấy writes theo thứ tự an toàn không?

3

Operation có bị interleave giữa chừng không?

4

1. Bài Toán Thực Tế - Notification Worker Báo Sent Nhưng ID Null

Một service gửi notification chạy ổn trên local. Sau khi lên server nhiều core và tải cao, log thỉnh thoảng ghi sent=true nhưng messageId=null. Không có exception, không reproduce ổn định.

Sai: không có happens-before
public class NotificationState {
    private boolean sent = false;
    private String messageId;

    public void markSent(String id) {
        this.messageId = id;
        this.sent = true;
    }

    public void printIfSent() {
        if (sent) {
            log.info("messageId={}", messageId);
        }
    }
}
Fix đơn giản: volatile flag đúng thứ tự
public class NotificationState {
    private String messageId;
    private volatile boolean sent = false;

    public void markSent(String id) {
        this.messageId = id;
        this.sent = true;
    }

    public void printIfSent() {
        if (sent) {
            log.info("messageId={}", messageId);
        }
    }
}
Điểm tinh tế

volatile sent chỉ đúng nếu write data xảy ra trước volatile write. Nếu đổi thứ tự thành sent = true rồi mới set messageId, volatile không cứu được logic sai.

🗺️ Nếu chỉ có X giờ thì học gì
1 giờ (CORE ★): Happens-before (3) là khái niệm gốc; visibility vs ordering vs atomicity (2); volatile recipe publish-bằng-flag (4.1). Nắm được là đọc/sửa được 80% bug JMM.
+1 giờ (CORE ★): synchronized = lock + visibility đi chung (5), AtomicXxx/CAS (6.1), safe publication + final fields (7.1-7.2). Phỏng vấn senior hỏi thẳng.
+1.5 giờ (SUPPLEMENTARY): wait/notify trong while (5.2), ABA (6.3), this-escape (7.3), double-checked locking đúng (7.4).
Khi cần (ADVANCED): false sharing + @Contended (8.1), VarHandle (8.2), deep immutability (8.3).
⏸ Pause & Predict
Nhìn class NotificationState bản "Sai" ở trên. Trả lời ra giấy:
  1. Vì sao thread đọc có thể thấy sent==true nhưng messageId==null, dù code ghi messageId TRƯỚC sent? (gợi ý: reordering + visibility)
  2. Bản "Fix" chỉ đặt volatile lên sent, KHÔNG lên messageId. Vì sao vẫn đúng? happens-before bắc cầu thế nào?
  3. Nếu lập trình viên đổi thứ tự hai dòng trong markSent thành sent=true; messageId=id;, volatile còn cứu được không? Vì sao đây là bug logic chứ không phải bug JMM?
Đáp án ở mục 2.1 (reordering), 3 (happens-before bắc cầu: data-write → volatile-write → volatile-read → data-read). Đừng đọc tiếp khi chưa viết được câu 2.

2. Vì Sao Cần Java Memory Model?

Máy hiện đại có nhiều tầng cache. Mỗi core có thể có cache riêng. Compiler/JIT và CPU có thể reorder instruction để tối ưu. Nếu mọi read/write đều phải đi thẳng tới main memory và giữ đúng thứ tự tuyệt đối, performance sẽ tệ. JMM cho phép tối ưu bên trong ranh giới an toàn, còn developer dùng synchronization để đặt ranh giới đó.

Không có synchronization, thread B không được đảm bảo thấy write mới nhất của thread A.

2.1 Reordering: Code Viết Một Đằng, Runtime Có Thể Sắp Xếp Một Nẻo

Compiler/JIT/CPU được phép reorder nếu trong single-thread kết quả quan sát được không đổi. Vấn đề là multi-thread có thể quan sát trạng thái trung gian nếu thiếu happens-before.

// Thread A
data = loadData();
ready = true;

// Thread B
if (ready) {
    use(data);
}

Nếu ready không volatile và không có lock, Thread B có thể thấy ready=true nhưng data vẫn là null hoặc stale. Đây không phải JVM "sai"; code của bạn không tạo guarantee.

Lớp tối ưuCó thể làm gìJMM xử lý bằng
Compiler/JITReorder, inline, eliminate dead codeKhông vượt qua happens-before boundary
CPUStore buffer, load reorder, cacheMemory barriers/fences
RuntimeLock optimization, escape analysisPreserve JMM semantics

2.2 Memory Barriers: Ý Nghĩa Thực Dụng

Bạn không thường viết memory barrier trực tiếp trong Java. Bạn dùng volatile, synchronized, atomic classes, concurrent collections. JVM dịch chúng thành cơ chế phù hợp để chặn reorder và flush/invalidate cache khi cần.

Java constructEffect đơn giản hóaÝ nghĩa

Volatile readAcquireReads sau đó không bị kéo lên trước volatile read.
Monitor exitReleaseFlush writes trước khi unlock.
Monitor enterAcquireĐọc state fresh sau khi lock.

3. Happens-Before

Happens-before là quan hệ đảm bảo visibility và ordering. Nếu A happens-before B, mọi effect của A phải visible với B.

RuleNghĩaVí dụ

Monitor lockunlock(m) hb lock(m) kế tiếpsynchronized(lock)
VolatileWrite volatile hb read volatile sau đóFlag shutdown worker
Thread startstart() hb action trong thread mớiSet config trước khi start thread
Thread joinAction trong thread hb join() returnMain thấy result sau join
TransitivityA hb B, B hb C thì A hb CVolatile flag publish non-volatile data trước nó
Transitivity làm non-volatile write trước volatile write visible sau volatile read.

4. volatile

volatile đảm bảo visibility và ordering cho biến đó. Nó không đảm bảo mutual exclusion và không biến compound operation thành atomic.

Đúng: shutdown flag
class Worker implements Runnable {
    private volatile boolean running = true;

    public void stop() {
        running = false;
    }

    public void run() {
        while (running) {
            doWork();
        }
    }
}
Sai: compound operation
class Counter {
    private volatile int count = 0;

    public void increment() {
        count++; // read + add + write, không atomic
    }
}

Dùng volatile khi có một giá trị/flag được một hoặc nhiều thread ghi, thread khác đọc, và operation chỉ là read/write đơn giản. Với invariant nhiều field hoặc increment, dùng lock/atomic/immutable state.

4.1 Volatile Recipe: Publish Data Bằng Flag

Pattern an toàn: ghi toàn bộ data non-volatile trước, rồi volatile write vào flag/reference. Reader volatile read flag/reference trước, rồi đọc data.

public class ConfigHolder {
    private Config config;
    private volatile boolean loaded;

    public void load() {
        Config parsed = parseConfig();
        this.config = parsed;   // normal write
        this.loaded = true;     // volatile write publishes previous writes
    }

    public Optional<Config> current() {
        if (!loaded) {          // volatile read
            return Optional.empty();
        }
        return Optional.of(config);
    }
}

4.2 Volatile Reference Tốt Hơn Nhiều Field Volatile

Nếu state gồm nhiều field phải nhất quán với nhau, đừng rải nhiều volatile field. Tạo immutable state object và publish bằng một volatile reference.

Sai: nhiều volatile field có thể lệch snapshot
private volatile BigDecimal discountRate;
private volatile Instant validUntil;
private volatile String campaignId;
Đúng: immutable snapshot
private volatile CampaignSnapshot campaign = CampaignSnapshot.empty();

public void reload(CampaignSnapshot next) {
    campaign = next;
}

4.3 Volatile Không Phải Lock Nhẹ

Nhiều người dùng volatile như "synchronized nhanh hơn". Đây là cách nghĩ sai. Volatile không đảm bảo critical section, không bảo vệ invariant nhiều field và không làm read-modify-write atomic.

🔧 Counter-example — khi nào volatile SAI
  • volatile int count; count++; — vẫn mất update. count++ là read-modify-write 3 bước; volatile chỉ lo visibility, không lo atomicity. Dùng AtomicInteger.
  • Invariant nhiều field: volatile BigDecimal low, high; với ràng buộc low <= high — hai thread cập nhật xen kẽ vẫn phá invariant. Cần lock bao cả cặp.
  • Check-then-act: if (!initialized) { init(); initialized = true; } với volatile initialized — hai thread cùng thấy false, cùng init. Volatile không tạo critical section.

Quy tắc: volatile chỉ đủ khi (1) chỉ một thread ghi, hoặc (2) thao tác ghi là một single write độc lập (publish flag, snapshot reference). Có read-modify-write hay invariant nhiều biến → cần lock/Atomic.

L1 — Biết tên

Biết "volatile làm biến hiển thị giữa các thread". Symptom hiểu giả: dùng volatile cho counter, hoặc tin volatile = lock nhẹ.

L2 — Explain được

Phân biệt visibility (volatile cho) vs atomicity (không cho) vs mutual exclusion (không cho). Giải thích happens-before của volatile write→read và vì sao nó bắc cầu kéo theo các write thường trước đó.

L3 — Apply được

Chọn đúng tầng guarantee cho từng bug: flag dừng → volatile; publish immutable snapshot → volatile reference; counter → Atomic/LongAdder; invariant nhiều field → synchronized. Giải thích trade-off và chứng minh tính đúng bằng happens-before.

5. synchronized

synchronized cung cấp hai thứ: mutual exclusion và visibility. Khi thread exit synchronized block, writes được flush; thread khác enter cùng monitor sẽ thấy chúng.

public class CartTotal {
    private final Object lock = new Object();
    private BigDecimal subtotal = BigDecimal.ZERO;
    private BigDecimal discount = BigDecimal.ZERO;

    public void applyLine(BigDecimal price, BigDecimal discountLine) {
        synchronized (lock) {
            subtotal = subtotal.add(price);
            discount = discount.add(discountLine);
        }
    }

    public BigDecimal total() {
        synchronized (lock) {
            return subtotal.subtract(discount);
        }
    }
}
Lock object

Dùng private final lock object thay vì lock trên string literal, boxed integer, local object mới, hoặc object public. Lock public có thể bị code ngoài vô tình giữ và gây deadlock.

5.1 Monitor Rules: Lock Và Visibility Đi Chung

Khi một thread release monitor bằng cách thoát khỏi synchronized, mọi write trong critical section happens-before thread khác acquire cùng monitor sau đó. Điều này chỉ đúng nếu hai bên dùng cùng một lock object.

Tình huốngCó an toàn không?Lý do

Writer synchronized, reader đọc trực tiếp không lockKhông đủReader không acquire monitor nên không có visibility guarantee
Writer lock A, reader lock BKhông đủHai monitor khác nhau, không liên quan
Lock trên object publicNguy hiểmCode ngoài có thể giữ lock, gây latency hoặc deadlock

5.2 wait/notify: Luôn Wait Trong Vòng while

wait release monitor và đưa thread vào trạng thái chờ. Khi được notify hoặc notifyAll, thread phải acquire lại monitor trước khi chạy tiếp. Điều quan trọng: wakeup không đồng nghĩa condition đã đúng, nên phải kiểm tra condition trong vòng while.

public class BoundedBuffer<T> {
    private final Queue<T> queue = new ArrayDeque<>();
    private final int capacity;

    public BoundedBuffer(int capacity) {
        this.capacity = capacity;
    }

    public synchronized void put(T item) throws InterruptedException {
        while (queue.size() == capacity) {
            wait();
        }
        queue.add(item);
        notifyAll();
    }

    public synchronized T take() throws InterruptedException {
        while (queue.isEmpty()) {
            wait();
        }
        T item = queue.remove();
        notifyAll();
        return item;
    }
}

Trong application hiện đại, bạn thường dùng BlockingQueue, Semaphore, CountDownLatch hoặc CompletableFuture thay vì tự viết wait/notify. Nhưng hiểu monitor rule giúp đọc thread dump và giải thích vì sao notify sai condition có thể gây treo.

6. AtomicXxx CORE ★

⏸ Pause & Predict
Trước khi đọc về CAS:
  1. AtomicInteger.incrementAndGet() không dùng lock mà vẫn atomic. Cơ chế nào ở tầng CPU làm được điều đó?
  2. Khi 64 thread cùng incrementAndGet() một AtomicLong liên tục, vì sao throughput có thể TỆ đi? LongAdder sửa điều này bằng cách nào?
  3. CAS so sánh "giá trị có còn là X không". Tình huống nào giá trị quay về X nhưng thực ra đã đổi giữa chừng, khiến CAS "đúng nhầm"? (tên vấn đề này là gì?)
Đáp án: CAS = lệnh cmpxchg của CPU (6.1); contention cao → CAS spin retry tốn CPU, LongAdder tách thành nhiều cell cộng dồn (6.2); ABA problem (6.3) dùng AtomicStampedReference để vá.

Atomic classes dùng CAS: compare current value với expected, nếu khớp thì set new value trong một hardware-level atomic operation. Nếu fail, retry hoặc return false.

6.1 CAS: Compare-And-Set

public class RateCounter {
    private final AtomicInteger inFlight = new AtomicInteger();

    public boolean tryAcquire(int max) {
        while (true) {
            int current = inFlight.get();
            if (current >= max) return false;
            if (inFlight.compareAndSet(current, current + 1)) {
                return true;
            }
        }
    }

    public void release() {
        inFlight.decrementAndGet();
    }
}

CAS rất mạnh cho state transition nhỏ, nhưng không miễn phí. Khi contention cao, nhiều thread có thể fail CAS liên tục và retry, tiêu thụ CPU. Với critical section phức tạp, lock đôi khi rõ và hiệu quả hơn.

ClassDùng choGhi nhớ
AtomicInteger/LongCounter chính xác, sequence trong JVMCAS có thể spin khi contention cao
LongAdderMetrics counter contention caosum() không phải snapshot tuyệt đối
AtomicReferenceSwap immutable state/configCAS trên reference
AtomicBooleanOne-time action, guardcompareAndSet(false, true)

6.2 LongAdder: Counter Cho Contention Cao

AtomicLong có một biến trung tâm, nhiều thread cùng CAS vào một điểm. LongAdder chia counter thành nhiều cell để giảm contention, sau đó cộng lại khi sum().

AtomicLong: exact, contention cao hơn
private final AtomicLong requests = new AtomicLong();

public void record() {
    requests.incrementAndGet();
}
LongAdder: tốt cho metrics
private final LongAdder requests = new LongAdder();

public void record() {
    requests.increment();
}

Dùng LongAdder cho metrics, counters không cần snapshot tuyệt đối chính xác từng nano giây. Dùng AtomicLong cho sequence/generation number cần giá trị chính xác ngay.

6.3 ABA Problem

CAS kiểm tra "giá trị hiện tại có bằng expected không". Nếu giá trị đi từ A sang B rồi quay lại A, CAS không biết đã có thay đổi ở giữa. Đây là ABA problem.

// Conceptual example
Thread 1 reads value A
Thread 2 changes A -> B -> A
Thread 1 CAS(A, C) succeeds, but state had changed in between

Trong code application bình thường, ABA ít gặp hơn so với lock-free data structure. Nếu cần version, dùng AtomicStampedReference hoặc thêm version field vào immutable state.

record VersionedRules(long version, Rules rules) {}

private final AtomicReference<VersionedRules> current =
    new AtomicReference<>(new VersionedRules(0, Rules.empty()));

public boolean update(Rules next) {
    VersionedRules old = current.get();
    VersionedRules updated = new VersionedRules(old.version() + 1, next);
    return current.compareAndSet(old, updated);
}

7. Safe Publication

Safe publication nghĩa là object được publish cho thread khác sau khi constructor hoàn tất và các field visible đúng. Nếu publish reference không an toàn, thread khác có thể thấy object ở trạng thái chưa đầy đủ.

7.1 Final Fields

public final class ProductSnapshot {
    private final String sku;
    private final BigDecimal price;

    public ProductSnapshot(String sku, BigDecimal price) {
        this.sku = Objects.requireNonNull(sku);
        this.price = Objects.requireNonNull(price);
    }
}

Final fields có guarantee đặc biệt: nếu constructor không để this escape, thread khác thấy reference sẽ thấy final fields initialized đúng. Guarantee này không biến object thành immutable hoàn toàn. Nếu field final trỏ tới một List mutable và list đó bị mutate sau khi publish, thread khác vẫn có thể thấy trạng thái thay đổi không kiểm soát.

Sai: final reference, mutable content
public final class CatalogSnapshot {
    private final List<Product> products;

    public CatalogSnapshot(List<Product> products) {
        this.products = products; // caller vẫn có thể mutate list
    }
}
Đúng: defensive copy
public final class CatalogSnapshot {
    private final List<Product> products;

    public CatalogSnapshot(List<Product> products) {
        this.products = List.copyOf(products);
    }
}

7.2 Volatile Reference + Immutable State

public class PricingRules {
    private volatile RulesSnapshot current = RulesSnapshot.empty();

    public void reload(RulesSnapshot snapshot) {
        current = snapshot; // volatile write publish toàn snapshot immutable
    }

    public Money calculate(Cart cart) {
        return current.apply(cart); // volatile read
    }
}

7.3 This Escape: Constructor Chưa Xong Đã Bị Nhìn Thấy

this escape xảy ra khi object tự đưa reference của mình ra ngoài trong lúc constructor chưa hoàn tất. Ví dụ thường gặp: đăng ký listener, start thread, submit task, đưa this vào static registry, hoặc gọi method override-able từ constructor.

Sai: start thread trong constructor
public class ReportWarmup {
    private final Map<String, Template> templates;

    public ReportWarmup(Executor executor) {
        executor.execute(this::warmup); // this escape
        this.templates = loadTemplates();
    }

    private void warmup() {
        templates.size(); // có thể NPE
    }
}
Đúng: factory publish sau init
public class ReportWarmup {
    private final Map<String, Template> templates;

    private ReportWarmup(Map<String, Template> templates) {
        this.templates = Map.copyOf(templates);
    }

    public static ReportWarmup createAndStart(Executor executor) {
        ReportWarmup warmup = new ReportWarmup(loadTemplates());
        executor.execute(warmup::warmup);
        return warmup;
    }
}

7.4 Double-Checked Locking Đúng

Double-checked locking chỉ đúng khi reference là volatile. Volatile ở đây không để bảo vệ critical section, mà để chặn reorder và đảm bảo thread đọc thấy object đã constructed đầy đủ.

public final class TemplateRegistry {
    private static volatile TemplateRegistry instance;

    private TemplateRegistry() {}

    public static TemplateRegistry getInstance() {
        TemplateRegistry result = instance;
        if (result == null) {
            synchronized (TemplateRegistry.class) {
                result = instance;
                if (result == null) {
                    result = new TemplateRegistry();
                    instance = result;
                }
            }
        }
        return result;
    }
}

Trong code thực tế, nếu không có lý do đặc biệt, ưu tiên static holder hoặc dependency injection hơn là tự viết singleton. Nhưng khi interviewer hỏi DCL, điểm cần nói là: thiếu volatile thì assignment reference có thể visible trước khi constructor effect visible.

7.5 Concurrent Collections

ConcurrentHashMap.put và get có synchronization nội bộ phù hợp để publish object giữa threads. Nhưng object value vẫn nên immutable hoặc thread-safe.

8. Advanced JMM: False Sharing, VarHandle, Deep Immutability

Các phần dưới đây không phải lúc nào cũng xuất hiện trong vòng đầu interview, nhưng thường xuất hiện khi interviewer muốn phân biệt người biết keyword với người từng debug concurrency/performance thật.

8.1 False Sharing: Khi Hai Biến Khác Nhau Vẫn Làm Chậm Nhau

CPU cache làm việc theo cache line, thường 64 bytes. Nếu hai thread update hai biến khác nhau nhưng hai biến nằm trên cùng cache line, các core vẫn phải invalidate cache line của nhau. Đây là false sharing: không sai correctness, nhưng performance tụt mạnh.

Dễ false sharing
final class Counters {
    volatile long requests;
    volatile long errors;
}
Giảm contention bằng LongAdder
final class Metrics {
    final LongAdder requests = new LongAdder();
    final LongAdder errors = new LongAdder();
}

LongAdder nhanh hơn AtomicLong dưới contention cao vì nó phân tán update vào nhiều cell, giảm việc nhiều thread CAS vào cùng một memory location. Trade-off là sum() không phải snapshot tuyệt đối tại một điểm thời gian.

Khi nào cần quan tâm?

Phần lớn business code không cần tự padding cache line. Nhưng metrics counter cực nóng, ring buffer, queue lock-free, high-throughput event processing có thể bị false sharing. Nếu chưa profile, đừng tối ưu kiểu này trước.

8.2 VarHandle: API Thấp Hơn Atomic, Cao Hơn Unsafe

VarHandle xuất hiện từ Java 9, cung cấp cách access field/array với memory ordering mode rõ hơn: plain, opaque, acquire/release, volatile, compare-and-set. Application backend thông thường ít cần viết trực tiếp, nhưng nên biết khi đọc code JDK, framework, queue hoặc library concurrency.

public final class Sequence {
    private volatile long value;
    private static final VarHandle VALUE;

    static {
        try {
            VALUE = MethodHandles.lookup()
                .findVarHandle(Sequence.class, "value", long.class);
        } catch (ReflectiveOperationException ex) {
            throw new ExceptionInInitializerError(ex);
        }
    }

    public long next() {
        return (long) VALUE.getAndAdd(this, 1L) + 1L;
    }

    public boolean moveIfCurrent(long expected, long next) {
        return VALUE.compareAndSet(this, expected, next);
    }
}

Acquire/ReleaseMột chiều: release publish, acquire consumeQueue/ring buffer tối ưu cao

Trong interview, không cần tự phô diễn VarHandle nếu câu hỏi ở mức application. Câu trả lời tốt là: Atomic classes đủ cho đa số app; VarHandle hữu ích khi xây primitive concurrency hoặc cần control memory ordering tinh hơn.

8.3 Deep Immutability: final Không Đồng Nghĩa Bất Biến Sâu

final field đảm bảo reference không đổi sau constructor, nhưng object mà reference trỏ tới có thể vẫn mutable. Deep immutability nghĩa là toàn bộ object graph visible từ object đó không bị mutate sau khi publish.

9. Patterns Và Anti-Patterns

P1

Volatile + Immutable Snapshot

Dùng cho hot reload config/rules. Update tạo object mới, gán vào volatile reference. Reader không cần lock và luôn thấy snapshot nhất quán.

P2

Initialization-on-demand Holder

public class ConfigRegistry {
    private ConfigRegistry() {}

    private static class Holder {
        static final ConfigRegistry INSTANCE = new ConfigRegistry();
    }

    public static ConfigRegistry instance() {
        return Holder.INSTANCE;
    }
}
AP1

Double-checked Locking Thiếu volatile

new Object() có thể bị reorder: allocate memory, assign reference, rồi mới chạy constructor. Thiếu volatile, thread khác có thể thấy reference không null nhưng object chưa initialized.

AP2

Lock Sai Object

Không lock trên new Object() trong method, string literal, boxed primitive, hoặc object public. Dùng private final lock.

AP3

Mix Lock Và Atomic Không Có Invariant Rõ

Nếu một invariant dùng nhiều field, đừng update một phần bằng lock và một phần bằng atomic rồi hy vọng chúng nhất quán. Chọn một ownership model: hoặc mọi field của invariant nằm trong cùng lock, hoặc state được gói vào một immutable object và swap bằng AtomicReference.

P3

Read-Once Local Snapshot

RulesSnapshot snapshot = current;
return snapshot.evaluate(request);

Khi đọc volatile reference nhiều lần trong một method, có thể mỗi lần đọc thấy snapshot khác nhau. Đọc một lần vào local variable giúp logic nhất quán trong phạm vi request hiện tại.

10. Debug Lab: Tự Chứng Minh JMM Bug

JMM bug khó học vì nhiều ví dụ có thể "không lỗi" trên laptop. Mục tiêu của lab không phải chứng minh JVM sai, mà là tập cách nghĩ: nếu thiếu happens-before, code không có quyền dựa vào visibility hoặc ordering.

10.1 Litmus Test: Reordering Có Thể Làm Cả Hai Đọc 0

public class ReorderingLab {
    static int x, y;
    static int r1, r2;

    public static void main(String[] args) throws Exception {
        for (int i = 0; i < 1_000_000; i++) {
            x = y = r1 = r2 = 0;

            Thread t1 = new Thread(() -> {
                x = 1;
                r1 = y;
            });

            Thread t2 = new Thread(() -> {
                y = 1;
                r2 = x;
            });

            t1.start();
            t2.start();
            t1.join();
            t2.join();

            if (r1 == 0 && r2 == 0) {
                System.out.println("Observed: r1=0, r2=0 at iteration " + i);
                break;
            }
        }
    }
}

Không phải lần chạy nào cũng thấy kết quả lạ. Nhưng về mặt JMM, kết quả đó được phép vì hai thread không có synchronization với nhau. Thêm volatile cho x và y, hoặc dùng lock chung, sẽ thay đổi guarantee.

10.2 Litmus Test: Volatile Counter Vẫn Sai

Đây là bài test chống hiểu nhầm phổ biến nhất: volatile không làm ++ atomic. Nó chỉ làm read/write visible và có ordering guarantee.

public class VolatileCounterLab {
    private volatile int count = 0;

    public void increment() {
        count++; // read, add, write: không atomic
    }

    public int count() {
        return count;
    }
}

Fix đúng phụ thuộc use case: AtomicInteger nếu cần exact counter, LongAdder nếu là metrics dưới contention cao, hoặc lock nếu counter là một phần của invariant nhiều field.

10.3 Litmus Test: Unsafe Publication

public class UnsafePublicationLab {
    static Config config;

    static class Config {
        int timeoutMillis;
        Map<String, String> labels;

        Config() {
            timeoutMillis = 500;
            labels = Map.of("source", "api");
        }
    }

    static void writer() {
        config = new Config(); // không có safe publication
    }

    static void reader() {
        Config local = config;
        if (local != null && local.labels == null) {
            System.out.println("Partially visible object");
        }
    }
}

Không phải cứ chạy là thấy lỗi, nhưng JMM không đảm bảo reader thấy constructor effects nếu publish không an toàn. Fix bằng static initializer, final fields không để this escape, volatile reference, lock, hoặc concurrent collection.

11. Interview Framework

Surfacevolatile làm gì?

volatile đảm bảo visibility và ordering cho biến đó. Write volatile visible với read volatile sau đó. Nó không đảm bảo atomicity cho operation như count++ và không mutual exclusion.

DeepHappens-before là gì?

Là quan hệ đảm bảo effect của action trước visible với action sau. Các rule quan trọng: program order, unlock-lock cùng monitor, volatile write-read, thread start, thread join, transitivity. Không có happens-before thì thread khác không được đảm bảo thấy write.

ArchitectureThiết kế hot-reload pricing rules thread-safe

Represent rules bằng immutable snapshot. Loader build snapshot mới đầy đủ, validate, rồi publish bằng volatile reference hoặc AtomicReference. Reader đọc snapshot một lần vào local variable rồi dùng. Không mutate map/list bên trong snapshot sau khi publish.

🛠️ Lab bắt buộc — quan sát JMM tận mắt (2 bài)

Lab 1 — Reproduce visibility bug. Viết class với boolean running = true (KHÔNG volatile). Một worker thread while(running){}, main thread sleep 1s rồi set running=false. Chạy với -server: worker có thể KHÔNG bao giờ dừng (JIT cache cờ vào register). Thêm volatile → dừng ngay. Tiêu chí: giải thích vì sao thiếu visibility làm vòng lặp không thấy update.

Lab 2 — Mất update vì thiếu atomicity. 8 thread cùng count++ trên một volatile int 100.000 lần mỗi thread. Assert kết quả < 800.000 (volatile không cứu). Đổi sang AtomicInteger.incrementAndGet() → đúng 800.000. Tiêu chí: phân biệt rõ "volatile cho visibility, không cho atomicity".

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. Happens-before nghĩa là gì? Vì sao nó là khái niệm gốc, quan trọng hơn từ khóa volatile?
  2. Tách rõ visibility, ordering, atomicity. Cho một bug chỉ thiếu visibility và một bug chỉ thiếu atomicity.
  3. Vì sao volatile sent=true lại làm thread khác thấy được messageId dù messageId không volatile?
  4. CAS hoạt động thế nào? ABA problem là gì và vá ra sao?
  5. "Safe publication" là gì? Vì sao một object khởi tạo trong constructor vẫn có thể bị thread khác thấy ở trạng thái nửa vời?
Bí câu nào = quay lại đúng section. Trôi cả 5 = bạn nắm JMM ở mức hiếm gặp ở mid-level.
