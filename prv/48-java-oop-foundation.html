Vì Sao File Này Tồn Tại?

Sau 10+ năm review code, 90% bug "khó tìm" của junior bắt nguồn từ không thấm OOP:

  • Class chỉ là "túi đựng method", không có invariant
  • Field public để truy cập tiện, dẫn đến state mutation hỗn loạn
  • Inheritance dùng vì "tiện kế thừa code", không vì "is-a" relationship
  • Service class với 50 method static — đó là procedural code đội lốt OOP
  • Bug xảy ra → không biết bắt đầu debug từ đâu vì class không có boundary rõ ràng
💡 Mục tiêu sau khi đọc xong

Bạn sẽ refactor được code procedural thành OOP đúng nghĩa, justified mỗi lần dùng extends, và recognize được "anemic domain model" trong code review của đồng đội.

1. Bài Toán Thực Tế — Procedural Code Đội Lốt OOP

Tình huống: Junior viết "Service" cho banking, code chạy được nhưng senior reject PR

❌ Code junior — chạy được, đúng output, nhưng senior reject ngay
public class TransferService {
    public static double balance1 = 0;
    public static double balance2 = 0;

    public static void deposit1(double amount) { balance1 += amount; }
    public static void withdraw1(double amount) { balance1 -= amount; }
    public static void deposit2(double amount) { balance2 += amount; }
    public static void withdraw2(double amount) { balance2 -= amount; }

    public static void transfer(double amount) {
        balance1 -= amount;
        balance2 += amount;
    }

    public static void main(String[] args) {
        deposit1(1000);
        transfer(300);
        System.out.println("A: " + balance1 + ", B: " + balance2);
    }
}
💬 Senior comment trên PR (ghi đầy đủ trong feedback thật)
  1. Đây là procedural code đội lốt class. TransferService không có state riêng — toàn static field/method. Tách method ra thành Account.deposit() đi.
  2. Field balance1, balance2 là public + static → bất kỳ ai cũng modify được, không có invariant. Nếu có người set balance1 = -1000 thì sao? Encapsulation đâu?
  3. double cho tiền tệ → mất chữ thập phân. Banking dùng BigDecimal.
  4. Nếu thêm account thứ 3 phải copy-paste deposit/withdraw lần nữa? Inheritance/parametrize đi.
  5. Transfer không atomic — nếu crash giữa 2 dòng thì sao? Đây là banking, hỏi nghiêm túc.
  6. Test thế nào? Static method khó mock. Inject dependency đi.

Refactor OOP đúng

✅ Account là entity có state + invariant
public class Account {
    private final String accountId;
    private BigDecimal balance;
    private final String currency;

    public Account(String accountId, BigDecimal initialBalance, String currency) {
        if (accountId == null || accountId.isBlank())
            throw new IllegalArgumentException("accountId required");
        if (initialBalance == null || initialBalance.signum() < 0)
            throw new IllegalArgumentException("balance must be >= 0");
        this.accountId = accountId;
        this.balance = initialBalance;
        this.currency = currency;
    }

    public void deposit(BigDecimal amount) {
        validatePositive(amount);
        this.balance = this.balance.add(amount);
    }

    public void withdraw(BigDecimal amount) {
        validatePositive(amount);
        if (this.balance.compareTo(amount) < 0)
            throw new InsufficientFundsException(accountId, amount, balance);
        this.balance = this.balance.subtract(amount);
    }

    private void validatePositive(BigDecimal amount) {
        if (amount == null || amount.signum() <= 0)
            throw new IllegalArgumentException("amount must be > 0");
    }

    public BigDecimal getBalance() { return balance; }
    public String getAccountId() { return accountId; }
    public String getCurrency() { return currency; }
}
✅ TransferService chỉ orchestrate, không lưu state
public class TransferService {
    public void transfer(Account from, Account to, BigDecimal amount) {
        if (!from.getCurrency().equals(to.getCurrency()))
            throw new CurrencyMismatchException(from, to);
        from.withdraw(amount);
        to.deposit(amount);
        // Atomicity ở layer @Transactional bên ngoài
    }
}

2. Mental Model — 4 Trụ Cột OOP

Pillar 1

Encapsulation

Class là một con thú có boundary. Field là nội tạng (private), method là cách bên ngoài tương tác.

Pillar 2

Inheritance

"Is-a" relationship, KHÔNG "tiện share code". Favor composition over inheritance.

Pillar 3

Polymorphism

Cùng API, behavior khác nhau theo type runtime. Cho phép Open-Closed.

Pillar 4

Abstraction

Caller chỉ thấy WHAT, không thấy HOW. Hide complexity, expose intent.

Trụ cột 1: Encapsulation — "Class là một con thú có boundary"

Mental model: Hãy nghĩ class như con thú trong chuồng. Field là nội tạng (private), method là cách bên ngoài tương tác (public). Nếu để nội tạng public, ai cũng "đụng vào" được → con thú dễ chết.

❌ Encapsulation phá vỡ
public class BankAccount {
    public BigDecimal balance;  // Ai cũng set được
    public String status;       // Có thể set
                                // "ALIEN_STATUS"
}

// Caller làm bậy:
account.balance =
    new BigDecimal("-9999999");
// Tài khoản âm? Logic không cấm được
account.status = "BANANA";
// Status không hợp lệ?
✅ Encapsulation đúng
public class BankAccount {
    private BigDecimal balance;
    private AccountStatus status;  // enum

    public void debit(BigDecimal amount) {
      if (status != AccountStatus.ACTIVE)
        throw new AccountNotActiveException();
      if (balance.compareTo(amount) < 0)
        throw new InsufficientFundsException();
      balance = balance.subtract(amount);
    }

    public BigDecimal getBalance() {
      return balance;
    }
}

3 quy tắc encapsulation

  1. Field luôn private, hoặc protected nếu thật sự cần kế thừa thấy
  2. Cung cấp method thay vì cho truy cập trực tiếp field
  3. Validate trong constructor + setter — class tự đảm bảo state luôn hợp lệ

Anti-pattern: getter/setter "theo phản xạ" cho mọi field

// ❌ JavaBean syndrome — getter/setter cho mọi field, encapsulation = 0
public class Order {
    private List<OrderItem> items;
    public List<OrderItem> getItems() { return items; }
    public void setItems(List<OrderItem> items) { this.items = items; }
}

// Caller phá:
order.getItems().clear(); // Items biến mất, Order không biết!
order.setItems(null);     // NPE chực chờ

// ✅ Cho method có ý nghĩa, không expose state thô
public class Order {
    private final List<OrderItem> items = new ArrayList<>();

    public void addItem(OrderItem item) {
        validate(item);
        items.add(item);
    }

    public void removeItem(String itemId) { /* ... */ }

    public List<OrderItem> getItems() {
        return Collections.unmodifiableList(items); // Read-only view
    }
}
📖 Quy tắc 10 năm

Trước khi viết getter/setter, hỏi "Caller cần làm gì? Có thể expose method có ý nghĩa thay vì raw access không?"

Trụ cột 2: Inheritance — "is-a", KHÔNG "has code-a"

Mental model: Inheritance là "con là một loại của cha" (Dog is-a Animal). KHÔNG phải "tôi muốn dùng code của class kia".

❌ Inheritance để "share code"
public class StringUtils { /* utility */ }

public class UserController
    extends StringUtils {
  // UserController là StringUtils?? KHÔNG!
  public ResponseEntity<User> getUser() {
    String trimmed = trim(input);
    // "tiện" gọi method của parent
  }
}
✅ Composition — DÙNG, không LÀ
public class UserController {
  public ResponseEntity<User> getUser() {
    String trimmed = StringUtils.trim(input);
    // static call — clean
  }
}

Liskov Substitution Principle (LSP) — Vi phạm kinh điển

Subclass phải có thể thay thế parent ở mọi nơi parent xuất hiện, không phá behavior.

// ❌ Vi phạm LSP — Square không thể thay Rectangle
public class Rectangle {
    protected int width, height;
    public void setWidth(int w) { this.width = w; }
    public void setHeight(int h) { this.height = h; }
    public int area() { return width * height; }
}

public class Square extends Rectangle {
    @Override
    public void setWidth(int w) { this.width = w; this.height = w; }  // Force = nhau
    @Override
    public void setHeight(int h) { this.width = h; this.height = h; }
}

// Caller dùng Rectangle:
Rectangle r = new Square();  // Polymorphism — OK?
r.setWidth(5);
r.setHeight(10);
assert r.area() == 50;  // ❌ FAIL! Vì Square ép width = height, area = 100
⚠️ Bài học

Nếu subclass cần override để "chỉnh" behavior parent → có lẽ không phải "is-a", mà là quan hệ khác. Square và Rectangle về toán học cùng họ Shape, không phải Square is-a Rectangle về behavior.

Inheritance vs Composition — quy tắc Effective Java

📖 Joshua Bloch — Effective Java

"Favor composition over inheritance."

// ❌ Inheritance — fragile, coupling chặt
public class CountingHashSet<E> extends HashSet<E> {
    private int addCount = 0;

    @Override public boolean add(E e) { addCount++; return super.add(e); }
    @Override public boolean addAll(Collection<? extends E> c) {
        addCount += c.size();
        return super.addAll(c); // BUG! HashSet.addAll() gọi add() bên trong → addCount đếm 2 lần
    }
}

// ✅ Composition — wrap, không kế thừa
public class CountingSet<E> {
    private final Set<E> set;
    private int addCount = 0;

    public CountingSet(Set<E> set) { this.set = set; }
    public boolean add(E e) { addCount++; return set.add(e); }
    public boolean addAll(Collection<? extends E> c) {
        addCount += c.size(); return set.addAll(c);
    }
    // Không phụ thuộc internal behavior của HashSet
}
Khi dùng inheritance?Khi dùng composition?
"is-a" thực sự đúng (Dog is-a Animal, EmailNotification is-a Notification)"has-a" hoặc "uses-a"
Không thể đạt bằng composition (hiếm)Cần flexibility (đổi implementation runtime)
Subclass không phá invariant của parentCode reuse mà không cần "is-a"
~10% trường hợp~90% trường hợp trong code application

Trụ cột 3: Polymorphism — "Cùng API, khác behavior"

Mental model: Polymorphism cho phép caller code một lần, hoạt động với nhiều type khác nhau mà không biết type cụ thể.

Service phụ thuộc interface, không biết implementation cụ thể → swap dễ, test dễ
// Interface định nghĩa "API"
public interface NotificationChannel {
    void send(String userId, String message);
}

// 3 implementation khác nhau
public class EmailChannel implements NotificationChannel {
    public void send(String userId, String message) { /* Gửi email */ }
}

public class SmsChannel implements NotificationChannel {
    public void send(String userId, String message) { /* Gửi SMS */ }
}

public class PushChannel implements NotificationChannel {
    public void send(String userId, String message) { /* Gửi push */ }
}

// Caller — không biết Email/SMS/Push, chỉ biết NotificationChannel
public class NotificationService {
    private final List<NotificationChannel> channels;

    public NotificationService(List<NotificationChannel> channels) {
        this.channels = channels;
    }

    public void notifyAll(String userId, String message) {
        for (NotificationChannel channel : channels) {
            channel.send(userId, message); // Polymorphism!
        }
    }
}

Lợi ích

  1. Open-Closed Principle: thêm WhatsAppChannel mới → không phải sửa NotificationService. Chỉ thêm class mới.
  2. Test dễ: mock NotificationChannel thay vì gọi email server thật
  3. Strategy pattern built-in: switch behavior runtime bằng cách inject implementation khác

Anti-pattern: instanceof + cast

// ❌ Type checking thủ công — kill polymorphism
public void send(NotificationChannel channel, String userId, String message) {
    if (channel instanceof EmailChannel) {
        EmailChannel email = (EmailChannel) channel;
        email.send(userId, message);
        email.markAsEmail();
    } else if (channel instanceof SmsChannel) {
        SmsChannel sms = (SmsChannel) channel;
        sms.send(userId, message);
        sms.markAsSMS();
    }
    // Thêm channel mới → phải sửa method này. Đây là Open-Closed violation.
}

// ✅ Polymorphism — method chung trong interface
public interface NotificationChannel {
    void send(String userId, String message);
    void markSent();  // Mỗi implementation tự xử lý
}

Trụ cột 4: Abstraction — "Hide complexity, expose intent"

Mental model: Abstraction là "caller chỉ thấy WHAT, không thấy HOW". Class hide chi tiết phức tạp, expose interface đơn giản có ý nghĩa.

❌ Caller phải hiểu HOW — leak
public class TransferService {
  public void transfer(String fromId, String toId,
      BigDecimal amount, Connection conn,
      TransactionTemplate tx, Logger log,
      AuditLogger audit) {
    tx.execute(status -> {
      try {
        conn.prepareStatement(
          "UPDATE accounts SET balance=...")
          .executeUpdate();
        // 50 dòng code DB
      } catch (SQLException e) { /*...*/ }
    });
  }
}
// Caller phải biết Connection, TransactionTemplate,
// Logger... → leak abstraction
✅ Caller chỉ thấy intent
public class TransferService {
  public void transfer(TransferRequest request) {
    // Chi tiết DB/transaction/log ẩn bên trong
  }
}

// Caller chỉ cần biết WHAT:
transferService.transfer(
  new TransferRequest(fromId, toId, amount));

Quy tắc abstraction tốt

  1. Method name describe WHAT, không describe HOW: transfer() đúng, executeUpdateAndCommit() sai
  2. Parameter ít (≤ 4): nhiều hơn → wrap thành object (TransferRequest)
  3. Hide internal state: caller không biết class lưu data thế nào
  4. Fail at right level: không leak SQLException ra ngoài, wrap thành TransferFailedException

3. Concept Quan Trọng — Junior PHẢI Master

this, super, static, final

public class Account {
    private static final BigDecimal MIN_BALANCE = BigDecimal.ZERO;  // static final = constant

    private final String accountId;  // final = không đổi sau khi set
    private BigDecimal balance;

    public Account(String accountId, BigDecimal balance) {
        this.accountId = accountId;  // this.x = field, x = parameter (disambiguate)
        this.balance = balance;
    }

    public Account(String accountId) {
        this(accountId, BigDecimal.ZERO);  // gọi constructor khác — constructor chaining
    }

    public static Account empty(String id) {  // static factory method
        return new Account(id);
    }
}

public class SavingsAccount extends Account {
    private BigDecimal interestRate;

    public SavingsAccount(String id, BigDecimal balance, BigDecimal rate) {
        super(id, balance);  // super = gọi constructor parent, PHẢI là dòng đầu
        this.interestRate = rate;
    }

    @Override
    public void deposit(BigDecimal amount) {
        super.deposit(amount);  // gọi method parent
        // Có thể thêm logic riêng
    }
}

Abstract Class vs Interface — Khi Nào Dùng Cái Nào?

// Abstract class — có state + behavior chung, là "is-a" relationship
public abstract class BankAccount {
    protected BigDecimal balance;
    protected String accountId;

    // Concrete method — shared
    public BigDecimal getBalance() { return balance; }

    // Abstract method — subclass phải override
    public abstract BigDecimal calculateInterest();
}

public class SavingsAccount extends BankAccount {
    public BigDecimal calculateInterest() {
        return balance.multiply(new BigDecimal("0.05"));
    }
}

public class CheckingAccount extends BankAccount {
    public BigDecimal calculateInterest() { return BigDecimal.ZERO; }
}
// Interface — chỉ contract, không có state. Có thể implement nhiều interface
public interface Auditable {
    AuditLog getAuditLog();
}

public interface Encryptable {
    void encrypt();
    void decrypt();
}

public class SavingsAccount extends BankAccount implements Auditable, Encryptable {
    // Java 8+: interface có default method
}

  • Default: dùng interface
  • Khi cần share state + behavior: abstract class
  • Khi cả 2 đều phù hợp: interface (flexible hơn — implement được nhiều)

Constructor & Initialization Order

public class Parent {
    private int x = 10;
    public Parent() { System.out.println("Parent constructor"); }
    { System.out.println("Parent instance initializer"); }
    static { System.out.println("Parent static block"); }
}

public class Child extends Parent {
    private int y = 20;
    public Child() {
        super();  // Implicit nếu không viết
        System.out.println("Child constructor");
    }
    { System.out.println("Child instance initializer"); }
    static { System.out.println("Child static block"); }
}

// Test
Child c = new Child();
🧠 Output thứ tự — nhớ luật
  1. Parent static block ← static block parent (chỉ chạy 1 lần khi class load)
  2. Child static block ← static block child
  3. Parent instance initializer ← instance init parent
  4. Parent constructor ← constructor parent
  5. Child instance initializer ← instance init child
  6. Child constructor ← constructor child

Nhớ thứ tự: static (parent → child) → instance init parent → constructor parent → instance init child → constructor child.

Inner Class & Static Nested Class

public class Outer {
    private int outerField = 10;

    // Inner class — gắn vào instance Outer, có thể truy cập outerField
    public class Inner {
        public void show() {
            System.out.println(outerField);  // OK
        }
    }

    // Static nested class — KHÔNG gắn instance, không truy cập outerField
    public static class StaticNested {
        public void show() {
            // System.out.println(outerField);  // ❌ Compile error
        }
    }
}

// Tạo instance:
Outer outer = new Outer();
Outer.Inner inner = outer.new Inner();         // Phải có outer instance
Outer.StaticNested nested = new Outer.StaticNested(); // Không cần outer
💡 Thực tế dùng khi nào?
  • Static nested: Builder pattern, Holder pattern
  • Inner: hiếm dùng, dễ memory leak (giữ reference outer)
  • Anonymous class: cho event handler, listener (Java 8+ thay bằng lambda nếu functional interface)

4. SOLID Cho Junior

5 nguyên tắc Robert C. Martin. Junior nắm 3 cái đầu (S, O, L) là đủ để code OOP đúng.

S — Single Responsibility Principle

"Mỗi class chỉ có một lý do để thay đổi."
// ❌ Class kiêm 3 trách nhiệm
public class User {
    private String email;
    public void save() { /* DB code */ }                // Persistence
    public void sendWelcomeEmail() { /* SMTP code */ }  // Communication
    public boolean validate() { /* validation */ }      // Business rule
}
// Thay đổi DB → sửa class này. Thay SMTP → sửa class này. Thay rule → sửa class này.
// 3 lý do khác nhau để thay đổi → vi phạm SRP.

// ✅ Tách trách nhiệm
public class User { /* chỉ data */ }
public class UserRepository { public void save(User u) { } }
public class EmailService { public void sendWelcomeEmail(User u) { } }
public class UserValidator { public boolean validate(User u) { } }

O — Open-Closed Principle

Open for extension, closed for modification.
// ❌ Thêm channel mới → phải sửa class
public class NotificationService {
    public void notify(String type, String msg) {
        if (type.equals("EMAIL")) sendEmail(msg);
        else if (type.equals("SMS")) sendSms(msg);
        // Thêm "PUSH" → sửa class này
    }
}

// ✅ Thêm channel mới = thêm class mới, không sửa code cũ
public interface NotificationChannel { void send(String msg); }
public class EmailChannel implements NotificationChannel { /* ... */ }
public class SmsChannel implements NotificationChannel { /* ... */ }
// Thêm: public class PushChannel implements NotificationChannel

L — Liskov Substitution Principle

Subclass thay thế được parent, không phá behavior. (Ví dụ Square/Rectangle ở trên đã giải thích đầy đủ.)

I — Interface Segregation Principle

Interface to → tách thành nhiều interface nhỏ.
// ❌ Fat interface
public interface Worker {
    void work();
    void eat();
    void sleep();
}
public class Robot implements Worker {
    public void work() { }
    public void eat() { throw new UnsupportedOperationException(); }  // ❌ Robot không ăn
    public void sleep() { throw new UnsupportedOperationException(); }
}

// ✅ Tách
public interface Workable { void work(); }
public interface Eatable { void eat(); }
public interface Sleepable { void sleep(); }
public class Robot implements Workable { public void work() { } }
public class Human implements Workable, Eatable, Sleepable { /* */ }

D — Dependency Inversion Principle

Phụ thuộc abstraction (interface), không phụ thuộc concrete class.
// ❌ UserService phụ thuộc concrete MySqlUserRepository
public class UserService {
    private MySqlUserRepository repo = new MySqlUserRepository();
}
// Đổi sang Postgres → phải sửa code

// ✅ Phụ thuộc interface, inject từ ngoài
public class UserService {
    private final UserRepository repo;  // interface
    public UserService(UserRepository repo) { this.repo = repo; }
}
✨ Vì sao Spring DI mạnh

DIP là nền tảng của Spring DI. Junior hiểu DIP rồi vào Spring sẽ rất nhanh — bạn sẽ thấy @Autowired trên interface chính là DIP applied at framework level.

5. Trade-offs & Anti-Patterns Phổ Biến

Anti-pattern #1: God Class

// ❌ Class 2000 dòng, làm đủ thứ
public class UserManager {
    public void createUser() { }
    public void deleteUser() { }
    public void sendEmail() { }
    public void validatePayment() { }
    public void exportToCsv() { }
    public void calculateTax() { }
    // ... 50 method khác
}

Refactor: tách theo SRP — UserService, EmailService, PaymentValidator, CsvExporter, TaxCalculator.

Anti-pattern #2: Anemic Domain Model

❌ Entity chỉ getter/setter
public class Order {
  private List<OrderItem> items;
  private OrderStatus status;
  // chỉ getter/setter
}

// Logic dồn về service
public class OrderService {
  public void cancel(Order order) {
    if (order.getStatus() != PENDING)
      throw new IllegalStateException();
    order.setStatus(CANCELLED);
  }
}
✅ Rich Domain Model
public class Order {
  private OrderStatus status;

  public void cancel() {
    if (status != PENDING)
      throw new InvalidOrderStateException(
        "Cannot cancel: " + status);
    status = CANCELLED;
  }
}
// Behavior gắn vào entity

Anti-pattern #3: Constructor 10 Tham Số

// ❌
public Order(Long id, String userId, BigDecimal total, String currency, OrderStatus status,
             LocalDateTime createdAt, LocalDateTime updatedAt, String shippingAddress,
             String billingAddress, String discountCode) { }

// ✅ Builder pattern
Order order = Order.builder()
    .id(123L)
    .userId("user1")
    .total(BigDecimal.valueOf(100))
    .currency("VND")
    .build();

Anti-pattern #4: Cyclic Dependency

// ❌ A phụ thuộc B, B phụ thuộc A
public class A { private B b; }
public class B { private A a; }
// Khó test, khó refactor, dễ infinite loop

Refactor: tách interface, hoặc dùng event để decouple.

Anti-pattern #5: Thiếu equals/hashCode khi làm Map key

// ❌ Custom class không override → HashMap.get() trả null
class UserId { String value; }

Map<UserId, User> map = new HashMap<>();
map.put(new UserId("1"), user);
map.get(new UserId("1"));  // null!

// ✅ Override hoặc dùng Java 16 record
record UserId(String value) { }

7. Interview Framework

Tầng 1 — SurfaceJunior
Q: 4 trụ cột OOP là gì?
"Encapsulation (đóng gói state, expose method có ý nghĩa, validate ở constructor), Inheritance (subclass extends parent với 'is-a' relationship), Polymorphism (cùng interface, behavior khác nhau theo type runtime), Abstraction (hide complexity, expose intent đơn giản). 4 cái này tổng hợp lại để code dễ extend, dễ test, dễ maintain."
Q: Abstract class vs Interface khác nhau thế nào?
"Abstract class có thể có state (field) + concrete method, là 'is-a' relationship, Java cấm extends nhiều. Interface là contract, không state (Java 8+ có default method), 1 class implement nhiều interface được. Thực tế: default dùng interface, abstract class khi cần share state + behavior + 'is-a' rõ."
Tầng 2 — Deep DiveMid bridge
Q: 'Favor composition over inheritance' — tại sao?
"3 lý do chính:

1. Fragile base class: subclass phụ thuộc internal behavior parent. Parent đổi → subclass break (ví dụ HashSet.addAll() gọi add() bên trong, override add() trong subclass → addAll() behave khác).
2. Tight coupling: inheritance fix relationship compile-time. Composition cho phép swap behavior runtime (Strategy pattern).
3. Multiple inheritance limit: Java cấm extends nhiều class. Composition cho phép 'use' nhiều class.

Quy tắc: 'is-a' chắc chắn → inheritance. Còn lại → composition. 90% trường hợp app dùng composition."
Q: Liskov Substitution Principle — ví dụ vi phạm?
"LSP nói subclass phải thay parent được mà không phá behavior expected.

Ví dụ kinh điển: Square extends Rectangle. Rectangle có setWidth/setHeight độc lập. Square ép width = height. Caller code:

Rectangle r = new Square(); r.setWidth(5); r.setHeight(10); assert r.area() == 50; // FAIL với Square (area = 100)

Square không 'is-a' Rectangle về behavior, dù về toán học là vậy. Đây là dấu hiệu inheritance sai. Refactor: cả 2 implement Shape interface, không kế thừa nhau."
Tầng 3 — ArchitectureSenior signal
Q: Tại sao Spring khuyến khích DI qua constructor, không qua field?
"4 lý do:

1. Immutable: field final chỉ set qua constructor → instance không bị thay đổi dependency runtime.
2. Required dependency rõ ràng: nhìn constructor signature thấy ngay class cần gì. Field injection ẩn dependency.
3. Test dễ: new object với dependency mock, không cần Spring context hay reflection.
4. Circular dependency phát hiện sớm: constructor injection fail compile/startup time, field injection fail runtime ở chỗ ngẫu nhiên.

DIP + constructor injection là combo. Field @Autowired là legacy, nên tránh trong code mới."
