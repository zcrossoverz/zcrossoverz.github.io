Linux for Backend Dev — Production Survival Kit

Junior backend chỉ dùng Linux khi ssh vào server. Khi gặp incident 3h sáng — CPU 100%, disk full, app không respond — biết 5 lệnh không đủ. File này biến Linux từ "lệnh thuộc lòng" thành mental model + workflow debug production.

CORE ★ 1 tuần reflex + practice trên server thật Domain: E-commerce / SaaS production

⚠️ Cảnh báo "hiểu giả"

  • Biết ls / cd / cat / grep — nhưng gặp kill -9 PID không hiểu khác kill PID gì
  • Dùng tail -f log — không biết tail -f | grep ERROR với --line-buffered để hot filter
  • Biết top — không đọc được load average 1/5/15 phút nghĩa gì
  • Biết tên ss / netstat — không biết LISTEN vs ESTABLISHED vs TIME_WAIT trong output
🚨 3 cấp — bạn đang ở đâu?

L1 (Recognize): biết tên 10 lệnh. Production incident gọi senior ngay.
L2 (Explain): reflex 30 lệnh. Dùng pipe chained. Diagnose CPU/disk/network bằng workflow chuẩn.
L3 (Apply): debug production 3h sáng <15 phút. Write shell script automation. Đọc systemd unit + log journalctl thuần thục.

1. 5 Tình huống production junior vướng CORE ★

📖 Tình huống #1 — "App chậm, CPU 100%"

Bối cảnh: 9h sáng peak traffic, app response 5-10s. Junior ssh vào server, không biết bắt đầu từ đâu.

Senior workflow:

top -o %CPU                       # sort theo CPU
# PID 1234 java 99% CPU  ← thủ phạm

# Dive vào process Java đó
top -H -p 1234                    # show thread của process
# PID 5678 99% CPU  ← thread cụ thể

# Convert thread ID sang hex (cho Java thread dump)
printf '%x\n' 5678                # → 0x162e

# Java thread dump
jstack 1234 | grep -A 30 'nid=0x162e'
# → 1 thread đang infinite loop ở OrderService.search()

Bài học: CPU 100% trên Java thường = 1 thread loop. Không phải tăng tài nguyên — phải tìm thread đó.

📖 Tình huống #2 — "Disk đầy"
df -h                             # disk usage tổng
# /dev/sda1   80G   78G   2G  98%  ← gần full

# Tìm folder ngốn nhất
du -h --max-depth=1 /var | sort -hr | head
# 50G /var/log
# 20G /var/lib/docker

# Tìm file lớn nhất trong /var/log
find /var/log -type f -size +1G -exec ls -lh {} \;
# 30G /var/log/app/access.log

# Fix tạm: truncate log (KHÔNG rm — app còn write)
truncate -s 0 /var/log/app/access.log

# Hoặc setup logrotate (long-term fix)
# /etc/logrotate.d/app: rotate daily, keep 7 days, compress

Bonus pattern: lsof | grep deleted → tìm file đã xóa nhưng process còn giữ handle (memory leak điển hình — disk full nhưng du không thấy).

📖 Tình huống #3 — "App không respond"
# 1. App có chạy không?
ps aux | grep java
systemctl status myapp

# 2. App listen đúng port không?
ss -tlnp | grep :8080
# nothing — app chết

# 3. Tại sao chết? Check log
journalctl -u myapp -n 100 --no-pager
# OutOfMemoryError: Java heap space
# Process killed by OOM killer

# 4. Diagnose: OOM. Cần tăng heap hoặc fix memory leak.
# Quick fix: tăng -Xmx
# Long-term: heap dump on OOM, MAT analyze
dmesg | grep -i "killed process"  # confirm OOM kill
📖 Tình huống #4 — "Tìm log lỗi hôm qua"
# Log rotated daily — tìm trong log cũ
zgrep "ERROR" /var/log/app/app.log.1.gz | head -20

# Hoặc tất cả log trong tuần
zgrep -h "ERROR" /var/log/app/app.log*.gz | wc -l

# Filter theo time range
journalctl -u myapp --since "2026-05-23 00:00" --until "2026-05-24 00:00" | grep ERROR

# Pattern phổ biến: top 10 error message
grep ERROR /var/log/app/*.log | awk -F': ' '{print $NF}' | sort | uniq -c | sort -rn | head
📖 Tình huống #5 — "Service A không gọi được service B"
# 1. DNS resolve được không?
dig service-b.internal
nslookup service-b.internal

# 2. Connect TCP được không?
nc -zv service-b.internal 8080
# Connection refused → service B chết
# Connection timed out → network firewall

# 3. HTTP response thế nào?
curl -v http://service-b.internal:8080/health
# 200 OK → service B ok, vấn đề khác
# 503 / timeout → service B down

# 4. Trace network path
traceroute service-b.internal
mtr service-b.internal              # interactive trace

# 5. Nếu service A bị block:
ss -anp | grep ESTABLISHED | grep service-b   # connection từ A
tcpdump -i any port 8080 -nn       # capture packet (cần root)
⏸ Pause & Predict
Trả lời:
  1. Tình huống #1: tại sao jstack cần thread ID hex (0x162e) thay vì decimal (5678)?
  2. Tình huống #2: tại sao truncate -s 0 tốt hơn rm trên file đang ghi?
  3. Tình huống #5: curl -v trả 200 nhưng app A vẫn fail — 3 nguyên nhân tiềm năng?

2. Mental Model — Linux as System CORE ★

2.1 "Everything is a file"

Triết lý Unix: device, socket, pipe, process — đều biểu diễn dưới dạng file. Cùng API (read/write/close) cho mọi thứ.

/proc/1234/        ← Process 1234 info "as file"
/proc/1234/status     basic info
/proc/1234/limits     ulimit settings
/proc/1234/fd/        open file descriptors

/sys/                 ← Kernel state as file
/dev/                 ← Device as file
/dev/null               discard sink
/dev/urandom            random source

Insight: echo "data" > /proc/sys/net/... = config kernel runtime. cat /proc/1234/limits = xem limit của process. Không cần "tool đặc biệt" — chỉ cần biết path.

2.2 Process Tree

Mọi process có parent (PPID). PID 1 = init (hoặc systemd). Tree visualize:

pstree -p
ps -ef --forest
htop                              # F5 = tree view

Khi process die: con thành "orphan", PPID tự reassign về 1 (init adopt). Pattern này = phát hiện service crash khi PPID đổi đột ngột.

2.3 File Permissions

-rwxr-xr--  user group  file
^^^                       file type: - file, d dir, l link
   ^^^                    owner: rwx
      ^^^                 group: r-x
         ^^^              other: r--

Octal: r=4, w=2, x=1. 755 = rwxr-xr-x. chmod 600 ~/.ssh/id_rsa = chỉ owner đọc/ghi.

2.4 stdin / stdout / stderr

command > output.txt              # stdout → file
command 2> error.log              # stderr → file
command > out 2> err              # split
command > out 2>&1                # merge stderr vào stdout
command &> all.log                # bash shortcut cho cả 2
command 2>/dev/null               # ignore stderr (silence warnings)

Bẫy production: Docker container default redirect stderr → /dev/null. Đây là tại sao e.printStackTrace() trong Java mất log. Phải dùng SLF4J → stdout.

3. Daily Commands — 30 Lệnh Reflex CORE ★

Navigation & File Ops

ls -lah                # list all, human-readable size
cd -                   # về thư mục trước
pwd                    # path hiện tại
find /path -name "*.log" -mtime -7   # file .log sửa trong 7 ngày
find /path -size +100M               # file > 100MB
cp -r src dst          # copy recursive
mv old new             # rename / move
rm -rf folder          # ⚠️ recursive force — kiểm path TRƯỚC khi enter

Search & Process Text

# Tìm file
find . -name "*.java" | xargs grep -l "OrderService"
# Tìm trong content
grep -rn "TODO" src/                # recursive, line number
grep -A 3 -B 3 "ERROR" app.log      # 3 line trước+sau

# Process text
cut -d',' -f2 data.csv | sort -u    # column 2, unique
awk '{print $5, $9}' access.log     # column 5+9
sed 's/old/new/g' file > file.new   # replace
sort | uniq -c | sort -rn           # đếm + sort giảm dần

Pipes & Redirection — Sức Mạnh Thật Sự

# Top 10 IP gọi nhiều nhất
awk '{print $1}' access.log | sort | uniq -c | sort -rn | head -10

# Đếm số lỗi mỗi giờ trong app.log
grep ERROR app.log | awk '{print $2}' | cut -d':' -f1 | sort | uniq -c

# Tìm 5 file lớn nhất trong /var
find /var -type f -exec du -h {} + | sort -rh | head -5

# Disk usage subfolder, sort
du -h --max-depth=1 /var | sort -hr

# Số connection theo state
ss -tan | awk 'NR>1 {print $1}' | sort | uniq -c

Process Management

ps aux | grep java                  # tìm process
ps -ef --forest                     # tree
kill PID                            # SIGTERM (graceful)
kill -9 PID                         # SIGKILL (force, KHÔNG cleanup)
kill -HUP PID                       # reload config (nhiều app support)
pkill -f "java.*OrderApp"           # kill by name pattern

# Background / foreground
nohup ./script.sh &                 # chạy background, không die khi logout
disown                              # detach từ shell
jobs                                # list background
fg %1                               # foreground job 1

# Run với priority
nice -n 19 ./heavy-job              # low priority (19 thấp nhất)
ionice -c 3 ./io-job                # low I/O priority

Resource Monitoring

top                                 # real-time, q quit
htop                                # nicer top, F5 tree
free -h                             # memory
df -h                               # disk usage
du -sh *                            # size từng folder
iostat -x 1                         # disk I/O detailed
vmstat 1                            # cpu + memory + IO
uptime                              # load avg 1/5/15 min
# load avg 0.5 → CPU 50% busy với 1 core. >1 = oversubscribed

Networking

# Listening ports
ss -tlnp                            # t=TCP, l=listening, n=numeric, p=process
# All connections
ss -tan
# Connect test
nc -zv host 8080                    # zero IO + verbose
curl -v http://host:8080/health
curl -w "@curl-format.txt" -o /dev/null http://host  # timing breakdown

# DNS
dig service.internal
nslookup service.internal
host service.internal

# Trace
traceroute host
mtr host                            # interactive

# Open files / sockets per process
lsof -p PID                         # all
lsof -i :8080                       # who use port 8080

Archive & Compression

tar -czf out.tar.gz folder          # create + gzip
tar -xzf out.tar.gz                 # extract
tar -tzf out.tar.gz | head          # list contents

gzip file                           # → file.gz, xóa original
gunzip file.gz                      # ngược lại
zcat file.gz | grep ERROR           # đọc gz không extract
zgrep ERROR /var/log/*.gz           # grep multi gz files

SSH & SCP

ssh user@host                       # login
ssh -i ~/.ssh/key.pem user@host     # với key
ssh -L 8080:localhost:8080 host     # local port forward
ssh -J jump-host target             # ssh qua jump host

scp file user@host:/path            # copy lên server
scp -r folder user@host:/path       # recursive
rsync -avz src/ user@host:/dst/     # tốt hơn scp — incremental

Systemd & Cron

# Systemd
systemctl status myapp
systemctl restart myapp
systemctl enable myapp              # auto-start at boot
journalctl -u myapp -f              # log realtime
journalctl -u myapp --since "1h ago"

# Cron
crontab -l                          # list
crontab -e                          # edit
# Format: minute hour day month dayofweek command
# Mỗi 5 phút:   */5 * * * * /script.sh
# 3:00 AM daily: 0 3 * * * /backup.sh

4. Production Debug Workflow — Senior style CORE ★

Workflow chuẩn — 3 phút đầu khi nhận alert

# 1. Quick health check (10 giây)
uptime                              # load avg cao?
free -h                             # mem dùng?
df -h /var /tmp                     # disk?

# 2. Top consumers (30 giây)
top -bn1 -o %CPU | head -20         # top CPU
top -bn1 -o %MEM | head -20         # top MEM

# 3. App status (30 giây)
systemctl status myapp
ss -tlnp | grep :8080               # port listen?

# 4. Recent log (1 phút)
journalctl -u myapp -n 200 --no-pager | grep -iE "error|exception|killed"

# 5. Now you have signal — dive deeper based on what found
🎯 Workflow theo symptom

CPU spike: top -H -p PID → jstack PID (Java) → tìm hot thread

Memory leak: jmap -histo:live PID → jmap -dump:file=heap.hprof PID → MAT

Disk full: du -h --max-depth=1 /var | sort -hr → tìm folder ngốn nhất → find -size +1G

Network slow: mtr → identify hop chậm → tcpdump nếu cần packet

App không respond: ss -tlnp → journalctl -u app → dmesg | grep killed

5. Shell Scripting Essentials SUPPLEMENTARY

#!/bin/bash
set -euo pipefail                   # exit on error, undefined var, pipe fail

# Variables
NAME="world"
echo "Hello, $NAME"

# Conditionals
if [ -f /etc/passwd ]; then
    echo "exists"
elif [ -d /tmp ]; then
    echo "tmp is dir"
else
    echo "neither"
fi

# Loop
for f in *.log; do
    echo "Processing $f"
    gzip "$f"
done

# Function
log_info() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] INFO: $1"
}
log_info "App started"

# Error handling
if ! mysqldump db > backup.sql; then
    log_info "Backup failed"
    exit 1
fi

# Trap (cleanup on exit)
trap 'rm -f /tmp/work.lock' EXIT

Rule: mọi script production phải có set -euo pipefail ở đầu — bắt lỗi sớm thay vì silent fail.

7. Docker Basics — Container = Linux Namespace ADVANCED

Docker container KHÔNG phải VM — chỉ là process Linux với namespace + cgroup isolation. Hiểu Linux = hiểu Docker.

docker ps -a                        # list all containers
docker logs -f myapp                # follow log
docker exec -it myapp bash          # shell vào container
docker inspect myapp                # full config + state
docker stats                        # resource per container

# Debug container chạy chậm
docker top myapp                    # processes inside
docker exec myapp top               # tương tự, từ inside

# Container chết — vì sao?
docker logs myapp --tail 50
docker inspect myapp | grep -A 5 State
# OOMKilled: true → container vượt memory limit
# ExitCode: 137 → killed by SIGKILL (thường là OOM)
# ExitCode: 1   → app crash với exception

# Disk: image + container layer
docker system df                    # disk usage
docker system prune -af             # cleanup unused

Insight: OOMKilled: true trong docker inspect = container vượt --memory limit. Khác với JVM OOM (heap full). 2 loại khác nhau, fix khác nhau.

🚨 Symptom → Debug Recipes CORE ★

🚨 Recipe #1 — "App start nhưng port không listen"

Symptom: systemctl status myapp show running. curl localhost:8080 connection refused.

ss -tlnp | grep :8080            # có ai listen 8080?
# Empty → app chưa bind port hoặc bind sai

journalctl -u myapp -n 50         # check log
# "Port 8080 already in use" → port conflict
# "Bind to 127.0.0.1" → bind localhost, không accept external

lsof -i :8080                     # ai chiếm port?

3 root cause: port conflict (process khác chiếm), bind sai IP (127.0.0.1 vs 0.0.0.0), firewall block.

🚨 Recipe #2 — "Disk full nhưng du không thấy"

Symptom: df -h show 95% full. du -sh / tổng chỉ 30G. Đâu ra 50G?

# Tìm file đã delete nhưng process còn handle
lsof | grep deleted | head
# COMMAND  PID  USER  FD  TYPE  SIZE  NAME
# java     1234 app   23  REG   50G   /var/log/app.log (deleted)

# Process đang giữ 50G của file đã rm
# Fix: restart process HOẶC truncate via /proc
> /proc/1234/fd/23                # truncate file qua fd

Pattern: ai đó rm log file đang ghi → inode còn vì process giữ handle → disk không free.

🚨 Recipe #3 — "load avg cao nhưng CPU không cao"

Symptom: uptime show load avg 8.0 trên 4-core. top show CPU chỉ 30%.

Root cause: load avg = CPU running + I/O waiting. Cao mà CPU thấp = process đang chờ I/O.

vmstat 1                          # check 'b' column (blocked, waiting IO)
iostat -x 1                       # %iowait cao? device nào busy?
ps -eo pid,stat,cmd | grep ' D '  # process trong state D (uninterruptible IO wait)

Thường là DB slow query, network drive lag, hoặc disk dying.

🚨 Recipe #4 — "Service A timeout khi gọi service B trong cùng K8s cluster"

Symptom: Service A báo timeout. kubectl exec -it pod-a -- curl service-b:8080 hoạt động. Reproduce khi load cao.

# Trong pod A
ss -tan | awk '{print $1}' | sort | uniq -c
# nhiều TIME_WAIT → connection pool exhausted

# Check connection limit
cat /proc/sys/net/netfilter/nf_conntrack_max
cat /proc/sys/net/netfilter/nf_conntrack_count    # current
# count gần max → conntrack table full → drop connection

3 nguyên nhân thường: connection pool nhỏ, conntrack table full, ephemeral port range hết.

🎤 Interview Framework CORE ★

Tier 1 — Surface

Q: kill vs kill -9?

kill PID gửi SIGTERM — process catch + cleanup (close file, flush log). kill -9 PID gửi SIGKILL — kernel kill ngay, KHÔNG cleanup. Dùng -9 chỉ khi SIGTERM không work sau 30s.

Tier 2 — Deep Dive

Q: load average 1.5 trên 4-core nghĩa là gì?

Load avg = số process running + waiting I/O. 1.5/4 = ~37% busy. >1.0 PER CORE = oversubscribed. Cao mà CPU thấp = process chờ I/O. Phân biệt bằng vmstat column 'b' và iostat %iowait.

Tier 3 — Architecture

Q: Production incident — "service A timeout khi gọi service B". Debug workflow?

Layer-by-layer:

  1. Service B alive? systemctl status + port listen check (ss)
  2. Network reachable? nc/curl từ pod A → mtr nếu có hop
  3. Connection pool? ss -tan đếm TIME_WAIT / ESTABLISHED
  4. Conntrack? /proc/sys/net/netfilter/nf_conntrack_count vs max
  5. Ephemeral port? /proc/sys/net/ipv4/ip_local_port_range
  6. DNS? dig + check /etc/resolv.conf timeout
  7. App-level? jstack pool size, JFR for downstream latency

🧠 Feynman Test — 5 câu

  1. "Everything is a file" — giải thích bằng 1 ví dụ /proc.
  2. load avg vs CPU% khác gì? Khi nào load cao mà CPU thấp?
  3. SIGTERM vs SIGKILL — khi nào dùng cái nào?
  4. Disk full nhưng du không thấy — recipe debug?
  5. Docker container OOMKilled khác Java heap OOM thế nào?
Pass: 5/5 trong < 10 phút.
