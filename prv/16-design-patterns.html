#16 — Design Patterns Trong Thực Tế

Patterns không phải để "thuộc lòng GoF" — mà là vocabulary chung để team communicate. Khi senior nói "đây nên là Strategy", junior hiểu ngay. File này dạy 6 pattern phổ biến nhất qua ví dụ e-commerce, ride-sharing, content streaming, kèm "vì sao chọn pattern này".

Vì Sao Học Patterns?

Có 2 sai lầm phổ biến với design patterns:

  1. Pattern Fever: học GoF rồi áp dụng mọi nơi → over-engineer simple problems
  2. Pattern Allergy: tránh patterns → code rối, junior khác không hiểu intent
🎯 Cách dùng đúng

Patterns là solution name cho recurring problem. Bạn không "đi tìm chỗ áp dụng Strategy" — bạn gặp 5 phương thức tính phí khác nhau và recognize đây là vấn đề Strategy giải quyết tốt. Mục tiêu: học để recognize pattern trong code đồng đội + trong yêu cầu mới, không phải để "thêm pattern vào code".

File này chọn 6 patterns được dùng nhiều nhất trong production Java:

  • Strategy — chọn 1 trong nhiều algorithm runtime
  • Factory — tạo object phức tạp
  • Builder — construct object với nhiều optional fields
  • Observer / Event — decouple producer khỏi consumers
  • Decorator — add behavior mà không sửa class gốc
  • Chain of Responsibility — pipeline xử lý

1. Bài Toán Thực Tế — Shipping Fee Của Sàn TMĐT

Năm 1: Chỉ có Standard Shipping

Sàn ShopVN khởi nghiệp với 1 phương thức ship: Standard (giao 3-5 ngày). Code 50 dòng trong OrderService.calculateShippingFee() — fee = base 30,000 VND + 5,000 VND/kg.

Năm 2: Express Shipping

Add Express (giao 1-2 ngày, fee gấp 1.5x). OrderService giờ có if-else:

if (shippingType.equals("STANDARD")) {
    fee = base + weight * 5000;
} else if (shippingType.equals("EXPRESS")) {
    fee = (base + weight * 5000) * 1.5;
}

Năm 3: Same-Day, International, COD-Pickup

Bây giờ OrderService.calculateShippingFee() dài 800 dòng, 6 nesting levels:

  • Same-Day: chỉ available nội thành HN/HCM, fee tính theo zone (innerCity vs outerCity)
  • International: dùng API DHL/FedEx, fee phụ thuộc country + customs declaration
  • COD-Pickup: ship miễn phí nhưng phụ thu 2% giá trị đơn hàng

Năm 4: "Add Pickup-At-Store trong 1 tuần, không sửa code cũ"

🔥 Triệu chứng của code không có pattern
  • Thêm method #6 phải hiểu toàn bộ 800 dòng để không break
  • Regression test cả 5 methods cũ → 2 ngày test manual
  • Fee logic scatter: 1 phần trong service, 1 phần trong DB config, 1 phần trong DHL SDK
  • Junior touch nhầm tab → SameDay tính nhầm fee như Express → mất tiền

5 câu hỏi suy ngẫm

  1. Pattern nào tách fee calculation logic để mỗi shipping method độc lập?
  2. Làm sao tránh massive if-else chain?
  3. Processor objects nên create 1 lần (singleton) hay per-request?
  4. Làm sao test shipping logic mà không gọi thật DHL/FedEx API?
  5. Pattern nào cho phép add logging/metrics mà không sửa processor code?

2. 6 Patterns Thực Dụng

Strategy Pattern — "Cùng task, khác algorithm"

Strategy: 1 interface, N implementations. Context không biết implementation cụ thể.
// Interface — định nghĩa "task"
public interface ShippingFeeStrategy {
    Money calculateFee(ShippingContext ctx);
}

// Implementations — mỗi method là 1 class
public class StandardShippingStrategy implements ShippingFeeStrategy {
    private static final Money BASE = Money.of(30_000, VND);
    private static final Money PER_KG = Money.of(5_000, VND);

    @Override
    public Money calculateFee(ShippingContext ctx) {
        return BASE.add(PER_KG.multiply(ctx.weightKg()));
    }
}

public class ExpressShippingStrategy implements ShippingFeeStrategy {
    private final ShippingFeeStrategy standardCalc;

    public ExpressShippingStrategy(ShippingFeeStrategy standardCalc) {
        this.standardCalc = standardCalc;
    }

    @Override
    public Money calculateFee(ShippingContext ctx) {
        // Express = 1.5x standard. Compose strategy thay vì duplicate logic.
        return standardCalc.calculateFee(ctx).multiply(BigDecimal.valueOf(1.5));
    }
}

public class SameDayShippingStrategy implements ShippingFeeStrategy {
    @Override
    public Money calculateFee(ShippingContext ctx) {
        if (!ctx.city().isInnerCityHCM() && !ctx.city().isInnerCityHN()) {
            throw new UnsupportedShippingException("Same-day chỉ trong nội thành HCM/HN");
        }
        return Money.of(80_000, VND);  // Flat fee
    }
}

public class InternationalShippingStrategy implements ShippingFeeStrategy {
    private final DhlApiClient dhl;

    public InternationalShippingStrategy(DhlApiClient dhl) { this.dhl = dhl; }

    @Override
    public Money calculateFee(ShippingContext ctx) {
        // External API call — strategy hide complexity từ caller
        return dhl.quote(ctx.destinationCountry(), ctx.weightKg(), ctx.declaredValue());
    }
}

// Context — không biết implementation cụ thể
public class CheckoutService {
    private final ShippingFeeStrategy shippingStrategy;

    public CheckoutService(ShippingFeeStrategy shippingStrategy) {
        this.shippingStrategy = shippingStrategy;
    }

    public Money totalCost(Order order) {
        Money items = order.itemsTotal();
        Money shipping = shippingStrategy.calculateFee(order.shippingContext());
        return items.add(shipping);
    }
}
🎯 Vì sao Strategy > if-else?
  • Open-Closed: thêm method #6 (Pickup-At-Store) = thêm 1 class mới, KHÔNG sửa CheckoutService. Code cũ không bị regression.
  • Testability: mỗi strategy test riêng. StandardShippingStrategyTest không cần biết Express tồn tại.
  • Single Responsibility: mỗi class chỉ 1 reason to change (logic của method đó). Khi DHL đổi pricing → chỉ sửa InternationalShippingStrategy.
  • Polymorphism > conditional: code đọc tự nhiên hơn. strategy.calculateFee(ctx) rõ intent hơn switch(type).
⚠️ Khi nào KHÔNG dùng Strategy?

2–3 case đơn giản, logic 1-2 dòng → if-else đủ. Đừng over-engineer:

// OK — không cần Strategy
Money fee = isVip ? Money.ZERO : Money.of(20_000, VND);

Factory Pattern — "Centralize object creation"

Khi đã có 5+ strategies, làm sao caller chọn đúng implementation dựa trên runtime input (như shippingType từ user)?

❌ Caller tự chọn
// Logic chọn rải rác khắp codebase
String type = request.shippingType();
ShippingFeeStrategy strategy;
if (type.equals("STANDARD"))
    strategy = new StandardShippingStrategy();
else if (type.equals("EXPRESS"))
    strategy = new ExpressShippingStrategy(...);
else if (type.equals("SAME_DAY"))
    strategy = new SameDayShippingStrategy();
// ... 5 lần, mỗi nơi caller chọn
✅ Centralized factory
@Component
public class ShippingStrategyFactory {
    private final Map<ShippingType, ShippingFeeStrategy>
        strategies;

    public ShippingStrategyFactory(
            List<ShippingFeeStrategy> allStrategies) {
        // Spring auto-inject TẤT CẢ implementations
        this.strategies = allStrategies.stream()
            .collect(toMap(
                ShippingFeeStrategy::supportedType,
                Function.identity()
            ));
    }

    public ShippingFeeStrategy get(ShippingType type) {
        var strategy = strategies.get(type);
        if (strategy == null) throw new UnsupportedShippingException(type);
        return strategy;
    }
}
🎯 Vì sao centralized factory?
  • Single source of truth: thêm method mới = đăng ký 1 chỗ. Không cần update 10 callers.
  • Dependency injection friendly: factory bean wire dependencies cho strategies 1 lần. Caller không cần biết DHL client tồn tại.
  • Testability: trong test, inject MockShippingStrategyFactory trả về mock strategy. Caller code không đổi.
  • Spring magic: inject List<ShippingFeeStrategy> tự động lấy mọi @Component implementing interface — auto-discover, không hardcode.
⚠️ Khi nào factory là overkill?

1 class duy nhất, constructor đơn giản → new SomeClass() trực tiếp. Factory chỉ có giá trị khi:

  • Có nhiều implementations để select runtime
  • Object cần wire complex dependencies
  • Cần thay implementation để test (mock factory)

Builder Pattern — "Construct complex objects readably"

Tình huống: API search cho ride-sharing app (Grab, Uber). Search request có ~12 optional filters:

❌ Constructor 12 tham số — unreadable, dễ swap nhầm
SearchRequest req = new SearchRequest(
    "HCM",              // pickupCity
    LocalDateTime.now(), // pickupTime
    "Q1",                // pickupDistrict
    "Q7",                // dropoffDistrict
    VehicleType.CAR,     // vehicle
    4,                   // maxPassengers
    null,                // promoCode
    true,                // allowSharedRide
    BigDecimal.valueOf(100_000), // maxPrice
    "VND",               // currency
    PaymentMethod.WALLET, // payment
    false                // hasPet
);
// Junior swap nhầm pickup vs dropoff district → bug khó tìm
✅ Builder — fluent, self-documenting
SearchRequest req = SearchRequest.builder()
    .pickup("HCM", "Q1")
    .dropoff("HCM", "Q7")
    .pickupTime(LocalDateTime.now())
    .vehicle(VehicleType.CAR)
    .maxPassengers(4)
    .allowSharedRide(true)
    .maxPrice(Money.of(100_000, VND))
    .payment(PaymentMethod.WALLET)
    .build();  // Validate tại đây
public class SearchRequest {
    private final String pickupCity, pickupDistrict, dropoffCity, dropoffDistrict;
    private final LocalDateTime pickupTime;
    private final VehicleType vehicle;
    private final int maxPassengers;
    private final boolean allowSharedRide;
    private final Money maxPrice;
    private final PaymentMethod payment;

    private SearchRequest(Builder b) {
        this.pickupCity = b.pickupCity;
        this.pickupDistrict = b.pickupDistrict;
        // ... copy from builder
    }

    public static Builder builder() { return new Builder(); }

    public static class Builder {
        private String pickupCity, pickupDistrict, dropoffCity, dropoffDistrict;
        private LocalDateTime pickupTime;
        private VehicleType vehicle = VehicleType.CAR;  // default
        private int maxPassengers = 1;
        private boolean allowSharedRide = false;
        private Money maxPrice;
        private PaymentMethod payment = PaymentMethod.WALLET;

        public Builder pickup(String city, String district) {
            this.pickupCity = city;
            this.pickupDistrict = district;
            return this;
        }

        public Builder dropoff(String city, String district) {
            this.dropoffCity = city;
            this.dropoffDistrict = district;
            return this;
        }

        public Builder pickupTime(LocalDateTime t) { this.pickupTime = t; return this; }
        public Builder vehicle(VehicleType v) { this.vehicle = v; return this; }
        public Builder maxPassengers(int n) { this.maxPassengers = n; return this; }
        public Builder allowSharedRide(boolean b) { this.allowSharedRide = b; return this; }
        public Builder maxPrice(Money m) { this.maxPrice = m; return this; }
        public Builder payment(PaymentMethod p) { this.payment = p; return this; }

        public SearchRequest build() {
            // Validate khi build — fail fast
            validateRequired();
            validateBusinessRules();
            return new SearchRequest(this);
        }

        private void validateRequired() {
            if (pickupCity == null) throw new IllegalArgumentException("pickup required");
            if (dropoffCity == null) throw new IllegalArgumentException("dropoff required");
            if (pickupTime == null) throw new IllegalArgumentException("pickupTime required");
        }

        private void validateBusinessRules() {
            if (pickupTime.isBefore(LocalDateTime.now())) {
                throw new IllegalArgumentException("Cannot book past time");
            }
            if (maxPassengers < 1 || maxPassengers > vehicle.maxCapacity()) {
                throw new IllegalArgumentException("Invalid passenger count for " + vehicle);
            }
            if (vehicle == VehicleType.MOTORBIKE && allowSharedRide) {
                throw new IllegalArgumentException("Motorbike cannot share ride");
            }
        }
    }
}
🎯 Vì sao Builder > constructor / setter approach?
  • Readability: caller nhìn biết ngay field nào set, default nào dùng. Constructor 12 tham số = guess game.
  • Immutability: SearchRequest không có setter, không bị mutate sau construction. Thread-safe.
  • Validation at boundary: build() validate trước khi tạo object → invalid object never exists. Setter pattern allow create rồi mutate thành invalid state.
  • Optional fields: builder có default, không cần truyền null. Caller chỉ set field cần thiết.
  • Composable validation: cross-field rule (motorbike + sharedRide) check ở build() — single place.
💡 Lombok shortcut

Trong project thật, dùng @Builder của Lombok auto-generate code này. Nhưng vẫn nên hiểu cơ chế bên trong để debug khi cần.

Observer / Event Pattern — "Decouple producer khỏi consumers"

Tình huống: Khi 1 order được đặt thành công, hệ thống cần làm 5 việc:

  1. Trừ stock trong warehouse
  2. Gửi email confirmation
  3. Push notification cho seller
  4. Update analytics dashboard (orders/min)
  5. Trigger fraud detection (review patterns)
❌ Tightly-coupled orchestration
@Service
public class PlaceOrderService {
  public void placeOrder(...) {
    Order order = createOrder(...);
    orderRepo.save(order);

    // Coupling tight với 5 services
    inventoryService.reduceStock(order);
    emailService.sendConfirmation(order);
    pushService.notifySeller(order);
    analyticsService.recordOrder(order);
    fraudService.review(order);
  }
}
✅ Event-driven
@Service
public class PlaceOrderService {
  private final ApplicationEventPublisher events;

  public void placeOrder(...) {
    Order order = createOrder(...);
    orderRepo.save(order);

    // Chỉ 1 event — không biết ai listen
    events.publishEvent(new OrderPlacedEvent(order));
  }
}

// Mỗi listener là 1 class riêng
@Component
public class InventoryListener {
  @EventListener
  @Async
  public void on(OrderPlacedEvent e) {
    inventoryService.reduceStock(e.order());
  }
}
// ... 4 listener khác tương tự
🎯 Vì sao Event-Driven > Direct call?
  • Add subscriber không sửa publisher: thêm "loyalty points update" sau order = thêm 1 listener mới, KHÔNG sửa PlaceOrderService. Open-Closed applied.
  • Independent failure: fraud service crash không làm fail order. Trong tightly-coupled version, exception ở fraud sẽ rollback order.
  • Independent scale: analytics chậm (DB heavy) không block inventory (fast). Mỗi listener có thread pool riêng nếu cần.
  • Testability: test PlaceOrderService chỉ cần verify event published (1 mock). Không cần mock 5 services.
⚠️ Trade-off — Sync vs Async

Spring @EventListener mặc định synchronous — chạy cùng thread publisher. Nếu listener chậm → block. Dùng @Async cho listeners không critical (notification, analytics). Critical listeners (inventory — phải atomic với order) giữ sync.

Với event cross-service (Kafka), pattern tương tự nhưng eventual consistency — listener có thể trễ vài ms đến giây. Cần idempotent consumer (xem #18).

Decorator Pattern — "Add cross-cutting concerns mà không sửa class gốc"

Tình huống: bạn có HttpClient gọi API của 3rd-party (DHL, Stripe, ...). Muốn thêm:

  • Logging: log request/response
  • Retry: retry 3 lần nếu 5xx
  • Caching: cache GET response 60s
  • Metrics: count requests, measure latency
❌ Inheritance explosion
class LoggingDhlClient extends DhlClient { ... }
class RetryingLoggingDhlClient
    extends LoggingDhlClient { ... }
class CachingRetryingLoggingDhlClient
    extends RetryingLoggingDhlClient { ... }
// N clients × M behaviors = N×M classes
// Đổi thứ tự Retry trước Logging? Class mới
✅ Composition (Decorator)
HttpClient client = new DhlClient(config);
client = new LoggingClient(client);
client = new RetryingClient(client, 3);
client = new CachingClient(client, Duration.ofSeconds(60));
client = new MetricsClient(client, registry);

// Dùng như HttpClient bình thường
client.get("/quote");
// Order: Metrics → Caching → Retry → Logging → DHL
//        (outer)                              (inner)
public interface HttpClient {
    HttpResponse get(String url);
    HttpResponse post(String url, String body);
}

// Decorator base
public abstract class HttpClientDecorator implements HttpClient {
    protected final HttpClient delegate;
    protected HttpClientDecorator(HttpClient delegate) { this.delegate = delegate; }
}

public class LoggingClient extends HttpClientDecorator {
    private static final Logger log = LoggerFactory.getLogger(LoggingClient.class);
    public LoggingClient(HttpClient delegate) { super(delegate); }

    @Override
    public HttpResponse get(String url) {
        log.info("GET {}", url);
        try {
            HttpResponse resp = delegate.get(url);
            log.info("GET {} → {}", url, resp.status());
            return resp;
        } catch (Exception e) {
            log.error("GET {} failed: {}", url, e.getMessage());
            throw e;
        }
    }
    // ... post similar
}

public class RetryingClient extends HttpClientDecorator {
    private final int maxAttempts;

    public RetryingClient(HttpClient delegate, int maxAttempts) {
        super(delegate);
        this.maxAttempts = maxAttempts;
    }

    @Override
    public HttpResponse get(String url) {
        Exception lastError = null;
        for (int i = 0; i < maxAttempts; i++) {
            try {
                HttpResponse resp = delegate.get(url);
                if (resp.status() < 500) return resp;  // 2xx/3xx/4xx — không retry
            } catch (Exception e) {
                lastError = e;
            }
            sleep(100 * (1L << i));  // Exponential backoff
        }
        throw new RuntimeException("Failed after " + maxAttempts + " attempts", lastError);
    }
}

public class CachingClient extends HttpClientDecorator {
    private final Cache<String, HttpResponse> cache;

    public CachingClient(HttpClient delegate, Duration ttl) {
        super(delegate);
        this.cache = Caffeine.newBuilder().expireAfterWrite(ttl).maximumSize(1000).build();
    }

    @Override
    public HttpResponse get(String url) {
        return cache.get(url, k -> delegate.get(k));
    }

    @Override
    public HttpResponse post(String url, String body) {
        return delegate.post(url, body);  // POST không cache
    }
}
🎯 Vì sao Decorator > Inheritance?
  • Composability: bạn có thể stack any combination của decorators. Khách hàng A muốn Logging + Retry, khách hàng B muốn Metrics + Caching → cùng base class, khác stack.
  • Reusable across types: LoggingClient dùng được với DHL, Stripe, FedEx — không cần class riêng LoggingDhlClient, LoggingStripeClient.
  • Runtime composition: order có thể đổi runtime dựa trên config. Inheritance fix compile-time.
  • Single Responsibility: mỗi decorator chỉ 1 concern. Test riêng dễ.
💡 Spring AOP = Decorator dynamic

Spring @Async, @Cacheable, @Retryable, @Transactional đều là decorator dynamic (qua proxy). Bạn không viết decorator class, Spring tạo proxy decorator runtime. Cùng concept, ít boilerplate.

Chain of Responsibility — "Pipeline xử lý"

Tình huống: trước khi 1 message được post lên 1 social platform (Facebook/Twitter clone), cần qua nhiều validation step:

  1. Authentication: user logged in?
  2. Rate limit: user post > 100 messages/hour?
  3. Content moderation: contain hate speech / NSFW?
  4. Spam detection: là spam pattern không?
  5. Mention validation: @mentions hợp lệ?
// Handler base
public abstract class PostHandler {
    protected PostHandler next;

    public PostHandler setNext(PostHandler next) {
        this.next = next;
        return next;  // Fluent — chain.setNext(a).setNext(b)
    }

    public final void handle(PostRequest req) {
        process(req);
        if (next != null) next.handle(req);
    }

    protected abstract void process(PostRequest req);
}

public class AuthHandler extends PostHandler {
    @Override
    protected void process(PostRequest req) {
        if (req.userId() == null) throw new UnauthenticatedException();
    }
}

public class RateLimitHandler extends PostHandler {
    private final RedisRateLimiter limiter;

    public RateLimitHandler(RedisRateLimiter limiter) { this.limiter = limiter; }

    @Override
    protected void process(PostRequest req) {
        if (!limiter.tryAcquire(req.userId(), "post", 100, Duration.ofHours(1))) {
            throw new RateLimitExceededException();
        }
    }
}

public class ModerationHandler extends PostHandler {
    private final ContentModerationApi moderation;

    @Override
    protected void process(PostRequest req) {
        var result = moderation.classify(req.content());
        if (result.containsHateSpeech() || result.containsNsfw()) {
            throw new ContentRejectedException(result.reason());
        }
    }
}

public class SpamHandler extends PostHandler {
    @Override
    protected void process(PostRequest req) {
        if (SpamDetector.isSpam(req.content())) throw new SpamDetectedException();
    }
}

// Configuration — build chain
@Configuration
public class PostHandlerChainConfig {
    @Bean
    public PostHandler postHandlerChain(AuthHandler auth, RateLimitHandler rateLimit,
                                         ModerationHandler mod, SpamHandler spam) {
        auth.setNext(rateLimit).setNext(mod).setNext(spam);
        return auth;
    }
}

// Usage
@Service
public class PostService {
    private final PostHandler chain;

    public PostService(PostHandler chain) { this.chain = chain; }

    public void post(PostRequest req) {
        chain.handle(req);  // Run through all handlers
        savePost(req);      // Reach here only if all handlers pass
    }
}
🎯 Vì sao Chain > monolithic validator?
  • Add/remove rules dễ: thêm "ProfanityFilter" = thêm 1 handler vào chain, không sửa code cũ.
  • Test riêng: SpamHandler test với spam input, không cần setup full chain.
  • Configurable order: dev environment có thể skip ModerationHandler (slow API call). Production có full chain.
  • Single Responsibility: mỗi handler 1 concern. Đổi rule spam không động đến code rate limit.
  • Reusable: AuthHandler dùng được cho post, comment, like — không duplicate.

4. Anti-Patterns Phổ Biến

#1 Pattern Fever — Áp dụng cho mọi vấn đề

❌ Factory cho 2-case logic
public class NotifierFactory {
  public Notifier get(boolean isVip) {
    return isVip
      ? new VipNotifier()
      : new RegularNotifier();
  }
}
// Indirection không có benefit
✅ Trực tiếp
Notifier n = isVip
  ? new VipNotifier()
  : new RegularNotifier();
🎯 Vì sao tránh?

Pattern thêm indirection. Indirection có cost (đọc code phải mở thêm file). Cost này chỉ đáng nếu có benefit (extensibility, testability, ...). Với 2-case stable logic → không có benefit → factory chỉ tạo noise.

#2 Strategy với State — Recreate object liên tục

// ❌ New strategy mỗi request — waste object creation
@Service
public class PriceService {
    public Money calculate(Order o, PromotionType type) {
        PricingStrategy strategy = switch (type) {
            case FLASH_SALE -> new FlashSalePricingStrategy(promoRepo);  // NEW mỗi call
            case LOYALTY -> new LoyaltyPricingStrategy(loyaltyRepo);
            default -> new RegularPricingStrategy();
        };
        return strategy.apply(o);
    }
}

// ✅ Inject strategies as beans, dùng factory để select
@Service
public class PriceService {
    private final PricingStrategyFactory factory;
    public Money calculate(Order o, PromotionType type) {
        return factory.get(type).apply(o);
    }
}
🎯 Vì sao sai?

Strategies stateless → tạo 1 lần, reuse mãi. New instance mỗi request gây GC pressure, đặc biệt khi strategy có dependencies (phải re-locate). Dùng Spring beans = lifecycle managed, dependencies injected 1 lần.

#3 Observer sync — Listener làm tắc nghẽn publisher

// ❌ Sync listener, SMS API chậm
@EventListener(OrderPlacedEvent.class)
public void on(OrderPlacedEvent e) {
    smsGateway.send(e.customerPhone(), "Order confirmed");
    // SMS gateway average 3s, sometimes 30s timeout
    // → Order API trả về sau 30s → customer retry → duplicate order
}

// ✅ Async
@EventListener(OrderPlacedEvent.class)
@Async
public void on(OrderPlacedEvent e) {
    smsGateway.send(...);
    // SMS chạy thread riêng, không block order flow
}
🎯 Vì sao mặc định Spring sync?

Sync events có transactional guarantees — listener chạy trong cùng @Transactional với publisher. Khi listener throw → cả transaction rollback. Tốt cho listener critical (inventory phải atomic với order). BAD cho listener slow hoặc không critical → đánh dấu @Async + tách thread pool.

#4 Builder không validate — Tạo invalid object

// ❌ build() không check
public SearchRequest build() {
    return new SearchRequest(this);  // Không validate
}

// Caller tạo object invalid:
SearchRequest req = SearchRequest.builder().build();
// pickup=null, dropoff=null → bug ở step xử lý sau

// ✅ Validate at build()
public SearchRequest build() {
    if (pickupCity == null) throw new IllegalArgumentException("pickup required");
    if (dropoffCity == null) throw new IllegalArgumentException("dropoff required");
    return new SearchRequest(this);
}
🎯 Vì sao validate ở build()?

Principle "fail fast at boundary". Object invalid không bao giờ tồn tại nếu validate ở build(). Nếu để invalid object sống và validate sau (ở use case, controller) → error message xa khỏi root cause → debug khó.

#5 Decorator + Inheritance lẫn lộn

// ❌ Inheritance cho cross-cutting
class LoggingDhlClient extends DhlClient { ... }
class RetryingDhlClient extends LoggingDhlClient { ... }
// Đổi thứ tự retry-logging hoặc thêm caching? Phải tạo class mới

// ✅ Composition
HttpClient client = new DhlClient(config);
client = new RetryingClient(client, 3);
client = new LoggingClient(client);
// Đổi thứ tự dễ, thêm decorator dễ
🎯 Inheritance vs Composition cho cross-cutting

Inheritance fix relationship compile-time. Cross-cutting concerns (logging, retry, cache) là orthogonal — không tự nhiên thuộc về 1 class cụ thể. Composition + interface cho phép mix-and-match runtime.

5. Interview Framework

Tầng 1 — SurfaceMid gate
Q: Giải thích Strategy pattern. Khi nào dùng thay if-else?
"Strategy đóng gói algorithm trong 1 class riêng implement interface chung. Context chỉ dùng interface, không biết implementation cụ thể. Dùng khi có 5+ variant hoặc algorithm hay thay đổi, vì thêm variant mới = thêm class, không sửa context (Open-Closed). Replace if-else bằng polymorphism. Trade-off: thêm classes → cost nhỏ. Benefit: testable riêng, isolated change. Với 2-3 case đơn giản → if-else vẫn tốt hơn."
Q: Builder pattern là gì, vì sao dùng thay vì constructor?
"Builder cung cấp fluent API để tạo object phức tạp với nhiều optional parameters. Constructor 10 tham số là unreadable và dễ swap nhầm vị trí (cùng kiểu String). Builder làm optional fields explicit qua method name. Validation ở build() đảm bảo invalid object never exists. Dễ test (chỉ set fields cần cho test). Lombok @Builder auto-generate."
Tầng 2 — Deep DiveMid → Senior
Q: Design payment processing system dùng Strategy + Factory + Decorator. Methods: Momo, VNPay. Add logging và retry.
"Strategy: PaymentGateway interface với MomoPaymentGateway, VNPayPaymentGateway implementations.
Factory: PaymentGatewayFactory map từ PaymentMethod enum → bean. Inject List<PaymentGateway> từ Spring, auto-discover.
Decorator: LoggingPaymentGateway wrap gateway, log request/response. RetryingPaymentGateway wrap thêm để retry 3 lần với exponential backoff.
Usage: factory.get(MOMO) trả gateway đã wrap decorators (config qua Spring AOP hoặc explicit wrap trong factory).
Thêm method mới: tạo class implement PaymentGateway, Spring auto-pick up. Factory tự register. Decorators apply automatically. 0 dòng sửa code cũ."
Q: Có 4 sync event listeners khi payment completed. 1 listener (fraud detection) chậm 5s. Fix sao không break listeners khác?
"Option 1 — Async chỉ listener slow: đánh dấu fraud listener @Async. Payment trả về customer ngay, fraud detection chạy background. Nếu fraud detect anomaly, tạo separate alert (không revert payment đã confirmed).
Option 2 — All async: tất cả listeners @Async. Payment trả về customer ngay sau khi save. Trade-off: nếu inventory update fail (out of stock), order đã confirmed → cần compensation flow.
Option 3 — Move to message queue: publish event lên Kafka, mỗi listener là service riêng consume. Decouple hoàn toàn — fraud service deploy độc lập, scale độc lập.
Recommendation: Option 1 cho near-realtime systems. Option 3 cho microservices architecture cần independent deployability."
Tầng 3 — ArchitectureSenior
Q: Design pipeline xử lý request lên platform Q&A (giống StackOverflow). Có 6 step: auth, rate-limit, spam check, content moderation, duplicate detection, tag validation. Mỗi step có thể async hoặc sync. Step nào fail thì abort.
"Chain of Responsibility với extension:

1. Handler abstraction: mỗi step implement QuestionHandler interface có method handle(QuestionRequest req) trả về HandlerResult (PASS/FAIL/SKIP).

2. Configurable chain: chain order config qua Spring profile. Production: full 6 steps. Dev: skip moderation + duplicate detection (slow). Test: chỉ auth.

3. Hybrid sync/async:
• Sync (must pass before save): auth, rate-limit, spam check
• Async (run after save, can flag/revert later): moderation, duplicate detection, tag validation
Đặt 2 chain riêng: preSaveChain và postSaveChain.

4. Result aggregation: nếu pre-save fail → throw exception → caller nhận error. Post-save fail → mark question là "under review" trong DB, gửi event QuestionFlaggedEvent → notification system báo moderator.

5. Observability: mỗi handler emit metric (handler_name, duration, result). Grafana dashboard show bottleneck handler. Add new handler #7 = thêm 1 class + register vào chain config.

6. Testability: handler test riêng với mock dependencies. Chain test với fake handlers verify order + abort behavior."
