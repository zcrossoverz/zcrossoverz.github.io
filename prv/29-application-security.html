Application Security — OWASP, Injection, CSRF, IDOR

Security là nơi hiểu giả dễ chết người nhất: "có dùng prepared statement", "có HTTPS" nghe đủ — nhưng "vì sao parameterized query chặn injection ở tầng nào?", "CSRF cần token cho form nhưng KHÔNG cần cho JWT API — vì sao?", "IDOR khác broken auth chỗ nào?" thì lộ. File này dạy cơ chế tấn công và phòng thủ, trên domain ShopFlow — tài khoản & đơn hàng của sàn e-commerce.

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Dùng prepared statement là an toàn SQL injection. Escape input cũng được."
Thực tế: Parameterized query an toàn vì tách câu lệnh khỏi dữ liệu — DB nhận query đã compile + data là tham số, data không bao giờ được parse như SQL. Tự escape thì luôn sót case. Nhưng nối chuỗi vào tên bảng/cột (không tham số hoá được) vẫn injectable.
Tưởng hiểu: "CSRF với CORS là một, đều về cross-origin."
Thực tế: CSRF = trình duyệt tự gửi cookie/session khi bị dụ submit form sang site bạn → cần CSRF token. CORS = cơ chế trình duyệt cho phép/chặn JS đọc response cross-origin. JWT trong header (không phải cookie) thì KHÔNG bị CSRF (trình duyệt không tự gắn header).
Tưởng hiểu: "Đã login (authentication) là an toàn truy cập."
Thực tế: Authentication (bạn là ai) ≠ authorization (bạn được làm gì). IDOR: user A login hợp lệ nhưng đổi orderId trên URL để xem đơn của user B. Phải check ownership ở server, không tin ID client gửi.

1. Bài toán thực tế — rò rỉ & chiếm đơn CORE ★

ShopFlow gặp 3 sự cố: (1) một query search nối chuỗi input → kẻ tấn công dump cả bảng user; (2) user đổi ?orderId=1002 trên URL xem được đơn người khác; (3) form đổi email bị site khác submit hộ qua cookie → tài khoản bị chiếm.

⏸ Pause & Predict — trước khi đọc lời giải
Đặt tên đúng từng lỗ hổng và cách chặn:
  1. Sự cố (1) tên là gì? Vì sao "escape dấu nháy" không đủ mà parameterized query mới đúng?
  2. Sự cố (2) tên là gì? Vì sao "đã login" không ngăn được? Check gì ở server?
  3. Sự cố (3) tên là gì? Vì sao JWT trong header sẽ không bị, nhưng session cookie thì bị?
Đáp án: (1) SQL Injection — parameterized tách lệnh khỏi data (mục 3); (2) IDOR — check ownership server, không tin ID client (mục 6); (3) CSRF — cần token; cookie tự gửi còn JWT header thì không (mục 5). Đừng đọc tiếp khi chưa đặt đúng 3 tên.

2. OWASP Top 10 — cốt lõi SUPPLEMENTARY

Một số mục hay gặp trong backend e-commerce: Broken Access Control (gồm IDOR), Injection (SQL/NoSQL/command), Cryptographic Failures (PII không mã hoá), Security Misconfiguration, Identification & Authentication Failures, SSRF. Học theo cơ chế chứ không thuộc danh sách.

3. SQL Injection — vì sao parameterized fix CORE ★

❌ Nối chuỗi — injectable

"SELECT * FROM users WHERE email='" + input + "'"
// input = x' OR '1'='1  → trả mọi user

✅ Parameterized — tách lệnh khỏi data

jdbc.query("SELECT * FROM users WHERE email = ?", input);
// query compile trước; input là DATA, không parse như SQL
Vẫn injectable dù dùng PreparedStatement

Tham số chỉ thay cho giá trị, không thay cho tên bảng/cột/ORDER BY. "ORDER BY " + sortColumn vẫn injectable → phải whitelist sortColumn trong tập cho phép.

5. CSRF vs CORS — hay nhầm CORE ★

L1 — Biết tên

Biết "CSRF cần token". Symptom hiểu giả: nghĩ CSRF và CORS là một, hoặc bật CSRF token cho REST API stateless dùng JWT (thừa).

L2 — Explain được

CSRF lợi dụng việc trình duyệt tự gửi cookie khi request sang origin của bạn → kẻ tấn công dụ submit form. Token chống vì site khác không đoán được token. CORS là cơ chế khác: kiểm soát JS đọc response cross-origin, không phải để chống CSRF.

L3 — Apply được

Session cookie → cần CSRF token (hoặc SameSite=Strict/Lax). JWT trong Authorization header → không bị CSRF (trình duyệt không tự gắn header) nên không cần CSRF token. Chọn cơ chế chống theo cách lưu credential.

6. IDOR — đừng tin client ID CORE ★

// ❌ tin orderId từ URL → user xem đơn người khác
Order o = orderRepo.findById(orderId).orElseThrow();

// ✅ check ownership: ràng buộc theo user đang đăng nhập
Order o = orderRepo.findByIdAndCustomerId(orderId, currentUser.getId())
                   .orElseThrow(NotFoundException::new);   // không lộ tồn tại

Trả 404 chứ không 403 để không tiết lộ resource có tồn tại.

7. Rate limiting + PII masking SUPPLEMENTARY

  • Rate limit login (theo IP + theo account) chống brute-force/credential stuffing; thêm exponential backoff, captcha sau N lần.
  • PII masking trong log: không bao giờ log email đầy đủ, số thẻ, địa chỉ, token. Mask a***@mail.com, chỉ giữ 4 số cuối thẻ.
  • Security headers: CSP, HSTS, X-Content-Type-Options, X-Frame-Options.

⚠️ Anti-patterns CORE ★

🔧 Counter-example — khi "phòng thủ" lại tạo lỗ hổng / sai chỗ
  • Tự escape thay parameterized: blacklist ký tự luôn sót (Unicode, comment, encoding). Phòng thủ đúng tầng là tách lệnh/data, không lọc input.
  • Verbose error: trả stack trace/SQL ra client = chỉ đường cho attacker. Log chi tiết server, trả message chung cho client.
  • Validate ở client là đủ: client validation chỉ cho UX; attacker bỏ qua client → mọi check bảo mật phải ở server.
  • Bật CSRF token cho JWT API stateless: thừa, gây hiểu sai mô hình; ngược lại quên SameSite cho session cookie mới là lỗ hổng.

Quy tắc: chặn ở đúng tầng (parameterized cho injection, output encoding cho XSS, ownership check cho IDOR), và không tin bất cứ gì từ client.

🎯 3-tier interview framework

Tầng 1 — SurfaceJunior gate

Q: Chặn SQL injection thế nào?

Parameterized query/PreparedStatement — tách câu lệnh khỏi dữ liệu, data không bao giờ parse như SQL. Whitelist cho phần không tham số hoá được (tên cột, ORDER BY). Không tự escape.

Tầng 2 — Deep DiveSenior gate

Q: Thiết kế API xử lý PII an toàn.

TLS bắt buộc; authorization theo ownership (chống IDOR); PII mã hoá at-rest, masking trong log; rate limit; input validation server-side + output encoding; security headers; audit log truy cập PII; least-privilege DB account.

Tầng 3 — ArchitectureSenior+

Q: CSRF vs CORS — giải thích sự nhầm lẫn phổ biến.

CSRF: trình duyệt tự gửi cookie → cần token/SameSite; ảnh hưởng app dùng session cookie. CORS: cơ chế trình duyệt cho JS đọc response cross-origin — KHÔNG phải chống CSRF. JWT header không bị CSRF. Nhầm hai cái → bật sai cơ chế, vừa thừa vừa hở.

🔧 Lab bắt buộc — khai thác & vá (2 bài)

Lab 1 — SQL Injection. Viết một endpoint search nối chuỗi input vào SQL. Khai thác bằng ' OR '1'='1 để dump bảng. Vá bằng parameterized query, thử lại → vô hại. Thêm case ORDER BY động → chứng minh vẫn injectable, vá bằng whitelist. Tiêu chí: giải thích tách lệnh/data.

Lab 2 — IDOR. Tạo endpoint GET /orders/{id} chỉ findById. Login user A, đổi id sang đơn của user B → xem được (lỗ hổng). Vá bằng findByIdAndCustomerId(currentUser), trả 404. Tiêu chí: phân biệt authentication vs authorization.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. Vì sao parameterized query chặn SQL injection? Trường hợp nào vẫn injectable?
  2. IDOR là gì? Vì sao "đã login" không ngăn được? Check gì ở server?
  3. CSRF vs CORS khác nhau ra sao? Vì sao JWT header không bị CSRF?
  4. XSS chặn chủ yếu ở đâu — input hay output? Vì sao?
  5. Vì sao mọi check bảo mật phải ở server dù client đã validate?
Bí câu nào = quay lại đúng section. Trôi cả 5 = không bị hiểu giả về security.
