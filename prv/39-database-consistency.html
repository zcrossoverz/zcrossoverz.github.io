Database Consistency — CAP, MVCC, Isolation Anomalies

"READ COMMITTED tránh dirty read" là câu ai cũng thuộc — nhưng vì sao hai transaction cùng commit mà số dư vẫn sai? Vì sao REPEATABLE_READ của PostgreSQL khác hẳn SERIALIZABLE của Oracle dù tên giống? File này tách anomaly có tên cụ thể (lost update, write skew, read skew) khỏi cách nói mơ hồ "race condition", dạy MVCC tận xmin/xmax + UNDO/SCN, kèm code optimistic/pessimistic/SSI/idempotency thật, trên domain ShopFlow — ví store-credit & tồn kho của sàn e-commerce.

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Cứ gặp lỗi đồng thời thì gọi là race condition, set SERIALIZABLE là xong."
Thực tế: Mỗi hiện tượng có tên riêng (Lost Update, Write Skew, Read Skew) và cách fix khác nhau. SERIALIZABLE fix được nhưng abort transaction khi conflict → không có @Retryable thì user nhận 500 ngẫu nhiên.
Tưởng hiểu: "REPEATABLE_READ ở PostgreSQL và Oracle giống nhau vì cùng tên SQL standard."
Thực tế: PostgreSQL REPEATABLE_READ = Snapshot Isolation; Oracle không có REPEATABLE_READ (nhảy thẳng SERIALIZABLE, mà bản chất cũng chỉ là SI). PostgreSQL SERIALIZABLE = SSI (true serializability). Tên giống, hành vi khác → bug khi đổi DB.
Tưởng hiểu: "CAP nói chọn 2 trong 3 (C, A, P)."
Thực tế: Phát biểu đúng: khi có partition (P) mới phải chọn giữa C và A. P không phải thứ "chọn" — nó là sự cố. Single-node DB không có P nên CAP không áp dụng; dùng PACELC để nói cả lúc bình thường (Latency vs Consistency).

1. Bài toán thực tế — store credit trừ 1 lần dù dùng 2 lần CORE ★

ShopFlow cho khách dùng "store credit" (ví khuyến mãi) khi checkout. Mặc định PostgreSQL READ COMMITTED.

@Transactional(isolation = Isolation.READ_COMMITTED)
public void spendCredit(Long walletId, BigDecimal amount) {
    Wallet w = walletRepo.findById(walletId).orElseThrow();   // Bước 1: đọc balance
    if (w.getBalance().compareTo(amount) < 0)                 // Bước 2: kiểm tra đủ
        throw new InsufficientCreditException();
    w.setBalance(w.getBalance().subtract(amount));            // Bước 3: trừ (tính trong app)
    walletRepo.save(w);                                       // Bước 4: ghi
}

Khách double-click checkout → 2 request song song, mỗi cái dùng 500K. Ví có 600K. Kết quả: cả 2 thành công, ví âm 400K.

Lost Update: hai TX đọc cùng giá trị, ghi đè nhau. READ COMMITTED KHÔNG chặn.
⏸ Pause & Predict — trước khi đọc lời giải
Viết ra giấy:
  1. Đây KHÔNG chỉ là "race condition" — nó có tên cụ thể trong lý thuyết DB. Tên là gì?
  2. READ COMMITTED "tránh dirty read" — vậy vì sao 2 transaction commit thành công mà balance vẫn sai? READ COMMITTED KHÔNG chặn anomaly nào ở đây?
  3. Đổi lên REPEATABLE_READ của PostgreSQL thì điều gì xảy ra với transaction thứ hai? User thấy gì nếu code không retry?
Đáp án: đây là Lost Update (2.4); READ COMMITTED chỉ chặn dirty read, không chặn lost update; PostgreSQL REPEATABLE_READ sẽ báo "could not serialize access due to concurrent update" → cần retry (2.3, 3.1). Đừng đọc tiếp khi chưa đặt đúng tên anomaly.

2.1 CAP & PACELC đúng cách SUPPLEMENTARY

Phát biểu đúng của CAP

Khi xảy ra network partition (P), hệ phân tán buộc chọn giữa Consistency (C) và Availability (A). Không có chuyện "chọn 2 trong 3" lúc bình thường — P là sự cố, không phải lựa chọn.

PACELC (đầy đủ hơn CAP):
  if Partition: chọn Availability vs Consistency
  else (bình thường): chọn Latency vs Consistency

DynamoDB                : PA/EL  (ưu tiên A khi partition, L khi bình thường)
Spanner                 : PC/EC  (ưu tiên C luôn, đánh đổi latency qua TrueTime)
PostgreSQL 1 node       : không có P → CAP không áp dụng
PostgreSQL sync replica : PC/EC  (sync replica làm chậm write nhưng đảm bảo consistency)
MongoDB replica set     : PA/EL  (eventual trên secondary)

Vì sao tiền/tồn kho chọn PC/EC: thà không available còn hơn inconsistent — partition mà chọn AP → hai DC cùng trừ một ví/kho → double-spend/oversell mất tiền thật. Dữ liệu analytics/notification thì PA/EL chấp nhận được.

2.3 MVCC — PostgreSQL vs Oracle CORE ★

Ý tưởng: thay vì lock row khi write, tạo version mới. Reader đọc version cũ, writer tạo version mới → không ai block ai ("writers don't block readers").

PostgreSQL: mỗi row ẩn 2 cột xmin (transaction tạo) và xmax (transaction xoá).

balance = 600K (xmin=50, xmax=NULL → visible cho mọi snapshot sau 50)

TX1 (xid=100) update → 100K:
   row cũ: 600K, xmin=50,  xmax=100   (TX1 đánh dấu xoá)
   row mới: 100K, xmin=100, xmax=NULL
TX1 commit.

TX2 (REPEATABLE_READ, snapshot lúc bắt đầu) update dựa trên 600K:
   → PostgreSQL: ERROR: could not serialize access due to concurrent update
   → application phải RETRY (optimistic concurrency control)

READ COMMITTED thì TX2 đọc LẠI row mới (100K) trước khi update → không lỗi nhưng dễ logic sai
nếu app đã tính toán dựa trên giá trị cũ.

L1 — Biết tên

Biết "MVCC = nhiều version, không lock đọc". Symptom hiểu giả: không giải thích được vì sao PostgreSQL cần VACUUM hay Oracle bị ORA-01555.

L2 — Explain được

PostgreSQL giữ version cũ trong table (xmin/xmax) → cần VACUUM dọn dead tuple. Oracle update in-place, version cũ vào UNDO segment; SCN xác định điểm nhìn → ORA-01555 "snapshot too old" khi UNDO bị ghi đè quá sớm.

L3 — Apply được

PostgreSQL: tune autovacuum, monitor table bloat, tránh long transaction giữ snapshot. Oracle: tune UNDO_RETENTION, kích thước UNDO tablespace. Đọc đúng error để biết chỉnh chỗ nào, không đoán mò.

3 gotcha vendor

(1) Oracle SERIALIZABLE thực chất là Snapshot Isolation → vẫn có Write Skew. (2) PostgreSQL SERIALIZABLE là SSI → true serializability, detect conflict lúc commit và abort. (3) MySQL REPEATABLE_READ (default) dùng gap lock nên chặn được phantom phần lớn trường hợp — khác SQL standard. Lưu ý: KHÔNG level nào tự chặn Lost Update — cần explicit lock.

2.6 Write Skew — anomaly Snapshot Isolation không fix CORE ★

-- Invariant: luôn ≥ 1 bác sĩ on-call. Hiện có Alice + Bob (on_duty=true).

-- TX1 (Alice xin off):                        -- TX2 (Bob xin off) chạy đồng thời:
BEGIN;                                          BEGIN;
SELECT count(*) FROM oncall                     SELECT count(*) FROM oncall
  WHERE shift='night' AND on_duty=true;  -- =2    WHERE shift='night' AND on_duty=true; -- =2 (snapshot cũ)
-- count >= 2 → OK cho off                       -- count >= 2 → OK cho off
UPDATE oncall SET on_duty=false                 UPDATE oncall SET on_duty=false
  WHERE name='Alice';                             WHERE name='Bob';
COMMIT;                                          COMMIT;
-- KẾT QUẢ: không ai on-call → vi phạm invariant

Vì sao SI không bắt: hai TX update disjoint rows (Alice vs Bob) → không có write-write conflict trên cùng row; mỗi TX đọc snapshot riêng thấy điều kiện thoả. SI chỉ detect conflict trên cùng row.

🔧 Counter-example — khi nâng isolation KHÔNG phải cách đúng

Phản xạ "thấy lỗi đồng thời → set SERIALIZABLE toàn bộ service" là sai hướng:

  • SSI abort khi conflict: không bọc @Retryable → transaction bị abort → user nhận 500 ngẫu nhiên.
  • Overhead: SSI phải track predicate lock → chậm đáng kể dưới tải cao.
  • Thường chỉ cần lock đúng chỗ: Lost Update fix bằng SELECT FOR UPDATE với READ COMMITTED là đủ và deterministic hơn.

Cách đúng: chọn isolation tối thiểu cần; materialize conflict bằng SELECT FOR UPDATE trên row "đại diện" (vd advisory lock theo shift_id) cho Write Skew. Xem 3.3.

3.1 Optimistic lock (@Version) — contention thấp CORE ★

@Entity @Table(name = "wallets")
public class Wallet {
    @Id private Long id;
    private BigDecimal balance;
    @Version private Long version;     // Hibernate tự increment + check
}

@Service
public class WalletService {
    @Transactional
    @Retryable(retryFor = ObjectOptimisticLockingFailureException.class,
               maxAttempts = 5,
               backoff = @Backoff(delay = 50, multiplier = 2, maxDelay = 500))
    public void spend(Long walletId, BigDecimal amount) {
        Wallet w = walletRepo.findById(walletId).orElseThrow();
        if (w.getBalance().compareTo(amount) < 0) throw new InsufficientCreditException();
        w.setBalance(w.getBalance().subtract(amount));
        walletRepo.save(w);            // UPDATE ... SET balance=?, version=version+1
    }                                  //         WHERE id=? AND version=?
                                       // row count = 0 → OptimisticLockException → retry

    @Recover
    public void onExhausted(ObjectOptimisticLockingFailureException ex, Long walletId, BigDecimal amount) {
        log.error("High contention on wallet {} after retries", walletId, ex);
        throw new HighContentionException("Đang bận, thử lại sau");
    }
}

Generated SQL: UPDATE wallets SET balance=?, version=version+1 WHERE id=? AND version=? — nếu version đã đổi (TX khác ghi trước) → 0 row → Hibernate ném exception → @Retryable đọc lại bản mới và thử lại. Trade-off: không lock (scalable) nhưng contention cao → nhiều retry tốn CPU; không hợp long transaction.

3.2 Pessimistic lock (FOR UPDATE) + chống deadlock CORE ★

public interface WalletRepository extends JpaRepository<Wallet, Long> {
    @Lock(LockModeType.PESSIMISTIC_WRITE)          // → SELECT ... FOR UPDATE
    @Query("SELECT w FROM Wallet w WHERE w.id = :id")
    Optional<Wallet> findByIdForUpdate(@Param("id") Long id);
}

@Transactional(isolation = Isolation.READ_COMMITTED)
public void transfer(Long fromId, Long toId, BigDecimal amount) {
    // LUÔN lock theo thứ tự ID cố định → không bao giờ deadlock chéo
    Long firstId  = Math.min(fromId, toId);
    Long secondId = Math.max(fromId, toId);
    Wallet first  = walletRepo.findByIdForUpdate(firstId).orElseThrow();
    Wallet second = walletRepo.findByIdForUpdate(secondId).orElseThrow();

    Wallet from = fromId.equals(firstId) ? first : second;
    Wallet to   = fromId.equals(firstId) ? second : first;
    if (from.getBalance().compareTo(amount) < 0) throw new InsufficientCreditException();
    from.setBalance(from.getBalance().subtract(amount));
    to.setBalance(to.getBalance().add(amount));
    walletRepo.save(from); walletRepo.save(to);
}
Deadlock symptom + fix

Triệu chứng: log deadlock detected, DB tự hủy một transaction ngẫu nhiên dưới tải. Nguyên nhân: TX1 lock A rồi xin B; TX2 lock B rồi xin A → chờ nhau. Fix: luôn acquire lock theo thứ tự cố định (MIN(id) trước MAX(id)) như trên. Pessimistic hợp khi contention cao (row "nóng") vì deterministic, không retry-storm.

3.3 SERIALIZABLE (PostgreSQL SSI) fix Write Skew SUPPLEMENTARY

@Transactional(isolation = Isolation.SERIALIZABLE)
@Retryable(retryFor = CannotSerializeTransactionException.class,
           maxAttempts = 3, backoff = @Backoff(delay = 100))
public void requestTimeOff(Long doctorId, String shift) {
    long onCall = oncallRepo.countByShiftAndOnDuty(shift, true);
    if (onCall < 2) throw new MinimumStaffingViolation();
    Oncall rec = oncallRepo.findByDoctorIdAndShift(doctorId, shift);
    rec.setOnDuty(false);
    oncallRepo.save(rec);
}
// PostgreSQL SSI: track predicate dependency; tại commit nếu phát hiện cycle trong
// dependency graph → abort 1 TX với "could not serialize access due to read/write
// dependencies among transactions" → @Retryable thử lại. KHÔNG có TX nào chắc chắn succeed.

Thay thế: SELECT ... FOR UPDATE trên các row đọc để materialize conflict (biến write-skew thành write-write conflict mà SI bắt được) — đôi khi đơn giản & nhanh hơn SSI.

3.4 Idempotency key — chống duplicate request CORE ★

Kể cả SERIALIZABLE, client submit 2 lần (timeout, double-click) vẫn có thể tạo 2 kết quả với id khác nhau. Idempotency key (unique constraint) giải quyết — và phải nằm cùng transaction với thao tác chính.

@Transactional(isolation = Isolation.READ_COMMITTED)
public OrderResult placeOrder(String idempotencyKey, OrderRequest req) {
    try {
        idempotencyRepo.saveAndFlush(new IdempotencyKey(idempotencyKey)); // INSERT unique
    } catch (DataIntegrityViolationException dup) {                        // đã tồn tại
        IdempotencyKey k = idempotencyRepo.findByKeyValue(idempotencyKey);
        return OrderResult.from(orderRepo.findById(k.getResourceId()).orElseThrow());
    }
    Order order = doPlaceOrder(req);                 // CÙNG transaction với insert key
    IdempotencyKey k = idempotencyRepo.findByKeyValue(idempotencyKey);
    k.setResourceType("ORDER"); k.setResourceId(order.getId());
    idempotencyRepo.save(k);
    return OrderResult.from(order);
}
Quan trọng

INSERT key và thao tác chính phải cùng transaction. Nếu tách ra, crash giữa chừng sẽ tạo key mồ côi (key đã ghi nhưng order chưa tạo) → request retry tưởng đã xử lý mà thực ra chưa.

3.5 Reconciliation — phát hiện inconsistency chủ động SUPPLEMENTARY

-- Chạy hằng đêm: tổng nợ phải = tổng có (ledger cân bằng)
SELECT
  SUM(CASE WHEN type='DEBIT'  THEN amount ELSE 0 END) AS total_debit,
  SUM(CASE WHEN type='CREDIT' THEN amount ELSE 0 END) AS total_credit,
  SUM(CASE WHEN type='DEBIT'  THEN amount ELSE -amount END) AS imbalance
FROM ledger_entries
WHERE created_date = CURRENT_DATE - INTERVAL '1 day';
-- imbalance != 0 → có anomaly đã xảy ra → alert ops team điều tra

Chọn cơ chế:
  Không shared mutable state           → READ COMMITTED (mặc định, nhanh)
  Update cùng row, contention thấp     → Optimistic (@Version) + retry

  Invariant phức tạp (Write Skew)      → SERIALIZABLE (SSI) + retry, hoặc FOR UPDATE materialize
  Cross-service / cross-shard          → Saga + Outbox (xem #13, #18)

5. 3-tier interview framework

Tầng 1 — SurfaceMid gate

Q: 4 isolation level chặn anomaly nào? Optimistic vs pessimistic?

RU: không chặn gì; RC: chặn dirty read; RR: + non-repeatable; SER: + phantom/write skew. Không level nào tự chặn lost update. Optimistic (@Version) = không lock, detect lúc commit, cần retry, contention thấp; Pessimistic (FOR UPDATE) = lock khi đọc, contention cao, rủi ro deadlock nếu sai lock order.

Tầng 2 — Deep DiveSenior gate

Q: Vì sao PostgreSQL REPEATABLE_READ không chặn Write Skew? Và vì sao chuyển tiền nên dùng FOR UPDATE thay vì RR retry?

RR = Snapshot Isolation, chỉ detect write-write conflict cùng row. Write Skew ghi disjoint rows (Alice vs Bob) sau khi đọc overlapping → SI không thấy conflict → cả hai commit, phá invariant. Fix: SERIALIZABLE (SSI predicate lock) hoặc SELECT FOR UPDATE materialize. Với chuyển tiền: FOR UPDATE cho hành vi deterministic (TX2 block rồi đọc giá trị mới, fail đúng logic) thay vì RR ném "could not serialize" buộc app retry → UX kém nếu quên retry.

Tầng 3 — ArchitectureSenior+

Q: Thiết kế hệ trừ tồn kho/ví 10K TPS, no lost update, no write skew.

Shard theo wallet_id (~1K TPS/shard, 10 shard). Per-shard: READ COMMITTED + SELECT FOR UPDATE, lock theo id order (no cross-shard deadlock). Cross-shard (chuyển A shard1 → B shard3): không lock 2 shard trong 1 TX → Saga + Outbox (debit A + outbox event → credit B + ack → compensation nếu fail). Idempotency key chống duplicate. Monitor: p99 lock wait, deadlock count (alert >0), retry rate (alert >5%).

🔧 Lab bắt buộc — reproduce anomaly (2 bài)

Lab 1 — Lost Update tận mắt. Mở 2 psql session, cùng BEGIN READ COMMITTED, cùng đọc balance=600, cả hai tính 600-500 trong "app" rồi UPDATE balance=100 WHERE id=1, commit. Quan sát balance sai.
