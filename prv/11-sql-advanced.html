Tổng Quan — Vì Sao SQL Critical Cho Senior

Junior viết SQL "chạy được". Senior viết SQL "chạy được trên 50M rows trong 50ms". Khác biệt nằm ở:

  • Hiểu cách DB lưu data (B-Tree, heap pages)
  • Predict được DB chọn plan nào cho query (index scan vs seq scan)
  • Đọc EXPLAIN ANALYZE fluent, không sợ
  • Biết khi nào index không help (functions in WHERE, leftmost prefix violation)
  • Trade-off giữa read perf vs write perf (mỗi index = +1 cost mỗi insert)
📜 Lịch sử của Index — Từ Hash Tables đến B-Tree
1960s-1970s — Database đầu (IBM IMS, hierarchical) dùng hash index cho equality lookup. Không support range query (BETWEEN, <, >). Mỗi insert hash key có thể trigger rehash toàn bộ — expensive.

1972 — Rudolf Bayer và Ed McCreight publish paper "Organization and Maintenance of Large Ordered Indices" introduce B-Tree. Self-balancing tree với 3 properties critical:
  • Mọi leaf cùng depth → predictable O(log n)
  • Node fan-out cao (100-1000 keys/node) → tree shallow, ít disk I/O
  • Range query natural (next sibling pointer)

1980s — B+Tree variant (data chỉ ở leaves, internal nodes chỉ keys) thành standard cho relational DB. Oracle, DB2 dùng.

1990s-2000s — PostgreSQL, MySQL InnoDB inherit B+Tree. Một số DB experiment với LSM-Tree (Cassandra, LevelDB) cho write-heavy workload — trade read perf cho write throughput.

Modern — B-Tree vẫn dominant cho OLTP (online transactional). LSM-Tree cho time-series/analytics. Hybrid (TokuDB, MyRocks) cho mixed workload.

Insight: B-Tree thắng vì balance giữa read/write/range query. Hiểu cấu trúc giúp predict performance trước khi chạy.

1. Bài Toán Thực Tế — Order Search API Của Sàn TMĐT

Sàn ShopVN có table orders 50M rows (5 năm operation, ~30K orders/ngày × 1800 ngày). API GET /api/customers/{id}/orders?status=DELIVERED&from=2026-01-01 mất 8 giây. SLA yêu cầu 500ms. Query trông đơn giản:

SELECT id, customer_id, total_amount, status, created_at, shipping_address
FROM orders
WHERE customer_id = ?
  AND status = 'DELIVERED'
  AND created_at > ?
ORDER BY created_at DESC
LIMIT 20;

Không có N+1, có LIMIT, primary key đã có index trên id. Nhưng query planner chọn... full table scan. Vì sao?

⏸ Pause & Predict — BẮT BUỘC dừng
Trước khi đọc tiếp, tự trả lời (viết ra giấy):
  1. 3 thông tin từ EXPLAIN ANALYZE output sẽ confirm đây là full scan problem?
  2. Tại sao chỉ add CREATE INDEX idx_cust ON orders(customer_id) KHÔNG đủ?
  3. Nếu tạo CREATE INDEX idx_cust_status_date ON orders(customer_id, status, created_at DESC) — tại sao column order matter? Reverse thành (created_at, status, customer_id) sẽ ra sao?
  4. Sau khi add composite index, latency drop xuống 200ms. Nhưng 1 tháng sau lại tăng 5s. Data volume không đổi. Chuyện gì xảy ra?
Nếu trả lời được hết 4 câu → bạn solid về SQL, đọc lướt file. Nếu <3 câu → đọc kỹ.

2. B-Tree Index CORE

★ CORE — Foundation cho mọi thứ về index. Hiểu B-Tree → tự predict được DB performance trước khi run query.

B-Tree index không phải sorted list. Đó là self-balancing tree giữ lookup O(log n) kể cả khi data grow:

🎯 Tại sao B-Tree shallow trên data lớn?

Mỗi node có thể chứa 100-1000 keys (fan-out cao). Với fan-out 100:

  • Depth 1 → 100 leaves → 100 keys
  • Depth 2 → 100² = 10K leaves → 1M keys
  • Depth 3 → 100³ = 1M leaves → 100M keys
  • Depth 4 → 100⁴ = 100M leaves → 10B keys

50M rows → tree chỉ depth 3-4. Lookup = 3-4 disk I/O (vs 50M nếu sequential scan). Đây là magic của B-Tree.

Index Scan vs Heap Fetch — 2-Phase Lookup

Leaf nodes của index chứa row pointer (TID), không phải full row. Sau khi tìm xong index, DB phải fetch heap page để get full row data:

Index Scan:
  Phase 1: Navigate B-Tree → tìm TIDs    (3-4 I/Os)
  Phase 2: Fetch heap pages              (1 I/O per matching row, có thể merge)

→ Total cost cho 20 rows: ~10-25 I/Os
→ Vs Seq Scan 50M rows: ~600K I/Os (mỗi page chứa ~80 rows)

→ Index scan nhanh hơn 60,000x cho query selective
🧠 Misconception — "Index = sorted list of values"
"Index là 1 file sorted theo column value, tìm bằng binary search"
B-Tree không phải sorted array. Sorted array có cost insert/delete O(n) (shift elements). B-Tree balance dynamic — insert/delete O(log n) với rebalancing. Đây là lý do DB có thể handle hàng triệu writes/second mà index vẫn fast.
🔬 Thử Phá Vỡ — Index trên column có 2 unique values

"Tôi tạo index trên column is_deleted (boolean true/false) để query WHERE is_deleted = false nhanh hơn." Đoán: index có giúp không?

Click để xem analysis

KHÔNG giúp, có khi còn hại. Reasoning:

  1. Low selectivity: 95% rows có is_deleted=false → DB scan 95% rows kể cả qua index → cost cao hơn full scan (extra index lookup + heap fetch).
  2. Planner thông minh: PostgreSQL optimizer dù có index sẽ chọn seq scan vì cost lower.
  3. Index nhưng vô dụng: tốn storage + write cost (mỗi insert/update phải maintain index) nhưng read không benefit.

Fix: dùng partial index: CREATE INDEX idx_active ON orders(created_at) WHERE is_deleted = false. Index chỉ chứa rows hợp lệ → smaller, faster. Hoặc combine vào composite index nếu thường filter together.

3. Composite Index + Leftmost Prefix Rule CORE

★ CORE — #1 source of SQL bug junior. Hiểu rule này = fix được 80% production query slow.

Composite index (customer_id, status, created_at) tạo B-Tree sort theo thứ tự:

customer_id=CUST_001:
  status=CANCELLED:
    2026-01-02, 2026-01-08, ...
  status=DELIVERED:
    2026-01-01, 2026-01-05, 2026-01-10, ...   ← sorted by created_at
  status=PROCESSING:
    2026-01-03, 2026-01-07, ...

customer_id=CUST_002:
  status=DELIVERED:
    ...

Leftmost Prefix Rule — Học thuộc

WHERE clauseDùng được index?Vì sao
WHERE customer_id = ?✅First column
WHERE customer_id = ? AND status = ?✅First 2 columns
WHERE customer_id = ? AND status = ? AND created_at > ?✅All 3 columns
WHERE customer_id = ? AND created_at > ?⚠️Skip status — index partial useful (chỉ first column)
WHERE status = ? AND customer_id = ?✅ (modern planner reorder)Planner tự reorder predicates — equivalent
WHERE status = ?❌Skip leading column customer_id
WHERE created_at > ?❌Skip 2 leading columns
🎯 Vì sao "leftmost prefix" matter?

Vì B-Tree sort hierarchically: trước sort theo column 1, trong mỗi group lại sort theo column 2, etc. Tương tự sort tên người: trước theo họ, trong cùng họ thì theo tên.

Tìm "tất cả người tên 'Nhân'" mà không biết họ → phải scan toàn bộ danh bạ. Không có shortcut. Đây là lý do skip leading column = không dùng được index.

4. Selectivity & Column Order CORE

★ CORE — Quyết định performance composite index. Sai column order = index gần như vô dụng.

Selectivity: column "selective" nếu WHERE values gần unique.

ColumnUnique valuesSelectivityRows/value avg
customer_id500K (50M / 100)0.01 (1%)~100 (rất selective)
status4 (PENDING/PROCESSING/DELIVERED/CANCELLED)0.25 (25%)~12.5M (low)
created_at~50M (millisecond precision)~1.0 (gần unique)~1 (cực selective)

Rule: trong composite index, đặt column SELECTIVE NHẤT đầu tiên (đối với equality filters). Lý do:

❌ (status, customer_id, created_at)
CREATE INDEX idx_bad ON orders(
  status,        -- 25M rows where DELIVERED
  customer_id,   -- narrow to ~50 rows
  created_at
);

-- Query: WHERE customer_id=X AND status=DELIVERED
-- Step 1: Index seek to status=DELIVERED → 25M entries
-- Step 2: Within those, filter customer_id → 50 rows
-- Step 3: Filter created_at, sort, limit
--
-- Index scans 25M index entries → SLOW
✅ (customer_id, status, created_at)
CREATE INDEX idx_good ON orders(
  customer_id,   -- narrow to ~100 rows
  status,        -- narrow to ~50 rows
  created_at     -- sort key
);

-- Query: WHERE customer_id=X AND status=DELIVERED
-- Step 1: Index seek to customer_id=X → 100 entries
-- Step 2: Within those, filter status → 50 entries
-- Step 3: Already sorted by created_at DESC, take LIMIT 20
--
-- Index scans 100 entries → FAST (10K× ít hơn)
🎯 Quy tắc thực dụng cho column order
  1. Equality filters trước, range/sort sau. WHERE a = ? AND b BETWEEN x AND y → index (a, b).
  2. Trong equality filters: high selectivity → low. Như example trên.
  3. Sort/ORDER BY column ở cuối. Index đã sorted theo column này → no extra sort needed.
  4. Pin column thường xuất hiện trong query lên trên. Index serve được nhiều query khác nhau cùng leading prefix.
🔬 Thử Phá Vỡ — Mixed equality + range filter

Query: WHERE customer_id = ? AND created_at > ? AND status = ?. Index nào tốt hơn?

  • A: (customer_id, created_at, status)
  • B: (customer_id, status, created_at)
Click để xem analysis

B tốt hơn. Lý do: range filter (created_at > ?) khi đặt trong middle composite index sẽ break leftmost prefix cho columns sau nó.

Cụ thể với A: (customer_id, created_at, status):

  • Index seek to customer_id = X → OK
  • Index seek to created_at > ? trong subset đó → OK, returns N rows
  • KHÔNG thể seek tiếp đến status = ? vì rows sau range filter scattered → phải scan tất cả N rows và filter status manually

Với B: (customer_id, status, created_at):

  • Index seek to customer_id = X → OK
  • Index seek to status = ? trong subset → OK
  • Index seek to created_at > ? trong subset đó → OK (range filter ở cuối — fine)

Rule: equality columns trước, range column ở cuối.

5. Covering Index SUPPLEMENTARY

SUPPLEMENTARY — Performance optimization cho hot queries. Học sau khi nắm core. Senior signal nếu áp dụng đúng.

Covering index = index chứa đủ data để answer query không cần heap fetch. Khi DB plan thấy mọi column SELECT đã có trong index → "Index Only Scan".

Tạo covering index — PostgreSQL syntax
CREATE INDEX idx_orders_covering ON orders(customer_id, status, created_at DESC)
INCLUDE (total_amount, shipping_address);
-- "INCLUDE" columns stored trong leaf, không tham gia sort
-- Không dùng cho WHERE filter, chỉ để avoid heap fetch

-- Query nay là Index Only Scan
SELECT id, customer_id, status, created_at, total_amount, shipping_address
FROM orders
WHERE customer_id = ? AND status = ? AND created_at > ?
ORDER BY created_at DESC LIMIT 20;

Benchmark expected (50M rows table):

Index strategyp95 latencyI/O per query
No index (seq scan)8000ms600K reads
Basic composite index + heap fetch180ms~50 reads
Covering index (Index Only Scan)15ms~5 reads
🎯 Trade-off của covering index
  • Storage: index size tăng 2-5× (vì store INCLUDE columns). Có thể vài chục GB extra.
  • Write cost: mỗi UPDATE column trong INCLUDE phải update index → write amplification.
  • Maintenance: REINDEX time tăng tỷ lệ với size.
  • Khi nào worth: hot query (≥1000 QPS), read-heavy (10:1 read/write ratio), latency critical.
  • Khi nào KHÔNG worth: rare query, write-heavy table, storage constrained.
🧠 Misconception — "SELECT * có thể dùng covering index"
"Tôi tạo covering index với INCLUDE (a, b, c), query SELECT * sẽ tự dùng Index Only Scan"
SELECT * yêu cầu tất cả columns. Nếu table có 20 columns mà INCLUDE chỉ có 3 → DB vẫn phải fetch heap cho 17 columns còn lại → Index Scan (not Index Only). Covering index defeated. Fix: explicit list columns trong SELECT, chỉ list những column index cover.

6. EXPLAIN ANALYZE — Tool Debug Chính CORE

★ CORE — Tool quan trọng nhất debug query. Mỗi senior phải đọc fluent, hiểu mọi keyword.

🎯 Edge Cases — EXPLAIN ANALYZE Production Reality
  • EXPLAIN (không ANALYZE) lying: estimate based on stats. Có thể wildly wrong nếu stats stale. Luôn dùng ANALYZE để see actual.
  • EXPLAIN ANALYZE chạy thật query: UPDATE/DELETE/INSERT sẽ modify data! Phải wrap trong transaction + ROLLBACK: BEGIN; EXPLAIN ANALYZE UPDATE ...; ROLLBACK;
  • Cold cache vs warm cache: lần chạy đầu chậm (disk reads), lần 2 nhanh (cached). Chạy 3-5 lần lấy median.
  • Plan thay đổi theo data size: dev DB 1K rows luôn dùng index. Prod 50M rows có thể chọn seq scan. Test trên realistic data size.
  • Statistics expire: PostgreSQL ANALYZE auto-run nhưng có thể lag. Force: ANALYZE orders; trước khi explain.
  • Parameter sniffing: query với hard-coded literal vs prepared statement có thể chọn plan khác. Test cả 2.
  • Parallel scan: query lớn có thể parallel → output có Workers Launched. Concurrency limit affect.
  • Generic plan vs custom plan: prepared statement chạy >5 lần có thể switch sang generic plan (worse). Set plan_cache_mode = force_custom_plan để test.

7. Query Planner Heuristics SUPPLEMENTARY

SUPPLEMENTARY — Hiểu vì sao planner chọn plan này. Helpful cho debug, không bắt buộc nắm chi tiết.

PostgreSQL planner dùng cost estimates, không phải actual time. Cost = combination của:

  • I/O cost: random_page_cost = 4.0, seq_page_cost = 1.0 (defaults). Random reads expensive hơn sequential 4×.
  • CPU cost: cpu_tuple_cost, cpu_index_tuple_cost, cpu_operator_cost.
  • Estimated rows: từ pg_statistics (column histograms, n_distinct).

Planner pick plan với lowest total cost. Decision tree:

Pseudo-decision logic
For each query:
  Generate candidate plans (Seq Scan, Index Scan, Index Only Scan, Bitmap Scan, ...)
  For each plan:
    Estimate cost based on stats + cost constants
  Pick lowest-cost plan

If filtering returns > ~10% of rows → Seq Scan often wins (avoid random I/O)
If filtering returns < 5% of rows → Index Scan wins
If query needs only indexed columns → Index Only Scan wins (no heap fetch)
⚠️ Khi planner sai — Stale statistics

PostgreSQL auto-vacuum chạy ANALYZE định kỳ, nhưng có delay. Sau big batch insert/delete:

  • Old stats: table có 1M rows, customer_id selectivity 0.001
  • Reality after batch: table có 100M rows, customer_id selectivity 0.0001
  • Planner estimate dựa old stats → wrong plan choice → query slow

Fix: ANALYZE table_name; sau batch operations. Hoặc tune autovacuum_analyze_threshold.

8. Slow Query Monitoring SUPPLEMENTARY

SUPPLEMENTARY — Production setup essential. Junior chưa cần, senior phải có.

P6Spy — Java JDBC Interceptor

# application.yml
spring:
  datasource:
    url: jdbc:p6spy:postgresql://localhost:5432/shopvn
    driver-class-name: com.p6spy.engine.spy.P6SpyDriver

# spy.properties (classpath)
moduleLogWithPrepare=true
excludecategories=info,debug
logfile=logs/p6spy-slow.log
filter=true
slowQueryThresholdMs=200
@Slf4j
@Component
public class SlowQueryListener implements JdbcEventListener {
    private static final long THRESHOLD_MS = 200;

    @Override
    public void onAfterAnyExecute(StatementInformation info, long timeNs, SQLException e) {
        long elapsedMs = TimeUnit.NANOSECONDS.toMillis(timeNs);
        if (elapsedMs > THRESHOLD_MS) {
            log.warn("SLOW_QUERY [{}ms] {}", elapsedMs, info.getSqlWithValues());
            slowQueryMetric.increment(elapsedMs);
            // Optionally: send to Sentry/Datadog
        }
    }
}

PostgreSQL pg_stat_statements

-- Enable extension (DBA action)
CREATE EXTENSION pg_stat_statements;

-- Top 20 slowest queries (mean time)
SELECT
    query,
    calls,
    total_exec_time,
    mean_exec_time,
    rows / calls AS avg_rows
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 20;

-- Top 20 queries by total time impact
SELECT query, total_exec_time, calls
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;

9. Index Maintenance ADVANCED

ADVANCED — DBA-level operation. Backend dev nên biết khi nào cần, không cần master detail.

Indexes fragmented theo thời gian (insert/update/delete). Periodic maintenance:

-- Refresh statistics — quick, an toàn
ANALYZE orders;

-- Rebuild fragmented index (PostgreSQL 12+ — CONCURRENTLY no lock)
REINDEX INDEX CONCURRENTLY idx_orders_cust_status_date;

-- Find unused indexes (candidate to drop)
SELECT schemaname, tablename, indexname, idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0
ORDER BY tablename;

-- Vacuum (reclaim space from deleted rows)
VACUUM ANALYZE orders;
📅 Maintenance schedule recommended
  • Daily 3AM: ANALYZE all hot tables (cheap, <1 min)
  • Weekly: Identify unused indexes, review with team
  • Monthly: REINDEX critical indexes (CONCURRENTLY) trong low-traffic window
  • Quarterly: Full VACUUM ANALYZE, review index strategy với DBA

10. Anti-Patterns CORE

#1 Over-indexing (every column has index)

-- ❌ 6 indexes trên 1 table
CREATE INDEX idx_orders_id ON orders(id);
CREATE INDEX idx_orders_customer ON orders(customer_id);
CREATE INDEX idx_orders_status ON orders(status);
CREATE INDEX idx_orders_created ON orders(created_at);
CREATE INDEX idx_orders_amount ON orders(total_amount);
CREATE INDEX idx_orders_address ON orders(shipping_address);
🔥 Consequence

Mỗi INSERT/UPDATE/DELETE phải maintain 6 indexes:

  • 1 index: INSERT 100K rows = 2s
  • 6 indexes: INSERT 100K rows = 6.2s (3× slower)

Fix: chỉ index columns thường filter/sort. Composite index cover được nhiều use case.

#2 Wrong column order (high selectivity first when should be low)

Đã cover ở section 4. Fix: analyze query patterns + put narrowing column first.

#3 Function on indexed column defeats index

❌ LOWER() defeats index
CREATE INDEX idx_email ON users(email);

-- Query (case-insensitive search)
SELECT * FROM users
WHERE LOWER(email) = 'foo@bar.com';
-- DB phải apply LOWER cho EVERY row → seq scan
✅ Functional index
CREATE INDEX idx_email_lower ON users(LOWER(email));

-- Query uses functional index
SELECT * FROM users
WHERE LOWER(email) = 'foo@bar.com';
-- Index seek O(log n)

Alternative: store data normalized (always lowercase) ở write time. Better long-term.

#4 SELECT * defeats covering index

Đã cover ở Misconception ở section 5. Fix: explicit columns trong SELECT.

#5 Missing index on foreign key (catastrophic for JOIN)

@Entity
public class OrderItem {
    @ManyToOne
    @JoinColumn(name = "order_id")  // ❌ Không có index trên order_id!
    private Order order;
}

// Query JOIN
SELECT oi.* FROM order_items oi
JOIN orders o ON oi.order_id = o.id
WHERE o.status = 'DELIVERED';
🔥 Hậu quả

Không có index trên order_items.order_id → JOIN thành nested loop:

  • For each order DELIVERED (1M rows):
  •   Full scan order_items (50M rows) tìm match
  • = 50 trillion comparisons → query mất 30+ phút thay vì 200ms

Fix: luôn index foreign keys.

#6 OR condition often defeats index

-- ❌ Có thể không dùng index hiệu quả
SELECT * FROM orders WHERE customer_id = ? OR email = ?

-- ✅ UNION 2 query, mỗi query dùng index riêng
SELECT * FROM orders WHERE customer_id = ?
UNION
SELECT * FROM orders WHERE email = ?

#7 NOT IN / != defeats index (most cases)

-- ❌ Index không help
SELECT * FROM orders WHERE status != 'CANCELLED'
-- Hầu hết rows match → seq scan tốt hơn

-- ✅ Nếu có vài statuses, list explicitly
SELECT * FROM orders WHERE status IN ('DELIVERED', 'PROCESSING', 'PENDING')
-- Index seek per status

#8 LIKE '%pattern' (leading wildcard) defeats index

-- ❌ Leading wildcard — full scan
SELECT * FROM products WHERE name LIKE '%iPhone%'

-- ✅ Trailing wildcard — index seek
SELECT * FROM products WHERE name LIKE 'iPhone%'

-- Nếu cần full-text — dùng full-text index
CREATE INDEX idx_name_gin ON products USING gin(name gin_trgm_ops);
SELECT * FROM products WHERE name ILIKE '%iPhone%';

12. Interview Framework

Tầng 1 — SurfaceMid gate
Q: Khác biệt giữa index scan và full table scan?
"Index scan dùng B-Tree index navigate đến rows matching → O(log n) cost. Full table scan đọc mỗi row → O(n). Index scan tốt cho query selective (small result set, < 5-10% rows). Nếu filter return > 10-15% rows, planner thường chọn seq scan vì avoiding random I/O overhead."
Q: Tại sao composite index (customer_id, status, created_at) không help query WHERE status = ? AND created_at > ??
"Leftmost prefix rule: composite index dùng từ trái sang phải. Query skip customer_id (column đầu) → index không seek được. DB phải scan toàn bộ index hoặc fallback seq scan. Fix: nếu query này thường xuyên, tạo index (status, created_at) riêng, hoặc đổi index thành (status, customer_id, created_at) nếu most queries dùng status làm anchor."
Tầng 2 — Deep DiveMid → Senior
Q: Design index strategy cho order search API. Queries filter by customer_id, status, date range. 50M rows.
"Index recommended: CREATE INDEX idx_orders_cust_status_date ON orders(customer_id, status, created_at DESC)

Reasoning column order:
• customer_id selectivity 0.001 (500K customers, 50M rows) → narrow to ~100 rows immediately, first
• status selectivity 0.25 (4 values) → narrow to ~25 rows, second
• created_at DESC → match ORDER BY direction, avoid extra sort step

Optionally add covering: INCLUDE (total_amount, shipping_address) nếu SELECT columns này thường xuyên → enable Index Only Scan, 10× faster.

Trade-offs:
• Write cost: mỗi INSERT/UPDATE tốn thêm 1 index maintenance → ~5-10% slower writes
• Storage: ~2GB index (400MB per column).
• Query thay đổi pattern (search by status only) → cần index riêng.

Monitoring: set up pg_stat_statements + alert nếu p95 latency > 500ms."
Q: Covering index là gì? Khi nào worth dùng?
"Covering index chứa đủ data trong index leaf để answer query mà không cần heap fetch. PostgreSQL syntax: INCLUDE (col1, col2) ở cuối CREATE INDEX. Plan output: 'Index Only Scan' thay vì 'Index Scan'.

Worth khi:
• Hot query (≥ 100 QPS), latency critical
• Read-heavy ratio (10:1)
• Query select few specific columns (không SELECT *)
• INCLUDE columns small (avoid bloat)

KHÔNG worth khi:
• Rare query
• Write-heavy table
• SELECT * (defeats purpose)
• Storage budget constrained (covering index 2-5× larger)

Production benchmark expected: query với covering 15ms vs same query without 180ms → 10× speedup."
Tầng 3 — ArchitectureSenior
Q: Orders table grows 10M → 500M rows. Index strategy hiện tại latency từ 100ms → 5s. Propose solution với indexing, partitioning, caching trade-offs.
"Multi-layer approach:

Layer 1 — Verify index strategy:
• Run EXPLAIN ANALYZE trên production query → confirm seq scan hay slow index scan
• Check stats: ANALYZE table → planner có lựa chọn dựa stats up-to-date
• Verify covering index cover most SELECT columns

Layer 2 — Range partitioning by created_at:
• PARTITION BY RANGE (created_at) — monthly partitions
• Mới queries (last 30 days) → only scan 1 partition (~3M rows thay 500M)
• Old data archive sang partition tách (parquet S3 hoặc cheaper storage)
• Trade-off: cross-partition JOIN slower, schema changes complex

Layer 3 — Read replica:
• Reporting queries (long-running analytics) → route to read replica
• Primary handle writes + critical real-time reads
• Trade-off: replication lag ~100ms-1s, eventual consistency

Layer 4 — Application caching:
• Caffeine L1 cache cho top 1% hot customers (LRU 10K entries)
• Redis L2 cache cho frequent queries (TTL 5 phút)
• Cache invalidation on order status change via event

Roadmap implementation:
• Week 1: optimize index (lowest risk, big impact)
• Week 2-3: implement caching (medium risk)
• Month 2-3: partition table (high risk, requires careful migration)
• Month 3-4: read replica setup (infrastructure work)

Expected final: p95 latency 50ms (hot), 500ms (cold), system handle 10× current load."
