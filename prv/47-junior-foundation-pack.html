📜 Lịch sử Java Collections — Tại sao có nhiều variants?
1995 — Java 1.0 release với chỉ Vector, Hashtable, Stack — tất cả synchronized. Performance kém vì lock overhead cho mọi operation.

1998 (Java 2 / JDK 1.2) — Joshua Bloch design Collections Framework chính thức. Introduce ArrayList, HashMap, LinkedList, TreeMap, HashSet — non-synchronized versions. Mục tiêu: speed + flexibility, để developer chọn khi nào cần thread-safety.

2004 (Java 5) — Doug Lea contribute java.util.concurrent: ConcurrentHashMap, BlockingQueue, lock-free data structures. Replace nhu cầu Collections.synchronizedMap (lock toàn map → contention cao).

2008 (Java 6) — Add ArrayDeque — efficient replacement cho Stack (legacy synchronized). Mọi textbook vẫn dạy Stack dù ArrayDeque faster.

2014 (Java 8) — HashMap collision optimization: chain dài > 8 convert sang Red-Black Tree. Worst case từ O(n) xuống O(log n) — fix hash collision DoS attack. Stream API + lambdas thay đổi cách code collections.

2016 (Java 9) — Factory methods List.of(), Map.of() — immutable collections concise. Replace Arrays.asList() (fixed-size, not truly immutable).

2021 (Java 16) — record classes — auto generate equals/hashCode/toString. Trước đó phải dùng Lombok hoặc 30-line boilerplate.

Insight quan trọng: variants nhiều vì không có "one collection fits all". Mỗi cái optimize cho 1 access pattern. Chọn đúng = 10-1000× perf khác biệt. Tool thay đổi (Java 17/21 records, sequenced collections), mental model về Big-O và data structure stable — học 1 lần dùng mãi.

1. Bài Toán Thực Tế — Junior Fail Coding Interview Tại Grab

Đề bài (Grab interview)

"Cho list of orders, group theo customerId rồi return customerId nào có tổng amount > 1tr VND."

❌ Code junior viết — chạy được, nhưng interviewer "kill" ngay trong 30 giây
public List<String> findTopCustomers(List<Order> orders) {
    List<String> result = new ArrayList<>();
    HashMap<String, Long> sums = new HashMap<>();

    for (int i = 0; i < orders.size(); i++) {
        Order o = orders.get(i);
        if (sums.containsKey(o.getCustomerId())) {
            sums.put(o.getCustomerId(), sums.get(o.getCustomerId()) + o.getAmount());
        } else {
            sums.put(o.getCustomerId(), o.getAmount());
        }
    }

    for (String customerId : sums.keySet()) {
        if (sums.get(customerId) > 1000000) {
            result.add(customerId);
        }
    }
    return result;
}

containsKey + get + put (3 lookups)"1 lookup là đủ với merge() hoặc getOrDefault(). Junior thiếu fluency với Map API."
for (String c : sums.keySet()) rồi sums.get(c)"Lại 2 lookups. entrySet() là chuẩn. Java basics chưa solid."
Trả ArrayList không sized"Đoán capacity được không? Nếu không, JVM sẽ resize nhiều lần."

Đây là gap "senior thấy ngay trong 30 giây code". Không cần biết DP/Backtracking, chỉ cần fluent với Java Collections đã khác biệt 1 level.

⏸ Pause & Predict — BẮT BUỘC dừng
Tự test 5 câu hỏi sau (viết ra giấy):
  1. ArrayList.get(i) là O(?) — LinkedList.get(i) là O(?). Khi nào nên dùng LinkedList?
  2. HashMap.put(k, v) average O(1) — worst case O(?). Java 8 fix xuống còn O(?).
  3. Integer a = 127; Integer b = 127; a == b → true hay false? Còn a = 128; b = 128?
  4. try { return 1; } finally { return 2; } — return giá trị nào?
  5. Recursion tree height — base case là gì? Vì sao base case quan trọng hơn recursive case?
Trả lời được hết 5 câu < 5 phút → solid foundation, đọc lướt. Nếu <3 câu → file này dành cho bạn.

🧠 Misconception — "LinkedList nhanh hơn ArrayList cho insert"
"LinkedList O(1) insert, ArrayList O(n) insert → LinkedList nhanh hơn"
2 sai sót: (1) LinkedList insert O(1) chỉ khi đã có Iterator ở vị trí đó. Insert by index = traverse + insert = O(n). (2) ArrayList có thể slower về complexity (System.arraycopy), nhưng cache locality giúp memcpy native cực nhanh — thực tế ArrayList insert middle nhanh hơn LinkedList với n < 10K vì L1/L2 cache prefetch contiguous memory hiệu quả. LinkedList nodes scattered → cache miss mỗi access. Benchmark: ArrayList add(0, x) n=1000 = 2μs; LinkedList tương tự = 8μs. Default: ArrayList wins 95% cases.
🔬 Thử Phá Vỡ — HashSet contains check trên custom class

Code này có bug. Tìm ra:

class Product { String id; Product(String id) { this.id = id; } }

Set<Product> seen = new HashSet<>();
seen.add(new Product("P001"));
boolean found = seen.contains(new Product("P001"));
System.out.println(found);  // ?
Click để xem analysis

Output: false. Bug: Product không override equals/hashCode. Default từ Object = reference equality.

new Product("P001") tạo instance mới → reference khác → contains() không match.

Worse: hash khác → HashSet nhìn vào bucket khác → contains trả false ngay cả khi instance same content tồn tại.

Fix: override equals/hashCode dùng id field, hoặc dùng Java 16 record:

record Product(String id) {}
// equals/hashCode auto-generated từ all fields

3. Big-O Reflex — Tính Nhẩm Trong Đầu CORE

★ CORE — Senior expect bạn tự nhẩm complexity. Không cần ai nhắc.

Senior interviewer mong candidate tự nhẩm complexity trong lúc code, không cần được nhắc. Đây không phải "lý thuyết khô" — đây là phản xạ.

Recursion 2 nhánh, depth log n (balanced tree)O(n)Tree traversal balanced
Recursion full split (Fibonacci naive)O(2ⁿ)Exponential — cần memo

Trick "code-as-you-think" complexity

Khi viết code, ngầm thêm comment // O(?) trên mỗi line:

public List<String> findDuplicates(List<String> words) {
    Set<String> seen = new HashSet<>();           // O(1)
    Set<String> duplicates = new HashSet<>();     // O(1)

    for (String word : words) {                   // n iterations
        if (!seen.add(word)) {                    // O(1) add
            duplicates.add(word);                 // O(1) add
        }
    }
    // Total: O(n) time, O(n) space
    return new ArrayList<>(duplicates);           // O(k) where k = #dupes
}

Tập 1 tuần thói quen này → tự nhẩm Big-O thành reflex.

Bẫy junior hay vướng

❌ Junior nghĩ O(n²), thực ra O(n³)
// Tìm duplicates trong 2D array
for (int i = 0; i < n; i++) {     // n

// list.contains là O(n) — junior thường quên

for (int i = 0; i < n; i++) {     // n

    if (lookup.contains(arr[i][j])){// O(1)
      // ...
    }
  }
}
// Total O(n²) — giảm 1 bậc complexity
// Pre-convert list to set = 1-time O(n) cost
❌ String concat = O(n²) hidden
String result = "";
for (String s : list) {
    result += s;
    // Mỗi += tạo new String,
    // copy hết n ký tự đã có
}
// String immutable — total O(n²)
// 1000 items × avg 10 chars = 10000 chars
// Total copies: ~10000² / 2 = 50M ops

    sb.append(s);

// per append → total O(n)
// Same task ~10000 ops
🎯 Edge Cases — Big-O Production Reality
  • HashMap worst case: hash collision rate cao → degrade O(n). Java 8 treeify (chain > 8) → O(log n). Custom hashCode bad → still degrade.
  • Amortized vs worst case: ArrayList add O(1) amortized, nhưng resize step là O(n). Mostly OK, gây latency spike khi capacity double.
  • Hidden allocations: list.toArray() O(n) — không chỉ access, mà copy. Tránh trong hot loop.
  • Stream lazy evaluation: stream().filter().map().collect() không phải 3 passes — chỉ 1 pass với fused operations. Counting passes wrong → Big-O wrong.
  • Sorting stable vs unstable: Arrays.sort(int[]) dùng dual-pivot quicksort O(n log n) avg, worst O(n²) với adversarial input. Arrays.sort(Object[]) dùng TimSort O(n log n) guarantee.
  • Memory locality matters: O(n) algorithm trên ArrayList có thể faster hơn O(log n) algorithm trên TreeMap với n < 1000 vì cache hits. Big-O không phải all.
  • JIT optimization: hot loops được JIT compile sau ~10K iterations. First runs slow, sau đó faster 10×. Benchmark cần warm-up.
  • GC pressure: object allocation trong hot loop → GC pause spike → latency unpredictable. Pre-allocate buffers.

4. equals / hashCode / Comparable — Contract CORE

★ CORE — HashMap/HashSet phụ thuộc hoàn toàn. Sai contract = bug khó tìm vì code "nhìn có vẻ đúng".
⚠️ 3 contracts phải nhớ
  1. equals() phải reflexive, symmetric, transitive, consistent
  2. equals() == true → hashCode() PHẢI equal (collision OK — không cần ngược lại)
  3. compareTo() consistent với equals (nên có, không bắt buộc)

Template chuẩn cho POJO làm Map key

public class CustomerId {
    private final String value;

    public CustomerId(String value) { this.value = Objects.requireNonNull(value); }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof CustomerId)) return false;
        CustomerId other = (CustomerId) o;
        return value.equals(other.value);
    }

    @Override
    public int hashCode() {
        return Objects.hash(value);
    }

    @Override
    public String toString() { return value; }
}

Java 16+ — Record giải quyết tự động

public record CustomerId(String value) { }
// equals, hashCode, toString tự generate
// ĐÂY là chuẩn từ Java 16 trở đi

Bẫy Integer cached — câu phỏng vấn kinh điển

Integer a = 127, b = 127;
System.out.println(a == b);  // true  ← cached!
Integer c = 128, d = 128;
System.out.println(c == d);  // false ← out of cache range [-128, 127]

// ✅ Luôn dùng .equals() cho boxed types
System.out.println(c.equals(d)); // true
🎯 Tại sao Java cache [-128, 127]?

JVM cache instance Integer trong khoảng này (như String.intern()) vì 90% code dùng số nhỏ. Tiết kiệm memory + GC. Range có thể tăng bằng JVM flag -XX:AutoBoxCacheMax=10000, nhưng đừng dựa vào — hãy luôn .equals().

🧠 Misconception — "Override equals đủ, không cần hashCode"
"Tôi override equals thôi, hashCode không quan trọng vì tôi không dùng HashMap"
Sai 2 lý do: (1) Spring/Hibernate/Jackson dùng HashMap/HashSet internally. ManyToMany relationship của JPA = dùng Set → hashCode call. Bug data corruption khó debug. (2) Tools như Collections.frequency, retainAll, deduplication code đều giả định contract đúng. Vi phạm = ngẫu nhiên fail.

Rule cứng: override equals MỘT trong 2 = override CẢ HAI. Hoặc dùng record (Java 16+) tự generate cả cặp. IDE warning nếu thiếu.

🎯 Edge Cases — equals/hashCode Production Reality
  • Mutable key trong HashMap: insert key K (hash=5), modify K (hash→8), get(K) → not found. Rule: HashMap key phải immutable hoặc never mutate after put.
  • BigDecimal.equals so cả scale: new BigDecimal("1.0").equals(new BigDecimal("1.00")) = false. Dùng compareTo() == 0 cho numeric equality.
  • Long vs Integer: new Long(5L).equals(new Integer(5)) = false vì khác type. Common bug khi mix types từ JSON parsing.
  • Array equals = reference: int[]{1,2}.equals(int[]{1,2}) = false. Dùng Arrays.equals().
  • HashCode = 0 cho all instances: legal nhưng degrade HashMap đến O(n). Java 8 treeify thành O(log n), still bad. IDE-generated thường OK.
  • Inheritance break symmetry: parent.equals(child) vs child.equals(parent) có thể return khác → vi phạm contract. Use getClass() check thay instanceof nếu có inheritance.
  • JPA entity proxy: Hibernate lazy proxy có thể equals false với eager-loaded entity dù same DB row. Override equals dùng business key, không reference.
  • Volatile state trong equals: include timestamp/last-modified trong equals → object equality changes over time → HashMap behavior bizarre. Stick to immutable identifying fields.

5. Generics — PECS Rule SUPPLEMENTARY

SUPPLEMENTARY — Đọc API Java/Spring dễ hơn. Senior signal nếu hiểu wildcards.

Khi đọc Spring/Java standard library, bạn sẽ gặp hàng trăm signature như:

public static <T> void copy(List<? super T> dest, List<? extends T> src);
public static <T extends Comparable<? super T>> void sort(List<T> list);

Không hiểu PECS = không đọc được Java standard library. PECS = Producer Extends, Consumer Super.

🧠 Mental model PECS

Producer = source bạn đọc dữ liệu ra → dùng extends
Consumer = sink bạn ghi dữ liệu vào → dùng super

Producer — đọc từ collection, dùng extends
public double sumOfList(List<? extends Number> list) {
    double sum = 0;
    for (Number n : list) sum += n.doubleValue();  // ✅ Đọc OK
    // list.add(...);  ❌ Compile error
    return sum;
}

sumOfList(List.of(1, 2, 3));            // List
sumOfList(List.of(1.5, 2.5));           // List
sumOfList(List.of(new BigDecimal("1"))); // List
Consumer — ghi vào collection, dùng super
public void addIntegers(List<? super Integer> list) {
    list.add(1);  // ✅ OK
    list.add(2);
    // Integer x = list.get(0);  ❌ chỉ get được Object
}

List<Object> objs = new ArrayList<>();
addIntegers(objs);  // ✅

6. Recursion — Leap of Faith ADVANCED

ADVANCED — Nền LeetCode tree/graph. Lỗ hổng lớn nhất của candidate VN khi grind LeetCode.
✨ Mental model đúng — "Leap of Faith"

Khi viết hàm recursion, GIẢ ĐỊNH recursive call đã đúng cho input nhỏ hơn. Bạn chỉ cần đảm bảo:

  1. Base case (input nhỏ nhất) trả đúng
  2. Bước recursive: kết hợp kết quả con đúng cho input lớn hơn

KHÔNG bao giờ debug bằng cách trace từng stack frame.

Ví dụ — count nodes trong binary tree (comment tree feed của social platform)

public int countComments(CommentNode root) {
    // Base case: cây rỗng có 0 node
    if (root == null) return 0;

    // LEAP OF FAITH:
    //   countComments(root.left)  → giả định trả đúng số comments ở subtree trái
    //   countComments(root.right) → giả định trả đúng số comments ở subtree phải
    // Câu hỏi duy nhất: kết hợp 2 kết quả + node hiện tại sao cho đúng?

    return 1 + countComments(root.left) + countComments(root.right);
}

Tự kiểm tra recursion đúng — 3 câu hỏi

  1. Base case có terminate không? (đảm bảo không infinite loop)
  2. Mỗi recursive call có làm input "nhỏ hơn"? (giảm về base case)
  3. Kết hợp con đã đúng cho cha? (đúng nếu giả định con đúng)

Nếu cả 3 đều YES → recursion đúng. Không cần trace.

3 pattern recursion phổ biến trong LeetCode

// Pattern A: "Mỗi node tự xử lý + delegate cho con"
int treePattern(TreeNode root) {
    if (root == null) return /* base */;
    int leftResult = treePattern(root.left);
    int rightResult = treePattern(root.right);
    return /* combine: leftResult + rightResult + root.val */;
}

// Pattern B: "Backtracking — chọn rồi unchoose"
void backtrack(State current, List<Result> results) {
    if (isComplete(current)) {
        results.add(snapshot(current));
        return;
    }
    for (Choice c : possibleChoices(current)) {
        current.choose(c);
        backtrack(current, results);
        current.unchoose(c); // ← Bước này hay quên
    }
}

    return result;
}

7. Exception Hygiene — Fail Loud, Fail Fast CORE

★ CORE — 80% production debug đến từ exception handling sai.

4 anti-patterns junior phải tránh

// ❌ 1. Swallow exception — bug bị nuốt mất
try { riskyCall(); } catch (Exception e) { /* empty */ }

// ❌ 2. Catch quá rộng rồi log thôi, không rethrow
try { riskyCall(); }
catch (Exception e) {
    log.error("Error", e);
    // không rethrow → caller nghĩ thành công
}

// ❌ 3. Bắt rồi rethrow mất stack trace gốc
try { riskyCall(); }
catch (IOException e) {
    throw new RuntimeException(e.getMessage()); // ← MẤT cause!
}

// ✅ Đúng — giữ cause:
catch (IOException e) {
    throw new RuntimeException("Failed to process: " + path, e);
}

// ❌ 4. throws Exception — quá lười
public void process() throws Exception { }

// ✅ Đúng:
public void process() throws ProcessingException, ValidationException { }
📖 Quy tắc 10 năm exp

"Nếu không biết nên handle exception thế nào, let it bubble up. Để layer cao nhất (controller / global handler) quyết định. Đừng catch chỉ để log."

8. Anti-Patterns Junior Phổ Biến

#1 Premature optimization

// ❌ Pre-size 10M khi data chỉ 100
HashMap<String, Object> map = new HashMap<>(10_000_000); // Thừa memory

// ✅ Default, optimize khi đã biết bottleneck
HashMap<String, Object> map = new HashMap<>();

#2 Modify collection while iterating

❌ ConcurrentModificationException
List<Integer> list = new ArrayList<>(
    List.of(1, 2, 3, 4, 5));
for (Integer n : list) {
    if (n % 2 == 0) list.remove(n); // CME!
}
✅ removeIf / Iterator.remove
// Java 8+ idiomatic
list.removeIf(n -> n % 2 == 0);

// Hoặc iterator
Iterator<Integer> it = list.iterator();
while (it.hasNext()) {
    if (it.next() % 2 == 0) it.remove();
}

#3 Comparing wrappers với ==

// ❌ false vì khác reference (Long không cache)
Long a = 200L, b = 200L;
if (a == b) { ... }  // false!

// ✅ Equals hoặc unbox
if (a.equals(b)) { ... }
if (a.longValue() == b.longValue()) { ... }

#4 Stream khi loop đơn giản hơn

Stream tốt cho transform 1-2 step, có collector ngắn. Chain > 4 step + cần debug → for-loop dễ đọc hơn. Stream không phải lúc nào cũng "senior".

#5 Mutable state trong "stateless" service

// ❌ Field shared across requests
@Service
public class OrderProcessor {
    private List<Order> currentBatch = new ArrayList<>(); // SHARED!
    public void process(List<Order> input) {
        currentBatch.addAll(input); // race condition
    }
}

// ✅ Local variable
@Service
public class OrderProcessor {
    public void process(List<Order> input) {
        List<Order> currentBatch = new ArrayList<>(input);
    }
}

10. Interview Framework

Tầng 1 — SurfaceJunior gate
Q: ArrayList và LinkedList khác gì?
"ArrayList lưu element trong array contiguous, get(i) là O(1), insert/remove ở giữa O(n) vì shift elements. LinkedList dùng doubly-linked node, get(i) là O(n) vì traverse, add/remove ở 2 đầu O(1).

Thực tế: gần như luôn dùng ArrayList vì cache locality tốt hơn — CPU cache prefetch contiguous memory hiệu quả. LinkedList chỉ đáng dùng khi đã có Iterator ở vị trí cần insert/remove. Stack/queue dùng ArrayDeque, không LinkedList."
Q: HashMap hoạt động thế nào?
"Array of buckets, default 16. Mỗi key được hash, modulo capacity ra bucket index. Collision (2 keys cùng bucket): Java 7 linked list, Java 8+ tự convert sang Red-Black Tree khi chain > 8 → worst case O(log n).

Số element > 75% capacity (load factor) → resize gấp đôi + rehash toàn bộ — O(n) đắt. Nếu biết trước size, new HashMap<>((int)(expectedSize / 0.75f) + 1) tránh resize."
Tầng 2 — Deep DiveMid
Q: Recursion vs Iteration — khi nào chọn cái nào?
"Recursion phù hợp khi bài toán có cấu trúc đệ quy tự nhiên — tree/graph traversal, divide & conquer, backtracking. Code ngắn, sạch.

Iteration tốt hơn khi: depth lớn (Java default stack ~256KB-1MB, depth 10K là tràn), hot loop performance critical (function call overhead).

Mọi recursion convert sang iteration bằng explicit stack — DFS với Deque<Node>. JVM không có tail call optimization (Scala có) → deep recursion bắt buộc iteration."
Q: Stream performance — đáng dùng?
"Trade-off: declarative, dễ đọc, parallel dễ. Nhưng overhead từ lambda + boxing — hot loop trên primitive (int, long), IntStream cần thiết, không Stream<Integer> (autoboxing).

Code thường: gap không đáng kể (~10-30% chậm hơn loop). Hot path benchmark: loop nhanh hơn rõ. Quy tắc: prefer readability, profile trước optimize, chỉ parallelStream khi data lớn (>10K) và operation CPU-bound."
Tầng 3 — ArchitectureSenior bridge
Q: Khi nào HashMap không đủ, cần data structure khác?
"5 tình huống chính:

1. Cần ordered iteration: LinkedHashMap (insertion/access order) hoặc TreeMap (sorted by key).
2. Cần range query: 'all keys giữa A và B' → TreeMap.subMap(). HashMap không hỗ trợ.
3. Cần thread-safe: ConcurrentHashMap — CAS + synchronized per bucket, throughput cao hơn Collections.synchronizedMap 10-50×.
4. Memory critical primitive keys: HashMap<Integer, Integer> autobox tốn memory; Eclipse Collections / fastutil IntIntHashMap saves ~50%.
5. Approximate membership scale lớn: Bloom Filter (Guava) — false positive nhưng O(1) memory cố định, dùng cho '100M users seen?'.

Senior phải biết cả 5, không chỉ HashMap."
