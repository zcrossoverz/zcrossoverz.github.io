#17 — Microservices Patterns

Microservices không chỉ là "tách monolith thành nhiều service". Đó là distributed systems engineering — phải hiểu service discovery, gateway, resilience patterns, cascading failures. File này dùng ví dụ ride-sharing app (kiểu Grab/Uber) để demo các pattern.

Vì Sao Patterns Quan Trọng Trong Microservices?

Monolith có 1 process, 1 DB, 1 deployment. Mọi call là method call trong cùng JVM — không bao giờ fail vì network. Khi tách thành microservices:

Khía cạnhMonolithMicroservices
Inter-component callMethod call (ns)HTTP/gRPC (ms-s, có thể fail)
TransactionDB transactionDistributed — không có 2PC trong thực tế
Deploy1 artifactN artifacts, phải coordinate version
DebuggingStack trace 1 file logTrace cross 5 services, 5 log streams
Service locationCompile-timeRuntime — service IP có thể đổi
Failure modeApp crash hếtPartial failure — 1 service down, others survive (hopefully)

Mỗi khía cạnh trên cần 1 pattern để giải quyết. File này dạy 6 patterns cốt lõi + deep dive về cascading failures (sai lầm chết người nhất khi mới chuyển sang microservices).

1. Bài Toán Thực Tế — Monolith-to-Microservices Migration Của Ride-Sharing App

Year 1: Monolith

RideZ là startup ride-sharing. Bắt đầu với 1 Spring Boot monolith 500K LOC chứa tất cả features (trips, drivers, payment, notification, pricing, location tracking). Deploy mất 2 giờ (full test suite). Outage 1 module ảnh hưởng cả app.

Year 2: Tách thành 15 microservices

  • TripService, DriverService, RiderService
  • PaymentService, PricingService (surge pricing)
  • NotificationService (push to driver/rider)
  • LocationService (geo tracking realtime)
  • RatingService, FraudService, PromoService, ...

Month 6: Chaos

🔥 Triệu chứng "migration gone wrong"
  • Engineers hardcode IP: http://10.0.0.15:8080/drivers — server restart đổi IP → broken
  • Auth không thống nhất: TripService validate JWT với Secret A, DriverService với Secret B → maintenance nightmare
  • Service A gọi B sync, B gọi C, C gọi D — chain 4 timeouts = 4× latency
  • No request tracing: user complain "ride request fail", engineer không biết service nào fail
  • Service A crash → cascade B/C/D timeout (cùng waiting A)
  • Mỗi service own DB: tạo trip phải atomically update TripService DB + LocationService DB + DriverService DB → không có distributed TX → data inconsistency

5 câu hỏi cần trả lời

  1. Làm sao tìm DriverService nếu IP đổi (scaling, restart)?
  2. Làm sao tránh cascading failures khi A → B → C → D?
  3. Nếu TripService và PaymentService phải cùng success/fail atomic, làm sao consistent không dùng 2PC?
  4. Làm sao biết service nào tốn 5s trong 1 API call 10s?
  5. Granularity đúng: 15 services (nanoservices) hay 4 bounded contexts?
🗺️ Nếu chỉ có X giờ thì học gì
1.5 giờ (CORE ★): Saga (compensating transaction thay 2PC), Circuit Breaker (fail-fast chống cascading), API Gateway. Đây là 3 thứ interview microservices hỏi gần như chắc chắn.
+1 giờ (CORE ★): Cascading failure timeline + cách chặn (3), Service Discovery, anti-pattern Distributed Monolith & Shared Database.
+1.5 giờ (SUPPLEMENTARY): CQRS, Bulkhead, Retry với backoff+jitter, Sidecar/Service Mesh.
Khi cần (ADVANCED): Orchestration (Camunda BPMN) vs Choreography sâu, distributed tracing setup, API versioning strategy.
⏸ Pause & Predict
Nhìn câu hỏi #3: TripService và PaymentService phải cùng success/fail. Trả lời ra giấy:
  1. Vì sao KHÔNG dùng 2PC (two-phase commit) ở đây? Nêu 2 lý do cụ thể (availability, coupling).
  2. Saga thay thế bằng cái gì? Nếu PaymentService fail sau khi TripService đã tạo trip, "rollback" thực ra là làm gì?
  3. Service A gọi B sync, B gọi C, C gọi D. A crash. Vì sao B, C, D cũng chết theo, và pattern nào chặn được? (gợi ý: cái gì cạn trước — thread hay memory?)
Đáp án: 2PC cần mọi node sống đồng thời + coordinator = giảm availability và tight coupling (mục Saga); Saga dùng compensating action (hoàn tiền/hủy trip) chứ không có rollback thật; cascading do thread pool của B cạn vì giữ kết nối chờ A (mục 3 + Circuit Breaker + Bulkhead).

2. 6 Patterns Cốt Lõi

Service Discovery — Tìm Service Dynamic

Vấn đề: trong K8s, pod restart → IP mới. Scaling out → IP mới. Hardcoded IP = broken.

🎯 Client-side vs Server-side discovery
  • Client-side (Netflix Ribbon, Eureka client): client fetch instance list từ registry, tự load balance. Nhanh (no extra hop), nhưng client code phức tạp + mỗi ngôn ngữ phải có lib riêng.
  • Server-side (K8s Service, AWS ELB): client gọi 1 stable virtual IP, LB query registry route request. Đơn giản với client (chỉ gọi DNS name), nhưng thêm 1 network hop.
  • Modern preference: K8s + DNS. http://driver-service:8080 tự resolve qua kube-dns → load balance to pod IPs. Service code không cần lib gì.

API Gateway — Single Entry Point

Vấn đề: 15 microservices, mỗi service tự validate JWT, tự rate limit, tự version API. Client phải biết IP/URL của 15 services. Mobile app build cứng URL pattern → service đổi URL = rebuild app.

// Spring Cloud Gateway — config routes
@Configuration
public class GatewayConfig {
    @Bean
    public RouteLocator routes(RouteLocatorBuilder builder) {
        return builder.routes()
            .route("trip_route", r -> r
                .path("/api/trips/**")
                .filters(f -> f
                    .rewritePath("/api/trips/(?<path>.*)", "/${path}")
                    .addRequestHeader("X-Request-ID", UUID.randomUUID().toString())
                    .filter(new JwtAuthFilter(jwtDecoder))
                    .requestRateLimiter(c -> c
                        .setKeyResolver(ex -> Mono.just(ex.getRequest().getHeaders().getFirst("Authorization")))
                        .setRateLimiter(new RedisRateLimiter(100, 200)))  // 100 req/sec sustained, 200 burst
                )
                .uri("lb://trip-service"))  // lb:// = load-balanced via service discovery
            .route("driver_route", r -> r
                .path("/api/drivers/**")
                .filters(f -> f.rewritePath("/api/drivers/(?<path>.*)", "/${path}"))
                .uri("lb://driver-service"))
            .build();
    }
}
🎯 Vì sao Gateway centralize cross-cutting concerns?
  • DRY: implement auth/rate-limit 1 lần, 15 services free. Không có gateway = mỗi service copy code auth.
  • Security at edge: invalid request bị reject ngay tại gateway, không reach internal services → giảm attack surface.
  • Client simplicity: mobile app chỉ biết api.ridez.vn, không care service nào ở đâu. Backend refactor không cần update app.
  • Observability: 1 chỗ log all traffic → tổng số request, top API, error rate dễ track.
⚠️ Gateway anti-pattern: Smart Gateway

Đừng đưa business logic vào gateway (validate phone format theo VN regulation, calculate price, ...). Gateway chỉ làm cross-cutting (auth, rate, routing, tracing). Business logic ở service. Gateway smart = bottleneck + single point of failure.

Sidecar Pattern / Service Mesh

Vấn đề: Cross-cutting concerns ở service-to-service communication (retry, mTLS, circuit breaker, tracing) — phải implement trong mỗi service code. Java có lib, Go có lib, Python có lib — duplicate effort, không consistent.

Giải pháp Sidecar: deploy proxy (Envoy) cùng pod với mỗi service. Service chỉ gửi HTTP plain, proxy xử lý mọi cross-cutting concerns.

🎯 Vì sao Sidecar quan trọng khi scale?
  • Polyglot: team Java, Go, Python đều dùng cùng resilience policy. Không cần implement retry/circuit breaker N lần cho N ngôn ngữ.
  • Consistent policy: thay đổi retry strategy = update mesh config, không touch service code. Apply uniformly.
  • Zero-trust networking: mTLS bật globally — encrypt service-to-service traffic mà service không biết.
  • Trade-off: thêm 1 latency hop (~1-2ms), tốn RAM (mỗi pod thêm 100MB cho sidecar). Chỉ đáng khi scale > 30 services.

CQRS — Tách Read và Write Model

Vấn đề: TripService write data normalized (trip + trip_lines + driver_id + rider_id). Reporting query "tổng trips/ngày theo city" cần JOIN 5 tables → query 3 giây.

// COMMAND side
@Service
public class CreateTripCommandHandler {
    @Transactional
    public void handle(CreateTripCommand cmd) {
        Trip trip = Trip.create(cmd);
        tripRepo.save(trip);  // Normalized DB
        eventBus.publish(new TripCreatedEvent(trip.id(), trip.riderId(), trip.city(), trip.amount()));
    }
}

// QUERY side — separate service, separate DB
@Service
public class TripReadProjection {
    private final TripReadRepository readRepo;  // ElasticSearch / denormalized PostgreSQL

    @KafkaListener(topics = "trips")
    public void on(TripCreatedEvent event) {
        // Update read model — denormalized, ready for fast query
        readRepo.save(new TripReadView(
            event.tripId(),
            event.city(),
            event.amount(),
            event.timestamp(),
            // Pre-aggregate hourly bucket cho dashboard query
            event.timestamp().truncatedTo(ChronoUnit.HOURS)
        ));
    }
}

@RestController
public class ReportingController {
    private final TripReadRepository readRepo;

    @GetMapping("/api/reports/trips-by-city")
    public Map<String, Long> tripsByCity(@RequestParam LocalDate date) {
        // Query read model — single index scan, no JOIN
        return readRepo.countByDateGroupByCity(date);  // 50ms vs 3s với write DB
    }
}
🎯 Vì sao CQRS có trade-off worth it?
  • Read và write có pattern khác nhau: write optimal cho integrity (normalize, FK), read optimal cho speed (denormalize, pre-aggregate). Một schema không thể tối ưu cả hai.
  • Independent scaling: read 1000× write trong analytics workload. Scale read DB lên 5 replica, write DB giữ 1 primary.
  • Independent technology: write PostgreSQL (ACID), read ElasticSearch (full-text search) hoặc ClickHouse (analytics).
  • Cost: eventual consistency (read model lag ms-s). Không phù hợp use case cần read-your-write (user vừa update, refresh phải thấy ngay).

Saga Pattern — Distributed Transaction Without 2PC

Vấn đề: Booking trip = phải:

  1. Reserve driver (DriverService)
  2. Charge rider (PaymentService)
  3. Create trip (TripService)

3 services, 3 DB khác nhau. Không có DB transaction span 3 services. Nếu step 2 fail sau step 1, driver vẫn reserved nhưng không charge → driver waste time.

Choice A — Orchestration (Central Coordinator)

@Service
public class BookTripOrchestrator {
    public TripId book(BookTripCommand cmd) {
        DriverId driver = null;
        PaymentId payment = null;
        try {
            // Step 1
            driver = driverService.reserve(cmd.pickup(), cmd.vehicleType());

            // Step 2
            payment = paymentService.charge(cmd.riderId(), cmd.estimatedFare());

            // Step 3
            return tripService.create(cmd, driver, payment);
        } catch (Exception e) {
            // Compensate in reverse order
            if (payment != null) paymentService.refund(payment);
            if (driver != null) driverService.release(driver);
            throw new BookingFailedException(e);
        }
    }
}

Choice B — Choreography (Event-Driven)

// TripService
@Transactional
public void requestTrip(BookTripCommand cmd) {
    Trip trip = Trip.requested(cmd);
    tripRepo.save(trip);
    eventBus.publish(new TripRequestedEvent(trip.id(), cmd.pickup(), cmd.vehicleType()));
}

// DriverService listens
@KafkaListener(topics = "trip-events")
public void on(TripRequestedEvent e) {
    try {
        DriverId driver = driverService.reserve(e.pickup(), e.vehicleType());
        eventBus.publish(new DriverReservedEvent(e.tripId(), driver));
    } catch (NoDriverAvailableException ex) {
        eventBus.publish(new DriverReservationFailedEvent(e.tripId()));
    }
}

// PaymentService listens
@KafkaListener(topics = "trip-events")
public void on(DriverReservedEvent e) {
    try {
        PaymentId p = paymentService.charge(e.riderId(), e.estimatedFare());
        eventBus.publish(new PaymentChargedEvent(e.tripId(), p));
    } catch (PaymentFailedException ex) {
        eventBus.publish(new PaymentFailedEvent(e.tripId()));
    }
}

// TripService listens cho confirm hoặc cancel
@KafkaListener(topics = "trip-events")
public void on(PaymentChargedEvent e) { tripRepo.confirm(e.tripId()); }

@KafkaListener(topics = "trip-events")
public void on(PaymentFailedEvent e) {
    tripRepo.cancel(e.tripId());
    eventBus.publish(new TripCancelledEvent(e.tripId()));  // Trigger driver release
}

Flow visibility✅ Rõ ràng (1 nơi)❌ Phân tán — đọc 5 services mới hiểu
Add new stepSửa orchestratorThêm 1 listener mới
DebuggingDễ — single logKhó — phải trace events across services
CouplingOrchestrator coupling với mọi serviceLoose — services chỉ biết events
ScaleOrchestrator có thể là bottleneckTự nhiên scale (async)
Workflow complexTốt — visual workflow (Camunda BPMN)Khó — phải reason về interleaving
🎯 Khi nào chọn cái nào?
  • Orchestration: workflow phức tạp 5+ steps, business rule "thực sự" có order rõ ràng (booking trip, order fulfillment nhiều bước), audit trail quan trọng (compliance). Camunda BPMN giúp visual hóa.
  • Choreography: workflow đơn giản 2-3 steps, services autonomous, low coupling priority. Phù hợp event-driven architecture.
  • Hybrid: lớn thì hybrid OK — choreography cho main flow, orchestration cho complex sub-flow.

L1 — Biết tên

Biết "Saga thay 2PC cho distributed transaction". Symptom hiểu giả: vẽ được sơ đồ nhưng không nói được điều gì xảy ra khi step 3/5 fail, hoặc tin Saga cho ACID.

L2 — Explain được

Giải thích mỗi local transaction publish event/command kế tiếp; khi một step fail thì chạy compensating transaction ngược chiều; Saga cho eventual consistency + atomicity nghiệp vụ, KHÔNG cho isolation → phải xử lý trạng thái trung gian (đã trừ kho nhưng chưa thu tiền).

L3 — Apply được

Chọn orchestration vs choreography theo độ phức tạp; thiết kế compensation idempotent (chạy 2 lần không sai); xử lý "compensation cũng fail" bằng retry + dead-letter + alert; chừa semantic lock cho dữ liệu trung gian. Chứng minh được vì sao không cần 2PC.

🛠️ Lab bắt buộc — build một Saga nhỏ (orchestration)

Bài toán: đặt vé trip gồm 3 bước qua 3 service giả lập (in-memory, không cần hạ tầng thật): reserveSeat() → chargePayment() → confirmTrip(). Mỗi service có hàm forward và hàm compensate (releaseSeat, refund, cancelTrip).

  1. Viết một SagaOrchestrator chạy 3 bước tuần tự, lưu lại các bước đã hoàn thành.
  2. Ép chargePayment() ném exception. Orchestrator phải gọi compensate cho các bước ĐÃ làm theo thứ tự ngược (chỉ releaseSeat), KHÔNG đụng bước chưa chạy.
  3. Làm refund() idempotent: gọi 2 lần chỉ hoàn tiền 1 lần. Test bằng cách gọi compensate lặp.
  4. Thêm trường hợp refund() fail lần đầu, thành công lần retry → log + không mất tiền.

Tiêu chí đạt: giải thích được "Saga không có rollback, chỉ có compensation", và vì sao mọi compensation phải idempotent.

3. Cascading Failure — Sai Lầm Chết Người Nhất

Đây là pattern bạn phải hiểu rõ nếu làm microservices. Hiểu sai = production outage tốn triệu đô.

Timeline 1 cascading failure thực tế

T+0s NotificationService DB slow (missing index). Response time 50ms → 3000ms

T+5s TripService call NotificationService → thread blocked 3s
10 concurrent requests → 10 threads blocked

T+15s WARN TripService thread pool exhausted (200/200 blocked on Notification)
New trip requests queued → timeout

T+30s ERROR API Gateway timeout waiting TripService (5s)
Returns 504 Gateway Timeout to mobile app

T+45s ERROR RETRY STORM: 10,000 users × 3 retries = 30,000 requests
Tệ hơn nhiều so với normal traffic

T+60s ERROR DriverService start failing (overwhelmed)
PricingService queue full → drop events

T+120s ERROR COMPLETE OUTAGE. 5/5 services down. Root cause: 1 missing index in NotificationService DB.
🔥 Mechanism

1 slow service → block caller threads → caller's thread pool exhausted → caller fails → caller's caller fails → retry storm khuếch đại traffic → toàn bộ hệ thống chết. Đây là cascading failure.

Circuit Breaker — Fail-Fast Khi Service Sick

// Resilience4j configuration
@Configuration
public class CircuitBreakerConfig {
    @Bean
    public CircuitBreakerRegistry registry() {
        return CircuitBreakerRegistry.of(
            CircuitBreakerConfig.custom()
                // Khi nào CLOSED → OPEN
                .failureRateThreshold(50)              // 50% requests fail → open
                .slowCallRateThreshold(80)             // 80% slow → open
                .slowCallDurationThreshold(Duration.ofSeconds(2))
                .slidingWindowType(COUNT_BASED)
                .slidingWindowSize(10)                 // Đánh giá trên 10 requests gần nhất

                // OPEN behavior
                .waitDurationInOpenState(Duration.ofSeconds(30))  // 30s → HALF_OPEN

                // HALF_OPEN behavior
                .permittedNumberOfCallsInHalfOpenState(3)
                // 3 calls success → CLOSED, fail → OPEN lại
                .build()
        );
    }
}

@Service
@Slf4j
public class DriverServiceClient {
    private final CircuitBreaker circuitBreaker;
    private final RestTemplate restTemplate;

    public DriverInfo findNearbyDriver(Location pickup) {
        return circuitBreaker.executeSupplier(() ->
            restTemplate.getForObject(
                "http://driver-service/api/drivers/nearby?lat={lat}&lng={lng}",
                DriverInfo.class, pickup.lat(), pickup.lng())
        );
    }

    // Fallback khi circuit OPEN
    public DriverInfo findNearbyDriverFallback(Location pickup, Throwable t) {
        log.warn("DriverService unavailable, returning cached/default. Error: {}", t.getMessage());
        // Trả về estimated driver từ cache, hoặc default response
        return DriverInfo.unavailable("Service temporarily unavailable, please retry");
    }
}
🎯 Vì sao Circuit Breaker cứu hệ thống?
  • Fail-fast: khi DriverService down, caller không wait 30s timeout → trả về ngay (1ms) → caller threads không bị block.
  • Cho service breathing room: nếu service đang struggle (high latency), block traffic 30s giúp service recover (clear queue, GC).
  • Graceful degradation: fallback trả response "limited service" thay vì error. App vẫn usable (limited mode).

Bulkhead — Isolate Thread Pools

❌ Shared thread pool
// Default Spring có 1 pool 200 threads
// Cả call Driver + Payment dùng chung
//
// DriverService slow → 200 threads
// đều bị block → Payment cũng không call được
✅ Isolated pools
@Bean
public ThreadPoolBulkhead driverBulkhead() {
  return ThreadPoolBulkhead.of("driver",
    ThreadPoolBulkheadConfig.custom()
      .maxThreadPoolSize(20)  // Riêng 20 threads
      .coreThreadPoolSize(10)
      .queueCapacity(30)
      .build());
}

@Bean
public ThreadPoolBulkhead paymentBulkhead() {
  return ThreadPoolBulkhead.of("payment", ...);
  // Independent pool — Driver slow không
  // ảnh hưởng Payment threads
}
🎯 Vì sao tên "Bulkhead"?

Lấy từ tàu thủy: thân tàu chia thành nhiều khoang kín (bulkheads). Khoang này ngập nước, khoang khác vẫn nổi. Tương tự: 1 service slow chỉ "ngập" pool của nó, không sink toàn bộ app.

Retry với Exponential Backoff + Jitter

// ❌ SAI — instant retry, no limit
@Retryable(maxAttempts = 5)
public void callDriverService() { /* ... */ }
// 10,000 clients × 5 retries = 50,000 requests trong vài giây → RETRY STORM

// ✅ ĐÚNG — exponential backoff + jitter + circuit breaker
@Retryable(
    maxAttempts = 3,
    backoff = @Backoff(
        delay = 1000,         // First retry sau 1s
        multiplier = 2,       // 1s → 2s → 4s
        maxDelay = 10000,     // Cap ở 10s
        random = true         // Jitter: 1s ± random → spread retries không cùng lúc
    )
)
@CircuitBreaker(name = "driverService")  // Nếu fail nhiều → stop retry hoàn toàn
public void callDriverService() { /* ... */ }
🎯 Vì sao cần jitter?

Không có jitter: 10,000 clients fail cùng T+0, retry cùng T+1s, T+2s, T+4s → 3 wave spike đồng loạt. Có jitter (random ±50%): retry phân bố [0.5s, 1.5s] → traffic smooth, dễ recover.

🔥 Combo "trinity" chống cascading failure

Timeout + Circuit Breaker + Bulkhead. Thiếu 1 trong 3, hệ thống dễ chết:

  • Timeout: limit max wait per call (1-2s)
  • Circuit Breaker: stop wasting calls khi downstream sick
  • Bulkhead: isolate thread pools, contain failure

Plus: retry với jitter, fallback degradation, observability để detect sớm.

4. Anti-Patterns

🔧 Counter-example — khi nào KHÔNG nên microservices

Microservices không phải mặc định "kiến trúc tốt". Nó đánh đổi độ phức tạp vận hành lấy khả năng scale độc lập. Chọn microservices là sai khi:

  • Team nhỏ / sản phẩm sớm: 3-5 kỹ sư mà 15 service = mỗi người gánh nhiều repo, distributed bug, không ai hiểu toàn cảnh. Modular monolith (module rõ ràng trong 1 deploy) tốt hơn.
  • Domain chưa ổn định: bounded context còn đổi liên tục → cắt service sai chỗ, sau phải gộp/tách lại rất đắt. Monolith cho phép đổi ranh giới gần như free.
  • Transaction mạnh là cốt lõi: nghiệp vụ cần ACID xuyên nhiều thực thể liên tục → Saga + eventual consistency thêm hàng tá edge case so với một DB transaction.
  • Chưa có nền tảng vận hành: không có CI/CD, observability, service discovery → microservices phơi bày mọi điểm yếu đó cùng lúc.

Quy tắc: "monolith first" — tách service khi có lý do cụ thể (một phần scale khác hẳn phần còn lại, team cần deploy độc lập), không phải vì microservices đang là trend.

#1 Distributed Monolith

ServiceA → sync call → ServiceB → sync call → ServiceC → sync call → ServiceD
Latency: 4 × network hop + 4 × timeout potential
Coupling: deploy A buộc deploy B/C/D cùng version
🎯 Vì sao tệ?

Có cost của microservices (network, ops, debugging) nhưng không có benefit (independent deploy, scaling). Worst of both worlds. Fix: dùng async/events thay sync calls, cache data local để tránh chain.

#2 Nanoservices — Tách quá nhỏ

1 service 50-100 LOC, không đủ work để justify network + ops overhead. Mỗi request touch 8 services → 8× latency.

Fix: group by bounded context, không phải feature.

#3 Shared Database

ServiceA ──┐
            ├→ shared_db
ServiceB ──┘

Schema change tightly couple 2 services. Mất hết benefit của microservices. Fix: mỗi service own DB, sync qua event hoặc API.

#4 No API Versioning

Service v2 đổi response schema → mobile app v1 (cũ) bị break. Fix: URI versioning (/api/v1/, /api/v2/), support multiple versions, có deprecation timeline.

#5 No Distributed Tracing

User complain "trip booking 10s mới response", engineer mở 5 service logs grep theo trip ID → không biết bottleneck. Fix: Jaeger/Zipkin + propagate X-Trace-ID qua MDC + đặt trace ID trong mọi outbound call.

@Component
public class TraceIdFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res,
            FilterChain chain) throws IOException, ServletException {
        String traceId = req.getHeader("X-Trace-ID");
        if (traceId == null) traceId = UUID.randomUUID().toString();
        MDC.put("traceId", traceId);
        res.addHeader("X-Trace-ID", traceId);
        try {
            chain.doFilter(req, res);
        } finally {
            MDC.remove("traceId");
        }
    }
}

// Outbound: propagate trace ID
HttpHeaders headers = new HttpHeaders();
headers.set("X-Trace-ID", MDC.get("traceId"));
HttpEntity<?> entity = new HttpEntity<>(null, headers);
restTemplate.exchange(driverUrl, GET, entity, DriverInfo.class);

5. Interview Framework

Tầng 1 — SurfaceMid gate
Q: Service discovery là gì, vì sao không hardcode IP?
"Service discovery dynamically tìm services khi chúng scale/restart. Hardcoding break khi instance die hoặc IP đổi. Service registry (Eureka, Consul, K8s DNS) maintain list active instances. Client query registry hoặc gọi load balancer (server-side). Health check remove dead instances. Modern: K8s + DNS đơn giản nhất, không cần lib trong code."
Q: Giải thích API Gateway pattern.
"Single entry point cho mọi request. Gateway handle auth (JWT validation), rate limiting, request routing, SSL termination, tracing. Prevent client biết internal service locations. Simplify cross-cutting concerns — implement 1 chỗ thay vì duplicate trong 15 services. Trade-off: gateway là single point of failure (cần HA setup) và có thể là bottleneck → phải scale horizontal."
Tầng 2 — Deep DiveMid → Senior
Q: Design booking trip API microservices. TripService, DriverService, PaymentService. Ensure atomicity.
"Không có DB transaction span 3 services. 2 approach:

Orchestration (recommended for complex): BookingOrchestrator coordinate steps (reserve driver → charge payment → create trip). Mỗi step có compensation (release driver, refund). Camunda BPMN cho visual workflow + state persistence (recover sau crash).

Choreography: TripService publish TripRequestedEvent. DriverService listen, reserve, emit DriverReservedEvent. PaymentService listen, charge, emit PaymentChargedEvent. TripService confirm. Nếu fail giữa chừng, emit compensation events.

Trade-off: Orchestration visible nhưng tight coupling với services. Choreography decoupled nhưng flow implicit, debug khó. Choice: orchestration cho 5+ steps complex workflow (order fulfillment, booking trip), choreography cho 2-3 steps đơn giản."
Q: Cách prevent cascading failures khi A → B → C?
"Trinity: Timeout + Circuit Breaker + Bulkhead.

Timeout: mỗi call có max wait (1-2s). Tránh thread block vô hạn.
Circuit Breaker: nếu downstream fail rate > 50% trong 10 requests → OPEN state, fail-fast 30s. HALF_OPEN cho 3 test calls, success → CLOSED (recovered).
Bulkhead: mỗi external service call có thread pool riêng (10-20 threads). Nếu DriverService slow chỉ ảnh hưởng pool của nó, không drain pool của PaymentService.

Thêm: retry với exponential backoff + jitter (tránh retry storm), fallback (degrade gracefully thay error), observability (Prometheus metric → alert nếu circuit OPEN).

Critical: thiếu 1 trong 3 của trinity → hệ thống dễ cascade. Có đủ 3 → 1 service down chỉ ảnh hưởng 1 chức năng, app vẫn hoạt động limited."
Tầng 3 — ArchitectureSenior
Q: RideZ có 20 microservices. Design observability stack: (a) service nào slow? (b) request nào fail? (c) vì sao request X fail across 5 services?
"4 trụ cột observability:

1. Distributed Tracing (Jaeger/Tempo): trace ID propagate qua header X-Request-ID + MDC. Mỗi service log span (start/end timestamp, parent span ID). UI hiển thị waterfall: Gateway[1ms] → TripService[50ms] → DriverService[200ms slow!]. Trả lời (a) và (c).

2. Structured Logs (ELK / Loki): centralized JSON logs với traceId, service, level, message. Query trong Kibana: traceId:xyz → toàn bộ logs của request đó across services. Trả lời (b) và (c).

3. Metrics (Prometheus + Grafana): mỗi service expose /actuator/prometheus với P50/P95/P99 latency, error rate, RPS. Dashboard show top 10 slowest service realtime. Trả lời (a).

4. Alerts (Alertmanager + PagerDuty): rule như "error rate > 5% trong 5 min → page on-call", "P99 latency > 2s → warning". On-call nhận alert, mở Grafana dashboard → biết service nào, mở Jaeger trace → biết bottleneck. Mở Kibana logs → biết exception cụ thể.

End-to-end example: alert "TripService error rate 8%" → Grafana cho thấy spike từ 14:32 → Jaeger trace 14:32 cho thấy DriverService timeout 5s → Kibana log DriverService 14:32 cho thấy SQLException: deadlock. Root cause: DriverService DB deadlock. Total debug time: 5 phút thay vì hours.

Investment: setup tốn 2-4 tuần engineering. ROI: mỗi outage giảm từ hours xuống minutes, mỗi P1 incident saving thousand $ cost."
🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. Vì sao microservices KHÔNG dùng 2PC cho transaction xuyên service? Saga thay bằng gì?
  2. Saga "không có rollback, chỉ có compensation" nghĩa là gì? Vì sao mọi compensation phải idempotent?
  3. Mô tả một cascading failure từ đầu đến cuối: cái gì cạn trước, vì sao lan, và "trinity" Timeout+Circuit Breaker+Bulkhead chặn ở đâu.
  4. Circuit Breaker có 3 trạng thái nào? Khi nào chuyển OPEN → HALF_OPEN → CLOSED?
  5. Nêu 2 tình huống KHÔNG nên dùng microservices và lý do.
Bí câu nào = quay lại đúng section. Trôi cả 5 = sẵn sàng cho vòng phỏng vấn microservices.
