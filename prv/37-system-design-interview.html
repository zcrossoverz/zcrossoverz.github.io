System Design Interview — Framework 5 bước, không thuộc design "hoành tráng"

System design interview KHÔNG đánh giá "thiết kế đẹp" — đánh giá process. Đa số candidate fail không phải vì design sai, mà vì: jump-to-design không clarify, vẽ box vô lý không estimate trước, deep dive sai chỗ, hết giờ vẫn ở high-level. File này dạy process qua framework UMPIRE-tương-đương cho system design: Clarify → Estimate → HLD → Deep Dive → Trade-off. Phân biệt với cả đống file "10 system design examples" trên blog — file đó cho bạn biết bài, file này cho bạn biết approach.

🎯 Mục tiêu
Reflex 5-step framework + Estimate trong 5 phút + Handle deep-dive probe
📅 Thời lượng
2 tuần đọc + 3 mock thật sự
📚 Prereq
Hiểu cơ bản #13 Distributed Tx, #14 Redis, #20 Kafka, #19 API Design, #33 Idempotency
🔥 Độ ưu tiên

⚠️ Cảnh báo "hiểu giả" — System Design dễ hiểu nhầm cấp 3

Vì sao System Design đặc biệt dễ tạo hiểu giả:
  • Đọc blog "design X" cảm thấy hiểu — Tutorial design hệ thống nhìn rõ ràng, ai đọc cũng gật gù. Nhưng vào interview với bài lạ → không biết bắt đầu từ đâu vì chỉ recognize pattern, không apply được.
  • Tutorials dạy "đáp án" — Bạn nhớ "Twitter dùng fan-out write" nhưng không biết vì sao hay khi nào. Interviewer hỏi variant → bí.
  • Capacity estimation không thực hành — Đọc "10K TPS = 3 shard" mà chưa tự tính. Interview phải tính nhẩm trên whiteboard.
  • Vocabulary tốn nhiều bandwidth não — "CAP theorem", "consistency model", "fan-out" — nhớ tên ≠ biết apply.

📖 5 quy tắc đọc

  1. Mỗi step (3-7) phải tự practice 1 lần. Đọc Step 1 (Clarify) → mở giấy → tự viết clarifying questions cho 1 bài bất kỳ (vd "design chat app"). Không tự làm = chưa internalize.
  2. Estimate (Step 2) phải tay tính trên giấy. Đọc bảng latency hierarchy + capacity formula → tự tính cho 1 ví dụ. Tay không gõ = não không nhớ con số.
  3. Section 8 (URL shortener) phải làm trước khi đọc đáp án. Đóng file ở Section 7, mở giấy, tự design URL shortener trong 35 phút. Mở lại, so sánh.
  4. Section "5 anti-behaviors" mở lại sau mỗi mock. Sau khi mock với peer, check xem dính bao nhiêu anti-behavior. Đây là feedback loop chính.
  5. 3 labs cuối file bắt buộc. 3 mock thật, 1 lab capacity estimation. Skip = ở L1.

🗺️ Roadmap framework 5-step (45-phút phỏng vấn chuẩn)

S1
Clarify · 5 min · CORE · Functional + Non-functional + Constraints, write down assumptions
S2
Estimate · 5 min · HIGH-SIGNAL · QPS, storage, bandwidth, cache size — math nhẩm
S3
High-Level Design · 10 min · CORE · Box + arrow + data flow, không deep dive
S4
Deep Dive · 15 min · HIGH-SIGNAL · 1 hardest problem — interviewer pick, không phải bạn
S5
Trade-offs & Wrap · 5 min · CORE · Bottleneck, scale 10×, cuts
+5
Buffer · 5 min · Follow-up Q&A, clarification, missed scenarios
Quy tắc cứng: KHÔNG skip step. Junior hay skip Step 1 (jump-to-design) → interviewer ghi "doesn't clarify". Hay skip Step 2 (no estimate) → "doesn't quantify". Hay quên Step 5 (no trade-off) → "doesn't think about alternatives". Mỗi step có signal riêng, mất 1 = giảm 1 grade.

1. 45 phút phỏng vấn — bài tương đương CORE

Bạn vào phòng phỏng vấn senior backend tại 1 tech company. Sau intro 5 phút, interviewer mở vấn đề:

"Design a URL shortener like bit.ly. Should support:
 - Shorten long URL → short URL (e.g., bit.ly/Xy7Q)
 - 100M new URLs per month
 - Read-heavy: 100:1 read-to-write ratio
 - Click analytics (count clicks per short URL)
 - Custom alias (user can choose alias if available)
 - 99.9% uptime
 You have 45 minutes. Where do you start?"

4 câu hỏi buộc tư duy

  1. Trong 5 phút đầu, bạn nói gì? Vẽ gì? Hỏi gì?
  2. Capacity estimation: 100M URL/tháng = bao nhiêu QPS? Storage 5 năm bao nhiêu? Cache size?
  3. Generating short URL: dùng auto-increment ID + base62 encode hay hash MD5 + take first 7 chars? Trade-off?
  4. Read-heavy → cần cache. Cache miss strategy thế nào? TTL bao lâu?

⏸️ Pause & Predict — Trước đọc Section 2

Viết ra giấy 3 clarifying questions bạn sẽ hỏi interviewer. Sau đó so sánh với câu trả lời ở Section 3 (Step 1 — Clarify).

Đa số candidate L1 viết được 1-2 câu rồi bí. L2 sẽ có ≥ 3 câu cho mỗi trục (Functional / Non-functional / Constraints).

2. Vì sao cần framework? CORE

Có 3 loại failure mode phổ biến mà framework giúp bạn tránh:

Failure mode 1: Jump-to-Design

❌ Candidate điển hình

Interviewer: "Design URL shortener"
Candidate: [bắt đầu vẽ ngay]
  "OK, có web server, có database lưu mapping..."
  [vẽ box load balancer, web server, DB]

Sau 10 phút vẽ, interviewer ngắt:
  "Hold on. Bạn assume capacity gì?
   Read-heavy hay write-heavy?
   Latency requirement?"

Candidate: [bí, design phải scrap, làm lại]

Symptom: Hết 15 phút mà chỉ có high-level vague. Còn 30 phút cho deep-dive + trade-off = không đủ.

Failure mode 2: Capacity estimation đoán mò

❌ Candidate điển hình

Interviewer: "100M URL/tháng, calculate storage cho 5 năm."
Candidate: "Khoảng... vài GB? 10 GB?"
Interviewer: "Show math."
Candidate: [im lặng]

Đáp án đúng: 100M × 60 tháng × ~500 bytes/URL = ~3 TB
(Off by 2 orders of magnitude)

Symptom: "doesn't quantify" feedback. Senior signal mất.

Failure mode 3: Deep dive sai chỗ

❌ Candidate điển hình

Interviewer hỏi probe "Walk me through how you handle X" — candidate deep dive vào chỗ candidate quen, không phải chỗ interviewer muốn. Interviewer ngắt 3 lần, candidate vẫn deep dive sai.

Symptom: "doesn't listen", "fixed mindset on one approach". Lost senior signal.

Framework giải quyết 3 failure mode này

StepPhòng failure
S1 — ClarifyBuộc 5 phút clarify TRƯỚC design → không jump-to-design
S2 — EstimateMath nhẩm có template → không đoán mò
S3 — HLDHigh level trước, deep dive sau → giữ thứ tự đúng
S4 — Deep DiveInterviewer pick chỗ deep dive → bạn không tự chọn nhầm

Insight: Framework KHÔNG phải "rule cứng". Interviewer giỏi sẽ check xem bạn có loop lại Step 1 nếu phát hiện hiểu nhầm ở Step 4. Đây là dấu hiệu thinking thật, không phải đọc thuộc framework.

3. Step 1 — Clarify (5 phút) CORE

Quy tắc: 5 phút clarify TRƯỚC khi vẽ gì. Mọi câu hỏi đi vào 1 trong 3 trục: Functional / Non-functional / Constraints. Viết assumptions ra giấy, đọc lại cho interviewer confirm.

3 trục clarifying

① Functional Requirements (features cần)

Cho URL shortener — bạn sẽ hỏi:

? Core feature là gì? — shorten URL + redirect là minimum
? User account cần không? — anonymous OK hay phải đăng ký?
? Custom alias (user chọn short URL) — phase 1 hay phase 2?
? Click analytics (count, geo, device) — chi tiết bao nhiêu?
? Expiration: short URL có expire không? TTL bao lâu?
? Bulk shortening API cho dev — có cần không?
? Branded domain (myco.link/Xy7Q) — multi-tenant?
? Mobile deep link — có cần handle không?

② Non-functional Requirements (scale, latency, consistency)

? Scale — bao nhiêu URL/tháng? Read:Write ratio?
  (Interview thường: 100M/tháng, 100:1 read:write)
? Latency — read URL phải < bao nhiêu ms?
  (Industry: redirect < 100ms total)
? Uptime — 99.9% (~8.76h/year) hay 99.99% (~52min/year)?
? Consistency — eventually OK hay phải strong cho click count?
? Geo — single region hay multi-region?

③ Constraints (tech, team, regulatory)

? Tech stack — có preference không?
? Cloud — AWS/GCP/Azure?
? Team size — 2 dev hay 20 dev?
? Budget — startup tight hay scaleup ample?
? Compliance — GDPR data residency?

Quy tắc 3 clarifying questions per trục

Đa số candidate thuộc 2 thái cực: hỏi 0-1 câu (jump to design) hoặc hỏi 10+ câu (interviewer ngán). Sweet spot: 2-3 câu mỗi trục, total 6-9 câu trong 5 phút.

Mẫu chuẩn — đáp án cho Section 1 (URL shortener):
"Tôi sẽ assume sau khi clarify:
• Functional: shorten + redirect là core. Custom alias phase 1. Click count đơn giản, không cần geo/device detail phase 1.
• Non-functional: 100M URL/tháng, 100:1 read:write → 10B reads/tháng. Redirect P99 < 100ms. Uptime 99.9%. Eventually consistent click count OK.
• Constraints: AWS, team ~5 dev, budget moderate. Multi-region phase 2.
Tôi sẽ design theo assumptions này. Bạn confirm OK không?"

4. Step 2 — Estimate (5 phút) HIGH-SIGNAL

Quy tắc: Math nhẩm trên whiteboard, show work. Interviewer đánh giá phương pháp, không phải con số chính xác. ±50% acceptable.

Latency hierarchy — memorize

"Numbers Every Programmer Should Know" (Jeff Dean / Peter Norvig):

L1 cache reference                    0.5 ns
Branch mispredict                     5 ns
L2 cache reference                    7 ns
Mutex lock/unlock                     100 ns
Main memory reference                 100 ns
Compress 1KB with snappy              10 μs
Send 1KB over 1 Gbps network          10 μs
Read 4KB random from SSD              100 μs
Read 1MB sequential from memory       250 μs
Round trip within datacenter          500 μs
Read 1MB sequential from SSD          1 ms
Disk seek                             10 ms
Read 1MB sequential from disk         20 ms
Round trip cross-continent            150 ms

Key takeaway: Network 150× chậm memory. Disk 100× chậm SSD.

Capacity formula template

// 1. QPS
Average QPS = Daily writes / 86,400 sec
Peak QPS = Average × 3-5 (use interviewer's peak)

// 2. Storage
Annual = records/day × avg_size × 365
With retention N năm × N

// 3. Bandwidth
Bandwidth = QPS × (avg_req_size + avg_resp_size)

// 4. Cache (Pareto 80/20)
Cache size = hot_set_size × avg_object_size
Hot set = 20% of total

// 5. DB Connections
Connections = (Peak QPS × query_complexity_ms) / 1000
App servers = total_connections / 20 (typical pool)

Estimate cho URL shortener — show work

Given: 100M URL/tháng, 100:1 read:write, retention 5 năm

QPS:
  Writes = 100M / (30 × 86,400) = ~40 writes/sec
  Reads  = 100M × 100 / (30 × 86,400) = ~4,000 reads/sec
  Peak (×3) = 120 writes/sec, 12,000 reads/sec

Storage (5 năm):
  Total URLs = 100M × 60 tháng = 6B URLs
  Per record: short_code (7B) + long_url (~500B) + metadata (~100B) ≈ 600B
  Total = 6B × 600B = 3.6 TB

  Click events (if log raw): 100M × 60 × 100 (read ratio) = 600B events
  Per event ~50B → 30 TB
  → KHÔNG log raw, dùng aggregation: count column update mỗi click

Bandwidth:
  Read: 12K req/sec × (50B req + 300B redirect response) = ~4 MB/sec
  Write: 120 req/sec × (500B + 100B) = ~70 KB/sec
  → Nhỏ, không bottleneck

Cache (80/20):
  Top 20% URLs = 1.2B URLs hot
  Hmm, 1.2B × 600B = 720 GB → too big cho Redis
  → Cache top 10M URLs (recent + popular)
  10M × 600B = 6 GB → fit single Redis instance, 12 GB cho HA replica

DB Connections:
  Peak 12K read/sec, mỗi query ~5ms (with index)
  Connections needed = 12K × 5/1000 = 60
  With 20/pool = 3 app servers minimum
  HA buffer: 5-7 servers

Bottleneck: Cache eviction strategy (LRU) + DB read pattern
Lưu ý interview reality: KHÔNG cần memorize công thức. Cần memorize: ① latency hierarchy (rough), ② 86400 = sec/day, ③ Pareto 80/20, ④ peak ≈ 3-5× average. Còn lại derive on-the-fly.

  2. Storage cho 10 năm tweets (assume 100B/tweet, 1M tweets/day).

6. Step 4 — Deep Dive (15 phút) HIGH-SIGNAL

Quy tắc: Interviewer PICK chỗ deep dive — không phải bạn. Nếu interviewer im, hỏi: "Bạn muốn tôi đi sâu component nào? Short code generation, cache strategy, hay click counter?"

4 chỗ deep dive thường gặp cho URL shortener

Deep dive 1: Short code generation

❌ Approach 1: MD5 hash + take 7 chars

short_code = MD5(long_url)[0:7]

Vấn đề:
- Collision rate cao (birthday paradox
  với 7 char base62 → 3.5 tỉ space)
- Same URL → same short code (đôi khi
  muốn khác, đôi khi không — không control)
- Custom alias conflict logic phức tạp
- Không retry logic được vì hash deterministic

✅ Approach 2: Counter + Base62 encode

id = ZooKeeper.nextSequence()  // distributed counter
short_code = base62(id)

Pros:
- Unique guaranteed (counter monotonic)
- Sequential ID → predict được capacity
- Custom alias check riêng dễ
- Có thể partition counter cho scale

Cons:
- Counter có thể bottleneck → batch allocate
  (mỗi app server lấy 1000 IDs một lần)

Approach 3 (hybrid): Counter + random salt — get best of both.

Deep dive 2: Cache strategy

Read path:
  GET /Xy7Q9aB
    → Check Redis: GET url:Xy7Q9aB
    → HIT: return long_url, async publish click event
    → MISS: query Postgres, populate cache với TTL 24h, return

Cache eviction: LRU (Redis default)
Cache size: 6 GB (top 10M URLs)
Cache hit rate target: > 95% (vì hot URLs chiếm hầu hết traffic)

Cache invalidation:
  DELETE /Xy7Q9aB → DELETE url:Xy7Q9aB từ Redis, soft-delete trong DB
  Update long_url (hiếm) → DEL cache, update DB

Cache stampede problem:
  Top URL có TTL 24h, expire → 1000s concurrent requests miss cache → DB hammered
  Solution:
  - Probabilistic early expiry (refresh before actual expiry)
  - Lock + double-check pattern
  - Cache-aside với async refresh

Deep dive 3: Click counter (high write rate)

Problem: 12K reads/sec → 12K click_count++ writes/sec
         Direct DB UPDATE → row lock contention → slow

Solution: Async aggregation
  1. App server publish click event to Kafka (fire-and-forget)
  2. Consumer batch reads events, aggregate trong 1-min window
  3. Single UPDATE per (short_code, time_bucket) commit

Trade-off: Click count lag 1-2 phút (eventual)
  → User analytics dashboard có disclaimer "near real-time"

Alternative: Redis INCR (atomic, fast)
  redis.INCR("clicks:Xy7Q9aB")
  Periodic flush từ Redis → Postgres (every 5 min)
  Risk: Redis crash mất 5 phút data
  Mitigation: Redis AOF persistence + replication

Deep dive 4: Sharding chiến lược (scale 10×)

Initial: single Postgres handles 12K reads/sec dễ
Scale 10×: 120K reads/sec → cần shard

Sharding key options:
  A) Hash(short_code) → uniform distribution
     - Range query khó (list URLs của user)
     - Resharding khó (rehash toàn bộ)

  B) Range by short_code prefix
     - Hot spot nếu URL mới luôn đến cuối range
     - Range query OK

  C) Hash(user_id) → URLs cùng user 1 shard
     - Anonymous URLs (user_id null) → 1 shard, hot spot
     - List user URLs nhanh

Khuyến nghị: A) Hash(short_code) với consistent hashing
  → Resharding dễ hơn naive hash
  → User-level queries: cache trong Redis sorted set
  → Cross-shard query: rare (analytics) → batch job

Khi không sure, narrate

Khi không sure, narrate quá trình suy nghĩ. Interviewer rất thích thấy: "Tôi đang nghĩ về 2 option. Option A pros... cons... Option B... Tôi nghiêng về A vì..." — đây là senior signal mạnh hơn "biết đáp án".

7. Step 5 — Trade-offs & Wrap (5 phút) CORE

5 phút cuối là chỗ ghi điểm senior. 3 việc bắt buộc:

① Bottleneck identification

"Bottleneck trong design hiện tại:
 1. Counter sequence — single point cho ID generation.
    Mitigation: distributed counter (ZooKeeper) hoặc batch allocate.

 2. Cache hit rate phải ≥ 95% — nếu drop → DB overloaded.
    Mitigation: warm cache, monitor hit rate, alert.

 3. Single region — geo-distant users high latency.
    Mitigation: multi-region với edge POPs (CloudFront caching)."

② Scale 10× scenario

"Nếu scale 10× (1B URL/tháng, 120K read/sec):
 - App servers: stateless, scale horizontal. Add 50+ instances.
 - DB: shard Postgres bằng hash(short_code). 10 shards.
 - Cache: Redis cluster mode, 60 GB.
 - Kafka: tăng partitions (currently 10 → 100), brokers (3 → 9).
 - CDN: aggressive edge caching cho top 1M URLs.
 - Counter: distributed via Snowflake-like (Twitter pattern)."

③ Corners cut + future enhancement

"Để ship phase 1 trong 6 tuần, tôi cut:
 - Custom domain (myco.link) → phase 2
 - Detailed analytics (geo, device, time series) → phase 2
 - URL preview thumbnail → phase 3
 - Spam URL detection → phase 2 (sau khi có user feedback)
 - Admin dashboard → phase 2

 Nếu có thêm 2 sprint, ưu tiên:
 1. Spam detection (Google Safe Browsing API integrate)
 2. Detailed analytics
 3. Custom domain"
Senior signal: Việc tự nêu trade-off + roadmap mà interviewer chưa hỏi = "thinks like a PM/architect". Đây là điểm cộng senior level. Junior chờ hỏi mới trả lời.

9. 6 patterns hay dùng trong system design SUPP

PatternKhi nào dùngTrade-off
Cache-aside Read-heavy, data tolerates staleness + Simple, miss path explicit · − Stampede risk khi expire
Write-through Cache phải up-to-date với DB + Strong consistency · − Slow writes (write 2 lần)
Outbox pattern Event publishing + DB change atomic + At-least-once delivery · − Phải có poller, extra latency
Idempotency key Retryable writes (payment, order create) + Dedup safe · − Extra storage cho keys (→ #33)
Rate limiting (Token bucket) Prevent abuse, fair sharing + Burst tolerant · − Memory per user, distributed sync tricky
CQRS Read pattern khác hẳn write pattern + Optimize riêng read/write · − Phức tạp, eventual sync

Khi gặp bài lạ, hỏi mình: "Pattern nào fit problem này?" Reflex 6 pattern này phủ ~70% bài interview.

10. Interview behaviors — 5 anti + 5 winning HIGH-SIGNAL

Cùng design có thể "Hire" hoặc "No Hire" tuỳ behavior. 5 anti-pattern phải tự bắt:

❌ Anti #1: Im lặng (silent thinker)

Tự suy nghĩ 5 phút, vẽ thầm, không narrate. Interviewer không biết bạn nghĩ gì → "no signal".

Fix: Narrate liên tục, kể cả khi chưa chắc: "Tôi đang nghĩ về 2 option..."

❌ Anti #2: Jump-to-code (java engineer mode)

Mới 5 phút đã vào code Java cụ thể, class diagram, method signature. Quên estimate, quên trade-off.

Fix: Code chỉ ở Deep Dive nếu interviewer ask. HLD ở box + arrow level.

❌ Anti #3: Fixed mindset on 1 approach

Interviewer hint "What if we used NoSQL?" — bạn defend Postgres tới cùng. Không listen.

Fix: Explore option interviewer gợi ý, dù bạn không initially chọn. "Hmm, NoSQL có thể work nếu... trade-off là..."

❌ Anti #4: Knowledge dump (vocabulary parade)

Nhét đủ buzzword: "CAP theorem", "PACELC", "vector clock", "Paxos" — nhưng không apply.

Fix: Mention pattern chỉ khi relevant. "Đây là eventual consistency vì..." — explain why, không chỉ name-drop.

❌ Anti #5: Quên time management

Spend 25 phút clarify + estimate. Còn 20 phút cho HLD + Deep Dive + Trade-off. Interviewer pull back.

Fix: Mặc đồng hồ trong đầu: 5/5/10/15/5/5. Check thường xuyên.

5 winning behaviors

  1. State assumptions explicitly + confirm. "Tôi assume X. OK chứ?" — clear contract.
  2. Draw before code. Interviewer redirect dễ ở giai đoạn vẽ. Code muộn = lock-in sớm.
  3. Show math, không đoán. "Let me work this out: 100M / 30 / 86400 = ..."
  4. Unprompted trade-off. "Trade-off của approach này là... Alternative là..."
  5. Ask for feedback. Mỗi 10 phút: "Direction OK chứ? Bạn muốn focus đâu?"

🎯 3-tier interview framework

Tier 1 — Surface (Junior signal)

Q1: Tại sao cần load balancer + multiple app servers?

Trả lời: "Single app server có giới hạn: CPU, memory, connection pool. Khi traffic vượt giới hạn → request queue, latency tăng, hoặc crash. Multiple app server scale horizontal — load balancer phân phối traffic đều. Plus: HA — 1 server down, traffic redirect server khác. Stateless app servers là điều kiện để scale horizontal."

Q2: Cache miss strategy — write-through vs write-back vs cache-aside?

Trả lời: "Cache-aside: app code đọc cache trước, miss thì query DB + populate cache. Đơn giản, phổ biến nhất. Write-through: ghi vào cache + DB cùng lúc → cache luôn fresh nhưng write chậm. Write-back: ghi cache trước, async flush DB → fast write nhưng risk mất data nếu cache crash. URL shortener dùng cache-aside vì read-heavy + ít update."

Tier 2 — Deep Dive (Mid vs Senior)

Q3: Eventual consistency vs Strong consistency — chọn khi nào?

"Strong consistency: mọi read thấy write mới nhất. Cần cho payment, inventory (oversell prevention), authentication. Cost: latency cao, throughput thấp (vì coordination).

Eventual consistency: read có thể stale ngắn hạn, eventually converge. Cho social feed, click count, view count. Throughput cao hơn nhiều.

URL shortener: click count eventual OK (1-2 phút lag acceptable). Long_url mapping strong OK vì update hiếm. Dùng cache 24h TTL = lag tối đa 24h cho update (rare).

Trade-off cụ thể: Strong consistency thường yêu cầu single-leader DB hoặc consensus protocol (Raft/Paxos) → bottleneck. Eventual cho phép multi-leader, multi-region, lower latency."

Q4: Sharding strategy — hash vs range vs directory?

"Hash sharding: shard = hash(key) % N. Uniform distribution. Cons: range query khó (phải scatter-gather), resharding khó.

Range sharding: shard theo key range (vd userId 1-1M shard 1, 1M-2M shard 2). Pros: range query dễ. Cons: hot spot nếu key skewed (user mới luôn đến shard cuối).

Directory sharding: lookup table mapping key → shard. Flexible, có thể move dữ liệu. Cons: lookup table là bottleneck/SPOF.

URL shortener: hash sharding với consistent hashing (resharding chỉ move 1/N data thay vì rehash toàn bộ). User-level queries (rare) batch query cross-shard.

Twitter/Facebook timeline: directory-like — user shard map cached, user data sticky shard."

Tier 3 — Architecture (Senior signal)

Q5: Design rate limiter cho 100M users, 1000 requests/sec/user limit, multi-region active-active.

Phút 1 — Clarify:
  - Rate limit type: per-user per-second hay sliding window?
  - Burst tolerance: 1000 sustained hay burst OK?
  - Strict cross-region hay region-local OK?
  - Hard limit (reject) hay soft (throttle/queue)?

Phút 2 — Approach options:
  A) In-memory token bucket per app server
     - Fast, no network
     - Cons: không enforce cross-server limit
  B) Redis-based distributed counter
     - Atomic INCR with TTL
     - Latency ~1ms thêm
     - Cons: Redis bottleneck cho 100K req/sec
  C) Approximate counting (Count-Min Sketch)
     - Lightweight, distributed easy
     - Cons: approximate, not exact

Phút 3 — Architecture (chọn B + tier):
  ┌────────┐  ┌────────────────┐
  │ Client │─►│ Edge gateway   │ (CDN level — drop obvious abuse)
  └────────┘  └───────┬────────┘
                      ▼
              ┌───────────────┐
              │ App server    │
              │ + local cache │
              │ (last second  │
              │  bucket)      │
              └───────┬───────┘
                      ▼
              ┌───────────────┐
              │ Redis cluster │ (Lua script atomic)
              │ rate:userId   │
              │ TTL 1 sec     │
              └───────────────┘

Phút 4 — Lua script atomic:
  local current = redis.call('INCR', KEYS[1])
  if current == 1 then redis.call('EXPIRE', KEYS[1], 1) end
  if current > tonumber(ARGV[1]) then return 0 else return 1 end

Phút 5 — Multi-region trade-off:
  Strict global counter: cross-region latency 150ms → unacceptable
  Region-local quota (per-region 1000/sec): user có thể abuse cross-region
    → Mitigation: detect anomaly + post-fact suspend
  Hybrid: 80% quota local + 20% global pool (CRDT counter)

  Trade-off: latency-vs-fairness. Production thường chọn region-local
  với detection layer sau.

🧠 Feynman test — Verify hiểu thật

🎓 Giải thích cho candidate khác — không nhìn file

Đóng tab. Trả lời 5 câu, < 30 giây mỗi câu:

  1. "5 step của framework là gì? Time budget mỗi step?" — Clarify (5'), Estimate (5'), HLD (10'), Deep Dive (15'), Trade-off (5'), Buffer (5'). Tổng 45 phút.
  2. "Vì sao phải clarify trước khi design?" — Tránh jump-to-design lạc hướng. Interviewer có specific scenario trong đầu. Skip clarify = design giả thuyết, có thể wrong. 5 phút clarify tiết kiệm 15 phút sửa.
  3. "3 con số nhẩm cần thuộc cho estimate?" — 86400 (sec/day). Pareto 80/20. Peak ≈ 3-5× average. Latency hierarchy rough (memory 100ns, network DC 500μs, disk 10ms).
  4. "Cache-aside vs write-through khác gì?" — Cache-aside: app đọc cache, miss query DB + populate. Write-through: ghi cache + DB cùng lúc. Cache-aside đơn giản hơn, write-through ensure cache fresh nhưng slow write. Read-heavy thường cache-aside.
  5. "Eventual vs Strong consistency — ví dụ chọn cho mỗi?" — Strong: payment, inventory, auth (correctness critical). Eventual: social feed, click count, view count (throughput critical, lag OK).

Bí 1 câu = hiểu giả. Quay lại section tương ứng.
