Docker & Kubernetes — Container, Probe, Resource, Zero-Downtime

K8s là chủ đề ai cũng "biết" — gõ kubectl apply được, vẽ được Pod/Service. Nhưng probe lên thật: "liveness probe check DB connection thì điều gì xảy ra khi DB chậm?", "requests khác limits chỗ nào và cái nào gây OOMKilled?", "maxUnavailable=0 mà vẫn downtime 30s thì thiếu gì?" — là lộ hiểu giả. File này dạy cả cơ chế (scheduler, QoS, rolling update) lẫn vận hành thật (đọc CrashLoopBackOff, OOMKilled, endpoints trống), với manifest production đầy đủ, trên domain ShopFlow — checkout service của một sàn e-commerce.

🎯 Mục tiêu
Viết Dockerfile + manifest production, hiểu probe/QoS, deploy zero-downtime
📅 Thời lượng
1 tuần · Sau #6 Spring Boot, #22 Resilience
📚 Prereq
Linux cơ bản · HTTP health check · Spring Boot Actuator
🔥 Độ ưu tiên
🟠 Cao — gần như mọi backend production chạy trên K8s

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy phổ biến

Tưởng hiểu: "Liveness và readiness gần giống nhau, đều check sức khoẻ app, để cả hai cùng gọi /health là xong."
Thực tế: Hai probe gây hậu quả ngược nhau khi fail — liveness fail → kill + restart pod; readiness fail → rút pod khỏi Service (không restart). Để liveness check DB → DB chậm → pod khoẻ bị giết hàng loạt → mất luôn cả nhóm.
Tưởng hiểu: "Secret an toàn hơn ConfigMap vì nó được mã hoá."
Thực tế: Secret mặc định chỉ base64 (obfuscation), KHÔNG mã hoá at-rest trong etcd. Ai đọc được etcd hoặc có quyền get secret là thấy plaintext. Mã hoá thật cần KMS / EncryptionConfiguration.
Tưởng hiểu: "Đặt maxUnavailable: 0 là chắc chắn zero-downtime."
Thực tế: Rolling update chỉ đảm bảo có pod READY, không đảm bảo pod cũ drain sạch request đang chạy. Thiếu graceful shutdown (xử lý SIGTERM + terminationGracePeriodSeconds + preStop) thì request in-flight vẫn bị cắt → vẫn lỗi lúc deploy.

1. Bài toán thực tế — checkout service crash lúc flash sale CORE ★

ShopFlow là sàn e-commerce. Dịch vụ checkout-api thỉnh thoảng crash vài giờ một lần. K8s dashboard cho thấy pod CrashLoopBackOff; memory leo dần tới khi OOM killer giết container. Mỗi lần deploy mất đúng 30 giây downtime. Tệ hơn: feature bật ở staging nhưng tắt ở production vì config lệch nhau.

14:02:11 WARN checkout-api-7c9 Liveness probe failed: Get "http://10.1.4.7:8080/health": context deadline exceeded (DB query 3s > probe timeout 2s)
14:02:11 ERROR kubelet Container checkout-api failed liveness probe, will be restarted
14:03:40 ERROR checkout-api-7c9 Last State: Terminated · Reason: OOMKilled · Exit Code: 137
14:03:41 WARN Back-off restarting failed container checkout-api → CrashLoopBackOff
⏸ Pause & Predict — trước khi đọc lời giải
Nhìn log trên, viết ra giấy:
  1. Có hai nguyên nhân crash khác hẳn nhau trong log. Tách chúng ra: cái nào do probe, cái nào do resource? Dấu hiệu phân biệt?
  2. Vì sao để liveness probe gọi xuống DB lại biến "DB chậm" thành "pod bị giết"? Sửa bằng probe nào?
  3. exit 137 nghĩa là gì? Tăng limit memory có chắc hết OOMKilled không, hay cần làm gì khác trước?
Đáp án rải ở 2.6 (OOMKilled = SIGKILL khi vượt memory limit; 137 = 128+9 = SIGKILL), 2.7 (liveness chỉ check nội bộ, readiness mới check DB). Đừng đọc tiếp khi chưa tách được 2 nguyên nhân.

5 câu Senior phải trả lời

  1. Vì sao pod vào CrashLoopBackOff? Phân biệt app crash vs resource starvation thế nào?
  2. Liveness vs readiness khác gì? Vì sao check DB trong liveness gây thêm crash?
  3. Requests vs limits cấu hình ra sao để tránh OOM mà không lãng phí hạ tầng?
  4. Vì sao config lệch giữa môi trường? Có nên để mật khẩu DB trong ConfigMap không?
  5. Chiến lược zero-downtime deploy là gì? Làm sao 30s downtime → 0?

2.1 Container vs Virtual Machine SUPPLEMENTARY

WHY trước WHAT: vấn đề gốc là "code chạy trên máy tôi nhưng fail trên server" (lệch dependency, OS, config). VM giải bằng cách đóng gói cả OS trong hypervisor — nặng, chậm khởi động. Container giải nhẹ hơn: là một process bị cô lập qua Linux namespace + cgroup, đóng gói filesystem theo lớp (Union FS), và dùng chung kernel với host.

ContainerVirtual Machine
Cô lậpProcess + namespace (pid/net/mount) + cgroup, chung kernel hostOS đầy đủ (guest kernel) trong hypervisor
Khởi động~mili-giây đến giâychục giây đến phút
Memory overhead~1/10 VM (không nhân bản OS)nặng (mỗi VM một OS)
FilesystemUnion FS theo lớp — layer dependency cache đượcDisk image đầy đủ
Mức cô lập bảo mậtYếu hơn — kernel bug/container-escape ảnh hưởng host + container khácMạnh hơn (ranh giới hypervisor)
Điểm dễ quên

Container chia sẻ kernel nghĩa là một lỗ hổng kernel hoặc container-escape có thể ảnh hưởng host và mọi container khác. Đó là lý do không chạy bằng root trong container (xem 3.1) và dùng securityContext.

2.2 Pod — đơn vị deploy nhỏ nhất CORE ★

Pod = 1 hoặc nhiều container + chung network namespace (mọi container trong pod nói chuyện qua localhost) + chung volume. Nhiều container một pod = sidecar pattern: ví dụ app container + log-shipper sidecar, hoặc app + service-mesh proxy. Pod là phù du (ephemeral) — bị xoá/tạo lại liên tục, IP đổi mỗi lần.

Pod "checkout-api-7c9"
 ├─ container: checkout-api   (8080)  ┐ chung localhost
 ├─ container: log-sidecar             ┘ chung volume /var/log
 └─ shared: network namespace + emptyDir volume

2.3 Deployment → ReplicaSet → Pod CORE ★

Bạn không tạo Pod trực tiếp trong production. Deployment khai báo trạng thái mong muốn (image, số replica) → tạo ReplicaSet giữ đúng N replica sống → ReplicaSet tạo Pod. Khi đổi image, Deployment tạo ReplicaSet mới, chuyển dần traffic từ cũ sang mới (rolling update).

Deployment (desired: image v1.2.3, replicas=3)
   └─ ReplicaSet v1.2.3 (đảm bảo luôn có 3 Pod)
        └─ Pod  Pod  Pod   (container đang chạy thật)
Update image → Deployment tạo ReplicaSet v1.2.4, scale down v1.2.3 dần

2.4 Service — entry point ổn định CORE ★

Vì Pod phù du (restart → đổi IP), không thể gọi thẳng Pod IP. Service cho một địa chỉ DNS + IP ổn định, route tới Pod theo label selector (không phải IP), và tự cập nhật danh sách endpoints khi pod lên/xuống.

⏸ Pause & Predict
Bạn deploy 3 pod checkout-api. Một pod restart và đổi IP từ 10.1.2.3 sang 10.1.2.9.
  1. Nếu service khác đang hardcode http://10.1.2.3:8080, điều gì xảy ra? ClusterIP sửa vấn đề này thế nào?
  2. Service biết route tới pod nào bằng cơ chế gì (không phải IP)?
  3. Nếu kubectl get endpoints trả về rỗng dù pod đang chạy, nguyên nhân thường gặp là gì?
Pod IP phù du → hardcode = vỡ khi restart. Service route theo label selector khớp Pod label. Endpoints rỗng thường do readiness probe fail → pod chưa Ready → bị loại khỏi endpoints (xem 2.7, 3.7).

2.5 ConfigMap vs Secret SUPPLEMENTARY

  • ConfigMap: cấu hình không nhạy cảm (profile active, feature flag, TTL cache, log level). Plaintext. Mount thành file hoặc inject env. Đây là lý do "config lệch môi trường" ở phần 1 — mỗi môi trường một ConfigMap.
  • Secret: credential, token, key. Base64 — obfuscation chứ không phải mã hoá; không encrypt at-rest mặc định. Cũng mount file/env.
🔧 Counter-example — khi nào KHÔNG nên dùng Secret kiểu mặc định

Đừng coi Secret mặc định là "đã an toàn":

  • Compliance / dữ liệu thật: Secret base64 nằm plaintext trong etcd. Production cần EncryptionConfiguration (encrypt etcd) hoặc external secret manager (Vault, cloud KMS, External Secrets Operator).
  • Mật khẩu DB / API key trong ConfigMap: không bao giờ — quyền đọc ConfigMap thường rộng hơn Secret, lộ ngay.
  • Secret commit vào Git: kể cả "đã base64" vẫn là plaintext. Dùng SealedSecrets / SOPS để commit an toàn.

Quy tắc: ConfigMap cho cái lộ ra không sao; Secret cho cái lộ ra là sự cố — và Secret mặc định vẫn cần thêm một lớp mã hoá thật.

2.6 Requests vs Limits + QoS CORE ★

Request = mức tối thiểu được đảm bảo; scheduler dùng request để xếp pod lên node (đảm bảo node còn đủ chỗ). Không set request → pod dễ bị evict khi node quá tải. Limit = trần cứng:

  • Vượt memory limit → container bị OOMKilled (SIGKILL, exit 137). Không "throttle" được memory.
  • Vượt CPU limit → bị throttle (chạy chậm lại), KHÔNG bị giết.

L1 — Biết tên

Biết "requests và limits là cấu hình tài nguyên". Symptom hiểu giả: đặt request = limit = số đại khái, không lý giải được vì sao pod bị OOMKilled hay Evicted, hay vì sao JVM heap không tôn trọng limit.

L2 — Explain được

Phân biệt: vượt memory limit → kill (137); vượt CPU limit → throttle. Hiểu QoS class quyết định thứ tự bị evict khi node thiếu tài nguyên. Hiểu JVM cần -XX:+UseContainerSupport -XX:MaxRAMPercentage để nhìn đúng limit thay vì RAM của cả node.

L3 — Apply được

Chọn request theo P95 thực đo, limit theo P99.5; đặt service quan trọng (checkout/order/inventory) ở Guaranteed QoS; gắn LimitRange để pod quên set vẫn có default; chẩn đoán "OOMKilled" = sửa leak hoặc tăng limit; "Evicted" = QoS thấp + node thiếu RAM → nâng QoS/right-size.

QoS class (ảnh hưởng thứ tự evict khi node hết tài nguyên):
1. Guaranteed : requests == limits cho CẢ cpu và memory  → evict CUỐI CÙNG → checkout/order/inventory
2. Burstable  : requests < limits (hoặc chỉ 1 trong 2)     → evict trước Guaranteed → reporting/notification
3. BestEffort : KHÔNG set requests/limits                  → evict ĐẦU TIÊN → KHÔNG dùng cho service quan trọng

2.7 Liveness / Readiness / Startup probe CORE ★

ProbeCâu hỏiFail → hậu quảCó nên check DB/downstream?
LivenessContainer còn sống?Kill + restart container❌ KHÔNG — DB chậm → giết oan pod khoẻ
ReadinessNhận traffic được chưa?Rút khỏi Service endpoints (KHÔNG restart)✅ Nên — chỗ đúng để check downstream sẵn sàng
StartupApp khởi động xong chưa?Hoãn liveness/readiness tới khi xongApp khởi động chậm (JVM nặng) dùng cái này để liveness không giết sớm
Anti-pattern kinh điển + symptom

Triệu chứng: log Liveness probe failed đồng loạt nhiều pod đúng lúc DB/downstream chậm, theo sau là restart hàng loạt → mất luôn cả nhóm pod đang khoẻ. Nguyên nhân: liveness check DB. Fix: liveness chỉ check nội bộ (/actuator/health/liveness — chỉ kiểm app process còn phản hồi); chuyển check DB/cache sang readiness (/actuator/health/readiness). Spring Boot tách sẵn 2 group này.

2.8 Rolling update + zero-downtime CORE ★

strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1         # cho phép tạo thêm tối đa 1 pod trong lúc rollout (cost: +1 pod)
    maxUnavailable: 0   # luôn giữ đủ pod READY → không tụt capacity

Rolling = thay pod dần. Nhưng "có pod READY" chưa đủ để zero-downtime. Cần thêm graceful shutdown: khi nhận SIGTERM, app ngừng nhận request mới, xử lý nốt request đang chạy, rồi mới thoát — trong khoảng terminationGracePeriodSeconds. Thêm preStop sleep vài giây để load balancer/kube-proxy kịp ngừng gửi traffic tới pod sắp chết (xoá khỏi endpoints có độ trễ lan truyền).

⏸ Pause & Predict
Đã đặt maxUnavailable: 0 mà deploy vẫn rớt vài request (HTTP 502) mỗi lần. Vì sao?
  1. Khi K8s gửi SIGTERM cho pod cũ, request đang xử lý dở ra sao nếu app thoát ngay lập tức?
  2. Có một "race" giữa lúc pod bị xoá khỏi endpoints và lúc nó thật sự dừng nhận traffic. preStop sleep giải quyết race đó thế nào?
  3. Spring Boot cần bật gì để hỗ trợ graceful shutdown (gợi ý: một property)?
maxUnavailable=0 chỉ lo capacity, không lo in-flight request. Thiếu graceful shutdown → app thoát ngay → request dở bị cắt = 502. Fix: bật server.shutdown=graceful + spring.lifecycle.timeout-per-shutdown-phase, set terminationGracePeriodSeconds: 30, thêm preStop sleep 5 để LB kịp drain.

2.9 Horizontal Pod Autoscaler SUPPLEMENTARY

HPA theo dõi metric (CPU/memory/custom) và tăng/giảm số replica tự động — cần metrics-server chạy. Ví dụ CPU trung bình > 70% → thêm pod tới maxReplicas. Cấu hình behavior để scale up nhanh (bám spike) nhưng scale down chậm (tránh giật — flapping). Manifest đầy đủ ở 3.4.

3.1 Dockerfile multi-stage + non-root CORE ★

# ───── Stage 1: build — chứa Maven + JDK, sẽ bị bỏ đi ─────
FROM maven:3.9-eclipse-temurin-21 AS builder
WORKDIR /build
COPY pom.xml .
RUN mvn dependency:go-offline          # WHY: cache layer dependency (ít đổi hơn source)
COPY src ./src
RUN mvn clean package -DskipTests

# ───── Stage 2: runtime — chỉ JRE + JAR → image 1.2GB còn ~250MB ─────
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app

# WHY: chạy non-root — container escape khi chạy root = chiếm root host
RUN addgroup -S shop && adduser -S appuser -G shop
USER appuser

# WHY: chỉ copy JAR từ builder; toàn bộ Maven/source bị bỏ → image nhỏ, ít attack surface
COPY --from=builder /build/target/*.jar app.jar

# WHY exec form (mảng) chứ không shell form: PID 1 nhận được SIGTERM → graceful shutdown
# WHY UseContainerSupport + MaxRAMPercentage: JVM tôn trọng memory LIMIT của container,
#     không nhìn RAM của cả node (nếu không, JVM xin quá limit → OOMKilled)
ENTRYPOINT ["java", "-XX:+UseContainerSupport", "-XX:MaxRAMPercentage=75", "-jar", "app.jar"]

EXPOSE 8080
HEALTHCHECK --interval=10s --timeout=3s --start-period=30s --retries=3 \
  CMD wget -qO- http://localhost:8080/actuator/health || exit 1
LABEL service="checkout-api" version="1.2.3"
Vì sao multi-stage quan trọng

Image nhỏ = pull nhanh (giây thay vì phút khi scale/rollout), startup nhanh, bề mặt tấn công nhỏ (ít package = ít CVE). Tách layer dependency (ít đổi) khỏi layer source (đổi luôn) tận dụng Docker cache → build nhanh hơn nhiều lần.

3.2 Deployment manifest đầy đủ CORE ★

apiVersion: apps/v1
kind: Deployment
metadata:
  name: checkout-api
  namespace: shop
  labels: { app: checkout-api, team: checkout }
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate: { maxSurge: 1, maxUnavailable: 0 }   # zero-downtime
  selector:
    matchLabels: { app: checkout-api }
  template:
    metadata:
      labels: { app: checkout-api }
      annotations:                                       # cho Prometheus scrape
        prometheus.io/scrape: "true"
        prometheus.io/port: "8080"
        prometheus.io/path: "/actuator/prometheus"
    spec:
      terminationGracePeriodSeconds: 30                  # cửa sổ graceful shutdown
      containers:
      - name: checkout-api
        image: shopflow/checkout-api:v1.2.3
        imagePullPolicy: IfNotPresent
        ports:
        - { name: http, containerPort: 8080 }
        env:
        - name: SPRING_PROFILES_ACTIVE                   # config không nhạy cảm ← ConfigMap
          valueFrom: { configMapKeyRef: { name: checkout-config, key: environment } }
        - name: LOGGING_LEVEL_ROOT
          valueFrom: { configMapKeyRef: { name: checkout-config, key: log.level } }
        - name: DB_PASSWORD                              # nhạy cảm ← Secret
          valueFrom: { secretKeyRef: { name: checkout-secrets, key: db-password } }
        - name: PAYMENT_API_KEY
          valueFrom: { secretKeyRef: { name: checkout-secrets, key: payment-api-key } }
        resources:
          requests: { memory: "512Mi", cpu: "250m" }     # đảm bảo cho scheduler
          limits:   { memory: "1Gi",  cpu: "1000m" }     # trần cứng (mem vượt → OOMKill)
        livenessProbe:                                   # CHỈ nội bộ — không check DB
          httpGet: { path: /actuator/health/liveness, port: http }
          initialDelaySeconds: 60
          periodSeconds: 10
          timeoutSeconds: 3
          failureThreshold: 3
        readinessProbe:                                  # check downstream sẵn sàng
          httpGet: { path: /actuator/health/readiness, port: http }
          initialDelaySeconds: 30
          periodSeconds: 5
          timeoutSeconds: 2
          failureThreshold: 2
        startupProbe:                                    # app khởi động chậm
          httpGet: { path: /actuator/health/liveness, port: http }
          failureThreshold: 30                           # 30 × 10s = tối đa 5 phút khởi động
          periodSeconds: 10
        lifecycle:
          preStop:
            exec: { command: ["sh", "-c", "sleep 5"] }   # cho LB kịp drain trước khi SIGTERM
      affinity:                                          # rải pod ra nhiều node (HA)
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 100
            podAffinityTerm:
              labelSelector: { matchLabels: { app: checkout-api } }
              topologyKey: kubernetes.io/hostname

3.4 HPA · PodDisruptionBudget SUPPLEMENTARY

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata: { name: checkout-api-hpa, namespace: shop }
spec:
  scaleTargetRef: { apiVersion: apps/v1, kind: Deployment, name: checkout-api }
  minReplicas: 3                         # tối thiểu HA
  maxReplicas: 10                        # chặn chi phí leo thang
  metrics:
  - type: Resource
    resource: { name: cpu, target: { type: Utilization, averageUtilization: 70 } }
  - type: Resource
    resource: { name: memory, target: { type: Utilization, averageUtilization: 80 } }
  behavior:
    scaleUp:   { stabilizationWindowSeconds: 30 }    # bám spike flash sale nhanh
    scaleDown: { stabilizationWindowSeconds: 300 }   # giảm chậm, tránh flapping
---
apiVersion: policy/v1
kind: PodDisruptionBudget               # giữ HA khi drain node (bảo trì, upgrade)
metadata: { name: checkout-api-pdb, namespace: shop }
spec:
  minAvailable: 2                        # luôn giữ ≥2 pod checkout sống khi voluntary disruption
  selector: { matchLabels: { app: checkout-api } }

3.5 NetworkPolicy — micro-segmentation ADVANCED

Mặc định K8s cho mọi pod gọi mọi pod. Một pod bị chiếm = lateral movement khắp cluster. NetworkPolicy = firewall mức pod (whitelist): chỉ cho traffic cần thiết.

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: checkout-api-netpol, namespace: shop }
spec:
  podSelector: { matchLabels: { app: checkout-api } }
  policyTypes: [Ingress, Egress]
  ingress:                                       # CHỈ api-gateway được gọi vào checkout
  - from: [{ podSelector: { matchLabels: { app: api-gateway } } }]
    ports: [{ port: 8080 }]
  egress:                                        # checkout CHỈ gọi ra DB + notification + DNS
  - to: [{ podSelector: { matchLabels: { app: postgres-checkout } } }]
    ports: [{ port: 5432 }]
  - to: [{ podSelector: { matchLabels: { app: notification-service } } }]
    ports: [{ port: 8080 }]
  - to: [{ namespaceSelector: {}, podSelector: { matchLabels: { k8s-app: kube-dns } } }]
    ports: [{ port: 53, protocol: UDP }]         # DNS cho service discovery

3.6 ResourceQuota · LimitRange · QoS ADVANCED

ResourceQuota chặn một team/namespace nuốt cả cluster. LimitRange đặt default cho pod quên set request/limit (tránh pod BestEffort lọt vào).

apiVersion: v1
kind: ResourceQuota
metadata: { name: checkout-quota, namespace: shop }
spec:
  hard:
    requests.cpu: "20"          # team checkout tối đa 20 core (request)
    requests.memory: "40Gi"
    limits.cpu: "40"
    limits.memory: "80Gi"
    pods: "100"
---
apiVersion: v1
kind: LimitRange
metadata: { name: checkout-defaults, namespace: shop }
spec:
  limits:
  - type: Container
    default:        { memory: "512Mi", cpu: "500m" }   # limit mặc định nếu pod quên set
    defaultRequest: { memory: "256Mi", cpu: "100m" }   # → không pod nào rơi vào BestEffort
Quy tắc QoS cho ShopFlow:
  checkout, order, inventory  → Guaranteed (requests == limits) — không bao giờ evict trước
  reporting, notification     → Burstable OK
  job thử nghiệm/throwaway    → KHÔNG để BestEffort cho service hứng traffic thật

kubectl describe pod checkout-api-xxx -n shop | grep -A5 "Last State"

5. 3-tier interview framework

Tầng 1 — SurfaceJunior gate

Q: Liveness vs readiness probe khác gì?

Liveness = "container còn sống?" → fail thì kill+restart. Readiness = "nhận traffic được chưa?" → fail thì rút khỏi Service endpoints (không restart). Không check downstream (DB) trong liveness vì DB chậm sẽ giết oan pod khoẻ; check downstream thuộc readiness.

Tầng 2 — Deep DiveSenior gate

Q: Thiết kế zero-downtime deploy cho checkout service đang xử lý request dở (in-flight).

Rolling update maxUnavailable: 0, maxSurge: 1; bật graceful shutdown ở app (server.shutdown=graceful, bắt SIGTERM ngừng nhận request mới, xử lý nốt in-flight); readiness probe chính xác; preStop sleep 5 cho LB kịp drain (xoá endpoint có độ trễ); terminationGracePeriodSeconds ≥ thời gian request dài nhất; PodDisruptionBudget minAvailable giữ tối thiểu pod khi drain node.

Tầng 3 — ArchitectureSenior+

Q: 5000 pod, tổng memory request 10TB nhưng ngân sách chỉ 5TB. Tối ưu chi phí.

Request đang over-provision = lãng phí. Load test để biết P95/P99 memory thật → request = P95, limit = P99.5 (bắt runaway nhưng cho phép spike bình thường). Kết hợp HPA (horizontal) + cluster autoscaler + cân nhắc VPA. Service phi-critical dùng Burstable + spot/preemptible instance; service quan trọng giữ Guaranteed. Đặt LimitRange + ResourceQuota để team không over-request. Theo dõi utilization thực và right-size định kỳ.

🔧 Lab bắt buộc — reproduce & fix (2 bài)

Lab 1 — Gây & sửa OOMKilled. Deploy một Spring Boot app với limits.memory: 256Mi nhưng app cấp phát ~400MB lúc khởi động. Quan sát pod vào CrashLoopBackOff; kubectl describe thấy Reason: OOMKilled, Exit Code: 137. Sửa theo 2 hướng: (a) tăng limit hợp lý sau khi đo, (b) thêm -XX:+UseContainerSupport -XX:MaxRAMPercentage=75 để JVM tôn trọng container. Tiêu chí: giải thích vì sao thiếu UseContainerSupport thì JVM nhìn RAM cả node và xin quá limit.

Lab 2 — Liveness sai → restart bão. Cấu hình liveness probe gọi một endpoint check DB; tắt DB; quan sát toàn bộ pod bị kill+restart liên tục dù app vẫn "sống". Chuyển check DB sang readiness, bật lại; quan sát pod bị rút khỏi Service (ngừng nhận traffic) nhưng KHÔNG restart, và tự Ready lại khi DB lên. Tiêu chí: nói được "fail readiness ≠ fail liveness" và hậu quả mỗi loại.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. Container khác VM ở chỗ nào? Vì sao container khởi động nhanh hơn nhưng cô lập yếu hơn?
