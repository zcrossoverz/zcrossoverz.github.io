AWS cho Java Backend — Chọn Đúng Dịch Vụ, Đúng Chi Phí

AWS có hàng trăm dịch vụ; hiểu giả là "biết tên" mà không biết khi nào dùng cái nào và chi phí ẩn ở đâu. "EC2 vs Lambda vs Fargate khác gì?", "vì sao Lambda timeout 15s?", "DynamoDB khi nào thay RDS?", "vì sao bill S3 tăng vì request chứ không phải dung lượng?" — là các câu phân loại junior/senior. ⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Lambda rẻ và tự scale nên dùng cho mọi thứ."
Thực tế: Lambda có timeout (tối đa 15 phút), cold start, giới hạn payload, và đắt khi chạy liên tục tải cao. Hợp cho event-driven/burst ngắn; không hợp cho job dài, kết nối DB lâu, hay tải cao ổn định (lúc đó ECS/EC2 rẻ hơn).
Tưởng hiểu: "Bill S3 cao là do lưu nhiều dữ liệu."
Thực tế: S3 tính cả số request (GET/PUT) và data transfer ra ngoài, không chỉ dung lượng. App gọi S3 hàng triệu lần/ngày hoặc serve ảnh trực tiếp (không CDN) → bill request/egress vượt cả bill storage.
Tưởng hiểu: "DynamoDB là NoSQL nên linh hoạt, query kiểu gì cũng được."
Thực tế: DynamoDB bắt bạn thiết kế theo access pattern trước (partition key + sort key). Query không theo key = scan toàn bảng (chậm, đắt). Nó nhanh & scale vô hạn NẾU thiết kế đúng pattern; "linh hoạt như SQL" là hiểu sai.

🗺️ Roadmap — nếu chỉ có X giờ thì học gì

Thứ tự ưu tiên
1.5 giờ (CORE ★): Compute EC2/Lambda/ECS/EKS — khi nào cái nào (3), DB RDS vs DynamoDB (5), IAM least-privilege (2). Đây là quyết định kiến trúc + bảo mật cốt lõi.
+1 giờ (CORE ★): S3 + chi phí request/egress + CDN (4), messaging SQS/SNS (6).
+1 giờ (SUPPLEMENTARY): VPC/subnet/security group, ElastiCache, EventBridge, ALB.
Khi cần (ADVANCED): cost optimization (spot, savings plan), multi-AZ/region, EKS chi tiết, DynamoDB single-table design.

1. Bài toán thực tế — bill 3×, Lambda timeout, service không gọi nhau CORE ★

ShopFlow trên AWS gặp 3 việc: (1) bill tháng tăng 3× không rõ vì sao; (2) một Lambda xử lý export báo cáo timeout sau 15s; (3) service A trong VPC không gọi được service B (connection refused).

⏸ Pause & Predict — trước khi đọc lời giải
Viết ra giấy:
  1. Bill tăng 3× — liệt kê 3 chi phí ẩn thường gặp (gợi ý: không phải dung lượng/CPU).
  2. Lambda timeout export báo cáo — vì sao Lambda sai công cụ cho việc này? Dùng gì thay?
  3. Service A không gọi được B trong VPC — thường do cấu hình mạng nào (gợi ý: 2 lớp firewall)?
Đáp án: chi phí ẩn = data egress, S3/API request count, NAT gateway, idle resource (mục 4, anti-patterns); Lambda timeout 15 phút max + không hợp job dài → ECS/Batch (mục 3); A→B chặn bởi security group / NACL / subnet routing (mục 2). Đừng đọc tiếp khi chưa nêu 3 chi phí ẩn.

2. IAM + VPC CORE ★

IAM = ai (principal) được làm gì (action) trên tài nguyên nào (resource), theo policy. Nguyên tắc: least privilege — cấp đúng quyền tối thiểu; dùng role cho service (không hardcode access key). VPC = mạng riêng: subnet public (có internet) / private (qua NAT), security group (firewall mức instance, stateful) + NACL (mức subnet, stateless).

Anti-pattern + symptom

Triệu chứng: service A gọi B "connection timed out". Nguyên nhân thường gặp: security group của B không cho phép inbound từ SG của A, hoặc B ở private subnet không có route. Fix: mở SG inbound theo SG nguồn (không mở 0.0.0.0/0), kiểm route table.

3. Compute: EC2 / Lambda / ECS / EKS CORE ★

L1 — Biết tên

Biết 4 cái tồn tại. Symptom hiểu giả: không nói được khi nào chọn cái nào hay vì sao Lambda không hợp job dài.

L2 — Explain được

EC2 = VM, toàn quyền, tự quản. Lambda = serverless, trả theo lần gọi, hợp event/burst ngắn nhưng có cold start + timeout 15 phút. ECS Fargate = container không cần quản node. EKS = Kubernetes managed (khi cần hệ sinh thái K8s).

L3 — Apply được

Chọn: API tải ổn định → ECS Fargate/EC2; xử lý sự kiện ngắn (resize ảnh, webhook) → Lambda; job dài (export, ETL) → ECS task/Batch (không Lambda); đã chuẩn K8s đa cloud → EKS. Cân nhắc cost: Lambda đắt khi chạy liên tục tải cao.

4. Storage: S3 / EBS / EFS SUPPLEMENTARY

Dịch vụLoạiDùng cho
S3Object storageẢnh sản phẩm, file, backup, static asset (serve qua CloudFront)
EBSBlock storage gắn 1 EC2Ổ đĩa cho instance (DB tự quản, OS)
EFSFile system chia sẻ nhiều instanceKhi nhiều pod/instance cần cùng filesystem

Chi phí S3: dung lượng + số request + data egress. Serve ảnh trực tiếp từ S3 ra internet → egress đắt; đặt CloudFront (CDN) trước → cache, giảm egress & request. Dùng storage class (S3-IA, Glacier) cho data ít truy cập.

5. DB: RDS / DynamoDB / ElastiCache CORE ★

  • RDS (Postgres/MySQL managed): khi cần SQL, quan hệ, query linh hoạt, transaction. Multi-AZ cho HA.
  • DynamoDB: NoSQL key-value, scale vô hạn, latency thấp ổn định — NHƯNG phải thiết kế theo access pattern (partition/sort key). Hợp cart, session, sự kiện nhiều ghi; không hợp query ad-hoc.
  • ElastiCache (Redis): cache, session, rate limit, leaderboard.
🔧 Counter-example — khi DynamoDB là lựa chọn SAI
  • Query đa chiều, ad-hoc: báo cáo "đơn theo nhiều bộ lọc tuỳ ý" → DynamoDB phải scan hoặc tạo nhiều GSI tốn kém. RDS với index linh hoạt hơn.
  • Transaction phức tạp nhiều bảng: DynamoDB transaction hạn chế; quan hệ nhiều thực thể → RDS.
  • Team chưa quen single-table design: mô hình sai từ đầu rất đắt để sửa (phải migrate).

Quy tắc: DynamoDB tỏa sáng khi access pattern biết trước & ổn định + cần scale cực lớn. Khi pattern còn đổi hoặc cần query linh hoạt → RDS. Chọn theo access pattern, không theo "NoSQL nghe hiện đại".

6. Messaging: SQS / SNS / EventBridge SUPPLEMENTARY

  • SQS: queue point-to-point, một consumer xử lý mỗi message; decouple + buffer + retry (DLQ).
  • SNS: pub/sub fan-out, một message tới nhiều subscriber.
  • EventBridge: event bus với routing theo rule (tích hợp SaaS, schema).
  • MSK: Kafka managed (khi cần replay, ordering, throughput Kafka).

Pattern phổ biến: SNS → nhiều SQS (fan-out + buffer riêng cho từng consumer).

🎯 3-tier interview framework

Tầng 1 — SurfaceJunior gate

Q: EC2 vs Lambda khác gì?

EC2 = VM chạy liên tục, tự quản, hợp tải ổn định. Lambda = serverless, trả theo lần gọi, tự scale, hợp event/burst ngắn nhưng có cold start + timeout 15 phút, không hợp job dài/tải cao liên tục.

Tầng 2 — Deep DiveSenior gate

Q: Bill AWS tăng 3×, điều tra thế nào?

Cost Explorer phân tích theo service/tag → tìm thủ phạm: data egress, S3/API request count, NAT gateway, idle EC2/RDS, Lambda chạy liên tục. Fix: CDN giảm egress, tắt idle, right-size, savings plan, cost alert + tag bắt buộc.

Tầng 3 — ArchitectureSenior+

Q: Thiết kế kiến trúc e-commerce AWS chịu tải, tối ưu chi phí.

ALB → ECS Fargate (auto-scale) cho API; RDS Multi-AZ + read replica cho đơn; DynamoDB cho cart/session; ElastiCache cho cache; S3 + CloudFront cho ảnh; SQS/SNS decouple; Lambda cho event ngắn (resize ảnh). IAM role least-privilege, VPC private subnet, cost alert + spot cho phi-critical.

🔧 Lab bắt buộc — quyết định & đo (2 bài)

Lab 1 — Lambda vs ECS cho 2 việc. Lấy 2 task: (a) resize ảnh khi upload (sự kiện ngắn), (b) export 1 triệu đơn ra CSV (chạy lâu). Lý giải bằng giấy: cái nào Lambda, cái nào ECS/Batch, dựa trên timeout/cost/cold start. Tiêu chí: nêu đúng giới hạn 15 phút và chi phí chạy liên tục.

Lab 2 — DynamoDB access pattern. Cho yêu cầu: "lấy đơn theo customerId, sắp theo ngày" và "lấy đơn theo orderId". Thiết kế partition key/sort key + GSI để cả hai query không scan. Thêm yêu cầu mới "lọc đơn theo status bất kỳ" → chỉ ra vì sao cần GSI hoặc nên dùng RDS. Tiêu chí: giải thích access-pattern-first.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. EC2 / Lambda / ECS / EKS — khi nào dùng cái nào?
  2. Bill S3 cao có thể do gì ngoài dung lượng? CDN giúp thế nào?
  3. DynamoDB khác RDS ở triết lý thiết kế nào? Khi nào DynamoDB là sai?
  4. Service A không gọi được B trong VPC — kiểm tra gì?
  5. Vì sao dùng IAM role thay vì hardcode access key?
Bí câu nào = quay lại đúng section. Trôi cả 5 = ra quyết định AWS theo trade-off, không cargo-cult.
