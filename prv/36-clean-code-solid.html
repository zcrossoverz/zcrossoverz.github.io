Clean Code & SOLID — Smell, Refactor, và Khi Nào Đừng

SOLID là thứ ai cũng đọc thuộc 5 chữ cái nhưng ít người dùng đúng — và over-apply SOLID còn hại hơn không biết nó. File này dạy mỗi nguyên tắc qua WHY (vấn đề gì sinh ra nó) + code FAIL/SUCCESS, kèm một refactoring walkthrough thật (method 200 dòng → clean), và counter-example (khi nào áp dụng là over-engineering), trên domain ShopFlow — OrderPricingService tính giá/khuyến mãi của sàn e-commerce.

🎯 Mục tiêu
Nhận diện smell, refactor có kiểm soát, biết khi nào KHÔNG abstract
📅 Thời lượng
1 tuần · Sau #16 Design Patterns, #15 Clean Arch
📚 Prereq
OOP · interface/polymorphism · unit test
🔥 Độ ưu tiên
🟠 Cao — code review & design hỏi mỗi vòng senior

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Thuộc 5 chữ S-O-L-I-D và định nghĩa mỗi cái là hiểu SOLID."
Thực tế: Hiểu thật = nhìn một đoạn code chỉ ra nó vi phạm nguyên tắc nào và hậu quả cụ thể khi thêm feature. Đọc định nghĩa mà không nói được "vi phạm O thì sao" = hiểu giả.
Tưởng hiểu: "Càng nhiều interface, abstraction, design pattern thì code càng SOLID, càng tốt."
Thực tế: Abstraction sớm/thừa là code smell ngược: 1 implementation mà có interface + factory + strategy = đọc 5 file để hiểu 1 việc. SOLID phục vụ thay đổi; chỗ không đổi thì abstraction chỉ thêm nhiễu.
Tưởng hiểu: "Refactor là dọn code cho đẹp, làm lúc nào cũng được."
Thực tế: Refactor KHÔNG đổi hành vi — nên bắt buộc có test phủ trước khi đổi. Refactor không test = viết lại + cầu may. Đây là khác biệt giữa refactoring và "rewrite".

🗺️ Roadmap — nếu chỉ có X giờ thì học gì

Thứ tự ưu tiên
1 giờ (CORE ★): SRP + OCP (2.2, 2.3) — hai nguyên tắc dùng nhiều nhất; nhận diện 7 code smell (2.1). Đủ để pass phần "tìm lỗi SOLID" của interview.
+1 giờ (CORE ★): Refactor messy → clean theo bước có test (3), Tell-Don't-Ask (2.8), khi nào KHÔNG áp dụng SOLID (4).
+1 giờ (SUPPLEMENTARY): LSP, ISP, DIP với code (2.4–2.6); Extract Method/Class/Parameter Object (2.7).
Khi cần (ADVANCED): Replace conditional with polymorphism, đánh giá ROI abstraction theo "rule of three".

1. Bài toán thực tế — PR tính giá 200 dòng CORE ★

Bạn review một PR: method calculateOrderTotal() dài 200 dòng, nhận 7 tham số, có magic number 0.05, 0.1 rải khắp, trộn validate + tính phí + ghi DB + gửi email trong một chỗ. Mỗi lần thêm một loại khuyến mãi phải sửa đúng method này và làm vỡ 3 chỗ khác.

⏸ Pause & Predict — trước khi đọc lời giải
Nhìn mô tả PR trên, viết ra giấy:
  1. Liệt kê đúng tên ít nhất 4 code smell trong đoạn này (không nói chung "code xấu").
  2. "Thêm 1 loại khuyến mãi phải sửa method này" — đây vi phạm nguyên tắc SOLID nào? Hậu quả khi có 10 loại khuyến mãi?
  3. Nếu refactor, bước ĐẦU TIÊN phải làm là gì trước khi đụng vào code (gợi ý: không phải tách method)?
Đáp án: Long Method, Long Parameter List, Magic Number, Mixed Concerns (2.1); vi phạm OCP (sửa thay vì mở rộng) + SRP (2.2, 2.3); bước đầu là viết test phủ hành vi hiện tại (4 — refactor không test). Đừng đọc tiếp khi chưa đặt tên được 4 smell.

4 câu thinking

  1. Code smell nào đang có và mỗi cái gây hại gì khi maintain?
  2. Vi phạm nguyên tắc SOLID nào? Thêm feature thì vỡ ở đâu?
  3. Refactor theo bước nào để KHÔNG đổi hành vi?
  4. Khi nào dừng — abstraction tới đâu là đủ, đâu là over-engineering?

2.2 S — Single Responsibility CORE ★

WHY: một class chỉ nên có một lý do để thay đổi. OrderPricingService vừa tính giá vừa gửi email có 2 lý do đổi: rule giá (Finance) và template email (Marketing) → Marketing sửa email làm vỡ logic tính tiền.

❌ Fail — 2 lý do đổi trong 1 class

class OrderPricingService {
  BigDecimal price(Order o) { /* tính giá */ }
  void emailReceipt(Order o) { /* template email */ }
  void saveHistory(Order o) { /* ghi DB */ }
}

✅ Success — mỗi class 1 trách nhiệm

class OrderPricingService { BigDecimal price(Order o){...} }
class ReceiptNotifier     { void email(Order o){...} }
class OrderHistoryStore   { void save(Order o){...} }

2.3 O — Open/Closed CORE ★

WHY: mở để mở rộng, đóng để sửa đổi. Thêm loại khuyến mãi không nên sửa chuỗi if/else khổng lồ (rủi ro vỡ cái cũ) mà thêm một implementation mới.

❌ Fail — mỗi loại mới sửa method này

BigDecimal discount(Order o) {
  if (o.type()==PERCENT)  return o.total().multiply(RATE);
  if (o.type()==FIXED)    return o.fixedOff();
  if (o.type()==BLACKFRI) return o.total().multiply(BF_RATE);
  // type mới → SỬA method này → rủi ro vỡ cũ
  return ZERO;
}

✅ Success — thêm rule = thêm class

interface DiscountRule { BigDecimal apply(Order o); }
class PercentRule    implements DiscountRule {...}
class FixedRule      implements DiscountRule {...}
class BlackFridayRule implements DiscountRule {...}  // THÊM, không sửa

class Pricing {
  private final List<DiscountRule> rules;            // inject
  BigDecimal discount(Order o){
    return rules.stream().map(r->r.apply(o))
                .reduce(ZERO, BigDecimal::add);
  }
}

2.4 L — Liskov Substitution SUPPLEMENTARY

WHY: subclass phải thay được superclass mà không phá kỳ vọng của client.

❌ Fail — subclass siết precondition

class Voucher { void redeem(){...} }
class SeasonalVoucher extends Voucher {
  @Override void redeem(){
    if(!isSeason()) throw new IllegalStateException(); // client vỡ
    super.redeem();
  }
}
// Voucher v = new SeasonalVoucher();
// v.redeem();  // ném exception — ngoài kỳ vọng "redeem được mọi lúc"

✅ Success — giữ hợp đồng, đổi hành vi hợp lệ

class SeasonalVoucher extends Voucher {
  @Override void redeem(){
    if(!isSeason()) applyOffSeasonAdjustment();  // vẫn redeem được
    super.redeem();
  }
}

Dấu hiệu vi phạm LSP: subclass ném exception cho method base cho phép, hoặc trả giá trị ngoài hợp đồng base. Nếu phải instanceof để xử lý subclass đặc biệt → thường đã vi phạm LSP.

2.5 I — Interface Segregation SUPPLEMENTARY

WHY: đừng ép client phụ thuộc method nó không dùng (fat interface). Interface 30 method khiến test phải mock cả 30 và implementation phải viết stub thừa.

❌ Fat interface

interface OrderGateway {
  Order place(...); List<Order> history(...);
  Report monthlyReport(...); void notify(...);
  void audit(...);                       // mọi impl phải làm hết
}

✅ Tách theo nhu cầu

interface OrderPlacer  { Order place(...); }
interface OrderReader  { List<Order> history(...); }
interface Reporter     { Report monthly(...); }
// client chỉ depend cái cần

2.6 D — Dependency Inversion CORE ★

WHY: module cấp cao không phụ thuộc cấp thấp; cả hai phụ thuộc abstraction. Nhờ vậy đổi implementation (DB, provider) hoặc inject mock khi test mà không sửa logic.

❌ Phụ thuộc class cụ thể

class PricingService {
  private JpaOrderRepository repo;     // concrete!
  // đổi sang Mongo = phải sửa PricingService
}

✅ Phụ thuộc abstraction

interface OrderRepository {
  Order findById(long id); void save(Order o);
}
class PricingService {
  private final OrderRepository repo;  // abstraction
  PricingService(OrderRepository repo){ this.repo = repo; }
  // test: inject mock; prod: Jpa; tương lai: Mongo — không sửa logic
}

2.7 Refactoring techniques SUPPLEMENTARY

  • Extract Method: tách một khối thành method có tên rõ → method gốc đọc như mục lục.
  • Extract Class: nhóm field/method liên quan ra class riêng (tách God Class).
  • Replace Magic Number with Constant: 0.08 → VAT_RATE.
  • Replace Conditional with Polymorphism: switch(type) dài → mỗi nhánh thành một subclass/strategy (chính là OCP).
  • Introduce Parameter Object: 7 tham số → một PricingRequest.

2.8 Tell-Don't-Ask SUPPLEMENTARY

❌ Ask — lôi data ra ngoài xử lý

if (order.getStatus() == PAID
    && order.getItems().size() > 0) {
    order.setStatus(SHIPPED);
}  // invariant rải ra ngoài → dễ quên ở nơi khác

✅ Tell — giao việc cho object giữ data

order.markShipped();
// invariant (đã PAID, có item) nằm TRONG Order
// → không thể quên check ở nơi khác

3. Refactor messy → clean (có code, có test) CORE ★

❌ Trước — Long Method + Mixed Concerns + Magic Number

BigDecimal calculateOrderTotal(String customerId, List<Item> items,
    String couponCode, boolean isVip, String channel,
    LocalDate date, String currency) {
  // validate
  if (items == null || items.isEmpty()) throw ...;
  if (customerId == null) throw ...;
  // tính subtotal
  BigDecimal sub = ZERO;
  for (Item i : items) sub = sub.add(i.price().multiply(qty(i)));
  // khuyến mãi (magic numbers, if-else)
  BigDecimal disc = ZERO;
  if (couponCode != null) disc = sub.multiply(new BigDecimal("0.1"));
  if (isVip) disc = disc.add(sub.multiply(new BigDecimal("0.05")));
  // thuế
  BigDecimal tax = sub.subtract(disc).multiply(new BigDecimal("0.08"));
  BigDecimal total = sub.subtract(disc).add(tax);
  // side-effect: ghi DB + gửi mail ngay trong hàm tính
  orderRepo.save(...);
  emailService.send(customerId, "Total: " + total);
  return total;
}

✅ Sau — tách trách nhiệm, OCP cho discount

// B1 (trước khi sửa): viết characterization test phủ hành vi hiện tại
// B2: Parameter Object
record PricingRequest(String customerId, List<Item> items,
    String coupon, boolean vip, String channel,
    LocalDate date, String currency) {}

// B3: Strategy cho discount (OCP)
interface DiscountRule { BigDecimal apply(PricingRequest r, BigDecimal sub); }

// B4: tách validation (SRP)
class OrderValidator { void validate(PricingRequest r){...} }

// B5: hàm tính THUẦN, không side-effect
class PricingCalculator {
  private final List<DiscountRule> rules;
  private static final BigDecimal VAT = new BigDecimal("0.08");
  Money total(PricingRequest r){
    var sub = r.items().stream().map(Item::lineTotal)
                 .reduce(ZERO, BigDecimal::add);
    var disc = rules.stream().map(x->x.apply(r, sub))
                 .reduce(ZERO, BigDecimal::add);
    var taxed = sub.subtract(disc);
    return new Money(taxed.add(taxed.multiply(VAT)));
  }
}

// B6: Facade orchestrator — nối các phần, side-effect tách riêng
class PlaceOrderUseCase {
  Order place(PricingRequest r){
    validator.validate(r);
    Money total = calculator.total(r);
    Order o = orderRepo.save(Order.of(r, total));
    events.publish(new OrderPlaced(o.id()));   // notify async, không trong hàm tính
    return o;
  }
}
Nguyên tắc vàng

Mỗi bước nhỏ + chạy test ngay sau mỗi bước. Test xanh sau mỗi commit = bạn luôn có điểm quay lui. Đây là khác biệt giữa "refactor" (an toàn, không đổi hành vi) và "rewrite" (rủi ro).

4. Khi SOLID phản tác dụng CORE ★

🔧 Counter-example — khi nào KHÔNG áp dụng SOLID/abstraction

Over-engineering hại không kém dưới-engineering:

  • Premature abstraction: tạo interface + factory cho cái chỉ có một implementation và chưa có dấu hiệu sẽ có cái thứ hai. Theo "rule of three": chờ đến lần thứ ba lặp lại mới abstract.
  • SOLID over-engineering: một CRUD đơn giản mà có 6 lớp (controller → service → usecase → gateway → mapper → repository) cho một findById = đọc 6 file để hiểu 1 việc.
  • Tách SRP quá mịn: mỗi class 5 dòng, 40 class cho một flow → "shotgun surgery" ngược: thêm 1 field phải đụng 6 class.
  • Refactor không test: đổi hành vi âm thầm → bug. Characterization test trước.

Quy tắc: abstraction phục vụ thay đổi đã thấy hoặc rất khả năng xảy ra. Không có trục thay đổi → YAGNI. Senior thể hiện ở chỗ biết dừng, kèm lý do.

Anti-patternHậu quảCách đúng
Premature abstraction5 file cho 1 việc, khó đọcRule of three; abstract khi có trục đổi
Refactor không testĐổi hành vi âm thầm → bugCharacterization test trước
God classMerge conflict, không ai dám sửaExtract class theo trách nhiệm
Feature envyLogic đặt sai chỗMove method tới class giữ data

5. 3-tier interview framework

Tầng 1 — SurfaceMid gate

Q: Cho đoạn code, chỉ ra vi phạm SOLID.

Nói tên nguyên tắc + bằng chứng cụ thể: "method này thêm loại giảm giá phải sửa chuỗi if-else → vi phạm OCP", "class này vừa tính tiền vừa gửi mail → vi phạm SRP". Không nói chung "code xấu".

Tầng 2 — Deep DiveSenior gate

Q: Refactor method 200 dòng này từng bước.

Characterization test phủ trước → Parameter Object → Strategy cho discount (OCP) → tách validation (SRP) → tách side-effect (DB/email ra khỏi hàm tính) → Facade orchestrator. Chạy test sau mỗi bước. Nhấn mạnh "không đổi hành vi" = refactor, không phải rewrite.

Tầng 3 — ArchitectureSenior+

Q: Khi nào KHÔNG nên áp dụng SOLID?

Khi không có trục thay đổi (YAGNI), code đơn giản/throwaway, 1 implementation duy nhất, hoặc abstraction làm tăng chi phí đọc > lợi ích. Rule of three. Senior cân bằng giữa flexibility và simplicity, không cargo-cult pattern.

🔧 Lab bắt buộc — refactor có kiểm soát (2 bài)

Lab 1 — OCP qua Strategy. Lấy một method tính khuyến mãi với if(type=="PERCENT")...else if(type=="FIXED").... Viết characterization test phủ các nhánh. Refactor sang DiscountRule interface + các implementation, inject list. Thêm một rule mới mà KHÔNG sửa code cũ. Tiêu chí: chứng minh test cũ vẫn xanh và rule mới chỉ thêm file.

Lab 2 — Nhận diện over-engineering. Lấy một đoạn code "SOLID quá đà" (interface + factory cho 1 implementation). Viết lại đơn giản hơn (xoá abstraction thừa) mà vẫn pass test. Tiêu chí: giải thích "rule of three" và vì sao bản đơn giản dễ đọc hơn ở quy mô hiện tại.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. SRP nói "một lý do để thay đổi" nghĩa là gì? Cho ví dụ một class vi phạm.
  2. OCP: "mở để mở rộng, đóng để sửa đổi" — refactor if-else thành cái gì để đạt OCP?
  3. Tại sao phải viết test TRƯỚC khi refactor? Khác gì với rewrite?
  4. Cho 2 ví dụ over-engineering SOLID và vì sao chúng hại.
  5. Tell-Don't-Ask là gì? Cho một cặp code Ask vs Tell.
Bí câu nào = quay lại đúng section. Trôi cả 5 = bạn vừa biết áp dụng vừa biết khi nào dừng — đúng tầm senior.
