Application Security — OWASP, Injection, CSRF, IDOR

Security là nơi hiểu giả dễ chết người nhất: "có dùng prepared statement", "có HTTPS" nghe đủ — nhưng "vì sao parameterized query chặn injection ở tầng nào?", "CSRF cần token cho form nhưng KHÔNG cho JWT API — vì sao?", "IDOR khác broken auth chỗ nào?" thì lộ. File này giải thích cơ chế từng lỗ hổng và vì sao cách chặn nằm ở đúng tầng — đọc một lần là hiểu. Domain: ShopFlow — tài khoản & đơn hàng của sàn e-commerce.

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Dùng prepared statement là an toàn SQL injection. Escape input cũng được."
Thực tế: Parameterized query an toàn vì tách câu lệnh khỏi dữ liệu ở tầng protocol — DB nhận query đã được phân tích cú pháp (parse) trước, rồi data đến sau như tham số thuần, không bao giờ được parse như SQL. Tự escape thì luôn sót case (Unicode, encoding lồng). Và nối chuỗi vào tên bảng/cột/ORDER BY (không tham số hoá được) vẫn injectable dù dùng PreparedStatement.
Tưởng hiểu: "CSRF với CORS là một, đều về cross-origin."
Thực tế: Ngược nhau. CSRF lợi dụng việc trình duyệt tự gửi cookie → kẻ tấn công khiến trình duyệt nạn nhân gửi request hợp lệ ngoài ý muốn. CORS là cơ chế trình duyệt nới lỏng/giới hạn việc JS đọc response cross-origin. Bật CORS không chống CSRF; thậm chí cấu hình CORS lỏng còn mở thêm rủi ro.
Tưởng hiểu: "Đã login (authentication) là an toàn truy cập."
Thực tế: Authentication (bạn là ai) ≠ authorization (bạn được làm gì với resource nào). IDOR là lỗ hổng authorization: user A login hợp lệ nhưng đổi orderId trên URL để xem đơn của user B. Login không ngăn được; phải check ownership ở server.

1. Bài toán thực tế — rò rỉ & chiếm đơn CORE ★

ShopFlow gặp 3 sự cố trong một tháng. (1) Một endpoint tìm kiếm ghép trực tiếp ô input người dùng vào câu SQL; kẻ tấn công nhập một chuỗi khéo léo khiến câu lệnh trả về toàn bộ bảng users kèm hash mật khẩu. (2) Một khách quan sát URL /orders?orderId=1002 của mình, đổi thành 1003 và xem được đơn của người khác — không cần hack gì, chỉ đổi số. (3) Form đổi email bị một website lạ submit hộ: khi nạn nhân đang đăng nhập ShopFlow mà ghé trang đó, trình duyệt tự đính kèm session cookie nên ShopFlow tưởng chính chủ yêu cầu, đổi email và kẻ tấn công chiếm tài khoản.

⏸ Pause & Predict — trước khi đọc lời giải
Đặt tên đúng từng lỗ hổng và cách chặn:
  1. Sự cố (1) tên là gì? Vì sao "escape dấu nháy" không đủ mà parameterized query mới đúng?
  2. Sự cố (2) tên là gì? Vì sao "đã login" không ngăn được? Check gì ở server?
  3. Sự cố (3) tên là gì? Vì sao JWT trong header sẽ không bị, nhưng session cookie thì bị?
Đáp án: (1) SQL Injection — parameterized tách lệnh khỏi data (2.2); (2) IDOR — authorization, check ownership server, không tin ID client (2.5); (3) CSRF — cookie tự gửi tạo "ambient authority"; JWT header thì trình duyệt không tự gắn (2.4). Đừng đọc tiếp khi chưa đặt đúng 3 tên.

2.1 OWASP Top 10 — học theo cơ chế SUPPLEMENTARY

OWASP Top 10 là danh sách 10 nhóm rủi ro web phổ biến nhất, cập nhật vài năm một lần. Đừng học thuộc danh sách — học theo cơ chế. Trong backend e-commerce hay gặp nhất: Broken Access Control (gồm IDOR — vượt quyền truy cập), Injection (SQL/NoSQL/command — chèn lệnh qua input), Cryptographic Failures (PII không mã hoá, dùng thuật toán yếu — xem #30), Security Misconfiguration (mặc định không an toàn, lộ stack trace), Identification & Authentication Failures (thiếu rate limit, session yếu), và SSRF (lừa server gọi tới địa chỉ nội bộ). Hiểu cơ chế của vài cái cốt lõi (injection, access control, crypto) thì suy ra được phần còn lại.

2.2 SQL Injection — cơ chế & vì sao parameterized fix CORE ★

SQL Injection xảy ra khi dữ liệu người dùng bị diễn giải như câu lệnh SQL thay vì dữ liệu. Khi bạn nối chuỗi "... WHERE email='" + input + "'", toàn bộ chuỗi (gồm cả input) được gửi tới DB như một câu lệnh và DB parse nó. Nếu input là x' OR '1'='1' --, câu lệnh thành WHERE email='x' OR '1'='1' --': mệnh đề OR '1'='1' luôn đúng nên trả về mọi dòng, còn -- biến phần sau thành comment. Dữ liệu đã "thoát ra" thành cấu trúc lệnh.

Vì sao parameterized query chặn được — ở tầng nào: với "... WHERE email = ?", driver gửi câu lệnh có placeholder tới DB để parse & compile trước, rồi gửi giá trị input riêng như một tham số đã được đánh dấu là "dữ liệu thuần". Lúc này DB đã xác định xong cấu trúc lệnh; giá trị tham số không bao giờ được parse lại như SQL — dù input có chứa ' OR '1'='1 thì nó chỉ được so sánh như một chuỗi literal. Việc tách "code" và "data" này diễn ra ở tầng giao thức DB, nên an toàn theo thiết kế chứ không nhờ "lọc đúng".

❌ Nối chuỗi — injectable

"SELECT * FROM users WHERE email='" + input + "'"
// input = x' OR '1'='1' --  → trả mọi user (1=1 luôn đúng)

✅ Parameterized — tách lệnh khỏi data

jdbc.query("SELECT * FROM users WHERE email = ?", input);
// query parse TRƯỚC; input là DATA thuần, DB không parse nó như SQL
Vẫn injectable dù dùng PreparedStatement

Placeholder ? chỉ thay được cho giá trị, không thay được cho định danh (tên bảng/cột) hay từ khoá. Vì vậy "ORDER BY " + sortColumn hay "... " + tableName vẫn injectable kể cả khi phần còn lại tham số hoá. Cách chặn: whitelist — chỉ chấp nhận sortColumn nằm trong tập cố định (Set.of("created_at","total")). ORM (JPA/Hibernate) mặc định dùng parameterized nên an toàn, nhưng createNativeQuery() + nối chuỗi thì lại hở.

Second-order injection

Dữ liệu có thể "sạch" lúc lưu (đã parameterized khi INSERT) nhưng vẫn nguy hiểm khi lấy ra dùng lại trong một câu SQL khác bằng cách nối chuỗi, hoặc render vào HTML/log mà không encode. Bài học: an toàn phải đảm bảo tại mỗi điểm sử dụng (mỗi câu query, mỗi lần render), không chỉ tại điểm nhập.

2.3 XSS — 3 biến thể + 4 tầng chặn SUPPLEMENTARY

XSS (Cross-Site Scripting) là khi kẻ tấn công khiến script của chúng chạy trong trình duyệt của nạn nhân, trong ngữ cảnh trang web của bạn — từ đó đánh cắp cookie/session, thực hiện hành động thay nạn nhân. Bản chất giống injection nhưng "đích" là trình duyệt thay vì DB: input người dùng bị diễn giải như mã HTML/JS thay vì văn bản. Ba biến thể khác nhau ở nơi script được tiêm vào:

LoạiCơ chế
ReflectedScript nằm trong URL/request và bị server phản chiếu thẳng vào trang trả về (vd /search?q=<script>...</script> rồi hiển thị lại q). Cần lừa nạn nhân bấm link.
StoredScript được lưu vào DB (tên hiển thị, review sản phẩm) rồi hiện cho mọi người xem → nguy hiểm nhất vì chạy với nhiều nạn nhân (vd admin mở danh sách → script chạy trong phiên admin).
DOM-basedLỗ hổng nằm ở JS phía client: code lấy input rồi ghi vào DOM bằng innerHTML (parse HTML). Dùng textContent thì an toàn vì không parse.

Tầng chặn chính là output encoding, không phải input validation. Lý do: cùng một input có thể an toàn ở chỗ này nhưng nguy hiểm ở chỗ khác tuỳ ngữ cảnh render (trong HTML body, trong attribute, trong JS, trong URL). Nên cách đúng là encode đúng ngữ cảnh ngay lúc render — biến < thành &lt; khi đưa vào HTML — để trình duyệt hiển thị nó như văn bản chứ không thực thi. Bốn tầng phòng thủ chồng lên nhau (defense in depth):

1. Input validation   — whitelist ký tự hợp lệ (giảm bề mặt, nhưng KHÔNG đủ một mình)
2. Output encoding     — encode theo ngữ cảnh khi render (OWASP Java Encoder) ← TẦNG CHÍNH
3. Content-Security-Policy (header) — trình duyệt từ chối chạy inline script dù bị tiêm
4. HttpOnly cookie flag — JavaScript không đọc được cookie → script (nếu lọt) cũng khó trộm session

2.4 CSRF vs CORS + SameSite CORE ★

CSRF (Cross-Site Request Forgery) lợi dụng một đặc tính của trình duyệt: khi gửi request tới shopflow.com, trình duyệt tự động đính kèm cookie của shopflow.com — bất kể request đó được kích hoạt từ trang nào. Đây gọi là "ambient authority" (quyền có sẵn theo ngữ cảnh). Kẻ tấn công khai thác bằng cách: tạo một trang có form (hoặc request ngầm) trỏ tới shopflow.com/account/email; khi nạn nhân (đang đăng nhập ShopFlow) ghé trang đó, form auto-submit, trình duyệt kèm cookie session, và ShopFlow tưởng chính chủ → thực hiện hành động ngoài ý muốn.

CSRF attack:
1. Nạn nhân đang login ShopFlow (trình duyệt giữ session cookie)
2. Nạn nhân vào attacker.com — trang này có form ẩn POST tới shopflow.com/account/email
3. Form auto-submit; trình duyệt TỰ ĐỘNG kèm session cookie
4. ShopFlow nhận request "đã xác thực" → đổi email nạn nhân

CSRF token chặn thế nào: server nhúng vào form một token ngẫu nhiên, lưu bản đối chiếu trong session (không nằm trong cookie). Khi submit, request phải kèm token này trong body. Trang của kẻ tấn công không đọc được token đó (vì same-origin policy chặn đọc nội dung trang cross-origin), nên không thể gửi kèm → request giả thiếu token → bị từ chối.

Vì sao JWT trong header miễn nhiễm CSRF: nếu bạn xác thực bằng JWT đặt trong header Authorization: Bearer ... (không phải cookie), thì trình duyệt không tự động đính kèm header này cho request cross-origin — chỉ JS của chính bạn mới gắn được. Kẻ tấn công không lấy được token để gắn → request giả thiếu auth → fail. Đó là lý do API stateless dùng JWT header thường không cần CSRF token (bật vào là thừa, gây hiểu sai mô hình).

CORS khác hẳn: CORS (Cross-Origin Resource Sharing) là cơ chế trình duyệt quyết định liệu JavaScript của một origin có được đọc response từ origin khác hay không. Nó không liên quan tới chống CSRF; thậm chí cấu hình CORS quá lỏng (Access-Control-Allow-Origin: * kèm credentials) còn mở thêm rủi ro. Nhầm CSRF với CORS → bật sai cơ chế.

SameSite cookie attribute (lớp chống CSRF ở tầng cookie):
  Strict — không bao giờ gửi cookie cho request khởi nguồn từ site khác
  Lax    — gửi cho cross-site nhưng chỉ với navigation an toàn (GET click link)  ← default hợp lý
  None   — gửi mọi nơi (bắt buộc kèm Secure = chỉ HTTPS)

L1 — Biết tên

Biết "CSRF cần token". Symptom hiểu giả: nghĩ CSRF và CORS là một, hoặc bật CSRF token cho REST API stateless dùng JWT (thừa).

L2 — Explain được

Giải thích CSRF = lợi dụng ambient authority của cookie; token chống vì site khác không đọc được token; CORS là cơ chế đọc response cross-origin, không phải chống CSRF.

L3 — Apply được

Chọn cơ chế theo cách lưu credential: session cookie → CSRF token hoặc SameSite=Lax/Strict + HttpOnly; JWT trong header → không cần CSRF token. Cấu hình CORS chặt (whitelist origin cụ thể).

2.5 IDOR — authentication ≠ authorization CORE ★

IDOR (Insecure Direct Object Reference) là một dạng Broken Access Control. Mấu chốt là phân biệt hai khái niệm hay bị gộp: authentication = xác minh "bạn là ai" (đăng nhập đúng); authorization = quyết định "bạn được phép làm gì với resource nào". IDOR là lỗ hổng authorization: user đã authenticate hợp lệ, nhưng server không kiểm tra rằng resource họ yêu cầu có thuộc về họ không. Code điển hình: findById(orderId) trả về đơn ứng với id trên URL mà không hỏi "đơn này có phải của người đang đăng nhập không". Đổi orderId = xem đơn người khác.

// ❌ tin orderId từ URL → user xem đơn người khác (chỉ authenticate, không authorize)
Order o = orderRepo.findById(orderId).orElseThrow();

// ✅ check ownership: ràng buộc resource theo user đang đăng nhập
Order o = orderRepo.findByIdAndCustomerId(orderId, currentUser.getId())
                   .orElseThrow(NotFoundException::new);   // trả 404, không lộ tồn tại

Fix bắt buộc ở backend (service/data layer) — không phải ở frontend (ẩn nút thì attacker vẫn gọi API trực tiếp) hay ở log (đã bị khai thác rồi). Chi tiết nhỏ nhưng quan trọng: trả 404 Not Found thay vì 403 Forbidden khi resource không thuộc về user — vì 403 vô tình tiết lộ "resource này có tồn tại, chỉ là bạn không được xem", giúp attacker dò ra id hợp lệ.

3.1 Controller: parameterized + validation CORE ★

@RestController
@RequestMapping("/orders")
class OrderController {
  private final OrderService service;

  @PostMapping
  ResponseEntity<OrderResponse> create(
      @Valid @RequestBody OrderRequest req,            // Bean Validation server-side
      @AuthenticationPrincipal UserPrincipal user) {
    OrderResponse r = service.place(user.id(), req);   // dùng user TỪ TOKEN, không từ body
    return ResponseEntity.status(CREATED).body(r);
  }

  @GetMapping("/{id}")
  OrderResponse get(@PathVariable Long id, @AuthenticationPrincipal UserPrincipal user) {
    return service.getOwned(id, user.id());            // check ownership chống IDOR
  }
}

record OrderRequest(
    @NotBlank String sku,
    @Min(1) @Max(999) int quantity,                    // validate biên ở SERVER
    @Pattern(regexp = "[A-Z0-9-]{0,16}") String coupon // whitelist ký tự
) {}

Lưu ý hai chỗ thường sai: (a) lấy customerId từ token (user.id()) chứ KHÔNG từ body request — nếu tin body, attacker đặt đơn cho người khác; (b) validation phải ở server, vì validation phía client chỉ phục vụ UX, attacker bỏ qua client dễ dàng.

3.2 Security headers filter SUPPLEMENTARY

@Component
class SecurityHeadersFilter extends OncePerRequestFilter {
  @Override protected void doFilterInternal(HttpServletRequest req,
      HttpServletResponse res, FilterChain chain) throws ... {
    res.setHeader("Content-Security-Policy", "default-src 'self'");   // chặn inline script (XSS)
    res.setHeader("X-Content-Type-Options", "nosniff");              // chống MIME sniffing
    res.setHeader("X-Frame-Options", "DENY");                        // chống clickjacking
    res.setHeader("Strict-Transport-Security", "max-age=31536000");  // HSTS — ép HTTPS
    res.setHeader("Referrer-Policy", "no-referrer");
    chain.doFilter(req, res);
  }
}
// Cookie nên có: Set-Cookie: session=...; HttpOnly; Secure; SameSite=Lax

3.3 PII masking + rate limit SUPPLEMENTARY

Hai biện pháp nhỏ nhưng quan trọng: masking đảm bảo dữ liệu nhạy cảm (email, số thẻ, địa chỉ, token) không bị ghi nguyên văn vào log — vì log thường được tập trung, lưu lâu, nhiều người đọc, dễ rò rỉ. Rate limit login chống brute-force (thử mật khẩu hàng loạt) và credential stuffing (thử các cặp email/mật khẩu rò rỉ từ nơi khác) bằng cách giới hạn số lần thử theo IP và theo tài khoản.

static String maskEmail(String e) {           // a***@mail.com
  int at = e.indexOf('@');
  return at <= 1 ? "***" : e.charAt(0) + "***" + e.substring(at);
}
static String maskCard(String c) {            // **** **** **** 1234 (giữ 4 số cuối)
  return "**** **** **** " + c.substring(c.length() - 4);
}
log.info("order placed customer={} card={}", maskEmail(email), maskCard(card));

// rate limit (Redis token bucket): chặn brute-force theo CẢ ip và account
boolean allow(String ip, String account) {
  return bucket("login:ip:" + ip).tryConsume(1)
      && bucket("login:acct:" + account).tryConsume(1);
}
// vượt N lần → exponential backoff + captcha; khoá tạm account khi nghi credential stuffing

4. Anti-patterns CORE ★

🔧 Counter-example — khi "phòng thủ" lại tạo lỗ hổng / sai chỗ
  • Tự escape thay parameterized: blacklist ký tự luôn sót (Unicode, comment lồng, double-encoding). Phòng thủ đúng tầng là tách lệnh/data ở protocol, không lọc input.
  • Verbose error: trả stack trace/câu SQL ra client = chỉ đường cho attacker (lộ schema, lộ framework). Log chi tiết ở server, trả message chung cho client.
  • Validate ở client là đủ: client validation chỉ cho UX; attacker gọi thẳng API → mọi check bảo mật phải lặp lại ở server.
  • Bật CSRF token cho JWT API stateless: thừa và gây hiểu sai mô hình; ngược lại quên SameSite/HttpOnly cho session cookie mới là lỗ hổng thật.

Quy tắc bao trùm: chặn ở đúng tầng cho từng lỗ hổng (parameterized cho injection, output encoding cho XSS, ownership check cho IDOR), và không tin bất cứ gì đến từ client.

5. 3-tier interview framework

Tầng 1 — SurfaceJunior gate

Q: Chặn SQL injection thế nào?

Parameterized query/PreparedStatement — DB parse câu lệnh trước, input đến sau như data thuần, không bao giờ parse như SQL (tách code/data ở tầng protocol). Whitelist cho phần không tham số hoá được (tên cột, ORDER BY). Không tự escape.

Tầng 2 — Deep DiveSenior gate

Q: Thiết kế API xử lý PII an toàn.

TLS bắt buộc; authorization theo ownership (chống IDOR); PII mã hoá at-rest + masking trong log; rate limit login; input validation server-side + output encoding (chống XSS); security headers (CSP/HSTS/X-Frame); cookie HttpOnly+Secure+SameSite; audit log truy cập PII; least-privilege cho DB account.

Tầng 3 — ArchitectureSenior+

Q: CSRF vs CORS — giải thích sự nhầm lẫn phổ biến.

CSRF: trình duyệt tự gửi cookie (ambient authority) → kẻ tấn công khiến nạn nhân gửi request hợp lệ ngoài ý muốn → cần token/SameSite; chỉ ảnh hưởng app dùng session cookie. CORS: cơ chế trình duyệt cho phép JS đọc response cross-origin — KHÔNG chống CSRF. JWT header miễn nhiễm CSRF (trình duyệt không tự gắn). Nhầm hai cái → bật sai cơ chế, vừa thừa vừa hở.

🔧 Lab bắt buộc — khai thác & vá (2 bài)

Lab 1 — SQL Injection. Viết một endpoint search nối chuỗi input vào SQL. Khai thác bằng ' OR '1'='1 để dump bảng. Vá bằng parameterized query, thử lại → vô hại. Thêm case ORDER BY động → chứng minh vẫn injectable, vá bằng whitelist. Tiêu chí: giải thích tách lệnh/data ở tầng nào.

Lab 2 — IDOR. Tạo endpoint GET /orders/{id} chỉ findById. Login user A, đổi id sang đơn của user B → xem được (lỗ hổng). Vá bằng findByIdAndCustomerId(currentUser), trả 404. Tiêu chí: phân biệt authentication vs authorization, vì sao 404 không 403.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. Vì sao parameterized query chặn SQL injection (ở tầng nào)? Trường hợp nào vẫn injectable?
  2. IDOR là gì? Phân biệt authentication vs authorization. Vì sao "đã login" không ngăn được?
  3. CSRF hoạt động thế nào (ambient authority)? Vì sao JWT header miễn nhiễm? CORS khác gì? SameSite làm gì?
  4. XSS chặn chủ yếu ở đâu — input hay output? 4 tầng phòng thủ là gì?
  5. Vì sao mọi check bảo mật phải lặp lại ở server dù client đã validate?
