Idempotency — Khi retry mà KHÔNG biến double-charge thành thảm hoạ

Idempotency là pattern quan trọng nhất trong backend xử lý tiền/order/booking. Hỏng = mất tiền khách hàng thật. Nhưng nó cũng là chủ đề dễ hiểu giả nhất: ai cũng "biết" UUID + DB check, nhưng khi probe sâu (race condition? TTL? retryable vs non-retryable failure? client-side responsibility?) thì lộ ngay. File này ép bạn dừng lại, tự reproduce race condition, và thấy được khi nào pattern bị implement sai mà code vẫn chạy.

Vì sao Idempotency dễ tạo hiểu giả:
  • Vocabulary nghe đơn giản ("idempotent = gọi nhiều lần OK") → não thưởng cho cảm giác "đã hiểu" mà chưa thấy edge case.
  • Pseudocode chuẩn nhìn rất "hợp lý" — nhưng implementation thật sai vì race condition không hiện ngay.
  • Bug chỉ xuất hiện dưới concurrent load — dev test 1 request thấy OK, deploy production thấy double-charge.
  • "Retryable vs non-retryable" là khái niệm subtle. Đa số dev miss → retry "insufficient funds" infinitely.

1. Bài toán: Double-charge từ network retry CORE

Một e-commerce platform xử lý 10K đơn/giờ. Customer ấn "Pay $99" cho 1 order:

[T+0.0s] Mobile app gửi POST /payments {amount:99, orderId:'ord_abc'}
[T+0.5s] Server nhận, charge card → thành công, balance trừ $99
[T+1.0s] Server bắt đầu gửi response
[T+2.0s] ❌ Network timeout — app không nhận được response
[T+2.0s] App logic: "Timeout → retry"
[T+2.1s] App gửi LẠI POST /payments {amount:99, orderId:'ord_abc'}
[T+2.6s] Server nhận, charge card LẦN 2 → thành công, balance trừ THÊM $99
[T+3.0s] App nhận response 200 OK của lần 2
[T+3.5s] Customer thấy "Payment success" → mở app sau, thấy bị charge $198

❌ Customer mất 99 dollar không lý do. Refund process tốn 3-5 ngày.

4 câu hỏi buộc tư duy

  1. Server đã charge xong, response bị lost. Khi client retry, server phải làm gì để KHÔNG charge lại?
  2. Làm sao server biết request lần 2 là "retry của lần 1" — không phải "payment mới cho cùng order"?
  3. Nếu user thực sự muốn pay 2 lần (lần 2 cho order khác), làm sao phân biệt với retry?
  4. Service A gọi service B qua message queue. B process xong nhưng ACK bị drop. A retry → B nhận lần 2. B phải handle thế nào?

2. Định nghĩa chính xác — không phải "gọi nhiều lần OK" CORE

Đa số tài liệu định nghĩa idempotency mơ hồ. Để hiểu đúng, dùng definition toán học:

f(f(x)) = f(x)

Một operation f là idempotent nếu: áp dụng f LẦN 2 lên kết quả của f
cho ra cùng kết quả. Equivalently: gọi f với cùng input N lần
cho ra cùng final state như gọi 1 lần.

3 trường hợp dễ confuse

❌ KHÔNG phải idempotent

// Tăng counter
void increment(Counter c) {
    c.value++;
}
// Gọi 1 lần: value = 1
// Gọi 2 lần: value = 2
// 2 lần ≠ 1 lần → KHÔNG idempotent

// Append log
void log(String msg) {
    file.appendLine(msg);
}
// 2 dòng log ≠ 1 dòng log → KHÔNG idempotent

✅ LÀ idempotent

// Set absolute value
void setBalance(Account a, int newBalance) {
    a.balance = newBalance;
}
// Gọi 1 lần với 100: balance = 100
// Gọi 2 lần với 100: balance = 100
// Cùng final state → idempotent

// Delete by ID
void delete(String id) {
    repo.deleteById(id); // 404 nếu không tồn tại
}
// Gọi 1 lần: xoá
// Gọi 2 lần: vẫn không tồn tại → cùng state
Lưu ý chính xác: Idempotent KHÔNG có nghĩa là "không có side effect". delete có side effect (xoá data lần đầu), nhưng cùng final state dù gọi N lần → idempotent. Vấn đề là final state, không phải không thay đổi gì.

Idempotent vs Safe — phân biệt

PropertyĐịnh nghĩaVí dụ
SafeKhông thay đổi server stateGET (chỉ đọc)
IdempotentGọi N lần = gọi 1 lần (về final state)GET, PUT, DELETE
BothVừa safe vừa idempotentGET
Idempotent nhưng không safeCó thay đổi state nhưng cùng state sau N lầnDELETE, PUT

3. HTTP method semantics — POST KHÔNG idempotent mặc định CORE

MethodIdempotent theo spec?Cơ chếVí dụ
GET✅ CóChỉ đọc, không thay đổi stateGET /orders/123
PUT✅ CóReplace toàn bộ resource với payloadPUT /users/42 {name:"Alice"} → state cuối luôn là name="Alice"
DELETE✅ CóXoá rồi, xoá lại = 404 hoặc no-opDELETE /orders/123
POST❌ KHÔNGMỗi lần gọi có thể tạo resource mớiPOST /payments → có thể charge 2 lần
PATCH⚠️ TuỳTuỳ payload — increment KHÔNG idempotent, set KHÔNG idempotentPATCH /counter {op:"increment"} ≠ idempotent
Insight quan trọng: HTTP spec định nghĩa idempotency cho method, nhưng implementation thực tế tuỳ developer. Bạn có thể viết GET /charge?amount=99 mà trừ tiền (sai spec, nhưng vẫn chạy). Spec chỉ là contract — server phải tự đảm bảo.

Vì sao POST cần idempotency key

POST tạo resource mới mỗi lần → tự nó không idempotent. Để biến POST thành idempotent, dùng Idempotency-Key header (chuẩn Stripe, được nhiều fintech áp dụng):

POST /payments HTTP/1.1
Idempotency-Key: 8e2c4a9f-3b1d-4f2e-9c1a-5d8f7e6b3a2c
Content-Type: application/json

{"amount": 99, "orderId": "ord_abc"}

# Server đảm bảo: cùng Idempotency-Key → cùng response, không xử lý lại

Idempotency-Key thực ra biến POST thành conditional create: "tạo payment NẾU key này chưa từng được dùng, ngược lại return result cũ".

⏸️ Pause & Predict

Tại sao bước "insert record state=PROCESSING trước khi execute" lại quan trọng? Viết 2-3 dòng giải thích.

Đáp án: Nếu insert SAU khi execute, có khoảng trống giữa "execute xong" và "insert key". Service crash trong khoảng đó → operation đã thực hiện nhưng key chưa lưu → client retry → thực hiện lại → duplicate. Insert TRƯỚC với state=PROCESSING đảm bảo: dù service crash giữa chừng, retry sẽ thấy state PROCESSING/COMPLETED và xử lý đúng (không re-execute).

5. At-most-once / At-least-once / Exactly-once CORE

GuaranteeBehaviorCostUse case
At-most-once (fire & forget) Gửi 1 lần, không retry. Có thể bị mất. Rẻ nhất, nhanh nhất Metrics, analytics, telemetry — mất 1 data point không sao
At-least-once Retry đến khi ACK. Có thể process N lần. Trung bình. CẦN idempotent consumer. Hầu hết messaging (Kafka default), payment retry
Exactly-once Process đúng 1 lần, guaranteed Đắt — distributed coordination (2PC, Kafka Transactions) Critical financial settlement, ledger entries
Pattern thực tế: "At-least-once + Idempotent consumer = effectively exactly-once" — đây là pattern phổ biến nhất production. Đơn giản, throughput cao, đúng đắn. Tránh Kafka Transactions / 2PC trừ khi thực sự cần atomic cross-partition writes.

Vì sao Exactly-once "thật" khó

Distributed systems theorem (Two Generals Problem, FLP impossibility): trong môi trường network có thể fail, KHÔNG thể đảm bảo cả 2 bên đồng ý "đã xử lý đúng 1 lần" mà không có infinite messages. Mọi "exactly-once" trong thực tế đều là kết hợp:

  • At-least-once delivery (retry)
  • + Idempotent processing (dedup)
  • = Effectively exactly-once từ góc nhìn consumer

🔨 Thử Phá Vỡ: Kafka Transactions có thật sự exactly-once không?

Kafka Transactions đảm bảo atomic producer write + consumer read offset. Nhưng consumer-side processing (call DB, gọi API ngoài) KHÔNG nằm trong Kafka transaction → vẫn cần idempotency layer. "Exactly-once" của Kafka thực ra là "atomic producer + read isolation" — không cover toàn bộ processing chain.

6. REST API implementation — Idempotency-Key header CORE

Controller — entry point

@RestController
@RequestMapping("/api/payments")
public class PaymentController {

    private final IdempotencyService idempotencyService;
    private final PaymentService paymentService;

    @PostMapping
    public ResponseEntity<PaymentResponse> createPayment(
            // Client TỰ tạo key, gửi lên. Server KHÔNG generate hộ.
            // Lý do: nếu server generate, retry sẽ gửi key khác → không dedup được.
            @RequestHeader("Idempotency-Key") String idempotencyKey,
            @RequestBody @Valid PaymentRequest request,
            @AuthenticationPrincipal UserPrincipal user) {

        // Validate format — phòng key tuỳ tiện (vd dùng số tăng dần)
        if (!isValidUuid(idempotencyKey)) {
            throw new BadRequestException("Idempotency-Key must be valid UUID v4");
        }

        // Scope per-user: cùng key từ user khác KHÔNG conflict
        String scopedKey = user.getId() + ":" + idempotencyKey;

        return idempotencyService.executeIdempotent(
            scopedKey,
            () -> paymentService.processPayment(request),
            PaymentResponse.class
        );
    }

    private boolean isValidUuid(String s) {
        try { UUID.fromString(s); return true; }
        catch (IllegalArgumentException e) { return false; }
    }
}
Pitfall #1 thường gặp: Lấy idempotency key từ body request, KHÔNG phải header. Sai vì: nếu body lớn, key bị bury. Logging tự động (access log) capture được header nhưng không body. Thuê 3rd-party (Stripe, Paddle) đều dùng header.
Pitfall #2: Không scope key per-user. User A và User B vô tình dùng cùng UUID (xác suất thấp nhưng có) → user B bị block. Fix: prefix key với userId.

7. Idempotency Service — Core logic (chứa bug TOCTOU dễ mắc) HIGH-SIGNAL

❌ Version 1 — Bug TOCTOU race condition

Đây là code đa số dev viết lần đầu. Nó nhìn đúng nhưng có race condition fatal:

// ❌ SAI — chứa Time-Of-Check / Time-Of-Use race
@Transactional
public <T> ResponseEntity<T> executeIdempotent(String key, Supplier<T> op, Class<T> type) {
    // Step 1: CHECK
    if (repo.existsByKey(key)) {
        IdempotencyRecord record = repo.findByKey(key).get();
        return ResponseEntity.status(record.getHttpStatus()).body(deserialize(record));
    }

    // ⚠️ KHOẢNG TRỐNG GIỮA CHECK VÀ INSERT — race condition

    // Step 2: USE (insert)
    repo.insert(IdempotencyRecord.builder().key(key).status(PROCESSING).build());

    // Step 3: execute
    T result = op.get();
    repo.updateCompleted(key, result);
    return ResponseEntity.ok(result);
}

// Concurrent scenario:
// Thread A: existsByKey(K) → false
// Thread B: existsByKey(K) → false (vì A chưa insert)
// Thread A: insert(K) → OK
// Thread B: insert(K) → OK??? hoặc duplicate key error
// Cả 2 thread execute → DOUBLE CHARGE

✅ Version 2 — Insert-first với unique constraint

@Service
@Slf4j
public class IdempotencyService {

    private final IdempotencyKeyRepository repo;
    private final ObjectMapper objectMapper;

    @Value("${idempotency.ttl-hours:24}")
    private int ttlHours;

    @Transactional
    public <T> ResponseEntity<T> executeIdempotent(
            String key, Supplier<T> operation, Class<T> responseType) {

        // Insert TRƯỚC với state=PROCESSING.
        // DB unique constraint là atomic gate — chỉ 1 thread thắng.
        // Race winner → tiếp tục execute.
        // Race loser → DuplicateKeyException → đọc record của winner.
        try {
            repo.insert(IdempotencyRecord.builder()
                .key(key)
                .status(IdempotencyStatus.PROCESSING)
                .createdAt(Instant.now())
                .expiresAt(Instant.now().plus(ttlHours, ChronoUnit.HOURS))
                .build());

            // Thread thắng race → execute
            return executeAndSave(key, operation, responseType);

        } catch (DataIntegrityViolationException e) {
            // Thread thua race (hoặc retry) — đọc record của winner
            IdempotencyRecord record = repo.findByKey(key)
                .orElseThrow(() -> new IllegalStateException("Key vanished after constraint hit"));

            return handleExistingRecord(record, operation, responseType, key);
        }
    }

    private <T> ResponseEntity<T> handleExistingRecord(
            IdempotencyRecord record, Supplier<T> operation,
            Class<T> responseType, String key) {

        switch (record.getStatus()) {
            case COMPLETED:
                // Return cached — KHÔNG re-execute
                T cached = deserialize(record.getResponseBody(), responseType);
                return ResponseEntity.status(record.getHttpStatus())
                    .header("Idempotency-Replayed", "true")
                    .body(cached);

            case PROCESSING:
                // Concurrent retry — chưa biết kết quả
                throw new ConflictException(
                    "Request with this key is being processed. Retry in a few seconds.");

            case FAILED:
                if (record.isRetryable()) {
                    // Transient failure — cho phép retry
                    repo.updateStatus(key, IdempotencyStatus.PROCESSING);
                    return executeAndSave(key, operation, responseType);
                } else {
                    // Permanent failure (Insufficient Funds, Card Declined)
                    // → return error cũ, KHÔNG retry
                    T error = deserialize(record.getResponseBody(), responseType);
                    return ResponseEntity.status(record.getHttpStatus()).body(error);
                }
            default:
                throw new IllegalStateException("Unknown status: " + record.getStatus());
        }
    }

    private <T> ResponseEntity<T> executeAndSave(
            String key, Supplier<T> operation, Class<T> responseType) {
        try {
            T result = operation.get();
            repo.updateCompleted(key, 200, serialize(result), false);
            return ResponseEntity.ok(result);

        } catch (InsufficientFundsException | CardDeclinedException e) {
            // Business error — non-retryable
            repo.updateFailed(key, 422, serialize(ErrorResponse.of(e)), false);
            throw e;

        } catch (Exception e) {
            // Technical error (network, timeout) — retryable
            repo.updateFailed(key, 500, serialize(ErrorResponse.of(e)), true);
            throw e;
        }
    }
}

Vì sao Insert-first thắng Check-then-Insert

ApproachAtomicityRace-safe?
Check exists → Insert if not2 step, có khoảng trống❌ KHÔNG — TOCTOU
Insert with unique constraint → catch exception1 atomic DB operation✅ CÓ — DB lock đảm bảo
SELECT FOR UPDATE → InsertRow lock✅ CÓ — nhưng dài hơn, performance kém
Application-level lock (synchronized)JVM-only❌ KHÔNG cho multi-instance
Redis SETNXAtomic✅ CÓ — nhưng cần backup DB
Quy tắc cứng: Mọi pattern "check rồi act" trên shared state đều có TOCTOU race. Phải dùng atomic primitive: DB unique constraint, Compare-And-Set (CAS), DB row lock, distributed lock. Idempotency tự nó là pattern chống race — implementation cũng không được có race.

8. DB schema + TTL cleanup CORE

CREATE TABLE idempotency_keys (
    id              BIGSERIAL PRIMARY KEY,
    key             VARCHAR(255) NOT NULL,
    status          VARCHAR(20)  NOT NULL,          -- PROCESSING / COMPLETED / FAILED
    http_status     INT,
    response_body   TEXT,                            -- JSON serialized response
    is_retryable    BOOLEAN      NOT NULL DEFAULT FALSE,
    created_at      TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    expires_at      TIMESTAMPTZ  NOT NULL,

    -- ★ Heart of idempotency — unique constraint atomic
    CONSTRAINT uq_idempotency_key UNIQUE (key)
);

-- Index cho cleanup job (partial index — chỉ rows cần xoá)
CREATE INDEX idx_idempotency_expires
    ON idempotency_keys (expires_at)
    WHERE status IN ('COMPLETED', 'FAILED');

-- Cleanup job (run every hour hoặc daily)
DELETE FROM idempotency_keys
WHERE expires_at < NOW()
  AND status IN ('COMPLETED', 'FAILED');
-- KHÔNG xoá PROCESSING — có thể service đang chạy, sẽ chuyển state sớm

🔨 Bẫy TTL — viết ra giấy trước khi đọc

TTL quá ngắn (1 phút): user retry sau 2 phút → key expired → treat as new request → DUPLICATE payment.
TTL quá dài (1 năm): table phình to + user thật sự muốn pay lại sau 1 tuần thì bị block.
Sweet spot: 24-72h cho POST. Tail end longer cho settlement/audit.

Partition table — khi data lớn

10K TPS × 24h × 86400s/h ≈ 8.6 triệu record/ngày. Dùng PostgreSQL partition by created_at daily, drop partition cũ thay vì DELETE:

-- Daily partition — drop O(1), delete O(n)
CREATE TABLE idempotency_keys (
    id              BIGSERIAL,
    key             VARCHAR(255) NOT NULL,
    -- ... cùng schema
    created_at      TIMESTAMPTZ  NOT NULL DEFAULT NOW()
) PARTITION BY RANGE (created_at);

-- Partition cho hôm nay
CREATE TABLE idempotency_keys_2026_05_23
    PARTITION OF idempotency_keys
    FOR VALUES FROM ('2026-05-23') TO ('2026-05-24');

-- Cleanup: DROP partition cũ — instant
DROP TABLE idempotency_keys_2026_05_20;

9. Idempotent Kafka Consumer HIGH-SIGNAL

Kafka mặc định at-least-once delivery. Consumer có thể nhận duplicate khi: rebalance, network blip, consumer crash giữa process và commit offset. Phải idempotent ở consumer side.

@Component
@Slf4j
public class OrderEventConsumer {

    private final OrderProcessingService processingService;
    private final ProcessedEventRepository processedRepo;

    @KafkaListener(topics = "order-events", groupId = "order-processor")
    @Transactional
    public void handle(OrderEvent event) {
        // event.eventId là UUID do producer tạo TẠI BUSINESS EVENT TIME,
        // KHÔNG phải Kafka offset (offset có thể reset hoặc duplicate ở producer side).
        String eventId = event.getEventId();

        // Check dedup
        if (processedRepo.existsById(eventId)) {
            log.info("Duplicate event {} skipped", eventId);
            return; // Idempotent: ignore
        }

        // Process business logic
        processingService.handle(event);

        // Mark processed — TRONG CÙNG TRANSACTION với business logic
        // Lý do: nếu save processed_event fail, business cũng rollback
        //        → retry sẽ re-process đúng (chưa có bản ghi processed)
        processedRepo.save(ProcessedEvent.builder()
            .eventId(eventId)
            .processedAt(Instant.now())
            .topic("order-events")
            .build());
    }
}

3 quy tắc cứng cho Kafka idempotency

  1. Event ID là business-level identifier — UUID do producer gán ở thời điểm tạo event, KHÔNG phải Kafka offset. Offset reset / rebalance làm offset không reliable.
  2. processed_events save cùng transaction với business logic. Save riêng → có thể save xong business nhưng save dedup fail → retry process lại.
  3. processed_events có TTL. Không lưu mãi mãi (table sẽ vô hạn). TTL 3-7 ngày + cleanup job. Producer phải đảm bảo retry trong window TTL.
Edge case nguy hiểm: Nếu business logic gọi external API (vd charge card qua Stripe), API call KHÔNG nằm trong DB transaction. Crash sau khi charge nhưng trước khi save dedup → retry charge lại. Fix: dùng Idempotency-Key của Stripe (Stripe có sẵn pattern này) + save Stripe response trong DB transaction.

10. Client-side responsibility — phần dev backend hay quên CORE

Idempotency chỉ work nếu client làm đúng phần của mình. Backend dev thường viết server đúng nhưng client sai → server vô dụng.

Quy tắc client

  1. Generate key MỘT LẦN khi user initiate action. Ví dụ: user mở checkout page → generate UUID, lưu localStorage / SecureStorage.
  2. Reuse cùng key cho mọi retry của cùng action. Network timeout → retry với CÙNG key. KHÔNG generate key mới.
  3. Generate key MỚI khi user thực sự muốn làm lại action. User pay xong, mở app sau, click pay lần nữa cho order khác → key mới.
  4. Clear key sau khi nhận được terminal response. 200 OK, 422 validation error, 403 forbidden — đều terminal, clear key. Chỉ retry với cùng key khi nhận timeout / 5xx.
// Mobile/Web client pseudocode
class PaymentClient {

    String generateKeyOnce(String userId, String orderId) {
        String stored = secureStorage.get("payment_key:" + orderId);
        if (stored != null) return stored;

        String newKey = UUID.randomUUID().toString();
        secureStorage.put("payment_key:" + orderId, newKey);
        return newKey;
    }

    Future<PaymentResponse> payWithRetry(PaymentRequest req) {
        String key = generateKeyOnce(currentUserId, req.orderId);

        for (int attempt = 0; attempt < 3; attempt++) {
            try {
                Response resp = http.post("/api/payments")
                    .header("Idempotency-Key", key)
                    .body(req)
                    .timeout(5000)
                    .execute();

                // Terminal response — không retry nữa
                if (resp.status() >= 200 && resp.status() < 500) {
                    secureStorage.remove("payment_key:" + req.orderId);
                    return resp.body();
                }
                // 5xx → retry với CÙNG key
            } catch (TimeoutException e) {
                // Retry với CÙNG key — server sẽ dedup
                continue;
            }
        }
        throw new MaxRetryException();
    }
}

❌ Client anti-pattern thường gặp

  • Generate UUID mỗi lần gửi: retry sinh ra request "mới", server không dedup được. Đây là lỗi phổ biến nhất — dev backend làm xong server, không hướng dẫn client.
  • Không persist key: app crash giữa retry → mất key → retry sinh key mới → duplicate.
  • Reuse key sau khi nhận 200: user click pay 2 lần thật sự (cho 2 order khác nhau) nhưng client gửi cùng key → lần 2 trả về response của lần 1 → user nghĩ pay được order mới, thực ra không.

⚠️ 5 anti-patterns kinh điển CORE

❌ #1: Check-then-Insert (TOCTOU)

Symptom: Dưới concurrent load (2 retry gần nhau), double-charge. Test 1 request OK, deploy production thấy lỗi.

Fix: Insert-first với DB unique constraint. Xem Section 7.

❌ #2: Key scoped globally thay vì per-user

Symptom: User A và User B cùng tạo UUID v4 (xác suất 1 trên 2^122 nhưng vẫn có) → user B request bị block bằng response của user A → leak data + UX broken.

Fix: Scope key per-user: scopedKey = userId + ":" + idempotencyKey.

❌ #3: Chỉ lưu "đã xử lý" KHÔNG lưu response body

// ❌ SAI
if (alreadyProcessed(key)) return ResponseEntity.ok().build(); // Empty body!

// Retry nhận 200 nhưng không có data → client crash hoặc UX broken

Fix: Lưu toàn bộ response (status + body + headers nếu cần). Return chính xác response của lần đầu.

❌ #4: TTL sai (quá ngắn hoặc quá dài)

Symptom (TTL quá ngắn, 1 phút): Network timeout 30s + user delay tap retry 90s sau → key expired → retry treated as new request → DUPLICATE.

Symptom (TTL quá dài, 1 năm): Table phình to. User legitimately pay cùng amount sau 6 tháng → bị block.

Fix: 24-72h cho payment. 3-7 ngày cho Kafka consumer.

❌ #5: Lưu record SAU khi process

// ❌ SAI
T result = operation.get(); // Process — đã charge!
                            // ← SERVICE CRASH HERE
repo.insert(key, result);   // Không kịp lưu

// Retry → process lại → DUPLICATE CHARGE

Fix: Insert state=PROCESSING TRƯỚC khi process. Update COMPLETED sau khi process xong. Crash giữa chừng → record vẫn còn → retry không re-process.

🔨 Anti-pattern nguy hiểm nhất?

#1 (TOCTOU). Lý do:

  • Bug chỉ hiện dưới concurrent load — dev test 1 request thấy OK.
  • Code "có vẻ đúng" theo logic con người — không có lỗi compile, không có lỗi test đơn lẻ.
  • Hậu quả: mất tiền khách hàng thật, refund process dài, đền bù tốn kém.
  • Bị repeat — junior tiếp theo viết code mới cũng dễ mắc lại.

❌ Khi nào KHÔNG cần Idempotency Key CORE

Idempotency-Key có cost: DB write extra mỗi request, cleanup job, complexity ở client. Không phải request nào cũng cần:

Tình huốngCần key?Lý do
GET /orders/123❌Read-only, đã idempotent native
PUT /users/42 {full payload}❌PUT đã idempotent theo spec — replace toàn bộ
DELETE /sessions/abc❌DELETE idempotent — xoá rồi không sao
POST /payments✅Tạo tài chính, side effect, cần dedup
POST /orders✅Tạo order — duplicate = order ma
POST /search-queries (log analytics)❌Mất 1 record analytic không sao — fire&forget OK
POST /signup (tạo user)⚠️Username unique constraint đã đủ — duplicate sẽ fail tự nhiên
PATCH /counter {op:increment}✅Increment KHÔNG idempotent — cần key

Quy tắc: Nếu request có side effect tạo resource quan trọng + client có thể retry → cần Idempotency-Key. Còn lại không cần.

🎯 3-tier interview framework

Tier 1 — Surface (Junior)

Q1: Idempotency là gì? Cho ví dụ HTTP method nào idempotent.

Trả lời: "Idempotency = gọi N lần cùng input cho cùng final state như gọi 1 lần. Theo HTTP spec: GET, PUT, DELETE idempotent. POST không idempotent — mỗi lần có thể tạo resource mới. Đó là lý do POST cần Idempotency-Key header để dedup ở server."

Q2: Vì sao retry POST có thể double-charge?

Trả lời: "Khi server process xong nhưng response bị lost (network timeout), client retry. Server không biết request lần 2 là retry hay payment mới → process lại → trừ tiền 2 lần. Solution: client gửi Idempotency-Key, server check key → return cached response nếu đã process."

Tier 2 — Deep Dive (Mid vs Senior)

Q3: Implement idempotency key pattern. Vì sao "check then insert" sai?

"Implement bằng cách client tạo UUID v4 gửi qua header Idempotency-Key. Server scope key per-user, insert vào DB với state=PROCESSING — dùng DB unique constraint, KHÔNG check-then-insert.

Vì sao check-then-insert sai: Đây là TOCTOU race condition. Thread A check exists → false. Thread B check exists → cũng false (A chưa insert). Cả 2 cùng insert → cùng execute → double-charge. DB unique constraint là atomic gate — chỉ 1 thread thắng race, thread thua nhận DuplicateKeyException → đọc response của winner.

Trade-off: thêm 1 DB write/request, cần cleanup job cho expired keys, client phải biết generate & reuse key."

Q4: At-least-once vs Exactly-once. Implement exactly-once trong Kafka thế nào?

"Kafka mặc định at-least-once — consumer có thể nhận duplicate khi rebalance / commit offset fail. Để đạt 'effectively exactly-once' (không phải 'true exactly-once'):

① Producer gán event_id (UUID) cho mỗi message — business-level ID, KHÔNG dùng Kafka offset.
② Consumer maintain processed_events table. Process logic: check exists → skip nếu đã có; ngược lại process + insert processed_event TRONG CÙNG TRANSACTION với business logic.
③ TTL 3-7 ngày cho processed_events, partition by date để cleanup nhanh.

Cùng transaction rất quan trọng: nếu save business OK nhưng save dedup fail → retry process lại = duplicate. Cùng transaction đảm bảo atomic.

Kafka Transactions cho 'true exactly-once' nhưng throughput giảm 20-30%, chỉ dùng khi thực sự cần atomic cross-partition write."

Tier 3 — Architecture (Senior)

Q5: Design idempotency cho 10,000 TPS payment gateway, multi-region active-active.

Phút 1 — Clarify:
  - 10K TPS sustained hay peak? Single region hay multi?
  - Consistency requirement: strong cross-region hay eventual OK?
  - Idempotency window — bao lâu? (Stripe = 24h)
  - Có dùng nhà cung cấp ngoài (Stripe, Adyen) hay tự build?

Phút 2 — Bottleneck identify:
  10K TPS = 10K DB writes/sec cho idempotency record + N writes cho business.
  Single PG primary có thể nghẽn. Cần:
  - Read replica cho cached response lookup (eventual OK)
  - Hoặc Redis cache trước DB

Phút 3 — Architecture proposed:
  ┌────────┐   ┌─────────────┐   ┌──────────┐
  │ Client │──►│ API Gateway │──►│ Service  │
  └────────┘   └─────────────┘   └────┬─────┘
                                       │
              ┌────────────────────────┤
              ▼                        ▼
       ┌─────────────┐          ┌──────────────┐
       │ Redis cache │          │ Postgres     │
       │ (1-2 hours) │          │ (24h retain) │
       └─────────────┘          └──────────────┘

  Flow:
  1. Service check Redis: HIT → return cached, DONE
  2. MISS → INSERT vào PG với unique constraint
  3. INSERT thành công → execute business → update Redis + PG
  4. INSERT fail (duplicate) → đọc PG record → cache vào Redis

Phút 4 — Multi-region challenge:
  Idempotency key phải globally unique. Options:
  A) Global DB (CockroachDB, Spanner) — strong consistency, latency cao
  B) Region-affinity routing — hash(userId) → fixed region. Same key
     luôn đến cùng region → local DB đủ. Risk: region down = không
     pay được. Fix: failover + replication async.
  C) Accept ngắn hạn double-charge cross-region nếu network split.
     Phát hiện qua reconciliation, refund proactive.

Phút 5 — Trade-offs:
  + Redis fast nhưng volatile — phải có PG fallback
  + Multi-region active-active phức tạp; nếu volume cho phép,
    active-passive đủ
  + Idempotency window 24h là sweet spot; longer = table phình to
  + Monitor: false-negative (dedup miss → duplicate charge — alarm CRITICAL)
            false-positive (dedup nhầm → client retry bị block — alarm WARN)

🔨 Lab 1 — Reproduce TOCTOU race condition (3 giờ)

Mục tiêu: Tự mắt thấy double-charge dưới concurrent load.

Steps:

  1. Tạo Spring Boot project. POST /api/payments với Idempotency-Key header.
  2. Implement Version 1 (Check-then-Insert) — code có sẵn ở Section 7.
  3. Tạo bảng payments để track số lần charge (mỗi charge = 1 row).
  4. Viết script bằng curl / k6 / JMeter: gửi 100 request với cùng Idempotency-Key, parallel 50 concurrent.
  5. Đếm rows trong payments sau test: phải > 1 (race condition đã trigger).
  6. Implement Version 2 (Insert-first with unique constraint). Re-run test. Confirm chỉ có đúng 1 row.
  7. So sánh log: Version 1 thấy multiple "Processing payment", Version 2 thấy 1 process + N "Idempotency-Replayed: true".

Acceptance: Reproduce bug Version 1 thành công + fix với Version 2 + log evidence. Push commit lên GitHub.

🔨 Lab 2 — Idempotent Kafka consumer with crash simulation (4 giờ)

Mục tiêu: Build at-least-once + idempotent consumer, test resilience.

Steps:

  1. Setup Kafka local (docker-compose). Topic order-events.
  2. Producer gán event_id = UUID cho mỗi event.
  3. Consumer với processed_events table + transactional handler (code Section 9).
  4. Send 1000 events. Confirm consumer process đúng 1000.
  5. Test duplicate: producer gửi cùng event_id 5 lần → consumer chỉ process 1 lần.
  6. Test crash: kill consumer process sau khi process business nhưng trước commit offset (dùng Thread.sleep + kill -9 trong handler). Restart consumer → kiểm tra không re-process (đã có row trong processed_events).
  7. Test crash trước save dedup: ngược lại — kill trước khi save processed_event. Restart → consumer re-process. Confirm business logic re-execute correctly (vì cùng transaction nên đã rollback).

Acceptance: Cả 3 scenario (normal, duplicate, crash mid-handler) đều đúng. Document trong README.

🧠 Feynman test — Verify hiểu thật

🎓 Giải thích cho dev chưa biết — không nhìn file

Đóng tab. Tự trả lời 5 câu, < 30 giây mỗi câu:

  1. "Vì sao check-then-insert sai?" — TOCTOU race. 2 thread cùng check thấy not exists, cùng insert, cùng execute → duplicate. DB unique constraint là atomic primitive cần thiết.
  2. "Vì sao insert state PROCESSING trước khi execute, không phải sau?" — Crash giữa execute và insert sẽ làm record không lưu → retry process lại → duplicate. Insert trước với PROCESSING đảm bảo có record dù crash giữa chừng.
  3. "Insufficient Funds có nên retry không? Network Timeout có nên retry không?" — Insufficient Funds = business error, KHÔNG retry (cache error, return same response). Network Timeout = technical error, retry OK (reset state về PROCESSING). Phân biệt qua is_retryable flag.
  4. "At-least-once + idempotent = exactly-once?" — Effectively yes, từ góc consumer. Producer retry (at-least-once delivery), consumer dedup bằng event_id. Cùng transaction với business logic. Đây là pattern phổ biến nhất, đơn giản hơn Kafka Transactions.
  5. "Client phải làm gì để idempotency work?" — Generate UUID 1 lần khi initiate action, persist locally. Reuse cùng key cho mọi retry. Generate key MỚI khi user thực sự muốn action mới. Clear key sau terminal response.

Bí 1 câu → đó là chỗ hiểu giả. Quay lại section tương ứng.
