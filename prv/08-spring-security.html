#08 - Spring Security: JWT, OAuth2, Filter Chain, Method Security, CORS

Security trong Spring không phải chỉ là thêm .authenticated(). Bạn cần hiểu request đi qua filter nào, ai set SecurityContext, role/authority được check ở đâu, và vì sao authorization theo ownership phải nằm gần business resource.

Security Là Pipeline Trước Controller

Spring Security đứng trước DispatcherServlet. Request phải đi qua filter chain để xác thực, set identity, xử lý exception và kiểm tra quyền. Controller không nên tự parse token thủ công ở từng endpoint.

Học Spring Security đúng là học 2 câu hỏi: user là ai? và user được làm gì trên resource nào? JWT/OAuth2 trả lời phần xác thực; method security và domain check trả lời phần phân quyền.

1. Bài Toán Thực Tế - SaaS Workspace Lộ Document Của Team Khác

Một SaaS quản lý tài liệu có API GET /workspaces/{workspaceId}/documents/{id}. Security incident: user có token hợp lệ nhưng thuộc workspace A vẫn đọc được document workspace B nếu đoán được ID.

Sai: chỉ authenticated, không check ownership
@RestController
class DocumentController {
    @GetMapping("/workspaces/{workspaceId}/documents/{id}")
    public DocumentDto get(@PathVariable String workspaceId,
                           @PathVariable String id,
                           @RequestHeader("Authorization") String token) {
        log.info("token={}", token); // lộ token trong log
        return documentService.findById(id);
    }
}
Đúng: identity từ context + ownership check
@RestController
class DocumentController {
    @GetMapping("/workspaces/{workspaceId}/documents/{id}")
    @PreAuthorize("@workspaceSecurity.canRead(#workspaceId, #id, authentication)")
    public DocumentDto get(@PathVariable String workspaceId,
                           @PathVariable String id) {
        return documentService.findById(workspaceId, id);
    }
}

2. Mental Model

2.1 Security Filter Chain

JWT filter không "cho phép" request; nó chỉ xác thực và set SecurityContext. AuthorizationFilter mới quyết định allow/deny.

Flow khi token hợp lệ: JWT filter đọc header, verify token, load user details, tạo Authentication, set vào SecurityContextHolder. Flow khi token invalid: không set authentication, request tiếp tục, authorization rule trả 401/403.

2.2 JWT

JWT là token format gồm header, payload, signature. Payload chỉ được base64url encoded, không được mã hóa. Ai có token đều đọc được payload, nên không đưa password, secret, token khác, dữ liệu cá nhân nhạy cảm vào claims.

Header: {"alg":"RS256","typ":"JWT"}
Payload: {"sub":"user-123","roles":["ROLE_EDITOR"],"workspace":"ws-9","exp":1760000000}
Signature: sign(base64(header) + "." + base64(payload), privateKey)
JWT logout problem

JWT access token stateless nên server không có session để xóa. Giải pháp thường là access token ngắn hạn + refresh token lưu server-side; nếu cần revoke ngay, thêm blacklist bằng Redis với TTL bằng thời gian còn lại của token.

2.3 SecurityContext

SecurityContextHolder giữ identity của request hiện tại, mặc định bằng ThreadLocal trong servlet stack. Vì vậy async execution cần chú ý context propagation.

Authentication authentication =
    SecurityContextHolder.getContext().getAuthentication();

String username = authentication.getName();
Collection<? extends GrantedAuthority> roles = authentication.getAuthorities();

2.4 Method Security

URL security trả lời endpoint nào cần auth. Method security trả lời business action cụ thể có được phép không. Với multi-tenant SaaS, authorization theo ownership là bắt buộc.

@PreAuthorize("hasRole('ADMIN') or @workspaceSecurity.isMember(#workspaceId, authentication)")
public List<DocumentDto> listDocuments(String workspaceId) {
    return documentQuery.findByWorkspace(workspaceId);
}

2.5 CORS

CORS là browser security model. Khi frontend từ domain khác gọi API, browser gửi preflight OPTIONS để hỏi server có cho origin/method/header đó không. Client không tự "fix CORS" được nếu server không trả header đúng.

2.6 401 vs 403, EntryPoint vs AccessDeniedHandler

Phân biệt 401 và 403 là kiến thức bắt buộc vì nó cho biết request fail ở authentication hay authorization.

StatusNghĩaSpring component thường xử lý
401 UnauthorizedChưa xác thực hoặc token invalid/expiredAuthenticationEntryPoint
403 ForbiddenĐã xác thực nhưng không đủ quyềnAccessDeniedHandler

Nếu user đã login nhưng nhận 401, kiểm tra token parsing/filter. Nếu nhận 403, kiểm tra authority, role prefix, ownership check hoặc method security.

2.7 OAuth2/OIDC Vừa Đủ Cho Resource Server

Trong nhiều backend, service của bạn là resource server: nhận access token, verify signature qua JWKS, map claims thành authorities, rồi authorization theo endpoint/resource. Không cần thuộc toàn bộ spec để làm daily work, nhưng cần biết token từ đâu đến và verify thế nào.

FlowDùng khiGhi nhớ
Authorization Code + PKCEWeb/mobile user loginClient đổi code lấy token, backend thường chỉ verify access token.
Client CredentialsService-to-serviceKhông có user, authorization dựa vào client/app identity.
Refresh tokenDuy trì session dài hơn access tokenNên lưu/rotate server-side nếu cần revoke.
JWKSVerify JWT ký bằng asymmetric keyCần cache key và xử lý key rotation.

2.8 CSRF: Khi Nào Tắt, Khi Nào Không

CSRF quan trọng khi browser tự gửi credential như cookie/session. Với REST API stateless chỉ dùng bearer token trong Authorization header và không dùng cookie auth, thường tắt CSRF. Nếu app dùng session cookie/form login, không tắt CSRF bừa.

REST bearer token

Thường csrf.disable(), vì attacker site không tự gắn được Authorization header hợp lệ.

Session/cookie auth

Giữ CSRF protection, vì browser tự gửi cookie theo domain.

3. Production-Grade Implementation

3.1 SecurityFilterChain Cho REST API

@Configuration
@EnableWebSecurity
@EnableMethodSecurity
public class SecurityConfig {
    private final JwtAuthenticationFilter jwtFilter;

    @Bean
    SecurityFilterChain apiSecurity(HttpSecurity http) throws Exception {
        return http
            .csrf(csrf -> csrf.disable())
            .cors(cors -> cors.configurationSource(corsConfigurationSource()))
            .sessionManagement(session ->
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers(HttpMethod.OPTIONS, "/**").permitAll()
                .requestMatchers("/api/auth/**", "/actuator/health").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated())
            .exceptionHandling(ex -> ex
                .authenticationEntryPoint(json401())
                .accessDeniedHandler(json403()))
            .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class)
            .build();
    }
}

3.2 JWT Filter

@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {
    private final JwtService jwtService;
    private final UserDetailsService userDetailsService;

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain chain) throws ServletException, IOException {

        String header = request.getHeader(HttpHeaders.AUTHORIZATION);
        if (header == null || !header.startsWith("Bearer ")) {
            chain.doFilter(request, response);
            return;
        }

        String token = header.substring(7);
        try {
            String username = jwtService.extractSubject(token);
            if (SecurityContextHolder.getContext().getAuthentication() == null) {
                UserDetails user = userDetailsService.loadUserByUsername(username);
                if (jwtService.isValid(token, user)) {
                    var auth = new UsernamePasswordAuthenticationToken(
                        user, null, user.getAuthorities());
                    auth.setDetails(new WebAuthenticationDetailsSource().buildDetails(request));
                    SecurityContextHolder.getContext().setAuthentication(auth);
                }
            }
        } catch (JwtException ex) {
            logger.warn("Invalid JWT for path " + request.getRequestURI());
        }

        chain.doFilter(request, response);
    }
}

3.3 Ownership Check

@Service("workspaceSecurity")
public class WorkspaceSecurity {
    private final WorkspaceMemberRepository members;
    private final DocumentRepository documents;

    public boolean canRead(String workspaceId, String documentId, Authentication auth) {
        if (auth == null || !auth.isAuthenticated()) {
            return false;
        }
        String userId = ((AppUser) auth.getPrincipal()).id();
        return members.existsByWorkspaceIdAndUserId(workspaceId, userId)
            && documents.existsByIdAndWorkspaceId(documentId, workspaceId);
    }
}

3.4 CORS Config

@Bean
CorsConfigurationSource corsConfigurationSource() {
    CorsConfiguration config = new CorsConfiguration();
    config.setAllowedOrigins(List.of(
        "https://app.example-saas.com",
        "https://admin.example-saas.com"));
    config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE", "OPTIONS"));
    config.setAllowedHeaders(List.of("Authorization", "Content-Type", "X-Correlation-ID"));
    config.setExposedHeaders(List.of("X-Correlation-ID"));
    config.setAllowCredentials(true);
    config.setMaxAge(3600L);

    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/api/**", config);
    return source;
}

3.5 Security Testing: Đừng Chỉ Test Happy Path

Security rule rất dễ hỏng khi refactor matcher, role name hoặc method security expression. Cần test cả unauthenticated, authenticated nhưng thiếu quyền, và đúng quyền đúng resource.

@WebMvcTest(DocumentController.class)
class DocumentControllerSecurityTest {
    @Autowired MockMvc mvc;

    @Test
    void shouldReturnUnauthorizedWithoutToken() throws Exception {
        mvc.perform(get("/workspaces/ws-1/documents/doc-1"))
            .andExpect(status().isUnauthorized());
    }

    @Test
    @WithMockUser(roles = "USER")
    void shouldReturnForbiddenWhenNotWorkspaceMember() throws Exception {
        mvc.perform(get("/workspaces/ws-2/documents/doc-9"))
            .andExpect(status().isForbidden());
    }
}

Với resource server JWT, có thể dùng test support để tạo JWT có claims/authorities mong muốn thay vì gọi identity provider thật trong unit/slice test.

5. Trade-offs & Anti-Patterns

AP1

Log Token

Không log Authorization header, refresh token, cookie session. Log userId, requestId, path, status là đủ cho debug thông thường.

AP2

Chỉ Check Role, Không Check Resource

ROLE_USER không có nghĩa được đọc mọi document. Với multi-tenant app, luôn check workspace membership và resource ownership.

AP3

CORS * Với Credentials

Browser không cho Access-Control-Allow-Origin: * đi cùng credentials. Dùng explicit origins hoặc origin patterns có kiểm soát.

6. Interview Framework

SurfaceJWT là gì?

JWT là token format gồm header, payload và signature. Signature giúp phát hiện token bị sửa. Payload đọc được, không phải encrypted, nên không chứa secret. JWT thường dùng cho stateless authentication.

DeepJWT invalid thì filter chain xử lý thế nào?

JWT filter không set SecurityContext nếu token invalid. Request đi tiếp, anonymous authentication có thể được set, rồi AuthorizationFilter kiểm tra rule. Nếu endpoint cần auth, ExceptionTranslationFilter gọi entry point trả 401.

ArchitectureThiết kế auth cho SaaS multi-tenant

Access token ngắn hạn chứa userId và high-level roles; workspace membership lấy từ DB/cache server-side để revoke nhanh. Method security gọi service kiểm tra membership/resource ownership. Refresh token lưu server-side theo device, có rotation và revoke. Audit security events nhưng không log token.

Checklist Thành Thạo

  • Vẽ được request qua SecurityFilterChain và vị trí JWT filter.
  • Giải thích được JWT payload readable, logout/revoke cần cơ chế riêng.
  • Phân biệt hasRole và hasAuthority, đặc biệt prefix ROLE_.
  • Viết được @PreAuthorize check cả role lẫn ownership.
  • Giải thích được CORS preflight và vì sao server phải cấu hình CORS.
  • Phân biệt 401/403 và biết vai trò của AuthenticationEntryPoint/AccessDeniedHandler.
  • Nắm OAuth2/OIDC ở mức resource server: verify JWT qua JWKS, map claims, xử lý key rotation.
  • Viết được security tests cho unauthenticated, forbidden và đúng ownership.
