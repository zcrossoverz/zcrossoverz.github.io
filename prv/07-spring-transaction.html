Spring Transaction — Isolation, Propagation, Rollback

Junior add @Transactional vô tội vạ — không hiểu propagation, không biết checked exception không rollback default, không hiểu tại sao "order created" log có nhưng DB không có row. File này bóc internals + 5 anti-pattern mất tiền/data.

CORE ★ 1 tuần đọc + practice nested + event listener Domain: E-commerce checkout

⚠️ Cảnh báo "hiểu giả"

  • Add @Transactional mọi service method — không phân biệt read vs write
  • Biết "rollback khi exception" — không biết checked exception KHÔNG rollback default
  • Dùng REQUIRES_NEW "để chắc chắn audit ghi được" — không hiểu nó tạo physical connection mới (deadlock risk)
  • Đặt @Transactional trên private method hoặc gọi this.method() trong cùng class → không có hiệu lực (vì proxy)
🚨 3 cấp

L1: add @Transactional service method. Không biết propagation.
L2: hiểu 7 propagation, chọn theo case. rollbackFor=Exception.class cho checked. Read-only cho query.
L3: design transaction boundary qua aggregate. Dùng @TransactionalEventListener(AFTER_COMMIT) cho side effect. Debug long transaction qua DB lock.

1. Bài toán thực tế — Order created, audit log biến mất CORE ★

📖 Tình huống — E-commerce checkout
// ❌ Code junior
@Service
public class CheckoutService {
    public Order checkout(CheckoutRequest req) {
        Order order = orderRepo.save(new Order(req));      // (1) DB write
        inventoryService.reserve(req.items());             // (2) external call
        auditService.log("Order created: " + order.getId()); // (3) DB write audit
        return order;
    }
}

Vấn đề observed:

  • App log: "Order created: ORD_123"
  • DB orders table: có row ORD_123 ✅
  • DB audit_log table: KHÔNG có row tương ứng ❌
  • Inventory: stock đã trừ

Root cause: Method không có @Transactional → mỗi DB call commit riêng (auto-commit). Nếu auditService.log() throw exception sau khi order đã commit → audit fail nhưng order vẫn còn. Customer service không trace được.

Junior fix sai: add @Transactional rồi nghĩ xong.

@Transactional
public Order checkout(CheckoutRequest req) { ... }   // ← chưa đủ

Vấn đề mới: inventoryService.reserve() là external call (HTTP/Kafka) — nằm TRONG transaction = giữ DB lock cả thời gian network call. 1 user 5 giây = lock connection 5 giây. 100 user concurrent = pool exhausted.

Fix đúng (3 nguyên tắc):

@Service
public class CheckoutService {
    @Transactional   // 1. Transaction boundary rõ
    public Order checkout(CheckoutRequest req) {
        Order order = orderRepo.save(new Order(req));
        auditService.log(order);                       // 2. DB write trong cùng TX
        events.publish(new OrderCreated(order));       // 3. External call SAU commit qua event
        return order;
    }
}

@Component
public class InventoryEventHandler {
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    @Async
    public void onOrderCreated(OrderCreated event) {
        inventoryService.reserve(event.items());       // external call NGOÀI TX
    }
}

Khác biệt: DB write atomic (order + audit cùng commit hoặc cùng rollback). External call chạy sau commit qua event → không giữ DB lock.

⏸ Pause & Predict
  1. Vì sao "external call trong transaction" là anti-pattern? Hậu quả production cụ thể?
  2. Nếu auditService.log() throw IOException (checked) — transaction có rollback không với default config?
  3. @TransactionalEventListener(AFTER_COMMIT) nếu listener throw exception — transaction đã commit có rollback không?
Đáp án ở Section 2.3 (rollback rules) và Section 3 (event listener).

2. Mental Model CORE ★

Transaction = unit of work có ACID: Atomicity (all-or-nothing), Consistency (invariant maintained), Isolation (concurrent TX không thấy nhau), Durability (commit = persistent).

2.1 Isolation Levels — Tường ngăn giữa transactions

LevelDirty readNon-repeatable readPhantom readUse case
READ_UNCOMMITTED✅ có✅✅Gần như không dùng
READ_COMMITTED❌✅✅Default Postgres/Oracle
REPEATABLE_READ❌❌✅ (Postgres) / ❌ (MySQL)Default MySQL InnoDB
SERIALIZABLE❌❌❌Financial, cần strict ordering

3 anomaly:

  • Dirty read: TX A đọc data TX B chưa commit. B rollback → A có data ma.
  • Non-repeatable read: TX A đọc row 2 lần, giữa lúc đó TX B update + commit → A thấy 2 giá trị khác.
  • Phantom read: TX A query range 2 lần, giữa lúc đó TX B insert row mới fit range → A thấy "phantom" rows.

Rule: default DB (READ_COMMITTED hoặc REPEATABLE_READ) đủ cho 95% case. Tăng lên SERIALIZABLE chỉ khi cần (vd: financial reconciliation). Cao isolation = chậm + nhiều lock.

2.2 Propagation — 7 loại

PropagationHiện có TX?Không có TX?
REQUIRED (default)JoinCreate new
SUPPORTSJoinRun without TX
MANDATORYJoinThrow exception
REQUIRES_NEWSuspend, create new physical TXCreate new
NOT_SUPPORTEDSuspend, run withoutRun without
NEVERThrow exceptionRun without
NESTEDSavepoint trong TX hiện tạiCreate new
🎯 Khi nào dùng cái nào
  • REQUIRED: 95% case business logic. Default.
  • REQUIRES_NEW: audit log MUST commit dù parent rollback (vd: log failed login attempt). Cẩn thận: dùng connection thứ 2 từ pool, deadlock risk.
  • NESTED: batch processing — fail 1 item, savepoint rollback chỉ item đó, tiếp tục batch.
  • SUPPORTS / NOT_SUPPORTED: rất hiếm dùng.
@Service
public class AuditService {
    // REQUIRES_NEW — luôn commit dù caller rollback
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void logFailedLogin(String username) {
        auditRepo.save(new AuditEntry("LOGIN_FAILED", username));
    }
}

2.3 Rollback Rules — Bẫy lớn nhất

🚨 Default rollback rule

Spring rollback khi gặp RuntimeException hoặc Error. KHÔNG rollback cho checked exception.

// ❌ IOException là checked → KHÔNG rollback default
@Transactional
public Order checkout(CheckoutRequest req) throws IOException {
    Order o = orderRepo.save(...);
    sendInvoiceEmail(o);   // throws IOException
    // Nếu email fail → exception bubble up, NHƯNG order vẫn committed!
    return o;
}

// ✅ Explicit rollbackFor
@Transactional(rollbackFor = Exception.class)
public Order checkout(...) throws IOException { ... }

// ✅ Hoặc wrap checked thành unchecked
@Transactional
public Order checkout(...) {
    try { sendInvoiceEmail(o); }
    catch (IOException e) { throw new EmailException(e); }
}

Modern best practice: dùng unchecked exception cho business error. Convert checked thành domain exception layer thấp. Tránh throws IOException trong service signature.

3. Production Patterns CORE ★

3.1 Read-Only Transaction

@Transactional(readOnly = true)
public List<Order> findByCustomer(String customerId) {
    return orderRepo.findByCustomer(customerId);
}

Tại sao quan trọng: Hibernate skip dirty checking (không snapshot entity), DB có thể route sang read replica, optimize internal flush. Cho mọi query method.

3.2 NESTED — Batch processing

@Transactional
public BatchResult processBatch(List<Order> orders) {
    int success = 0, fail = 0;
    for (Order o : orders) {
        try {
            processOne(o);     // NESTED — savepoint
            success++;
        } catch (Exception e) {
            fail++;            // savepoint rollback, batch tiếp tục
        }
    }
    return new BatchResult(success, fail);
}

@Transactional(propagation = Propagation.NESTED)
public void processOne(Order o) { ... }

Cần JDBC driver support savepoint (Postgres/MySQL/Oracle ok).

3.3 @TransactionalEventListener — Event chỉ publish SAU commit

// Publisher
@Service
public class OrderService {
    @Transactional
    public Order create(CreateRequest req) {
        Order o = orderRepo.save(new Order(req));
        events.publishEvent(new OrderCreated(o));   // chưa fire listener
        return o;
        // Sau khi method exit + transaction commit thành công → listener mới chạy
    }
}

// Listener
@Component
public class OrderEventHandler {
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    @Async
    public void onOrderCreated(OrderCreated event) {
        emailService.sendConfirmation(event.order());
        analyticsService.track(event.order());
    }
}
⚠️ Quan trọng
  • AFTER_COMMIT: listener chỉ chạy nếu TX commit thành công → email không gửi nếu order rollback
  • BEFORE_COMMIT: chạy trước commit, có thể block commit
  • AFTER_ROLLBACK: chạy khi rollback — cho cleanup compensating
  • @Async để listener không giữ caller. Listener exception KHÔNG rollback TX đã commit.

4. 5 Anti-Patterns CORE ★

❌ #1 — Checked exception không rollback

Đã giải thích ở 2.3. Fix: rollbackFor=Exception.class hoặc dùng unchecked.

❌ #2 — Long transaction giữ lock
// ❌ External call + DB write trong cùng TX
@Transactional
public Order checkout(...) {
    Order o = orderRepo.save(...);
    paymentGateway.charge(...);    // 5s HTTP call
    emailService.send(...);         // 2s SMTP
    // Tổng 7s giữ DB connection + row lock
}

Symptom: connection pool exhausted dưới load. hikaricp_connections_pending spike.

Fix: tách DB write vs external. External qua @TransactionalEventListener(AFTER_COMMIT).

❌ #3 — N+1 query trong transaction
@Transactional(readOnly = true)
public List<OrderSummary> summaries() {
    return orderRepo.findAll().stream()        // 1 query
        .map(o -> new OrderSummary(o.getCustomer().getName(), ...))  // N queries cho customer
        .toList();
}

1 query + N lazy load = giữ TX rất lâu. Fix: JOIN FETCH hoặc @EntityGraph.

❌ #4 — REQUIRES_NEW vô tội vạ

Mỗi REQUIRES_NEW = connection mới từ pool. Nested 5 lần REQUIRES_NEW = 5 connection. Deadlock risk khi 2 TX cùng cần connection thứ 2.

Rule: chỉ REQUIRES_NEW khi audit/logging cần commit độc lập. Default REQUIRED.

❌ #5 — @Transactional trên private hoặc this.method() trong cùng class
@Service
public class OrderService {
    public void outer() {
        this.inner();     // ← KHÔNG qua proxy → @Transactional bị ignore
    }

    @Transactional
    public void inner() { ... }   // chỉ work khi gọi từ NGOÀI class

    @Transactional
    private void privateMethod() { ... }   // ❌ private không proxy được
}

Root cause: Spring AOP dùng proxy. Self-invocation bypass proxy.

Fix: tách inner method sang service khác, hoặc inject self qua @Autowired private OrderService self; rồi gọi self.inner().

🎯 3 Cấp Mastery CORE ★

Propagation choice

L1

Default REQUIRED. Add @Transactional mọi service method.

L2

Hiểu 7 loại. Dùng REQUIRES_NEW cho audit độc lập, NESTED cho batch, readOnly cho query.

L3

Design transaction boundary qua DDD aggregate. @TransactionalEventListener cho side effect. Monitor long TX qua DB lock view.

Rollback rules

L1

Nghĩ "exception nào cũng rollback". Bị bug checked exception không rollback.

L2

Biết default chỉ rollback unchecked. Dùng rollbackFor khi cần. Tránh checked trong service signature.

L3

Design exception hierarchy domain-unchecked. Custom RollbackException hierarchy cho compliance/audit case.

🚨 Symptom → Debug Recipes CORE ★

🚨 Recipe #1 — "Log nói thành công, DB không có"

Log INFO trong TX trước commit. Sau đó throw exception → rollback. Log đã ghi, DB không.

Fix: Log INFO ở @TransactionalEventListener(AFTER_COMMIT), không phải giữa method.

🚨 Recipe #2 — "@Transactional bị ignore"

3 nguyên nhân thường:

  1. Self-invocation (this.method() trong cùng class)
  2. Method private hoặc final — proxy không override được
  3. Class không là Spring bean (new bằng tay)
🚨 Recipe #3 — "Connection pool exhausted"

Symptom: hikaricp_connections_pending spike, app timeout.

Root cause typical: long TX (external call trong TX), N+1 query, hoặc REQUIRES_NEW vô tội vạ giữ multiple connection.

-- Postgres tìm long-running TX
SELECT pid, now() - xact_start AS duration, query
FROM pg_stat_activity WHERE state <> 'idle' AND xact_start IS NOT NULL
ORDER BY duration DESC LIMIT 10;
🚨 Recipe #4 — "Deadlock detected"

2 TX update cùng 2 row theo thứ tự ngược → deadlock.

Fix: luôn update theo thứ tự cố định (vd: order theo id ASC). Hoặc dùng SELECT FOR UPDATE NOWAIT + retry.

🎤 Interview Framework CORE ★

Q: 4 Isolation level + 3 anomaly?

READ_UNCOMMITTED / READ_COMMITTED / REPEATABLE_READ / SERIALIZABLE. Anomaly: dirty read, non-repeatable read, phantom read. Postgres default READ_COMMITTED, MySQL InnoDB default REPEATABLE_READ.

Q: REQUIRES_NEW vs NESTED?

REQUIRES_NEW: suspend parent TX, mở physical TX mới với connection riêng từ pool. Commit/rollback độc lập. Deadlock risk khi cần connection thứ 2.

NESTED: savepoint trong TX hiện tại, dùng cùng connection. Rollback savepoint không rollback toàn TX. Cần JDBC support savepoint.

Q: Design transaction strategy cho checkout flow (order + payment + inventory + email)?

  1. DB write (order): @Transactional ngắn — chỉ DB
  2. Payment external: ngoài TX, dùng saga + outbox cho consistency
  3. Inventory reserve: event-driven AFTER_COMMIT
  4. Email/notification: @TransactionalEventListener(AFTER_COMMIT) + @Async
  5. Audit log: cùng TX (REQUIRED) — chỉ giữ nếu order commit
  6. Compensation: AFTER_ROLLBACK handler cho cleanup nếu cần
