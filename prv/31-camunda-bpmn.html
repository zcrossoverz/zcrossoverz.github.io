Camunda & BPMN — Workflow Engine Đúng Cách

BPMN nhìn như "vẽ flowchart cho đẹp" — hiểu giả ở đây làm workflow thành mớ rối khó sửa. File này giải thích process engine quản state & persistence ra sao (vì sao recover được sau crash), user task khác service task ở đâu, và vì sao service task phải idempotent — đọc một lần là biết khi nào nên dùng engine. Domain: ShopFlow — quy trình duyệt hoàn tiền (refund approval) của sàn e-commerce.

🎯 Mục tiêu
Mô hình quy trình có người + tự động, bền với crash, dễ sửa
📅 Thời lượng
1 tuần · Sau #18 Event-Driven, #17 Microservices
📚 Prereq
Spring Boot · async · state machine
🔥 Độ ưu tiên

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "BPMN chỉ là sơ đồ để giải thích cho business, code vẫn viết tay."
Thực tế: Với một process engine (Camunda), BPMN là executable — engine chạy chính sơ đồ đó như một chương trình: nó lưu state ở mỗi bước vào DB, recover sau crash, và biết đợi người duyệt hay đợi timer. Coi BPMN chỉ là "tài liệu" là bỏ phí giá trị chính của engine: persistence của trạng thái lâu dài + khả năng theo dõi.
Tưởng hiểu: "Service task chỉ là gọi một hàm, chạy lại cũng không sao."
Thực tế: Engine đảm bảo at-least-once — khi crash/timeout giữa chừng, nó retry service task. Nếu task không idempotent (vd gọi PSP hoàn tiền), retry sẽ hoàn 2 lần = mất tiền. Vì vậy service task BẮT BUỘC idempotent.
Tưởng hiểu: "Nhồi business logic vào expression trong BPMN cho gọn."
Thực tế: Logic phức tạp nhét trong expression XML thì không unit test được, không debug được, không version đẹp trong Git. BPMN nên giữ luồng (thứ tự, rẽ nhánh, chờ); logic để trong code (delegate/external task) — nơi test được.

1. Bài toán thực tế — refund approval rối CORE ★

ShopFlow xử lý hoàn tiền với quy tắc: hoàn < 500K thì tự duyệt; ≥ 500K cần quản lý duyệt tay; nếu quá 48h chưa duyệt thì tự escalate lên cấp cao hơn; sau khi duyệt thì gọi PSP hoàn tiền. Bản tự code bằng if-else + cron job + cột flag trạng thái gặp ba vấn đề: (1) khó biết một đơn đang ở bước nào (phải đọc nhiều flag rời rạc); (2) khi service restart giữa lúc một đơn đang chờ duyệt, state nằm rải rác dễ mất hoặc sai; (3) bước "gọi PSP hoàn tiền" khi bị retry sau lỗi mạng thì hoàn tiền 2 lần.

⏸ Pause & Predict — trước khi đọc lời giải
Viết ra giấy:
  1. Quy trình này có "chờ người duyệt" (có thể vài ngày) và "chờ timer 48h" — vì sao tự code bằng flag + cron dễ mất state khi service restart?
  2. Bước "gọi PSP hoàn tiền" bị retry sau crash → hoàn 2 lần. Thuộc tính nào của task ngăn được?
  3. Process engine cho bạn cái gì mà if-else + DB flag không cho (gợi ý: 2 thứ)?
Đáp án: engine lưu state mỗi bước vào DB nên restart không mất (2.1); service task phải idempotent vì engine at-least-once (2.5); engine cho persistence của trạng thái lâu dài + visibility (đơn đang ở bước nào). Đừng đọc tiếp khi chưa thấy giá trị "state bền".

2.1 BPMN + process engine — cơ chế state CORE ★

BPMN (Business Process Model and Notation) là một ngôn ngữ vẽ quy trình bằng các ký hiệu chuẩn: start/end event, task (việc cần làm), gateway (rẽ nhánh — exclusive đi một đường, parallel đi nhiều đường), sequence flow (mũi tên nối), timer/message event (chờ thời gian/tin nhắn). Điểm then chốt: với một process engine như Camunda, BPMN không chỉ là hình vẽ mà là thứ chạy được.

Cơ chế quan trọng nhất cần hiểu là cách engine quản state. Khi bạn khởi một quy trình, engine tạo một process instance (một "thẻ chạy" cho một đơn refund cụ thể) và di chuyển một token qua các bước theo sơ đồ. Tại mỗi wait state — chỗ phải đợi (đợi người duyệt, đợi timer, đợi message) — engine lưu toàn bộ trạng thái (token đang ở đâu + các biến của process) xuống database, rồi giải phóng thread. Hệ quả: một quy trình chờ người duyệt 3 ngày không giữ một thread nào — nó chỉ là một dòng trong DB; và nếu service crash rồi khởi động lại, engine load lại state từ DB và tiếp tục đúng chỗ đang dừng. Đây chính là điều "if-else + flag + cron" tự code rất khó làm đúng.

L1 — Biết tên

Vẽ được sơ đồ BPMN. Symptom hiểu giả: không giải thích được engine lưu state ở đâu hay vì sao recover được sau crash.

L2 — Explain được

Engine persist token + biến process vào DB tại mỗi wait state (user task, timer, message). Quy trình lâu dài (chờ duyệt 3 ngày) không giữ thread — chỉ là row trong DB. Restart → load lại → tiếp tục đúng chỗ.

L3 — Apply được

Thiết kế quy trình: tách wait state rõ ràng, service task idempotent, dùng external task worker để scale; chọn engine khi quy trình dài + có người + cần audit/visibility; tự code khi luồng ngắn đơn giản.

2.2 User task vs Service task CORE ★

Hai loại task khác nhau ở ai thực hiện và engine hành xử thế nào:

2.3 Timer + error handling SUPPLEMENTARY

Timer event cho phép quy trình "chờ một khoảng thời gian" (48h) hoặc tới một mốc, rồi tự kích hoạt một nhánh (escalate/timeout) — bạn không cần viết cron riêng và tự quản. Engine lưu thời điểm hết hạn vào DB và tự đánh thức process instance đúng lúc.

Error handling phân biệt hai loại lỗi: business error (vd PSP từ chối hoàn tiền — đây là một kết quả nghiệp vụ hợp lệ, nên dùng error boundary event để rẽ sang nhánh xử lý như "chuyển sang xử lý thủ công"); và technical error (vd timeout mạng — đây là sự cố tạm thời, engine nên retry với backoff, và nếu hết số lần retry thì tạo một incident cần người can thiệp). Phân biệt đúng giúp quy trình vừa robust vừa không nuốt lỗi.

2.4 Message correlation SUPPLEMENTARY

Khi PSP gửi callback "đã hoàn tiền cho refund #R123" về, engine cần biết callback này thuộc về process instance nào trong hàng nghìn instance đang chờ. Đó là việc của message correlation: bạn khai báo một business key (vd refundId) làm khoá khớp; khi message đến với refundId = R123, engine tìm đúng instance đang chờ message với business key đó và đánh thức nó. Nếu correlation sai (vd khớp nhầm khoá hoặc không có instance chờ), message rơi vào instance khác hoặc bị bỏ — gây bug khó tìm.

2.5 Vì sao service task phải idempotent CORE ★

Process engine đảm bảo at-least-once cho service task — nghĩa là một task được đảm bảo chạy ít nhất một lần, nhưng có thể nhiều hơn. Vì sao "có thể nhiều hơn": giả sử service task gọi PSP hoàn tiền thành công, nhưng ngay sau đó (trước khi engine kịp ghi nhận "đã xong" vào DB) service crash. Khi khởi động lại, engine thấy task này "chưa được đánh dấu hoàn thành" nên chạy lại — và lần này PSP hoàn tiền lần thứ hai. Đây không phải bug của engine mà là bản chất của hệ phân tán: không thể đảm bảo "đúng một lần" giữa hai hệ thống (engine + PSP) khi có thể crash giữa chừng. Cách đúng là làm service task idempotent — gọi PSP với một idempotency key (vd refundId) để PSP nhận diện và chỉ hoàn tiền một lần dù bị gọi nhiều lần. Đây là lý do mọi service task (nhất là cái có side-effect tiền/thông báo) bắt buộc idempotent.

3.1 Process definition (BPMN) SUPPLEMENTARY

refund-approval.bpmn (mô tả luồng):

  (start) → [exclusive gateway: amount < 500K ?]
              ├─ yes → (service task: auto-approve)
              └─ no  → (user task: manager review)   ← WAIT STATE, lưu DB
                          ⏲ timer boundary 48h → (service task: escalate)
              ↓
          (service task: callPspRefund)              ← idempotent, engine có thể retry
              ↓ (error boundary: PspDeclined → user task: manual handling)
          (message event: chờ PSP callback "refunded")  ← correlate theo refundId
              ↓
            (end)

// gateway condition để TRONG flow; business logic NẶNG để trong delegate (3.2), không nhồi expression.

3.2 Service task delegate (idempotent) CORE ★

@Component("callPspRefund")
class CallPspRefundDelegate implements JavaDelegate {
  @Override public void execute(DelegateExecution exec) {
    String refundId = (String) exec.getVariable("refundId");
    BigDecimal amount = (BigDecimal) exec.getVariable("amount");

    // engine đảm bảo AT-LEAST-ONCE → có thể chạy lại sau crash/timeout
    // → BẮT BUỘC idempotent: PSP nhận idempotency key, gọi 2 lần chỉ hoàn 1 lần
    PspRefundResult r = pspClient.refund(
        RefundRequest.builder().idempotencyKey(refundId).amount(amount).build());

    if (r.status() == DECLINED)
      throw new BpmnError("PspDeclined");          // → error boundary event xử lý (business error)
    exec.setVariable("pspRefundRef", r.reference());
  }
}

3.3 External task worker + correlation SUPPLEMENTARY

// External task pattern: worker PULL task từ engine → scale độc lập, ngôn ngữ tuỳ ý
@ExternalTaskSubscription("callPspRefund")
class RefundWorker implements ExternalTaskHandler {
  public void execute(ExternalTask task, ExternalTaskService svc) {
    try {
      var ref = pspClient.refund(task.getVariable("refundId"), ...);  // idempotent
      svc.complete(task, Map.of("pspRefundRef", ref));
    } catch (TransientException e) {                                   // technical error → retry
      svc.handleFailure(task, "psp timeout", e.toString(),
          task.getRetries()==null?3:task.getRetries()-1, 5000L);      // còn retry + backoff
    }
  }
}

// PSP callback đến → correlate message tới ĐÚNG instance theo business key
runtimeService.createMessageCorrelation("PspRefunded")
    .processInstanceBusinessKey(refundId)         // khớp đúng process instance đang chờ
    .setVariable("settledAt", Instant.now())
    .correlate();

4. Anti-patterns CORE ★

🔧 Counter-example — khi dùng BPMN engine là SAI / dùng sai
  • Service task không idempotent: engine retry → hoàn tiền/gửi mail 2 lần. Mọi service task phải idempotent (key).
  • Task quá mịn: chia 30 task cho một việc tuyến tính đơn giản → sơ đồ rối, mỗi bước một transaction (chậm). Engine hợp với quy trình có wait state & con người, không phải mọi vòng for.
  • Business logic trong expression XML: không test/debug được. Để trong delegate/external task.
  • Dùng engine cho luồng ngắn không có người: over-engineering — một REST call đồng bộ đơn giản không cần workflow engine (thêm cả một DB engine + học BPMN).

Quy tắc: chọn workflow engine khi quy trình dài, có bước con người, cần audit/visibility, nhiều rẽ nhánh & timer. Luồng ngắn thuần tự động → code thường rẻ & rõ hơn (hoặc Saga nhẹ cho microservices).

5. 3-tier interview framework

Tầng 1 — SurfaceMid gate

Q: User task vs service task?

User task = engine tạo việc cho con người & chờ họ complete (wait state, có thể lâu, lưu DB). Service task = engine gọi code tự động; phải idempotent vì engine đảm bảo at-least-once (có thể retry).

Tầng 2 — Deep DiveSenior gate

Q: Vì sao dùng Camunda thay vì tự code refund flow?

Quy trình dài (chờ người duyệt, timer 48h) → engine persist state vào DB ở mỗi wait state, recover sau crash, không giữ thread; visibility (đơn đang ở bước nào); audit; rẽ nhánh + escalation khai báo được. Tự code = flag + cron dễ mất state, khó theo dõi.

Tầng 3 — ArchitectureSenior+

Q: Khi nào KHÔNG dùng workflow engine?

Luồng ngắn, đồng bộ, không có con người, ít rẽ nhánh → code thường rẻ & rõ hơn. Engine thêm hạ tầng (DB engine, học BPMN). Dùng khi quy trình dài + người + audit + nhiều biến thể; cân nhắc orchestration nhẹ (Saga) cho microservices thuần tự động.

🔧 Lab bắt buộc — build & phá (2 bài)

Lab 1 — Refund process. Vẽ & chạy một BPMN: start → exclusive gateway (<500K tự duyệt / ≥500K user task) → service task gọi "PSP refund" (mock) → end. Thêm timer boundary 48h trên user task → escalate. Tiêu chí: kill app giữa lúc chờ duyệt, khởi động lại, chứng minh instance vẫn ở đúng bước (state đã lưu DB).

Lab 2 — Idempotency của service task. Làm service task "PSP refund" non-idempotent (cộng tiền mỗi lần gọi). Ép nó throw sau khi đã hoàn → engine retry → hoàn 2 lần. Sửa bằng idempotency key. Tiêu chí: giải thích at-least-once của engine và vì sao không thể "exactly-once" giữa engine và PSP.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. Process engine lưu state ở đâu (wait state → DB)? Vì sao recover được sau crash và không giữ thread khi chờ lâu?
  2. User task vs service task khác gì? Cái nào là wait state?
  3. Vì sao service task BẮT BUỘC idempotent (at-least-once)? Vì sao không thể exactly-once?
  4. Message correlation giải quyết vấn đề gì khi PSP callback về?
  5. Khi nào KHÔNG nên dùng workflow engine?
Bí câu nào = quay lại đúng section. Trôi cả 5 không nhìn tài liệu = biết khi nào dùng & dùng đúng engine.
