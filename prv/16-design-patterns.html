Patterns rất dễ "đọc trôi" — feels obvious sau khi xem giải pháp. Nhưng recognize pattern trong code đồng đội + trong yêu cầu mới mới là kỹ năng thật. 5 quy tắc:

  1. ⏸ Pause & Predict: trước khi đọc giải pháp, tự design solution. So sánh với pattern.
  2. 🧠 Misconception Check: patterns có nhiều myth (overuse, factory cho mọi thứ)
  3. 🔬 Thử Phá Vỡ: nghĩ "pattern này fail khi nào?" trước khi xem analysis

  5. 🎓 Feynman: cuối mỗi section, đóng máy, giải thích "vì sao pattern này, không phải pattern khác" cho người không biết code

Vì Sao Patterns Quan Trọng?

2 sai lầm phổ biến với design patterns:

  1. Pattern Fever: học GoF rồi áp dụng mọi nơi → over-engineer simple problems
  2. Pattern Allergy: tránh patterns → code rối, junior khác không hiểu intent
🎯 Cách dùng đúng

Patterns là solution name cho recurring problem. Bạn không "đi tìm chỗ áp dụng Strategy" — bạn gặp 5 phương thức tính phí khác nhau và recognize đây là vấn đề Strategy giải quyết tốt. Mục tiêu: học để recognize pattern trong code đồng đội + trong yêu cầu mới, không phải để "thêm pattern vào code".

📜 Lịch sử Design Patterns — Từ kiến trúc xây dựng đến software
1977 — Christopher Alexander (architect, không phải software engineer) publish "A Pattern Language" — book về design patterns cho kiến trúc xây dựng. Mô tả 253 patterns cho design cities, buildings, rooms. Cốt lõi: "every pattern describes a problem which occurs over and over again... and then describes the core of the solution".

1987 — Kent Beck và Ward Cunningham apply ideas của Alexander vào software, present tại OOPSLA conference. First documented use trong programming context.

1994 — "Gang of Four" (Erich Gamma, Richard Helm, Ralph Johnson, John Vlissides) publish "Design Patterns: Elements of Reusable Object-Oriented Software". 23 patterns được catalog chính thức. Sách dùng C++ và Smalltalk, nhưng patterns universal.

1995-2000s — Patterns explode trong Java community (Sun Microsystems, IBM). EJB, J2EE patterns book series. Sometimes over-applied → "Pattern Fever" era.

2004 — Joshua Bloch publish "Effective Java" — moderate approach. Item 1: "Consider static factory methods instead of constructors". Item 2: "Consider a builder when faced with many constructor parameters". Patterns from GoF revisited với Java-specific lens.

2010s — Spring Framework biến patterns thành built-in: @Component (Factory + Singleton), @EventListener (Observer), @Async/@Transactional (Decorator via AOP). Patterns vẫn matter, nhưng implementation invisible.

2020s — Functional programming influence: nhiều patterns (Strategy, Command, Visitor) trở thành lambdas/functions. Java 8+ Function<T,R> thay class implement Strategy interface.

Insight cốt lõi: GoF patterns không phải "rules to follow" — là "names for solutions that already work". Recognize chúng trong existing codebases (Spring source, JDK) giúp bạn understand intent nhanh hơn. Apply chúng thoughtfully để code đồng đội recognize cùng intent.

1. Bài Toán Thực Tế — Shipping Fee Của Sàn TMĐT Mọc Tự Phát

Năm 1 — Chỉ có Standard Shipping

Sàn ShopVN khởi nghiệp với 1 phương thức ship: Standard (giao 3-5 ngày). Code 50 dòng trong OrderService.calculateShippingFee() — fee = base 30,000 VND + 5,000 VND/kg.

Năm 2 — Express Shipping

Add Express (giao 1-2 ngày, fee gấp 1.5x). OrderService giờ có if-else:

// Year 2 code
public Money calculateShippingFee(Order order) {
    String type = order.shippingType();
    if (type.equals("STANDARD")) {
        return base.add(perKg.multiply(order.weightKg()));
    } else if (type.equals("EXPRESS")) {
        return base.add(perKg.multiply(order.weightKg())).multiply(1.5);
    }
    throw new IllegalArgumentException();
}

Năm 3 — Same-Day, International, COD-Pickup

Bây giờ OrderService.calculateShippingFee() dài 800 dòng, 6 nesting levels:

Năm 4 — Requirement: "Add Pickup-At-Store trong 1 tuần, không sửa code cũ"

🔥 Triệu chứng của code không có pattern
  • Thêm method #6 phải hiểu toàn bộ 800 dòng để không break
  • Regression test cả 5 methods cũ → 2 ngày test manual
  • Fee logic scatter: 1 phần trong service, 1 phần trong DB config, 1 phần trong DHL SDK
  • Junior touch nhầm tab → SameDay tính nhầm fee như Express → mất tiền
  • Unit test class này = setup 30 mock objects, test runtime > 60s
  • Code review > 3 ngày vì reviewer phải mentally trace 800-line method
⏸ Pause & Predict — BẮT BUỘC dừng
Trước khi đọc giải pháp, tự trả lời (viết ra giấy):
  1. Pattern nào tách fee calculation logic để mỗi shipping method độc lập?
  2. Làm sao tránh massive if-else chain mà không tạo "factory cho mọi thứ"?
  3. Processor objects nên create 1 lần (singleton) hay per-request? Vì sao?
  4. Làm sao test shipping logic mà không gọi thật DHL/FedEx API?
  5. Pattern nào cho phép add logging/metrics/retry cho mọi shipping methods mà không sửa từng class?
Trả lời được 4/5 → solid foundation patterns. <3 → đọc kỹ.

2. Strategy Pattern CORE

★ CORE — Pattern phổ biến nhất production. Xuất hiện 90% Spring codebase. Replace if-else với polymorphism.
📜 Strategy Pattern — From GoF 1994
GoF Definition (1994): "Define a family of algorithms, encapsulate each one, and make them interchangeable. Strategy lets the algorithm vary independently from clients that use it."

Original example trong GoF book: text formatting algorithms (line breaking) — different strategies cho different output devices.

Java/Spring evolution: với functional interfaces từ Java 8, Strategy có thể là 1 lambda Function<T,R> thay vì interface + class. Spring @Qualifier + List<StrategyInterface> = idiomatic Strategy injection.

Insight: bất kỳ chỗ nào code có if (type.equals(X)) ... else if ... với 4+ branches, mỗi branch có logic không trivial — đó là signal cho Strategy.
Strategy: 1 interface, N implementations. Context không biết implementation cụ thể.

Implementation Full Depth

// 1. Interface định nghĩa "task"
public interface ShippingFeeStrategy {
    Money calculateFee(ShippingContext ctx);
    ShippingType supportedType();  // Để factory map đúng strategy
}

public record ShippingContext(
    Money declaredValue,
    BigDecimal weightKg,
    Address from,
    Address to,
    boolean fragile,
    boolean isInternational
) {}

// 2. Implementations — mỗi method là 1 class
@Component
public class StandardShippingStrategy implements ShippingFeeStrategy {
    private static final Money BASE = Money.of(30_000, VND);
    private static final Money PER_KG = Money.of(5_000, VND);

    @Override
    public Money calculateFee(ShippingContext ctx) {
        Money byWeight = PER_KG.multiply(ctx.weightKg().intValue());
        Money fee = BASE.add(byWeight);

        // Surcharge cho fragile items
        if (ctx.fragile()) {
            fee = fee.add(Money.of(20_000, VND));
        }
        return fee;
    }

    @Override
    public ShippingType supportedType() { return ShippingType.STANDARD; }
}

@Component
public class ExpressShippingStrategy implements ShippingFeeStrategy {
    private final ShippingFeeStrategy standardCalc;
    private static final BigDecimal EXPRESS_MULTIPLIER = BigDecimal.valueOf(1.5);

    public ExpressShippingStrategy(@Qualifier("standardShippingStrategy") ShippingFeeStrategy standardCalc) {
        this.standardCalc = standardCalc;
    }

    @Override
    public Money calculateFee(ShippingContext ctx) {
        // Compose strategy thay vì duplicate logic — Strategy có thể nested
        return standardCalc.calculateFee(ctx).multiply(EXPRESS_MULTIPLIER);
    }

    @Override
    public ShippingType supportedType() { return ShippingType.EXPRESS; }
}

@Component
public class SameDayShippingStrategy implements ShippingFeeStrategy {
    private static final Money INNER_CITY_FEE = Money.of(80_000, VND);

    @Override
    public Money calculateFee(ShippingContext ctx) {
        if (!ctx.to().isInnerCityHCM() && !ctx.to().isInnerCityHN()) {
            throw new UnsupportedShippingException(
                "Same-day chỉ trong nội thành HCM/HN, address: " + ctx.to());
        }
        // Flat fee, plus weight surcharge nếu > 5kg
        Money fee = INNER_CITY_FEE;
        if (ctx.weightKg().compareTo(BigDecimal.valueOf(5)) > 0) {
            BigDecimal extraKg = ctx.weightKg().subtract(BigDecimal.valueOf(5));
            fee = fee.add(Money.of(10_000, VND).multiply(extraKg.intValue()));
        }
        return fee;
    }

    @Override
    public ShippingType supportedType() { return ShippingType.SAME_DAY; }
}

@Component
public class InternationalShippingStrategy implements ShippingFeeStrategy {
    private final DhlApiClient dhl;
    private final FedexApiClient fedex;
    private final CircuitBreaker dhlCircuitBreaker;

    public InternationalShippingStrategy(DhlApiClient dhl, FedexApiClient fedex,
            CircuitBreakerRegistry registry) {
        this.dhl = dhl;
        this.fedex = fedex;
        this.dhlCircuitBreaker = registry.circuitBreaker("dhl");
    }

    @Override
    public Money calculateFee(ShippingContext ctx) {
        // Try DHL first, fallback to FedEx if circuit broken
        try {
            return dhlCircuitBreaker.executeSupplier(
                () -> dhl.quote(ctx.to().country(), ctx.weightKg(), ctx.declaredValue()));
        } catch (Exception e) {
            log.warn("DHL unavailable, fallback to FedEx", e);
            return fedex.quote(ctx.to().country(), ctx.weightKg(), ctx.declaredValue());
        }
    }

    @Override
    public ShippingType supportedType() { return ShippingType.INTERNATIONAL; }
}

// 3. Strategy factory — auto-discover all strategies
@Component
public class ShippingStrategyRegistry {
    private final Map<ShippingType, ShippingFeeStrategy> strategies;

    // Spring inject ALL @Component implementing ShippingFeeStrategy
    public ShippingStrategyRegistry(List<ShippingFeeStrategy> allStrategies) {
        this.strategies = allStrategies.stream()
            .collect(Collectors.toMap(
                ShippingFeeStrategy::supportedType,
                Function.identity(),
                (a, b) -> {
                    throw new IllegalStateException(
                        "Duplicate strategy for type: " + a.supportedType());
                }
            ));
    }

    public ShippingFeeStrategy get(ShippingType type) {
        ShippingFeeStrategy strategy = strategies.get(type);
        if (strategy == null) {
            throw new UnsupportedShippingException("No strategy for type: " + type);
        }
        return strategy;
    }
}

// 4. Context — không biết implementation cụ thể
@Service
public class CheckoutService {
    private final ShippingStrategyRegistry strategyRegistry;

    public Money totalCost(Order order) {
        Money items = order.itemsTotal();
        ShippingFeeStrategy strategy = strategyRegistry.get(order.shippingType());
        Money shipping = strategy.calculateFee(order.shippingContext());
        return items.add(shipping);
    }
}
🎯 Vì sao Strategy > if-else?
  • Open-Closed Principle: thêm method #6 (Pickup-At-Store) = thêm 1 class mới + Spring auto-discover. KHÔNG sửa CheckoutService. Code cũ không bị regression risk.
  • Testability: mỗi strategy test riêng. StandardShippingStrategyTest không cần biết Express tồn tại. Mock dependencies dễ.
  • Single Responsibility: mỗi class chỉ 1 reason to change. DHL đổi pricing → chỉ sửa InternationalShippingStrategy.
  • Polymorphism > conditional: code đọc tự nhiên hơn. strategy.calculateFee(ctx) rõ intent hơn switch(type).
  • Composable: Express dùng Standard + multiplier. Code reuse mà không inherit.
  • Auto-discovery: Spring inject List<ShippingFeeStrategy> tự pick up mọi @Component implementing interface — zero config thêm strategy.
🧠 Misconception — "Strategy cho mọi if-else"
"Tôi học Strategy, giờ replace mọi if-else trong code thành Strategy."
2-3 case đơn giản, logic 1-2 dòng → if-else đủ. Strategy có cost: thêm interface + N classes + factory + DI wiring. Benefits chỉ pay off khi:
(1) Algorithm variants > 4-5
(2) Mỗi variant có logic phức tạp (> 20 lines)
(3) Variants có thể thêm/sửa độc lập theo thời gian
(4) Cần testability per-variant

Ví dụ KHÔNG cần Strategy: Money fee = isVip ? Money.ZERO : Money.of(20_000, VND); — if-else 1-line, không thêm variant trong tương lai. Strategy đây = over-engineering.
🔬 Thử Phá Vỡ — Strategy có shared state

"Tôi cần track count requests per strategy. Add private int counter vào mỗi strategy class." Đoán: pattern nào sẽ break?

Click để xem analysis

Concurrent access bug. Spring beans default singleton-scoped → 1 instance share across all threads. counter++ không atomic → lost updates trong high concurrency.

Worse: strategies thường stateless trong design. Junior add field → break implicit contract → next dev assume stateless → bug subtle.

Fix options:

  1. AtomicInteger counter nếu cần count tại strategy
  2. Better: extract counting concern ra Decorator hoặc AOP aspect — strategies remain stateless
  3. Best: use Micrometer metrics — strategies không own monitoring concern

Insight: Strategies should be stateless để safe trong DI containers. Stateful concerns extract ra (Decorator, AOP, Observer).

🎯 Edge Cases — Strategy Production Reality
  • Strategy với external dependency: InternationalShippingStrategy call DHL API. API down → strategy throw. Wrap với circuit breaker, có fallback strategy.
  • Strategy selection key conflict: 2 strategies cùng supportedType() = STANDARD. Spring inject cả 2, registry build map throw. Fix: validate ở registry constructor + clear error message.
  • Strategy missing for new type: enum ShippingType.PICKUP_AT_STORE added, quên implement strategy → runtime exception khi user chọn. Fix: integration test verify mọi enum value có strategy.
  • Strategy throws unexpected exception: client expect UnsupportedShippingException, strategy throw NullPointerException. Convention: define exception types in interface contract; wrap raw exceptions.
  • Performance — Map lookup overhead: factory lookup O(1) but reflection-heavy nếu dùng @Qualifier dynamic. Cache Map<Type, Strategy> tại startup.
  • Strategy phụ thuộc context không có sẵn: SameDay cần current time để check business hours. Inject Clock (testable), không dùng System.currentTimeMillis() trực tiếp.
  • Strategy ordering matter: chain of strategies (try A, fallback B, fallback C). Use List<Strategy> ordered theo @Order annotation, không Set.
  • Hot-swap strategy at runtime: feature flag enable new strategy without restart. Use Spring @ConditionalOnProperty hoặc dynamic registry với refresh.

3. Factory Pattern CORE

★ CORE — Centralize object creation. Foundation cho DI containers. Strategy + Factory phổ biến đi cùng.
📜 Factory Pattern — Original GoF + Java evolution
GoF 1994 — Factory Method và Abstract Factory là 2 patterns riêng. Factory Method: subclass quyết định concrete class. Abstract Factory: family of related products.

Java idiom (Bloch 2001+): "Static factory method" — Effective Java Item 1. Integer.valueOf(int), List.of(...), Optional.of(...). Hơn constructor: tên có ý nghĩa, có thể cache instance, return subtype.

Spring Framework: BeanFactory và ApplicationContext = Factory pattern at framework level. @Component, @Bean = declarative factory.

Modern Java (16+): records với static factory methods phổ biến hơn constructor cho immutable types. Pattern matching giảm need cho some factory uses.

Insight: Factory benefit lớn khi: (1) creation logic phức tạp, (2) cần cache/reuse instances, (3) return type là interface (caller không biết concrete class), (4) cần testability (mock factory).
❌ Caller tự chọn — logic scatter

String type = request.shippingType();
ShippingFeeStrategy strategy;
if (type.equals("STANDARD"))
    strategy = new StandardShippingStrategy(...);
else if (type.equals("EXPRESS"))
    strategy = new ExpressShippingStrategy(
        new StandardShippingStrategy(...));
else if (type.equals("INTERNATIONAL"))
    strategy = new InternationalShippingStrategy(
        dhlClient, fedexClient, circuitBreakerRegistry);
// ... 5 nơi khác duplicate

// Test phải mock 10 dependencies mỗi caller
✅ Centralized factory
@Component
public class ShippingStrategyRegistry {
    private final Map<ShippingType, ShippingFeeStrategy> strategies;

    // Spring auto-inject TẤT CẢ @Component
    // implementing ShippingFeeStrategy
    public ShippingStrategyRegistry(
            List<ShippingFeeStrategy> allStrategies) {
        this.strategies = allStrategies.stream()
            .collect(toMap(
                ShippingFeeStrategy::supportedType,
                Function.identity()
            ));
    }

    public ShippingFeeStrategy get(ShippingType type) {
        var strategy = strategies.get(type);
        if (strategy == null) {
            throw new UnsupportedShippingException(type);
        }
        return strategy;
    }
}

// Caller — 1 line, testable
ShippingFeeStrategy strategy = registry.get(type);
Money fee = strategy.calculateFee(ctx);
🎯 Vì sao centralized factory?
  • Single source of truth: thêm method mới = đăng ký 1 chỗ (auto via Spring). Không cần update 10 callers.
  • Dependency injection friendly: factory wire dependencies cho strategies 1 lần (constructor). Caller không cần biết DHL client tồn tại.
  • Testability: trong test, inject MockShippingStrategyRegistry trả về mock strategy. Caller code không đổi, không cần mock cascade dependencies.
  • Spring magic: inject List<ShippingFeeStrategy> tự động lấy mọi @Component implementing interface — auto-discover, không hardcode.
  • Error consistency: factory throw UnsupportedException với clear message. Mỗi caller không phải repeat error handling.

Static Factory Method — Effective Java Item 1

// Alternative: static factory method on the class itself
public final class Money {
    private final BigDecimal amount;
    private final Currency currency;

    // Private constructor — force factory method
    private Money(BigDecimal amount, Currency currency) {
        this.amount = amount;
        this.currency = currency;
    }

    // Static factory methods — named, no exception
    public static Money of(BigDecimal amount, Currency currency) {
        return new Money(amount, currency);
    }

    public static Money zero(Currency currency) {
        // Có thể cache instance phổ biến
        return CACHED_ZEROS.computeIfAbsent(currency, c -> new Money(BigDecimal.ZERO, c));
    }

    public static Money vnd(long amount) {
        return of(BigDecimal.valueOf(amount), VND);
    }
}

// Usage
Money fee = Money.vnd(50_000);
Money zero = Money.zero(VND);  // Return cached instance
// vs Money fee = new Money(BigDecimal.valueOf(50000), VND);  // Less expressive
🎯 Static factory vs constructor — 5 lợi thế (Effective Java)
  1. Có tên ý nghĩa: Money.vnd(50_000) rõ hơn new Money(BigDecimal.valueOf(50000), Currency.getInstance("VND"))
  2. Không bắt buộc tạo new instance: cache (Integer.valueOf), singleton, immutable reuse
  3. Return subtype: Collections.unmodifiableList return private subclass
  4. Type inference cho generics: trước Java 7 — Map.of() vs new HashMap<String, List<Integer>>()
  5. Class chưa load tại compile time: factory có thể load class dynamically
🎯 Edge Cases — Factory Production Reality
  • Factory return null: easy bug nếu factory tạo null cho unknown type. Never return null; throw exception hoặc return Optional.
  • Factory tạo prototype thay singleton: request.getStrategy(type) create new instance mỗi call → GC pressure. Spring default singleton — verify scope.
  • Circular dependency in factory: StrategyA depends on factory, factory depends on StrategyA → Spring fail at startup. Restructure hoặc @Lazy.
  • Factory in static field: private static final Factory FACTORY = new Factory(); — class load order issue, hard to test. Use Spring beans.
  • Hot reload strategies: feature flag enable new strategy. Factory cache map at startup → restart required. Use refresh mechanism (Spring Cloud).
  • Factory throws checked exception: create() throws IOException — caller forced try-catch. Wrap checked → runtime exception in factory.
  • Generic factory với type erasure: Factory<T>.create() không know T at runtime. Pass Class<T> param hoặc TypeReference.
  • Factory test với fake instances: production factory inject real beans. Tests cần TestFactory với fake strategies. Use @TestConfiguration.

4. Builder Pattern SUPPLEMENTARY

SUPPLEMENTARY — Clean object creation. Lombok @Builder auto-generate.
📜 Builder — From GoF to Effective Java Item 2
GoF 1994: Builder for complex object construction with multiple steps. Director + Builder.

Joshua Bloch — Effective Java 2001+: Item 2 — "Consider a builder when faced with many constructor parameters". Modify GoF builder thành fluent API style. Inspired Java Builder pattern dominant today.

Lombok 2009: @Builder annotation auto-generate builder boilerplate. Reduce 50+ lines to 1 annotation.

Java 14+ Records: cho simple immutable values, records replace Builder. Builder still needed cho complex objects với optional fields, validation, computed defaults.

Insight: Builder shines khi: (1) constructor có 4+ parameters, đặc biệt khi nhiều same-type (dễ swap nhầm), (2) optional parameters, (3) cross-field validation, (4) immutable object cần fluent API.

Tình huống: API search cho ride-sharing app (Grab, Uber). Search request có ~12 optional filters:

❌ Constructor 12 tham số — unreadable, dễ swap nhầm
SearchRequest req = new SearchRequest(
    "HCM",                // pickupCity
    "Q1",                 // pickupDistrict
    "HCM",                // dropoffCity
    "Q7",                 // dropoffDistrict
    LocalDateTime.now(),  // pickupTime
    VehicleType.CAR,
    4,                    // maxPassengers
    null,                 // promoCode
    true,                 // allowSharedRide
    BigDecimal.valueOf(100_000), // maxPrice
    "VND",                // currency
    false                 // hasPet
);
// Junior swap nhầm "Q1" và "Q7" → bug khó tìm
// Compiler không help (cùng kiểu String)
✅ Builder — fluent, self-documenting
SearchRequest req = SearchRequest.builder()
    .pickup("HCM", "Q1")          // Named, không thể swap với dropoff
    .dropoff("HCM", "Q7")
    .pickupTime(LocalDateTime.now())
    .vehicle(VehicleType.CAR)
    .maxPassengers(4)
    .allowSharedRide(true)
    .maxPrice(Money.of(100_000, VND))
    .build();  // Validate tại đây — fail fast
public class SearchRequest {
    private final String pickupCity, pickupDistrict, dropoffCity, dropoffDistrict;
    private final LocalDateTime pickupTime;
    private final VehicleType vehicle;
    private final int maxPassengers;
    private final boolean allowSharedRide;
    private final Money maxPrice;

    private SearchRequest(Builder b) {
        this.pickupCity = b.pickupCity;
        this.pickupDistrict = b.pickupDistrict;
        this.dropoffCity = b.dropoffCity;
        this.dropoffDistrict = b.dropoffDistrict;
        this.pickupTime = b.pickupTime;
        this.vehicle = b.vehicle;
        this.maxPassengers = b.maxPassengers;
        this.allowSharedRide = b.allowSharedRide;
        this.maxPrice = b.maxPrice;
    }

    public static Builder builder() { return new Builder(); }

    public static class Builder {
        private String pickupCity, pickupDistrict, dropoffCity, dropoffDistrict;
        private LocalDateTime pickupTime;
        private VehicleType vehicle = VehicleType.CAR;  // default
        private int maxPassengers = 1;
        private boolean allowSharedRide = false;
        private Money maxPrice;

        // Fluent setters — combine related fields
        public Builder pickup(String city, String district) {
            this.pickupCity = city;
            this.pickupDistrict = district;
            return this;
        }

        public Builder dropoff(String city, String district) {
            this.dropoffCity = city;
            this.dropoffDistrict = district;
            return this;
        }

        public Builder pickupTime(LocalDateTime t) { this.pickupTime = t; return this; }
        public Builder vehicle(VehicleType v) { this.vehicle = v; return this; }
        public Builder maxPassengers(int n) { this.maxPassengers = n; return this; }
        public Builder allowSharedRide(boolean b) { this.allowSharedRide = b; return this; }
        public Builder maxPrice(Money m) { this.maxPrice = m; return this; }

        public SearchRequest build() {
            // Validate at boundary — fail fast
            validateRequired();
            validateBusinessRules();
            return new SearchRequest(this);
        }

        private void validateRequired() {
            if (pickupCity == null) throw new IllegalArgumentException("pickup required");
            if (dropoffCity == null) throw new IllegalArgumentException("dropoff required");
            if (pickupTime == null) throw new IllegalArgumentException("pickupTime required");
        }

        private void validateBusinessRules() {
            // Cross-field validation
            if (pickupTime.isBefore(LocalDateTime.now())) {
                throw new IllegalArgumentException("Cannot book past time");
            }
            if (maxPassengers < 1 || maxPassengers > vehicle.maxCapacity()) {
                throw new IllegalArgumentException(
                    String.format("Invalid passenger count %d for %s (max %d)",
                        maxPassengers, vehicle, vehicle.maxCapacity()));
            }
            if (vehicle == VehicleType.MOTORBIKE && allowSharedRide) {
                throw new IllegalArgumentException("Motorbike cannot share ride");
            }
        }
    }
}
🎯 Vì sao Builder > constructor / setter approach?
  • Readability: caller nhìn biết ngay field nào set, default nào dùng. Constructor 12 tham số = guess game.
  • Immutability: SearchRequest không có setter, không bị mutate sau construction. Thread-safe.
  • Validation at boundary: build() validate trước khi tạo object → invalid object never exists. Setter pattern allow create rồi mutate thành invalid state.
  • Optional fields: builder có default, không cần truyền null. Caller chỉ set field cần thiết.
  • Composable validation: cross-field rule (motorbike + sharedRide) check ở build() — single place.
  • Type safety on similar params: pickup(city, district) impossible to swap với dropoff(city, district) — different method names.
💡 Lombok shortcut

Trong project thật, dùng @Builder của Lombok auto-generate code này (50 lines → 1 annotation). Nhưng vẫn nên hiểu cơ chế bên trong để debug + customize khi cần validation/computed defaults.

5. Observer / Event Pattern CORE

★ CORE — Foundation cho event-driven. Spring Events, Kafka pub/sub đều dùng pattern này.
📜 Observer — From MVC to Reactive
1979 — Trygve Reenskaug tại Xerox PARC introduce MVC (Model-View-Controller) cho Smalltalk-80. View observe Model — first major use of Observer pattern.

GoF 1994: Observer formalized. "Define a one-to-many dependency between objects so that when one object changes state, all its dependents are notified."

Java 1.0 (1996): java.util.Observable và Observer interface. Deprecated trong Java 9 vì thread-safety issues, không serializable.

2000s: Event-Driven Architecture grow. Java Swing, GUI frameworks heavily Observer.

2009-2014: Reactive Streams spec — backpressure-aware Observer. Erik Meijer (Microsoft → Netflix Rx). RxJava popularize trong Java.

Spring 3+: ApplicationEventPublisher + @EventListener = built-in Observer. Spring 4.2+ thêm @TransactionalEventListener.

Modern: Distributed systems extend Observer to network — Kafka pub/sub, AWS SNS, EventBridge. Same pattern, network-scale.

Insight: Observer evolution mirror sự phát triển của software architecture: in-process → GUI → web app → microservices → distributed. Pattern stable, scale change.

Tình huống: Khi 1 order được đặt thành công, hệ thống cần làm 5 việc:

  1. Trừ stock trong warehouse
  2. Gửi email confirmation
  3. Push notification cho seller
  4. Update analytics dashboard (orders/min)
  5. Trigger fraud detection
❌ Tightly-coupled orchestration
@Service
public class PlaceOrderService {

  private final EmailService email;

  private final AnalyticsService analytics;
  private final FraudService fraud;

  @Transactional

    Order order = createOrder(cart);

✅ Event-driven
@Service
public class PlaceOrderService {
  private final ApplicationEventPublisher events;

  @Transactional
  public Order placeOrder(Cart cart) {
    Order order = createOrder(cart);
    orderRepo.save(order);

    // Chỉ 1 event — không biết ai listen
    events.publishEvent(
      new OrderPlacedEvent(order));

    return order;
  }
}

@Component

    inventoryService.reduceStock(e.order());
  }
}

🎯 Vì sao Event-Driven > Direct call?
  • Add subscriber không sửa publisher: thêm "loyalty points update" sau order = thêm 1 listener mới, KHÔNG sửa PlaceOrderService. Open-Closed applied.
  • Independent failure: fraud service crash không làm fail order. Trong tightly-coupled version, exception ở fraud sẽ rollback order.
  • Independent scale: analytics chậm (DB heavy) không block inventory (fast). Mỗi listener có thread pool riêng nếu cần.
  • Testability: test PlaceOrderService chỉ cần verify event published (1 mock). Không cần mock 5 services.
  • Cross-module decoupling: listeners có thể trong modules khác (loyalty, fraud, marketing) — không tạo cyclic dependencies.
🧠 Misconception — "@EventListener sync = same transaction"
"Spring @EventListener chạy trong cùng @Transactional của publisher — nếu listener throw, publisher rollback."
Đúng nhưng nguy hiểm: nếu sync listener throw → toàn bộ transaction rollback. Inventory exception → order không saved. Email service down → order fail!

Best practice:
(1) Default @EventListener sync but listener should NOT throw — try-catch internally, log, send to DLQ.
(2) Critical listeners (inventory deduction MUST happen with order) → keep sync, careful exception handling.
(3) Non-critical (notifications, analytics) → @Async để không block + không break transaction.
(4) Use @TransactionalEventListener(phase = AFTER_COMMIT) để chỉ fire sau khi transaction commit thành công — guarantee event reflects committed state.
🎯 Edge Cases — Observer/Event Production Reality
  • Sync listener slow → block publisher: SMS gateway 5s → order API 5s. Mark non-critical listeners @Async.
  • Event before commit: events.publishEvent(...) trong middle of transaction. Listener read DB → not yet committed. Use @TransactionalEventListener(AFTER_COMMIT).
  • Event lost on app crash: in-memory events lost if app crash trước listener execute. Use Outbox pattern + persistent event store.
  • Cycle: listener A trigger event → listener B trigger event → listener A again: infinite loop, stack overflow. Add cycle detection hoặc deduplication via event ID.
  • Listener execution order matters: inventory must reduce before email confirms "shipping soon". Use @Order annotation hoặc orchestration.
  • Schema evolution: event class add field → old listeners break if deserialize strict. Use backward-compatible serialization (optional fields).
  • Distributed event = network failure: Kafka pub down → events accumulate. Local pub/sub OK. Hybrid: local for sync concerns, Kafka for cross-service.
  • Listener idempotency: at-least-once delivery → listener may process duplicate. Track processed event IDs.
  • Event explosion: 10 events fire per order → 50 listeners → 500 invocations/order. Audit listener count, batch hoặc consolidate events.

6. Decorator Pattern SUPPLEMENTARY

SUPPLEMENTARY — Add cross-cutting concerns. Spring AOP = dynamic decorator.

Tình huống: bạn có HttpClient gọi API của 3rd-party (DHL, Stripe, ...). Muốn thêm: Logging, Retry, Caching, Metrics.

❌ Inheritance explosion

class CachingRetryingLoggingDhlClient

// Đổi thứ tự Retry trước Logging? Class mới
✅ Composition (Decorator)

client.get("/quote");
// Order: Metrics → Caching → Retry → Logging → DHL
public interface HttpClient {
    HttpResponse get(String url);
    HttpResponse post(String url, String body);
}

// Decorator base
public abstract class HttpClientDecorator implements HttpClient {
    protected final HttpClient delegate;
    protected HttpClientDecorator(HttpClient delegate) { this.delegate = delegate; }
}

public class LoggingClient extends HttpClientDecorator {
    private static final Logger log = LoggerFactory.getLogger(LoggingClient.class);
    public LoggingClient(HttpClient delegate) { super(delegate); }

    @Override
    public HttpResponse get(String url) {
        log.info("GET {}", url);
        long start = System.nanoTime();
        try {
            HttpResponse resp = delegate.get(url);
            log.info("GET {} → {} ({}ms)", url, resp.status(),
                (System.nanoTime() - start) / 1_000_000);
            return resp;
        } catch (Exception e) {
            log.error("GET {} failed: {}", url, e.getMessage());
            throw e;
        }
    }
}

public class RetryingClient extends HttpClientDecorator {
    private final int maxAttempts;

    public RetryingClient(HttpClient delegate, int maxAttempts) {
        super(delegate);
        this.maxAttempts = maxAttempts;
    }

    @Override
    public HttpResponse get(String url) {
        Exception lastError = null;
        for (int i = 0; i < maxAttempts; i++) {
            try {
                HttpResponse resp = delegate.get(url);
                if (resp.status() < 500) return resp;  // 2xx/3xx/4xx — không retry
            } catch (Exception e) {
                lastError = e;
            }
            try {
                Thread.sleep(100 * (1L << i));  // Exponential backoff
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                throw new RuntimeException(e);
            }
        }
        throw new RuntimeException("Failed after " + maxAttempts + " attempts", lastError);
    }
}

public class CachingClient extends HttpClientDecorator {
    private final Cache<String, HttpResponse> cache;

    public CachingClient(HttpClient delegate, Duration ttl) {
        super(delegate);
        this.cache = Caffeine.newBuilder().expireAfterWrite(ttl).maximumSize(1000).build();
    }

    @Override
    public HttpResponse get(String url) {
        return cache.get(url, k -> delegate.get(k));
    }

    @Override
    public HttpResponse post(String url, String body) {
        return delegate.post(url, body);  // POST không cache
    }
}
🎯 Vì sao Decorator > Inheritance?
  • Composability: stack any combination của decorators. Customer A muốn Logging+Retry, B muốn Metrics+Caching → cùng base, khác stack.
  • Reusable across types: LoggingClient dùng được với DHL, Stripe, FedEx — không class riêng cho mỗi cái.
  • Runtime composition: order có thể đổi runtime dựa config. Inheritance fix compile-time.
  • Single Responsibility: mỗi decorator chỉ 1 concern. Test riêng dễ.
💡 Spring AOP = Decorator dynamic

Spring @Async, @Cacheable, @Retryable, @Transactional đều là decorator dynamic (qua proxy). Bạn không viết decorator class, Spring tạo proxy decorator runtime. Cùng concept, ít boilerplate.

7. Chain of Responsibility ADVANCED

ADVANCED — Pipeline processing. Spring filter chain, HTTP middleware đều dùng pattern này.

Tình huống: trước khi 1 message được post lên social platform, cần qua nhiều validation step:

  1. Authentication: user logged in?
  2. Rate limit: user post > 100 messages/hour?
  3. Content moderation: contain hate speech / NSFW?
  4. Spam detection: spam pattern không?
  5. Mention validation: @mentions hợp lệ?
// Handler base
public abstract class PostHandler {
    protected PostHandler next;

    public PostHandler setNext(PostHandler next) {
        this.next = next;
        return next;  // Fluent — chain.setNext(a).setNext(b)
    }

    public final void handle(PostRequest req) {
        process(req);
        if (next != null) next.handle(req);
    }

    protected abstract void process(PostRequest req);
}

@Component
public class AuthHandler extends PostHandler {
    @Override
    protected void process(PostRequest req) {
        if (req.userId() == null) throw new UnauthenticatedException();
    }
}

@Component
public class RateLimitHandler extends PostHandler {
    private final RedisRateLimiter limiter;

    public RateLimitHandler(RedisRateLimiter limiter) { this.limiter = limiter; }

    @Override
    protected void process(PostRequest req) {
        if (!limiter.tryAcquire(req.userId(), "post", 100, Duration.ofHours(1))) {
            throw new RateLimitExceededException();
        }
    }
}

@Component
public class ModerationHandler extends PostHandler {
    private final ContentModerationApi moderation;

    @Override
    protected void process(PostRequest req) {
        var result = moderation.classify(req.content());
        if (result.containsHateSpeech() || result.containsNsfw()) {
            throw new ContentRejectedException(result.reason());
        }
    }
}

// Configuration — build chain
@Configuration
public class PostHandlerChainConfig {
    @Bean
    public PostHandler postHandlerChain(AuthHandler auth, RateLimitHandler rateLimit,
                                         ModerationHandler mod, SpamHandler spam) {
        auth.setNext(rateLimit).setNext(mod).setNext(spam);
        return auth;
    }
}

// Usage
@Service
public class PostService {
    private final PostHandler chain;

    public PostService(PostHandler chain) { this.chain = chain; }

    public void post(PostRequest req) {
        chain.handle(req);  // Run through all handlers
        savePost(req);      // Reach here only if all handlers pass
    }
}
🎯 Vì sao Chain > monolithic validator?
  • Add/remove rules dễ: thêm "ProfanityFilter" = thêm 1 handler vào chain, không sửa code cũ.
  • Test riêng: SpamHandler test với spam input, không cần setup full chain.
  • Configurable order: dev environment skip ModerationHandler (slow API). Production full chain.
  • Single Responsibility: mỗi handler 1 concern.
  • Reusable: AuthHandler dùng được cho post, comment, like — không duplicate.

8b. When NOT to Use — Counter-examples cho từng pattern CORE ★

Pattern dùng sai chỗ tệ hơn không dùng — code phình to, abstraction rỗng, người maintain ghét. Đây là cảnh báo cụ thể: khi nào SKIP pattern và viết straight code.

❌ Strategy Pattern — KHÔNG dùng khi…
  • Chỉ có 2 case và không bao giờ thêm — if (type == "PREMIUM") discount = 0.2 else discount = 0.1 đơn giản và rõ. Strategy = 3 file (interface + 2 impl + factory) cho 1 dòng if
  • Logic không có behavior, chỉ là data lookup — nếu strategy chỉ trả về 1 giá trị từ config, dùng Map<String, Config>. Strategy thêm overhead mà không có polymorphic behavior
  • Strategies share quá nhiều state — nếu 5 strategy đều dùng cùng 8 helper method, abstraction sai — chia theo behavior chứ không phải theo data variant

Symptom của over-use: StrategyFactory dài 200 dòng, mỗi strategy implement 1 method trả 1 dòng kết quả, mọi PR add feature đều tạo 1 strategy mới.

❌ Factory Pattern — KHÔNG dùng khi…
  • Constructor đã đủ rõ ràng — new User(email, name) dễ đọc hơn UserFactory.create(email, name). Factory chỉ thêm value khi có logic chọn implementation hoặc validation phức tạp
  • Chỉ 1 implementation — Factory cho interface có 1 impl = abstraction giả. Spring DI tự inject được, không cần factory wrapper
  • Pure data object (DTO, value object) — Builder hợp lý hơn nếu nhiều field; constructor đủ nếu < 4 field. Factory không thêm gì

Symptom của over-use: OrderFactory.createOrder(...) bên trong chỉ có return new Order(...). Mọi class đều có FactoryXxx tương ứng.

❌ Builder Pattern — KHÔNG dùng khi…
  • Class có ≤ 3 field bắt buộc, không có optional — constructor tốt hơn. Builder là overkill
  • Immutable record với tất cả field bắt buộc — Java record syntax đã rõ. Builder thêm boilerplate không có lợi
  • Validation phức tạp giữa các field — Builder không enforce được "phải set A trước B". Step builder hoặc factory với validated input tốt hơn

Symptom của over-use: Builder cho class 2 field. Builder không có validate trong build() → tạo object invalid silently.

❌ Observer / Event Pattern — KHÔNG dùng khi…
  • Cần biết kết quả xử lý của subscriber — event là fire-and-forget, không trả response. Nếu cần "gọi service B và nhận kết quả", dùng direct call hoặc reactive Mono
  • Strict ordering matter — observer thường async, không guarantee thứ tự. Nếu cần "A xong rồi mới B", dùng explicit workflow/saga
  • Trong cùng transaction boundary — Spring ApplicationEventPublisher default sync, listener chạy trong cùng TX → exception rollback. Dùng @TransactionalEventListener(phase=AFTER_COMMIT) nếu muốn async, nhưng phải biết

Symptom của over-use: Mọi state change publish event. Codebase có 50 event class, debug flow cần đọc 8 listener.

❌ Decorator Pattern — KHÔNG dùng khi…
  • Chỉ 1-2 hành vi cần wrap — Java AOP (@Around) hoặc functional composition đơn giản hơn. Decorator hierarchy 5 lớp khó trace
  • Decorator chain order quan trọng và không rõ — new Logging(new Caching(new RateLimit(api))) vs ngược lại có behavior khác. Nếu team không hiểu order, dùng pipeline framework hoặc reactive stream
  • Cần state chia sẻ giữa decorator — decorator nên stateless. Nếu phải chia sẻ state, sai pattern

Symptom của over-use: Stack trace có 8 lớp decorator, mỗi lớp wrap chỉ để log "method called".

❌ Chain of Responsibility — KHÔNG dùng khi…
  • Mỗi handler luôn được chạy — không phải chain, đây là pipeline. Dùng Stream.reduce hoặc functional composition
  • Decision tại 1 điểm rõ ràng — strategy pattern hoặc switch expression đơn giản và rõ hơn
  • Order handler thay đổi theo runtime — chain hard-coded order không phù hợp. Cần rule engine (Drools) hoặc DSL

Symptom của over-use: Chain dài 12 handler, debug phải set breakpoint từng handler để xem ai handle. Không log handler nào skip vs accept.

🎯 3 cấp độ Mastery cho Design Patterns

L1 — Recognize (Junior)

Biết tên các pattern, đọc code nhận ra "đây là Strategy". Có xu hướng "học pattern xong áp dụng mọi nơi" — pattern fever.

Symptom của L1: code review comment "nên dùng Factory cho cái này" mà không nói tại sao.

L2 — Explain (Mid)

Hiểu vấn đề mà pattern giải quyết (open-closed, varying algorithm). Biết phân biệt pattern nào cho tình huống nào theo bảng so sánh.

Symptom của L2: dùng pattern khi đã thấy "nếu thêm 1 case nữa thì if/else trở thành mớ bòng bong".

L3 — Apply (Senior)

Biết khi nào KHÔNG dùng pattern (6 counter-example ở trên là phản xạ). Khi review code, từ chối abstraction sai (Strategy cho 2 case, Factory wrap 1 dòng new). Refactor pattern lỗi thời về straight code.

Symptom của L3: pull request xoá pattern thừa thay vì thêm pattern mới.

9. Anti-Patterns Phổ Biến

#1 Pattern Fever — áp dụng cho mọi vấn đề

❌ Factory cho 2-case logic
public class NotifierFactory {
  public Notifier get(boolean isVip) {
    return isVip
      ? new VipNotifier()
      : new RegularNotifier();
  }
}
// Indirection không có benefit
✅ Trực tiếp
Notifier n = isVip
  ? new VipNotifier()
  : new RegularNotifier();

#2 Strategy với State — Recreate object liên tục

Strategies stateless → tạo 1 lần, reuse mãi. New instance mỗi request gây GC pressure. Dùng Spring beans = lifecycle managed.

#3 Observer sync — Listener làm tắc nghẽn publisher

SMS API chậm 5s → sync listener block order flow. Mark @Async cho non-critical.

#4 Builder không validate — Tạo invalid object

build() phải validate. Principle "fail fast at boundary".

#5 Decorator + Inheritance lẫn lộn

Cross-cutting concerns orthogonal — composition cho phép mix-and-match runtime, inheritance fix compile-time.

#6 Chain quá dài

20 handlers trong chain → mỗi request tốn 20 method calls. Performance critical paths cần consolidate.

#7 Pattern thay vì simple code

2-line method không cần pattern. Junior thấy senior dùng patterns → copy pattern vào mọi nơi → over-engineer.

  • Bonus: Observer pattern publish event after successful post
Acceptance: 95%+ test coverage. Add new handler = single file change, no other modifications.

11. Interview Framework

Tầng 1 — SurfaceMid gate
Q: Giải thích Strategy pattern. Khi nào dùng thay if-else?
"Strategy đóng gói algorithm trong 1 class riêng implement interface chung. Context chỉ dùng interface, không biết implementation cụ thể.

Dùng khi có 5+ variant hoặc algorithm hay thay đổi, vì thêm variant mới = thêm class, không sửa context (Open-Closed). Replace if-else bằng polymorphism. Trade-off: thêm classes → cost nhỏ. Benefits: testable riêng, isolated change, auto-discover via DI.

Với 2-3 case đơn giản → if-else vẫn tốt hơn. Premature application của Strategy = over-engineering."
Q: Builder pattern là gì, vì sao dùng thay vì constructor?
"Builder cung cấp fluent API để tạo object phức tạp với nhiều optional parameters. Constructor 10 tham số là unreadable và dễ swap nhầm vị trí (cùng kiểu String).

Lợi ích: (1) Readability với named methods, (2) Immutability — object không setter, (3) Validation tại build() đảm bảo invalid object never exists, (4) Optional fields với defaults, (5) Cross-field validation single place.

Lombok @Builder auto-generate. Records (Java 16+) replace builder cho simple immutable; builder vẫn cần cho complex objects."
Tầng 2 — Deep DiveMid → Senior
Q: Design payment processing dùng Strategy + Factory + Decorator. Methods: Momo, VNPay. Add logging + retry.
"Strategy: PaymentGateway interface với MomoPaymentGateway, VNPayPaymentGateway implementations.
Factory: PaymentGatewayRegistry map từ PaymentMethod enum → bean. Inject List<PaymentGateway> từ Spring, auto-discover.
Decorator: LoggingPaymentGateway wrap gateway, log request/response. RetryingPaymentGateway wrap thêm để retry 3 lần với exponential backoff.
Usage: registry.get(MOMO) trả gateway đã wrap decorators (config qua Spring AOP hoặc explicit wrap trong registry).

Thêm method mới: tạo class implement PaymentGateway, Spring auto-pick up. Registry tự register. Decorators apply automatically. 0 dòng sửa code cũ.

Production concerns: idempotency key, circuit breaker per gateway (DHL down không break VNPay), correlation ID trace cross gateway calls."
Q: 4 sync event listeners khi payment completed. 1 listener (fraud) chậm 5s. Fix sao không break listeners khác?
"Option 1 — Async chỉ listener slow: đánh dấu fraud listener @Async. Payment trả về customer ngay, fraud detection chạy background. Nếu fraud detect anomaly → separate alert (không revert payment đã confirmed).

Option 2 — All async: tất cả listeners @Async. Payment trả về ngay sau save. Trade-off: nếu inventory update fail, order đã confirmed → cần compensation flow (SAGA).

Option 3 — Move to message queue: publish event lên Kafka, mỗi listener là service riêng consume. Decouple hoàn toàn — fraud service deploy độc lập, scale độc lập.

Recommendation: Option 1 cho near-realtime systems trong same JVM. Option 3 cho microservices architecture cần independent deployability. Plus: @TransactionalEventListener(AFTER_COMMIT) để listeners chỉ fire sau commit successful."
Tầng 3 — ArchitectureSenior
Q: Design pipeline xử lý request lên platform Q&A (StackOverflow-like). 6 steps: auth, rate-limit, spam check, content moderation, duplicate detection, tag validation. Mỗi step có thể async/sync. Step fail thì abort.
"Chain of Responsibility với extension:

1. Handler abstraction: mỗi step implement QuestionHandler interface có method handle(QuestionRequest req) trả về HandlerResult (PASS/FAIL/SKIP).

2. Configurable chain: chain order config qua Spring profile. Production: full 6 steps. Dev: skip moderation + duplicate detection (slow). Test: chỉ auth.

3. Hybrid sync/async:
• Sync (must pass before save): auth, rate-limit, spam check
• Async (run after save, can flag/revert later): moderation, duplicate detection, tag validation
Đặt 2 chain riêng: preSaveChain và postSaveChain.

4. Result aggregation: nếu pre-save fail → throw exception → caller nhận error. Post-save fail → mark question "under review" trong DB, gửi event QuestionFlaggedEvent → notification system báo moderator.

5. Observability: mỗi handler emit metric (handler_name, duration, result). Grafana dashboard show bottleneck handler. Add new handler #7 = thêm 1 class + register vào chain config.

6. Testability: handler test riêng với mock dependencies. Chain test với fake handlers verify order + abort behavior.

7. Combine patterns: Strategy cho handler implementations, Factory để build chain, Observer cho events sau success, Decorator cho cross-cutting (logging, metrics)."
