CI/CD Pipeline — Deploy An Toàn, Rollback Trong Giây

"Có pipeline tự build & deploy" là CI/CD ở mức bề mặt. Lên thật: "blue-green và canary khác nhau ở rủi ro nào?", "vì sao đổi schema DB và đổi code phải tách 2 lần deploy?", "feature flag không xoá thành nợ kỹ thuật ra sao?" — là lộ hiểu giả. File này dạy cả cơ chế (deployment strategy, expand-contract migration) lẫn vận hành (rollback, approval gate), trên domain ShopFlow — checkout-api của sàn e-commerce.

🎯 Mục tiêu
Thiết kế pipeline deploy zero-downtime, rollback nhanh, migration an toàn
📅 Thời lượng

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Deploy = release. Push code lên là user dùng feature mới."
Thực tế: Deploy (đưa code lên server) và release (bật feature cho user) là 2 việc tách được bằng feature flag. Tách ra cho phép deploy ngày thường, bật feature theo lịch, tắt ngay khi lỗi mà không rollback.
Tưởng hiểu: "Đổi schema DB và đổi code cứ deploy chung một lần cho gọn."
Thực tế: Lúc rolling update, code cũ và code mới chạy đồng thời. Drop/rename cột ngay = pod cũ vỡ. Phải expand-contract: thêm cột (tương thích ngược) → deploy code → mới xoá cột ở lần sau.
Tưởng hiểu: "Canary và blue-green na ná nhau, đều deploy an toàn."
Thực tế: Blue-green switch 100% tức thì (rollback nhanh nhưng lỗi ảnh hưởng toàn bộ user ngay). Canary cho 1%→10%→100% dần (bắt lỗi sớm trên ít user, nhưng cần đo metric để quyết định). Khác nhau ở "blast radius".

1. Bài toán thực tế — deploy thứ Sáu = nín thở CORE ★

ShopFlow deploy bằng tay, mỗi lần ~30 phút downtime, cả team thức canh. Một lần đổi schema (rename cột) làm pod cũ vỡ giữa rolling update → lỗi 500 hàng loạt. Rollback phải build lại từ đầu, mất 40 phút. Feature mới phải deploy đúng đêm ra mắt vì không tách được deploy khỏi release.

⏸ Pause & Predict — trước khi đọc lời giải
Viết ra giấy:
  1. "Rename cột làm pod cũ vỡ giữa rolling update" — vì sao? Pod cũ và pod mới có chạy đồng thời không? Cách migration nào tránh được?
  2. "Feature phải deploy đúng đêm ra mắt" — kỹ thuật nào cho phép deploy trước, bật sau?
  3. Muốn rollback trong <1 phút thay vì build lại 40 phút, cần chuẩn bị gì từ trước (gợi ý: artifact)?
Đáp án: rolling update chạy song song code cũ+mới → cần expand-contract migration (mục 6); feature flag tách deploy/release (mục 5); rollback nhanh = giữ artifact/image version cũ sẵn sàng (mục 7), không build lại.

2. CI vs Continuous Delivery vs Continuous Deployment SUPPLEMENTARY

Continuous Integration : mỗi push → build + test tự động, merge thường xuyên
Continuous Delivery    : luôn ở trạng thái deploy được, nhưng bấm nút mới deploy
Continuous Deployment  : pass pipeline → tự động deploy production, không cần bấm

3. Pipeline stages + fail-fast CORE ★

checkout → build → unit test → lint/static analysis → package (image)
        → integration test → security scan → deploy staging → smoke test
        → [approval] → deploy prod (canary) → monitor → promote/rollback

Fail-fast: chạy stage rẻ & nhanh trước (unit test) để feedback sớm; stage đắt (integration, e2e) sau. Một stage đỏ chặn các stage sau.

4. Blue-green / Canary / Rolling CORE ★

Chiến lượcCách hoạt độngBlast radius / rollback
RollingThay pod dần (mặc định K8s)Cũ+mới chạy chung lúc rollout; rollback = rollout undo
Blue-Green2 môi trường, switch 100% tức thìLỗi ảnh hưởng toàn bộ ngay; rollback = switch lại (giây), tốn gấp đôi tài nguyên
Canary1%→10%→100% theo metricLỗi chỉ chạm ít user; cần đo error rate/latency để quyết promote

L1 — Biết tên

Biết tên 3 chiến lược. Symptom hiểu giả: không nói được khác biệt blast radius hay khi nào chọn cái nào.

L2 — Explain được

Giải thích blue-green = switch toàn bộ (rollback nhanh, blast radius lớn, tốn 2× tài nguyên); canary = phơi bày dần dựa trên metric (bắt lỗi sớm, cần observability); rolling = mặc định nhưng phải lo tương thích cũ/mới.

L3 — Apply được

Chọn theo ngữ cảnh: thay đổi rủi ro cao + có metric tốt → canary; cần rollback tức thì + đủ tài nguyên → blue-green; thay đổi nhỏ thường ngày → rolling. Ghép với feature flag để giảm rủi ro thêm.

5. Feature flags — tách deploy khỏi release CORE ★

Code mới deploy nhưng nằm sau cờ if (flags.isOn("new-checkout")). Bật cho 5% user, theo dõi, bật dần. Lỗi → tắt cờ ngay (không rollback). Cho phép trunk-based development (merge code chưa hoàn thiện nhưng tắt cờ).

🔧 Counter-example — feature flag thành nợ kỹ thuật

Flag không miễn phí. Lạm dụng gây hại:

  • Flag không xoá: để mãi sau khi rollout 100% → code đầy if(flag) chết, mỗi flag nhân đôi đường code cần test.
  • Flag lồng nhau: 5 flag = 32 tổ hợp trạng thái, không ai test hết → bug tổ hợp.
  • Dùng flag thay cho config có cấu trúc: mọi thứ thành flag → mất khả năng suy luận hệ thống.

Quy tắc: flag là tạm thời — có ngày hết hạn và task dọn dẹp ngay khi rollout xong. Flag dài hạn (kill-switch, entitlement) thì tách loại rõ ràng.

6. DB migration backward-compatible (expand-contract) CORE ★

Vì rolling update chạy code cũ+mới đồng thời, migration phải tương thích ngược qua nhiều bước:

Đổi tên cột full_name → display_name (KHÔNG drop ngay):
1. EXPAND   : thêm cột display_name, code mới ghi cả 2, đọc display_name fallback full_name
2. MIGRATE  : backfill display_name từ full_name (batch)
3. (deploy code chỉ dùng display_name)
4. CONTRACT : sau khi chắc không pod nào dùng full_name → drop full_name
Anti-pattern + symptom

Triệu chứng: ngay sau deploy, log đầy column "full_name" does not exist từ pod CŨ chưa bị thay. Nguyên nhân: drop cột cùng lúc deploy code. Fix: tách drop sang lần deploy sau (contract), dùng Liquibase/Flyway có changeset versioned + rollback.

7. Rollback theo loại SUPPLEMENTARY

Loại thay đổiCách rollback
Code (stateless)kubectl rollout undo / switch blue-green / tắt feature flag — giây
DB schema (đã expand)Rollback code an toàn vì schema còn tương thích ngược
DB schema (đã contract/drop)KHÔNG rollback dễ — đây là lý do tách contract sang lần sau
Data migrationCần compensating script; backup trước

8. GitHub Actions + GitOps SUPPLEMENTARY

name: ci-cd
on: { push: { branches: [main] } }
jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: mvn -B verify              # build + unit + integration (Testcontainers)
      - run: docker build -t shopflow/checkout-api:${{ github.sha }} .
      - run: trivy image shopflow/checkout-api:${{ github.sha }}   # security scan
      - run: docker push shopflow/checkout-api:${{ github.sha }}
  # GitOps: cập nhật image tag trong repo manifest; ArgoCD tự sync về cluster

GitOps (ArgoCD): trạng thái mong muốn nằm trong Git; ArgoCD liên tục so sánh và sync cluster về đúng Git → audit trail + rollback = git revert.

9. Supply-chain security + approval gate ADVANCED

  • Scan image (Trivy/Grype) bắt CVE trước khi push.
  • SBOM + image signing (cosign) — chứng minh image build từ source tin cậy.
  • Secrets không nằm trong pipeline log — dùng OIDC/secret manager, không echo.
  • Approval gate cho production: yêu cầu duyệt thủ công cho thay đổi rủi ro cao.

🎯 3-tier interview framework

Tầng 1 — SurfaceJunior gate

Q: CI khác CD thế nào?

CI = mỗi push build+test+merge thường xuyên. Continuous Delivery = luôn deploy được, bấm nút mới deploy. Continuous Deployment = pass pipeline thì tự deploy production.

Tầng 2 — Deep DiveSenior gate

Q: Deploy đổi schema không downtime, có rollback.

Expand-contract: thêm cột tương thích ngược → deploy code đọc/ghi cả hai → backfill → deploy code chỉ dùng cột mới → drop cột ở lần sau. Code rollback an toàn vì schema còn tương thích. Dùng Flyway/Liquibase versioned.

Tầng 3 — ArchitectureSenior+

Q: Thiết kế progressive delivery cho thay đổi rủi ro cao.

Canary 1%→10%→50%→100% gắn metric gate (error rate, p99, business KPI); auto-rollback nếu vượt ngưỡng; bọc feature flag để tắt nhanh không cần rollback; GitOps để audit + revert; approval gate cho bước 100%.

🔧 Lab bắt buộc — thực hành deploy an toàn (2 bài)

Lab 1 — Expand-contract migration. Với một bảng users(full_name), thực hiện đổi tên sang display_name qua 4 bước expand-contract trên local. Giữa bước, chạy đồng thời "code cũ" (đọc full_name) và "code mới" (đọc display_name) → chứng minh cả hai chạy được. Tiêu chí: giải thích vì sao KHÔNG drop ở bước 1.

Lab 2 — Feature flag tắt nhanh. Bọc một feature sau flag (config/env). Bật cho "10% request" bằng hash userId. Giả lập lỗi → tắt flag (không redeploy) và xác nhận hết lỗi tức thì. Sau đó viết task "dọn flag". Tiêu chí: nói được khác biệt deploy vs release.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. Deploy và release khác gì? Feature flag tách chúng thế nào?
  2. Blue-green vs canary khác nhau ở blast radius và rollback ra sao?
  3. Vì sao drop cột cùng lúc deploy code gây lỗi? Expand-contract sửa thế nào?
  4. Làm sao rollback trong giây thay vì build lại 40 phút?
  5. Khi nào feature flag trở thành nợ kỹ thuật?
Bí câu nào = quay lại đúng section. Trôi cả 5 = nắm chắc CI/CD mức senior.
