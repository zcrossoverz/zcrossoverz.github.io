Tổng Quan — Vì Sao Redis Critical

Redis = in-memory data store. Đọc/ghi 100K-1M ops/sec per node (vs PostgreSQL 5-10K). Latency <1ms (vs DB 10-100ms). Cost: data trong RAM (volatile), nhỏ hơn DB capacity.

Modern backend dùng Redis cho:

  • Cache: giảm load DB cho hot data
  • Session store: user session, JWT blacklist
  • Rate limiter: per-user request quota
  • Distributed lock: coordinate access trong distributed system
  • Counter/leaderboard: real-time stats (likes, views, ranks)
  • Pub/Sub: lightweight messaging giữa services
  • Job queue: lightweight task queue (Sidekiq, Bull)
📜 Lịch sử Redis — Từ side project đến industry standard
2009 — Salvatore Sanfilippo (alias "antirez", Italian developer) tạo Redis ban đầu để solve scaling issue cho startup Lloogg (real-time web analytics). Frustrated với MySQL không handle được write throughput cao. Build C-based in-memory key-value store với data structures.

2010 — VMware hire Sanfilippo, sponsor Redis development full-time. Redis grow rapidly trong web 2.0 era — Twitter, Instagram, GitHub dùng cho cache + counter.

2013 — Pivotal spin off, sau đó Redis Labs (now Redis Inc.) founded để commercialize. Open source core vẫn maintain.

2015 — Redis Cluster GA — sharding built-in. Trước đó phải dùng client-side sharding hoặc Twemproxy.

2018 — Redis Streams introduced (Redis 5.0) — competitor cho Kafka cho lightweight messaging use case.

2020 — Martin Kleppmann publish famous "How to do distributed locking" critique của Redlock algorithm. Sanfilippo respond. Cuộc tranh luận nổi tiếng giúp community hiểu sâu hơn về distributed locks limitations.

2024 — Redis Inc. license change từ BSD sang dual SSPL/RSAL. AWS, Google fork Valkey để maintain Apache 2.0 version. Industry divided. Underlying tech vẫn cùng.

Insight: Redis success không phải vì invent new algorithm (most data structures classical) — mà vì practical engineering: single-thread model simple, ridiculously fast, rich data structures. Trade-off: single-thread cho throughput per-core. Multi-core via clustering/replication.

1. Bài Toán Thực Tế — Flash Sale Của Sàn TMĐT Sập

Sàn ShopVN tổ chức flash sale: iPhone 15 giảm 50%, 100 units, bắt đầu 12:00 trưa. Marketing push notification cho 5 triệu users.

Timeline disaster

11:58 INFO System normal, 5K QPS
11:59 INFO Pre-warming, cache loaded
12:00:00 WARN Traffic spike: 50K QPS
12:00:03 WARN Traffic: 250K QPS, DB connections 95% used
12:00:08 ERROR Traffic: 800K QPS, DB connection pool exhausted
12:00:15 ERROR Cache stampede: hot key "product:iphone15" expired, 50K concurrent DB queries
12:00:30 ERROR 502 Bad Gateway errors, DB crash
12:01:00 ERROR Distributed lock for inventory deduction: 500 concurrent threads holding lock with 5s timeout, deadlock-like state
12:05:00 ERROR Inventory shows 50 sold, payment confirmed 87. Oversell by 37 units.
12:15:00 ERROR Customer complaints, refund process triggered for 37 orders
🔥 5 bug Redis-related cùng lúc xảy ra
  1. Cache stampede: hot key expire → 50K concurrent DB queries
  2. Distributed lock bug: SETNX không có TTL → lock held forever khi worker crash
  3. Cache không cover all hot data: product detail cached, inventory count không cached
  4. No fallback: Redis down → app crash thay vì degrade
  5. Memory not limited: Redis OOM khi spike traffic, evict random keys
⏸ Pause & Predict — BẮT BUỘC dừng
Trước khi đọc giải pháp, tự trả lời:
  1. Cache stampede: 1000 requests arrive cùng moment cache expire → tất cả miss → 1000 DB queries. Cách prevent?
  2. Distributed lock: SETNX key 1 để lock. Worker process xong → DEL key. Vấn đề gì nếu worker crash giữa chừng?
  3. Inventory deduction: cần atomic check-and-decrement (nếu stock > 0 thì giảm 1). Làm sao với Redis?
  4. Redis down: app fall back thế nào? Show error, fallback to DB, hoặc serve stale data?
  5. Data structure: cache product detail (10 fields), recent orders (last 50), leaderboard top sellers — chọn structure nào cho mỗi case?
Nếu trả lời được 4/5 → bạn solid về Redis production. Nếu <3 → đọc kỹ.

Case Study — Leaderboard với Sorted Set

Social platform cần "Top 100 most-liked posts trong 24h". Với 1M posts:

❌ DB query mỗi request
SELECT post_id, like_count
FROM posts
WHERE created_at > NOW() - INTERVAL '24h'
ORDER BY like_count DESC
LIMIT 100;
-- Scan 1M rows, sort, take top 100
-- ~500ms per query
-- Index on (created_at, like_count) help
-- nhưng vẫn slow vì sort large set
-- 10K QPS → DB chết
✅ Sorted Set
# When post receive like
ZINCRBY top_posts_24h 1 "post_42"

# When fetching leaderboard
ZREVRANGE top_posts_24h 0 99 WITHSCORES
# O(log n + m) = O(log 1M + 100)
# ~1ms per query
# Reset daily: DEL + recompute
🎯 Vì sao Sorted Set magic cho leaderboard?

Sorted Set = skip list + hash table internally. ZADD insert + maintain sort order O(log n). ZRANGE get top N O(log n + N). ZINCRBY atomic update score + reorder. Tất cả memory, không disk I/O. DB sort 1M rows tốn 500ms; sorted set query 1ms — 500× faster. Trade-off: phải maintain song song với DB (truth source), sync via event.

🧠 Misconception — "Lưu mọi thứ vào String JSON, đơn giản"
"Tôi serialize entire user object thành JSON, GET/SET với String. Đơn giản, không cần học data structures khác."
3 vấn đề với cách này:
(1) Update 1 field = deserialize full JSON, modify, re-serialize, SET back → O(n) thay vì O(1) với Hash.
(2) Concurrent updates → lost write (last writer wins). Hash atomic per field.
(3) Wasted bandwidth: GET 10KB JSON khi chỉ cần 1 field.
Hash dùng nhiều ngay nếu object có 5+ fields và update partial common.

3. Caching Patterns CORE

★ CORE — Pattern phổ biến nhất production. Phải master cả 4 patterns + trade-offs.

Pattern 1: Cache-Aside (Lazy Loading) — phổ biến nhất

@Service
public class ProductService {
    private final RedisTemplate<String, String> redis;
    private final ProductRepository repo;
    private final ObjectMapper json;

    public Product findById(String productId) {
        String key = "product:" + productId;

        // 1. Try cache
        String cached = redis.opsForValue().get(key);
        if (cached != null) {
            return json.readValue(cached, Product.class);
        }

        // 2. Cache miss — load from DB
        Product product = repo.findById(productId)
            .orElseThrow(() -> new ProductNotFoundException(productId));

        // 3. Write to cache (with TTL)
        redis.opsForValue().set(key, json.writeValueAsString(product), Duration.ofMinutes(5));

        return product;
    }

    // Update: write to DB, invalidate cache
    public Product update(String productId, ProductUpdate update) {
        Product product = repo.update(productId, update);
        redis.delete("product:" + productId);  // Invalidate
        return product;
    }
}
🎯 Vì sao Cache-Aside phổ biến nhất?
  • App control flow: app quyết định khi nào cache, không phụ thuộc framework magic
  • Cache failure tolerant: cache down → fall back DB tự nhiên (slow but works)
  • Selective caching: chọn data nào cache, TTL khác nhau per case
  • Simple mental model: GET → MISS → LOAD → SET

Trade-off: invalidation phải explicit (dễ quên), first-time hit luôn slow (warm-up).

Pattern 2: Read-Through Cache

Cache layer transparent giữa app và DB. App chỉ gọi cache; cache miss → cache tự load từ DB. Spring @Cacheable dùng pattern này.

@Cacheable(value = "products", key = "#productId")
public Product findById(String productId) {
    return repo.findById(productId).orElseThrow();
}
// Spring intercept: check cache, miss → call method, store result
🎯 Cache-Aside vs Read-Through?
  • Read-Through (Spring @Cacheable): declarative, ít code, nhưng "magic" — debug hard, flexibility kém
  • Cache-Aside (manual): nhiều code hơn, control nhiều hơn — production thường prefer

Khi nào dùng cái nào: simple CRUD, prototype → Spring annotation. Production critical path → manual cache-aside.

Pattern 3: Write-Through

Write to cache → cache write to DB synchronously. Cache always fresh.

public Product save(Product product) {
    redis.opsForValue().set("product:" + product.id(), serialize(product), Duration.ofMinutes(5));
    return repo.save(product);  // Sync write to DB
}

Pros: cache always consistent với DB. Cons: write latency tăng (cache + DB), cache pollution (write rare-read data into cache).

Pattern 4: Write-Behind (Write-Back)

Write to cache → return immediately. Cache async batch write to DB.

// Write goes to cache only
public void recordPageView(String pageId) {
    redis.opsForValue().increment("pageviews:" + pageId);
}

// Async job batch-write to DB
@Scheduled(fixedDelay = 60000)
public void flushPageViewsToDb() {
    Set<String> keys = redis.keys("pageviews:*");
    for (String key : keys) {
        Long count = redis.opsForValue().get(key);
        pageViewRepo.upsert(extractPageId(key), count);
        redis.delete(key);
    }
}

Pros: blazing fast writes. Cons: data loss risk nếu cache crash trước flush. Chỉ dùng cho non-critical data (analytics, counters).

4. Cache Stampede Prevention CORE

★ CORE — Bug kinh điển khi hot key expire. Hiểu xảy ra như thế nào và 3 cách prevent.

The Problem

Hot key "product:iphone15" có TTL 5 phút. Tại moment expire:

  1. Request 1 arrive → cache MISS → start DB query (50ms)
  2. Request 2 arrive (10μs sau) → cache vẫn MISS → start DB query
  3. ...continue...
  4. Request 5000 (50ms sau, before Request 1 finishes) → vẫn MISS → start DB query
  5. = 5000 concurrent DB queries cho cùng key → DB overload

Solution 1: Lock pattern (Single-Flight)

public Product findByIdSafe(String productId) {
    String key = "product:" + productId;
    String lockKey = "lock:" + key;

    // Check cache
    String cached = redis.opsForValue().get(key);
    if (cached != null) return deserialize(cached);

    // Cache miss — try acquire lock
    Boolean locked = redis.opsForValue().setIfAbsent(lockKey, "1", Duration.ofSeconds(10));

    if (Boolean.TRUE.equals(locked)) {
        try {
            // We won the lock — load from DB
            Product product = repo.findById(productId).orElseThrow();
            redis.opsForValue().set(key, serialize(product), Duration.ofMinutes(5));
            return product;
        } finally {
            redis.delete(lockKey);
        }
    } else {
        // Lock held by someone else — wait briefly + retry cache
        Thread.sleep(50);
        String retryCache = redis.opsForValue().get(key);
        if (retryCache != null) return deserialize(retryCache);
        // Fallback: query DB (degraded path)
        return repo.findById(productId).orElseThrow();
    }
}

Solution 2: Probabilistic Early Refresh (XFetch)

Refresh cache trước khi expire với probability tăng theo time-to-expire. Chỉ 1 request refresh, others serve stale.

public Product findByIdWithEarlyRefresh(String productId) {
    String key = "product:" + productId;
    CachedProduct cached = getCachedWithMetadata(key);

    if (cached == null) {
        return loadAndCache(productId);  // True miss
    }

    long ttl = cached.expiresAt() - System.currentTimeMillis();
    long delta = cached.lastFetchDuration();  // How long last DB query took
    double beta = 1.0;  // Tuning parameter

    // Probabilistic refresh — likelihood tăng khi gần expire
    // Formula: now() - delta * beta * ln(rand()) >= expiresAt
    if (System.currentTimeMillis() - delta * beta * Math.log(Math.random()) >= cached.expiresAt()) {
        // Refresh async, return stale value
        CompletableFuture.runAsync(() -> loadAndCache(productId));
    }

    return cached.product();
}

Solution 3: Two-Level TTL (Soft + Hard)

// Hard TTL: actual expiry (10 min)
// Soft TTL: stored in value, app refresh proactively at 8 min

public Product findById(String productId) {
    String key = "product:" + productId;
    CachedValue cached = getCachedWithSoftTtl(key);

    if (cached == null) {
        // Hard miss
        return loadAndCacheWithTtls(productId, Duration.ofMinutes(10), Duration.ofMinutes(8));
    }

    if (cached.isSoftExpired()) {
        // Async refresh — others continue to serve stale value
        CompletableFuture.runAsync(() -> loadAndCacheWithTtls(productId, ...));
    }
    return cached.product();
}
🎯 Khi chọn strategy nào?
  • Lock pattern: simple, work cho most cases. Trade-off: extra Redis round-trip cho mỗi miss.
  • Probabilistic early refresh: smooth, no thundering herd, no lock overhead. Trade-off: complex math, tuning.
  • Two-level TTL: simple, predictable. Trade-off: app code phức tạp hơn.

Production lớn (Facebook, Twitter) thường combine: lock cho first-load, probabilistic cho refresh.

5. Invalidation Strategies CORE

★ CORE — "Cache invalidation là 1 trong 2 hard things trong CS" (Phil Karlton). 4 strategies, mỗi cái có failure mode riêng.

Strategy 1: TTL Only — Simplest

redis.opsForValue().set(key, value, Duration.ofMinutes(5));
// Stale data tolerable trong 5 min window

Pros: simple, no coordination. Cons: stale data window (5 min), khó cho real-time data.

Strategy 2: Write-Through Invalidation

@Transactional
public Product update(String productId, ProductUpdate update) {
    Product product = repo.update(productId, update);
    redis.delete("product:" + productId);  // Invalidate ngay
    return product;
}

Pros: cache fresh ngay sau write. Cons: dual-write problem — nếu DB commit OK nhưng Redis DELETE fail (Redis down), cache có stale data đến TTL expire.

Strategy 3: Event-Driven Invalidation

// Producer
@Transactional
public Product update(String productId, ProductUpdate update) {
    Product product = repo.update(productId, update);
    // Save event vào outbox table (cùng transaction)
    outboxRepo.save(OutboxEvent.of(new ProductUpdatedEvent(productId)));
    return product;
}

// Consumer (separate service hoặc background poller)
@KafkaListener(topics = "product-events")
public void on(ProductUpdatedEvent event) {
    redis.delete("product:" + event.productId());
}

Pros: reliable invalidation (Outbox guarantees). Scale to many cache instances. Cons: complex setup (event infrastructure).

Strategy 4: Version-Based

// Key includes version
String key = "product:" + productId + ":v" + currentVersion;

// On update, bump version (atomic)
Long newVersion = redis.opsForValue().increment("product_version:" + productId);
// Old keys still exist, expire via TTL
// New reads use new key, miss → load fresh

Pros: no invalidation race condition. Cons: memory bloat (old versions linger), need version tracking.

🎯 Edge Cases — Cache Invalidation Production Reality
  • Race condition: Thread A read DB → Thread B update DB + DELETE cache → Thread A SET cache với old value. Now cache stale forever (until TTL). Fix: cache-aside với version check, hoặc dùng Redis transaction (MULTI/EXEC).
  • Distributed cache: 3 Redis instances. UPDATE invalidate chỉ 1 instance → 2 còn lại serve stale. Fix: invalidate all instances qua pub/sub event.
  • Cascade invalidation: product updated → invalidate product cache. Nhưng order details cache include product info → cũng stale. Track dependencies hoặc dùng tag-based invalidation.
  • Mass invalidation thundering herd: deploy new product catalog version → DELETE 100K keys một lúc → next reads all miss → DB stampede. Fix: gradual invalidation (10K/min) hoặc soft TTL.
  • Cache poison: bug code write wrong value to cache. TTL chưa hết → all users see wrong data. Fix: version key trong code (deploy = increment version → old cache abandoned).
  • Memory pressure: Redis hit maxmemory → eviction policy kick in. LRU evict hot keys không expected → cache miss spike. Monitor evictions.
  • Negative cache: 404 response cached → real product created sau → 404 vẫn return từ cache đến TTL. Fix: short TTL cho negative cache (30s), invalidate on insert.
  • Cache stale do clock skew: app server clock lệch Redis → TTL calculation sai. Sync time với NTP.

6. Distributed Lock CORE

★ CORE — Cực kỳ dễ bug. Hiểu SETNX gotchas + Redlock controversy là dấu hiệu senior.

Naive Implementation — Bug Everywhere

❌ Naive SETNX
// Acquire lock
Boolean ok = redis.opsForValue()
    .setIfAbsent("lock:order:123", "1");
if (Boolean.TRUE.equals(ok)) {
    try {
        processOrder();
    } finally {
        redis.delete("lock:order:123");
    }
}
// Bugs:
// 1. No TTL → worker crash → lock held forever
// 2. No owner check → DEL by wrong process
// 3. No retry strategy
✅ Proper distributed lock
String lockKey = "lock:order:123";
String lockValue = UUID.randomUUID().toString();
Duration ttl = Duration.ofSeconds(30);

// Atomic SET với NX (only if not exists) + EX (TTL)
Boolean acquired = redis.opsForValue()
    .setIfAbsent(lockKey, lockValue, ttl);

if (Boolean.TRUE.equals(acquired)) {
    try {
        processOrder();
    } finally {
        // Release ONLY if we still own (atomic Lua)
        String luaScript = """
            if redis.call('get', KEYS[1]) == ARGV[1]
            then return redis.call('del', KEYS[1])
            else return 0 end
            """;
        redis.execute(new DefaultRedisScript<>(luaScript, Long.class),
            List.of(lockKey), lockValue);
    }
}
🎯 Vì sao "check ownership" qua Lua script?

Scenario without ownership check:

  1. Worker A acquire lock với TTL 30s
  2. Worker A pause (GC, network) 35s
  3. Lock TTL expire
  4. Worker B acquire lock với same key
  5. Worker A resume → tưởng vẫn hold lock → call DELETE → delete lock của Worker B!

Ownership check (compare value before delete) ngăn this. Lua script ensure atomicity của check + delete (không race condition giữa GET và DEL).

Redlock Controversy

Sanfilippo propose Redlock algorithm cho distributed lock across multiple Redis instances. Martin Kleppmann publish famous critique 2016. Tóm tắt:

Concern (Kleppmann)Response (Sanfilippo)
Clock skew between Redis nodes affect TTL correctnessReasonable bounded clock skew acceptable cho most use cases
GC pause của client > TTL → lock expire while client think holdingAdd fencing token, validate at resource side
Network partition + master/slave failover → lock duplicateMulti-node algorithm reduces this risk
⚠️ Practical guidance

Distributed lock với Redis (single or Redlock) OK cho "efficiency" (avoid duplicate work) nhưng NOT OK cho "correctness" (mutual exclusion guarantee critical).

  • Efficiency use case: cache regeneration (2 workers regenerate cache cùng lúc = waste, không critical). Redis lock đủ.
  • Correctness use case: financial transactions, inventory deduction. Cần fencing tokens + resource-side validation. Hoặc dùng database transaction trực tiếp.

Library recommend: Redisson (Java) — implement Redlock + auto-renew + fair lock. Production-tested.

// Redisson example — production-ready
@Autowired RedissonClient redisson;

public void processOrder(String orderId) {
    RLock lock = redisson.getLock("order:lock:" + orderId);
    try {
        // Wait up to 5s, hold lock 30s, auto-renew nếu thread alive
        boolean acquired = lock.tryLock(5, 30, TimeUnit.SECONDS);
        if (!acquired) {
            throw new LockAcquisitionFailedException();
        }
        // Process safely
        processOrder();
    } finally {
        if (lock.isHeldByCurrentThread()) lock.unlock();
    }
}

Inventory Deduction — Atomic Without Lock

Cho flash sale "100 units", thay vì lock + check-decrement, dùng Lua atomic:

String luaScript = """
    local stock = tonumber(redis.call('get', KEYS[1]))
    if stock == nil or stock <= 0 then
        return -1
    end
    redis.call('decr', KEYS[1])
    return stock - 1
    """;

Long remaining = redis.execute(new DefaultRedisScript<>(luaScript, Long.class),
    List.of("inventory:iphone15"));

if (remaining < 0) {
    throw new OutOfStockException();
}
// remaining = current stock after decrement

Lua script execute atomically trong Redis single-thread → no race. Không cần distributed lock cho check-decrement pattern.

7. Pub/Sub & Streams SUPPLEMENTARY

SUPPLEMENTARY — Lightweight messaging. Khi nào dùng vs Kafka.

Pub/Sub — Fire-and-Forget

// Publisher
redis.convertAndSend("channel:notifications", "New order from user 123");

// Subscriber
@RedisListener(topic = "channel:notifications")
public void onNotification(String message) {
    log.info("Received: {}", message);
}
⚠️ Pub/Sub limitations
  • No persistence: subscriber offline → miss messages forever
  • No acknowledgment: publisher không biết subscribers nhận thành công
  • No replay: không thể re-process old messages

Phù hợp cho: real-time notifications không critical (chat presence, live update), cache invalidation events.

Streams (Redis 5+) — Kafka-like Persistence

// Publisher
Map<String, String> data = Map.of("orderId", "123", "amount", "500000");
redis.opsForStream().add("orders", data);

// Consumer with consumer group (load balance, ack)
redis.opsForStream().createGroup("orders", ReadOffset.from("0"), "order-processors");

List<MapRecord<String, String, String>> records = redis.opsForStream()
    .read(Consumer.from("order-processors", "worker-1"),
          StreamOffset.create("orders", ReadOffset.lastConsumed()));

for (var record : records) {
    process(record.getValue());
    redis.opsForStream().acknowledge("orders", "order-processors", record.getId());
}
🎯 Redis Streams vs Kafka?
  • Redis Streams: simpler setup, lower latency, suitable up to ~50K msg/sec. Persistence via RDB/AOF. Best cho moderate-scale event streaming.
  • Kafka: high throughput (millions msg/sec), better durability guarantees, ecosystem (Kafka Streams, Connect). Required cho large-scale.

Rule of thumb: if < 10K events/sec và đã có Redis → Streams. If > 100K events/sec → Kafka. Between: depends on existing infrastructure.

8. Persistence & HA ADVANCED

ADVANCED — Ops-level concern. Dev cần biết khi nào worry.

RDB vs AOF

RDB (snapshot)AOF (append-only log)
FormatBinary snapshotAppend every write command
RecoveryFast loadReplay log, slower
Data loss windowMinutes (interval)1 second (fsync every sec)
Disk usageCompactLarger (grows with writes)
Use caseBackup, lower durabilityHigh durability needed

Production recommendation: enable cả hai. RDB cho fast restore, AOF cho minimal data loss. Trade-off: write performance ~10% slower với AOF.

HA Setup — Redis Sentinel vs Redis Cluster

  • Sentinel: 1 master + N replicas + sentinel nodes monitor. Auto-failover. Same data, no sharding. Good cho <= 50GB data.
  • Cluster: sharded (16384 slots). Auto-failover per shard. Linear scale. Required cho data > 50GB hoặc throughput > 100K ops/sec.

9. Anti-Patterns CORE

#1 Using Redis as Primary Database

"Redis fast, lưu all data đó." Sai vì: data in RAM (lost if crash), expensive vs disk, không có ACID transactions full, query capabilities limited. Redis là cache/auxiliary, not primary store.

#2 KEYS command in production

# ❌ KEYS blocks entire Redis O(n)
KEYS user:*
# Production với 1M keys → 5s block → all clients timeout

# ✅ SCAN cursor-based
SCAN 0 MATCH user:* COUNT 100

#3 Storing huge values

1MB value = 1MB network transfer per GET. Redis single-thread blocked. Latency spike. Solution: split, paginate, hoặc reconsider use case.

#4 No memory limit

Redis grow indefinitely → OOM kill → data loss. Always set maxmemory + eviction policy (allkeys-lru for cache, noeviction for primary store).

#5 Hot key

1 key receive 90% traffic (celebrity user, hot product). Single shard overloaded. Solutions: client-side cache, replicate key with multiple suffixes (product:42:#1, product:42:#2), application-level sharding.

#6 Pipeline misuse

Forget to pipeline → N round trips for N ops. Use MULTI/EXEC hoặc client pipeline để batch.

#7 Not handling connection failures

Network blip → Redis client throw exception → app crash. Use connection pool with retry, circuit breaker, fallback to DB.

#8 Forgetting TTL on session keys

User session stored forever → memory bloat. Always TTL.

11. Interview Framework

Tầng 1 — SurfaceMid gate
Q: Cache-aside pattern là gì, vì sao phổ biến?
"Cache-aside: app explicit manage cache. Read flow: GET cache → MISS → SELECT DB → SET cache. Write flow: UPDATE DB → DELETE cache.

Phổ biến vì: app control (chọn data nào cache, TTL khác nhau), failure tolerant (Redis down → fall back DB), simple mental model. Trade-off: invalidation explicit (dễ quên), first-time always slow.

Alternatives: Read-through (Spring @Cacheable — transparent nhưng magic), Write-through (write to cache + DB sync), Write-behind (write cache, async write DB — fast but data loss risk)."
Q: Khi nào dùng Hash vs String?
"String cho simple key-value (counter, single-field cache, JSON-serialized object nhỏ).

Hash khi object có 5+ fields và need partial update. Lý do: với String JSON, update 1 field = deserialize all → modify → re-serialize → SET. O(n). Hash có HSET per field, O(1), atomic.

Plus: HGETALL bandwidth có thể hơn String GET 1 large JSON.

Example: session cache với 20 fields (last_active, ip, role, ...) → Hash. Cache product với 5 fields rarely update → String OK."
Tầng 2 — Deep DiveMid → Senior
Q: Cache stampede là gì, design solution.
"Cache stampede: hot key expire → nhiều concurrent requests miss cache → tất cả hit DB simultaneously → DB overload.

Concrete: product 'iPhone 15' TTL 5 min. Mỗi giây receive 5K requests. Tại moment expire, 5K requests cùng miss → 5K concurrent DB queries → connection pool exhausted → DB crash.

3 Solutions:

1. Lock pattern (single-flight): first request acquire Redis lock, load DB + populate cache. Others wait briefly + retry cache, hoặc serve stale fallback. Simple, work cho most cases.

2. Probabilistic early refresh (XFetch): refresh cache trước khi expire với probability tăng theo time-to-expire. Chỉ 1 request refresh, others serve stale. Smoother no thundering herd.

3. Two-level TTL: hard TTL 10 min (actual expiry), soft TTL 8 min (stored trong value). Khi soft expire → async refresh, return stale. No miss spike.

Production combine: lock cho first load, probabilistic cho refresh. Plus monitoring để detect stampede risk.

Additional: rate limit cache miss path, circuit breaker on DB if overloaded."
Q: Distributed lock với Redis — naive vs production-ready?
"Naive SETNX has 3 critical bugs:
1. No TTL → worker crash → lock held forever, deadlock
2. No ownership check → wrong process can DELETE other's lock
3. No retry strategy → caller hard-fails

Production fix:
• SET key uuid NX EX 30 — atomic SET with TTL + check-not-exists
• Release với Lua script: check ownership (GET match UUID) before DELETE — atomic in Redis single-thread
• Use library like Redisson for fair lock, auto-renewal, etc.

Redlock controversy: Kleppmann critique (clock skew, GC pause). Sanfilippo defend. Practical guidance:
• OK for efficiency (avoid duplicate work, e.g., cache regeneration)
• NOT OK for correctness (financial mutex). Use database transaction or fencing tokens.

Alternative for inventory deduction: Lua atomic DECR check (no lock needed). Redis single-thread guarantees atomicity."
Tầng 3 — ArchitectureSenior
Q: Design Redis architecture cho e-commerce platform handling flash sales (1M concurrent users, 100K QPS). Hot products, inventory deduction, cache invalidation, fault tolerance.
"Multi-tier Redis architecture:

Tier 1 — L1 Cache (in-process Caffeine)
• Per-app-instance JVM cache, ~10K entries, 60s TTL
• Latency < 1ms
• Reduce Redis load 70-80% for super-hot keys
• Invalidate via Redis pub/sub event

Tier 2 — Redis Cluster (L2)
• 6-node cluster (3 master + 3 replica), 16384 slots
• Sharded by product_id hash
• 100GB+ capacity, 1M ops/sec aggregate
• maxmemory + allkeys-lru eviction

Tier 3 — PostgreSQL (truth source)
• Behind L1 + L2
• Read-only access from app (writes through Redis for hot data)

Hot product handling:
• Replicate hot key with suffixes: product:iphone15:#1 through #10 (sharded by user_id % 10)
• L1 hot cache absorb most reads
• Pre-warm cache before flash sale start

Inventory deduction (flash sale 100 units):
• Lua atomic script: DECR if > 0
• Redis single-thread guarantees no oversell
• Async sync to DB for persistence + reporting
• Idempotency check to handle network retry

Cache invalidation:
• Event-driven: product update → publish to Kafka → consumer invalidate Redis + broadcast to L1
• Outbox pattern guarantee delivery
• < 200ms propagation cross all app instances

Cache stampede prevention:
• Two-level TTL (hard + soft)
• Lock pattern fallback
• Rate limit cache miss path to DB

Failure tolerance:
• Circuit breaker: Redis down → degrade to DB direct (limited capacity, queue requests)
• Stale-while-revalidate: serve old cache value if Redis temporarily unavailable
• Sentinel auto-failover for cluster nodes

Observability:
• Metrics: hit rate L1/L2, eviction rate, latency p50/p95/p99, hot keys (CLUSTER COUNTKEYSINSLOT)
• Alerts: hit rate < 80%, latency p99 > 10ms, evictions > 1000/min
• Tracing across L1 → L2 → DB chain

Capacity planning:
• L1: 10K entries × 5KB = 50MB per instance × 50 instances = 2.5GB total
• L2: 1M products × 5KB = 5GB; 10M sessions × 2KB = 20GB; total ~30GB → 6-node cluster với 8GB each
• Network: 100K QPS × 5KB avg = 500MB/sec → 4Gbps network adequate"
