#49 · Junior Foundation

Exception & Logging — Junior Survival Kit

80% thời gian debug production đến từ exception handle sai + log thiếu context. File này lấp đúng gap đó cho junior 0-2 năm.

CORE ★ 1 tuần đọc + setup logback thật Domain: E-commerce / SaaS

⚠️ Cảnh báo "hiểu giả" — đặc trưng Exception & Logging

Đây là 2 chủ đề junior dễ "đọc xong tưởng hiểu" nhất, vì syntax đơn giản và đã quen thuộc. Bạn rơi vào hiểu giả nếu:

  • Biết viết try-catch, nhưng KHÔNG giải thích được khác biệt catch(Exception e) vs catch(SQLException e) ngoài "specific hơn"
  • Biết dùng SLF4J, nhưng KHÔNG biết tại sao log.info("user=" + u) tệ hơn log.info("user={}", u)
  • Biết khái niệm MDC, nhưng chưa từng debug 1 request bằng traceId
  • Biết "đừng log password", nhưng không biết list cụ thể PII nào không được log
🚨 3 cấp độ — bạn đang ở đâu?

L1 (Recognize): biết tên try-catch, log levels. Code chạy nhưng debug khó.
L2 (Explain): giải thích được why của từng anti-pattern, dùng MDC + parameterized log.
L3 (Apply): production debug 3h sáng dùng traceId search Kibana ra root cause <15 phút.

📖 5 quy tắc đọc — KHÔNG đọc một mạch

  1. Mỗi Pause & Predict phải dừng lại, viết câu trả lời ra giấy/Notes. KHÔNG đọc đáp án ngay.
  2. Mỗi anti-pattern đọc xong, tự nói thành lời: "code này sai vì... nếu chạy production sẽ gây..."
  3. Code phải gõ tay (không copy-paste), tự chạy 1 file Spring Boot demo.
  4. Phần MDC bắt buộc làm Lab — setup filter, gửi 3 request đồng thời, verify traceId riêng.
  5. Cuối file: Feynman test 5 câu. Bí 1 câu = quay lại section tương ứng.

1. Bài toán thực tế — 3 câu chuyện chết production CORE ★

📖 Câu chuyện #1 — "Order biến mất, log chỉ có 1 dòng 'error'"

Bối cảnh: Order service của sàn TMĐT. 3 giờ sáng PagerDuty kêu: "order create error rate 12%". On-call ssh vào server, đọc log:

ERROR
ERROR
ERROR
something went wrong
process failed

Không có stack trace. Không user ID. Không order ID. Code:

try {
    orderService.create(req);
} catch (Exception e) {
    log.error("error");  // ← chỉ có thế này
    return ResponseEntity.status(500).build();
}

Hậu quả: Mất 6 tiếng reproduce trên staging. Cuối cùng phát hiện là NullPointerException do thiếu validate address == null trong checkout API mới deploy. Mất 1.2 tỷ doanh thu trong 6h.

Bài học: Log error mà không có context = không log gì. Phải có stack trace + business ID + user ID.

📖 Câu chuyện #2 — "Log file có 50K dòng password"

Bối cảnh: Junior debug login flow, để tiện thì:

log.info("Login attempt: email={}, password={}", email, password);

Code merge lên production. Log ELK index 7 ngày = 50K dòng password customer plaintext. Security team phát hiện trong audit định kỳ. Compliance team escalate:

  • GDPR violation tiềm năng — fine €20M hoặc 4% revenue
  • Phải force reset password 50K customer
  • Phải khai báo với cơ quan bảo vệ dữ liệu trong 72h
  • Junior bị warning, manager bị PIP

Bài học: 1 dòng log sai = thiệt hại lớn hơn 1 năm lương junior. Luôn có "PII never-log list" trong tay.

📖 Câu chuyện #3 — "e.printStackTrace() không vào log file"

Bối cảnh: Payment integration service. Junior viết:

try {
    paymentGateway.charge(orderId, amount);
} catch (Exception e) {
    e.printStackTrace();  // ← chỗ này
    throw new RuntimeException(e);
}

Production có bug rằng 0.5% request fail. On-call mở Kibana search "PaymentService" — không có ERROR log nào về charge. Tại sao? Vì printStackTrace() in ra System.err, mà container Docker đã redirect stderr ra /dev/null.

Hậu quả: Mất 2 tuần tracking bug. Cuối cùng phải dùng tcpdump capture HTTP request giữa app và payment gateway để biết bug.

Bài học: printStackTrace() KHÔNG bao giờ dùng trong production. Luôn dùng SLF4J logger.

⏸ Pause & Predict — trước khi đọc tiếp
Viết ra giấy:
  1. 3 câu chuyện trên, mỗi cái có 1 anti-pattern. Tên đúng anti-pattern là gì?
  2. Trong câu chuyện #2, nếu phải fix nhanh (hot-fix), bạn sẽ làm gì với 50K dòng password đã có trong ELK index?
  3. Câu chuyện #3, nếu KHÔNG có log, dùng cách nào diagnose bug 0.5% trong payment service? Đề xuất 3 cách.
Đáp án ở Section 5 (anti-patterns) và Section 9 (logging anti-patterns). Đừng đọc đáp án ngay.

2. Exception Hierarchy + Checked vs Unchecked CORE ★

Trước khi biết cách handle exception, phải biết exception là gì. Hierarchy quyết định "có buộc handle hay không", "khi nào catch", "khi nào throw".

Java Throwable Hierarchy

Throwable
├── Error                           ← Đừng bao giờ catch
│   ├── OutOfMemoryError            (JVM hết heap)
│   ├── StackOverflowError          (recursion vô hạn)
│   └── NoClassDefFoundError        (classpath sai)
│
└── Exception
    ├── RuntimeException            ← UNCHECKED — không bắt buộc handle
    │   ├── NullPointerException      (bug logic — quên null check)
    │   ├── IllegalArgumentException  (input validation fail)
    │   ├── IllegalStateException     (object state sai)
    │   ├── ClassCastException        (cast sai type)
    │   ├── IndexOutOfBoundsException
    │   └── 90% custom exception nên extend cái này
    │
    └── (Checked Exception)         ← CHECKED — phải handle hoặc throws
        ├── IOException              (file, network)
        ├── SQLException             (database)
        ├── InterruptedException     (thread)
        └── ClassNotFoundException

3 loại — học thuộc bảng này

LoạiNghĩaCatch hay không?Ví dụ
ErrorJVM problem, app không recover được❌ KHÔNG catchOutOfMemoryError
RuntimeException (unchecked)Bug logic hoặc validation. Không buộc compile-time handleCatch nếu biết phải làm gìNPE, IllegalArgument, custom business exception
Checked ExceptionExternal I/O. Caller được "buộc" handle hoặc throwsBẮT BUỘC handle hoặc declareIOException, SQLException
⏸ Pause & Predict — Checked vs Unchecked
Viết ra giấy:
  1. Spring framework wrap SQLException thành DataAccessException (unchecked). Tại sao?
  2. Khi bạn viết custom OrderNotFoundException, nên extend Exception hay RuntimeException? Vì sao?
  3. Java có thêm checked exception lần cuối khoảng năm 2005. Tại sao modern Java tránh checked exception?
Đáp án ngay dưới — viết câu trả lời trước.

📌 Đáp án Pause & Predict

  1. Spring wrap SQLException: vì SQLException quá generic (error code 1062 = duplicate key MySQL khác Postgres). Spring chuyển thành hierarchy nghĩa (DuplicateKeyException, DataIntegrityViolationException, ...) → caller switch theo loại, không phải parse error code thủ công. Đồng thời chuyển sang unchecked vì caller hiếm khi recover được — chỉ retry hoặc fail.
  2. OrderNotFoundException → extends RuntimeException: vì caller thường không recover được (không tìm thấy order → trả 404). Force caller declare throws mọi nơi = noise. Modern style: 90% custom exception là unchecked, chỉ checked khi caller THỰC SỰ có thể recover (vd: retry I/O).
  3. Modern Java tránh checked: clutter signature (throws IOException lan khắp call chain), lambda + functional interface không support checked exception tốt (Function<T,R> không throws), caller thường wrap thành RuntimeException dù sao → noise không có giá trị. Spring/Jakarta EE đều dùng unchecked-first.

Counter-example — Khi nào KHÔNG dùng custom exception

❌ KHÔNG tạo custom exception khi…
  • Java đã có exception đúng nghĩa: IllegalArgumentException cho invalid argument, IllegalStateException cho state sai. Đừng tạo InvalidEmailException nếu chỉ cần throw new IllegalArgumentException("invalid email")
  • Exception chỉ throw 1 chỗ và caller không cần phân biệt: tạo class mới = boilerplate không có giá trị
  • Để "đẹp" mà không thêm field/errorCode: nếu custom exception chỉ extend RuntimeException thêm tên thì không khác gì RuntimeException với message khác nhau

Quy tắc: Tạo custom exception khi caller cần distinguish (catch riêng từng loại), HOẶC khi cần thêm structured data (errorCode, requiredAmount, availableAmount).

3. try / try-with-resources / multi-catch CORE ★

3.1 try-catch-finally — pattern cũ

FileInputStream fis = null;
try {
    fis = new FileInputStream("orders.csv");
    // process
} catch (FileNotFoundException e) {
    log.error("File not found", e);
    throw new ImportException(e);
} catch (IOException e) {
    log.error("IO error", e);
} finally {
    // Luôn chạy, dù có exception hay không
    if (fis != null) {
        try {
            fis.close();
        } catch (IOException e) {
            log.warn("Failed to close", e);
        }
    }
}

→ Verbose, dễ quên close, dễ NPE check. Java 7+ có try-with-resources thay thế.

3.2 try-with-resources (Java 7+) — pattern mới

// ✅ Auto-close mọi resource implements AutoCloseable
try (FileInputStream fis = new FileInputStream("orders.csv");
     BufferedReader reader = new BufferedReader(new InputStreamReader(fis))) {

    String line;
    while ((line = reader.readLine()) != null) {
        process(line);
    }
} catch (IOException e) {
    log.error("Order import failed: file=orders.csv", e);
    throw new ImportException("Failed to import orders", e);
}
// fis, reader auto-close kể cả có exception
// Close theo thứ tự NGƯỢC: reader trước, fis sau

3.3 Multi-catch (Java 7+)

try {
    riskyOperation();
} catch (IOException | SQLException e) {  // 1 block xử lý 2 type
    log.error("External resource failed", e);
    throw new ServiceException(e);
}

3.4 Bẫy finally — return bị override

🚨 Câu interview kinh điển
public int trickQuestion() {
    try {
        return 1;
    } finally {
        return 2;  // ❌ Finally override return của try!
    }
}
// Output: 2

Quy tắc: KHÔNG return trong finally. Chỉ cleanup, không thay đổi flow.

Tương tự: KHÔNG throw exception trong finally — sẽ swallow exception gốc.

⏸ Pause & Predict — Code này print gì?
public static String mystery() {
    try {
        throw new RuntimeException("from try");
    } catch (Exception e) {
        throw new RuntimeException("from catch");
    } finally {
        return "from finally";  // có chạy không?
    }
}
public static void main(String[] args) {
    System.out.println(mystery());
}
Predict output trước khi đọc đáp án.
✅ Đáp án

Output: "from finally". Cả 2 exception bị nuốt mất. Đây là tại sao KHÔNG return trong finally — exception bubble up bị mất silently, debugging gần như không thể.

4. Custom Exception cho E-commerce Domain CORE ★

Domain example: e-commerce order checkout. Các exception cần distinguish ở client + server.

Base class + concrete

// Base class cho mọi domain exception
public abstract class OrderException extends RuntimeException {
    private final String errorCode;

    protected OrderException(String errorCode, String message) {
        super(message);
        this.errorCode = errorCode;
    }

    protected OrderException(String errorCode, String message, Throwable cause) {
        super(message, cause);
        this.errorCode = errorCode;
    }

    public String getErrorCode() { return errorCode; }
}

// Concrete exception cho từng case
public class OrderNotFoundException extends OrderException {
    public OrderNotFoundException(String orderId) {
        super("ORDER_NOT_FOUND", "Order not found: " + orderId);
    }
}

public class InsufficientInventoryException extends OrderException {
    private final String productId;
    private final int requested;
    private final int available;

    public InsufficientInventoryException(String productId, int requested, int available) {
        super("INSUFFICIENT_INVENTORY",
            String.format("Product %s: requested %d, available %d", productId, requested, available));
        this.productId = productId;
        this.requested = requested;
        this.available = available;
    }

    public String getProductId() { return productId; }
    public int getRequested() { return requested; }
    public int getAvailable() { return available; }
}

public class PaymentDeclinedException extends OrderException {
    public PaymentDeclinedException(String orderId, String reason) {
        super("PAYMENT_DECLINED",
            "Payment declined for order " + orderId + ": " + reason);
    }
}

5 quy tắc viết custom exception

  1. Extend RuntimeException trừ khi caller thật sự cần handle (recovery)
  2. Có errorCode để client/log machine-readable (UI mapping, alert rule)
  3. Constructor giữ cause: super(message, cause) — KHÔNG nuốt mất cause
  4. Message có context: ID, value, expected vs actual — debug ngay từ log
  5. Field bổ sung nếu caller cần biết detail (như requested, available trên đây)

Spring @ControllerAdvice — Map exception → HTTP

@RestControllerAdvice
public class GlobalExceptionHandler {
    private static final Logger log = LoggerFactory.getLogger(GlobalExceptionHandler.class);

    @ExceptionHandler(OrderNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(OrderNotFoundException e) {
        log.warn("Order not found: {}", e.getMessage());  // WARN — expected
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
            .body(new ErrorResponse(e.getErrorCode(), e.getMessage()));
    }

    @ExceptionHandler(InsufficientInventoryException.class)
    public ResponseEntity<ErrorResponse> handleInsufficient(InsufficientInventoryException e) {
        log.warn("Insufficient inventory: product={}, requested={}, available={}",
            e.getProductId(), e.getRequested(), e.getAvailable());
        return ResponseEntity.status(HttpStatus.CONFLICT)
            .body(new ErrorResponse(e.getErrorCode(), e.getMessage(),
                Map.of("productId", e.getProductId(),
                       "requested", e.getRequested(),
                       "available", e.getAvailable())));
    }

    @ExceptionHandler(PaymentDeclinedException.class)
    public ResponseEntity<ErrorResponse> handlePayment(PaymentDeclinedException e) {
        log.warn("Payment declined: {}", e.getMessage());
        return ResponseEntity.status(HttpStatus.PAYMENT_REQUIRED)
            .body(new ErrorResponse(e.getErrorCode(), e.getMessage()));
    }

    // Catch-all cuối cùng — log ERROR vì unexpected
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleUnexpected(Exception e) {
        log.error("Unexpected error", e);  // full stack trace
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
            .body(new ErrorResponse("INTERNAL_ERROR", "Internal server error"));
        // ⚠️ KHÔNG leak stack trace vào response — security risk + privacy
    }
}
💡 Tại sao log level khác nhau cho từng exception?
  • WARN cho expected business exception (NotFound, InsufficientInventory, PaymentDeclined) — đây là "user input không hợp lệ", không phải bug của ta
  • ERROR cho unexpected (NPE, generic Exception) — đây là "code của ta có bug" hoặc "downstream chết", cần on-call

Nếu log mọi exception là ERROR → on-call bị spam, bỏ qua alert thật. Đây là "alert fatigue" — failure mode #1 của observability.

5. 7 Anti-Patterns Exception CORE ★

Junior nhìn từng cái nhận ra "tôi từng viết như vậy". Mỗi anti-pattern có symptom production cụ thể.

❌ Anti-pattern #1: Swallow exception (im lặng)
// ❌
try { riskyCall(); } catch (Exception e) { /* empty */ }

// ✅
try { riskyCall(); } catch (Exception e) {
    log.error("Risky call failed", e);
    throw new ServiceException("Risky operation failed", e);
}

Symptom production: User báo "checkout không tạo order" nhưng không có error log nào. Reproduce trên staging cũng không thấy. Vì exception bị nuốt silently.

Cách detect: code review tìm pattern catch (Exception e) { } — KHÔNG có log + KHÔNG throw = swallow.

❌ Anti-pattern #2: Mất cause khi rethrow
// ❌ Mất stack trace gốc
catch (IOException e) {
    throw new RuntimeException(e.getMessage());  // ← chỉ lấy message
}

// ✅ Giữ cause
catch (IOException e) {
    throw new RuntimeException("Failed to read file: " + path, e);  // ← truyền e
}

Symptom: Stack trace trong log chỉ thấy line trong service, không thấy IOException gốc. Debug như "tại sao service fail" mà không biết "vì sao IO fail".

❌ Anti-pattern #3: Catch quá rộng
// ❌ Catch Exception bắt cả NPE bug logic
try {
    user.getProfile().getAddress().getCity();  // NPE chỗ nào?
} catch (Exception e) {
    return "Unknown";  // Bug về NPE bị che mất
}

// ✅ Catch specific
try {
    apiClient.fetch(orderId);
} catch (TimeoutException e) {
    log.warn("API timeout for order {}, returning cached value", orderId);
    return cache.get(orderId);
}

Symptom: Bug logic (như NPE do quên null check) bị treat như "external error", user nhận response sai dữ liệu, không có alert.

❌ Anti-pattern #4: e.printStackTrace()
// ❌ In ra System.err — không vào log file production
catch (Exception e) {
    e.printStackTrace();
}

// ✅ Dùng logger
catch (Exception e) {
    log.error("Operation failed: orderId={}", orderId, e);
}

Symptom: Như câu chuyện #3 ở Section 1 — Docker redirect stderr → /dev/null → log mất hoàn toàn.

❌ Anti-pattern #5: throws Exception (quá generic)
// ❌ Lazy
public void processOrder() throws Exception { }
// Caller không biết exception nào có thể xảy ra

// ✅ Specific hoặc unchecked
public void processOrder() throws OrderProcessingException, ValidationException { }
// hoặc
public void processOrder() { /* throws unchecked, document trong Javadoc */ }

Symptom: Caller buộc catch Exception theo style cascade → tất cả callsite catch generic → mất signal về exception nào quan trọng.

❌ Anti-pattern #6: Catch rồi rethrow trống rỗng
// ❌ Rethrow không thêm context
catch (SQLException e) {
    throw new RuntimeException(e);
}

// ✅ Thêm context: ID, operation, what user was doing
catch (SQLException e) {
    throw new DatabaseException(
        "Failed to update inventory for product " + productId +
        " during checkout of order " + orderId, e);
}

Symptom: Log có stack trace nhưng phải đọc từng line code mới biết context. Senior debug 30 phút thay vì 30 giây.

❌ Anti-pattern #7: Exception cho control flow
// ❌ Dùng exception thay vì if/else
public boolean userExists(String id) {
    try {
        userRepo.findById(id).get();  // throws NoSuchElementException
        return true;
    } catch (NoSuchElementException e) {
        return false;
    }
}
// Exception là expensive (stack trace cost ~1-10μs) — KHÔNG cho luồng bình thường

// ✅ Optional / boolean
public boolean userExists(String id) {
    return userRepo.findById(id).isPresent();
}

Symptom: Performance test thấy method nhanh khi 100% exist, chậm 100× khi 50% not-exist. JFR profile show fillInStackTrace chiếm CPU. Fix bằng cách thay exception bằng Optional.

6. 5 Log Levels CORE ★

LevelKhi nào dùngProduction có log không?Volume ước tính
ERRORException không expected, on-call phải xử lý✅ Luôn<0.1% requests
WARNBất thường nhưng recovered: retry, fallback, deprecated, business validation fail✅ Luôn<5% requests
INFOBusiness event quan trọng: order created, user logged in, payment processed✅ Luôn1 dòng/request typical
DEBUGChi tiết kỹ thuật, chỉ bật khi debug ad-hoc❌ Không (hoặc tạm bật)5-20 dòng/request khi bật
TRACEPer-step detail, gần như không dùng❌ Khônghàng trăm dòng/request
⚠️ Junior anti-pattern: log mọi thứ là INFO

Junior thường log.info("Entering method X"), log.info("got data Y"), log.info("returning"). Sau 1 ngày: log file 50GB, không filter được, ELK index full.

Rule of thumb: 1 request = 1-3 INFO log (start, key milestone, end). Mọi chi tiết khác là DEBUG.

⏸ Pause & Predict — Đặt log level cho từng tình huống
Bạn sẽ dùng level nào (ERROR/WARN/INFO/DEBUG) cho mỗi case?
  1. User submit form với email sai format
  2. Order tạo thành công
  3. Payment gateway timeout, retry lần 2 thành công
  4. Database connection pool exhausted
  5. Stripe API trả 500 sau 3 lần retry
  6. Cache miss cho product detail
  7. JWT token expired (user phải re-login)
Tip: hỏi "on-call có cần thức dậy lúc 3h sáng vì cái này không?" nếu có → ERROR.
✅ Đáp án
  1. WARN hoặc DEBUG (validation fail là expected user behavior, không phải bug)
  2. INFO (business event quan trọng)
  3. WARN cho lần fail đầu, INFO cho lần retry thành công
  4. ERROR (production issue cần on-call)
  5. ERROR (downstream chết sau retry — escalate)
  6. DEBUG (cache miss là bình thường, chỉ log để debug cache hit rate)
  7. INFO hoặc WARN (expected behavior, security audit có thể cần)

7. SLF4J + Logback Setup SUPPLEMENTARY

Dependency (Spring Boot đã có sẵn)

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>

Cú pháp SLF4J

import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

@Service
public class OrderService {
    // private static final — convention chuẩn
    private static final Logger log = LoggerFactory.getLogger(OrderService.class);

    public Order createOrder(CreateOrderRequest req) {
        // Parameterized — KHÔNG concat string
        log.info("Order creation initiated: customerId={}, items={}, total={}",
            req.getCustomerId(), req.getItems().size(), req.getTotal());

        try {
            Order order = doCreate(req);
            log.info("Order created: orderId={}, total={}", order.getId(), order.getTotal());
            return order;
        } catch (InsufficientInventoryException e) {
            log.warn("Order failed (inventory): customerId={}, product={}",
                req.getCustomerId(), e.getProductId());
            throw e;
        } catch (Exception e) {
            log.error("Order failed unexpectedly: customerId={}", req.getCustomerId(), e);
            // ↑ Truyền `e` ở cuối → SLF4J log full stack trace
            throw e;
        }
    }
}

8. MDC — Mapped Diagnostic Context CORE ★

Section quan trọng nhất của Phần B. MDC là cách add context (traceId, customerId) vào mọi log trong cùng request mà không phải truyền tham số xuống mọi method.

Vì sao cần MDC?

Production debug = ssh vào log file (hoặc Kibana). Mỗi giây có hàng nghìn request, log interleave nhau. Câu hỏi: "log nào thuộc request của user A?"

Không MDC → impossible. Có MDC → search traceId:abc123 ra hết log của request đó, kể cả qua nhiều service.

Implementation — Servlet Filter

@Component
public class RequestLoggingFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res, FilterChain chain)
            throws ServletException, IOException {

        String traceId = req.getHeader("X-Trace-Id");
        if (traceId == null || traceId.isEmpty()) {
            traceId = UUID.randomUUID().toString();
        }
        String customerId = extractCustomerId(req);  // từ JWT/session

        try {
            MDC.put("traceId", traceId);
            if (customerId != null) MDC.put("customerId", customerId);

            res.setHeader("X-Trace-Id", traceId);  // echo back để client biết
            chain.doFilter(req, res);

        } finally {
            MDC.clear();  // ← BẮT BUỘC clear để không leak sang request khác
        }
    }
}

public void sendAsync(String orderId) {

            MDC.clear();
        }

9. 6 Anti-Patterns Logging CORE ★

❌ Anti-pattern #1: Log password / PII
// ❌ DỪNG NGAY!
log.info("Login attempt: username={}, password={}", username, password);
log.debug("User data: {}", user);  // user.toString() có thể chứa password, SSN, credit card

// ✅
log.info("Login attempt: username={}", username);
log.debug("User data: id={}, email={}", user.getId(), maskEmail(user.getEmail()));

PII NEVER log list:

  • Password, PIN, OTP, security question answer
  • Full credit card number (chỉ last 4)
  • SSN / CMND / CCCD / passport đầy đủ
  • Full address (chỉ city/country)
  • Authentication token (JWT, session, API key)
  • Health data, mental health, religion, ethnicity

Hậu quả nếu vi phạm: GDPR €20M hoặc 4% revenue, PCI-DSS fine, audit fail, mất khách hàng enterprise.

❌ Anti-pattern #2: Log trong loop
// ❌ Loop 1M items, mỗi item 1 log → log file 1M dòng
for (Order o : pendingOrders) {
    log.info("Processing order: {}", o.getId());
    process(o);
}

// ✅ Log batch, hoặc DEBUG
log.info("Processing {} pending orders", pendingOrders.size());
for (Order o : pendingOrders) {
    log.debug("Processing order: {}", o.getId());  // DEBUG chỉ bật khi cần
    process(o);
}
log.info("Processed {} orders successfully", pendingOrders.size());

Symptom: ELK index 100GB/ngày, query Kibana timeout, log retention chỉ giữ 1 ngày thay vì 30.

❌ Anti-pattern #3: Log exception nhiều lần
// ❌ Log ở mọi layer
public class Repository {
    public Order find(String id) {
        try { /* */ } catch (SQLException e) {
            log.error("DB error", e);   // log lần 1
            throw new DatabaseException(e);
        }
    }
}
public class Service {
    public Order get(String id) {
        try {
            return repo.find(id);
        } catch (DatabaseException e) {
            log.error("Service failed", e);  // log lần 2
            throw e;
        }
    }
}
public class Controller {
    public Order get(String id) {
        try {
            return service.get(id);
        } catch (Exception e) {
            log.error("Controller error", e);  // log lần 3
            throw e;
        }
    }
}
// → 1 exception, 3 stack trace trong log → noise + ELK index bloat

Quy tắc: Log exception 1 lần ở layer cao nhất bắt nó (thường là @ControllerAdvice). Layer dưới chỉ throw, không log.

❌ Anti-pattern #4: log.error() khi không phải error
// ❌ Validation fail là expected, không phải error
log.error("Invalid email: {}", email);

// ✅ WARN hoặc INFO
log.warn("Invalid email submitted: {}", email);

Symptom: "Alert fatigue" — Slack channel #alerts kêu 100 lần/giờ, on-call bỏ qua, đến khi có real ERROR cũng không ai nhìn.

❌ Anti-pattern #5: System.out.println / printStackTrace
// ❌ Junior thường viết
System.out.println("DEBUG: order = " + order);
e.printStackTrace();

// ✅ Luôn dùng logger
log.debug("Order: {}", order);
log.error("Failed", e);

Như câu chuyện #3 ở Section 1 — Docker stderr → /dev/null, mất log hoàn toàn.

❌ Anti-pattern #6: Log Exception thiếu stack trace
// ❌ Chỉ log message — mất stack trace
catch (Exception e) {
    log.error("Failed: " + e.getMessage());
}

// ✅ Truyền exception, SLF4J xử lý
catch (Exception e) {
    log.error("Failed processing order {}", orderId, e);  // exception ở cuối, không placeholder
}

Symptom: Log có "Failed: null" — vì e.getMessage() return null cho 1 số exception type (như NPE). Không có stack trace → không biết bug ở đâu.

🎯 3 Cấp Độ Mastery CORE ★

Exception Handling

L1 — Recognize (Junior)

Biết try-catch syntax. Phân biệt checked/unchecked. Dùng e.printStackTrace() khi debug. Tạo exception bằng throw new RuntimeException("error").

Symptom: code có pattern catch (Exception e) { e.printStackTrace(); }, không có @ControllerAdvice.

L2 — Explain (Mid)

Giải thích 7 anti-patterns + refactor. Hiểu fail-fast principle. Tạo custom exception hierarchy với errorCode + cause. Dùng try-with-resources. Hiểu vì sao Spring wrap SQLException → DataAccessException.

Symptom: code review từ chối PR có catch (Exception) không có context.

L3 — Apply (Senior)

Design exception strategy cho service: domain exception layer thấp, wrap thành business exception layer cao, @ControllerAdvice map → HTTP. Biết exception boundary. Custom exception có error catalog cho client + alert rule.

Symptom: service có error catalog doc (mỗi error code có description + recovery action + alert level), code recovery test bao phủ.

Logging

L1 — Recognize

Biết SLF4J + log levels. Đôi khi log password / dùng System.out. Log nhiều INFO. Không biết MDC.

Symptom: log của 1 request không track được, debug ssh vào server đọc thủ công.

L2 — Explain

Phân biệt được 5 log levels theo case. Dùng parameterized log. Implement MDC filter. Biết PII never-log list. Hiểu structured logging giúp gì.

Symptom: code có %X{traceId} trong pattern, có MDC filter, log level đúng theo nature event.

L3 — Apply

Build observability stack: structured JSON log + traceId xuyên service + alert rule trên Kibana/Datadog. Debug production 3h sáng <15 phút dùng traceId. Capacity plan log storage. Recipe debug 5 scenario là phản xạ.

Symptom: production incident có timeline đầy đủ từ trace, post-mortem có "log/metric đã có đủ để diagnose trong X phút".

MDC + Distributed Tracing

L1 — Recognize

Biết tên MDC. Chưa setup. Debug bằng grep service name + timestamp.

L2 — Explain

Hiểu MDC = ThreadLocal context. Setup filter cho traceId, customerId. Biết bẫy MDC + thread pool. Pass traceId qua HTTP header giữa service.

L3 — Apply

Distributed tracing với OpenTelemetry / Sleuth. TraceId propagate qua Kafka, async task. Sampling strategy. Pair với Jaeger/Tempo để visualize span tree.

🚨 Symptom → Debug Recipes CORE ★

🚨 Recipe #1 — "Order create thành công nhưng customer không thấy"

Symptom: Customer report tạo order xong không thấy trong "My Orders". App log show "Order created: orderId=ORD_123" nhưng DB không có row.

Debug:

  1. Tìm log của request đó bằng traceId hoặc customerId+timestamp
  2. Đọc kỹ stack trace: có exception nào sau "Order created" không? (có thể bị catch silently ở layer sau)
  3. Check @Transactional: log "Order created" trong transaction → nếu sau đó throw exception thì rollback, DB không có data nhưng log đã ghi
  4. Tool: enable Hibernate SQL log org.hibernate.SQL=DEBUG + org.hibernate.type.descriptor.sql=TRACE để xem actual INSERT có chạy không

Root cause typical: log INFO inside transaction trước khi commit. Sau đó downstream call fail, transaction rollback. Fix: log INFO ở @TransactionalEventListener(AFTER_COMMIT), không phải giữa transaction.

🚨 Recipe #2 — "Error rate spike nhưng không log nào"

Symptom: Datadog/Grafana báo error rate 12% nhưng grep ELK không thấy ERROR log nào.

Checklist:

  1. Có catch (Exception) { /* empty */ } ở đâu không? Code review pull request gần đây
  2. Có e.printStackTrace() không? Container stderr → /dev/null
  3. Có exception throw từ @Async task không? Default Spring sẽ log warning vào stdout nhưng nếu disabled thì mất
  4. Có async event listener throw exception? Spring 5+ default swallow nếu không có error handler
  5. Error rate count từ đâu (5xx HTTP, hay custom metric)? Mismatch có thể là metric đếm sai
🚨 Recipe #3 — "Log mất traceId, không trace được request"

Symptom: Log của 1 request scattered, không group được bằng traceId vì có log [traceId=-].

3 nguyên nhân:

  1. MDC chưa setup ở filter — log trước khi filter chạy không có traceId. Check filter order: traceId filter phải chạy đầu tiên
  2. Async task không inherit MDC — fix bằng MDC.getCopyOfContextMap() trước submit
  3. Scheduled task không có traceId — task không phải request, không có context. Fix: generate traceId mới ở start task
🚨 Recipe #4 — "Log file phình 100GB/ngày"

Symptom: ELK index full, log retention chỉ giữ 1 ngày thay vì 30. Cost storage tăng 10×.

Debug:

  1. Top 10 loggers by volume: grep -c "logger_name" log.json | sort | uniq -c | sort -rn | head
  2. Loop logging? Tìm logger nào log > 1000 lines/giây — thường là batch job log INFO mỗi item
  3. Library noise? org.hibernate.SQL, org.springframework.web có thể spam DEBUG. Set WARN
  4. Stack trace duplicate? Nếu cùng exception log nhiều layer → fix anti-pattern #3
🚨 Recipe #5 — "Production crash không có log lúc crash"

Symptom: App crash lúc 3h sáng. Log file dừng lại lúc 02:59:58. Không có ERROR/FATAL trước khi crash.

3 case:

  1. OOM kill (OS hoặc K8s) — JVM không kịp flush log. Check dmesg / kubectl describe pod → "OOMKilled"
  2. JVM crash native — sẽ có hs_err_pidXXX.log trong working dir. Native memory leak, JIT bug
  3. kill -9 from parent — supervisor (systemd, Docker) bị health check fail → SIGKILL. App không có chance log

Pattern phòng ngừa: Async appender với queueSize=512 + flush on shutdown. Heap dump on OOM. Health check endpoint không phụ thuộc DB.

🎤 Interview Framework CORE ★

Tier 1 — Surface (Junior screening)

Q: Checked vs Unchecked exception?

Trả lời mẫu: "Checked extends Exception (trừ RuntimeException) — compiler force handle hoặc throws. Ví dụ IOException, SQLException. Unchecked extends RuntimeException — không bắt buộc handle, dùng cho bug logic hoặc business exception. Ví dụ NPE, IllegalArgument.

Modern Java + Spring xu hướng unchecked vì checked clutter signature, caller thường wrap thành RuntimeException dù sao. Spring wrap SQLException → DataAccessException unchecked."

Q: try-with-resources hoạt động thế nào?

Trả lời mẫu: "Java 7+ feature. Resource trong try(...) phải implement AutoCloseable. JVM tự gọi close() ở finally implicit, kể cả có exception. Multi-resource close theo thứ tự ngược lại.

Lợi: ngắn gọn hơn try-finally truyền thống, không quên close, không null check, không nested try-catch cho close()."

Tier 2 — Deep Dive (Mid level)

Q: Tại sao log parameterized thay vì string concat?

Trả lời mẫu: "2 lý do:

1. Performance: log.debug("User " + user) luôn build string, kể cả khi DEBUG OFF. log.debug("User {}", user) chỉ build khi DEBUG ON. Hot path log nhiều → khác biệt rõ ràng.

2. Consistency: SLF4J chuẩn hóa format, parser ELK dễ hiểu placeholder.

Edge case: nếu argument có toString() đắt (object lớn), kể cả parameterized cũng tốn — phải if (log.isDebugEnabled()) guard."

Q: MDC là gì? Tại sao quan trọng cho microservices?

Trả lời mẫu: "MDC là ThreadLocal map gắn key-value vào mọi log trong thread hiện tại. Trong microservices, traceId truyền qua header HTTP, mỗi service đặt traceId vào MDC → log của tất cả service trong 1 request có cùng traceId → distributed tracing.

Pitfall: thread pool / async — MDC không tự pass. Phải copy contextMap trước submit task. Spring Cloud Sleuth + Brave / OpenTelemetry tự động hóa, nhưng manual code phải nhớ."

Tier 3 — Architecture (Senior)

Q: Design exception handling strategy cho e-commerce microservices.

Trả lời mẫu: "Layer-by-layer:

Domain layer: throw business exception (InsufficientInventory, PaymentDeclined) — extends RuntimeException + có errorCode.

Repository layer: catch SQLException → wrap thành DatabaseException (Spring DataAccessException nếu JPA). Không expose JDBC chi tiết.

Service layer: business logic, throw domain exception. Không catch trừ khi handle có ý nghĩa (retry, fallback).

Controller layer: @ControllerAdvice convert exception → ResponseEntity. ErrorResponse có errorCode + message + details + traceId. KHÔNG leak stack trace.

Cross-cutting:

  • All log có traceId qua MDC
  • Sentry/Rollbar capture stack trace cho ERROR
  • Metric counter cho mỗi error type (Prometheus)
  • Alert trên error rate > threshold (PagerDuty)

Có error catalog doc cho team — mỗi error code có description + recovery action + alert level."
