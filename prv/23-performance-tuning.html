#23 — Performance Tuning

Performance tuning senior không phải thuộc vài JVM flag hay tăng connection pool. Đó là quy trình điều tra: đo đúng, tái hiện đúng, profile đúng, thay đổi nhỏ có giả thuyết, verify bằng số liệu và rollback được.

Performance là chủ đề dễ nói hay nhưng khó làm đúng. Học theo 5 luật:

  1. Không tối ưu khi chưa có metric baseline.
  2. Mỗi thay đổi phải có hypothesis: “Nếu bottleneck là X, đổi Y sẽ làm metric Z giảm”.
  3. Luôn nhìn p95/p99, không chỉ average.
  4. Phân biệt CPU-bound, IO-bound, lock-bound, pool-bound, GC-bound, downstream-bound.
  5. Verify bằng load test hoặc production canary, không kết luận từ local run đơn lẻ.

🗺️ Roadmap — Theo độ ưu tiên

Nếu chỉ có một ngày, tập trung method + profiling + connection pool. Đây là phần interview senior hỏi nhiều nhất.
1 Measure before optimize — CORE: không đo là đoán.
2 Bottleneck taxonomy — CORE: chọn đúng hướng điều tra.
3 Java profiling: JFR, async-profiler, thread dump — CORE: debug production.
4 Connection pool tuning — CORE: daily Spring backend.
5 Query/cache/load-test tuning — SUPP: mở rộng từ bottleneck.
6 Coordinated omission, queueing theory, flame graph interpretation — ADV: senior+ signal.

Tổng quan & Vì sao quan trọng CORE

Một service có thể chậm vì SQL scan, connection pool queue, lock contention, GC pause, CPU serialization, remote API timeout, logging quá nhiều, hoặc đơn giản là load balancer gửi traffic lệch. Nếu bạn nhảy ngay vào “tăng thread pool” hoặc “thêm cache”, bạn có thể làm hệ thống tệ hơn.

📜 Lịch sử ra đời của profiling & tuning
Unix/server era — Kỹ sư bắt đầu với `top`, `vmstat`, `iostat`, log timestamp để biết CPU, memory, disk, network. Đây là nền của performance diagnosis.

JVM era — Java thêm JMX, thread dump, heap dump, GC log, JIT compilation. Vấn đề performance không chỉ nằm trong code Java mà còn ở runtime behavior.

Modern profiling — JFR, async-profiler, eBPF, distributed tracing giúp nhìn flame graph, allocation, lock, wall-clock và cross-service latency với overhead thấp hơn.

Insight: performance tuning tiến hóa từ “đo máy” sang “đo toàn request path”. Backend senior phải nối được app metrics, runtime profile, DB plan và downstream latency.
🎯 Senior answer khác gì junior answer?

Junior thường đưa checklist: cache, index, tăng pool. Senior bắt đầu bằng metric, workload, SLO, bottleneck hypothesis, profiling evidence, smallest safe change, rollout plan và regression guard.

1. Bài Toán Thực Tế — MarketHub sale lớn, product page p99 = 9s CORE

MarketHub có product detail API. Bình thường p95 là 180ms. Ngày sale lớn, p95 tăng 2.5s, p99 9s. CPU app chỉ 45%, database CPU 85%, connection pool active luôn max, thread dump nhiều thread `WAITING` ở Hikari pool. Team đề xuất tăng pool từ 40 lên 200 và thêm Redis cache cho mọi thứ.

Naive product detail — nhiều IO ẩn trong một request
public ProductPage getProductPage(String productId, String userId) {
    Product product = productRepository.findById(productId).orElseThrow();
    List<Review> reviews = reviewRepository.findLatest(productId, 20);
    Seller seller = sellerRepository.findById(product.sellerId()).orElseThrow();
    Inventory inventory = inventoryClient.get(productId);
    List<Product> related = recommendationClient.related(productId, userId);
    return ProductPage.of(product, reviews, seller, inventory, related);
}
🔥 Tăng pool có thể làm tệ hơn

Nếu database đã CPU 85% và query đang scan nhiều row, tăng connection pool sẽ cho nhiều query chạy đồng thời hơn, làm DB cạnh tranh CPU/IO nặng hơn, latency tăng thêm. Pool lớn không tạo thêm capacity ở database; nó chỉ tăng concurrency đổ vào bottleneck.

⏸ Pause & Predict (BẮT BUỘC dừng)
Trước khi chọn fix, hãy viết 5 metric bạn cần xem.
  1. App latency p50/p95/p99 theo endpoint.
  2. Hikari active/idle/pending/acquire time.
  3. Top SQL by total time và execution plan.
  4. Thread state/flame graph wall-clock.
  5. Downstream inventory/recommendation latency.
Nếu bạn chưa biết bottleneck là DB, pool, lock, CPU hay downstream, mọi fix chỉ là đoán.

2. Method — Measure, Profile, Fix, Verify CORE

★ CORE — Bắt buộc nắm chắc: Interview performance thường đánh vào phương pháp, không chỉ kỹ thuật.

Quy trình tuning tốt giống điều tra incident nhỏ. Bạn định nghĩa symptom, scope, baseline, workload, bottleneck hypothesis, rồi mới thay đổi. Mỗi lần chỉ đổi một nhóm biến đủ nhỏ để biết tác động đến đâu.

Step 1

Measure

Xác định SLO, p95/p99, throughput, error rate, saturation. So sánh trước/sau theo cùng workload.

Step 2

Classify

CPU, DB, network, lock, pool, GC, downstream, serialization hay logging. Mỗi loại cần công cụ khác nhau.

Step 3

Change

Thay đổi nhỏ có giả thuyết. Ví dụ index query X, giảm N+1, chỉnh pool size, thêm timeout, cache read-heavy.

Step 4

Verify

Load test/canary, kiểm tra regression, cost, memory, correctness, rollback plan.

❌ Tối ưu theo cảm giác
// "Chậm chắc do thiếu cache"
@Cacheable("product")

    return buildPage(productId, userId);

✅ Hypothesis rõ metric
Hypothesis:
- p99 high because DB query review_latest scans 1.8M rows.
Evidence:
- pg_stat_statements: 42% total DB time.
- EXPLAIN: Seq Scan + Sort.

🧠 Misconception Check — “Average latency ổn là được”
"Average chỉ 120ms nên service ổn, p99 5s chắc vài request lẻ."
User trải nghiệm tail latency. Nếu mỗi page gọi 8 backend calls, xác suất ít nhất một call chậm tăng mạnh. Average che spike, queueing, GC pause và hot partition. Production tuning phải nhìn p95/p99, saturation và error rate.

3. Bottleneck Taxonomy — Chậm ở đâu? CORE

Bottleneck là tài nguyên hoặc dependency giới hạn throughput. Nếu không phân loại đúng, bạn dùng sai tool. CPU-bound cần flame graph CPU; IO-bound cần latency breakdown; lock-bound cần thread dump/lock profiling; pool-bound cần pending/acquire metrics; GC-bound cần GC log/JFR allocation.

Loại bottleneckDấu hiệuToolFix thường gặp
CPU-boundCPU cao, run queue caoasync-profiler CPU, JFRGiảm serialization, algorithm, caching computed result
DB-boundDB CPU/IO cao, slow queryEXPLAIN, pg_stat_statementsIndex, query rewrite, batch, pagination
Pool-boundConnection pending cao, active=maxHikari metricsGiảm query time, size pool theo capacity, backpressure
Lock-boundThreads BLOCKED/WAITINGThread dump, JFR locksGiảm synchronized scope, concurrent data structure
GC-boundPause spike, allocation rate caoGC log, JFR allocationGiảm allocation, tune heap/GC, object reuse có kiểm soát
Downstream-boundExternal latency caoTracing, client metricsTimeout, circuit breaker, cache, async
⏸ Pause & Predict — Chẩn đoán từ metric
Nếu CPU app 30%, Hikari pending cao, DB CPU 95%, thread dump nhiều thread chờ connection, bottleneck là gì?
  1. App CPU-bound?
  2. Connection pool quá nhỏ?
  3. Database/query-bound làm connection bị giữ lâu?
Đừng vội tăng pool. Pool pending là triệu chứng; nguyên nhân có thể là query giữ connection quá lâu.

4. Profiling Java — JFR, async-profiler, thread dump CORE

Profiling trả lời “thời gian đi đâu?”. CPU profiler nhìn thread đang chạy trên CPU. Wall-clock profiler nhìn cả thời gian chờ IO/lock/sleep. Allocation profiler nhìn object nào tạo nhiều memory. Thread dump là snapshot, không phải profile dài hạn, nhưng rất hữu ích khi hệ thống đang treo.

JFR trong production canary
jcmd <pid> JFR.start name=prod-profile settings=profile duration=120s filename=/tmp/profile.jfr
jcmd <pid> JFR.check
jcmd <pid> JFR.stop name=prod-profile
async-profiler CPU và wall-clock
./profiler.sh -e cpu -d 60 -f /tmp/cpu.html <pid>
./profiler.sh -e wall -d 60 -f /tmp/wall.html <pid>
./profiler.sh -e alloc -d 60 -f /tmp/alloc.html <pid>
❌ Kết luận từ một thread dump
Thread dump shows many threads in HikariPool.getConnection().
Conclusion: increase pool size to 200.
✅ Kết hợp thread dump + metrics + DB evidence
Evidence:
- Hikari pending p95 = 120ms, active=max.
- Top query p95 = 900ms.
- DB CPU 92%, slow query scans reviews.
Conclusion:
- Fix query first; pool size is symptom amplifier.
🔬 Thử Phá Vỡ — Flame graph đọc sai

Câu hỏi: tìm 3 cách đọc flame graph sai dẫn đến fix sai.

Click để xem analysis (sau khi đã nghĩ)
  1. CPU profile cho IO problem: nếu service chờ DB, CPU flame graph có thể không thấy gì đáng kể. Cần wall-clock profile/tracing.
  2. Local workload khác production: local dataset nhỏ không kích hoạt query plan/GC giống production. Cần workload đại diện.
  3. Nhìn method rộng mà bỏ context: JSON serialization rộng có thể là do API trả payload quá lớn, không phải Jackson “chậm”. Fix có thể là pagination/field selection.

Insight: profiler là bằng chứng, không phải đáp án tự động. Phải nối profile với workload và metric.

⚠️ Profiling data có thể nhạy cảm

JFR/thread dump/log có thể chứa URL, query parameter, user id, token fragment hoặc SQL literal. Khi chia sẻ profile, sanitize dữ liệu nhạy cảm và tuân thủ policy nội bộ.

5. DB & Query Tuning — Tối ưu nơi giữ connection lâu nhất CORE

Backend Java thường chậm không vì Java chậm, mà vì request giữ DB connection quá lâu. Slow query làm pool active lâu, pending tăng, thread chờ, p99 phình. Query tuning không chỉ là thêm index; còn là giảm N+1, projection đúng field, pagination, batch write, transaction scope ngắn và plan ổn định.

❌ N+1 query khi build product page
List<Review> reviews = reviewRepository.findLatest(productId, 20);
for (Review review : reviews) {
    User user = userRepository.findById(review.userId()).orElseThrow();
    review.attachUser(user);
}
✅ Batch fetch/projection đúng nhu cầu
List<ReviewView> reviews =
    reviewRepository.findLatestReviewViews(productId, 20);
❌ Pagination bằng offset sâu
SELECT id, title, created_at
FROM product
WHERE category_id = :categoryId
ORDER BY created_at DESC
OFFSET 500000 LIMIT 50;
✅ Keyset pagination theo cursor
SELECT id, title, created_at
FROM product
WHERE category_id = :categoryId
  AND created_at < :lastSeenCreatedAt
ORDER BY created_at DESC
LIMIT 50;
EXPLAIN trước/sau index
EXPLAIN (ANALYZE, BUFFERS)
SELECT id, rating, comment, created_at
FROM review
WHERE product_id = 'p-123'
ORDER BY created_at DESC
LIMIT 20;

CREATE INDEX idx_review_product_created
ON review (product_id, created_at DESC);
💡 Query tuning checklist ngắn

Đọc plan thật với dữ liệu thật; nhìn rows estimated vs actual; tránh function trên indexed column nếu làm mất index usage; chỉ select field cần dùng; dùng keyset pagination cho feed sâu; giữ transaction ngắn; đo query p95 sau khi thêm index vì write cost có thể tăng.

6. Connection Pool Tuning — Little’s Law trong đời thật CORE

Connection pool không phải càng lớn càng tốt. Mỗi connection là khả năng chạy thêm work trên database. Nếu DB không đủ CPU/IO, thêm connection làm tăng contention. Pool size hợp lý dựa trên throughput mục tiêu, latency DB, số instance và capacity DB.

🎯 Little’s Law đơn giản hóa

Concurrency ≈ Throughput × Latency. Nếu mỗi instance cần 200 request/s và mỗi request giữ DB connection trung bình 50ms, concurrency DB trung bình khoảng 10. Bạn cần headroom, nhưng không cần pool 200 cho instance đó nếu DB không chịu nổi.

❌ Pool quá lớn để che query chậm
spring:
  datasource:
    hikari:
      maximum-pool-size: 200
      connection-timeout: 60000
✅ Pool có giới hạn + fail fast
spring:
  datasource:
    hikari:
      maximum-pool-size: 32
      minimum-idle: 8
      connection-timeout: 500
      validation-timeout: 250
      leak-detection-threshold: 2000

@Transactional

    product.markViewed();

public ProductPage build(String productId) {

7. Cache, Load Test & Verification SUPPLEMENTARY

Cache là công cụ mạnh nhưng dễ làm sai dữ liệu. Cần biết data có đọc nhiều hơn ghi không, stale bao lâu chấp nhận được, invalidation bằng gì, key cardinality thế nào, cache stampede xử lý ra sao. Load test cũng phải đại diện production: warm-up, realistic data distribution, think time, p95/p99, error rate, resource saturation.

❌ Cache không TTL/invalidation rõ
@Cacheable("productPage")
public ProductPage getProductPage(String productId, String userId) {
    return buildProductPage(productId, userId);
}
✅ Cache snapshot public, tránh user-specific leak
@Cacheable(cacheNames = "publicProductSnapshot", key = "#productId")
public ProductSnapshot getPublicSnapshot(String productId) {
    return productReader.findPublicSnapshot(productId);
}
🎯 Edge Cases — Production Reality

Performance fix phải kiểm tra cả correctness và vận hành:

  • Race condition: cache invalidation và DB update chạy ngược thứ tự; fix bằng versioned cache key hoặc event ordering.
  • Network failure: DB/network spike làm pool pending tăng; fix bằng timeout, backpressure và circuit breaker cho downstream.
  • Scale limit: index giúp read nhưng làm write sale traffic chậm; fix bằng đo write amplification và chọn index cần thiết.
  • Security: cache user-specific response bằng key thiếu `userId` làm lộ dữ liệu; fix tách public/private cache.
  • Data integrity: stale inventory khiến user thấy còn hàng sai; fix TTL ngắn hoặc source-of-truth check ở command path.
  • Time/timezone: load test dùng fixed time không kích hoạt daily partition/report query; fix dataset theo thời gian thật và UTC.
  • Concurrent access: cache stampede khi key hot expire; fix single-flight, soft TTL, refresh-ahead.
  • Resource exhaustion: pool queue dài che overload; fix fail fast, bulkhead và admission control.
  • Coordinated omission: load generator chờ response rồi mới gửi tiếp làm p99 giả thấp; fix dùng constant arrival rate tool.
⚠️ Load test không phải chỉ bắn RPS

Cần warm-up JVM/JIT/cache, dữ liệu đủ lớn, phân bố key giống production, dependency fake có latency thật, theo dõi cả app và DB. Một test local với 1,000 sản phẩm không chứng minh API sẽ ổn với 50 triệu sản phẩm.

8. Production Runbook — Điều tra p99 chậm từ đầu đến cuối ADVANCED

Interview senior thường không hỏi “JFR là gì?” đơn lẻ. Họ đưa một case mơ hồ: “API chậm sau deploy” hoặc “p99 tăng trong sale”. Bạn cần dẫn dắt investigation như một incident: khoanh vùng, đo, giả thuyết, kiểm chứng, fix nhỏ, verify và rollback.

CORE 15 phút đầu tiên

  1. Scope: endpoint nào, region nào, version nào, tenant/category nào, bắt đầu lúc nào.
  2. User impact: p95/p99, error rate, timeout, conversion/drop-off nếu có.
  3. Change correlation: deploy, config, schema migration, index, traffic campaign, cache flush.
  4. Saturation: CPU, memory, GC, thread, Hikari, DB CPU/IO, downstream latency.
  5. Quick mitigation: rollback, reduce traffic, disable optional feature, increase timeout only nếu hiểu impact, fallback degraded.
⚠️ Đừng tối ưu trong incident nếu có rollback rõ

Nếu p99 tăng ngay sau deploy và rollback rẻ, rollback trước có thể đúng hơn ngồi profile 30 phút trong khi user đang chịu lỗi. Sau khi ổn định mới capture evidence và fix root cause.

ADVANCED Little’s Law applied: pool sizing walkthrough

Giả sử một instance product-service phục vụ 300 RPS. Mỗi request gọi DB trung bình 2 lần, mỗi lần giữ connection 25ms sau khi query đã tối ưu. Tổng DB connection-seconds mỗi giây là `300 × 2 × 0.025 = 15`. Nghĩa là concurrency trung bình cần khoảng 15 connections. Bạn thêm headroom cho burst và tail, ví dụ pool 24-32. Nếu query giữ connection 250ms, concurrency thành 150; lúc đó pool 150 không phải fix, vì DB sẽ bị đập bởi query quá lâu.

Pool sizing thought process
Target per instance:
- HTTP throughput: 300 RPS
- DB calls/request: 2
- DB usage time/call p50: 25ms, p95: 80ms
Average concurrent DB usage:
  300 * 2 * 0.025 = 15
Practical pool:
  24-32, verify acquire p95 and DB CPU
If usage time jumps to 250ms:
  300 * 2 * 0.250 = 150
Conclusion:
  fix query/transaction first, not pool.

CORE GC/allocation runbook

Không phải GC pause nào cũng cần đổi GC algorithm. Nhiều case bắt đầu từ allocation rate quá cao: tạo DTO lớn, parse JSON nhiều lần, collect list khổng lồ, log string interpolation, load toàn bộ result set. Hãy xem allocation profile trước khi tune flag.

  1. Bật GC log/JFR trong khoảng có spike.
  2. Xem pause time, allocation rate, old-gen occupancy, promotion rate.
  3. Capture allocation flame graph nếu allocation rate cao.
  4. Fix code tạo object nhiều hoặc payload quá lớn trước khi đổi heap/GC.
  5. Verify p95/p99 và memory sau fix; tránh làm tăng latency vì pooling object thủ công quá tay.
❌ Load toàn bộ rồi mới cắt trang
List<Product> all = productRepository.findByCategory(categoryId);
return all.stream()
    .sorted(comparing(Product::createdAt).reversed())
    .skip(offset)
    .limit(50)
    .toList();
✅ Pagination ở database
return productRepository.findPageByCategory(
    categoryId,
    cursor,
    PageRequest.ofSize(50)
);

8. Risks:
   write cost +8%, monitor index bloat and review insert latency.
✅ Câu trả lời senior khi bị hỏi performance tuning

“Em bắt đầu bằng SLO và baseline, phân loại bottleneck bằng metrics/tracing/profile. Em không tăng pool hay thêm cache trước khi biết thời gian đi đâu. Mỗi fix có hypothesis và metric expected. Sau fix em chạy cùng workload, so p95/p99/error/saturation, kiểm tra correctness và rollback.”

Interview Drill Bank — Performance tình huống CORE

Performance interview đánh vào tư duy điều tra. Trả lời tốt cần metric, bottleneck hypothesis và verification. Tránh nhảy ngay vào “cache/index/tăng pool”.

CORE Q1. API chậm, bạn làm gì đầu tiên?

Answer mẫu: Xác định scope và baseline: endpoint nào, p50/p95/p99, throughput, error rate, bắt đầu lúc nào, deploy nào liên quan. Sau đó xem saturation: CPU, memory, GC, thread, DB, Hikari, downstream. Dùng tracing/profile để phân loại bottleneck rồi mới fix. Nếu impact lớn và rollback rẻ, rollback trước rồi điều tra sau.

Signal: interviewer muốn nghe method, không phải danh sách optimization.

CORE Q2. Hikari pending cao, có nên tăng pool?

Answer mẫu: Chưa chắc. Pending cao nghĩa là thread chờ connection, nhưng nguyên nhân có thể là query chậm, transaction giữ connection lâu, DB saturated, hoặc pool quá nhỏ thật. Xem usage time, acquire time, DB CPU/IO, top SQL, thread dump. Nếu DB đã CPU cao, tăng pool làm cạnh tranh nặng hơn. Fix query/transaction trước rồi size pool theo throughput × DB latency.

Signal: phải nói pool pending là symptom, không phải root cause mặc định.

CORE Q3. Average latency thấp nhưng p99 cao, vì sao nguy hiểm?

Answer mẫu: User thấy tail latency. Nếu page gọi nhiều backend calls, xác suất một call dính p99 tăng. Average che queueing, GC pause, lock contention, hot keys và downstream spike. SLO thường dùng percentile vì nó phản ánh trải nghiệm phần lớn user và tail. Cần nhìn p95/p99 cùng throughput/error/saturation.

Signal: phải nối percentile với user experience và fan-out.

SUPPLEMENTARY Q4. Khi nào dùng CPU profiler, khi nào dùng wall-clock profiler?

Answer mẫu: CPU profiler dùng khi CPU cao và muốn biết method nào tiêu CPU. Wall-clock profiler dùng khi latency cao nhưng CPU không cao, để thấy thời gian chờ IO, lock, sleep, connection pool. Allocation profiler dùng khi GC/allocation rate cao. Nếu service chờ DB, CPU flame graph có thể đánh lừa vì nó chỉ thấy phần đang chạy CPU.

Signal: biết chọn tool theo bottleneck.

CORE Q5. Query chậm làm connection pool nghẽn như thế nào?

Answer mẫu: Mỗi query giữ một DB connection đến khi hoàn tất. Nếu query p95 tăng từ 30ms lên 900ms, cùng RPS sẽ cần concurrency DB lớn hơn rất nhiều. Pool active=max, request mới pending, thread chờ, p99 tăng. Tăng pool chỉ cho thêm query chậm chạy đồng thời; nếu DB đã saturated, toàn hệ thống chậm hơn.

Signal: phải hiểu relationship query duration → connection usage → pending.

ADVANCED Q6. Coordinated omission là gì?

Answer mẫu: Là khi load generator chờ response xong mới gửi request tiếp, nên lúc server chậm nó cũng gửi ít request hơn và bỏ lỡ latency mà user thật sẽ gặp. Kết quả p99 bị giả thấp. Dùng constant arrival rate hoặc tool hỗ trợ correction để giữ arrival pattern độc lập với response time. Đây là lỗi load test phổ biến.

Signal: senior+ signal, nhất là với performance lab.

SUPPLEMENTARY Q7. Cache product page có rủi ro gì?

Answer mẫu: Nếu cache cả response user-specific bằng key chỉ có `productId`, có thể lộ dữ liệu hoặc trả giá/khuyến nghị sai. Cache stale inventory làm user thấy còn hàng sai. Cache stampede xảy ra khi key hot expire. Cần tách public snapshot và user-specific data, TTL/invalidation rõ, single-flight/refresh-ahead cho key hot, và command path vẫn check source-of-truth.

Signal: performance fix phải giữ correctness/security.

CORE Q8. Bạn đọc EXPLAIN như thế nào ở mức cơ bản?

Answer mẫu: Xem query dùng seq scan hay index, rows estimated vs actual có lệch không, sort/hash có spill không, filter loại bao nhiêu row, join order, buffers read/hit, execution time. Dùng `EXPLAIN ANALYZE` trên môi trường an toàn với dữ liệu đại diện. Không thêm index chỉ vì thấy seq scan; nếu table nhỏ hoặc selectivity thấp, seq scan có thể hợp lý.

Signal: biết plan là evidence, không phải ritual.

ADVANCED Q9. GC pause tăng sau deploy, bạn xử lý thế nào?

Answer mẫu: Xem GC log/JFR: pause time, allocation rate, heap occupancy, promotion, old-gen growth. Capture allocation profile để biết object nào tăng. So sánh payload size, batch size, DTO mapping, logging, result set. Fix allocation/code trước khi đổi GC flag. Nếu heap thật sự thiếu, tune heap/GC sau khi có evidence và verify p95/p99.

Signal: tránh “đổi G1/ZGC cho nhanh” khi chưa biết allocation.

CORE Q10. Một performance fix tốt cần report gì?

Answer mẫu: Goal/SLO, environment, dataset, workload, baseline, evidence, change, result, side effects và rollback. Ví dụ: p95 product detail 2.4s do review query seq scan; thêm index `(product_id, created_at desc)` và projection; p95 giảm 310ms, DB CPU giảm, write latency tăng 8% cần monitor. Không có baseline/result thì không chứng minh được fix.

Signal: biết chứng minh hiệu quả và trade-off.

Performance Cookbook — Symptom → Evidence → Fix SUPPLEMENTARY

Cookbook này giúp bạn phản xạ đúng trong interview. Đừng nhảy từ symptom sang fix; luôn chèn bước evidence.

CORE Symptom 1 — p99 tăng, CPU app thấp

Nghi vấn: IO/downstream/DB/pool/lock chứ không phải CPU-bound.

Evidence: tracing latency breakdown, Hikari pending/acquire, HTTP client latency, thread dump, wall-clock profile.

Fix: tùy evidence: timeout/fallback cho downstream, query tuning, giảm transaction scope, bulkhead, hoặc fix lock contention.

CORE Symptom 2 — DB CPU cao, pool active=max

Nghi vấn: slow query hoặc concurrency quá cao vào DB.

Evidence: top SQL by total time, EXPLAIN ANALYZE, buffer reads, rows actual vs estimated, Hikari usage time.

Fix: index/query rewrite/projection/keyset pagination/batch, sau đó mới tune pool. Nếu DB CPU đã cao, tăng pool thường làm tệ hơn.

SUPPLEMENTARY Symptom 3 — CPU app cao sau deploy

Nghi vấn: serialization payload lớn, algorithm thay đổi, compression, regex, mapping DTO, logging sync.

Evidence: async-profiler CPU flame graph, endpoint payload size, allocation profile nếu GC cũng tăng.

Fix: giảm payload, pagination/field selection, optimize hot method, tránh serialize object graph dư, cache computed result nếu data semantics cho phép.

SUPPLEMENTARY Symptom 4 — GC pause spike

Nghi vấn: allocation rate tăng, result set quá lớn, object churn, heap pressure, memory leak.

Evidence: JFR allocation, GC log, heap occupancy, old-gen trend, heap dump nếu nghi leak.

Fix: stream/paginate data, giảm DTO/list tạm, tránh load toàn bộ, tune heap/GC sau khi code fix rõ.

CORE Symptom 5 — Latency tốt ở load test nhưng production xấu

Nghi vấn: workload test không đại diện, dataset nhỏ, cache warm khác, key distribution khác, coordinated omission.

Evidence: so sánh production traffic distribution, hot keys, dataset size, arrival rate model, cache hit rate.

Fix: dùng constant arrival rate, Zipfian/hot-key distribution, warm-up đúng, dataset gần production, inject downstream latency thật.

ADVANCED Symptom 6 — Một category/product hot làm toàn service chậm

Nghi vấn: hot key, cache stampede, lock per key, DB row/page contention.

Evidence: latency grouped by product/category, cache miss burst, DB lock wait, request distribution.

Fix: request coalescing/single-flight, soft TTL, refresh-ahead, shard hot work, precompute read model, rate limit extreme hot key nếu cần.

SUPPLEMENTARY Symptom 7 — Thread count tăng liên tục

Nghi vấn: thread leak, unbounded executor, blocking call trong reactive pipeline, scheduler tạo thread mới.

Evidence: thread dump nhiều thời điểm, executor metrics, code path tạo executor/thread, JFR thread events.

Fix: bounded executor, reuse bean-managed pools, remove per-request executor creation, timeout blocking calls.

ADVANCED Symptom 8 — p99 chỉ xấu sau 30 phút chạy

Nghi vấn: memory leak, cache growth, connection leak, JIT/warmup không phải nguyên nhân chính, DB bloat/temp files.

Evidence: time-series heap, pool active/idle, cache size/eviction, GC old-gen trend, DB temp usage.

Fix: leak detection, bounded cache, close resources, pagination/streaming, cleanup temp data, long-duration soak test.

📖 Câu nhớ nhanh

Performance tuning tốt là chuỗi: symptom → evidence → bottleneck class → minimal fix → verification. Bỏ qua evidence là đoán.

Self-Test Scenarios — Chẩn đoán performance bằng evidence CORE

Tự viết hypothesis và metric cần xem trước. Nếu bạn nhảy thẳng tới fix, hãy quay lại bước evidence.

CORE Scenario 1 — Pool pending cao sau sale campaign

Metrics: Hikari active=max, pending p95 200ms, DB CPU 90%, top query `review_latest` chiếm 45% total time.

Câu hỏi: tăng pool có phải fix đầu tiên không?

Gợi ý: không. Fix query/index/projection trước. Pool pending là hậu quả connection bị giữ lâu. Sau khi query giảm, đo lại acquire time rồi mới tune pool.

SUPPLEMENTARY Scenario 2 — CPU high vì serialization

Metrics: CPU 85%, DB bình thường, async-profiler top frame nằm ở JSON serialization, response product page 2MB.

Gợi ý: không đổ lỗi “Jackson chậm” ngay. Kiểm tra payload dư, object graph quá lớn, reviews/recommendations quá nhiều, thiếu field selection. Fix có thể là projection DTO, pagination, bỏ field không dùng, gzip nếu network-bound.

ADVANCED Scenario 3 — p99 xấu nhưng profiler CPU sạch

Metrics: CPU 35%, p99 5s, wall-clock profile thấy nhiều thread chờ `CompletableFuture.get`, tracing chỉ ra recommendation service p99 4s.

Gợi ý: bottleneck downstream. Thêm timeout/fallback/bulkhead, cache recommendation nếu semantics cho phép, và không block thread vô hạn. CPU profiler không phải tool đúng cho IO wait.

CORE Scenario 4 — Cache hit cao nhưng data sai

Bối cảnh: cache hit 95%, latency giảm, nhưng user thấy inventory sai trong sale.

Gợi ý: performance fix phá correctness. Inventory là data nhạy với thời gian; cache TTL/invalidation phải phù hợp, command path cần check source-of-truth. Có thể cache public product info nhưng không cache availability quyết định mua quá lâu.

SUPPLEMENTARY Scenario 5 — Load test pass, production fail

Bối cảnh: k6 test 500 RPS local/staging pass, production sale fail ở 300 RPS.

Gợi ý: kiểm tra dataset size, hot key distribution, cache warm state, downstream latency, DB hardware, arrival model, coordinated omission, network. Staging 1M rows không đại diện production 100M rows. Load test phải mô phỏng shape, không chỉ RPS.

CORE Scenario 6 — Transaction giữ connection quá lâu

Bối cảnh: thread dump cho thấy nhiều request chờ Hikari. Code mở `@Transactional`, đọc product, gọi inventory service 800ms, rồi update view count.

Gợi ý: transaction đang giữ DB connection qua remote call. Tách read snapshot ra transaction ngắn, gọi inventory ngoài transaction, view count chuyển event/async hoặc update riêng. Đây là performance và reliability issue: remote latency ăn connection pool.

SUPPLEMENTARY Scenario 7 — Index mới làm write chậm

Bối cảnh: thêm 4 index để tăng tốc search, read p95 giảm nhưng product import chậm gấp đôi.

Gợi ý: index có write amplification. Performance fix phải đo cả read và write path. Giữ index phục vụ query critical, bỏ index trùng/ít dùng, cân nhắc partial index/materialized view/search engine nếu query pattern phức tạp. Report phải nêu trade-off.

ADVANCED Scenario 8 — Lock contention trong in-memory cache

Bối cảnh: CPU không quá cao nhưng thread dump nhiều thread BLOCKED ở synchronized cache loader. p99 spike khi key hot expire.

Gợi ý: đây là lock/cache stampede. Dùng single-flight per key thay vì global lock, soft TTL, refresh-ahead, hoặc cache library có loading semantics tốt. Profile lock/JFR monitor events để chứng minh trước khi refactor.

CORE Scenario 9 — Optimization làm mất tính đúng

Bối cảnh: team cache product price 10 phút để giảm DB load. Latency giảm mạnh, nhưng trong flash sale giá thay đổi theo phút và user thấy giá cũ.

Gợi ý: performance fix phải giữ business correctness. Tách cache public metadata ít thay đổi khỏi price/availability nhạy thời gian. Dùng TTL ngắn, event invalidation, versioned price, hoặc kiểm tra source-of-truth ở bước tạo đơn. Không chấp nhận p95 đẹp nếu dữ liệu sai.

SUPPLEMENTARY Scenario 10 — Logging làm p99 xấu

Bối cảnh: sau khi bật debug log payload lớn trong sale, CPU và disk IO tăng, p99 xấu nhưng DB bình thường.

Gợi ý: logging cũng là workload. Dùng structured log có sampling, không log payload lớn/PII, tránh sync appender trên hot path, đo log volume và IO wait. Performance tuning phải xem cả observability overhead.

Calibration: một thay đổi “chỉ thêm log” vẫn phải qua performance review nếu nằm trên hot path hoặc payload lớn. Observability tốt phải giúp debug mà không biến chính nó thành bottleneck CPU, disk IO, network hoặc chi phí lưu trữ. Log, metric và trace đều có ngân sách hiệu năng riêng. Khi tuning, hãy đo cả overhead của công cụ đo nếu nó nằm trên hot path, nhất là trong endpoint có traffic lớn và payload biến thiên mạnh theo user trong giờ cao điểm hoặc sale lớn toàn sàn kéo dài.

Interview Framework CORE

Tầng 1 — Surface Method gate
Q: API chậm, bạn làm gì đầu tiên?
Không tối ưu ngay. Đầu tiên xác định endpoint, SLO, p50/p95/p99, throughput, error rate, saturation. Sau đó phân loại bottleneck bằng metrics/tracing/profile: DB, CPU, pool, lock, GC, downstream. Có hypothesis rồi mới thay đổi và verify.
Tầng 2 — Deep Dive Pool tuning
Q: Hikari active=max và pending cao. Có nên tăng pool?
Chưa đủ dữ liệu. Pending cao có thể do pool nhỏ, nhưng cũng có thể do query chậm hoặc transaction giữ connection lâu. Xem DB CPU/IO, query time, usage time, acquire time, thread dump. Nếu DB đã saturated, tăng pool có thể làm tệ hơn. Fix query/transaction trước rồi size pool theo throughput × latency và DB capacity.
Tầng 3 — Architecture End-to-end investigation
Q: Product page p99 9s trong sale event. Thiết kế investigation plan.
Tách latency breakdown bằng tracing; xem Hikari metrics, top SQL, DB CPU/IO, downstream latency, JVM GC/thread dump. Reproduce bằng load test có key distribution sale. Nếu query scan review là bottleneck, thêm index/projection/keyset. Nếu recommendation downstream chậm, timeout/fallback. Canary rollout, compare p95/p99/error/resource, giữ rollback.
