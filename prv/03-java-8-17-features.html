#03 - Java 8-17 Features: Stream, Lambda, Optional, Record, Sealed, Pattern Matching

Modern Java không phải biến mọi thứ thành stream. Senior biết khi nào Stream làm code rõ hơn, khi nào for-loop tốt hơn; biết Optional là API contract chứ không phải thuốc trị null; biết record/sealed class giúp model dữ liệu và state machine chặt hơn.

Java 8-17 Là Chuyển Dịch Từ Imperative Sang Expressive

Java 8 đưa lambda, stream, Optional và default methods vào daily code. Java 14-17 bổ sung record, sealed class, switch expression và pattern matching. Những feature này giúp code ngắn hơn, nhưng nếu dùng máy móc có thể làm code khó debug, chậm hơn, hoặc giấu side effect.

Cách học đúng: mỗi feature phải trả lời ba câu. Nó giải quyết vấn đề gì? Nó có trade-off gì? Trong code backend production, dùng nó ở boundary nào là hợp lý?

Mục tiêu không phải "viết ít dòng hơn"

Modern Java tốt khi nó làm intent rõ hơn và giảm state ngoài ý muốn. Nếu stream pipeline dài 20 dòng, có side effect, bắt checked exception vòng vèo và khó debug, nó không modern hơn for-loop. Senior biết quay lại for-loop khi đó là lựa chọn rõ hơn.

1

Intent

Stream/Optional/record phải làm ý định code rõ hơn, không chỉ ngắn hơn.

2

Contracts

Optional, sealed class và record làm API contract rõ hơn ở compile time.

3

Cost model

Stream có allocation/boxing/debug cost; parallelStream có scheduling cost.

4

Boundaries

DTO/event/value object rất hợp record; entity/proxy/lifecycle object thì không.

1. Code Review Scenario - Báo Cáo Billing Viết Kiểu Java Cũ

Bạn review PR tạo báo cáo invoice cho một SaaS. Code chạy được nhưng nhiều null check, string status rải rác, mutable list external và thiếu xử lý empty case.

Trước: nested imperative
public List<String> paidInvoiceIds(List<Invoice> invoices) {
    List<String> result = new ArrayList<>();
    for (Invoice invoice : invoices) {
        if (invoice != null) {
            if (invoice.getAmount() != null) {
                if (invoice.getAmount().compareTo(BigDecimal.ZERO) > 0) {
                    if ("PAID".equals(invoice.getStatus())) {
                        result.add(invoice.getId());
                    }
                }
            }
        }
    }
    return result;
}
Sau: pipeline thể hiện intent
public List<String> paidInvoiceIds(List<Invoice> invoices) {
    return invoices.stream()
        .filter(Objects::nonNull)
        .filter(Invoice::hasPositiveAmount)
        .filter(invoice -> invoice.status() == InvoiceStatus.PAID)
        .map(Invoice::id)
        .toList();
}
Không phải ngắn hơn là tốt hơn

Stream tốt khi pipeline là transform/filter/reduce rõ ràng. Nếu logic cần nhiều nhánh, debug từng bước, break phức tạp, mutate state có kiểm soát, for-loop vẫn là lựa chọn tốt.

🗺️ Nếu chỉ có X giờ thì học gì
1 giờ (CORE ★): Stream lazy + terminal operation (2.1), Collectors groupingBy/toMap (2.4), Optional.map/flatMap đúng cách (3.1-3.2). Đây là phần code hằng ngày + hỏi nhiều nhất.
+1.5 giờ (CORE ★): Record cho DTO/value object (5.1), sealed + pattern matching cho state hữu hạn (5.2-5.3). Java 17 interview gần như chắc chắn hỏi.
+1.5 giờ (SUPPLEMENTARY): Primitive streams tránh boxing (2.3), lambda capture/effectively final (4.2), default methods (4.3), HashMap/ConcurrentHashMap internals (6).
Khi cần (ADVANCED): JPMS modules (5.4), parallelStream cost model, StampedLock-style optimization.
⏸ Pause & Predict
Nhìn pipeline paidInvoiceIds ở trên. Trả lời ra giấy:
  1. Nếu list có 1 triệu invoice mà chỉ 3 cái PAID, pipeline duyệt list mấy lần? (lazy hay eager?)
  2. Đổi .map(Invoice::id) thành .peek(System.out::println) mà KHÔNG có terminal operation thì in ra gì? Vì sao?
  3. Bản imperative bắt được null phần tử bằng invoice != null. Bản stream .filter(Objects::nonNull) đặt SAI vị trí (sau map) thì lỗi gì xảy ra?
Gợi ý: stream chỉ duyệt 1 lần và lazy — không có terminal op thì không chạy gì. Đáp án ở 2.1.

2. Stream API

2.1 Stream Là Pipeline Lazy

Stream không lưu dữ liệu như collection. Nó mô tả pipeline xử lý. Intermediate operations như filter, map, sorted lazy; terminal operation như collect, toList, count, findFirst mới kích hoạt chạy.

Lazy evaluation là lý do limit, findFirst, anyMatch có thể short-circuit.

Pipeline chạy theo từng element, không phải từng stage toàn bộ

Với pipeline stateless, stream thường xử lý một element đi qua nhiều operation rồi mới tới element tiếp theo. Điều này giúp short-circuit hiệu quả.

Optional<Invoice> first = invoices.stream()
    .filter(Invoice::isPaid)
    .filter(invoice -> invoice.total().compareTo(LIMIT) > 0)
    .findFirst();

Khi findFirst tìm được element đầu tiên thỏa điều kiện, stream dừng. Nó không cần filter toàn bộ list trước.

Spliterator: lý do parallel stream có thể split data

Bên dưới stream source có Spliterator, chịu trách nhiệm duyệt và có thể chia dữ liệu thành phần nhỏ cho parallel processing. Collection như ArrayList split tốt vì có index; LinkedList split kém hơn; I/O stream hoặc iterator không biết size thường parallel kém.

SourceParallel friendlinessLý do
ArrayListTốtRandom access, biết size, split dễ.
HashSetTrung bìnhSplit được nhưng order không ổn định.
LinkedListKém hơnKhông random access, split cost cao.
BufferedReader.lines()Thường không tốtI/O source, không muốn parallel common pool.

2.2 Stateless vs Stateful Operation

LoạiOperationĐặc điểmRủi ro
Statelessmap, filter, flatMapMỗi element xử lý độc lậpDễ parallel hơn nếu không side effect
Statefulsorted, distinct, limit, skipCần nhớ nhiều elementCó buffer, tốn memory, parallel không chắc nhanh

2.3 Primitive Streams: Tránh Boxing Không Cần Thiết

Stream<Integer>, Stream<Long>, Stream<Double> có boxing/unboxing overhead. Với aggregation số lượng lớn, dùng IntStream, LongStream, DoubleStream.

Tốn boxing
int total = orders.stream()
    .map(Order::quantity)      // Stream<Integer>
    .reduce(0, Integer::sum);
Primitive stream
int total = orders.stream()
    .mapToInt(Order::quantity)
    .sum();

Với money, vẫn dùng BigDecimal; primitive stream không thay thế decimal arithmetic. Nhưng với count, duration, quantity, primitive stream làm code vừa rõ vừa ít overhead.

2.4 Reduce, Collect, Grouping

BigDecimal total = invoices.stream()
    .map(Invoice::amount)
    .filter(Objects::nonNull)
    .reduce(BigDecimal.ZERO, BigDecimal::add);

Map<InvoiceStatus, BigDecimal> totalByStatus = invoices.stream()
    .collect(Collectors.groupingBy(
        Invoice::status,
        Collectors.mapping(
            Invoice::amount,
            Collectors.reducing(BigDecimal.ZERO, BigDecimal::add)
        )
    ));

reduce hợp với aggregation đơn giản. collect hợp với mutable result container như list, map, grouping, partitioning.

partitioningBy: chia 2 nhóm boolean

Map<Boolean, List<Invoice>> byPaymentState = invoices.stream()
    .collect(Collectors.partitioningBy(Invoice::isPaid));

List<Invoice> paid = byPaymentState.get(true);
List<Invoice> unpaid = byPaymentState.get(false);

teeing: tính hai aggregate trong một pass

record InvoiceStats(long count, BigDecimal total) {}

InvoiceStats stats = invoices.stream()
    .collect(Collectors.teeing(
        Collectors.counting(),
        Collectors.mapping(
            Invoice::amount,
            Collectors.reducing(BigDecimal.ZERO, BigDecimal::add)
        ),
        InvoiceStats::new
    ));

toMap cần merge function khi dữ liệu có thể duplicate

Map<String, Invoice> byExternalId = invoices.stream()
    .collect(Collectors.toMap(
        Invoice::externalId,
        Function.identity(),
        (older, newer) -> newer
    ));

3. Optional

Optional là cách làm absence explicit trong API. Nó không loại bỏ null khỏi toàn bộ chương trình, nhưng làm method contract rõ hơn: caller phải nghĩ tới case không có kết quả.

DùngKhông dùng
Return type của query có thể không tìm thấyField trong JPA entity
Transform result bằng map/flatMapParameter bắt caller wrap value
Fail rõ bằng orElseThrowOptional.get() không check
Lazy default bằng orElseGetCollection chứa Optional
 
public Optional<CustomerDto> findCustomer(String id) {
    return customerRepository.findById(id)
        .map(CustomerDto::from);
}

public CustomerDto requiredCustomer(String id) {
    return findCustomer(id)
        .orElseThrow(() -> new CustomerNotFoundException(id));
}

public String displayName(String id) {
    return findCustomer(id)
        .map(CustomerDto::displayName)
        .filter(name -> !name.isBlank())
        .orElseGet(() -> "Guest");
}
orElse vs orElseGet

orElse(defaultValue()) evaluate default ngay cả khi Optional có value. Nếu default là DB call, HTTP call hoặc object đắt, dùng orElseGet(() -> defaultValue()).

3.1 Optional.map vs flatMap

Dùng map khi function trả value thường. Dùng flatMap khi function đã trả Optional, để tránh Optional<Optional<T>>.

Sai: nested Optional
Optional<Optional<Address>> address =
    customerRepository.findById(id)
        .map(Customer::primaryAddress);
Đúng: flatMap
Optional<Address> address =
    customerRepository.findById(id)
        .flatMap(Customer::primaryAddress);

3.2 Optional Không Thay Validation

Optional biểu diễn absence của result, không biểu diễn validation error phong phú. Nếu cần trả nhiều lỗi validate, dùng result type riêng hoặc exception phù hợp ở boundary.

public sealed interface SignupResult permits SignupResult.Accepted, SignupResult.Rejected {
    record Accepted(UserId userId) implements SignupResult {}
    record Rejected(List<String> errors) implements SignupResult {}
}

3.3 Optional Trong Repository Và Service

Repository method như findById trả Optional là hợp lý. Service public API cần cân nhắc: nếu missing là expected, trả Optional; nếu missing là lỗi của use case, throw domain exception ngay để caller không phải handle ở mọi tầng.

MethodReturnLý do
findBySlugOptional<Article>Không tìm thấy là expected.
getRequiredWorkspaceWorkspace hoặc throwUse case không thể tiếp tục nếu thiếu.
validateSignupValidationResultCần chứa nhiều lỗi, không chỉ present/empty.
🔧 Counter-example — khi nào KHÔNG dùng Optional
  • Field của entity/DTO: private Optional<String> middleName; là anti-pattern — Optional không Serializable, tốn thêm 1 object mỗi field, và Jackson/JPA xử lý lúng túng. Dùng null + @Nullable hoặc giá trị mặc định.
  • Tham số method: void update(Optional<X> x) bắt caller bọc Optional.of(...) mỗi lần. Thà overload method hoặc nhận null.
  • Collection rỗng: Optional<List<T>> là thừa — trả List rỗng đã đủ "không có gì".
  • Trong vòng lặp nóng: mỗi Optional là một allocation; ở hot path đếm bằng triệu lần/giây, nó tạo GC pressure thật.

Quy tắc Brian Goetz: Optional sinh ra chỉ để làm return type cho "có thể không có kết quả", không phải để thay thế null ở mọi nơi.

L1 — Biết tên

Biết Optional "tránh NullPointerException". Symptom hiểu giả: gọi .get() ngay sau khi nhận Optional, hoặc nhét Optional vào field — vẫn NPE/ngược mục đích.

L2 — Explain được

Giải thích Optional là kiểu trả về báo "có thể vắng kết quả"; phân biệt map (T→R) vs flatMap (T→Optional<R>); biết orElse (luôn eval) vs orElseGet (lazy) khác nhau ở đâu.

L3 — Apply được

Thiết kế contract cả service: findBySlug trả Optional, getRequired throw domain exception, validation nhiều lỗi dùng sealed result type. Chọn đúng kiểu theo "missing là expected hay là lỗi use case".

4. Lambda, Method Reference, Functional Interface

Lambda giúp truyền behavior như value. Về mặt mental model, bạn đang đưa một function nhỏ vào API khác: filter cần predicate, map cần function, forEach cần consumer.

4.1 Functional Interfaces

InterfaceMethodNghĩaVí dụ
Supplier<T>get()Không nhận, trả valueGenerate request ID
Consumer<T>accept(T)Nhận value, không trảLog/audit một event
Function<T,R>apply(T)TransformEntity sang DTO
Predicate<T>test(T)Điều kiện true/falseLọc invoice paid
 
Predicate<Invoice> isPaid = invoice -> invoice.status() == InvoiceStatus.PAID;
Function<Invoice, InvoiceDto> toDto = InvoiceDto::from;
Consumer<InvoiceDto> trace = dto -> log.debug("invoiceId={}", dto.id());

List<InvoiceDto> dtos = invoices.stream()
    .filter(isPaid)
    .map(toDto)
    .peek(trace) // chỉ debug, không đặt business side effect quan trọng ở đây
    .toList();

Tự định nghĩa functional interface khi domain cần tên rõ

@FunctionalInterface
public interface PricePolicy {
    Money apply(Cart cart);
}

public Money calculate(Cart cart, PricePolicy policy) {
    return policy.apply(cart);
}

Dùng Function<Cart, Money> cũng được, nhưng PricePolicy truyền intent domain rõ hơn.

4.2 Lambda Capture Và Effectively Final

Lambda chỉ capture biến local effectively final. Java cố tình không cho bạn mutate biến local capture trong lambda vì nó dễ gây bug về lifecycle và concurrency.

Không compile
int count = 0;
invoices.forEach(invoice -> {
    if (invoice.isPaid()) {
        count++; // local variable must be effectively final
    }
});
Đúng: dùng stream operation
long count = invoices.stream()
    .filter(Invoice::isPaid)
    .count();

Nếu bạn dùng array hoặc AtomicInteger chỉ để "lách" effectively final trong stream, đó thường là code smell. Hãy dùng collector/reduce hoặc quay lại for-loop.

4.3 Default Methods

Default method cho phép thêm method vào interface mà không phá tất cả implementation cũ. Nó được dùng nhiều trong Java collections và framework APIs.

public interface InvoiceFormatter {
    String format(Invoice invoice);

    default List<String> formatAll(List<Invoice> invoices) {
        return invoices.stream()
            .map(this::format)
            .toList();
    }
}
Không biến interface thành abstract class trá hình

Default method nên cung cấp behavior phổ quát dựa trên method abstract cốt lõi. Nếu interface bắt đầu giữ quá nhiều logic, dependency và state giả, hãy cân nhắc class/strategy riêng.

5. Java 14-17: Record, Sealed, Pattern Matching CORE ★

⏸ Pause & Predict
Bạn định biến entity JPA Order và DTO InvoiceSummary thành record. Trả lời:
  1. Cái nào HỢP record, cái nào KHÔNG? Nêu đúng 2 lý do kỹ thuật record không hợp JPA entity.
  2. record Money(BigDecimal amount, List<Fee> fees) có thật sự immutable không, nếu caller giữ tham chiếu tới fees rồi add()? Sửa thế nào?
  3. Với một state máy có đúng 3 trạng thái (Pending/Paid/Failed), sealed interface + pattern matching cho bạn lợi gì mà enum + if-else không cho?
Gợi ý: entity cần no-arg constructor + mutable + proxy → record cấm. Record chỉ shallow-immutable → cần defensive copy. Sealed cho compiler exhaustiveness check. Đáp án ở 5.1-5.3.

5.1 Record Cho DTO Và Value Object

public record InvoiceSummary(
    String invoiceId,
    String customerId,
    BigDecimal total,
    InvoiceStatus status
) {
    public InvoiceSummary {
        Objects.requireNonNull(invoiceId);
        Objects.requireNonNull(customerId);
        Objects.requireNonNull(total);
        Objects.requireNonNull(status);
        if (total.signum() < 0) {
            throw new IllegalArgumentException("total must not be negative");
        }
    }
}

Record tự sinh constructor, accessor, equals/hashCode/toString. Không dùng record cho JPA entity vì entity cần lifecycle, proxy, no-arg constructor và thường mutable.

Record không có nghĩa là shallow object graph immutable tuyệt đối

Record fields là final, nhưng nếu field là mutable collection, content bên trong vẫn có thể bị đổi. Hãy defensive copy trong compact constructor.

Sai: list vẫn mutable từ ngoài
public record InvoiceBatch(List<InvoiceSummary> invoices) {}
Đúng: defensive copy
public record InvoiceBatch(List<InvoiceSummary> invoices) {
    public InvoiceBatch {
        invoices = List.copyOf(invoices);
    }
}

5.2 Sealed Class Cho State Hữu Hạn

public sealed interface PaymentAttempt
    permits PaymentAttempt.Approved, PaymentAttempt.Declined, PaymentAttempt.ProviderError {

    record Approved(String transactionId, Instant approvedAt) implements PaymentAttempt {}
    record Declined(String reasonCode, boolean retryable) implements PaymentAttempt {}
    record ProviderError(String provider, String message) implements PaymentAttempt {}
}

Sealed hierarchy giúp compiler biết toàn bộ subtype hợp lệ. Khi kết hợp switch expression, bạn giảm rủi ro quên case mới.

5.3 Pattern Matching Cho instanceof

public String describe(Object value) {
    if (value instanceof InvoiceSummary summary) {
        return "Invoice " + summary.invoiceId() + " = " + summary.total();
    }
    if (value instanceof CustomerDto customer) {
        return "Customer " + customer.id();
    }
    return "Unknown";
}

5.4 Switch Expression, var, Text Blocks

Switch expression giúp return value trực tiếp và tránh fall-through lỗi thời. var giảm noise khi type đã rõ từ vế phải. Text block giúp SQL/JSON mẫu dễ đọc hơn.

Switch expression

public String label(InvoiceStatus status) {
    return switch (status) {
        case DRAFT -> "Draft";
        case SENT -> "Sent";
        case PAID -> "Paid";
        case CANCELLED -> "Cancelled";
    };
}

var dùng có kiểm soát

Ổn: type rõ
var invoice = invoiceRepository.findById(id).orElseThrow();
var totalsByStatus = new EnumMap<InvoiceStatus, BigDecimal>(InvoiceStatus.class);
Kém: type không rõ
var data = client.fetch(id);
var result = processor.process(data);

Text block cho SQL/JSON

String sql = """
    select i.id, i.status, i.total_amount
    from invoices i
    where i.customer_id = :customerId
      and i.created_at >= :fromDate
    order by i.created_at desc
    """;

5.5 JPMS Module System: Biết Để Debug Dependency/Classpath

JPMS xuất hiện từ Java 9. Nhiều Spring application vẫn chạy trên classpath truyền thống, nhưng senior nên hiểu module path vì nó ảnh hưởng encapsulation, reflective access, split package và lỗi runtime khi migrate JDK.

module com.acme.reporting {
    requires java.sql;
    requires com.fasterxml.jackson.databind;

    exports com.acme.reporting.api;
    opens com.acme.reporting.dto to com.fasterxml.jackson.databind;
}

Không cần thuộc mọi option JPMS. Điểm quan trọng là hiểu khác biệt classpath/module path, vì sao reflection bị chặn mạnh hơn, và khi nào cần opens thay vì exports.

6. Collections Internals: Câu Hỏi Java Core Luôn Gặp

Stream và lambda là modern Java, nhưng senior interview vẫn thường quay về collections. Lý do đơn giản: hầu hết code backend dùng Map, List, Set, cache, grouping, deduplication. Nếu không hiểu cost model của collection, bạn có thể viết code đúng chức năng nhưng chậm, tốn memory hoặc sai trong concurrent access.

6.1 HashMap: Hash, Bucket, Resize, Treeify

HashMap lưu entry theo bucket. Key được hash để chọn bucket. Nếu nhiều key rơi vào cùng bucket, bucket chứa chain hoặc tree. Từ Java 8, collision chain dài có thể treeify thành red-black tree khi đủ điều kiện để tránh worst-case lookup quá tệ.

Khái niệmNghĩaBẫy phỏng vấn
hashCodeQuyết định bucket ban đầuKey mutable làm hash thay đổi sau khi put
equalsXác định key thật sự bằng nhauOverride equals mà quên hashCode
Load factorTỷ lệ size/capacity kích hoạt resizeMap lớn mà không set initial capacity gây resize nhiều
ResizeTạo table lớn hơn và rehash/bin splitChi phí cao nếu xảy ra trong hot path
TreeifyBucket collision dài chuyển sang treeKhông thay thế việc thiết kế hash tốt
 
public record ProductKey(String tenantId, String sku) {}

Map<ProductKey, Product> byKey = new HashMap<>(expectedSize * 4 / 3 + 1);
byKey.put(new ProductKey("tenant-a", "SKU-001"), product);
Mutable key là lỗi rất khó debug

Nếu object dùng làm key bị đổi field tham gia equals/hashCode sau khi put vào map, lookup có thể fail dù object vẫn nằm trong map. Vì vậy key nên immutable: record, final fields, hoặc value object không mutate.

Sai: mutable key
class SearchKey {
    String query;
    int page;
    // equals/hashCode dùng query + page
}

SearchKey key = new SearchKey("camera", 1);
cache.put(key, result);
key.page = 2;
cache.get(key); // có thể null
Đúng: immutable key
record SearchKey(String query, int page) {}

cache.put(new SearchKey("camera", 1), result);

6.2 ConcurrentHashMap: Atomic Theo Key, Không Phải Transaction Toàn Map

ConcurrentHashMap cho phép access concurrent an toàn hơn HashMap, nhưng không biến mọi operation nhiều bước thành atomic. Các method như putIfAbsent, compute, computeIfAbsent atomic theo key. Nếu invariant trải qua nhiều key, bạn vẫn cần thiết kế synchronization/transaction riêng.

Sai: check-then-act không atomic
if (!cache.containsKey(id)) {
    cache.put(id, loadProduct(id));
}
Đúng hơn: computeIfAbsent
Product product = cache.computeIfAbsent(id, this::loadProduct);

Function truyền vào computeIfAbsent nên nhanh, không giữ lock ngoài, không gọi ngược vào chính map theo cách phức tạp. Nếu load từ network/DB rất chậm, cân nhắc cache future hoặc thư viện cache có loading semantics rõ ràng để tránh block bin lâu.

Counter Theo Key

private final ConcurrentHashMap<String, LongAdder> views = new ConcurrentHashMap<>();

public void recordView(String productId) {
    views.computeIfAbsent(productId, ignored -> new LongAdder())
         .increment();
}

6.3 ArrayList vs LinkedList: Đừng Trả Lời Máy Móc

Câu trả lời "ArrayList đọc nhanh, LinkedList insert/delete nhanh" thường quá nông. Trong thực tế, ArrayList thường nhanh hơn cho rất nhiều workload vì dữ liệu contiguous, cache locality tốt, ít object allocation. LinkedList mỗi node là một object riêng, tốn memory và pointer chasing.

removeIf Thay Vì Remove Trong For-each

Sai: ConcurrentModificationException
for (Product product : products) {
    if (product.discontinued()) {
        products.remove(product);
    }
}
Đúng: removeIf
products.removeIf(Product::discontinued);

ConcurrentModificationException là fail-fast best effort, không phải cơ chế thread safety. Nó giúp phát hiện modify collection sai trong cùng thread/iterator, nhưng không đảm bảo bắt mọi lỗi concurrent.

6.4 TreeMap, HashMap, LinkedHashMap: Chọn Theo Contract

Map không chỉ khác nhau về performance. Chúng khác nhau về contract: có giữ order không, sort theo key không, cho null không, iteration cost thế nào.

NavigableMap<LocalDate, List<Shipment>> shipmentsByDate = new TreeMap<>();

    .subMap(startDate, true, endDate, true)

LRU Mini Bằng LinkedHashMap

public class SmallLruCache<K, V> extends LinkedHashMap<K, V> {
    private final int maxEntries;

    public SmallLruCache(int maxEntries) {
        super(16, 0.75f, true);
        this.maxEntries = maxEntries;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > maxEntries;
    }
}

Ví dụ trên phù hợp cache nhỏ trong một thread hoặc được bảo vệ bởi lock ngoài. Với production cache multi-thread, dùng thư viện cache chuyên dụng có eviction, TTL, metrics và concurrency semantics rõ ràng.

7. Code Production

7.1 Pipeline Báo Cáo Có Chunking

public BillingReport generateReport(Stream<Invoice> invoiceStream) {
    Map<InvoiceStatus, BigDecimal> totals = invoiceStream
        .filter(Objects::nonNull)
        .filter(invoice -> invoice.amount() != null)
        .collect(Collectors.groupingBy(
            Invoice::status,
            Collectors.mapping(
                Invoice::amount,
                Collectors.reducing(BigDecimal.ZERO, BigDecimal::add)
            )
        ));

    return BillingReport.from(totals);
}

Nếu source là database stream, giữ trong transaction và đóng stream đúng cách. Không collect hàng triệu row nếu có thể xử lý chunk.

7.2 Duplicate Key Khi Collect To Map

Map<String, Invoice> byId = invoices.stream()
    .collect(Collectors.toMap(
        Invoice::id,
        Function.identity(),
        (first, duplicate) -> first,
        LinkedHashMap::new
    ));

Không có merge function, duplicate key sẽ ném IllegalStateException. Đây là bug hay gặp khi dữ liệu production không sạch như test data.

7.3 Sealed Result Thay Vì Null/Exception Rải Rác

public sealed interface ImportResult permits ImportResult.Success, ImportResult.ValidationFailed, ImportResult.SystemFailed {
    record Success(int importedRows) implements ImportResult {}
    record ValidationFailed(List<String> errors) implements ImportResult {}
    record SystemFailed(String message) implements ImportResult {}
}

public HttpResponse handle(ImportResult result) {
    return switch (result) {
        case ImportResult.Success s -> HttpResponse.ok(s);
        case ImportResult.ValidationFailed v -> HttpResponse.badRequest(v.errors());
        case ImportResult.SystemFailed f -> HttpResponse.serverError(f.message());
    };
}

7.5 Refactor Một Hàm Thực Tế Theo 3 Mức

Không phải lúc nào cũng nhảy thẳng từ for-loop sang stream. Có thể refactor theo ba mức: tách predicate, đổi status string thành enum, sau đó mới cân nhắc stream.

private boolean isBillable(Invoice invoice) {
    return invoice != null
        && invoice.amount() != null
        && invoice.amount().signum() > 0
        && invoice.status() == InvoiceStatus.SENT;
}

public List<InvoiceId> billableInvoiceIds(List<Invoice> invoices) {
    return invoices.stream()
        .filter(this::isBillable)
        .map(Invoice::id)
        .toList();
}

8. Trade-offs & Anti-Patterns

AP1

Stream Với Side Effect

// Sai
List<String> ids = new ArrayList<>();
invoices.stream().filter(Invoice::isPaid).forEach(i -> ids.add(i.id()));

// Đúng
List<String> ids = invoices.stream()
    .filter(Invoice::isPaid)
    .map(Invoice::id)
    .toList();
AP2

parallelStream Cho I/O

parallelStream dùng common ForkJoinPool. Không dùng nó để gọi DB/HTTP trong lambda. Với I/O fan-out, dùng CompletableFuture và custom executor, hoặc WebFlux nếu reactive toàn đường đi.

AP3

Optional Everywhere

Optional không nên là field, parameter hoặc phần tử collection. Nó hợp nhất ở return type cho method có thể không có kết quả.

AP4

Record Cho Entity

Record immutable và final, không phù hợp với JPA proxy/lifecycle. Dùng record cho DTO, event, command, value object đơn giản.

AP5

Stream Quá Dài

Nếu pipeline cần nhiều biến tạm, nhiều nhánh, nhiều error handling và logging từng bước, tách method nhỏ hoặc dùng for-loop. Debuggability cũng là chất lượng code.

AP6

Optional.get()

optional.get() gần giống null dereference phiên bản mới. Nếu chắc chắn có value, hãy dùng orElseThrow với exception rõ. Nếu không chắc, handle empty case.

AP7

var Làm Mất Thông Tin

var tốt khi type rõ từ context. Nếu reader phải nhảy vào method khác để biết type, hãy viết type tường minh.

9. Interview Framework

SurfaceStream khác Collection?

Collection lưu dữ liệu; Stream là pipeline xử lý dữ liệu. Stream lazy, chỉ chạy khi có terminal operation, và thường chỉ consume một lần.

DeepparallelStream có luôn nhanh hơn?

Không. Nó có overhead split/merge, dùng common pool, không hợp dataset nhỏ, I/O-bound, operation có order dependency hoặc side effect. Nó hợp CPU-bound, stateless, dataset lớn và đo đạc chứng minh nhanh hơn.

ArchitectureModel result import file lớn

Dùng sealed result để biểu diễn success, validation failure, system failure. Stream file theo dòng/chunk để không load hết memory. Dùng record cho row DTO immutable. Enrich DB theo batch thay vì gọi repository từng dòng.
