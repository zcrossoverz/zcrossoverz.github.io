Observability — Log, Metric, Trace, và Alert Đúng

"Có log với dashboard" chưa phải observability. Lên thật: "log, metric, trace mỗi cái trả lời câu hỏi gì khác nhau?", "SLI/SLO/SLA khác nhau ra sao và error budget dùng để làm gì?", "alert dựa trên CPU 80% vì sao gây alert fatigue?" — là lộ hiểu giả. File này dạy 3 trụ cột + cách alert theo symptom, kèm config Logback/MDC/Micrometer thật, trên domain ShopFlow — sàn e-commerce nhiều microservice.

🎯 Mục tiêu
Debug production bằng evidence: trace → log → metric, alert đúng
📅 Thời lượng
1 tuần · Sau #17 Microservices, #24 K8s
📚 Prereq
HTTP · Spring Boot Actuator · log cơ bản
🔥 Độ ưu tiên
🟠 Cao — không quan sát được = không vận hành được

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Log, metric, trace na ná nhau, có log là đủ debug."
Thực tế: Mỗi trụ cột trả lời câu hỏi khác: metric "có vấn đề không & mức độ" (rẻ, tổng hợp); trace "vấn đề ở service nào trong chuỗi" (theo 1 request); log "chính xác chuyện gì xảy ra" (chi tiết, đắt). Thiếu trace thì trong 8 service không biết cái nào chậm.
Tưởng hiểu: "SLA, SLO, SLI là ba tên gọi của một thứ."
Thực tế: SLI = số đo thực (vd % request <300ms); SLO = mục tiêu nội bộ (99.9%); SLA = cam kết với khách hàng kèm phạt (thường lỏng hơn SLO). Error budget = 1−SLO, quyết định được phép ship rủi ro bao nhiêu.
Tưởng hiểu: "Alert càng nhiều càng an toàn, cứ CPU>80% là báo."
Thực tế: Alert theo cause (CPU cao) gây alert fatigue — CPU cao chưa chắc user bị ảnh hưởng. Alert theo symptom (error rate, latency user thấy) ít hơn, đúng hơn, đánh thức người đúng lúc.

🗺️ Roadmap — nếu chỉ có X giờ thì học gì

Thứ tự ưu tiên
1 giờ (CORE ★): 3 trụ cột mỗi cái trả lời gì (2.1), distributed tracing + correlation ID (2.3), structured logging (2.2). Đủ để debug production xuyên service.
+1 giờ (CORE ★): RED/USE metrics (2.4), SLI/SLO/error budget (2.5), alert theo symptom (2.6).
+1 giờ (SUPPLEMENTARY): Logback JSON + MDC (3.1, 3.2), Micrometer + Prometheus/Grafana (3.3).
Khi cần (ADVANCED): tail-based sampling, cardinality control, log masking PII, OpenTelemetry, tiered retention.

1. Bài toán thực tế — "request checkout chậm" mà mù CORE ★

User báo "đặt hàng mất 8 giây". ShopFlow có 8 microservice. Có log nhưng mỗi service một file, không nối được; không biết service nào tốn 5s trong 8s; không có alert nên đội biết khi user phàn nàn.

⏸ Pause & Predict — trước khi đọc lời giải
Viết ra giấy:
  1. Để trả lời "service nào tốn 5s trong 8s", cần trụ cột nào — log, metric hay trace? Vì sao 2 cái kia không trả lời được câu này?
  2. Làm sao nối log của cùng một request rải khắp 8 service? (gợi ý: một id đi xuyên suốt)
  3. Đội nên biết sự cố TRƯỚC khi user phàn nàn bằng cách nào, alert dựa trên cái gì để không nhiễu?
Đáp án: distributed tracing (2.3) trả lời "ở đâu chậm" bằng waterfall span; correlation/trace ID propagate qua header + MDC nối log (2.2, 2.3); alert theo symptom error rate/latency (2.6). Đừng đọc tiếp khi chưa map đúng câu hỏi ↔ trụ cột.

2.1 3 trụ cột — mỗi cái trả lời gì CORE ★

Trụ cộtTrả lời câu hỏiChi phí / đặc tính
Metrics"Có vấn đề không? Mức độ? Xu hướng?"Rẻ, tổng hợp số, lưu lâu; không cho chi tiết 1 request
Traces"Vấn đề ở service/bước nào trong chuỗi?"Theo 1 request xuyên service; thường sampling
Logs"Chính xác chuyện gì đã xảy ra?"Chi tiết nhất, đắt nhất để lưu/tìm
Luồng debug điển hình

Alert (metric) báo error rate tăng → mở trace tìm service chậm/lỗi → đọc log của trace đó để thấy exception cụ thể. Ba trụ cột phối hợp, không thay thế nhau. End-to-end: alert "checkout error 8%" → Grafana spike từ 14:32 → Jaeger trace 14:32 thấy inventory-service timeout 5s → Kibana log thấy SQLException: deadlock → root cause 5 phút thay vì hàng giờ.

2.2 Structured logging + log level CORE ★

Log JSON (key-value) thay vì chuỗi text → query được trong ELK/Loki (traceId:xyz AND level:ERROR). Dùng placeholder, không nối chuỗi.

// ✅ structured + rẻ (chỉ format khi level bật)
log.info("order placed orderId={} customerId={} total={}", id, cid, total);
// ❌ log.info("order placed " + id + " " + cid)  — nối chuỗi luôn, không query được field
LevelDùng khi
ERRORCần người xử lý: request thất bại không tự phục hồi
WARNBất thường nhưng đã handle (retry, fallback)
INFOSự kiện nghiệp vụ quan trọng (order placed, payment ok)
DEBUGChi tiết để debug, tắt ở prod (hoặc bật tạm theo logger)

2.3 Distributed tracing + correlation ID CORE ★

Một traceId sinh ở entry (gateway), propagate qua header (W3C traceparent) xuống mọi service; mỗi service ghi span (start/end, parentSpanId). UI (Jaeger/Tempo) vẽ waterfall: Gateway[1ms] → Order[50ms] → Inventory[200ms ← chậm!].

L1 — Biết tên

Biết "tracing theo dõi request xuyên service". Symptom hiểu giả: không nói được span/traceId propagate thế nào hay vì sao cần sampling.

L2 — Explain được

traceId sinh ở entry, đi theo header, MDC đưa traceId vào mọi dòng log; span có parent → dựng cây; sampling giảm chi phí (giữ 1–10% trace).

L3 — Apply được

Chuẩn hoá header propagation, MDC filter, tail-based sampling (giữ trace lỗi/chậm), nối trace↔log↔metric bằng exemplar; kiểm soát cardinality để Prometheus không nổ.

2.4 Metrics — RED & USE SUPPLEMENTARY

RED (cho service):  Rate (req/s) · Errors (% lỗi) · Duration (latency p50/p95/p99)
USE (cho hạ tầng):  Utilization · Saturation (queue/chờ) · Errors

Dùng percentile (p95/p99), không dùng trung bình — trung bình giấu đuôi chậm mà user thật sự cảm nhận.

2.5 SLI / SLO / SLA + error budget SUPPLEMENTARY

  • SLI (Indicator): số đo thực, vd "% request <300ms".
  • SLO (Objective): mục tiêu nội bộ, vd 99.9%.
  • SLA (Agreement): cam kết với khách hàng + phạt, thường lỏng hơn SLO.
  • Error budget = 1 − SLO (vd 0.1% = ~43 phút/tháng). Hết budget → đóng băng feature, tập trung độ ổn định.

2.6 Alerting — theo symptom CORE ★

🔧 Counter-example — khi thêm alert làm hệ thống TỆ hơn

Nhiều alert ≠ an toàn hơn. Alert sai gây alert fatigue → on-call bỏ qua cả alert thật:

  • Alert theo cause: "CPU > 80%" — CPU cao lúc batch chạy là bình thường, user không bị ảnh hưởng → đánh thức người vô ích.
  • Alert mọi metric: 200 alert rule → noise. On-call học cách ignore → bỏ sót sự cố thật.
  • Không có ngưỡng/độ trễ: spike 1 giây cũng báo → flapping.

Quy tắc: alert theo symptom user cảm nhận (error rate, latency p99, SLO burn rate), ít mà đúng; cause để chẩn đoán sau khi alert symptom nổ, không để alert.

3.2 MDC correlation filter CORE ★

@Component
class TraceIdFilter extends OncePerRequestFilter {
  @Override protected void doFilterInternal(HttpServletRequest req,
      HttpServletResponse res, FilterChain chain) throws ... {
    String traceId = req.getHeader("X-Trace-Id");
    MDC.put("traceId", traceId != null ? traceId : UUID.randomUUID().toString());
    try {
      chain.doFilter(req, res);          // mọi log trong request này tự mang traceId
    } finally {
      MDC.clear();                       // BẮT BUỘC — tránh leak qua thread pool reuse
    }
  }
}
// Khi gọi service khác: forward header để traceId xuyên suốt
// restTemplate/webClient thêm header X-Trace-Id = MDC.get("traceId")
Anti-pattern + symptom

Triệu chứng: log của request A bỗng mang traceId của request B. Nguyên nhân: quên MDC.clear() → thread pool reuse thread giữ MDC cũ → request sau "thừa kế" traceId request trước. Fix: luôn clear trong finally.

3.3 Micrometer + Prometheus alert SUPPLEMENTARY

// custom metric — Micrometer expose qua /actuator/prometheus
meterRegistry.counter("orders.placed", "channel", "web").increment();

Timer.Sample s = Timer.start(meterRegistry);
// ... xử lý checkout ...
s.stop(meterRegistry.timer("checkout.duration"));   // có p50/p95/p99 nếu cấu hình percentile
// ⚠️ KHÔNG dùng userId làm tag (label) → cardinality cao → Prometheus nổ bộ nhớ
# Prometheus alert rule — theo SYMPTOM, không theo cause
- alert: CheckoutErrorRateHigh
  expr: sum(rate(http_requests_total{job="checkout",status=~"5.."}[5m]))
      / sum(rate(http_requests_total{job="checkout"}[5m])) > 0.05
  for: 5m                       # phải kéo dài 5 phút (tránh flapping)
  labels: { severity: page }
  annotations: { summary: "checkout error rate > 5% trong 5 phút" }

5. 3-tier interview framework

Tầng 1 — SurfaceJunior gate

Q: 3 trụ cột observability là gì?

Logs (chuyện gì xảy ra, chi tiết), Metrics (có vấn đề không & mức độ, số tổng hợp), Traces (ở service nào trong chuỗi). Mỗi cái trả lời câu hỏi khác nhau, phối hợp khi debug.

Tầng 2 — Deep DiveSenior gate

Q: Thiết kế observability để trả lời "vì sao request X fail xuyên 5 service".

traceId propagate header + MDC → log JSON query theo traceId; trace UI waterfall tìm service lỗi/chậm; metric RED báo trước; alert SLO burn. End-to-end: alert → trace → log = phút thay vì giờ.

Tầng 3 — ArchitectureSenior+

Q: Quy mô lớn, chi phí observability tăng vọt. Tối ưu.

Tail-based sampling (giữ trace lỗi/chậm, bỏ trace bình thường), kiểm soát cardinality metric, log level động, tiered retention (log nóng 7 ngày, lạnh sang object storage), alert ít theo SLO burn rate thay vì mọi metric.

🔧 Lab bắt buộc — quan sát thật (2 bài)

Lab 1 — Correlation ID xuyên service. 2 service Spring Boot gọi nhau. Viết filter sinh/propagate X-Trace-Id qua header + đưa vào MDC. Gây một lỗi ở service 2, dùng traceId tìm toàn bộ log của request đó ở cả 2 service. Tiêu chí: chứng minh nối được log bằng traceId và giải thích vì sao phải MDC.clear().

Lab 2 — Alert symptom vs cause. Expose metric RED bằng Micrometer + Actuator. Tạo 2 alert: (a) CPU>80% (cause), (b) error rate>5% trong 5 phút (symptom). Chạy một batch nặng CPU nhưng không lỗi → quan sát alert (a) nổ vô ích, (b) im. Tiêu chí: giải thích vì sao symptom-based ít nhiễu hơn.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. Log, metric, trace mỗi cái trả lời câu hỏi gì? Vì sao thiếu trace thì khó debug microservice?
  2. traceId/correlation ID nối log xuyên service thế nào? Vì sao phải MDC.clear()?
  3. SLI/SLO/SLA khác nhau ra sao? Error budget dùng để làm gì?
  4. Vì sao dùng p99 thay vì trung bình cho latency?
  5. Vì sao alert theo symptom tốt hơn theo cause?
Bí câu nào = quay lại đúng section. Trôi cả 5 = nắm chắc observability mức senior.
