2. Process and port CORE

★ CORE: Biết app có chạy và listen port nào.

ps/top/ss/lsof giúp trả lời app còn sống không, PID nào, port nào, connection nào.

Đừng debug Spring khi process đã chết hoặc port không listen.

🎯 Vì sao phải nắm?

Incident đầu tiên luôn hỏi: service có sống không?

❌ Không biết kiểm tra PID
java -jar app.jar
# app fail nhưng không kiểm process
✅ Kiểm process/port
ps -ef | grep java
ss -ltnp | grep 8080

3. Logs and grep CORE

★ CORE: Log là nguồn evidence đầu tiên.

tail, grep, awk, journalctl là reflex production.

Tìm first error và context trước/sau, không chỉ dòng cuối.

🎯 Vì sao phải nắm?

Không đọc log được thì không debug được.

❌ Mở file log khổng lồ bằng editor
vim /var/log/app.log
✅ Tail/grep có context
tail -200 /var/log/app.log
grep -A 20 -B 5 "ERROR" /var/log/app.log

4. Disk and file handles SUPPLEMENTARY

SUPPLEMENTARY: Disk full làm app chết rất nhanh.

df xem filesystem, du xem folder, lsof deleted xem file đã xóa nhưng process còn giữ.

Không rm log đang mở nếu không hiểu file handle.

🎯 Vì sao phải nắm?

Disk full là incident junior gặp rất nhiều.

❌ rm log đang mở
rm /var/log/app.log
✅ truncate hoặc logrotate
truncate -s 0 /var/log/app.log
lsof | grep deleted

5. Network debugging ADVANCED

ADVANCED: Phân biệt DNS, TCP và HTTP.

ping/nslookup kiểm DNS/ICMP, nc/telnet kiểm TCP, curl kiểm HTTP.

Một service không gọi được service khác có thể fail ở nhiều layer.

🎯 Vì sao phải nắm?

Backend microservices không tránh khỏi network issue.

❌ Chỉ nói service B down
curl service-b:8080/health
# failed, kết luận service B chết
✅ Tách DNS/TCP/HTTP
nslookup service-b
nc -zv service-b 8080
curl -v http://service-b:8080/health

🎯 Edge Cases — Production Reality SUPPLEMENTARY

🎯 Edge Cases — Production Reality

Happy path chưa đủ. Junior cần học cách hỏi: input lạ, data lớn, concurrency, security và vận hành thì sao?

  • Race condition: Hai thao tác cùng chạy có thể ghi đè state; cần lock/atomic workflow khi phù hợp.
  • Scale limit: Dataset hoặc repo lớn làm thao tác vốn ổn ở local trở nên chậm.
  • Security: Không để lộ secret, token, PII trong log, command history hoặc commit.
  • Data integrity: Phải biết khi nào thao tác làm mất dữ liệu và cách rollback.
  • Time/timezone: Log, file timestamp, commit time và scheduler nên dùng UTC/ISO khi debug.
  • Concurrent access: Nhiều dev/process cùng chạm vào cùng resource cần quy ước rõ.
  • Recovery: Luôn có đường quay lại: backup, reflog, rollback, restore point hoặc test reproduction.

CORE Code Review Checklist cho topic này

Dùng checklist này khi review PR của chính mình. Nó cố tình cụ thể để tránh câu “nhìn ổn”.

  1. Tên biến/method có nói đúng intent không?
  2. Input null/empty/invalid được xử lý rõ chưa?
  3. Data lớn có làm complexity đổi cấp không?
  4. Có shared mutable state không?
  5. Lỗi có bị swallow hoặc che mất stack trace không?
  6. Log có đủ context nhưng không lộ secret không?
  7. Có rollback/recovery path nếu thao tác fail giữa chừng không?
  8. Có test chứng minh edge case quan trọng không?
  9. Code có phụ thuộc implementation thay vì abstraction không cần thiết không?
  10. Có comment giải thích mechanism khó, không comment hiển nhiên không?
  11. Có source official nào confirm behavior không?
  12. Có cách reproduce bug nếu người khác hỏi không?
  13. Có trade-off được ghi trong README/PR description không?
  14. Có tránh domain-specific assumption quá hẹp không?
  15. Có thể dạy lại cho junior khác trong 10 phút không?
❌ Không có evidence
I think this should work.\nNo test. No command output. No rollback note.
✅ Có evidence tối thiểu
Test: passes edge cases.\nCommand/output recorded.\nKnown trade-off documented.\nRollback/recovery step written.

Interview Drill Bank CORE

Luyện nói thành tiếng. Mỗi câu nên trả lời trong 1-3 phút, có ví dụ và trade-off.

CORE Q1. Vì sao Linux for Backend Dev quan trọng với junior backend?

Answer mẫu: Vì nó xuất hiện trong daily work và là nền để không gây lỗi khó debug. Nắm concept này giúp bạn tự xử lý lỗi thay vì chờ senior cứu.

Signal: Nối được kiến thức với production failure.

CORE Q2. Bạn debug khi không hiểu lỗi bắt đầu từ đâu thế nào?

Answer mẫu: Xác định state hiện tại, input, expected output, command/log gần nhất, rồi thu hẹp từng bước. Không sửa mù khi chưa có bằng chứng.

Signal: Có method điều tra.

SUPPLEMENTARY Q3. Khi nào nên viết runbook?

Answer mẫu: Khi workflow có nhiều bước, có thể fail, hoặc người khác cần lặp lại an toàn. Runbook tốt có precheck, command, expected output và rollback.

Signal: Có tư duy vận hành.

ADVANCED Q4. Dấu hiệu bạn thật sự hiểu một foundation topic?

Answer mẫu: Bạn giải thích được cơ chế, dự đoán edge case, viết test phá happy path và dạy lại người khác.

Signal: Không chỉ thuộc syntax.

CORE Q5. Nếu code chạy đúng ở local, vì sao senior vẫn có thể reject?

Answer mẫu: Vì local happy path chỉ chứng minh một input nhỏ chạy được. Senior review failure mode: input bẩn, data lớn, concurrency, rollback, observability, security và maintainability.

Signal: Bạn hiểu code review không phải bắt bẻ style.

CORE Q6. Một câu trả lời junior tốt khác gì câu trả lời thuộc lòng?

Answer mẫu: Nó có ví dụ cụ thể, trade-off, và nói được khi nào cách đó sai. Thuộc lòng thường chỉ có định nghĩa.

Signal: Có thể tự tạo ví dụ mới.

SUPPLEMENTARY Q7. Bạn sẽ viết checklist học topic này thế nào?

Answer mẫu: Chia thành concept, code example, edge case, exercise, Feynman explanation và source verification.

Signal: Có cách học có thể lặp lại.

SUPPLEMENTARY Q8. Khi gặp output lạ, bước đầu tiên là gì?

Answer mẫu: Đóng băng hiện trạng: input, command, log, version, branch, environment. Sau đó thu hẹp bằng reproduction nhỏ.

Signal: Không sửa mù.

ADVANCED Q9. Làm sao biết mình đã hiểu thật?

Answer mẫu: Bạn có thể phá code, sửa code, giải thích mechanism, và dạy người khác tránh lỗi tương tự.

Signal: Hiểu thật gắn với hành động.

ADVANCED Q10. Nếu phải mentor junior khác, bạn bắt đầu từ đâu?

Answer mẫu: Bắt đầu từ lỗi thật họ vừa gặp, vẽ mental model, sửa một ví dụ nhỏ, rồi cho họ tự làm biến thể.

Self-Test Scenarios SUPPLEMENTARY

Đọc bối cảnh, tự viết đáp án, rồi mới xem gợi ý. Đây là phần chống đọc hiểu giả mạnh nhất.

CORE Scenario 1 — Local pass, production fail

Bối cảnh: Feature chạy được trên máy cá nhân nhưng fail khi data lớn hoặc nhiều người dùng.

Tự trả lời: Bạn cần thêm bằng chứng gì?

Gợi ý: So sánh input, environment, concurrency, data size và logs.

SUPPLEMENTARY Scenario 2 — Junior panic khi gặp lỗi lạ

Bối cảnh: Console có stack trace/command output dài.

Tự trả lời: Bạn dạy bạn đó đọc từ đâu?

Gợi ý: Tìm first cause, command gần nhất, state trước/sau, rồi reproduce nhỏ.

ADVANCED Scenario 3 — Review PR foundation

Bối cảnh: Code chạy nhưng thiếu guard/rollback/log.

Tự trả lời: Comment thế nào để người viết học được?

Gợi ý: Nêu failure mode cụ thể, hậu quả và hướng sửa nhỏ.

CORE Scenario 4 — Code review bị hỏi 'vì sao?'

Bối cảnh: Bạn chọn một cách implement nhưng chỉ trả lời được 'em thấy người ta hay làm vậy'.

Tự trả lời: Làm sao nâng câu trả lời?

Gợi ý: Nói problem, constraint, trade-off, alternative và lý do chọn. Nếu chưa nói được, bạn mới copy pattern.

CORE Scenario 5 — Unit test pass nhưng integration fail

Bối cảnh: Test nhỏ mock hết dependency, lên môi trường thật fail vì data/permission/path khác.

Tự trả lời: Thiếu loại kiểm chứng nào?

Gợi ý: Cần test gần reality hơn: integration, contract, command transcript, hoặc environment checklist.

SUPPLEMENTARY Scenario 6 — Bug chỉ xảy ra khi chạy song song

Bối cảnh: Hai request hoặc hai dev thao tác cùng resource làm state lệch.

Tự trả lời: Bạn sẽ tìm race thế nào?

Gợi ý: Tìm shared mutable state, check-then-act, non-atomic operation, lock/transaction boundary.

SUPPLEMENTARY Scenario 7 — Log không đủ debug

Bối cảnh: Production chỉ có dòng 'failed' mà không có id, input, stack trace.

Tự trả lời: Log tối thiểu cần gì?

Gợi ý: Event name, correlation id, business id, outcome, duration, exception cause, không chứa secret.

ADVANCED Scenario 8 — Tự viết runbook cho lỗi đã gặp

Bối cảnh: Bạn vừa được senior cứu một lỗi trong 5 phút.

Tự trả lời: Biến nó thành tài sản học thế nào?

Gợi ý: Ghi symptom, command, expected output, root cause, fix, rollback và prevention checklist.
