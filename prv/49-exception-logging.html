Vì Sao File Này Tồn Tại?

3 câu chuyện thật trong nghề mình đã chứng kiến:

📖 Câu chuyện 1 — Stack trace biến mất

Junior catch Exception e rồi log.error("error") không có stack trace. Production crash, log chỉ có 1 dòng "error". Sếp gọi 2 giờ đêm, không tìm ra root cause, mất 8 tiếng vì phải reproduce.

📖 Câu chuyện 2 — Log password customer

Junior log log.info("User password: " + user.getPassword()) để debug. Code lên prod, log file có 50,000 dòng password customer. Audit phát hiện → công ty bị fine, junior bị warning.

📖 Câu chuyện 3 — e.printStackTrace()

Junior viết try { ... } catch (Exception e) { e.printStackTrace(); }. Stack trace in ra System.err, không vào log file. Production debug = ssh vào container đã chết → vô vọng.

3 câu chuyện trên xảy ra vì junior thiếu mental model về exception + logging. File này lấp gap đó.

Phần AException Handling

A.1 Bài Toán Thực Tế — Code Junior Phá Production

❌ Code junior — chạy được, nhưng production sẽ chết
public class TransferService {
    public void transfer(String fromId, String toId, BigDecimal amount) {
        try {
            Account from = accountRepo.findById(fromId);
            Account to = accountRepo.findById(toId);
            from.withdraw(amount);
            to.deposit(amount);
            accountRepo.save(from);
            accountRepo.save(to);
        } catch (Exception e) {
            // Im lặng, ai biết được
        }
    }
}
🔥 5 vấn đề chí mạng senior phát hiện ngay
  1. Swallow exception: catch Exception → empty block. Lỗi mất hoàn toàn, caller nghĩ thành công.
  2. Catch quá rộng: bắt Exception → bắt cả OutOfMemoryError-like cases (pattern xấu).
  3. No logging: ngay cả khi log, người sau không biết exception là gì.
  4. No re-throw: caller không biết thất bại → tài khoản A trừ rồi nhưng B chưa cộng → tiền biến mất.
  5. No transaction boundary: nếu save(to) fail, save(from) đã commit → tiền thật sự biến mất.
✅ Refactor đúng
@Service
public class TransferService {
    private static final Logger log = LoggerFactory.getLogger(TransferService.class);
    private final AccountRepository accountRepo;

    public TransferService(AccountRepository accountRepo) {
        this.accountRepo = accountRepo;
    }

    @Transactional
    public TransferResult transfer(String fromId, String toId, BigDecimal amount)
            throws AccountNotFoundException, InsufficientFundsException {

        Account from = accountRepo.findById(fromId)
            .orElseThrow(() -> new AccountNotFoundException(fromId));
        Account to = accountRepo.findById(toId)
            .orElseThrow(() -> new AccountNotFoundException(toId));

        from.withdraw(amount);  // throws InsufficientFundsException nếu không đủ
        to.deposit(amount);
        accountRepo.save(from);
        accountRepo.save(to);

        log.info("Transfer success: from={}, to={}, amount={}", fromId, toId, amount);
        return new TransferResult(from.getId(), to.getId(), amount);
    }
}

Khác biệt:

  • Throw checked exception specific (AccountNotFoundException, InsufficientFundsException)
  • @Transactional để đảm bảo atomicity
  • Log success với context đủ để debug
  • Caller buộc handle hoặc declare throws
  • Không catch — let it bubble up đến @ControllerAdvice

A.2 Mental Model — Exception Hierarchy

Exception Hierarchy — học thuộc, quan trọng cho mọi câu hỏi exception trong phỏng vấn
LoạiNghĩaKhi nào dùng
ErrorJVM problemKHÔNG catch (trừ khi biết rõ làm gì)
RuntimeException (unchecked)Bug logic, không bắt buộc handleCustom exception cho validation, business rule violation
Checked ExceptionRecoverable, caller có thể handleI/O, DB, external service — caller "có lựa chọn" handle hoặc bubble up

Checked vs Unchecked — Cuộc Tranh Luận Vĩnh Cửu

🏛️ Trường phái Checked (Java truyền thống)
  • Force caller think về error case
  • Compile-time safety
  • Self-documenting (nhìn signature biết exception nào có thể xảy ra)
🚀 Trường phái Unchecked (Spring, modern)
  • Checked exception lan ra signature, code clutter (throws everywhere)
  • Caller thường wrap thành RuntimeException dù sao
  • Spring chuyển hết SQLException → RuntimeException (DataAccessException)
📖 Quy tắc thực tế

• Custom business exception → unchecked (extends RuntimeException)
• Recoverable I/O ở layer thấp → checked (IOException)
• Application layer → bắt checked + wrap thành domain exception unchecked
• Controller layer → @ExceptionHandler convert thành HTTP response

✅ Pattern: bắt checked ở layer thấp, throw domain exception unchecked
public class FileStorageService {
    public byte[] read(String path) {
        try {
            return Files.readAllBytes(Path.of(path));
        } catch (IOException e) {
            throw new StorageException("Cannot read file: " + path, e);
        }
    }
}

public class StorageException extends RuntimeException {
    public StorageException(String message, Throwable cause) {
        super(message, cause);  // ← KHÔNG quên cause!
    }
}

A.3 Cú Pháp — try / catch / finally / try-with-resources

try-with-resources (Java 7+) — Auto-close resource

// ✅ Auto-close mọi resource implements AutoCloseable
try (FileInputStream fis = new FileInputStream("file.txt");
     BufferedReader reader = new BufferedReader(new InputStreamReader(fis))) {

    String line;
    while ((line = reader.readLine()) != null) {
        process(line);
    }
} catch (IOException e) {
    log.error("File processing failed", e);
    throw new ProcessingException("Failed to process file.txt", e);
}
// fis, reader auto-close kể cả có exception

Multi-catch (Java 7+)

try {
    riskyOperation();
} catch (IOException | SQLException e) {  // 1 block xử lý 2 type
    log.error("External resource failed", e);
    throw new RuntimeException(e);
}

Bẫy finally — return bị override

⚠️ Câu phỏng vấn kinh điển
public int badExample() {
    try {
        return 1;
    } finally {
        return 2;  // ❌ Finally override return của try!
    }
}
// Output: 2

Quy tắc: KHÔNG return trong finally. Chỉ cleanup, không thay đổi flow.

A.4 Custom Exception — Banking Domain

// Base class cho domain exception
public abstract class BankingException extends RuntimeException {
    private final String errorCode;

    protected BankingException(String errorCode, String message) {
        super(message);
        this.errorCode = errorCode;
    }

    protected BankingException(String errorCode, String message, Throwable cause) {
        super(message, cause);
        this.errorCode = errorCode;
    }

    public String getErrorCode() { return errorCode; }
}

// Concrete exceptions
public class AccountNotFoundException extends BankingException {
    public AccountNotFoundException(String accountId) {
        super("ACC_NOT_FOUND", "Account not found: " + accountId);
    }
}

public class InsufficientFundsException extends BankingException {
    private final BigDecimal required;
    private final BigDecimal available;

    public InsufficientFundsException(String accountId, BigDecimal required, BigDecimal available) {
        super("INSUFFICIENT_FUNDS",
            String.format("Account %s: required %s, available %s", accountId, required, available));
        this.required = required;
        this.available = available;
    }

    public BigDecimal getRequired() { return required; }
    public BigDecimal getAvailable() { return available; }
}
📋 5 quy tắc viết custom exception
  1. Extend RuntimeException trừ khi caller thật sự cần handle
  2. Có errorCode để client/log machine-readable
  3. Constructor giữ cause: super(message, cause) — KHÔNG nuốt mất cause
  4. Message có context: ID, value, expected vs actual
  5. Field bổ sung nếu caller cần biết detail (như required, available)

Spring @ControllerAdvice — Map Exception → HTTP

@RestControllerAdvice
public class GlobalExceptionHandler {
    private static final Logger log = LoggerFactory.getLogger(GlobalExceptionHandler.class);

    @ExceptionHandler(AccountNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleAccountNotFound(AccountNotFoundException e) {
        log.warn("Account not found: {}", e.getMessage());
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
            .body(new ErrorResponse(e.getErrorCode(), e.getMessage()));
    }

    @ExceptionHandler(InsufficientFundsException.class)
    public ResponseEntity<ErrorResponse> handleInsufficientFunds(InsufficientFundsException e) {
        log.warn("Insufficient funds: {}", e.getMessage());
        return ResponseEntity.status(HttpStatus.BAD_REQUEST)
            .body(new ErrorResponse(e.getErrorCode(), e.getMessage()));
    }

    @ExceptionHandler(Exception.class)  // Catch-all cuối cùng
    public ResponseEntity<ErrorResponse> handleUnexpected(Exception e) {
        log.error("Unexpected error", e);  // log full stack trace
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
            .body(new ErrorResponse("INTERNAL_ERROR", "Internal server error"));
        // ⚠️ KHÔNG leak stack trace ra response — security risk
    }
}

A.5 Anti-Patterns — 7 Sai Lầm Junior Phổ Biến

#1 Swallow exception

❌ Bug bị nuốt
try { riskyCall(); }
catch (Exception e) { /* empty */ }
// Không ai biết
✅ Log + rethrow
try { riskyCall(); }
catch (Exception e) {
    log.error("Risky call failed", e);
    throw new RuntimeException(e);
}

#2 Mất cause khi rethrow

❌ Mất stack trace gốc
catch (IOException e) {
    throw new RuntimeException(e.getMessage());
    // ← chỉ lấy message
}
✅ Giữ cause
catch (IOException e) {
    throw new RuntimeException(
        "Failed to read file: " + path, e);
    // ← truyền e
}

#3 Catch quá rộng

❌ Bắt cả NPE bug
try {
  user.getProfile()
      .getAddress().getCity();
} catch (Exception e) {
  return "Unknown";
  // Bug NPE bị che mất
}
✅ Catch specific
try {
  apiClient.fetch();
} catch (TimeoutException e) {
  log.warn("API timeout, return cache");
  return cache.get();
}

#4 e.printStackTrace()

❌ In ra System.err
catch (Exception e) {
    e.printStackTrace();
    // Production không thấy được
}
✅ Dùng logger
catch (Exception e) {
    log.error("Operation failed", e);
}

#5 throws Exception (lười)

❌ Quá generic
public void process() throws Exception { }
// Caller không biết exception nào
✅ Specific hoặc unchecked
public void process()
  throws ProcessingException,
         ValidationException { }
// Hoặc throws unchecked (không declare)

#6 Rethrow trống rỗng không thêm context

❌ Mất context
catch (SQLException e) {
    throw new RuntimeException(e);
}
✅ Thêm context
catch (SQLException e) {
    throw new DatabaseException(
      "Failed to update account "
      + accountId + " balance", e);
}

#7 Exception cho control flow

❌ Dùng exception thay if/else
public boolean isUserExists(String id) {
  try {
    userRepo.findById(id).get();
    return true;
  } catch (NoSuchElementException e) {
    return false;
  }
}
// Exception expensive (stack trace)
✅ Optional / boolean
public boolean isUserExists(String id) {
  return userRepo.findById(id).isPresent();
}
📖 Quy tắc 10 năm exp

"Nếu không biết nên handle exception thế nào, let it bubble up. Để layer cao nhất (controller / global handler) quyết định. Đừng catch chỉ để log."

B.2 Mental Model — 5 Log Levels

LevelKhi nào dùngProduction có log không?
ERRORException không expected, phải báo on-call✅ Luôn
WARNRecoverable issue, retry, fallback, deprecated API✅ Luôn
INFOBusiness event quan trọng (transfer, login, order create)✅ Luôn
DEBUGChi tiết để debug❌ Không (hoặc chỉ tạm)

⚠️ Junior anti-pattern

Dùng INFO cho mọi thứ → log file phình to, không tìm được error trong noise. Hỏi mình: "Việc này có cần ai đó nửa đêm xem không?" — nếu không, ít nhất là DEBUG.

B.3 SLF4J + Logback — Setup Chuẩn

Spring Boot đã include sẵn SLF4J + Logback — không cần thêm dependency.

Cú pháp dùng SLF4J

import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class TransferService {
    // private static final — convention chuẩn
    private static final Logger log = LoggerFactory.getLogger(TransferService.class);

    public void transfer(String fromId, String toId, BigDecimal amount) {
        // Parameterized logging — KHÔNG concat string
        log.info("Transfer initiated: from={}, to={}, amount={}", fromId, toId, amount);

        try {
            // logic
            log.info("Transfer success: txId={}", txId);
        } catch (InsufficientFundsException e) {
            log.warn("Transfer failed (insufficient funds): from={}, amount={}", fromId, amount);
            throw e;
        } catch (Exception e) {
            log.error("Transfer failed unexpectedly: from={}, to={}", fromId, toId, e);
            // ↑ Truyền `e` ở cuối → log full stack trace
            throw e;
        }
    }
}
⚠️ Quy tắc cú pháp — học thuộc
// ❌ String concat — luôn build string ngay cả khi log level OFF
log.debug("User: " + user.getName() + ", id: " + user.getId());

// ✅ Parameterized — chỉ build string nếu level ON
log.debug("User: {}, id: {}", user.getName(), user.getId());

// ✅ Exception ở argument cuối — log stack trace
log.error("Failed to process user: {}", userId, exception);

// ❌ Đừng put exception vào placeholder
log.error("Failed: {}", exception);  // chỉ in toString, không stack trace

Lombok @Slf4j shortcut

@Slf4j  // Lombok generate `private static final Logger log` tự động
@Service

        log.info("Working...");

Pattern format giải thích

%d{yyyy-MM-dd HH:mm:ss.SSS}timestamp
[%thread]thread name
%-5levellog level (5 ký tự, padded)
[%X{traceId:-}]MDC value (trace ID), default '-' nếu null
%logger{36}class name (max 36 chars)
%msglog message
%nnewline

B.5 MDC — Mapped Diagnostic Context (Quan Trọng Nhất)

MDC là cách add context (như traceId, userId) vào mọi log trong cùng request mà không cần truyền tham số.

@Component
public class RequestLoggingFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res, FilterChain chain)
            throws ServletException, IOException {

        String traceId = req.getHeader("X-Trace-Id");
        if (traceId == null) traceId = UUID.randomUUID().toString();

        String userId = extractUserId(req);  // từ JWT/session

        try {
            MDC.put("traceId", traceId);
            MDC.put("userId", userId);

            chain.doFilter(req, res);

        } finally {
            MDC.clear();  // ← QUAN TRỌNG: clear sau khi xong, tránh leak sang request khác
        }
    }
}

Ở các class khác, không cần truyền traceId — log tự có:

@Service
public class TransferService {
    public void transfer(...) {
        log.info("Transfer initiated");
        // Log sẽ tự có [traceId=abc123] [userId=USER_5523]
    }
}
⚠️ Bẫy MDC + thread pool

MDC dùng ThreadLocal, nếu submit task sang executor, MDC không pass tự động. Phải copy thủ công:

Map<String, String> contextMap = MDC.getCopyOfContextMap();
executor.submit(() -> {
    MDC.setContextMap(contextMap);
    try {
        doWork();
    } finally {
        MDC.clear();
    }
});

B.6 Anti-Patterns — Logging Sai Cách

#1 Log password / PII (CRITICAL — banking fine)

// ❌ DỪNG NGAY!
log.info("Login attempt: username={}, password={}", username, password);
log.debug("User data: {}", user);  // user.toString() có thể chứa password, SSN, credit card

// ✅
log.info("Login attempt: username={}", username);
log.debug("User data: id={}, email={}", user.getId(), maskEmail(user.getEmail()));
NEVER log (Banking PII)Hậu quả nếu vi phạm
Password, PIN, OTP• GDPR fine €20M hoặc 4% revenue
• PCI-DSS fine
• Audit fail
• Junior bị warning hoặc terminate
Full credit card number (chỉ last 4)
SSN/CMND/CCCD đầy đủ
Account balance đầy đủ (mask 4 số cuối)
Authentication token

#2 Log trong loop

// ❌ Loop 1M items, mỗi item 1 log → log file 1M dòng
for (Transaction t : transactions) {
    log.info("Processing transaction: {}", t.getId());
    process(t);
}

// ✅ Log batch, hoặc DEBUG
log.info("Processing {} transactions", transactions.size());
for (Transaction t : transactions) {
    log.debug("Processing transaction: {}", t.getId());  // DEBUG chỉ bật khi cần
    process(t);
}
log.info("Processed {} transactions successfully", transactions.size());

#3 Log exception nhiều lần khi bubble up

⚠️ Quy tắc

Log exception 1 lần ở layer cao nhất bắt nó (thường là @ControllerAdvice). Layer dưới chỉ throw, không log → tránh 1 exception có 3 stack trace trong log = noise.

#4 log.error() khi không phải error

// ❌ Validation fail là expected, không phải error
log.error("Invalid email: {}", email);

// ✅ WARN hoặc INFO

❌ Junior thường viết

    "DEBUG: user = " + user);

✅ Luôn dùng logger
log.debug("User: {}", user);

#6 Log exception thiếu stack trace

❌ Mất stack trace
catch (Exception e) {
  log.error("Failed: " + e.getMessage());
}
✅ Truyền exception
catch (Exception e) {

Exercise 2: Setup MDC + traceId (Medium)

Spring Boot project, implement:

  • Filter add traceId vào MDC mỗi request
  • Log pattern hiển thị traceId
  • Test: gửi 3 request đồng thời, verify mỗi request có traceId riêng trong log

Interview Framework

Tầng 1 — SurfaceJunior
Q: Checked vs Unchecked exception?
"Checked extends Exception (trừ RuntimeException) — compiler force handle hoặc throws. Ví dụ: IOException, SQLException. Unchecked extends RuntimeException — không bắt buộc handle. Ví dụ: NPE, IllegalArgument.

Modern Java + Spring xu hướng dùng unchecked nhiều hơn. Spring wrap SQLException → DataAccessException unchecked. Lý do: checked clutter signature, caller thường wrap thành RuntimeException dù sao."
Q: try-with-resources hoạt động thế nào?
"Java 7+ feature. Resource khai báo trong try (...) phải implement AutoCloseable. JVM tự gọi close() ở finally implicit, kể cả có exception. Multi-resource thì close theo thứ tự ngược lại.

So với try-finally truyền thống: ngắn gọn hơn, không quên close, không cần null check, không cần try-catch nested cho close()."
Tầng 2 — Deep DiveMid
Q: Tại sao log parameterized thay vì string concat?
"2 lý do:

1. Performance: log.debug('User ' + user) luôn build string, kể cả khi DEBUG OFF. log.debug('User {}', user) chỉ build khi DEBUG ON. Hot path log nhiều → khác biệt rõ.

2. Consistency: SLF4J chuẩn hóa format, parser ELK dễ hiểu placeholder.

Edge case: nếu argument có toString() đắt (object lớn), kể cả parameterized cũng tốn — phải if (log.isDebugEnabled()) guard."
Q: MDC là gì? Tại sao quan trọng cho microservices?
"MDC là ThreadLocal map gắn key-value vào mọi log trong thread hiện tại. Trong microservices, traceId truyền qua header HTTP, mỗi service đặt traceId vào MDC → log của tất cả service trong 1 request có cùng traceId → distributed tracing.

Pitfall: thread pool / async — MDC không tự pass. Phải copy contextMap trước submit task. Spring Cloud Sleuth + Brave/OpenTelemetry tự động hóa được, nhưng manual code phải nhớ."
Tầng 3 — ArchitectureSenior
Q: Design exception handling strategy cho microservices banking.
"Layer-by-layer:

Domain layer: throw business exception (InsufficientFunds, AccountFrozen) — extends RuntimeException + có errorCode.

Repository layer: catch SQLException → wrap thành DatabaseException (Spring DataAccessException nếu dùng JPA). Không expose JDBC chi tiết ra ngoài.

Service layer: business logic, throw domain exception. Không catch trừ khi handle có ý nghĩa (retry, fallback).

Controller layer: @ControllerAdvice convert exception → ResponseEntity. ErrorResponse có errorCode (machine-readable) + message (user-friendly). KHÔNG leak stack trace ra response.

Cross-cutting:
• All log có traceId qua MDC
• Sentry/Rollbar capture stack trace cho ERROR
• Metric counter cho mỗi error type (Prometheus)
• Alert trên error rate > threshold (PagerDuty)

Audit log (banking compliance): mọi sensitive operation log vào append-only audit table, không vào file log thường."
