Event-Driven Là Gì?

Event-Driven Architecture (EDA) là kiến trúc nơi components communicate qua events thay vì direct calls. 2 ý tưởng cốt lõi:

  1. Decouple producers from consumers: producer chỉ publish event, không biết ai consume.
  2. Events là first-class citizens: persist như data, replay được, queryable.

EDA thường đi cùng Event Sourcing (lưu events thay vì state) và CQRS (tách read/write model — xem #17).

🎯 Vì sao học EDA?
  • Microservices communication không thể chỉ là REST sync calls — sẽ cascade fail (xem #17)
  • Audit/compliance requirement (GDPR, financial regulation) cần full history of changes
  • Debug production bug cần replay actual sequence of events đã xảy ra
  • Analytics realtime cần stream of events flow vào ML/dashboard pipeline

1. Bài Toán Thực Tế — Order Tracking Cho Sàn TMĐT

Requirement

Customer service nhận khiếu nại: "Đơn hàng #12345 của tôi báo delivered ngày 25/3 nhưng tôi chưa nhận được. Lúc 14:32 ngày 24/3 ai đã update status?"

Engineer query DB:

SELECT * FROM orders WHERE id = 12345;
-- order_id | status     | updated_at         | updated_by
-- 12345    | DELIVERED  | 2026-03-25 09:15   | system

Chỉ thấy state cuối cùng. Không biết:

  • Trước đó status là gì? Khi nào thay đổi?
  • Ai đã update? Driver tự confirm hay system tự động?
  • Lúc 14:32 ngày 24/3 status là gì? GPS location lúc đó?

Workaround cũ — Audit log table

CREATE TABLE order_audit_log (
    id BIGSERIAL,
    order_id BIGINT,
    field_name VARCHAR(50),
    old_value TEXT,
    new_value TEXT,
    changed_by VARCHAR(50),
    changed_at TIMESTAMP
);
-- 1 row mỗi field change. Order có 10 fields × 5 updates avg = 50 rows/order
-- 10K orders/ngày × 365 ngày = 183M rows/năm
-- Query "state của order X lúc Y" phải full table scan + replay → slow
🔥 Vấn đề của approach truyền thống
  • Storage explosion: 183M rows audit table chỉ cho 1 năm
  • Query "state at time T" chậm: 30s scan + reconstruct
  • Schema thay đổi gây mất history: thêm column mới → audit log trước đó null cột này
  • Không thể replay business logic: chỉ biết field value thay đổi, không biết EVENT gì gây ra (cancel? refund? shipping update?)

5 câu hỏi suy ngẫm

  1. Làm sao lưu order history để trả lời "state lúc 14:32 ngày 24/3" trong < 100ms?
  2. Nếu event sourcing lưu chuỗi events, làm sao query nhanh "all orders > 5tr trong tháng 3"?
  3. Khi Kafka deliver cùng event 2 lần (at-least-once), làm sao tránh trừ stock 2 lần?
  4. Schema event thay đổi sau 2 năm (thêm field mới), replay event cũ thế nào?
  5. GDPR yêu cầu xóa data user — event store nên append-only hay allow delete?

🎯 Vì sao phân biệt quan trọng?

Junior hay nhầm MakePaymentEvent hoặc SendEmailEvent — tên giống command (intent). Nếu publish "event" này nhưng payment chưa thật sự thực hiện (validation fail), consumer tưởng đã thanh toán và update state sai. Đặt tên đúng: past tense cho event (PaymentMadeEvent, EmailSentEvent) — chỉ publish khi action đã xảy ra thật.

Event Sourcing — Lưu Chuỗi Events Thay Vì State

🎯 Vì sao Event Sourcing > Audit log truyền thống?
  • Single source of truth: state hiện tại derive từ events. Không có thứ "state" tách biệt với "audit log" — chúng là một.
  • Business-level events: event là OrderShipped (business action), không phải status field changed from PACKED to SHIPPED (technical detail). Customer service đọc hiểu ngay.
  • Time travel: replay events đến time T → state tại T. Customer service hỏi "lúc 14:32 ngày 24/3 status là gì?" → replay đến time đó.
  • Replay debug: production bug phát hiện → load events vào dev env, replay với logger debug → tìm ra root cause.
  • New views from old data: thêm analytics metric mới (ví dụ "average packing time") → backfill từ event history. Audit log không làm được.
⚠️ Costs
  • Storage: events tích lũy mãi mãi. 10K orders/ngày × 8 events/order = 29M events/năm. Cần partition / archive strategy.
  • Query phức tạp: không thể SELECT * WHERE status = 'DELIVERED' trực tiếp — phải có projection (read model) cập nhật từ events.
  • Schema evolution: event format thay đổi sau 2 năm — phải handle backward compatibility khi replay.
  • Eventual consistency: projection lag write vài ms-s. Không cho "read your write" UX.

Snapshot Pattern — Tối Ưu Replay

Order có 8 events lifetime — không vấn đề. Nhưng shopping cart hoặc user session có thể có hàng nghìn events (mỗi click add/remove item là 1 event). Replay 5K events mỗi lần load cart = chậm.

🎯 Snapshot strategy
  • Frequency trade-off: snapshot quá thường → tốn storage. Snapshot quá hiếm → replay nhiều events. Quy tắc: snapshot mỗi 100-1000 events, hoặc sau mỗi N giờ.
  • Async creation: tạo snapshot background job, không trong write path.
  • Versioning: snapshot serialize state object. Schema state thay đổi → cần snapshot migration logic hoặc throw away old snapshots + regenerate.

Idempotent Consumer — Tránh Process Duplicate Events

Kafka, RabbitMQ delivery guarantee at-least-once: message gửi ít nhất 1 lần, có thể nhiều lần (network retry). Consumer phải idempotent — process cùng event N lần kết quả phải giống nhau.

❌ Không idempotent
@KafkaListener(topics = "orders")
public void on(OrderPaidEvent e) {
    // Network retry → cùng event redelivered
    inventory.reduceStock(
        e.productId(), e.quantity());
    // Stock trừ 2 lần!
    // Customer order 1 cái, stock - 2
}
✅ Idempotent với deduplication
@KafkaListener(topics = "orders")
@Transactional
public void on(OrderPaidEvent e) {
  // Check đã processed chưa
  if (processedEvents.exists(e.eventId())) {
    return;  // Skip
  }
  inventory.reduceStock(
    e.productId(), e.quantity());
  // Mark processed (atomic với business logic)
  processedEvents.markProcessed(e.eventId());
}
@Repository
public class ProcessedEventStore {
    @Autowired private JdbcTemplate jdbc;

    public boolean exists(String eventId) {
        // Unique constraint on event_id → INSERT fail = duplicate
        Integer count = jdbc.queryForObject(
            "SELECT COUNT(*) FROM processed_events WHERE event_id = ?",
            Integer.class, eventId);
        return count != null && count > 0;
    }

    public void markProcessed(String eventId) {
        jdbc.update(
            "INSERT INTO processed_events(event_id, processed_at) VALUES (?, NOW())",
            eventId);
    }
}

-- Table schema
CREATE TABLE processed_events (
    event_id VARCHAR(64) PRIMARY KEY,  -- Unique constraint → atomic dedup
    processed_at TIMESTAMP NOT NULL,
    INDEX idx_processed_at (processed_at)  -- Để cleanup old entries
);
🎯 Vì sao dedupe + business logic phải atomic (cùng transaction)?

Nếu tách:

  1. Business logic commit (stock - 1)
  2. Crash trước khi markProcessed
  3. Kafka redelivery → check exists() = false → process lại → stock - 1 lần nữa

Đặt cả 2 trong cùng @Transactional: hoặc cả 2 commit (next message safe), hoặc cả 2 rollback (next retry sẽ xử lý đúng).

Event Schema Evolution

Events store mãi mãi. Nhưng schema thay đổi theo business:

// Version 1 (year 1)
public record OrderPlacedEvent(
    String orderId,
    String customerId,
    Money total
) {}

// Version 2 (year 2) — thêm promo tracking
public record OrderPlacedEvent(
    String orderId,
    String customerId,
    Money total,
    String promoCode,     // NEW field
    Money discountAmount  // NEW field
) {}

// Replay event V1 với V2 handler → NPE on promoCode!
🎯 Strategies cho schema evolution
  1. Additive only: chỉ thêm field optional, default null/empty. Đừng rename, remove, change type.
  2. Version field: mỗi event có field version. Handler check version, migrate trước khi process.
  3. Upcasting: load event V1 → upcast thành V2 với default values trong loader → handler chỉ xử lý V2.
  4. Format-agnostic: dùng JSON với optional fields thay vì rigid schemas. Trade-off: kích thước lớn hơn, validation weaker.
@Component
public class EventUpcaster {
    public OrderPlacedEvent upcast(JsonNode raw) {
        int version = raw.has("version") ? raw.get("version").asInt() : 1;
        if (version == 1) {
            // V1 → V2: thêm promo defaults
            return new OrderPlacedEvent(
                raw.get("orderId").asText(),
                raw.get("customerId").asText(),
                deserialize(raw.get("total"), Money.class),
                null,  // promoCode default
                Money.ZERO  // discountAmount default
            );
        }
        return deserialize(raw, OrderPlacedEvent.class);
    }
}

Choreography vs Orchestration

Đã cover trong #17. Tóm tắt nhanh:

🎭 Choreography (Event-driven)

Mỗi service publish event + listen events khác. Không có coordinator central. Flow emerge từ event chains.

Pros: loose coupling, scale tự nhiên, autonomous services
Cons: flow implicit, debug khó, dễ tạo circular event dependencies

🎬 Orchestration (Central coordinator)

1 orchestrator (Camunda BPMN) call services theo workflow định trước. Compensation logic explicit.

Pros: flow visible, debug dễ, compensation explicit
Cons: orchestrator coupling với mọi service, có thể bottleneck

Event Store Interface (Domain) + JPA Impl

// Domain layer — interface only
public interface EventStore {
    void appendEvents(String aggregateId, List<DomainEvent> events, long expectedVersion);
    List<DomainEvent> getEvents(String aggregateId);
    List<DomainEvent> getEventsSince(Instant timestamp);
    EventStreamWithSnapshot getEventStream(String aggregateId);
}

// Infrastructure layer — JPA impl
@Repository
public class JpaEventStore implements EventStore {
    private final EventJpaRepository jpaRepo;
    private final SnapshotRepository snapshots;
    private final ObjectMapper json;

    @Override
    @Transactional
    public void appendEvents(String aggregateId, List<DomainEvent> events, long expectedVersion) {
        // Optimistic concurrency — check current version
        long currentVersion = jpaRepo.maxSequenceForAggregate(aggregateId);
        if (currentVersion != expectedVersion) {
            throw new ConcurrencyException(
                "Aggregate " + aggregateId + " modified concurrently. Expected version "
                + expectedVersion + " but got " + currentVersion);
        }

        long nextSeq = currentVersion + 1;
        for (DomainEvent event : events) {
            EventJpaEntity entity = new EventJpaEntity();
            entity.setAggregateId(aggregateId);
            entity.setEventType(event.getClass().getSimpleName());
            entity.setSequence(nextSeq++);
            entity.setOccurredAt(event.getOccurredAt());
            entity.setEventData(serialize(event));
            jpaRepo.save(entity);
        }
    }

    @Override
    public List<DomainEvent> getEvents(String aggregateId) {
        return jpaRepo.findByAggregateIdOrderBySequenceAsc(aggregateId).stream()
            .map(this::deserialize)
            .toList();
    }

    private String serialize(DomainEvent e) {
        try { return json.writeValueAsString(e); }
        catch (JsonProcessingException ex) { throw new RuntimeException(ex); }
    }

    private DomainEvent deserialize(EventJpaEntity e) {
        try {
            Class<? extends DomainEvent> clazz = EventTypeRegistry.lookup(e.getEventType());
            return json.readValue(e.getEventData(), clazz);
        } catch (IOException ex) { throw new RuntimeException(ex); }
    }
}

Reconstruct Aggregate Từ Events

public class Order {
    private OrderId orderId;
    private OrderStatus status;
    private Money total;
    private List<DomainEvent> uncommittedEvents = new ArrayList<>();

    // Replay constructor
    public static Order rehydrateFromEvents(List<DomainEvent> events) {
        Order order = new Order();
        for (DomainEvent event : events) {
            order.apply(event);  // Apply each event to rebuild state
        }
        return order;
    }

    private void apply(DomainEvent event) {
        if (event instanceof OrderPlacedEvent e) {
            this.orderId = e.getOrderId();
            this.status = OrderStatus.PLACED;
            this.total = e.getTotal();
        } else if (event instanceof OrderShippedEvent e) {
            this.status = OrderStatus.SHIPPED;
        } else if (event instanceof OrderDeliveredEvent e) {
            this.status = OrderStatus.DELIVERED;
        } else if (event instanceof OrderCancelledEvent e) {
            this.status = OrderStatus.CANCELLED;
        }
    }

    // Business method — record events
    public void ship() {
        if (status != OrderStatus.PLACED) {
            throw new IllegalStateException("Cannot ship order in status " + status);
        }
        OrderShippedEvent event = new OrderShippedEvent(orderId, Instant.now());
        apply(event);  // Update state
        uncommittedEvents.add(event);  // Track for persist
    }

    public List<DomainEvent> pullUncommittedEvents() {
        var events = new ArrayList<>(uncommittedEvents);
        uncommittedEvents.clear();
        return events;
    }
}

4. Outbox Pattern — Guarantee Event Delivery

Tình huống: lưu Order vào DB + publish event lên Kafka. 2 operations khác system — không có distributed transaction. Có thể xảy ra:

  1. DB commit thành công
  2. Kafka publish fail (network)
  3. Order saved, nhưng downstream services không biết → inventory không trừ, email không gửi
@Service
public class OrderService {
    private final OrderRepository orderRepo;
    private final OutboxEventRepository outboxRepo;

    @Transactional  // Same DB transaction
    public OrderId placeOrder(PlaceOrderCommand cmd) {
        Order order = Order.create(cmd);
        orderRepo.save(order);  // Save business data

        // Save event vào outbox table cùng transaction
        OutboxEvent event = new OutboxEvent(
            UUID.randomUUID().toString(),
            "OrderPlacedEvent",
            serializeJson(new OrderPlacedEvent(order.getId(), ...)),
            Instant.now(),
            "PENDING"
        );
        outboxRepo.save(event);

        return order.getId();
    }
}

@Component
public class OutboxPoller {
    private final OutboxEventRepository outboxRepo;
    private final KafkaTemplate<String, String> kafka;

    @Scheduled(fixedDelay = 1000)  // Poll mỗi 1s
    public void publishPending() {
        List<OutboxEvent> pending = outboxRepo.findTop100ByStatus("PENDING");
        for (OutboxEvent event : pending) {
            try {
                kafka.send("order-events", event.getPayload()).get();
                event.markPublished();
                outboxRepo.save(event);
            } catch (Exception e) {
                log.error("Failed to publish event {}, will retry", event.getId(), e);
                // Để retry lần sau
            }
        }
    }
}

CREATE TABLE outbox_events (
    id VARCHAR(64) PRIMARY KEY,
    event_type VARCHAR(100) NOT NULL,
    payload TEXT NOT NULL,
    created_at TIMESTAMP NOT NULL,
    status VARCHAR(20) NOT NULL,  -- PENDING / PUBLISHED
    published_at TIMESTAMP,
    INDEX idx_status_created (status, created_at)
);
🎯 Vì sao Outbox Pattern critical?
  • Atomic: business data + event lưu cùng transaction. Hoặc cả 2 commit hoặc cả 2 rollback. Không có "data saved but event lost".
  • Reliable delivery: poller retry liên tục cho đến khi Kafka publish thành công. Crash giữa chừng? Restart, poller tiếp tục.
  • Decoupling from Kafka availability: Kafka down 10 phút → service vẫn ship orders. Outbox events tích trong table, publish khi Kafka recover.
  • Modern alternative: Debezium (CDC) đọc binlog của DB → publish lên Kafka. Không cần app code thực hiện outbox table.

5. Anti-Patterns

#1 Event đặt tên như command

// ❌ SendEmailEvent, MakePaymentEvent — sounds like intent
// Consumer process tưởng đã success, nhưng có thể chưa
class SendEmailEvent { ... }  // Đã gửi chưa?

// ✅ Past tense
class EmailSentEvent { ... }  // Đã gửi rồi
class PaymentReceivedEvent { ... }

#2 No schema evolution strategy

Add field mới mà không có version → replay events cũ NPE. Fix: version field + upcaster (xem phần 2).

#3 Large event payloads

// ❌ Lưu entire entity state vào event
class OrderUpdatedEvent {
    Order fullOrderState;  // 100 fields, 10KB JSON
}
// 1M events/day × 10KB = 10GB/day storage

// ✅ Chỉ lưu changed fields
class OrderShippedEvent {
    OrderId orderId;
    String trackingNumber;
    Instant shippedAt;
}

#4 Missing idempotency

Kafka at-least-once delivery → consumer phải dedupe. Đã cover trong phần 2.

#5 Coupling consumer với producer's event schema

// ❌ Consumer dùng SAME class với producer
// Producer service đổi field → consumer break
@KafkaListener(topics = "orders")
public void on(OrderPlacedEvent event) { ... }
// Producer + consumer dùng cùng OrderPlacedEvent class qua shared library

// ✅ Consumer có DTO riêng, deserialize từ JSON
class OrderPlacedEventDto {
    String orderId;
    String customerId;
    // Chỉ field consumer cần
}
@KafkaListener(topics = "orders")
public void on(OrderPlacedEventDto event) { ... }

#6 Synchronous event consumers

Consumer chạy slow business logic sync → block consumer thread → Kafka lag. Fix: consumer chỉ enqueue task, worker thread async process. Hoặc tăng partition + consumer instances.

#7 Not testing replay

Event sourcing nhưng chưa từng test replay. Production disaster recovery cần replay → fail. Fix: integration test replay 10K events, verify state correct.

6. Interview Framework

Tầng 1 — SurfaceMid gate
Q: Giải thích Event Sourcing. Vì sao lưu events thay vì state?
"Event Sourcing lưu chuỗi immutable events thay vì current state. State hiện tại = replay tất cả events. Benefits: full audit trail built-in, temporal queries (state at any time T), debugging bằng replay, có thể tạo view mới từ event history. Costs: storage tích lũy, query phức tạp hơn (cần projection), eventual consistency với read model."
Q: Idempotent consumer là gì, vì sao cần?
"Idempotent = process cùng message N lần kết quả giống nhau. Cần vì messaging system (Kafka, RabbitMQ) thường guarantee at-least-once delivery — event có thể deliver nhiều lần (network retry). Implementation: track processed event IDs trong DB/Redis với unique constraint. Critical: deduplication check + business logic trong cùng transaction để atomic — nếu không, crash giữa chừng vẫn có duplicate processing."
Tầng 2 — Deep DiveMid → Senior
Q: Design event store cho sàn TMĐT 50M events/năm. Query "order status lúc T" trong < 100ms?
"Combine 3 patterns:

1. Event store: append-only events table với index trên (aggregate_id, sequence). Partition theo created_at month để cleanup old data.

2. Snapshot pattern: snapshot order state mỗi 50 events. Query "state at T" = load snapshot ≤ T + replay events between snapshot và T (~10-50 events, fast).

3. Projection (read model): maintain materialized view order_current_state updated từ events. Query "current status of order X" → direct read (1ms). Chỉ dùng event replay cho historical queries.

Storage: PostgreSQL với BRIN index trên timestamp cho range query efficient. Archive events > 2 năm sang S3 + Athena cho compliance queries (chấp nhận slower).

Hot path query: projection (current state) — < 10ms. Cold path (historical "state at T"): snapshot + replay — < 100ms cho recent, vài seconds cho old."
Q: Schema evolution: event V1 {amount, currency}, V2 thêm {fee}. Replay V1 events thế nào?
"3 strategies:

1. Upcasting (recommended): trong event loader, detect version, transform V1 → V2 với default values (fee=0). Handler chỉ xử lý V2. Loader có version → version mapping rules.

2. Versioned handlers: 2 handlers cho 2 versions. Loader route đến đúng handler. Code phức tạp hơn nhưng explicit.

3. Forward migration: 1-time job replay tất cả V1 events, ghi lại thành V2 events vào event store. Sau đó chỉ có V2 events. Risk: huge ops on production data, không reversible.

Recommendation: upcasting cho live system, forward migration chỉ khi cần thiết và có backup.

Test critical: replay test trong CI — load 10K events V1, verify rehydrate state matches expected. Catch regression sớm."
Tầng 3 — ArchitectureSenior
Q: Design event-driven order processing cho sàn TMĐT. OrderService publish event, các services (Inventory, Payment, Notification, Analytics) consume. Đảm bảo no event loss + idempotency + replay capability.
"Architecture stack:

1. Producer side — Outbox Pattern: OrderService save order + outbox event trong cùng DB transaction. OutboxPoller scheduled job đọc outbox → publish Kafka → mark as published. Guarantee: nếu order saved, event eventually published (no loss).

2. Transport — Kafka: 1 topic order-events với partition theo orderId (orderings preserved per order). At-least-once delivery. Retention 30 ngày cho replay.

3. Consumer side — Idempotent + Dedupe: mỗi consumer service có processed_events(event_id PRIMARY KEY) table. Listener check exists → skip nếu có, process + insert trong cùng transaction.

4. Schema management: Schema Registry (Confluent) cho Avro/Protobuf. Backward compatible: thêm field optional, không break consumer cũ. Forward compatible: consumer ignore unknown fields.

5. Dead Letter Queue: consumer fail process 3 retries → publish vào order-events-dlq topic. Engineer manually inspect + reprocess.

6. Replay capability: Kafka retention 30 ngày + event store (PostgreSQL) cho long-term. Replay: reset consumer offset, hoặc dump event store → republish.

7. Observability: metric per consumer (lag, processing rate, error rate). Alert nếu consumer lag > 1000 messages hoặc error rate > 1%.

8. Audit + Compliance: event store là source of truth. GDPR deletion = encrypt event payload with user key, delete key (crypto-shredding) thay vì delete event row → preserve append-only property.

Total: 8 components nhưng standard pattern stack trong production. Mọi cloud (AWS, GCP, Azure) đều có managed equivalents."
