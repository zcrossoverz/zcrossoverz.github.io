#10 - Spring WebFlux: Reactive Programming, Reactor, Backpressure

WebFlux không phải "Spring MVC nhưng nhanh hơn". Nó là mô hình non-blocking, event-loop và reactive streams. Dùng đúng thì rất hợp cho API gateway, streaming, fan-out I/O. Dùng sai với blocking JDBC hoặc .block() thì còn khó debug hơn MVC.

Trọng tâm
I/O concurrency
Ví dụ
Travel search aggregator
Bẫy
block trong event loop
Liên quan
#1 Async, #22 Resilience, #20 Kafka

WebFlux Giải Quyết Bài Toán Thread Bị Block

Spring MVC dùng thread-per-request. Nếu request chủ yếu chờ network, thread ngồi chờ phần lớn thời gian. WebFlux dùng event loop và non-blocking I/O: thread gửi request đi, trả lại event loop, rồi xử lý tiếp khi response về.

Điểm quan trọng: WebFlux chỉ có lợi khi đường đi chính không blocking: WebClient, R2DBC, reactive Redis/Kafka. Nếu vẫn gọi JPA/JDBC/blocking SDK trực tiếp trên event loop, bạn phá mô hình.

Kiến Trúc Học Chống "Hiểu Giả"

1

Không đồng nhất async

Mono không giống CompletableFuture; nó lazy và có backpressure với Flux.

2

Thread awareness

Luôn biết operator nào chạy trên event loop, operator nào switch sang scheduler khác.

3

No manual subscribe

Service trả Mono/Flux; framework hoặc caller compose/subscribe.

4

Know when not to use

CRUD đơn giản với JPA blocking thường dùng MVC tốt hơn và dễ maintain hơn.

1. Bài Toán Thực Tế - Travel Search API Nghẽn Thread

API search du lịch phải gọi 4 provider: flight, hotel, promotion, weather. Spring MVC hiện tại gọi tuần tự, mỗi request mất khoảng 900ms, trong đó CPU thực sự làm việc chỉ vài chục ms.

@RestController
class TravelSearchController {
    @GetMapping("/api/search")
    public SearchResult search(SearchRequest request) {
        Flights flights = flightClient.search(request);     // 300ms blocking
        Hotels hotels = hotelClient.search(request);        // 250ms blocking
        Promotions promos = promotionClient.find(request);  // 180ms blocking
        Weather weather = weatherClient.forecast(request);  // 120ms blocking
        return SearchResult.of(flights, hotels, promos, weather);
    }
}
Tomcat threads: 200/200 active
Queue length: 1200
P99 latency: 11s
CPU: 35% nhưng request vẫn timeout vì thread đang chờ network.

2. Mental Model

2.1 Blocking vs Non-blocking I/O

WebFlux thắng khi request chủ yếu là I/O wait và concurrency cao.

2.2 Mono và Flux

TypeSố itemVí dụGhi nhớ
Mono<T>0 hoặc 1Find one hotel, call one APIGiống Optional async + lazy.
Flux<T>0 đến NStream search results, notificationsCó thể infinite, có backpressure.

Mono/Flux lazy: không chạy cho tới khi có subscriber. Trong controller WebFlux, framework subscribe khi gửi response.

2.3 Backpressure

Backpressure giải quyết producer nhanh hơn consumer. Subscriber nói request(n), publisher chỉ gửi số lượng được yêu cầu thay vì đẩy vô hạn vào memory.

StrategyHành viDùng khi
BUFFERGiữ item trong bufferKhông muốn mất data, volume có giới hạn.
DROPBỏ item khi consumer chậmRealtime UI, dữ liệu cũ không còn giá trị.
LATESTGiữ item mới nhấtProgress/status stream.
ERRORFail nhanh khi overflowMuốn bảo vệ hệ thống khỏi overload.

2.4 Schedulers, subscribeOn, publishOn

subscribeOn ảnh hưởng source subscription; thường dùng khi bọc blocking source. publishOn chuyển scheduler cho các operator phía sau nó.

Mono.fromCallable(() -> legacyBlockingClient.call())
    .subscribeOn(Schedulers.boundedElastic())
    .map(this::toDto);

Rule sống còn: không chạy JDBC, file I/O, Thread.sleep, .block() trên Netty event loop.

2.5 Operator Map: Chọn Đúng Operator Là Nửa Trận

OperatorDùng khiBẫy
mapTransform sync từ T sang RKhông gọi async API bên trong rồi trả Mono.
flatMapTransform async T sang Mono/FluxConcurrency mặc định có thể cao, cần giới hạn nếu fan-out lớn.
concatMapGiữ thứ tự, xử lý lần lượtThroughput thấp hơn flatMap.
zipChạy nhiều Mono độc lập rồi combineNếu một nguồn fail, toàn zip fail nếu không fallback.
switchIfEmptyFallback khi emptyKhác với error fallback.
onErrorResumeFallback khi errorĐừng nuốt lỗi critical thành success giả.

2.6 Reactor Context: ThreadLocal Không Còn Là Câu Trả Lời Mặc Định

Reactive pipeline có thể đổi thread qua scheduler. Vì vậy correlation ID, tenant ID hoặc security metadata không nên dựa hoàn toàn vào ThreadLocal tự viết. Reactor cung cấp Context để truyền metadata theo subscriber.

Mono<SearchResult> result = searchService.search(request)
    .contextWrite(ctx -> ctx.put("correlationId", correlationId));

Mono<String> currentCorrelationId() {
    return Mono.deferContextual(ctx ->
        Mono.just(ctx.getOrDefault("correlationId", "unknown")));
}

Trong Spring Security reactive, security context cũng đi theo reactive context, không giống servlet stack dùng ThreadLocal đơn giản.

3. Production-Grade Implementation

3.1 Fan-out Song Song Với WebClient

@RestController
public class TravelSearchReactiveController {
    private final WebClient flightClient;
    private final WebClient hotelClient;
    private final WebClient promotionClient;
    private final WebClient weatherClient;

    @GetMapping("/api/search")
    public Mono<SearchResult> search(SearchRequest request) {
        Mono<Flights> flights = flightClient.get()
            .uri(uriBuilder -> uriBuilder.path("/flights").queryParam("to", request.to()).build())
            .retrieve()
            .bodyToMono(Flights.class)
            .timeout(Duration.ofMillis(600))
            .onErrorReturn(Flights.empty());

        Mono<Hotels> hotels = hotelClient.get()
            .uri("/hotels?city={city}", request.city())
            .retrieve()
            .bodyToMono(Hotels.class)
            .timeout(Duration.ofMillis(600))
            .onErrorReturn(Hotels.empty());

        Mono<Promotions> promos = promotionClient.get()
            .uri("/promotions")
            .retrieve()
            .bodyToMono(Promotions.class)
            .timeout(Duration.ofMillis(300))
            .onErrorReturn(Promotions.empty());

        Mono<Weather> weather = weatherClient.get()
            .uri("/forecast?city={city}", request.city())
            .retrieve()
            .bodyToMono(Weather.class)
            .timeout(Duration.ofMillis(300))
            .onErrorReturn(Weather.unavailable());

        return Mono.zip(flights, hotels, promos, weather)
            .map(tuple -> SearchResult.of(
                tuple.getT1(), tuple.getT2(), tuple.getT3(), tuple.getT4()));
    }
}

3.2 WebClient Production Config

@Configuration
public class WebClientConfig {
    @Bean
    WebClient travelWebClient(WebClient.Builder builder) {
        ConnectionProvider provider = ConnectionProvider.builder("travel-provider")
            .maxConnections(200)
            .pendingAcquireTimeout(Duration.ofSeconds(2))
            .maxIdleTime(Duration.ofSeconds(30))
            .build();

        HttpClient httpClient = HttpClient.create(provider)
            .option(ChannelOption.CONNECT_TIMEOUT_MILLIS, 2000)
            .responseTimeout(Duration.ofSeconds(5));

        return builder
            .clientConnector(new ReactorClientHttpConnector(httpClient))
            .filter(correlationIdFilter())
            .build();
    }
}

3.3 Blocking Legacy Call: Cách Bọc Tạm Thời

@Service
public class LegacyHotelAdapter {
    private final LegacyHotelSdk sdk;

    public Mono<Hotels> search(SearchRequest request) {
        return Mono.fromCallable(() -> sdk.search(request))
            .subscribeOn(Schedulers.boundedElastic())
            .timeout(Duration.ofSeconds(2));
    }
}

Đây là giải pháp chuyển tiếp. Nếu phần lớn stack vẫn blocking, MVC có thể là lựa chọn tốt hơn WebFlux.

3.4 Streaming Notification Với Server-Sent Events

@GetMapping(value = "/api/search/{searchId}/events",
            produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ServerSentEvent<SearchProgress>> streamProgress(@PathVariable String searchId) {
    return progressService.stream(searchId)
        .map(progress -> ServerSentEvent.builder(progress)
            .event("search-progress")
            .id(progress.sequence())
            .build())
        .onBackpressureLatest();
}

3.5 Controlled Concurrency Với flatMap

public Flux<ProviderResult> searchManyProviders(List<Provider> providers, SearchRequest request) {
    return Flux.fromIterable(providers)
        .flatMap(provider -> provider.search(request)
            .timeout(Duration.ofMillis(700))
            .onErrorResume(ex -> Mono.just(ProviderResult.failed(provider.name()))),
            8); // tối đa 8 provider chạy song song
}

3.6 Error Handling, Timeout, Retry Có Kiểm Soát

Reactive error là terminal signal. Nếu một source trong zip fail mà không fallback, cả pipeline fail. Với fan-out provider, thường cần timeout ngắn, fallback rõ, và retry chỉ cho lỗi tạm thời.

Mono<ProviderResult> callProvider(Provider provider, SearchRequest request) {
    return provider.search(request)
        .timeout(Duration.ofMillis(700))
        .retryWhen(Retry.backoff(2, Duration.ofMillis(100))
            .filter(this::isTransientError))
        .onErrorResume(ex -> Mono.just(ProviderResult.failed(provider.name())));
}
Retry không miễn phí

Retry nhân tải lên dependency. Luôn có giới hạn số lần, backoff, timeout tổng và chỉ retry lỗi transient. Không retry validation error hoặc 4xx business error.

3.7 Test Reactive Flow Với StepVerifier

Đừng test WebFlux bằng cách gọi .block() khắp nơi. StepVerifier giúp assert signal: next, complete, error, timeout, virtual time.

@Test
void shouldFallbackWhenProviderTimeout() {
    Mono<ProviderResult> result = service.callProvider(slowProvider, request);

    StepVerifier.create(result)
        .expectNextMatches(value -> value.status() == ProviderStatus.FAILED)
        .verifyComplete();
}

Với endpoint, dùng WebTestClient. Với operator/pipeline thuần Reactor, dùng StepVerifier. Với blocking detection trong test, có thể thêm BlockHound ở mức vừa đủ.

5. Trade-offs & Anti-Patterns

AP1

.block() Trong WebFlux

.block() giữ event-loop thread và có thể gây starvation/deadlock. Compose bằng flatMap, zip, map thay vì block.

AP2

Manual subscribe() Trong Service

Manual subscribe biến flow thành fire-and-forget, lỗi khó propagate, test khó. Service nên return Mono/Flux.

AP3

Dùng WebFlux Cho CRUD JPA Đơn Giản

JPA blocking làm mất lợi ích chính của WebFlux. Nếu app chủ yếu CRUD với load vừa phải, Spring MVC đơn giản và dễ maintain hơn.

6. Interview Framework

SurfaceMono khác Flux?

Mono emit 0 hoặc 1 item; Flux emit 0 đến N item. Cả hai lazy và async. Flux hỗ trợ backpressure cho stream nhiều item.

DeepsubscribeOn khác publishOn?

subscribeOn quyết định scheduler cho source subscription, thường dùng để bọc blocking source. publishOn chuyển scheduler cho các operator phía sau nó. Cần biết để không block event loop.

ArchitectureKhi nào chọn WebFlux thay MVC?

Chọn WebFlux khi service I/O-bound, concurrency cao, fan-out nhiều API, streaming/SSE/WebSocket, và dependency có driver reactive. Không chọn khi team chưa quen reactive, logic CPU-bound, hoặc stack chủ yếu JPA/blocking SDK.

Checklist Thành Thạo

  • Giải thích được vì sao blocking I/O lãng phí thread trong MVC.
  • Viết được flow Mono.zip gọi nhiều service song song với timeout/fallback.
  • Biết Mono/Flux lazy và không manual subscribe trong service.
  • Phân biệt subscribeOn, publishOn, boundedElastic, parallel.
  • Nêu được 3 trường hợp không nên dùng WebFlux.
  • Chọn đúng map, flatMap, concatMap, zip, onErrorResume theo use case.
  • Biết dùng Reactor Context cho metadata qua async boundary.
  • Viết được test reactive bằng StepVerifier thay vì block bừa.
