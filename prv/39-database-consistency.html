Database Consistency — CAP, MVCC, Isolation Anomalies

"READ COMMITTED tránh dirty read" là câu ai cũng thuộc — nhưng vì sao hai transaction cùng commit mà số dư vẫn sai? Vì sao REPEATABLE_READ của PostgreSQL khác hẳn SERIALIZABLE của Oracle dù tên giống? File này tách anomaly có tên cụ thể (lost update, write skew, read skew) khỏi cách nói mơ hồ "race condition", dạy MVCC tận xmin/xmax, trên domain ShopFlow — ví store-credit & tồn kho của sàn e-commerce.

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Cứ gặp lỗi đồng thời thì gọi là race condition, set SERIALIZABLE là xong."
Thực tế: Mỗi hiện tượng có tên riêng (Lost Update, Write Skew, Read Skew) và cách fix khác nhau. SERIALIZABLE fix được nhưng abort transaction khi conflict → không có @Retryable thì user nhận 500.
Tưởng hiểu: "REPEATABLE_READ ở PostgreSQL và Oracle giống nhau vì cùng tên SQL standard."
Thực tế: PostgreSQL REPEATABLE_READ = Snapshot Isolation; Oracle không có REPEATABLE_READ (nhảy thẳng SERIALIZABLE, mà bản chất cũng chỉ là SI). Tên giống, hành vi khác → bug khi đổi DB.
Tưởng hiểu: "CAP nói chọn 2 trong 3 (C, A, P)."
Thực tế: Phát biểu đúng: khi có partition (P) mới phải chọn giữa C và A. P không phải thứ "chọn" — nó là sự cố. Single-node DB không có P nên CAP không áp dụng; dùng PACELC để nói về cả lúc bình thường.

1. Bài toán thực tế — store credit trừ 1 lần dù dùng 2 lần CORE ★

ShopFlow cho khách dùng "store credit" (ví khuyến mãi) khi checkout. Mặc định PostgreSQL READ COMMITTED.

@Transactional(isolation = Isolation.READ_COMMITTED)
public void spendCredit(Long walletId, BigDecimal amount) {
    Wallet w = walletRepo.findById(walletId).orElseThrow();   // đọc balance
    if (w.getBalance().compareTo(amount) < 0)                 // kiểm tra đủ
        throw new InsufficientCreditException();
    w.setBalance(w.getBalance().subtract(amount));            // trừ
    walletRepo.save(w);
}

Khách double-click checkout → 2 request song song, mỗi cái dùng 500K. Ví có 600K. Kết quả: cả 2 thành công, ví âm 400K.

⏸ Pause & Predict — trước khi đọc lời giải
Viết ra giấy:
  1. Đây KHÔNG chỉ là "race condition" — nó có tên cụ thể trong lý thuyết DB. Tên là gì?
  2. READ COMMITTED "tránh dirty read" — vậy vì sao 2 transaction commit thành công mà balance vẫn sai? READ COMMITTED KHÔNG chặn anomaly nào ở đây?
  3. Đổi lên REPEATABLE_READ của PostgreSQL thì điều gì xảy ra với transaction thứ hai? User thấy gì nếu code không retry?
Đáp án: đây là Lost Update (mục 5); READ COMMITTED chỉ chặn dirty read, không chặn lost update; PostgreSQL REPEATABLE_READ sẽ báo "could not serialize access due to concurrent update" → cần retry (mục 4, 8). Đừng đọc tiếp khi chưa đặt đúng tên anomaly.

5 câu Senior phải trả lời

  1. Vì sao READ COMMITTED không đủ dù 2 transaction đều commit?
  2. Đổi lên REPEATABLE_READ có fix không? PostgreSQL và Oracle có cùng hành vi?
  3. MVCC giải quyết điều gì? Không dùng read lock thì hậu quả là gì?
  4. CAP có áp dụng cho single-node DB không? Vì sao interviewer vẫn hỏi?
  5. Write Skew là gì? (bài on-call doctor — lost-update pattern không giải được)

2. CAP & PACELC đúng cách SUPPLEMENTARY

Phát biểu đúng của CAP

Khi xảy ra network partition (P), hệ phân tán buộc chọn giữa Consistency (C) và Availability (A). Không có chuyện "chọn 2 trong 3" lúc bình thường — P là sự cố, không phải lựa chọn.

PACELC (đầy đủ hơn):
  if Partition: chọn Availability vs Consistency
  else (bình thường): chọn Latency vs Consistency

DynamoDB          : PA/EL  (ưu tiên A khi partition, L khi bình thường)
Spanner           : PC/EC  (ưu tiên C luôn, đánh đổi latency)
PostgreSQL 1 node : không có P → CAP không áp dụng
PostgreSQL sync replica : PC/EC

Linearizability: write X=5 xong lúc T → mọi read sau T thấy X=5 (hoặc mới hơn). Snapshot Isolation: đọc từ snapshot lúc transaction bắt đầu, write không block read (MVCC) — nhưng không linearizable. Eventual: rồi sẽ hội tụ, không hứa khi nào.

4. MVCC — PostgreSQL vs Oracle CORE ★

Ý tưởng: thay vì lock row khi write, tạo version mới. Reader đọc version cũ, writer tạo version mới → không ai block ai.

PostgreSQL: mỗi row ẩn 2 cột xmin (transaction tạo) và xmax (transaction xoá).

balance = 600K (xmin=50, xmax=NULL → visible)
TX1 update → 100K:
   row cũ: 600K, xmin=50, xmax=100   (đánh dấu xoá bởi TX1)
   row mới: 100K, xmin=100, xmax=NULL
TX1 commit.
TX2 (REPEATABLE_READ, snapshot cũ) update dựa trên 600K
   → PostgreSQL: "ERROR: could not serialize access due to concurrent update" → retry

L1 — Biết tên

Biết "MVCC = nhiều version, không lock đọc". Symptom hiểu giả: không giải thích được vì sao PostgreSQL cần VACUUM hay Oracle bị ORA-01555.

L2 — Explain được

PostgreSQL giữ version cũ trong table (xmin/xmax) → cần VACUUM dọn dead tuple. Oracle update in-place, version cũ vào UNDO segment → ORA-01555 "snapshot too old" khi UNDO bị ghi đè quá sớm.

L3 — Apply được

PostgreSQL: tune autovacuum, monitor table bloat, tránh long transaction giữ snapshot. Oracle: tune UNDO_RETENTION, kích thước UNDO tablespace. Đọc đúng error để biết chỉnh chỗ nào.

6. Isolation levels theo vendor CORE ★

Level (SQL standard)DirtyNon-RepeatablePhantom
READ UNCOMMITTED❌❌❌
READ COMMITTED✅❌❌
REPEATABLE READ✅✅❌
SERIALIZABLE✅✅✅
3 gotcha vendor

(1) Oracle SERIALIZABLE thực chất là Snapshot Isolation → vẫn có Write Skew. (2) PostgreSQL SERIALIZABLE là SSI (Serializable Snapshot Isolation) → true serializability, detect conflict lúc commit. (3) MySQL REPEATABLE_READ (default) dùng gap lock nên chặn được phantom phần lớn trường hợp — khác SQL standard. Lưu ý: không level nào tự chặn Lost Update — cần explicit lock.

7. Write Skew — anomaly Snapshot Isolation không fix CORE ★

-- Invariant: luôn ≥ 1 bác sĩ on-call. Hiện có Alice + Bob.
-- TX1 (Alice off):              -- TX2 (Bob off) chạy đồng thời:
SELECT count(*) ... on_duty=true; -- =2  | SELECT count(*) ...; -- =2 (snapshot cũ)
-- count>=2 OK                          | -- count>=2 OK
UPDATE ... SET on_duty=false WHERE name='Alice'; | UPDATE ... name='Bob';
COMMIT;                                | COMMIT;
-- KẾT QUẢ: không ai on-call → vi phạm invariant

Vì sao SI không bắt: hai TX update disjoint rows (Alice vs Bob) → không write-write conflict cùng row; mỗi TX đọc snapshot riêng thấy điều kiện thoả. SI chỉ detect conflict trên cùng row.

🔧 Counter-example — khi nâng isolation KHÔNG phải cách đúng

Phản xạ "thấy lỗi đồng thời → set SERIALIZABLE toàn bộ service" là sai hướng:

  • SSI abort khi conflict: không bọc @Retryable → transaction bị abort → user nhận 500 ngẫu nhiên.
  • Overhead: SSI phải track predicate lock → chậm đáng kể dưới tải cao.
  • Thường chỉ cần lock đúng chỗ: Lost Update fix bằng SELECT FOR UPDATE với READ COMMITTED là đủ và deterministic hơn.

Cách đúng: chọn isolation tối thiểu cần; materialize conflict bằng SELECT FOR UPDATE trên row "đại diện" (vd advisory lock theo shift_id) cho Write Skew.

8. Optimistic vs Pessimistic lock CORE ★

Optimistic (@Version) — contention thấp

@Version private Long version;
// UPDATE ... SET balance=?, version=version+1
//   WHERE id=? AND version=?
// row count=0 → OptimisticLockException → @Retryable

Pessimistic (FOR UPDATE) — contention cao

@Lock(LockModeType.PESSIMISTIC_WRITE)
// SELECT * FROM wallet WHERE id=? FOR UPDATE
// row bị lock đến hết TX; lock theo MIN(id),MAX(id) tránh deadlock
Deadlock symptom + fix

Triệu chứng: log deadlock detected, transaction bị DB hủy ngẫu nhiên dưới tải. Nguyên nhân: TX1 lock A rồi xin B; TX2 lock B rồi xin A. Fix: luôn lock theo thứ tự cố định (vd MIN(id) trước MAX(id)).

9. Idempotency key — chống duplicate request SUPPLEMENTARY

Kể cả SERIALIZABLE, client submit 2 lần (timeout, double-click) vẫn có thể tạo 2 kết quả. Idempotency key (unique constraint) giải quyết — và phải nằm cùng transaction với thao tác chính, nếu không crash giữa chừng tạo key mồ côi.

try {
    idempotencyRepo.saveAndFlush(new IdempotencyKey(key)); // INSERT unique
} catch (DataIntegrityViolationException dup) {
    return existingResultFor(key);                          // đã xử lý → trả kết quả cũ
}
Order o = place(req);                                       // cùng transaction
link(key, o.getId());

🎯 3-tier interview framework

Tầng 1 — SurfaceMid gate

Q: 4 isolation level chặn anomaly nào? Optimistic vs pessimistic?

RU: không chặn gì; RC: chặn dirty read; RR: + non-repeatable; SER: + phantom/write skew. Không level nào tự chặn lost update. Optimistic = không lock, detect lúc commit, cần retry, contention thấp; Pessimistic = lock khi đọc, contention cao, rủi ro deadlock nếu sai order.

Tầng 2 — Deep DiveSenior gate

Q: Vì sao PostgreSQL REPEATABLE_READ không chặn Write Skew?

RR = Snapshot Isolation, chỉ detect write-write conflict cùng row. Write Skew ghi disjoint rows (Alice vs Bob) sau khi đọc overlapping → SI không thấy conflict → cả hai commit, phá invariant. Fix: SERIALIZABLE (SSI predicate lock) hoặc SELECT FOR UPDATE materialize conflict.

Tầng 3 — ArchitectureSenior+

Q: Thiết kế hệ trừ tồn kho/ví 10K TPS, no lost update, no write skew.

Shard theo wallet_id (~1K TPS/shard, 10 shard). Per-shard: READ COMMITTED + SELECT FOR UPDATE, lock theo id order. Cross-shard (chuyển A shard1 → B shard3): không lock 2 shard 1 TX → Saga + Outbox (debit + outbox event → credit + ack → compensation nếu fail). Monitor: p99 lock wait, deadlock count, retry rate.

🔧 Lab bắt buộc — reproduce anomaly (2 bài)

Lab 1 — Lost Update tận mắt. Mở 2 psql session, cùng BEGIN với READ COMMITTED, cùng đọc balance=600, cả hai UPDATE balance=balance-500 WHERE id=1 không dùng lock (đọc trước rồi set giá trị tính tay trong app), commit. Quan sát balance sai. Lặp lại với SELECT ... FOR UPDATE → TX2 block và đọc giá trị mới → fail đúng logic. Tiêu chí: gọi đúng tên anomaly và giải thích FOR UPDATE sửa thế nào.

Lab 2 — Write Skew + SSI. Tạo bảng oncall(name, on_duty) 2 dòng true. Hai session REPEATABLE_READ chạy kịch bản on-call ở mục 7 → cả hai commit, count=0 (bug). Đổi sang SERIALIZABLE → một transaction bị "could not serialize", phải retry. Tiêu chí: chỉ ra vì sao RR không bắt mà SSI bắt.
