Spring Core — IoC, DI, AOP, Bean Lifecycle & Scope (Từ cơ chế đến production bug)

Spring Core là nền tảng của mọi thứ trong Spring ecosystem — Spring Boot auto-config, Spring Data JPA, Spring Security, Spring Transaction đều đứng trên 4 trụ: IoC container, Dependency Injection, AOP proxy, Bean lifecycle. 100% phỏng vấn Spring hỏi 4 trụ này, nhưng bẫy nằm ở cơ chế bên trong: vì sao @Transactional trên private method không chạy? 1. Vì sao Spring Core là gốc? CORE

Nhiều junior Spring developer nghĩ "biết viết @Service @Autowired là đủ". Sai. Hiểu Spring ở mức "API surface" sẽ fail mọi bug production có dấu vết của cơ chế bên trong: AOP proxy, bean lifecycle, scope. File này tồn tại để chuyển bạn từ "user của Spring" thành "người hiểu Spring vận hành".

📜 Lịch sử Spring Framework — vì sao IoC ra đời

  • 1990s – early 2000s — Java EE (J2EE) phổ biến mô hình EJB (Enterprise JavaBeans). EJB heavy-weight, đòi phải implement nhiều interface (SessionBean, EntityBean), deploy vào application server (WebLogic, WebSphere). Unit test gần như không thể vì code phụ thuộc vào container.
  • 2002 — Rod Johnson xuất bản Expert One-on-One J2EE Design and Development (Wiley). Cuốn sách chỉ trích EJB và đề xuất "lightweight container" với code DI. Code mẫu trong sách chính là phôi của Spring Framework.
  • 2004 — Spring 1.0 release. POJOs + IoC + AOP, không cần EJB container. Triết lý "Don't repeat yourself" và "Convention over configuration".
  • 2006 — Spring 2.0: AspectJ integration, annotation-driven config (@Transactional, @Component).
  • 2009 — Spring 3.0: Java config (@Configuration), REST support, native annotation-driven.
  • 2013 — Spring Boot 1.0 launch tại SpringOne. Auto-configuration + embedded server + opinionated defaults. Spring Boot làm Spring trở thành de-facto standard cho Java backend.
  • 2017 — Spring 5.0: Reactive (WebFlux), Kotlin support, Java 8 baseline.
  • 2022 — Spring 6.0 + Spring Boot 3.0: Java 17 baseline, AOT (Ahead-of-Time) compilation cho GraalVM, Jakarta EE 9+ (đổi javax.* → jakarta.*).
  • 2024–2026 — Spring 6.1+: virtual threads support, observability improvements. Spring vẫn dominate Java backend với 60%+ market share (theo JetBrains Developer Ecosystem Report).

Verifiable: Rod Johnson — Expert One-on-One J2EE, ISBN 978-0764543852. Spring source — github.com/spring-projects/spring-framework. Release notes — spring.io/blog. JetBrains report — jetbrains.com/lp/devecosystem-2024.

"User của Spring" vs "Hiểu Spring"

❌ Chỉ là user

  • Biết viết @Service @Autowired
  • Copy template Spring Boot từ Stack Overflow
  • Bug "@Transactional không rollback" → Google → copy answer → vẫn không hiểu vì sao
  • Khi senior hỏi "AOP proxy hoạt động thế nào?" → "Cứ đặt @Transactional là chạy thôi"
  • Singleton service có mutable field, lâu lâu data leak → debug 2 ngày không ra

✅ Hiểu Spring

  • Vẽ được Bean lifecycle 7 bước, biết AOP proxy tạo ở bước 4
  • Bug "@Transactional private method" → reflex giải được trong 30 giây
  • Debug bằng cách inspect proxy class name → biết AOP chạy hay không
  • Stateless singleton là habit, không phải rule học thuộc
  • Self-invocation problem → reflex 3 cách fix + trade-off

⏸️ Pause & Predict

Trước khi đọc tiếp: nếu đặt @Transactional trên một private method, transaction có hoạt động không? Vì sao? Viết câu trả lời + cơ chế cụ thể. Đáp án ở cuối Section 7.

2. Bug e-contract — 4 hiểu lầm trong 1 service CORE

Team e-contract báo cáo: chữ ký số đôi khi bị ký trùng, nhưng không phải lúc nào cũng xảy ra. Production rồi, đã trên prod 2 tuần. Sau 2 ngày debug, developer tìm thấy đoạn code này:

@Service
public class SigningService {

    // ⚠️ Lỗi #1: Developer nghĩ đây là stateless service
    private List<String> signedDocuments = new ArrayList<>();
    private int requestCount = 0;

    public SignResult sign(SignRequest request) {
        requestCount++;
        signedDocuments.add(request.getDocumentId()); // Accumulates across requests!

        if (signedDocuments.contains(request.getDocumentId())) {
            // Logic này LUÔN true sau request đầu tiên
            return SignResult.duplicate();
        }
        return doActualSigning(request);
    }
}

@RestController
public class ContractController {

    // ⚠️ Lỗi #2: @Autowired field injection trên controller
    @Autowired
    private SigningService signingService;

    // ⚠️ Lỗi #3: AOP không hoạt động như mong đợi
    @AuditLog  // Custom annotation
    public SignResult signContract(@RequestBody SignRequest request) {
        return signingService.sign(request);
    }
}

// ⚠️ Lỗi #4: @Transactional trên private method
@Service
public class DocumentService {
    @Transactional
    private void saveDocumentInternal(Document doc) { // Không bao giờ có transaction!
        documentRepository.save(doc);
    }
}

❌ 4 hiểu lầm trong 50 dòng code

  1. Singleton scope là default → signedDocuments được SHARE giữa mọi request → User A ký xong, User B sign → thấy documentIds của User A → race condition + data leak.
  2. Field injection bằng @Autowired trực tiếp → field không thể final → mutable → khó test (phải dùng reflection), khó detect circular dependency, dependencies "hidden".
  3. AOP self-invocation → khi gọi signingService.sign() ngon, nhưng nếu trong SigningService có method khác gọi this.sign() → bypass proxy → @AuditLog không trigger.
  4. @Transactional trên private method → CGLIB không thể override private method → proxy không intercept được → transaction KHÔNG BAO GIỜ active → exception không rollback.

4 câu hỏi buộc bạn tư duy

  1. Tại sao signedDocuments accumulate qua các requests? Singleton scope hoạt động thế nào? (→ Section 6)
  2. @Transactional trên private method có hoạt động không? Tại sao liên quan đến AOP? (→ Section 7)
  3. @AuditLog AOP annotation trên method trong cùng class có hoạt động không? (→ Section 8)
  4. Field injection (@Autowired trực tiếp) có vấn đề gì so với constructor injection? (→ Section 4)
Chi phí thực: Bug này trên production 2 tuần, ~200 hợp đồng bị "ký trùng signal" trả về cho user dù chưa ký xong. Team phải migrate data, gửi mail xin lỗi 200 khách hàng, audit log lại toàn bộ chữ ký. 4 hiểu lầm × 50 dòng code = ~80 giờ remediation. File này tồn tại để bạn KHÔNG mắc phải.

3. IoC Container — "Hollywood Principle" CORE

Vì sao CORE: Mọi thứ trong Spring đứng trên IoC container. Hiểu container = hiểu thứ tự bean được tạo + khi nào dependencies sẵn sàng + tại sao AOP proxy tồn tại. Không hiểu = chỉ là user.

IoC = "Don't call us, we'll call you"

IoC (Inversion of Control) là nguyên tắc thiết kế: control flow bị đảo ngược — framework gọi code của bạn, không phải bạn gọi framework. DI (Dependency Injection) là implementation pattern của IoC.

❌ Truyền thống (bạn kiểm soát)

public class OrderService {
  private PaymentService ps;

  public OrderService() {
    // Bạn tự tạo dependency
    this.ps = new PaymentService();
    // → Phải biết PaymentService
    //   được tạo thế nào
    // → Tight coupling
    // → Khó test (không mock được)
  }
}

✅ IoC (container kiểm soát)

public class OrderService {
  private final PaymentService ps;

  // Container inject vào
  public OrderService(PaymentService ps) {
    this.ps = ps;
  }
  // → OrderService chỉ biết
  //   interface, không biết cách tạo
  // → Loose coupling
  // → Dễ test (mock PaymentService)
}

ApplicationContext — Cấu trúc thực sự

ApplicationContext
├── BeanFactory (core: create, store, retrieve beans)
│   ├── BeanDefinitionRegistry (đăng ký metadata của bean)
│   ├── DefaultListableBeanFactory (implementation chính)
│   └── Bean instances (singleton cache)
│
├── MessageSource (i18n)
├── ApplicationEventPublisher (event system)
└── ResourceLoader (file, classpath, URL)

Spring Boot start — 7 bước cụ thể

1. Scan classpath → tìm @Component, @Service, @Repository, @Controller
2. Đọc @Configuration classes → build BeanDefinition objects (metadata, CHƯA tạo instance)
3. Sort BeanDefinitions theo dependency order (topological sort)
4. Instantiate beans theo thứ tự
5. Inject dependencies (constructor → setter → field)
6. Chạy BeanPostProcessors (★ AOP proxies được tạo ở đây!)
7. Chạy @PostConstruct → ApplicationContextEvent.REFRESHED → READY
Key insight về bước 6: AOP proxy KHÔNG được tạo ở constructor — nó được tạo ở BeanPostProcessor.postProcessAfterInitialization(). Đây là lý do trong constructor, this = target object (chưa wrap proxy), KHÔNG phải proxy. Đó cũng là lý do bạn KHÔNG nên gọi @Transactional method từ constructor.

BeanDefinition vs Bean instance — khái niệm phân biệt

Junior thường confuse 2 khái niệm này:

  • BeanDefinition = metadata mô tả cách tạo bean (class, scope, init-method, dependencies). Được register ở bước 2. 1 BeanDefinition có thể tạo nhiều instance (prototype scope).
  • Bean instance = object thực tế trong memory. Singleton → 1 instance / context. Prototype → N instance.

Hiểu phân biệt này quan trọng khi:

  • Auto-configuration (Spring Boot) → register BeanDefinition, KHÔNG tạo instance ngay.
  • Conditional bean (@ConditionalOnProperty) → quyết định ở bước 2 (register hay không), không phải runtime.
  • Lazy initialization (@Lazy) → có BeanDefinition nhưng instance tạo khi lần đầu inject/getBean.

🔨 Thử Phá Vỡ: IoC có lỗ hổng nào?

  • Magic quá nhiều — junior không biết bean được tạo từ đâu. Khi debug bug "NoSuchBeanDefinitionException" → mất 30 phút tìm vì sao class không scan được. Fix: bật --debug mode để xem auto-config report.
  • Circular dependency dễ tạo ra — A inject B, B inject A → khi constructor injection thì fail-fast (good), khi field injection thì late-fail (bad).
  • Performance overhead start-up — scan classpath + reflection chậm. Spring Boot 3 có AOT để giảm xuống ~10× nhưng cấu hình phức tạp.
  • Test harness phức tạp — full @SpringBootTest mất 10–30 giây. Cần @WebMvcTest, @DataJpaTest để scoped test — nhưng phải hiểu rõ slice annotations.

4. DI — 3 cách inject, chỉ 1 đúng CORE

Vì sao CORE: 90% junior Spring developer dùng field injection — sai. Constructor injection là best practice không phải vì style mà vì 5 lý do kỹ thuật cụ thể: immutability, testability, fail-fast, explicit dependencies, circular detection.

Constructor injection — chuẩn vàng

@Service
@Slf4j
public class PaymentService {

    // final → immutable sau khi inject → thread-safe
    private final PaymentGatewayClient gatewayClient;
    private final TransactionRepository transactionRepository;
    private final FeeCalculator feeCalculator;
    private final MeterRegistry meterRegistry;
    private final ApplicationEventPublisher events;

    // @Autowired optional từ Spring 4.3+ khi chỉ có 1 constructor
    public PaymentService(
            PaymentGatewayClient gatewayClient,
            TransactionRepository transactionRepository,
            FeeCalculator feeCalculator,
            MeterRegistry meterRegistry,
            ApplicationEventPublisher events) {
        // Validate critical dependencies ngay trong constructor
        this.gatewayClient = Objects.requireNonNull(gatewayClient, "gatewayClient required");
        this.transactionRepository = Objects.requireNonNull(transactionRepository);
        this.feeCalculator = Objects.requireNonNull(feeCalculator);
        this.meterRegistry = Objects.requireNonNull(meterRegistry);
        this.events = Objects.requireNonNull(events);
    }
}

Lombok shortcut — chuẩn production

@Service
@RequiredArgsConstructor  // Lombok tạo constructor cho mọi final field
public class PaymentService {
    private final PaymentGatewayClient gatewayClient;
    private final TransactionRepository transactionRepository;
    // ... không cần viết constructor thủ công
}
Pitfall Lombok: @RequiredArgsConstructor không inject Objects.requireNonNull. Nếu cần validate null, hoặc viết constructor thủ công, hoặc dùng @NonNull trên field — Lombok sẽ generate null check trong constructor.

Field injection — vì sao tệ

@Service
public class BadPaymentService {
    @Autowired
    private PaymentGatewayClient gatewayClient; // Field injection

    // Vấn đề 1: Không thể mark final → mutable → thread-safety risk
    // Vấn đề 2: Unit test phải dùng reflection để inject mock:
    //          ReflectionTestUtils.setField(service, "gatewayClient", mock);
    // Vấn đề 3: Circular dependency khó detect — Spring "magic" làm nó silent
    // Vấn đề 4: Dependencies hidden — không nhìn thấy class cần gì
    // Vấn đề 5: SonarQube / IntelliJ inspections warn "field injection is discouraged"
}

Setter injection — chỉ khi optional dependency

Setter injection có chỗ đứng narrow: khi dependency thực sự optional. Ví dụ:

@Service
public class CacheService {
    private final RedisClient redis;
    private CacheMetrics metrics; // Optional — chỉ enable khi observability bật

    public CacheService(RedisClient redis) {
        this.redis = redis;
    }

    @Autowired(required = false)
    public void setMetrics(CacheMetrics metrics) {
        this.metrics = metrics;
    }
}

Trong 95% trường hợp, dùng constructor + @ConditionalOnProperty sạch hơn.

5. Bean Lifecycle — 7 bước từ sinh đến chết CORE

Vì sao CORE: Biết đúng vị trí của AOP proxy + @PostConstruct + Aware interfaces = debug được mọi bug "tại sao dependency null khi constructor?" và "tại sao proxy chưa được tạo?". Đây là lifecycle bí mật bên dưới mỗi @Service.

7 bước chuẩn — vẽ ra giấy

[1] Instantiation
    → Constructor được gọi
    → Chỉ có constructor injection available ở đây
    → this = target object thực, KHÔNG phải proxy

[2] Populate Properties
    → @Autowired fields được inject (sau constructor)
    → @Value được resolve

[3] BeanNameAware, BeanFactoryAware, ApplicationContextAware (nếu implement)
    → Bean biết tên của mình trong container
    → Hiếm dùng, chỉ cho framework code

[4] BeanPostProcessor.postProcessBeforeInitialization()
    → Hook cho framework

[5] @PostConstruct / InitializingBean.afterPropertiesSet() / init-method
    → Custom init logic
    → Validate config, warm up cache, establish connections

[6] BeanPostProcessor.postProcessAfterInitialization()
    → ★ AOP proxy tạo ở đây ★
    → Bean được wrap thành proxy nếu có advice
    → Sau bước này, bean trong container = proxy
    → Singleton cache lưu PROXY, không phải target

[7] Bean READY — đưa vào singleton cache, ApplicationContext trả bean này cho mọi @Autowired

--- runtime ---

[8] ApplicationContext.close() / @PreDestroy
    → Cleanup: đóng connection, flush buffer, cancel scheduled tasks
    → Sau đó: DisposableBean.destroy() / destroy-method
Bí mật bước 6: Vì proxy được tạo SAU @PostConstruct, nếu trong @PostConstruct bạn gọi this.someMethod() → gọi vào target, KHÔNG qua proxy → @Transactional / @Async / @Cacheable trong @PostConstruct sẽ KHÔNG hoạt động. Đây là gotcha rất hay gặp.

Constructor vs @PostConstruct — khi nào dùng cái nào

❌ Init logic trong constructor

@Service
public class CoreBankingConnector {
  private final ConnectionPool pool;
  private Connection conn;

  public CoreBankingConnector(ConnectionPool pool) {
    this.pool = pool;
    // ❌ I/O trong constructor:
    //   - dependencies có thể chưa
    //     hoàn tất setup
    //   - Test chậm (full I/O cho
    //     mỗi @Autowired)
    //   - Khó debug fail-fast
    this.conn = pool.acquire();
    if (!conn.ping()) throw ...;
  }
}

✅ Init logic trong @PostConstruct

@Service
@Slf4j
public class CoreBankingConnector {
  private final ConnectionPool pool;
  private Connection conn;

  public CoreBankingConnector(
      ConnectionPool pool) {
    this.pool = Objects.requireNonNull(pool);
    // Chỉ assign, không I/O
  }

  @PostConstruct
  public void initConnection() {
    log.info("Init connection...");
    this.conn = pool.acquire();
    if (!conn.ping()) {
      throw new IllegalStateException(
        "Cannot connect to core banking");
    }
    log.info("Connection established");
  }

  @PreDestroy
  public void close() {
    if (conn != null) conn.close();
  }
}

3 cách hook lifecycle — phân biệt

CáchInitDestroyPros / Cons
JSR-250 annotation @PostConstruct @PreDestroy ✅ Khuyến nghị. Standard Java EE. Spring 6.0+ Jakarta package: jakarta.annotation.PostConstruct.
Spring interface InitializingBean.afterPropertiesSet() DisposableBean.destroy() ⚠️ Couple với Spring. Chỉ dùng cho framework code.
XML / @Bean config init-method="..." destroy-method="..." Cho 3rd-party bean không thể annotate. Ví dụ: HikariDataSource có close() method.

⚠️ Edge cases lifecycle phải nhớ

  1. @PostConstruct trên method có override-able. Subclass override → BaseService init chạy method của subclass khi subclass CHƯA fully initialized → state corruption. Fix: @PostConstruct trên private method.
  2. @PostConstruct gọi @Transactional / @Async của chính mình. Bypass proxy (xem Section 7). Fix: inject self qua @Lazy hoặc tách service.
  3. @PreDestroy không chạy khi JVM crash. Cleanup logic KHÔNG được đảm bảo. Production: dùng external cleanup (k8s preStop hook) cộng @PreDestroy.
  4. Prototype scope KHÔNG có @PreDestroy. Spring không track lifecycle của prototype sau khi tạo. Phải tự cleanup hoặc dùng DestructionAwareBeanPostProcessor.
  5. Order khi có nhiều @PostConstruct trong inheritance. Parent @PostConstruct chạy TRƯỚC child. Order trong same class: không guarantee — chỉ dùng 1 @PostConstruct per class.
  6. Exception trong @PostConstruct. Bean fails to start → ApplicationContext startup fails → app không boot. Có thể cố ý "fail-fast" để catch config sai.
  7. @PostConstruct trên @Bean method. KHÔNG được — @Bean trong @Configuration không support @PostConstruct trực tiếp. Phải dùng @Bean(initMethod = "init").

6. Bean Scope — Vì sao Singleton sinh bug CORE

Vì sao CORE: 90% bug "data leak giữa users" / "race condition trong service" đều từ hiểu sai scope. Default = singleton — nghĩa là 1 instance share giữa tất cả request, tất cả thread. Mọi mutable field = bug tiềm năng.

5 scope chuẩn

ScopeLifecycleUse caseThread-safe?
singleton (DEFAULT)1 instance / containerService, Repository, Controller⚠️ PHẢI stateless
prototype1 instance / injection pointStateful helper (builder, parser)✅ Mỗi caller có instance riêng
request1 instance / HTTP requestRequestContext, current user✅ Per-request isolation
session1 instance / HTTP sessionShopping cart, multi-step form⚠️ Multi-tab có thể race
application1 instance / ServletContextHiếm dùng. Web-only.⚠️ Như singleton nhưng scope ServletContext

Vì sao bug Singleton ở Section 2?

SigningService là @Service → default singleton scope.

→ 1 instance duy nhất trong ApplicationContext.
→ Mỗi HTTP request đến /sign endpoint:
   Tomcat thread A: signingService.sign(reqA)
   Tomcat thread B: signingService.sign(reqB)  ← cùng instance!
   Tomcat thread C: signingService.sign(reqC)  ← cùng instance!

→ signedDocuments field = SHARED giữa A, B, C.
→ Thread A add docId_A. Thread B đọc list → thấy docId_A!
   → B return "duplicate" cho document chưa từng ký.
→ requestCount++ không atomic → race condition.

Đây là bug DATA LEAK + CONCURRENCY trong 5 dòng code.

Quy tắc cứng cho Singleton

  1. Không mutable field. Mọi state per-request phải là local variable.
  2. Dependencies = final. Inject 1 lần, dùng mãi.
  3. Configuration = final + @Value. Immutable từ properties.
  4. Caches phải là thread-safe collection. ConcurrentHashMap, không phải HashMap.
  5. Counters phải là atomic. AtomicLong, không phải long.

7. AOP — Proxy mechanism, không phải magic HIGH-SIGNAL

Vì sao HIGH-SIGNAL: AOP là kỹ thuật phân biệt mid vs senior. Senior debug được "@Transactional silent fail" trong 30 giây vì hiểu proxy. Junior tốn 2 ngày. Đây là điểm câu hỏi yêu thích của interviewer Spring.

AOP hoạt động qua PROXY

AOP trong Spring không phải bytecode manipulation (như AspectJ compile-time weaving). Nó tạo proxy object wraps bean của bạn:

Bạn inject:                Thực ra bạn nhận:
────────────               ─────────────────────────────────
PaymentService       →     PaymentService$$SpringCGLIB$$0
                           (subclass được tạo runtime bởi CGLIB
                            hoặc JDK Dynamic Proxy)

Khi bạn gọi:
  paymentService.charge(100)

Thực ra:
  proxy.charge(100)
    → before advice (logging, security check, tx begin)
    → TARGET.charge(100)  ← method thực tế
    → after advice (logging, tx commit/rollback)

2 loại proxy — phân biệt

LoạiKhi nào dùngGiới hạn
JDK Dynamic Proxy Bean implement interface → proxy implement cùng interface Bean PHẢI có interface. Inject phải qua interface (không qua impl class).
CGLIB Proxy Default từ Spring Boot 2+. Subclass của target class. Không proxy được: final class, final method, private method. Cần constructor không có business logic.

Default trong Spring Boot 2+ là CGLIB (vì developer hay quên tạo interface). Có thể force JDK qua spring.aop.proxy-target-class=false.

3 điều kiện làm @Transactional / @Async / @Cacheable SILENT FAIL

❌ Điều kiện 1: private method

@Service
public class OrderService {
    @Transactional
    private void saveOrder(Order order) { // KHÔNG BAO GIỜ có transaction!
        orderRepository.save(order);
        // Nếu exception → dữ liệu vẫn bị save (không có rollback)
    }
}

Cơ chế: CGLIB tạo subclass và override method để intercept. Nhưng private method KHÔNG được override (Java rule). Method bị bypass proxy → annotation ignored. Không có warning — Spring không thể detect ở compile-time.

❌ Điều kiện 2: final method / final class

@Service
public final class FinalService {  // ❌ CGLIB không thể subclass
    @Transactional
    public void process() { ... }   // Annotation silent ignored
}

Cơ chế: CGLIB subclass không thể override final method. Class final = không thể subclass. Application start sẽ có warning log nhưng dễ miss.

❌ Điều kiện 3: Self-invocation

@Service
public class TransactionService {

    @Transactional
    public void processPayment(Payment payment) {
        validatePayment(payment);
        savePayment(payment);
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void auditLog(Payment payment) {
        // ❌ KHÔNG có transaction mới khi gọi qua self-invocation
    }

    public void processWithAudit(Payment payment) {
        processPayment(payment);
        auditLog(payment); // ← Self-invocation: gọi this.auditLog(),
                          //   KHÔNG qua proxy → propagation NEW không trigger
    }
}

Cơ chế: Trong processWithAudit, this = target object (NOT proxy). Spring inject proxy vào field/constructor của caller, nhưng bên trong target object, this luôn là target. Method call this.auditLog() đi thẳng vào target, bypass proxy hoàn toàn.

📌 Đáp án Pause & Predict từ Section 1

Câu hỏi: Nếu đặt @Transactional trên private method, transaction có hoạt động không?
Câu trả lời: KHÔNG. Cơ chế: CGLIB proxy = subclass + override method. Java private method KHÔNG override được → CGLIB không intercept được → annotation ignored. Spring không log warning rõ ràng → bug silent. Fix: chuyển method thành public hoặc protected, hoặc move logic ra service khác.

Khi nào AspectJ thắng Spring AOP?

Spring AOP có giới hạn của proxy-based. Khi cần advice trong cùng class, advice trên private method, advice trên field access — phải dùng AspectJ với compile-time weaving (AspectJ Compiler) hoặc load-time weaving (LTW agent):

Spring AOP (proxy-based):
  ✅ Đơn giản: thêm @EnableAspectJAutoProxy
  ✅ Không cần build setup phức tạp
  ❌ Chỉ public method qua external call
  ❌ Self-invocation, final, private bypass

AspectJ (weaving):
  ✅ Mọi join point: private, final, constructor, field, self-invocation
  ✅ Performance tốt hơn (no proxy indirection)
  ❌ Build setup phức tạp: aspectjrt agent hoặc AJC compiler
  ❌ Debug stack trace khó hơn (woven code)
  ❌ Học cost cao

Quy tắc: 95% case Spring AOP đủ. Chỉ chuyển AspectJ khi self-invocation
unavoidable và refactor không khả thi.

8. Self-invocation problem — 3 cách fix HIGH-SIGNAL

Self-invocation là bug AOP phổ biến nhất. 3 cách fix dưới đây, theo thứ tự preferred:

✅ Cách 1 — Tách method ra class khác (PREFERRED)

Khi nào dùng: Default. Clean architecture luôn ưu tiên cách này.

@Service
public class AuditService {
    private final AuditRepository auditRepository;

    public AuditService(AuditRepository auditRepository) {
        this.auditRepository = auditRepository;
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void log(AuditEntry entry) {
        // Gọi từ class khác → qua proxy của AuditService → OK
        auditRepository.save(entry);
    }
}

@Service
public class PaymentServiceFixed {
    private final AuditService auditService; // Inject service khác

    public void process(Payment payment) {
        doProcess(payment);
        auditService.log(AuditEntry.from(payment)); // Qua proxy của AuditService → OK
    }
}

Pros: Clean separation. Tuân SRP. Dễ test.

Cons: Phải tạo thêm class — nhưng đây là feature, không phải bug.

⚠️ Cách 2 — Self-inject với @Lazy

Khi nào dùng: Khi không thể tách class (ví dụ: legacy code, refactor risk cao).

@Service
public class PaymentService {
    @Lazy // Tránh circular dependency khi self-inject
    @Autowired
    private PaymentService self; // Inject PROXY của chính mình

    public void process(Payment payment) {
        doProcess(payment);
        self.auditLog(payment); // Gọi qua proxy → AOP hoạt động
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void auditLog(Payment payment) { ... }
}

Pros: Không phải tách class.

Cons: Code smell — service inject chính nó. Reviewer ngạc nhiên. Test phức tạp.

❌ Cách 3 — AopContext.currentProxy() (AVOID)

Khi nào dùng: Hầu như không bao giờ. Chỉ cho framework code.

@EnableAspectJAutoProxy(exposeProxy = true) // Phải enable expose
@Configuration
public class AopConfig { }

@Service
public class PaymentService {
    public void process(Payment payment) {
        doProcess(payment);
        // Lấy proxy của chính mình từ ThreadLocal
        ((PaymentService) AopContext.currentProxy()).auditLog(payment);
    }
}

Cons: Khó đọc. Static call → khó test. Couple với Spring API. Nếu thấy code này trong PR, đề nghị refactor.

Pattern decision tree:
Self-invocation cần fix? → Có thể tách class không?
  Có → Cách 1 (PREFERRED, clean architecture).
  Không (legacy / hot path) → Cách 2 với comment giải thích.
  Vẫn không được → reconsider architecture. AspectJ weaving là last resort.

9. Production AOP — Custom audit log aspect CORE

Banking requirement: mọi thao tác tài chính phải có audit trail. Implement với custom annotation + AOP aspect:

@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface AuditLog {
    String action() default "";
    boolean logRequest() default true;
    boolean logResponse() default false; // Cẩn thận: response có thể chứa sensitive data
}

@Aspect
@Component
@Slf4j
public class AuditLogAspect {

    private final AuditRepository auditRepository;
    private final ObjectMapper objectMapper;

    public AuditLogAspect(AuditRepository auditRepository, ObjectMapper objectMapper) {
        this.auditRepository = auditRepository;
        this.objectMapper = objectMapper;
    }

    /**
     * @Around: wrap toàn bộ method execution.
     * Lý do chọn @Around thay vì @Before + @AfterReturning:
     * - Cần đo duration (cả thời điểm trước và sau)
     * - Cần xử lý exception nhất quán trong 1 chỗ
     * - Cần capture cả request và response
     */
    @Around("@annotation(auditLog)")
    public Object auditMethod(ProceedingJoinPoint joinPoint, AuditLog auditLog) throws Throwable {
        long startTime = System.currentTimeMillis();
        String methodName = joinPoint.getSignature().toShortString();
        String action = auditLog.action().isBlank() ? methodName : auditLog.action();
        String userId = getCurrentUserId();

        AuditEntry.Builder builder = AuditEntry.builder()
                .action(action)
                .userId(userId)
                .timestamp(Instant.now())
                .ipAddress(getClientIpAddress());

        if (auditLog.logRequest() && joinPoint.getArgs().length > 0) {
            try {
                String requestJson = objectMapper.writeValueAsString(joinPoint.getArgs()[0]);
                builder.request(maskSensitiveData(requestJson));
            } catch (JsonProcessingException e) {
                log.warn("Failed to serialize audit request for {}", methodName);
            }
        }

        Object result = null;
        String status = "SUCCESS";
        String errorMessage = null;

        try {
            result = joinPoint.proceed(); // ★ Gọi method thực tế
            return result;
        } catch (Throwable ex) {
            status = "FAILED";
            errorMessage = ex.getMessage();
            throw ex; // Re-throw để caller xử lý
        } finally {
            long duration = System.currentTimeMillis() - startTime;
            AuditEntry entry = builder
                    .status(status)
                    .errorMessage(errorMessage)
                    .durationMs(duration)
                    .build();
            auditRepository.saveAsync(entry); // Save async để không chậm main flow
            log.info("AUDIT: action={}, user={}, status={}, duration={}ms",
                     action, userId, status, duration);
        }
    }

    /**
     * @Pointcut: reusable expression để tái dụng
     */
    @Pointcut("within(com.banking.payment..*) || within(com.banking.contract..*)")
    public void bankingServiceLayer() {}

    @Before("bankingServiceLayer() && @annotation(org.springframework.transaction.annotation.Transactional)")
    public void logTransactionStart(JoinPoint jp) {
        log.debug("Transaction starting: {}", jp.getSignature().toShortString());
    }
}

10. Fix SigningService — stateless singleton CORE

@Service
@Slf4j
public class SigningServiceFixed {

    // ✅ Dependencies — inject một lần, dùng mãi (stateless, thread-safe)
    private final SignatureProvider signatureProvider;
    private final DocumentRepository documentRepository;
    private final IdempotencyService idempotencyService;

    // ✅ Configuration — immutable, inject từ properties
    @Value("${signing.timeout-seconds:30}")
    private int signingTimeoutSeconds;

    // ❌ KHÔNG CÓ: private List<String> signedDocuments, private int requestCount
    // State thuộc về REQUEST, không thuộc về SERVICE

    public SigningServiceFixed(
            SignatureProvider signatureProvider,
            DocumentRepository documentRepository,
            IdempotencyService idempotencyService) {
        this.signatureProvider = Objects.requireNonNull(signatureProvider);
        this.documentRepository = Objects.requireNonNull(documentRepository);
        this.idempotencyService = Objects.requireNonNull(idempotencyService);
    }

    @AuditLog(action = "SIGN_DOCUMENT", logRequest = true)
    @Transactional
    public SignResult sign(SignRequest request) {
        // State per-request: local variable trong stack → thread-safe by design
        String documentId = request.getDocumentId();
        String userId = request.getUserId();

        log.info("Signing request: docId={}, user={}", documentId, userId);

        // Idempotency check: dùng service riêng, không phải field của class
        if (idempotencyService.isDuplicate(request.getIdempotencyKey())) {
            return idempotencyService.getPreviousResult(request.getIdempotencyKey());
        }

        Document doc = documentRepository.findById(documentId)
                .orElseThrow(() -> new DocumentNotFoundException(documentId));

        SignResult result = signatureProvider.sign(doc, request.getCertificate());

        idempotencyService.saveResult(request.getIdempotencyKey(), result);
        return result;
    }
}
Mọi thứ đã đúng: Constructor injection (final fields) · Stateless (no mutable state) · @Transactional public method (proxy intercept OK) · @AuditLog gọi từ Controller external → qua proxy OK · Idempotency tách service riêng (thread-safe state managed externally, không phải field).

Test verify stateless

@SpringBootTest
class SigningServiceConcurrencyTest {

    @Autowired
    SigningServiceFixed signingService;

    @Test
    void shouldHandle100ConcurrentRequestsWithoutStateLeak() throws Exception {
        int requests = 100;
        ExecutorService executor = Executors.newFixedThreadPool(20);
        CountDownLatch latch = new CountDownLatch(requests);

        Set<String> resultDocIds = ConcurrentHashMap.newKeySet();

        for (int i = 0; i < requests; i++) {
            String docId = "doc-" + i;
            executor.submit(() -> {
                try {
                    SignRequest req = SignRequest.builder()
                            .documentId(docId)
                            .userId("user-" + Thread.currentThread().getId())
                            .idempotencyKey(UUID.randomUUID().toString())
                            .build();
                    SignResult r = signingService.sign(req);
                    resultDocIds.add(r.getDocumentId());
                } finally {
                    latch.countDown();
                }
            });
        }

        latch.await(10, TimeUnit.SECONDS);
        executor.shutdown();

        // Không có duplicate sai do state leak
        assertThat(resultDocIds).hasSize(requests);
    }
}

11. Khi nào THỰC SỰ cần prototype scope SUPP

90% bean nên là singleton. Prototype chỉ cần khi:

  • Stateful builder pattern: Mỗi caller cần object state riêng để accumulate (ReportBuilder, QueryBuilder).
  • Object KHÔNG thread-safe: Như SimpleDateFormat (Java 8+ dùng DateTimeFormatter thay), một số XML parser legacy.
  • Per-execution context: Workflow engine cần state per-flow-run.
@Component
@Scope("prototype") // Mỗi lần inject = instance mới
public class ReportBuilder {
    private final List<ReportSection> sections = new ArrayList<>(); // State per-build
    private String title;
    private LocalDate periodStart;
    private LocalDate periodEnd;

    public ReportBuilder addSection(ReportSection section) {
        sections.add(section);
        return this;
    }

    public ReportBuilder withTitle(String title) {
        this.title = title;
        return this;
    }

    public Report build() {
        return new Report(title, periodStart, periodEnd, sections);
    }
}
Phải nhớ: Prototype scope KHÔNG có @PreDestroy. Spring "release" reference ngay sau khi tạo → bạn phải tự cleanup nếu cần. Đó là trade-off chính.

⚠️ 7 anti-patterns kinh điển CORE

7 lỗi dưới đây chiếm ~80% bug Spring Core trong production. Memorize → tránh được 4 tuần debug.

❌ Anti-pattern #1: Stateful Singleton

@Service
public class FeeCalculator {
    private BigDecimal lastFee;     // Shared giữa mọi requests!
    private String lastCurrency;    // Data leak giữa users!

    public BigDecimal calculate(BigDecimal amount) {
        lastFee = amount.multiply(RATE);   // Thread A set 100
        lastCurrency = "VND";             // Thread B có thể thấy Thread A's lastFee
        return lastFee;
    }
}

Symptom: Data leak intermittent. Bug chỉ xảy ra dưới concurrent load.

Fix: Mọi state là local variable. Field chỉ chứa final dependencies + final config.

❌ Anti-pattern #2: @Transactional trên private/final method

@Service
public class OrderService {
    @Transactional
    private void saveOrder(Order order) { // KHÔNG BAO GIỜ có transaction!
        orderRepository.save(order);
    }
}

Symptom: Exception throw nhưng data vẫn save (no rollback).

Fix: Method PHẢI là public (hoặc protected nếu cần subclass override). KHÔNG final.

❌ Anti-pattern #3: Overuse AOP Pointcut

@Around("execution(* com.banking..*.*(..))")  // Mọi method trong package
public Object logEverything(ProceedingJoinPoint jp) throws Throwable { ... }

Symptom: Performance drop 30%+. Log spam. Có khi log cả toString(), equals(), hashCode().

Fix: Pointcut có target rõ ràng:

@Around("@annotation(com.banking.audit.AuditLog)")  // Chỉ method có annotation
@Around("within(@org.springframework.stereotype.Service *) && execution(public * *(..))")

❌ Anti-pattern #4: Circular Dependency qua Constructor

@Service
public class ServiceA {
    public ServiceA(ServiceB b) { ... } // A cần B
}
@Service
public class ServiceB {
    public ServiceB(ServiceA a) { ... } // B cần A → Circular!
}
// Spring ném: UnsatisfiedDependencyException

Symptom: App không boot. Stack trace mention "Circular dependency".

Fix: Circular dependency là design smell. Cách fix theo thứ tự ưu tiên:

  1. Tách dependency chung ra class C — A và B đều dùng C, không cần biết nhau.
  2. Dùng @Lazy trên 1 trong 2 → defer initialization (workaround, không phải fix).
  3. Refactor — tách thêm abstraction layer.

❌ Anti-pattern #5: @PostConstruct gọi Override-able Method

@Service
public class BaseService {
    @PostConstruct
    public void init() {
        loadConfiguration(); // loadConfiguration() có thể bị override!
    }

    public void loadConfiguration() { /* default */ }
}

@Service
public class ExtendedService extends BaseService {
    @Override
    public void loadConfiguration() {
        // Chạy trong @PostConstruct của BaseService
        // nhưng ExtendedService chưa được fully initialized!
        // → NullPointerException trên field của ExtendedService
    }
}

Fix: @PostConstruct gọi private method (không thể override).

❌ Anti-pattern #6: I/O trong Constructor

@Service
public class CacheService {
    public CacheService(RedisClient redis) {
        // ❌ I/O trong constructor:
        this.cache = redis.preload(); // Bean chưa fully constructed
        // Test chậm. Fail-fast khó debug.
    }
}

Fix: Constructor chỉ assign. I/O / init logic trong @PostConstruct.

❌ Anti-pattern #7: Inject ApplicationContext trực tiếp

@Service
public class GodService {
    @Autowired
    private ApplicationContext context; // ❌ Service Locator anti-pattern

    public void process() {
        SomeService s = context.getBean(SomeService.class); // ❌ Magic lookup
        s.doWork();
    }
}

Vì sao tệ: Hidden dependencies. Khó test. Tight coupling với Spring API.

Fix: Inject specific dependency qua constructor. Nếu cần lookup runtime, dùng Provider<T> hoặc Map<String, T> (Spring inject map of beans).

🔨 Thử Phá Vỡ: anti-pattern nào nguy hiểm nhất?

Câu trả lời: #1 (Stateful Singleton). Lý do:

  • Bug intermittent — chỉ xảy ra dưới concurrent load → khó reproduce.
  • Data leak qua users → security + compliance issue.
  • Test unit không catch được — chỉ stress test mới phát hiện.
  • Senior reviewer có thể miss khi review code (field nhìn "vô hại").

Đây là lý do quy tắc "Default scope = stateless" phải thành reflex, không phải checklist.

🎯 3-tier interview framework — Spring Core rubric

Spring Core gần như chắc chắn xuất hiện trong phỏng vấn Java backend. 3 tier dưới đây phân biệt junior / mid / senior signal.

Tier 1 — Surface (Junior signal)

Q1: IoC và DI khác nhau như thế nào?

Trả lời chuẩn: "IoC là nguyên tắc thiết kế: control flow bị đảo ngược — framework gọi code của bạn, không phải bạn gọi framework. DI là implementation pattern của IoC: dependencies được inject vào object thay vì object tự tạo. Spring Container là IoC container thực hiện DI. Nói cách khác: DI là cách Spring thực hiện IoC."

Q2: Singleton scope trong Spring khác Singleton design pattern thế nào?

Trả lời chuẩn: "Singleton design pattern: 1 instance per JVM (classloader). Spring singleton: 1 instance per ApplicationContext. Một JVM có thể có nhiều ApplicationContext (parent-child trong web app) → mỗi context có singleton riêng. Spring singleton cũng không enforce private constructor — developer có thể tạo thêm instance bên ngoài container."

Tier 2 — Deep Dive (Mid vs Senior)

Q3: Giải thích tại sao @Transactional không hoạt động khi gọi method trong cùng class. Cơ chế gì gây ra?

Framework trả lời:

"Tôi sẽ giải thích từ cơ chế proxy của Spring AOP.

Khi Spring thấy @Transactional trên một bean, nó không modify class của bạn. Thay vào đó, nó tạo một proxy object (CGLIB subclass hoặc JDK Dynamic Proxy) wraps bean của bạn. Proxy này intercept method calls để begin/commit/rollback transaction.

Khi external code inject OrderService, nó nhận proxy chứ không phải OrderService thực. Mọi method call → proxy → begin transaction → delegate sang target → commit/rollback.

Nhưng khi self-invocation: this.saveInternal() trong class — this là target object thực, không phải proxy. Call đi thẳng vào method, bypass proxy hoàn toàn → không có transaction.

Trade-off của approach proxy-based: đơn giản và không cần instrument bytecode, nhưng giới hạn: không hoạt động với private method, final method, self-invocation. Nếu cần AOP với self-invocation, dùng AspectJ compile-time weaving — không có limitation này nhưng phức tạp hơn nhiều.

Fix tôi thường dùng trong production: tách logic ra service khác, inject service đó vào. Clean architecture cũng recommend điều này."

Q4: Mô tả đầy đủ Bean lifecycle. Khi nào dùng @PostConstruct vs Constructor để init?

Framework trả lời:

"Constructor chạy đầu tiên — trước khi bất kỳ dependency nào được inject (nếu dùng field injection). Dùng constructor để: validate non-null dependencies, assign final fields. KHÔNG dùng để: gọi method phụ thuộc vào dependencies khác, thực hiện I/O.

@PostConstruct chạy sau khi tất cả dependencies đã inject. Đây là nơi đúng để: khởi tạo cache từ DB, establish external connections, validate business config, start background threads. Dùng constructor injection thì @PostConstruct có thể access tất cả dependencies an toàn.

Trade-off: Constructor init là fail-fast — lỗi xảy ra ngay khi create bean, stack trace rõ ràng hơn. @PostConstruct init có thể fail sau, khó trace hơn nhưng cần thiết khi init cần dependencies đã sẵn sàng.

Trong banking service của tôi: Constructor = validate required dependencies (không null), @PostConstruct = connect to core banking system, warm up caches. @PreDestroy = gracefully close connections và drain in-flight transactions."

Tier 3 — Architecture (Senior Level)

Q5: Thiết kế một plugin system cho banking platform — cho phép dynamically load/unload business rules (fee calculation) mà không cần restart service. Dùng Spring ApplicationContext.

Cách tiếp cận trong 5 phút phỏng vấn:

Phút 1 — Clarify & scope:
  • Rules thay đổi tần suất thế nào? (daily? realtime?)
  • Rules được define ở đâu? (code? DSL? DB config?)
  • Cần rollback nếu rule mới có lỗi?
  • Isolation: rule lỗi có ảnh hưởng service khác không?

Phút 2 — Core mechanism:
  Spring có ChildApplicationContext:
  Parent context: cố định, chứa core services
  Child context: dynamic, chứa fee rules

  Khi load rule mới:
  1. Compile/load rule class vào custom ClassLoader
  2. Tạo new child ApplicationContext với ClassLoader đó
  3. Register context với parent
  4. Atomically swap reference đến context mới

Phút 3 — Thread safety:
  AtomicReference<ApplicationContext> currentRuleContext
  → Reads không block (volatile reference)
  → Update: tạo context mới, CAS swap
  → Requests in-flight dùng context cũ đến khi complete
  → Shutdown old context sau grace period

Phút 4 — Error handling & rollback:
  Context mới được validate trước khi swap:
    - Smoke test các rule chính
    - Health check
  Nếu fail → giữ nguyên context cũ
  Lưu snapshot context cũ → rollback nếu cần

Phút 5 — Alternative & trade-offs:
  Alternative: Scripting engine (Groovy/MVEL/Drools)
  → không cần reload JVM
  Trade-off: ClassLoader leak nếu không close context cũ đúng cách
  Production: combine với feature flags để A/B test rules
  Simpler approach: load rules từ DB/config server
  → không cần dynamic ClassLoader

🔧 Build & drill — 3 lab biến hiểu thành reflex

🔨 Lab 1 — AOP Proxy Detective (2 giờ)

Mục tiêu: Tự mắt thấy proxy được tạo và cơ chế self-invocation bypass.

Steps:

  1. Tạo Spring Boot project mới.
  2. Tạo SimpleService với 2 method: publicMethod() annotate @Transactional, callPublicMethod() gọi this.publicMethod().
  3. Tạo SimpleAspect log "Before" và "After" cho publicMethod.
  4. Inject SimpleService vào Controller, gọi callPublicMethod().
  5. Quan sát log: aspect KHÔNG được trigger (self-invocation bypass).
  6. Gọi publicMethod() trực tiếp từ Controller → aspect TRIGGERED.
  7. Bật logging.level.org.springframework.aop=DEBUG → quan sát proxy class name $$SpringCGLIB$$0.

Acceptance: Mắt thấy 3 hiện tượng: (1) proxy class name, (2) self-invocation bypass, (3) external call trigger aspect.

🔨 Lab 2 — Singleton Race Condition Reproducer (3 giờ)

Mục tiêu: Tự mắt thấy data leak trong singleton service.

Steps:

  1. Tạo CounterService với field private int count = 0;, method increment() và getCount().
  2. Tạo Controller endpoint /increment gọi counterService.increment() và return getCount().
  3. Dùng ab hoặc wrk stress test: ab -c 50 -n 1000 http://localhost:8080/increment.
  4. So sánh: expected count = 1000, actual count thường < 1000 (race condition).
  5. Fix bằng AtomicInteger → re-run → count = 1000 exactly.
  6. Fix bằng synchronized → re-run → count = 1000 nhưng throughput thấp hơn.

Acceptance: Đo được throughput chênh lệch giữa int (sai), AtomicInteger (đúng, fast), synchronized (đúng, slow).

🔨 Lab 3 — Bean Lifecycle Debugger (2 giờ)

Mục tiêu: Vẽ được 7 bước lifecycle với data thực từ debugger.

Steps:

  1. Tạo LifecycleBean implement InitializingBean, DisposableBean, BeanNameAware.
  2. Thêm constructor (log "1. Constructor"), setter cho dependencies (log "2. Setter"), setBeanName (log "3. Aware"), @PostConstruct (log "4. PostConstruct"), afterPropertiesSet (log "5. InitializingBean"), @PreDestroy (log "6. PreDestroy"), destroy (log "7. DisposableBean").
  3. Tạo custom BeanPostProcessor in log trước/sau init.
  4. Run app, observe console log → xác định thứ tự thực sự.
  5. Đặt breakpoint trong từng method, inspect this để xem object state.
  6. Vẽ sequence diagram theo log observed.

Acceptance: Diagram khớp với 7 bước trong Section 5 (constructor → setter → aware → BPP before → init → BPP after → ready).

🧠 Feynman test — kết thúc file này

🎓 Tự test Feynman — giải thích cho junior chưa biết Spring

Đóng file, giải thích các câu sau cho 1 sinh viên năm 3 đã biết Java OOP nhưng chưa biết Spring:

  1. "Vì sao Spring tồn tại? Trước Spring, code Java EE thế nào?" — Trước Spring, code phải implement EJB interfaces, deploy vào application server. Test gần như không thể vì code phụ thuộc container. Spring làm "lightweight container": POJO + DI + AOP, không cần EJB → testable, đơn giản.
  2. "IoC khác DI thế nào?" — IoC là nguyên tắc thiết kế ("framework gọi code của bạn"). DI là implementation ("inject dependencies vào object"). DI là cách Spring thực hiện IoC.
  3. "Vì sao @Transactional trên private method KHÔNG chạy?" — Spring AOP qua proxy. CGLIB tạo subclass + override method. Java private method KHÔNG override được → proxy không intercept → annotation ignored. Silent bug.
  4. "Vì sao stateful singleton là bug?" — Singleton = 1 instance share giữa tất cả thread/request. Mutable field share = race condition + data leak giữa users. Fix: state là local variable, không phải field.
  5. "AOP proxy được tạo lúc nào trong lifecycle?" — Ở BeanPostProcessor.postProcessAfterInitialization() (bước 6), SAU @PostConstruct. Vì vậy gọi @Transactional method trong @PostConstruct của chính mình = bypass proxy → không có transaction.

Bí 1 câu → quay lại module tương ứng. Mượt cả 5 = đã internalize.
