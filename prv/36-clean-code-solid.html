Clean Code & SOLID — Smell, Refactor, và Khi Nào Đừng

SOLID là thứ ai cũng đọc thuộc 5 chữ cái nhưng ít người dùng đúng — và over-apply SOLID còn hại hơn không biết nó. File này dạy mỗi nguyên tắc qua WHY (vấn đề gì sinh ra nó), kèm counter-example (khi nào áp dụng là over-engineering), trên domain ShopFlow — OrderPricingService tính giá/khuyến mãi của sàn e-commerce.

🎯 Mục tiêu
Nhận diện smell, refactor có kiểm soát, biết khi nào KHÔNG abstract
📅 Thời lượng
1 tuần · Sau #16 Design Patterns, #15 Clean Arch
📚 Prereq
OOP · interface/polymorphism · unit test
🔥 Độ ưu tiên
🟠 Cao — code review & design hỏi mỗi vòng senior

⚠️ Cảnh báo "hiểu giả" — 3 cái bẫy

Tưởng hiểu: "Thuộc 5 chữ S-O-L-I-D và định nghĩa mỗi cái là hiểu SOLID."
Thực tế: Hiểu thật = nhìn một đoạn code chỉ ra nó vi phạm nguyên tắc nào và hậu quả cụ thể khi thêm feature. Đọc định nghĩa mà không nói được "vi phạm O thì sao" = hiểu giả.
Tưởng hiểu: "Càng nhiều interface, abstraction, design pattern thì code càng SOLID, càng tốt."
Thực tế: Abstraction sớm/thừa là code smell ngược: 1 implementation mà có interface, factory, strategy = đọc 5 file để hiểu 1 việc. SOLID phục vụ thay đổi; chỗ không đổi thì abstraction chỉ thêm nhiễu.
Tưởng hiểu: "Refactor là dọn code cho đẹp, làm lúc nào cũng được."
Thực tế: Refactor KHÔNG đổi hành vi — nên bắt buộc có test phủ trước khi đổi. Refactor không test = viết lại + cầu may. Đây là khác biệt giữa refactoring và "rewrite".

🗺️ Roadmap — nếu chỉ có X giờ thì học gì

Thứ tự ưu tiên
1 giờ (CORE ★): SRP + OCP (3) — hai nguyên tắc dùng nhiều nhất; nhận diện 7 code smell (2). Đủ để pass phần "tìm lỗi SOLID" của interview.
+1 giờ (CORE ★): Refactor messy → clean theo bước có test (6), Tell-Don't-Ask (5), khi nào KHÔNG áp dụng SOLID (anti-patterns).
+1 giờ (SUPPLEMENTARY): LSP, ISP, DIP với ví dụ; Extract Method/Class/Parameter Object.
Khi cần (ADVANCED): Replace conditional with polymorphism, đánh giá ROI abstraction theo "rule of three".

1. Bài toán thực tế — PR tính giá 200 dòng CORE ★

Bạn review một PR: method calculateOrderTotal() dài 200 dòng, nhận 7 tham số, có magic number 0.05, 0.1 rải khắp, trộn validate + tính phí + ghi DB + gửi email trong một chỗ. Mỗi lần thêm một loại khuyến mãi phải sửa đúng method này và làm vỡ 3 chỗ khác.

⏸ Pause & Predict — trước khi đọc lời giải
Nhìn mô tả PR trên, viết ra giấy:
  1. Liệt kê đúng tên ít nhất 4 code smell trong đoạn này (không nói chung "code xấu").
  2. "Thêm 1 loại khuyến mãi phải sửa method này" — đây vi phạm nguyên tắc SOLID nào? Hậu quả khi có 10 loại khuyến mãi?
  3. Nếu refactor, bước ĐẦU TIÊN phải làm là gì trước khi đụng vào code (gợi ý: không phải tách method)?
Đáp án: Long Method, Long Parameter List, Magic Number, Mixed Concerns (mục 2); vi phạm OCP (sửa thay vì mở rộng) + SRP (mục 3); bước đầu là viết test phủ hành vi hiện tại (anti-pattern "refactor không test"). Đừng đọc tiếp khi chưa đặt tên được 4 smell.

4 câu thinking

  1. Code smell nào đang có và mỗi cái gây hại gì khi maintain?
  2. Vi phạm nguyên tắc SOLID nào? Thêm feature thì vỡ ở đâu?
  3. Refactor theo bước nào để KHÔNG đổi hành vi?
  4. Khi nào dừng — abstraction tới đâu là đủ, đâu là over-engineering?

3. SOLID — WHY từng nguyên tắc CORE ★

S — Single Responsibility

WHY: một class chỉ nên có một lý do để thay đổi. OrderPricingService vừa tính giá vừa gửi email → bộ phận marketing đổi email làm vỡ logic tính tiền. Tách ra: pricing đổi vì rule giá, notifier đổi vì kênh gửi.

O — Open/Closed

WHY: mở để mở rộng, đóng để sửa đổi. Thêm loại khuyến mãi mới không nên sửa if/else khổng lồ (rủi ro vỡ cái cũ) mà thêm một DiscountStrategy mới.

interface DiscountRule { BigDecimal apply(Order o); }
// Thêm "BlackFridayRule" = thêm 1 class mới, KHÔNG sửa code cũ
List<DiscountRule> rules;   // inject; pricing chỉ lặp qua rules

L — Liskov Substitution

WHY: subclass phải thay được superclass mà không phá kỳ vọng. Cổ điển: Square extends Rectangle ghi đè setter làm vỡ bất biến diện tích. Trong e-commerce: FreeShippingOrder extends Order mà ném exception ở getShippingFee() = vi phạm LSP.

I — Interface Segregation

WHY: đừng ép client phụ thuộc method nó không dùng. Một interface OrderRepository 30 method khiến test phải mock cả 30. Tách thành interface nhỏ theo nhu cầu (read vs write).

D — Dependency Inversion

WHY: module cấp cao không phụ thuộc cấp thấp, cả hai phụ thuộc abstraction. PricingService phụ thuộc interface TaxProvider, không phụ thuộc VietnamVatApiClient cụ thể → đổi nhà cung cấp/test dễ.

L1 — Biết tên

Đọc thuộc 5 chữ S-O-L-I-D và định nghĩa. Symptom hiểu giả: nhìn code không chỉ ra được nó vi phạm cái nào.

L2 — Explain được

Cho một đoạn code, chỉ ra nguyên tắc bị vi phạm + hậu quả cụ thể khi thêm feature (vd "vi phạm OCP → mỗi khuyến mãi mới sửa if-else → vỡ cũ").

L3 — Apply được

Refactor có kiểm soát: viết test trước, tách theo SRP/OCP, và biết dừng đúng lúc — không abstract chỗ chỉ có 1 implementation. Cân nhắc ROI mỗi abstraction.

4. Refactoring techniques SUPPLEMENTARY

  • Extract Method: tách một khối thành method có tên rõ → method gốc đọc như mục lục.
  • Extract Class: nhóm field/method liên quan ra class riêng (tách God Class).
  • Replace Magic Number with Constant: 0.08 → VAT_RATE.
  • Replace Conditional with Polymorphism: switch(type) dài → mỗi nhánh thành một subclass/strategy.
  • Introduce Parameter Object: 7 tham số → một PricingRequest.

5. Tell-Don't-Ask SUPPLEMENTARY

❌ Ask — lôi data ra ngoài xử lý

if (order.getStatus() == PAID
    && order.getItems().size() > 0) {
    order.setStatus(SHIPPED);
}

✅ Tell — giao việc cho object giữ data

order.markShipped();
// invariant (đã PAID, có item) nằm TRONG Order
// → không thể quên check ở nơi khác

6. Refactor messy → clean (có test) CORE ★

Quy trình an toàn cho method calculateOrderTotal() 200 dòng:

  1. Phủ test trước — capture hành vi hiện tại (kể cả bug) bằng characterization test.
  2. Introduce Parameter Object — gom 7 tham số thành PricingRequest.
  3. Extract + Strategy — mỗi loại khuyến mãi thành DiscountRule (OCP).
  4. Extract Validation — tách validate ra OrderValidator (SRP).
  5. Tách side-effect — ghi DB và gửi email ra khỏi hàm tính (notification async).
  6. Orchestrator (Facade) — một service mỏng nối các phần; chạy lại test xanh sau mỗi bước.
Nguyên tắc vàng

Mỗi bước nhỏ + chạy test ngay. Test xanh sau mỗi commit = bạn luôn có điểm quay lui. Đây là khác biệt giữa "refactor" (an toàn) và "rewrite" (rủi ro).

⚠️ Khi SOLID phản tác dụng CORE ★

🔧 Counter-example — khi nào KHÔNG áp dụng SOLID/abstraction

Over-engineering hại không kém dưới-engineering:

  • Premature abstraction: tạo interface + factory cho cái chỉ có một implementation và chưa có dấu hiệu sẽ có cái thứ hai. Theo "rule of three": chờ đến lần thứ ba lặp lại mới abstract.
  • SOLID over-engineering: một CRUD đơn giản mà có 6 lớp (controller → service → usecase → gateway → mapper → repository) cho một findById = đọc 6 file để hiểu 1 việc.
  • Tách SRP quá mịn: mỗi class 5 dòng, 40 class cho một flow → "shotgun surgery" ngược: thêm 1 field phải đụng 6 class.
  • DIP cho thứ không bao giờ đổi: abstract java.time.Clock thì hợp lý (test time), nhưng abstract String formatting thì thừa.

Quy tắc: abstraction phục vụ thay đổi đã thấy hoặc rất khả năng xảy ra. Không có trục thay đổi → YAGNI. Senior thể hiện ở chỗ biết dừng, kèm lý do.

🎯 3-tier interview framework

Tầng 1 — SurfaceMid gate

Q: Cho đoạn code, chỉ ra vi phạm SOLID.

Nói tên nguyên tắc + bằng chứng cụ thể: "method này thêm loại giảm giá phải sửa chuỗi if-else → vi phạm OCP", "class này vừa tính tiền vừa gửi mail → vi phạm SRP". Không nói chung "code xấu".

Tầng 2 — Deep DiveSenior gate

Q: Refactor method 200 dòng này từng bước.

Test phủ trước → Parameter Object → Strategy cho discount (OCP) → tách validation (SRP) → tách side-effect → Facade orchestrator. Chạy test sau mỗi bước. Nhấn mạnh "không đổi hành vi".

Tầng 3 — ArchitectureSenior+

Q: Khi nào KHÔNG nên áp dụng SOLID?

Khi không có trục thay đổi (YAGNI), code đơn giản/throwaway, 1 implementation duy nhất, hoặc abstraction làm tăng chi phí đọc > lợi ích. Rule of three. Senior cân bằng giữa flexibility và simplicity, không cargo-cult pattern.

🔧 Lab bắt buộc — refactor có kiểm soát (2 bài)

Lab 1 — OCP qua Strategy. Lấy một method tính khuyến mãi với if(type=="PERCENT")...else if(type=="FIXED").... Viết characterization test phủ các nhánh. Refactor sang DiscountRule interface + các implementation, inject list. Thêm một rule mới mà KHÔNG sửa code cũ. Tiêu chí: chứng minh test cũ vẫn xanh và rule mới chỉ thêm file.

Lab 2 — Nhận diện over-engineering. Lấy một đoạn code "SOLID quá đà" (interface + factory cho 1 implementation). Viết lại đơn giản hơn (xoá abstraction thừa) mà vẫn pass test. Tiêu chí: giải thích "rule of three" và vì sao bản đơn giản dễ đọc hơn ở quy mô hiện tại.

🧠 Feynman Test — giải thích cho người chưa biết (< 60s mỗi câu)
  1. SRP nói "một lý do để thay đổi" nghĩa là gì? Cho ví dụ một class vi phạm.
  2. OCP: "mở để mở rộng, đóng để sửa đổi" — refactor if-else thành cái gì để đạt OCP?
  3. Tại sao phải viết test TRƯỚC khi refactor? Khác gì với rewrite?
  4. Cho 2 ví dụ over-engineering SOLID và vì sao chúng hại.
  5. Tell-Don't-Ask là gì? Cho một cặp code Ask vs Tell.
Bí câu nào = quay lại đúng section. Trôi cả 5 = bạn vừa biết áp dụng vừa biết khi nào dừng — đúng tầm senior.
