#50 · Junior Foundation

Git Daily Workflow — Survival Kit Cho Junior

Junior dùng Git mỗi ngày 10+ lần, nhưng 70% chỉ biết 5 lệnh (clone/add/commit/push/pull) — khi gặp conflict / rebase / undo là hoảng loạn. File này biến Git từ "magic" thành mental model rõ ràng.

CORE ★ 1 tuần đọc + practice trên repo thật Domain: E-commerce / SaaS

⚠️ Cảnh báo "hiểu giả" — đặc trưng Git

Git là chủ đề "ai cũng biết" nhưng ít người hiểu sâu. Bạn rơi vào hiểu giả nếu:

  • Dùng git pull mỗi ngày nhưng KHÔNG giải thích được nó = git fetch + git merge hay git fetch + git rebase
  • Biết git rebase nhưng KHÔNG giải thích được "Quy tắc vàng" của rebase
  • Biết git reset --hard nhưng làm xong mất code rồi mới hỏi "reflog là gì"
  • Biết git merge conflict nhưng resolve bằng cách chọn 1 bên ngẫu nhiên
🚨 3 cấp độ — bạn đang ở đâu?

L1 (Recognize): dùng được clone/add/commit/push. Conflict = hoảng. Không biết reflog.
L2 (Explain): hiểu Git DAG, phân biệt merge vs rebase, dùng được interactive rebase, recover được commit đã reset.
L3 (Apply): design Git workflow cho team, viết commit hook, biết khi nào dùng squash vs no-ff, recover repo bị force-push xóa branch.

1. Bài toán thực tế — 5 tình huống junior vướng CORE ★

📖 Tình huống #1 — "Em pull mà mất hết code"

Bối cảnh: Junior code 3h chưa commit. Senior bảo "pull đi". Sau pull, code biến mất, replace bằng version trên remote.

Root cause: code chưa commit + git pull = merge từ remote vào working tree → conflict được auto-resolve theo strategy nguy hiểm, hoặc bị overwrite.

Đúng cách: luôn git status trước, commit hoặc stash trước khi pull.

📖 Tình huống #2 — "Em commit vào nhầm branch main"

Bối cảnh: Junior quên check branch hiện tại, commit 5 file vào main thay vì feature branch.

Recovery: chưa push → git reset HEAD~1 (giữ file), checkout branch mới, commit lại. Đã push → tạo branch mới từ HEAD, revert commit trên main.

📖 Tình huống #3 — "Em rebase mà sợ"

Bối cảnh: Senior nói "rebase trước khi merge", junior nghe xong né tránh, đẩy bừa rồi senior phải fix.

Lý do sợ: không hiểu rebase ghi lại lịch sử như nào, nghĩ "rebase = mất code". Thực tế rebase = replay commit của bạn lên đầu branch đích.

📖 Tình huống #4 — "Em merge conflict, không biết làm gì"

Bối cảnh: Conflict marker <<<<<<< xuất hiện. Junior xóa hết marker, save, commit. Code lệch hoàn toàn.

Đúng cách: đọc cả 2 bên, hiểu intent, merge logic — không phải xóa marker rồi commit liều.

📖 Tình huống #5 — "Em vô tình xóa branch, mất 200 dòng code"

Bối cảnh: Junior git branch -D feature/x để "dọn dẹp", quên rằng branch chưa merge. 2 ngày code biến mất.

Recovery: git reflog → tìm commit hash cuối của branch → git checkout -b feature/x <hash>. Reflog giữ 90 ngày mặc định.

⏸ Pause & Predict — trước khi đọc tiếp
Viết ra giấy:
  1. Tình huống #1: tại sao pull có thể mất code chưa commit? Cơ chế merge của Git làm gì với working tree?
  2. Tình huống #5: reflog hoạt động thế nào? Tại sao xóa branch không mất commit?
  3. Nếu push commit nhầm lên main và 5 người đã pull, làm gì? (Hint: git revert vs git reset --force — chọn cái nào?)
Đáp án ở Section 2 (mental model) và Section 7 (undo).

2. Mental Model — Git Là Cây Commit (DAG) CORE ★

Trước khi học command, phải hiểu Git lưu cái gì + branch là gì.

3 Khu vực của Git

Working tree     ←  file bạn đang sửa trên disk
   ↓ git add
Index (staging)  ←  "snapshot dự kiến cho commit tiếp"
   ↓ git commit
Local repo       ←  lịch sử commit trên máy bạn
   ↓ git push
Remote repo      ←  GitHub / GitLab / Bitbucket

Hiểu 3 khu vực này = hiểu 80% command. Mọi command Git đều di chuyển dữ liệu giữa 4 chỗ này.

Commit là snapshot, KHÔNG phải diff

Mỗi commit = (snapshot toàn bộ tree, parent commit, author, message, timestamp). Hash SHA-1 tính từ nội dung — đổi 1 byte = hash khác hoàn toàn → immutable.

commit a1b2c3 (parent: 999888)
├── README.md       (blob: hash1)
├── src/
│   ├── App.java    (blob: hash2)
│   └── User.java   (blob: hash3)
└── pom.xml         (blob: hash4)

→ Đổi User.java → commit mới có hash khác, tree mới, nhưng các blob unchanged vẫn share.

Branch là pointer, không phải copy

main      → commit C (head của main)
              ↓
              B → A (commit gốc)
                  ↑
feature   → commit E → D (head của feature, parent là B)

git checkout feature  → HEAD trỏ vào "feature" → working tree = snapshot của E
git checkout main     → HEAD trỏ vào "main"    → working tree = snapshot của C

Branch chỉ là 1 file text 40 ký tự (hash của commit head). Tạo branch = tạo pointer, không copy code → nhanh, rẻ.

HEAD là con trỏ hiện tại

HEAD = "branch tôi đang ở". Khi git commit, commit mới được tạo, branch pointer + HEAD đều move forward.

DAG — Directed Acyclic Graph

Mỗi commit có 1+ parent. Merge commit có 2 parent. Toàn bộ history = DAG. Visualize: git log --graph --oneline --all.

⏸ Pause & Predict
Trả lời thành lời (không nhìn dưới):
  1. Khi bạn git checkout feature, file trên disk có thay đổi không? Vì sao?
  2. Nếu bạn xóa branch feature, các commit của nó còn không?
  3. 2 commit có cùng nội dung file, hash có giống nhau không?
Đáp án ngay dưới.
✅ Đáp án
  1. Có — Git update working tree để match snapshot của commit feature trỏ vào. File chưa commit bị overwrite nếu conflict.
  2. Còn — commit là object trong .git/objects/, branch chỉ là pointer. Xóa pointer không xóa object. Reflog vẫn tham chiếu trong 90 ngày → recover được bằng git checkout -b name <hash>.
  3. KHÔNG — hash tính từ (snapshot tree + parent + author + timestamp + message). 2 commit cùng diff nhưng khác timestamp → khác hash. Đây là tại sao rebase tạo commit mới (parent khác).

3. Daily Commands — 80% Junior Cần CORE ★

Setup ban đầu (1 lần)

git config --global user.name "Nguyen Van A"
git config --global user.email "a@example.com"
git config --global init.defaultBranch main
git config --global pull.rebase false  # hoặc 'true' nếu team dùng rebase
git config --global core.editor "code --wait"  # VS Code cho commit message
git config --global core.autocrlf input  # macOS/Linux; "true" cho Windows

Workflow hằng ngày

# 1. Bắt đầu ngày — sync latest
git checkout main
git pull

# 2. Tạo branch mới
git checkout -b feature/checkout-flow

# 3. Code, save, check status thường xuyên
git status                # XEM 100 lần/ngày — quan trọng nhất
git diff                  # Xem diff chưa stage
git diff --staged         # Xem diff đã stage

# 4. Add + commit
git add src/Checkout.java         # add file cụ thể, KHÔNG dùng `git add .` bừa
git add -p                        # interactive add (chọn từng hunk)
git commit -m "feat(checkout): add cart validation"

# 5. Push lên remote
git push -u origin feature/checkout-flow  # -u set upstream lần đầu
git push                                  # lần sau

# 6. Mở Pull Request trên GitHub/GitLab UI

Xem lịch sử

git log --oneline -10               # 10 commit gần nhất
git log --graph --oneline --all     # DAG đầy đủ
git log -p src/User.java            # lịch sử thay đổi của 1 file
git log --author="An"               # commit của 1 người
git log --since="2 weeks ago"
git show <hash>                     # detail 1 commit
git blame src/User.java             # ai sửa dòng nào, khi nào

Chuyển branch / commit

git checkout main                  # về branch
git checkout <hash>                # về commit cụ thể (detached HEAD)
git switch main                    # Git 2.23+ — clearer hơn checkout
git switch -c new-branch           # tạo + chuyển

4. Branching Strategies SUPPLEMENTARY

GitHub Flow (đơn giản, phổ biến)

main (luôn deployable, có CI/CD auto deploy)
  ├── feature/checkout-validation
  ├── feature/order-history-pagination
  ├── bugfix/cart-total-mismatch
  └── hotfix/critical-payment-bug

Workflow: branch từ main → code → PR → review → merge → delete branch

Git Flow (cũ, phức tạp — đa số team không dùng nữa)

main, develop, feature/*, release/*, hotfix/* — quá nhiều branch, overhead lớn. Chỉ dùng khi có scheduled release cứng (game, embedded).

Naming convention branch

feature/checkout-validation       # tính năng mới
bugfix/cart-total-mismatch        # bug bình thường
hotfix/critical-payment-bug       # bug cần fix gấp lên prod
refactor/extract-payment-service  # refactor không thay đổi behavior
docs/update-api-readme            # tài liệu
chore/upgrade-spring-boot         # nâng cấp dependency

Commit message — Conventional Commits

<type>(<scope>): <subject>

feat(checkout): add cart validation for max items
fix(payment): handle Stripe timeout gracefully
refactor(user): extract email validator
docs(readme): add docker setup guide
chore(deps): bump spring-boot to 3.2.0
test(order): add integration test for cancel flow

Type: feat / fix / refactor / docs / test / chore / perf. Subject ≤ 50 ký tự, imperative ("add" không phải "added").

5. Merge / Rebase / Cherry-pick CORE ★

Merge — Tạo merge commit

git checkout main
git merge feature/checkout
# Tạo merge commit M, 2 parent (HEAD của main, HEAD của feature)
    A---B---C  main
         \      \
          D---E--M   ← merge commit, parent = C và E
                feature

Fast-forward merge

Trước:  A---B  main
              \
               C---D  feature

Sau ff: A---B---C---D  main, feature (cả 2 trỏ vào D)

Khi main không có commit mới sau khi feature tách ra → merge = move pointer, KHÔNG tạo merge commit. Dùng git merge --no-ff để force merge commit (giữ history rõ).

Rebase — Replay commit lên branch khác

git checkout feature
git rebase main
# Replay D, E lên đầu main → tạo D', E' với parent mới
Trước:  A---B---C  main
              \
               D---E  feature

Sau rebase:  A---B---C  main
                      \
                       D'---E'  feature (commit MỚI, hash khác)

🚨 Quy tắc Vàng của Rebase

KHÔNG bao giờ rebase commit đã push lên branch shared

Rebase tạo commit mới (hash khác). Nếu commit cũ đã push lên main và người khác pulled → rebase = lịch sử khác nhau giữa local + remote → mỗi người push xong sẽ ghi đè lên nhau, code biến mất, team chửi.

Rebase OK khi: commit chỉ ở local hoặc trên feature branch của bạn (không ai khác đang work trên đó).

Interactive Rebase — Cleanup history trước PR

git rebase -i HEAD~5  # interactive cho 5 commit gần nhất

# Editor mở ra:
pick a1b2c3 feat(checkout): add validation
squash d4e5f6 fix typo
squash g7h8i9 fix another typo
reword j0k1l2 refactor: extract method
drop m3n4o5 wip debug

# Commands: pick / squash (s) / fixup (f) / reword (r) / drop (d) / edit (e)

Dùng để: gộp commit "fix typo" / "wip", sửa commit message, xóa commit debug.

Cherry-pick — Lấy 1 commit từ branch khác

git checkout main
git cherry-pick <hash-commit-từ-branch-khác>
# Apply commit đó lên main, tạo commit mới

Use case: hotfix trên feature branch nhưng cần đẩy lên main ngay; backport bugfix sang release branch cũ.

Counter-example — Khi nào KHÔNG dùng

❌ KHÔNG dùng rebase khi…
  • Branch đã push và có người khác work trên đó (Quy tắc Vàng)
  • Bạn không tự tin: nếu lo sợ, dùng merge — an toàn hơn
  • Team chưa thống nhất convention: hỏi senior trước, mỗi team có style khác nhau
❌ KHÔNG dùng cherry-pick khi…
  • Có thể merge cả branch: cherry-pick từng commit = dễ miss commit, dễ sót dependency
  • Số lượng commit nhiều: nếu cần > 5 commit, dùng rebase hoặc merge

6. Conflict Resolution CORE ★

Conflict xảy ra khi 2 branch sửa cùng 1 dòng trong cùng 1 file, hoặc 1 bên xóa file mà bên kia sửa.

Bước 1: Xem file nào conflict

git status
# Both modified: src/User.java
# Both modified: src/Order.java

Bước 2: Mở file, resolve

// src/User.java
public class User {
<<<<<<< HEAD
    private String email;     ← code của branch hiện tại (main)
=======
    private String emailAddress;  ← code đến từ branch merge (feature)
>>>>>>> feature/rename-email
}

Đọc cả 2 bên, hiểu ý đồ, viết version đúng:

public class User {
    private String emailAddress;  // Quyết: rename theo feature branch, đồng nhất với DB column
}

Bước 3: Mark resolved + commit

git add src/User.java         # mark resolved
git status                    # verify "All conflicts fixed"
git commit                    # editor mở với message merge default — accept hoặc edit

Tools support

git mergetool                 # mở merge tool config (vimdiff, kdiff3, vscode...)
git config --global merge.tool vscode  # set vscode làm default

Abort nếu rối

git merge --abort       # undo merge, về state trước merge
git rebase --abort      # tương tự cho rebase
⚠️ Anti-pattern: xóa hết marker, commit liều

Junior thường thấy <<< đáng sợ → xóa marker + 1 trong 2 bên random + commit. Code chạy được nhưng logic sai.

Đúng: đọc cả 2 bên, hiểu why, ask senior nếu không chắc. 10 phút hỏi tốt hơn 4h debug.

7. Undo — 5 Tình huống Junior Hoảng Loạn CORE ★

A. Sửa file mà chưa add, muốn về như cũ
git restore src/User.java        # Git 2.23+
# hoặc cũ hơn:
git checkout -- src/User.java
B. Add rồi nhưng chưa commit, muốn unstage
git restore --staged src/User.java
# hoặc cũ hơn:
git reset HEAD src/User.java
C. Commit rồi nhưng message sai
git commit --amend -m "feat(checkout): correct message"
# Chỉ amend khi CHƯA push. Đã push = đổi hash = rewrite history shared.
D. Commit nhầm vào main (CHƯA push)
# Tạo branch giữ commit
git branch feature/work-in-progress

# Reset main về commit trước
git reset --hard HEAD~1

# Chuyển sang branch mới có commit
git checkout feature/work-in-progress
E. Đã push commit nhầm lên main (NGUY HIỂM)

KHÔNG dùng git push --force trên main — sẽ xóa commit của người khác đã push sau bạn, team chửi.

# Dùng revert — tạo commit "ngược lại"
git revert <hash-commit-nhầm>
git push

Revert tạo commit mới đảo lại change → an toàn cho shared branch. History dài hơn nhưng đúng.

🆘 Bonus: Reflog cứu mạng

# "Em vô tình `git reset --hard` mất 3h code"
git reflog                          # xem mọi HEAD movement (90 ngày)
# Output:
# a1b2c3 HEAD@{0}: reset: moving to HEAD~3
# d4e5f6 HEAD@{1}: commit: WIP checkout flow  ← commit em cần
# g7h8i9 HEAD@{2}: commit: validation

git checkout d4e5f6                 # về commit đó (detached HEAD)
git checkout -b recover-branch     # tạo branch để giữ

Reflog giữ 90 ngày mặc định, kể cả khi branch bị xóa. Mọi action local đều log.

⏸ Pause & Predict
Trả lời:
  1. Khi nào git reset --hard mất code KHÔNG recovery được? (Hint: liên quan add)
  2. Push commit có data PII (password) lên public repo. Xóa thế nào? git push --force đủ không?
✅ Đáp án
  1. Reset --hard mất code KHI: file chưa được git add bao giờ (working tree only, không có trong objects). Reflog chỉ recovery được commit/staged content.
  2. Force push KHÔNG đủ — git history bị rewrite local nhưng GitHub vẫn cache, mọi clone trước đó vẫn có commit. Phải: rotate password ngay, dùng BFG Repo-Cleaner để rewrite history, contact GitHub support để force purge cache.

8. Stash + .gitignore SUPPLEMENTARY

Git Stash — Cất tạm

# Đang code dở, cần pull / switch branch
git stash                       # cất tạm working tree + index
git stash push -m "WIP checkout"  # với message
git checkout main
git pull                        # OK, không conflict
git checkout feature
git stash pop                   # lấy lại + xóa khỏi stash
# hoặc
git stash list                  # xem các stash
git stash apply stash@{0}       # apply nhưng giữ stash
git stash drop stash@{0}        # xóa stash

.gitignore — File KHÔNG commit

# Java
*.class
target/
*.jar
*.war

# IDE
.idea/
*.iml
.vscode/
*.swp

# Logs
logs/
*.log

# OS
.DS_Store
Thumbs.db

# Secrets
.env
application-secret.yml
*.pem
*.key

# Dependencies
node_modules/

Quy tắc: ignore mọi file generated (build output, IDE config), mọi file secret. Nếu lỡ commit file secret: revert + rotate secret + xóa khỏi history bằng BFG.

9. Enterprise Workflow ADVANCED

PR Workflow chuẩn

  1. Branch từ main latest: git checkout main && git pull && git checkout -b feature/x
  2. Code, commit (small, descriptive messages)
  3. Push, mở PR trên GitHub/GitLab
  4. CI chạy: tests, linter, security scan
  5. Code review: ít nhất 1 approver (2 cho production code)
  6. Fix review comments, push tiếp (CI re-run)
  7. Squash + merge (hoặc rebase + merge tùy team) → delete branch

Pre-commit Hooks

# .git/hooks/pre-commit (executable)
#!/bin/bash
# Run linter trước commit
mvn spotless:check || exit 1
mvn test || exit 1
echo "✅ Pre-commit checks passed"

Dùng pre-commit framework để share hooks giữa team qua file .pre-commit-config.yaml.

Sign Commit (Enterprise sometimes requires)

git config --global commit.gpgsign true
git config --global user.signingkey <gpg-key-id>
git commit -S -m "feat: add audit log"
# Sign verify trên GitHub: green badge "Verified"

Use case: compliance, audit trail, ngăn commit giả mạo identity (impersonation attack).

🎯 3 Cấp Độ Mastery CORE ★

Git Daily Workflow

L1 — Recognize (Junior)

Biết clone/add/commit/push/pull. Conflict = hoảng, gọi senior. Không biết reflog. Đôi khi commit nhầm main.

Symptom: mỗi conflict tốn 30 phút + sự giúp đỡ. Đã từng mất code vì reset --hard.

L2 — Explain (Mid)

Hiểu DAG, phân biệt merge vs rebase, dùng interactive rebase để cleanup history, dùng stash khi switch branch dở, recover commit bằng reflog.

Symptom: tự resolve conflict trong 10 phút, biết khi nào dùng merge vs rebase theo Quy tắc Vàng.

L3 — Apply (Senior)

Design Git workflow cho team (PR template, branch protection, CODEOWNERS), viết pre-commit hooks, recover repo bị force-push xóa branch, dùng git bisect tìm commit bug.

Symptom: debug 1 regression bằng git bisect ra commit gây bug trong 5 phút thay vì bisect tay.

Merge vs Rebase

L1

Mặc định dùng merge. Tránh rebase vì "sợ". Đôi khi rebase nhầm branch shared → team complain.

L2

Hiểu rebase = replay commit + tạo hash mới. Áp dụng Quy tắc Vàng. Dùng interactive rebase cho local branch trước PR.

L3

Team standardize policy: feature → rebase trước, squash khi merge vào main. Tùy chỉnh theo team style. Train junior tránh anti-pattern.

Recovery & Reflog

L1

Không biết reflog. Mất code = chấp nhận viết lại.

L2

Recover commit bị reset bằng reflog. Recover branch bị xóa bằng git checkout -b <hash>.

L3

Build script auto-backup branch trước force-push. Recovery từ FSCK orphan objects. Đào tạo team không panic.

🚨 Symptom → Debug Recipes CORE ★

🚨 Recipe #1 — "Push bị reject: non-fast-forward"

Symptom: ! [rejected] main -> main (non-fast-forward)

Root cause: Remote có commit mới mà local chưa có. Push của bạn sẽ ghi đè → Git từ chối.

Fix: git pull --rebase (hoặc git fetch && git rebase origin/main) → resolve conflict nếu có → push lại.

Anti-fix: KHÔNG dùng git push --force trên main — sẽ xóa commit người khác.

🚨 Recipe #2 — "Detached HEAD state"

Symptom: You are in 'detached HEAD' state khi checkout commit hash.

Nghĩa là: HEAD trỏ vào commit, không phải branch. Commit mới không thuộc branch nào — dễ mất khi switch.

Fix nếu cần giữ work: git switch -c new-branch-name ngay. Nếu chỉ xem: git switch - về branch trước.

🚨 Recipe #3 — "Merge conflict mỗi lần pull"

Symptom: Mỗi git pull đều có conflict ở cùng file (ví dụ package-lock.json, pom.xml).

Root cause: File generated được commit + 2 người regenerate cùng lúc.

Fix:

  1. Add file vào .gitignore nếu không cần commit (ví dụ build output)
  2. Nếu phải commit (ví dụ lock file): team agree về ai regenerate khi nào, dùng .gitattributes với merge driver "ours" hoặc "theirs"
🚨 Recipe #4 — "Branch xóa nhầm, mất code"

Symptom: git branch -D feature/x chưa merge → mất code.

Recover:

git reflog                           # tìm last commit của branch
# Output: a1b2c3 HEAD@{5}: checkout: moving from feature/x to main
git checkout -b feature/x a1b2c3     # recreate branch từ commit đó

Nếu reflog không có (branch chưa từng checkout local): git fsck --lost-found tìm orphan commit.

🚨 Recipe #5 — "Repo bị corrupt sau crash hệ điều hành"

Symptom: fatal: object <hash> is corrupted

Fix:

  1. Clone lại từ remote: git clone <url> ở thư mục khác, copy lại working changes
  2. Nếu remote cũng có vấn đề: git fsck --full để xem damaged objects
  3. Backup objects từ teammate (cùng team clone): copy .git/objects/<hash[:2]>/<hash[2:]> sang

🔧 4 Exercises BẮT BUỘC CORE ★

🧪 Exercise #1 — Daily Workflow (Easy, 30min)

Tạo repo local, làm full workflow:

  1. init, set user.name/email, add remote (GitHub repo trống)
  2. Tạo file, commit 5 lần với message theo conventional commits
  3. Push, kiểm tra GitHub thấy đủ 5 commit
  4. Tạo branch feature/x, commit thêm 2 lần, push, mở PR (vẫn trống nhưng có)
🧪 Exercise #2 — Conflict Resolution (Medium, 1h)

Tạo 2 branch sửa cùng dòng:

  1. main: sửa README.md dòng 1 thành "Hello world"
  2. feature/greeting: sửa cùng dòng thành "Xin chào"
  3. Merge feature vào main → conflict
  4. Resolve thành "Hello world / Xin chào" (giữ cả 2)
  5. Verify git log --graph show merge commit
🧪 Exercise #3 — Undo Scenarios (Hard, 1h)

Practice từng scenario:

  1. Sửa file, git restore về cũ
  2. Add file, git restore --staged để unstage
  3. Commit rồi --amend sửa message
  4. Commit 3 lần, reset --hard HEAD~3, recovery bằng reflog
  5. Tạo branch, commit, xóa branch -D, recovery bằng reflog
🧪 Exercise #4 — Interactive Rebase (Hard, 1.5h)

Practice cleanup history:

  1. Tạo branch, commit 5 lần: 1 feature thật + 4 "fix typo"
  2. git rebase -i HEAD~5
  3. Squash 4 typo vào feature commit
  4. Reword message cho clear
  5. Verify git log giờ chỉ có 1 commit clean
  6. Bonus: git bisect tìm commit gây bug giả lập

🎤 Interview Framework CORE ★

Tier 1 — Surface

Q: git pull khác git fetch như nào?

fetch = download latest từ remote vào local (origin/main), KHÔNG update branch local. pull = fetch + merge (hoặc rebase nếu config). Fetch an toàn hơn, pull có thể conflict.

Q: git revert khác git reset?

revert tạo commit mới đảo ngược change → an toàn cho shared branch (history forward). reset di chuyển HEAD pointer → rewrite history → chỉ dùng local.

Tier 2 — Deep Dive

Q: Quy tắc Vàng của rebase?

KHÔNG rebase commit đã push lên branch shared. Lý do: rebase tạo commit mới với hash khác. Người khác đã pull commit cũ → sau khi bạn force-push rebase, history họ lệch → conflict khủng + có thể mất commit.

Q: Reflog là gì? Khi nào dùng?

Reflog = log local mọi movement của HEAD và branch pointers (90 ngày default). Dùng để recover commit/branch sau khi reset --hard hoặc branch -D nhầm. Chỉ local — không trên remote.

Tier 3 — Architecture

Q: Design Git workflow cho team 20 dev microservices

1. Trunk-based hoặc GitHub Flow: main luôn deployable, feature branch ngắn (1-3 ngày).

2. Branch protection: main require PR + 1 review + CI pass + linear history (force rebase trước merge).

3. CODEOWNERS: auto-assign review theo service ownership.

4. Conventional commits + auto changelog: tool như release-please tự tạo version + changelog.

5. Pre-commit hooks: lint + test + secret scan (gitleaks).

6. Squash merge: PR = 1 commit clean vào main, history dễ đọc.

7. Hotfix process: branch từ tag release, fix, cherry-pick về main.

🧠 Feynman Test — 5 câu giải thích

  1. Branch trong Git là gì? Tại sao tạo branch không tốn dung lượng?
  2. Phân biệt working tree, index, repo. Mỗi git command move data giữa đâu?
  3. Quy tắc Vàng của rebase + tại sao quan trọng?
  4. Reflog dùng để làm gì? Cho 2 ví dụ cụ thể cứu mạng.
  5. Khi nào dùng revert vs reset? Liên quan shared branch thế nào?
